## 一、关键发现提炼与行动分级

### P0 — 立即采用（本周执行）

#### P0-8 ｜ Claude 稳定性 Fallback 策略设计
- **发现**：Claude Opus 4.6 出现大面积错误（HN 双帖讨论 + Anthropic 状态页确认），HN 用户反映"几乎每天都宕机"
- **风险**：我们 ACP 主力模型是 Claude，频繁宕机直接影响子 Agent 执行和项目交付
- **行动**：
  1. 在 ACP 任务调度层设计自动 fallback 机制：Claude 不可用时切换 GPT-5.4
  2. 结合 P0-5（GPT-5.4 mini/nano 评估），确定备选模型矩阵
  3. 监控本周 ACP 任务的 Claude 错误率
- **负责**：Leader + 技术架构专家
- **截止**：2026-03-21

#### P0-9 ｜ claude-mem 持久记忆插件集成评估
- **发现**：[claude-mem](https://github.com/thedotmack/claude-mem)（37,616⭐）— Claude Code 持久记忆插件，自动捕获会话上下文并压缩，跨会话注入，**已原生支持 OpenClaw 集成**
- **价值**：可能替代或增强我们现有的手动 memory/ 管理，减少 Leader 的记忆维护负担
- **行动**：安装并在一个 ACP 任务中试用，对比手动记忆管理效果
- **负责**：Leader
- **截止**：2026-03-22

#### P0-10 ｜ 功能正确性验证补盲
- **发现**：Harness Engineering 文章指出"内部质量好 ≠ 业务功能正确"——当前行业（包括我们）过度关注 lint/doc/type-check，忽视业务场景端到端验证
- **风险**：ACP 产出通过技术检查但业务逻辑不对，用户发现时已经太晚
- **行动**：在质量测试流程中增加"业务场景端到端验证"检查项，至少覆盖关键用户路径
- **负责**：Leader + 测试专家
- **截止**：2026-03-24

### P1 — 本周评估（形成方案）

#### P1-5 ｜ OpenViking 分层上下文架构深度研究
- **发现**：[OpenViking](https://github.com/volcengine/OpenViking)（15,130⭐）字节火山引擎出品，文件系统范式统一管理记忆/资源/技能，L0/L1/L2 三级按需加载
- **与我们的关系**：和我们的 HOT/WARM/项目级记忆架构理念一致但实现更系统化，有可视化检索轨迹
- **评估点**：是否可引入其分层管理理念优化我们的 memory/ 结构
- **截止**：2026-03-24

#### P1-6 ｜ forge-orchestrator 契约先行模式对标
- **发现**：[forge-orchestrator](https://github.com/tarunms7/forge-orchestrator)（34⭐但理念高度吻合）— 任务分解 → 接口契约生成 → 并行 Agent 执行 → 多级审查 → 自动 PR
- **与我们的关系**：与我们的"对抗式协作 + 指令包 7 层结构"高度一致，可借鉴其契约自动生成能力
- **评估点**：其接口契约生成逻辑能否提取为 Skill
- **截止**：2026-03-25

#### P1-7 ｜ OpenSpec 规格驱动开发工作流借鉴
- **发现**：[OpenSpec](https://github.com/Fission-AI/OpenSpec)（31,586⭐）— 规格驱动开发框架，`/propose → /apply → /archive` 三步工作流
- **与我们的关系**：我们的指令包 7 层结构+对抗式协作是 SDD 变体，OpenSpec 的标准化工作流可进一步提升指令包质量
- **截止**：2026-03-25
