## 一、AI Agent 组织管理

### 1.1 多 Agent 框架格局（2026 年 3 月）

行业已收敛出 **三大架构范式**，且正快速趋同：

| 框架 | 架构范式 | GitHub Stars | 核心特点 |
|------|----------|-------------|---------|
| **LangGraph** (LangChain) | 有状态图 | 100K+ (含 LangChain) | 检查点、时间旅行调试、人机回环、条件路由 |
| **AutoGen / AG2** (Microsoft) | 对话式 Agent | 54,700+ | 事件驱动、异步、GroupChat 协调、.NET 支持 |
| **CrewAI** | 角色制团队 | 44,500+ | 最快原型速度、YAML 配置、角色+目标+背景故事 |
| **OpenAI Agents SDK** | Handoff 链 | — | 生产级 Handoff 模式，锁定 OpenAI 模型 |
| **Anthropic Agent SDK** | 宪法 AI 安全 | — | Claude 模型深度集成，安全第一 |
| **Google ADK** | — | — | 2025.04 发布，Google 生态集成 |

**关键洞察**：
- **CrewAI 的角色制（Role-based）范式与我们 NDHY Agent Team 的架构高度类似**：每个 Agent 定义角色(role)、目标(goal)、背景故事(backstory)，通过"Crew"协作执行任务
- CrewAI 的三种执行模式（sequential/hierarchical/consensual）对应我们的串行链/分层调度/共识决策
- **但 CrewAI 的局限也是我们需要警惕的**：缺乏检查点（我们用文件系统弥补）、错误处理粗粒度、Agent 间直接通信能力弱

### 1.2 Claude Code Agent Teams — 直接影响我们

**核心发现**：Anthropic 在 Opus 4.6 发布时原生支持了 Agent Teams（实验性功能），这与我们通过 OpenClaw ACP 实现的多 Agent 协作直接相关。

**Subagents vs Agent Teams 对比**：

| 维度 | Subagents | Agent Teams |
|------|-----------|-------------|
| 通信模式 | 只能向父报告 | 成员间直接通信 |
| 协调方式 | 父 Agent 做中介 | 共享任务列表 + 邮箱系统 |
| Token 消耗 | 4-7x 单 Agent | ~15x 单 Agent |
| 适用场景 | 独立并行任务 | 跨层协调、对抗式验证 |

**对我们的影响**：
- 我们当前的 ACP 任务中已使用 `USE AGENT TEAMS` 指引，方向正确
- Agent Teams 的"共享任务列表"机制可以参考改进我们的项目状态跟踪
- **15x Token 消耗是关键约束**——需要严格区分何时用 subagent（独立任务）vs agent team（跨层协调）

## 二、提示词/上下文工程

### 2.1 从 Prompt Engineering → Context Design

**核心趋势**（SDG Group 报告，2026.03）：
- Prompt Engineering 正在演变为 **Context Design**
- 传统提示词是"静态指令"，Context Design 是"动态信息生态系统"
- Context Designer 需要编排来自多源的数据流，随交互动态更新
- 不再只是设定语气和角色，而是**管理一个持续演化的状态窗口**

**对我们的启示**：
- 我们的 SOUL.md / AGENTS.md / STANDARDS.md 体系已经在做 Context Design
- 但可以进一步进化：从"静态文件加载"到"动态上下文编排"
- 项目上下文的注入策略（指令包7层结构）本质就是 Context Engineering

### 2.2 Harness Engineering — 新概念（重要）

**Epsilla 博客（2026.03.12）总结的核心观点**：

> "工程的主战场正在从写代码迁移到设计环境、约束、反馈循环和治理机制。"

**Agent Harness = LLM 的操作系统**：
- 如果 LLM 是 CPU，Harness 就是 OS
- Harness Engineering ≠ Prompt Engineering
- 目标不是"这次做对"，而是"系统性地持续做对"

**核心要素**：
1. **上下文组织** — 结构化信息注入
2. **架构约束** — 单向依赖流（Types→Config→Repo→Service→Runtime→UI）
3. **自动验证** — 超越 lint/格式检查，验证业务行为正确性
4. **持续治理** — 可审计、可恢复、可扩展

**AGENTS.md 文件的谬误**（原文直指）：
> "一个综合性的 AGENTS.md 文件足以充当 Harness"是危险的误解。静态 markdown 规则会快速腐化，约束必须是结构化的、可验证的、可回收的。

**对我们的反思**：
- 我们正是在走 Harness Engineering 的路——AGENTS.md + SOUL.md + STANDARDS.md + Skill 体系
- 但文章指出的"AGENTS.md 谬误"值得警惕：我们的设定体系是否在持续腐化？
- **缺失的一环**：功能正确性验证（我们目前以代码质量检查为主，行为正确性验证不够）

### 2.3 AI 编码 Agent 的架构趋同

Dave Patten（Medium，2026.03.13）观察到所有主流编码 Agent 正在趋同：

**共同架构要素**：
1. **Repo Memory** — CLAUDE.md / AGENTS.md / GEMINI.md
2. **Tool Use** — Git、Shell、Test Runner、Browser
3. **Sub-Agents** — Planning / Coding / Testing / Review
4. **Long-Running Execution** — 分钟到小时级任务循环

**三种形态**：CLI-First（Claude Code）→ IDE-Native（Cursor）→ Cloud Engineering（Devin/Codex Cloud）

## 四、教育 AI

### 4.1 🔥 OpenMAIC — 清华开源 L4 级 AI 课堂（2 小时前发布）

**这是本次调研最重要的发现之一。**

- **来源**：清华大学 MAIC 团队，2 年研发
- **开源地址**：https://github.com/THU-MAIC/OpenMAIC
- **X 上数小时引发十几万人关注**

**核心能力**：
- 上传 PDF/讲义/论文 → 几分钟自动生成完整互动课堂
- **Agentic 课堂引擎**：AI 老师（TTS 语音讲解）+ AI 助教（关键知识点补充）+ AI 同学（随机提问引发讨论）
- 自动生成：Slides + Quiz + PBL 网页
- 成本：1 个课件 + 2 美元 + 30 分钟

**验证数据**：
- 清华校内随机对照实验：AI 教师效果 > 传统慕课，在激发主动讨论上甚至 > 真人授课
- 课程结业率 **40%**（传统 MOOC 不到 5%）
- 2025 年已上线教育部国家智慧教育平台，累计访问 2000 万次

**与 NDHY 的关联**：
- 我们做教育产品，这个项目的理念和技术路线与我们高度相关
- "多 Agent 课堂"模式（AI 老师 + AI 助教 + AI 同学）可直接参考
- Agentic 课堂引擎的设计思路可能是我们产品的重要参考

## 五、关键发现与建议

### 🔑 5 条对组织有价值的发现

**1. Harness Engineering 是我们正在做的事情的学术名称**
- 我们的 SOUL.md + AGENTS.md + STANDARDS.md + Skill 体系 = Agent Harness
- 但需要警惕"静态 markdown 腐化"风险，需要定期审计和更新
- **行动建议**：增加"功能正确性验证"环节，不只检查代码质量还要验证业务行为

**2. Claude Code Agent Teams 的 15x Token 消耗需要精细管理**
- 我们已经在用 Agent Teams，但成本控制需要更精细
- **行动建议**：建立 Token 消耗基线，区分 subagent 场景 vs agent team 场景

**3. OpenMAIC 对教育产品有直接参考价值**
- 多 Agent 课堂（老师+助教+同学）是被清华验证过的有效模式
- 40% 结业率 vs MOOC 5% — 互动是关键差异
- **行动建议**：深入研究 OpenMAIC 源码和论文

**4. Figma-Context-MCP 可以加速设计到代码的链路**
- 13,751 Stars，社区活跃
- **行动建议**：评估是否接入我们的开发工具链

**5. 编码 Agent 架构趋同意味着我们的 ACP 投资方向正确**
- Memory Files + Tool Use + Sub-Agents + Long-Running = 行业共识
- 我们通过 OpenClaw + Claude Code ACP 已经实现了这个架构
