## 一、问题全清单（按严重程度分类）

### 🟡 改进级（9 个）

#### 来自官网代码审查（7 个）

| # | 问题 | 模块 | 根因分析 |
|---|------|------|---------|
| Y-1 | Canvas 粒子动画缺少移动端性能保护，O(n²) 连线算法在低端设备掉帧 | 前端/动画 | **开发时只在桌面端调试**，未做移动端性能 profiling；动画实现优先视觉效果，未考虑设备分层 |
| Y-2 | Pipeline 动画 `setInterval` 未清理，离开视口后仍每 400ms 执行 DOM 操作 | 前端/动画 | **资源释放意识不足**——用了 IntersectionObserver 启动但未用它控制停止，只断开了 observer 自身 |
| Y-3 | 数字计数器对 "0 行" 无动画效果，缺少"归零"的视觉冲击 | 前端/交互 | 计数器统一逻辑未对"从 0 到 0"这一特殊情况做差异化处理 |
| Y-4 | CSP 策略中 `'unsafe-inline'` 削弱 XSS 防护 | 安全/部署 | **单文件架构的技术债**——CSS/JS 全部内联导致必须使用 unsafe-inline，是架构选型的代价 |
| Y-5 | HTML 文件缓存策略 1 小时过长，更新后用户看到旧版本 | 部署/缓存 | **缓存策略一刀切**——静态资源和 HTML 用了同一个 matcher 和同一个缓存时间 |
| Y-6 | 组织基建分组与 content-v2.md 定义不完全一致（3 组合并为 1 组） | 内容/一致性 | **缺少"文案契约"的约束意识**——开发为了视觉紧凑做了合并，但未确认文案是否允许偏差 |
| Y-7 | `X-XSS-Protection` 头已被现代浏览器废弃 | 安全/配置 | **安全头配置基于过时知识**——照搬旧模板，未验证各头的当前浏览器支持状态 |

## 二、根因归类与模式识别

从 19 个问题中识别出 **6 种根因模式**：

| 模式 | 出现次数 | 涉及问题 | 核心教训 |
|------|---------|---------|---------|
| **资源未释放/未清理** | 1 | Y-2 | setInterval/setTimeout/listener 启动后必须有对应的清理机制 |
| **框架默认行为覆盖契约** | 2 | Y-8, Y-9 | 使用验证框架（如 Pydantic）时，必须确认其默认错误响应是否与 API 契约一致 |
| **配置一刀切** | 2 | Y-5, Y-7 | 缓存、安全头等配置需按资源类型差异化，不能所有文件同策略 |
| **仅桌面端验证** | 1 | Y-1 | 展示型项目也必须做移动端性能基线测试 |
| **文案/契约一致性缺失** | 1 | Y-6 | 设计稿与文案契约有差异时必须确认，不能默默偏离 |
| **Mock/开发态残留** | 2 | G-7, G-8 | 发布前必须有"开发态清理"检查项，环境切换不应需要改代码 |

## 三、可复用防范 Checklist

> 以下 checklist 可直接用于未来项目的质量门控

### ✅ 前端动画/性能 Checklist

- [ ] Canvas/WebGL 动画是否有移动端降级策略？（粒子数减少 / 直接跳过）
- [ ] 是否支持 `prefers-reduced-motion` 媒体查询？
- [ ] 所有 `setInterval` / `setTimeout` 是否有对应的清理逻辑？
- [ ] 不可见区域的动画是否暂停？（用 IntersectionObserver 控制启停）
- [ ] O(n²) 及以上复杂度算法是否有移动端保护？
- [ ] 计数器/过渡动画是否处理了边界值（0→0、相同值等）？

### ✅ API 契约一致性 Checklist

- [ ] 验证框架（Pydantic/Zod 等）的默认错误响应格式是否与 API 契约一致？
- [ ] 框架级校验返回的 HTTP 状态码是否与契约约定一致？（特别是 422 vs 400）
- [ ] 自定义错误码是否在框架校验之前/之后都能正确返回？
- [ ] 空值/空数组的校验是否返回契约约定的专用错误码？
- [ ] 测试用例是否覆盖了"框架先于业务逻辑拦截"的场景？

### ✅ 部署/安全配置 Checklist

- [ ] HTML 文件与静态资源的缓存策略是否分开？（HTML 短缓存，资源长缓存）
- [ ] CSP 策略是否必须使用 `unsafe-inline`？如果必须，是否确认无 eval/innerHTML 风险？
- [ ] 安全头是否逐个验证了当前浏览器支持状态？（特别是 X-XSS-Protection 已废弃）
- [ ] Docker 日志是否配置了轮转和大小限制？
- [ ] 容器安全配置是否包含 read_only + no-new-privileges + 资源限制？

### ✅ 内容/文案一致性 Checklist

- [ ] 页面内容是否与文案契约/PRD 逐条对照验证？
- [ ] 展示优化（合并/精简）是否获得了文案负责人确认？
- [ ] `aria-label` 等辅助文本语言是否与页面语言一致？
- [ ] Open Graph / Twitter Card 元数据是否齐全？
- [ ] Footer 是否包含年份和版权信息？

### ✅ 发布前"开发态清理" Checklist

- [ ] Mock 开关是否通过环境变量控制，而非代码硬编码？
- [ ] Mock 数据与后端真实数据的差异是否已记录？
- [ ] 发布前是否完成了 `USE_MOCK=false` 的端到端验证？
- [ ] 开发用 console.log / debug 代码是否已清理？
- [ ] 环境切换是否零代码改动？

## 五、两份报告的共性教训

1. **"有条件放行"模式有效但需跟踪闭环**：两份报告都给出了"有条件通过"的结论，这说明需要一个机制跟踪"条件"的落实——否则条件会被遗忘。建议：每次"有条件放行"都生成一个跟踪 issue 列表。

2. **框架默认行为是隐形陷阱**：Pydantic 422 vs 契约 400 的问题说明，使用任何框架都应该在第一时间验证其默认行为是否与业务契约一致，而不是在测试阶段才发现。

3. **静态分析不能替代运行时验证**：官网审查通过代码分析发现了 setInterval 未清理，但提示词工程的"前端在 Mock 模式"意味着真实链路从未被运行过——静态通过 ≠ 运行时正确。

4. **安全配置需要定期审查**：X-XSS-Protection 已废弃、unsafe-inline 的权衡——安全头不是配一次就完的，需要随浏览器标准演进定期更新。

5. **"60 分版本"的质量底线**：提示词工程明确是 60 分版本，接受了无前端测试的现状。这是合理的 trade-off，但必须在报告中明确标注风险——"知道自己在冒什么险"比"不知道自己缺什么"好得多。
