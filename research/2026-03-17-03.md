## 一、AI Agent 组织管理

### 1.1 Claude Code Agent Teams（⭐⭐⭐⭐⭐ 重大进展）

Anthropic 在 Opus 4.6 发布时推出了 **Agent Teams** 实验性功能，这是多 Agent 协作领域的里程碑事件。

**核心能力**：
- 一个 session 作为 team lead，协调多个 teammate
- Teammate 之间可以**直接通信**（不需要经过 lead 中转）
- 共享任务列表（TaskList），Agent 可自行认领任务
- 每个 teammate 独立上下文窗口，各自负责自己的 scope

**vs Subagent 的关键区别**：
- Subagent = 你派出去跑腿的外包，只能向你汇报
- Agent Teams = 坐在同一间办公室的项目团队，可以互相沟通、挑战对方假设、协同收敛

**实战案例**：有人用 5 个 Agent Teams 运行了一个"游戏工作室"，Agent 自行认领任务、协调工作。关键发现：
- Agent 闲置时间是痛点 → 需要在 prompt 中加入"准备工作"指引
- 全 Opus 成本很高：6 agents × Opus = 显著 token 消耗

**对 NDHY 的启示**：
> 我们的 ACP-First + subagent 架构与 Agent Teams 理念高度一致。但原生 Agent Teams 提供了更强的 teammate 间直接通信能力。建议在复杂开发任务中试用 Agent Teams，观察与我们当前 subagent 串行/并行模式的效率差异。

## 二、Context Engineering（上下文工程）

### 2.1 学术论文：Context Engineering 成为独立学科（⭐⭐⭐⭐⭐）

arXiv 论文 [2603.09619]：**"Context Engineering: From Prompts to Corporate Multi-Agent Architecture"**

**核心框架**：提出了一个 4 层金字塔成熟度模型：
1. **Prompt Engineering (PE)** — 制作单次查询
2. **Context Engineering (CE)** — 设计、结构化和管理 Agent 决策的整个信息环境
3. **Intent Engineering (IE)** — 将组织目标、价值观、权衡层级编码到 Agent 基础设施中
4. **Specification Engineering (SE)** — 创建机器可读的企业策略和标准语料库

**5 个上下文质量标准**：相关性、充分性、隔离性、经济性、溯源性

**关键发现**：75% 的企业计划 2 年内部署 Agentic AI（Deloitte 2026），但部署后又撤回，因为面临扩展复杂性。Klarna 案例展示了"上下文缺陷 + 意图缺陷"的双重风险。

> **"控制 Agent 上下文的人控制其行为；控制其意图的人控制其战略；控制其规范的人控制其规模。"**

**对 NDHY 的启示**：
> 这篇论文的框架与我们的知识治理协议高度对齐！我们的"设定=宪法，技能=操作手册"就是 CE + SE 的实践。建议深读此论文，可能发现优化方向。

### 2.2 从 Prompt Engineer 到 Context Designer

SDG Group 文章指出 2026 年的关键转变：
- **传统 PE**：制作一条指令获取单次响应
- **Context Design**：构建动态信息生态系统，上下文窗口不是快照而是演进状态

这正是我们在做的事——SOUL.md / AGENTS.md / MEMORY.md 构成的就是 Agent 的"动态信息生态系统"。
