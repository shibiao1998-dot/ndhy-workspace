### 3. Harness Engineering — 从模型到控制系统的范式转移
- **OpenAI 论文**：小团队通过 Agent 构建 100 万行代码、1500 个自动 PR，零手写代码
- **核心观点**：Agent 的产出速度已超过人类审查能力，瓶颈从"写代码"转移到"设计控制系统"
- **AGENTS.md 谬误**：单一 Markdown 文件不够，需要结构化、可验证、可回收的约束系统
- **更强模型 ≠ 更松约束**：Agent 自主性越强，操作环境反而需要更严格的约束
- **功能正确性验证缺口**：当前实践过度关注内部质量（lint/doc），忽视业务功能正确性

## 三、关键趋势提炼

### 趋势 1：Context Engineering 取代 Prompt Engineering
> "2025 的框架是 Prompt Engineering，2026 的框架是 Context Engineering"

**核心变化**：从「如何写好一个提示词」转向「如何管理系统指令+检索知识+工具定义+对话摘要+任务元数据的完整上下文」。这不是改名，是本质性的范式升级。

**对我们的意义**：我们的 SOUL.md + AGENTS.md + Skills + memory/ 架构本质上就是 Context Engineering。我们已经在实践前沿。

### 趋势 2：Spec-Driven Development 成为主流
GSD（31,990⭐）和 OpenSpec（31,586⭐）两个项目在不到半年内各积累 3 万+ star，说明"先规格后编码"成为 AI 辅助开发的共识。

**对我们的意义**：我们的指令包 7 层结构+对抗式协作本质上就是 SDD 的变体。可以借鉴 OpenSpec 的 `/propose → /apply → /archive` 工作流进一步标准化。

### 趋势 3：Agent 控制系统 > Agent 本身
Epsilla 的 Harness Engineering 文章和 OpenAI 论文共同指出：Agent 吞吐量已超越人类审查能力，关键竞争力从「模型能力」转向「控制系统设计」。

**对我们的意义**：我们的分级验收、心跳巡查、ACP 铁律就是 Harness Engineering 的实践。但 Harness Engineering 提出的「功能正确性验证缺口」值得警惕——我们也可能过度关注内部质量而忽视业务正确性。

### 趋势 4：Claude Code Agent Teams 成为标配
多篇文章和教程表明 Agent Teams（多个 Claude 实例共享代码库、自主协调）已从实验走向主流。关键实践：
- 主 Agent 做团队领导：分解→委派→验证
- 共享任务列表和邮箱系统实现协调
- 每个 teammate 有聚焦范围，避免上下文膨胀

**对我们的意义**：我们已在 AGENTS.md 中强制 ACP 开发任务使用 Agent Teams，方向正确。

### 趋势 5：持久记忆成为 Agent 核心基础设施
claude-mem（37,616⭐）和 OpenViking（15,130⭐）的爆发说明「跨会话记忆」是 Agent 系统的核心痛点。OpenViking 的 L0/L1/L2 三级加载和我们的 HOT/WARM/项目级记忆架构高度相似。

## 五、对组织的 5 条核心收获

1. **🔑 我们的架构方向被行业验证**——Context Engineering、Spec-Driven Development、Harness Engineering 三大趋势与我们现有实践高度吻合。我们不是在追随，而是在引领。

2. **⚠️ 功能正确性验证是盲区**——Harness Engineering 指出的"内部质量好≠业务功能正确"问题，我们也存在。建议在质量测试流程中增加「业务场景端到端验证」环节。

3. **🛠️ claude-mem 值得集成评估**——37K+ star 的 Claude Code 持久记忆插件，已原生支持 OpenClaw。可能替代/增强我们现有的手动记忆管理。

4. **📊 OpenViking 的分层上下文架构值得研究**——字节出品，L0/L1/L2 三级加载与我们的 HOT/WARM/项目级类似但更系统化，有可视化检索轨迹。

5. **🏗️ forge-orchestrator 的契约先行模式可借鉴**——先生成接口契约再让 Agent 并行编码的设计，与我们的"对抗式协作"理念高度一致，且已有成熟实现。
