## 一、多 Agent 协作与团队管理

### 1.1 学术突破：Context Engineering 正式成为独立学科

**论文**: [Context Engineering: From Prompts to Corporate Multi-Agent Architecture](https://arxiv.org/abs/2603.09619) (2026-03-10)

核心观点：
- **Context Engineering (CE)** 被正式定义为独立学科——设计、结构化和管理 AI Agent 决策所处的整个信息环境
- 提出 **5 个上下文质量标准**：相关性 (relevance)、充分性 (sufficiency)、隔离性 (isolation)、经济性 (economy)、溯源性 (provenance)
- 构建 **4 层金字塔成熟度模型**：Prompt Engineering → Context Engineering → Intent Engineering → Specification Engineering
- 75% 的企业计划在 2 年内部署 Agentic AI（Deloitte 2026），但 KPMG 报告称部署"先猛后缩"——组织面对扩展复杂性时退缩
- **关键洞察**：「控制 Agent 的上下文就控制了它的行为；控制它的意图就控制了它的策略；控制它的规格就控制了它的规模」

**对我们的启示** 🔥：
- 我们的 SOUL.md / AGENTS.md / SKILL.md 三层体系本质上就是 CE + IE + SE 的实践
- 论文验证了我们"设定=宪法, 技能=操作手册, references=深度细节"的渐进式披露架构是正确方向
- 但我们缺少"Specification Engineering"——将组织政策和标准编码为机器可读规格的层级

### 1.3 Claude Code Agent Teams（实验性功能）

**来源**: [Claude Code Agent Teams 完全指南](https://claudefa.st/blog/guide/agents/agent-teams) (2026-03-13)

- Anthropic 随 Opus 4.6 发布了原生 Agent Teams 功能
- **核心区别**：subagent 是"分别派出的承包商"，Agent Teams 是"坐在同一间办公室的项目团队"
- 特性：共享任务列表、直接消息传递、独立上下文窗口、团队 Lead 协调
- **最佳场景**：研究探索（多角度调研）→ 协作阶段用 Agent Teams；实现阶段用 builder-validator 模式
- 文章明确提到"社区已在用 OpenClaw 和自定义编排脚本构建类似模式"

**对我们的启示** 🔥：
- 我们当前的 sessions_spawn + subagent 模式 = 承包商模式
- Agent Teams 的"共享任务列表+直接通信"模式值得关注——当前我们的子 Agent 之间无法直接通信
- 但 Agent Teams token 消耗显著更高，需要权衡

## 二、Context Engineering / Prompt Management

### 2.1 从 Prompt Engineering 到 Context Design

**来源**: [The Evolution of Prompt Engineering to Context Design in 2026](https://www.sdggroup.com/en/insights/blog/the-evolution-of-prompt-engineering-to-context-design-in-2026) (2026-03-11)

- Prompt Engineer 角色正在转型为 **Context Designer**——帮助 AI Agent 更好地理解上下文并高效自动化任务
- 不再是"写一条好的提示词"，而是"架构化地设计 AI 的信息环境"

### 2.2 GSD v2 的上下文管理实践

GSD-2（1,338 星）的演进特别值得关注：
- **v1**（prompt 框架）→ **v2**（Agent 应用程序）
- 核心解决问题：每个任务启动全新 session（上下文隔离）、程序化注入精确文件、Git worktree 隔离、崩溃恢复
- **"One command. Walk away. Come back to a built project with clean git history."**

**对我们的启示** 🔥：
- 我们的 MEMORY.md + memory/*.md 分层记忆体系 = GSD 的 .gsd/ 文件系统
- 但 GSD 做了我们还没做的：**成本追踪、卡顿检测（stuck loop detection）、崩溃恢复**
- GSD 的"每任务清空上下文"策略验证了我们 subagent-first 的做法是对的

## 六、对组织的关键发现与建议

### 🔥 Top 5 价值洞察

| # | 发现 | 对我们的意义 | 建议行动 | 关联专家 |
|---|------|-------------|---------|---------|
| 1 | **Context Engineering 成为独立学科**，提出 5 质量标准 + 4 层金字塔成熟度 | 我们的三层知识架构（设定/技能/references）被学术验证为正确方向 | 补齐"Specification Engineering"层——将组织规范编码为机器可读规格 | Leader、Skill 开发专家 |
| 2 | **Agent Teams 模式 vs Subagent 模式**：前者支持直接通信+共享任务，后者只能回报不能协作 | 我们当前是 subagent 模式，复杂任务中子 Agent 间信息传递依赖 Leader 中转 | 评估是否需要引入 Agent Teams 模式用于探索阶段任务（如需求挖掘、竞品调研） | Leader、技术架构专家 |
| 3 | **GSD-2 的上下文管理**：每任务独立 session + 程序化文件注入 + 卡顿检测 + 崩溃恢复 | 我们的子 Agent 超时处理已有规范，但缺少成本追踪和系统级卡顿检测 | 研究 GSD-2 的 stuck loop detection 机制，看能否应用到我们的子 Agent 巡查 | Leader、技术架构专家 |
| 4 | **Figma Code to Canvas**：Claude Code → Figma 设计稿的双向流 | 可大幅缩短"设计→开发→设计审查"的闭环周期 | 体验设计专家和前端开发专家应学习此工作流 | 体验设计专家、前端开发专家 |
| 5 | **教育 AI 政策加速**：教育部系列规范 + AI 课件工具爆发 | 教育是我们的目标领域，政策方向确认，工具链正在形成 | 教育领域专家关注政策文件和工具链进展 | 教育领域专家 |
