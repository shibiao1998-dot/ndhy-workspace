## 一、Harness Engineering：Agent 管理的范式转变

### 核心发现

**行业共识正在形成：Agent 的质量瓶颈不在模型，在 Harness。**

本周三篇重量级文章同时聚焦 "Harness Engineering"——围绕模型构建控制系统的工程实践：

1. **LangChain 博文《The Anatomy of an Agent Harness》**（2026-03-16）
   - 定义：`Agent = Model + Harness`，如果你不是模型，你就是 harness
   - Harness 包括：System Prompt、Tools/Skills/MCP、基础设施（文件系统/沙箱/浏览器）、编排逻辑（子 Agent 派发/路由）、Hooks/中间件
   - **关键洞察**：文件系统是最基础的 harness 原语——持久化存储、增量产出、多 Agent 协作表面
   - LangChain 正研究：数百 Agent 并行共享代码库编排、Agent 自我分析 trace 修复 harness 缺陷、动态即时组装工具和上下文
   - 🔗 https://blog.langchain.com/the-anatomy-of-an-agent-harness/

2. **HumanLayer 博文《Skill Issue: Harness Engineering for Coding Agents》**（2026-03-12）
   - **核心论点**："不是模型问题，是配置问题"——期待 GPT-6 解决一切是幻觉
   - Harness 的四个杠杆：System Prompt、Tools/MCP、Context、Sub-agents
   - HumanLayer 额外强调两个杠杆：**Hooks**（自动化集成+确定性控制流）和 **Skills**（渐进式知识披露）
   - **子 Agent 最佳实践**："frontend engineer / backend engineer / data analyst" 式角色分工**不 work**。有效的是将子 Agent 用作"上下文防火墙"——隔离离散任务到独立上下文窗口，防止中间噪音累积
   - Harness Engineering 是 Context Engineering 的子集
   - 🔗 https://www.humanlayer.dev/blog/skill-issue-harness-engineering-for-coding-agents

3. **Epsilla 博文《Harness Engineering: Why the Focus is Shifting from Models to Agent Control Systems》**（2026-03-12）
   - Prompt Engineering 是战术层（单次优化），Harness Engineering 是战略层（系统级控制）
   - 目标不是让 Agent "这次做对"，而是设计让 Agent "持续做对"的控制系统
   - 🔗 https://www.epsilla.com/blogs/2026-03-12-harness-engineering

4. **学术论文《Building Effective AI Coding Agents for the Terminal》**（arXiv 2603.05344, 2026-03-05）
   - OPENDEV 框架：Scaffolding + Harness + Context Engineering
   - 强制显式推理阶段 + 上下文效率优先
   - 🔗 https://arxiv.org/abs/2603.05344

### 🎯 对 NDHY 的价值

| 发现 | 我们的现状 | 改进方向 |
|------|-----------|---------|
| Agent = Model + Harness | 我们的 SOUL.md + STANDARDS.md + Skills 就是 Harness 的一种实现 | ✅ 方向正确，需要系统化命名和管理 |
| 子 Agent 角色分工不 work，要当上下文防火墙用 | 我们的 AGENTS.md 已定义角色分工模式 | ⚠️ 需审视：是否应更关注"上下文隔离"而非"角色扮演" |
| Skills = 渐进式知识披露 | 我们的 Skill 体系已实现 | ✅ 完全一致，行业验证了我们的方向 |
| Hooks 用于确定性控制流 | 我们缺少自动化 hook 机制 | 🔴 可探索：Claude Code hooks 集成 |
| 文件系统是最基础协作表面 | 我们用文件做记忆/协作 | ✅ 方向正确 |

## 二、Claude Code Agent Teams：原生多 Agent 协作

### 核心发现

**Anthropic 随 Opus 4.6 发布 Agent Teams 实验特性**——多个 Claude Code 实例原生协作。

- **InfoQ 报道**（2026-03-12）：Opus 4.6 引入 Adaptive Reasoning 和 Context Compaction，用于长时间运行的 Agent。Agent Teams 作为研究预览发布。
- **claudefast 完整指南**（2026-03-14）：
  - Team Lead 协调，Teammates 独立工作
  - vs Subagents：**Subagents 是承包商，各自跑腿；Agent Teams 是坐同一间办公室的团队**
  - 关键区别：Teammates 可直接互相发消息、共享发现、挑战彼此假设，不需通过主 Agent 中转
  - 适用场景：研究+评审、新模块开发（各自负责独立组件）、竞争性假设调试
  - **社区之前已用 OpenClaw 等工具独立构建类似模式**，Anthropic 看到需求后内建
- **Claude Flow**：开源编排框架，多 Claude Agent 协调工作流

### 🎯 对 NDHY 的价值

| 发现 | 影响 |
|------|------|
| Agent Teams = 并行执行 + 主动协调 | 我们的 ACP-First + Agent Team 策略与行业方向完全一致 |
| OpenClaw 被点名为社区先行者 | 我们的技术栈选择得到行业认可 |
| Context Compaction 解决长任务内存问题 | 可缓解我们 ACP 任务的 context overflow 问题 |
| Opus 4.6 + Agent Teams | 评估是否升级 Claude Code 到支持 Agent Teams 的版本 |

## 三、Context Engineering → Context Design：提示工程的进化

### 核心发现

**行业正式从 "Prompt Engineering" 过渡到 "Context Engineering / Context Design"。**

- **SDG Group《The Evolution of Prompt Engineering to Context Design in 2026》**：
  - Prompt Engineer 角色没有消失，而是进化为 **Context Designer**
  - 传统提示工程：设计单条静态指令获取单个响应
  - Context Design：编排多源动态信息生态系统，上下文窗口不再是快照而是持续演化的状态
  - 2022→2026 演进：单条提示 → Chain-of-Thought → 记忆与历史 → 上下文关联 → Context Design
  
- **instinctools《Context Engineering in AI: Techniques, Best Practices》**（2026-03-11）：
  - Context Engineering 的核心策略和技术最佳实践
  - 中间件基础设施层简化 Agent 上下文工程

### 🎯 对 NDHY 的价值

我们的 SOUL.md/AGENTS.md/STANDARDS.md + Skills 分层 + 渐进式披露架构，**本质上就是 Context Design**。行业在 2026 年 Q1 正式命名和系统化了我们已在实践的东西。

## 七、关键洞察与行动建议

### 🔑 Top 5 发现

1. **Harness Engineering 是 2026 年 Agent 领域最重要的概念转变**
   - 模型智能是基座，harness 让智能可用。我们的设定/技能/记忆体系是 harness 的优秀实现。
   
2. **子 Agent 角色分工的反直觉发现**
   - HumanLayer 经验：按"前端/后端/数据分析师"分角色子 Agent **不 work**。有效模式是子 Agent 作为"上下文防火墙"——隔离任务的上下文，防止父线程积累噪音。
   - **⚠️ 这挑战了我们当前 27 专家的角色化设计假设**，需要认真评估。

3. **Claude Code Agent Teams 是原生多 Agent 协作的里程碑**
   - 我们已用 OpenClaw + ACP 构建了类似能力。官方原生支持意味着更稳定、更高效。

4. **Context Engineering 正式取代 Prompt Engineering**
   - 我们的渐进式披露架构（设定→技能→References）就是 Context Design 的实践。行业验证。

5. **中国教育 AI 市场 380 亿，备课效率是第一痛点**
   - 与 NDHY 教育方向直接相关的市场信号。
