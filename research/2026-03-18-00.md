## 一、AI Agent 组织管理

### 1.1 行业趋势：从 Big Model 到 Big Harness

本周最重要的信号来自 **Harness Engineering**（线束工程）这个新概念的爆发：

- **OpenAI 论文**（2026年2月）：《Harness Engineering: Leveraging Codex in an Agent-First World》——一个小团队5个月内用 Agent 生成了100万行代码、约1500个自动化PR，**没写一行手动代码**
- **核心论点**：竞争优势不再属于拥有最大模型的人，而属于拥有最有效 harness 的人
- **Andrej Karpathy 名言**：「Don't tell it what to do, give it success criteria and watch it go.」
- **Epsilla 总结**：Agent harness = 包裹在 AI 模型外围的基础设施层，管理长时任务。如果 LLM 是 CPU，harness 就是操作系统

**⚠️ 关键警告——AGENTS.md 谬误**：
> Epsilla 文章明确指出：「一个综合性的 AGENTS.md 文件就是足够的 harness——这是危险的误解。单一 markdown 规则文件是脆弱的方案，会随系统演化迅速过时。操作约束必须是结构化的、可验证的、可回收的，不能活在静态文档里。」

**🔥 对我们的启发**：我们的 AGENTS.md + SOUL.md + STANDARDS.md 模式正处于这条演化路径上，但还不够。需要将约束从「静态文档」向「可验证的结构化约束」演进。

### 1.3 Claude Code Agent Teams（重要！）

Anthropic 随 Opus 4.6 发布了 **Agent Teams** 实验功能：
- **核心区别**：subagent = 独立承包商各干各的；Agent Team = 坐在同一间屋子里的项目组
- **能力**：成员间直接通信、共享任务列表、主动协调，无需主Agent做中介
- **最佳实践**：探索阶段用 Agent Teams（协作探索），实现阶段用 builder-validator 模式（质量门禁）
- **适用场景**：研究+审查、新模块开发、竞争假设调试、大型重构
- **注意**：协调开销大，Token消耗远高于单会话

### 1.4 新工具值得关注

- **rune-plugin**（GitHub）：Discipline Engineering for Claude Code — Plan, Work, Review, Audit using Agent Teams
- **doey**（GitHub）：tmux-based multi-agent Claude Code teams CLI tool
- **aperture**：确定性注意力和判断引擎，用于Agent系统

## 二、Context Engineering / 提示词工程演进

### 2.1 从 Prompt Engineering 到 Context Design

SDG Group 发布了一篇重要文章《The Evolution of Prompt Engineering to Context Design in 2026》：

- **演化路径**：2022 基础提示 → 2023 CoT思维链 → 2024 记忆&历史 → 2025 上下文链接 → **2026 Context Design**
- **核心转变**：从「设计单次指令获取一次响应」变为「编排多源数据流的动态信息生态系统」
- **Context Designer** 的工作不再是写好一条 prompt，而是管理一个随交互演化的状态——整合用户偏好、历史消息、变化中的话题细微差别

### 2.2 "Design the Arena, Not the Agent"

Whimsey Labs 创始人的文章，核心观点：

- AI 发展4阶段：搜索 → 副驾驶 → Agent → **组织**
- 每个阶段的价值重心不同：搜索优化模型，副驾驶优化集成，Agent优化harness，**组织优化协调框架**
- **Beads 模式**：Git-backed issue tracking，用结构化JSON在 `.beads` 目录存储工作状态，持久化在Agent上下文窗口之外。解决了Agent"自己勾掉没完成的任务"的经典问题

### 2.3 RCCF 框架（2026最新提示词最佳实践）

Role（角色）→ Context（上下文）→ Constraints（约束）→ Format（格式）

> 「现代LLM已经会推理了。它们不知道的是你的具体约束。」

## 六、对组织的关键发现和行动建议

### 🔥 发现1：Harness Engineering 是我们的核心能力方向

我们的 AGENTS.md / SOUL.md / STANDARDS.md / Skills 体系**本质上就是一个 Agent Harness**。但行业正在指出：静态markdown规则文件是脆弱的，需要向「可验证的结构化约束」演进。

**行动**：评估当前设定体系的可验证性，探索将关键约束从「规则描述」变为「可自动检查的断言」。

### 🔥 发现2：Agent Teams 是我们已在实践的模式

Claude Code Agent Teams 的设计理念——主Agent协调、成员间直接通信、共享任务列表——和我们的 Leader + 专家团队模式高度一致。我们是这个方向的早期实践者。

**行动**：持续优化我们的ACP调度模式，借鉴 Agent Teams 的「探索阶段协作 + 实现阶段门禁」最佳实践。

### 🔥 发现3：Context Engineering > Prompt Engineering

行业共识已从「写好prompt」转向「设计好context」。我们的渐进式披露（设定→技能→references）、记忆分层（HOT/WARM/PROJECT/EMERGENCY）正是 Context Engineering 的实践。

**行动**：参考 RCCF 框架审视我们的指令包结构是否完善。

### 🔥 发现4：Beads 模式值得借鉴

Git-backed 结构化任务状态持久化（存在Agent上下文窗口之外），解决Agent「虚假完成」问题。我们的 PROJECT.md + HEARTBEAT.md 是类似尝试，但可以更结构化。

**行动**：研究 Beads 模式，评估是否升级我们的任务跟踪机制。

### 🔥 发现5：Google Stitch 对教育产品设计有直接价值

免费的AI原型工具，即将支持React导出。对我们的教育产品快速原型阶段很有价值。

**行动**：5月Google I/O后评估Stitch正式版能力。
