> **提取日期**：2026-03-16

## 一、10 大协作关系的沟通时机和方式

### 1. 产品↔开发

| 沟通点 | 时机 | 方式 |
|--------|------|------|
| 需求评审 | Sprint Planning 前 1-2 天 | 需求评审会（走读 PRD + AC） |
| 可行性反馈 | 评审中即时 + 评审后 24h 内书面确认 | 异步文档评审 |
| 变更管理 | 变更发生时**立刻**通知 | 变更日志（原因+影响+审批人） |
| 持续澄清 | 开发过程中随时 | 即时消息（先查已有文档） |

**关键实践**：Definition of Ready（DoR）— 需求进入开发前必须满足条件清单

### 2. 产品↔测试

| 沟通点 | 时机 | 方式 |
|--------|------|------|
| AC 对齐 | 需求评审阶段（不是开发完了） | AC 走读（测试视角审查） |
| 测试策略 | Sprint 开始时 | 确定本轮测试重点 |
| 缺陷分级 | 缺陷报告时即判定 | 争议时 24h 内三方裁决 |
| UAT | 功能开发完 + 内部测试通过后 | UAT 检查清单 |
| Go/No-Go | 发布前 1 天 | 残留缺陷风险评估 |

**关键实践**：测试左移（Shift Left）— 测试在需求阶段就介入

### 3. UX/UI↔产品

| 沟通点 | 时机 | 方式 |
|--------|------|------|
| 需求对齐 | PRD 草稿完成后、正式评审前 | 设计 Brief（目标用户+场景+约束+标准） |
| 设计方向 | 画稿前确认方向正确 | 情绪板/概念草图 |
| 设计评审 | 低保真→高保真→终稿，至少 2 轮 | 渐进式评审 |
| 设计变更 | 需求变更时同步通知 UX | 评估设计影响 |

**关键实践**：设计决策记录（DDR）— 记录为什么选 A 不选 B，避免翻烧饼

### 4. UX/UI↔开发

| 沟通点 | 时机 | 方式 |
|--------|------|------|
| Design Handoff | 设计评审通过后、开发开始前 | Figma 标注 + Handoff 文档 |
| 技术约束 | Handoff 时或更早 | 开发评估实现成本 |
| Design Token | 项目初期定义，后续增量更新 | 代码化设计变量 |
| 设计走查 | 每个功能开发完成、提测前 | 走查 Checklist（逐屏对比） |

**关键实践**：全状态设计 — 每个页面/组件必须包含 5 种状态（默认/加载中/空/错误/成功）

### 5. UX↔视觉

| 沟通点 | 时机 | 方式 |
|--------|------|------|
| 交互稿交接 | 交互评审通过后立即 | 标注交互稿（hover/click/状态转换） |
| 视觉方向 | 项目初期确定 | 情绪板（Mood Board） |
| 视觉评审 | 初稿→修改→终稿，2-3 轮 | 视觉规范文档 |
| 一致性审查 | 视觉终稿确定后 | 跨页面统一审查 |

**关键实践**：设计系统（Design System）— 统一的视觉组件库 + 使用规范

### 6. 测试↔开发

| 沟通点 | 时机 | 方式 |
|--------|------|------|
| 缺陷报告 | 发现后立即（P0/P1 额外即时通知） | 缺陷追踪系统（结构化记录） |
| 缺陷确认 | 24h 内响应（P0 2h 内） | 确认/拒绝/需更多信息 |
| 修复通知 | 代码合并后主动通知 | 测试可以验证的通知 |
| 验证闭环 | 修复版本部署后 24h 内 | 修复验证结果 |

**关键实践**：缺陷报告模板 — 强制包含复现步骤、环境、预期/实际结果、截图/录屏

### 7. 架构↔各角色

| 对象 | 沟通内容 | 时机 |
|------|---------|------|
| 产品 | 技术约束、能力边界、技术债影响 | 需求评审阶段就介入 |
| 开发 | 架构决策（ADR）、技术规范 | 决策确定后 48h 内发布 |
| 测试 | 非功能需求测试策略 | 架构方案确定后 |
| UX | 技术限制对交互的影响 | 设计阶段 |

**关键实践**：ADR 标准化 — 所有重大架构决策必须有 ADR（Context → Decision → Consequences）

### 8. 代码审查↔开发

| 沟通点 | 时机 | 方式 |
|--------|------|------|
| PR 提交 | 功能完成 + 自测通过 + CI 绿灯后 | PR 描述模板 |
| 审查响应 | 提交后 4h 内开始审查 | 反馈分级：🔴Must Fix / 🟡Suggestion / 🟢Nit / 💚Praise |
| 修改回复 | 收到反馈后 24h 内 | 修改说明或讨论澄清 |
| 审查轮次 | 目标 ≤2 轮完成 | 超过 3 轮说明问题 |

**关键实践**：小 PR 原则 — 每个 PR ≤300 行变更，超过就拆分

### 9. 项目管理↔全员

| 沟通点 | 频率 | 方式 |
|--------|------|------|
| 站会 | 每日，≤15min | 异步文字更新状态 |
| 周会 | 每周 | 本周进展、下周计划、风险预警 |
| 阻塞升级 | 阻塞 2h 未解决时**立刻** | 结构化升级模板 |
| Sprint 回顾 | Sprint 结束后 24h 内 | Start-Stop-Continue |

**关键实践**：异步优先 — 能异步解决的不开会；阻塞 SLA（2h 自助→4h PM 介入→8h 管理层介入）

### 10. 全部角色的统一交接规范

所有角色间交接遵循 **DoR + DoD + 交接 Checklist** 三件套：
- **DoR**（Definition of Ready）：下游开始前，上游产出必须满足的条件
- **DoD**（Definition of Done）：上游交付时，产出必须达到的标准
- **交接 Checklist**：双方确认的交接物完整性检查清单

## 二、常见反模式及其解决方案

### 2.1 五大核心反模式

| 反模式 | 表现 | 根因 | 破解 |
|--------|------|------|------|
| **甩过墙** | 产出做完直接扔给下游，不管对方能不能接住 | 角色间缺乏 Overlap | 交接时双方 Walkthrough；DoR/DoD 门槛；交接 Checklist |
| **信息孤岛** | 关键信息只存在于个人/小组中 | 沟通渠道不畅、没有共享机制 | 单一事实源（SSOT）；跨角色评审；共享知识库 |
| **过度会议** | 太多同步会议，执行时间被挤占 | 异步协作能力弱、信任度低 | 异步优先；会议准入标准（RACI）；No-Meeting Day |
| **验收不对齐** | 交付物"完成"了但不符合期望 | AC 模糊、DoD 不统一 | DoD 团队统一；Three Amigos 对齐；过程中 Demo |
| **HiPPO** | 最高薪者意见自动胜出，不论证据 | 权力等级压制 | 数据驱动决策、ADR 记录决策依据 |

### 2.2 其他高频反模式

| 反模式 | 表现 | 破解 |
|--------|------|------|
| **Gold Plating（镀金）** | 加入用户没要求的功能/精度 | 严格按 AC 交付，超出范围走变更流程 |
| **Bike Shedding** | 简单问题讨论很久，重要问题快速通过 | 时间盒讨论，重要议题先行 |
| **Hero Culture** | 依赖个人能力而非流程 | 知识共享、流程标准化、Bus Factor > 1 |
| **Silent Disagreement** | 不同意但不说，执行时打折扣 | 安全发言环境、明确反对是义务不是挑衅 |

## 三、AI Agent 团队特别需要注意的协作要点

### 3.1 AI 团队 vs 人类团队关键差异

| 维度 | 人类团队 | AI Agent 团队 |
|------|---------|---------------|
| 沟通渠道 | 面对面、视频、IM 多种混合 | **全部通过文本指令和文档** |
| 上下文连续性 | 人脑持续记忆（有遗忘） | **每次会话从零开始**（靠文件记忆） |
| 并行度 | 受限于人数和注意力 | **可高度并行**，但需依赖管理 |
| 模糊理解 | 人可以"意会"模糊指令 | **AI 需要精确指令**，模糊→偏差 |
| 质量方差 | 人类有状态波动 | **AI 质量受指令质量和上下文质量决定** |

### 3.2 五大适配原则

| 原则 | 说明 |
|------|------|
| **文档即沟通** | 所有"会议"场景转化为结构化文档交接。评审 = 文档走读 + 结构化评论 |
| **上下文即信息对称** | 每次 Spawn 时必须提供完整上下文（SSOT），不能假设 Agent "还记得" |
| **契约即验收标准** | AC 必须精确到可自动验证。模糊的 AC 在 AI 团队会直接导致偏差 |
| **流转即交接协议** | 每对角色间的交接需要定义明确的交接物格式和完整性检查清单 |
| **异步是天然优势** | AI Agent 没有"开会疲劳"，但过度串行会浪费并行优势 |

### 3.3 人类实践→AI 团队适配映射

| 人类团队实践 | AI 团队适配 |
|-------------|-------------|
| Three Amigos 会议 | 三角色顺序评审 + 结构化反馈汇总 |
| 每日站会 | 项目管理专家定期读取各任务状态 |
| 设计评审会 | 设计稿 + 评审 Prompt → 结构化评审意见 |
| Bug Triage 会议 | 自动化缺陷分级规则 + 人工裁决争议 |
| 结对审查 | 独立审查 + 对话式讨论（审查者↔开发者 Agent 交互） |
| Sprint 回顾 | Leader 记忆系统中的模式识别和规则晋升 |

## 四、信息交接的标准格式和验收标准

### 4.1 交接物格式要求（按角色对）

| 上游→下游 | 交接物 | 格式要求 |
|-----------|--------|---------|
| 产品→UX | 设计 Brief | 目标用户、核心场景、约束条件、成功标准 |
| 产品→开发 | PRD + AC | AC 必须 SMART 化（具体/可衡量/明确/相关/可测） |
| UX→视觉 | 交互稿 | 线框图 + 交互说明（hover/click/状态转换标注） |
| 视觉→前端 | Design Handoff | Design Tokens + 组件规格（5 种状态）+ 交互规范 + 文案规范 + 标注 + 切图 |
| 架构→开发 | ADR + 技术规范 | Context → Decision → Consequences |
| 开发→审查 | PR | 变更说明 + 关联 Issue + 自检清单 + CI 绿灯 |
| 测试→开发 | 缺陷报告 | 复现步骤 + 环境信息 + 预期/实际结果 + 截图 |
| 开发→测试 | 修复通知 | 修复说明 + 影响范围 + 回归建议 |

### 4.2 验收标准（通用）

每个交接物的验收检查：
1. **完整性** — 所有必要信息都包含，没有遗漏
2. **一致性** — 与上游产出无矛盾
3. **可消费性** — 下游能直接使用，不需要额外澄清
4. **可验证性** — 有明确的通过/不通过标准

### 4.3 缺陷报告状态机

```
New → Confirmed → In Progress → Fixed → Verified → Closed
```

每个状态有明确的负责人和时限：
- New → Confirmed：24h 内（P0 2h 内）
- Fixed → Verified：24h 内
- 完整链路应在 3 个工作日内闭环（P0 除外，P0 当天闭环）
