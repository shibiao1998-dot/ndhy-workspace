## 一、Leader 调度时最容易犯的错误

### 1.1 顺序错误速查

| 错误 | 正确做法 | 违反的依赖类型 |
|------|---------|---------------|
| 产品定义没做就开始体验设计 | 产品定义 → 体验设计 | 产出-输入 |
| 体验设计没做就开始视觉设计 | 体验设计 → 视觉设计 | 产出-输入 |
| 技术架构没做就开始数据库设计 | 技术架构 → 数据库设计 | 决策依赖 |
| API 契约没定义就开始前后端开发 | API 设计 → 前后端开发 | 接口依赖 |
| 前后端没联调就开始系统测试 | 联调集成 → 系统测试 | 质量门控 |
| 代码审查没通过就合并代码 | 代码审查 → 合并 | 质量门控 |
| 需求不清楚就开始方案设计 | 需求分析 → 方案设计 | 认知依赖 |
| 没有回滚方案就发布上线 | 部署准备（含回滚）→ 发布 | 安全依赖 |

### 1.2 并行错误速查

| 错误 | 类型 | 正确做法 |
|------|------|---------|
| Phase 0 调研全做完才开始 Phase 1 | 过度串行 | 用户研究、市场评估、竞品分析可并行 |
| 所有设计做完才开始所有开发 | 过度串行 | 模块级滚动交付——A 模块设计完就开始 A 的开发 |
| 前端和后端串行开发 | 过度串行 | 有 API 契约后可并行，前端用 Mock |
| 体验设计和技术架构串行 | 过度串行 | 产品定义完成后两者可并行 |
| 体验设计和视觉设计并行 | 过度并行 | 必须串行——视觉需要交互骨架 |
| 数据库设计和后端开发并行 | 过度并行 | 后端 Repository 层依赖数据模型 |
| 联调和系统测试并行 | 过度并行 | 必须串行——联调不通测不了功能 |

### 1.3 反模式速查（12 条）

| 反模式 | 一句话危害 | 破解 |
|--------|----------|------|
| 跳过需求直接开发 | 做出来不是用户要的 | 即使看起来清楚，也过需求验证门 |
| 没设计就写代码 | 架构混乱、返工严重 | 先出架构方案和 API 契约 |
| 视觉设计先于体验设计 | 好看但不好用 | 先完成信息架构和交互设计 |
| 全部并行 | 互相等待、频繁返工 | 按依赖关系排序，只并行无依赖的 |
| 全部串行 | 项目周期过长 | 识别无依赖的任务并行执行 |
| 没 API 契约就前后端并行 | 联调时接口对不上 | 先定义 API 契约再并行 |
| 跳过代码审查 | 质量隐患、技术债务 | 必须至少一轮代码审查 |
| 联调前不做单元测试 | 联调问题难以归因 | 各模块先过自己的测试 |
| 边开发边改需求 | 需求蔓延、永远做不完 | 需求变更走正式流程，评估影响 |
| Leader 替代专家做决策 | 外行指挥内行 | 专业的事交给专业的角色 |
| 没有回滚方案就发布 | 出问题后手忙脚乱 | 发布前必须准备回滚方案 |
| 测试只测正常路径 | 上线后边界异常爆发 | 覆盖异常、边界、回归场景 |

## 二、串并行规则核心原则

### 2.1 三条铁律

1. **不理解就不设计，不设计就不开发，不测试就不发布** — 每步依赖前一步产出，不能跳步
2. **有契约才能并行，无契约必须串行** — 前后端并行的前提是 API 契约已定义
3. **专业的事交给专业的角色** — 执行顺序由流程设计专家设计，任务拆解由项目管理专家执行

### 2.2 可以并行的 5 个条件（全部满足才可并行）

| 条件 | 说明 |
|------|------|
| **无数据依赖** | B 不需要 A 的产出作为输入 |
| **无资源冲突** | A 和 B 不争夺相同资源 |
| **有明确接口** | 并行双方约定好交互接口 |
| **失败独立** | A 失败不会导致 B 也失败 |
| **可独立验收** | A 和 B 各自有独立的验收标准 |

### 2.3 必须串行的 5 个条件（任一成立就必须串行）

| 条件 | 说明 |
|------|------|
| **产出-输入依赖** | B 需要 A 的产出 |
| **决策依赖** | B 的执行取决于 A 的决策结果 |
| **共享状态** | A 和 B 修改同一数据/资源 |
| **质量门控** | B 的开始需要 A 通过质量检查 |
| **认知依赖** | 做 B 之前需要从 A 中获取理解 |

### 2.4 有条件并行（最容易出错的场景）

| 场景 | 并行前提 | 必须做的事 |
|------|---------|-----------|
| 体验设计 ∥ 技术架构 | 产品定义已完成 | 两者基于同一产品定义，通过对齐会议保持一致 |
| 数据库设计 ∥ API 设计 | 技术架构方向已定 | 并行执行，但需要**字段语义对齐**环节 |
| 前端 ∥ 后端 | API 契约已定义 | 基于契约各自开发，前端用 Mock 数据 |
| 多 Sprint Team 并行 | 各 Team 范围无重叠 | Sprint 同步会议避免冲突 |

### 2.5 四种依赖类型

| 类型 | 含义 | 示例 |
|------|------|------|
| **FS（完成-开始）** | A 完成后 B 才能开始 | 设计完成 → 开发开始 |
| **SS（开始-开始）** | A 开始后 B 可以开始 | 需求分析开始 → 技术调研同步开始 |
| **FF（完成-完成）** | A 完成时 B 也应完成 | 开发完成 → 文档也应同步完成 |
| **SF（开始-完成）** | A 开始时 B 应完成（少见） | 新系统上线 → 旧系统应下线 |

## 三、项目类型适用场景判断标准

### 3.1 瀑布模型（Waterfall）

| 维度 | 特征 |
|------|------|
| **适用场景** | 需求明确且不变（政府采购、合规系统、硬件集成） |
| **核心特征** | 线性顺序、每阶段完成后才进入下一阶段 |
| **串并行** | 严格串行，不允许阶段重叠 |
| **优势** | 文档完整、可预测性强 |
| **劣势** | 不能回退，需求变更成本极高 |
| **NDHY 适用判断** | 需求已被老板完全确认且不会变化的项目 |

### 3.2 Scrum 框架

| 维度 | 特征 |
|------|------|
| **适用场景** | 需求需要迭代探索、用户反馈驱动 |
| **核心特征** | 迭代增量、经验主义、2-4 周 Sprint |
| **串并行** | Sprint 内设计/开发/测试可并行；Sprint 之间串行 |
| **关键约束** | Sprint Goal 设定后不应更改；增量必须满足 DoD |
| **NDHY 适用判断** | 产品开发类项目的默认选择——大部分 NDHY 项目适用 |

### 3.3 看板（Kanban）

| 维度 | 特征 |
|------|------|
| **适用场景** | 持续流动的工作、没有固定迭代周期（运维、客服、技术债） |
| **核心特征** | WIP 限制、持续流动、拉动制 |
| **串并行** | 单个工作项按列串行；不同工作项在不同列并行 |
| **关键规则** | WIP 限制是核心——某列达上限时上游必须停止推送 |
| **NDHY 适用判断** | 技能开发、知识库维护、持续优化类工作适用 |

### 3.4 选择决策表

| 条件 | 推荐模型 |
|------|---------|
| 需求完全确定、不允许变更 | 瀑布 |
| 需要迭代探索、有用户反馈 | Scrum |
| 持续流动、无固定周期 | Kanban |
| 需要快速验证、MVP 思路 | Scrum（短 Sprint） |
| 紧急修复、热修补丁 | Kanban |

## 四、教育科技行业特有的工作流差异

### 4.1 关键差异

| 维度 | 普通产品 | 教育产品 |
|------|---------|---------|
| 用户群 | 单一用户群 | **双用户群**：教师（使用者）+ 学生（学习者） |
| 设计驱动 | 功能/体验驱动 | **教学目标驱动**：功能必须服务于教学目标 |
| 科学约束 | 商业逻辑 | **学习科学约束**：认知负荷理论、间隔重复等 |
| 发布节奏 | 随时发布 | **教学日历约束**：受学期/假期影响 |
| 内容审核 | 可选 | **不可跳过**：教育内容必须经学科专家审核 |

### 4.2 教育行业特殊约束

- **发布窗口**：新学期开始前 2-4 周是最佳发布窗口
- **避免期中发布**：期中考试前后避免重大更新
- **教师培训前置**：发布前必须完成教师培训
- **内容审核不可跳过**：教育内容必须经过学科专家审核

### 4.3 ADDIE 模型（教育内容开发行业标准）

```
分析（Analysis）→ 设计（Design）→ 开发（Development）→ 实施（Implementation）→ 评估（Evaluation）
```

- 每个步骤的产出都是下一步的输入（严格串行）
- 评估贯穿全程（形成性评估在每个阶段都有）
- 快速原型法是改进版——Design 阶段就原型验证

### 4.4 教育产品全链路额外阶段

| 额外阶段 | 普通产品不需要 | 说明 |
|---------|--------------|------|
| Phase 0 增加：课程标准分析 | ✅ | 必须对齐国家/地方课程标准 |
| Phase 1 增加：教学设计 | ✅ | 学习目标定义 → 教学策略选择 → 评估方式设计 |
| Phase 3 增加：教学内容制作 | ✅ | 与前后端开发并行 |
| Phase 4 增加：内容审核 | ✅ | 学科专家审核，与技术测试并行 |
| Phase 5 改为：种子学校试点 | ✅ | 不是直接全量发布 |

## 五、提示词工程/AI 内容生产的工作流要点

### 5.1 提示词工程 6 步流程

```
目标定义 → 信息收集 → 提示词设计 → 测试迭代 → 优化固化 → 部署监控
```

**串并行规则**：
- 步骤 1→2→3 **严格串行**（不知道目标不能收集信息，不知道信息不能设计提示词）
- 步骤 3↔4 **快速迭代循环**（设计→测试→调整→再测试）
- 步骤 5→6 **串行**（先固化再部署）
- 不同业务线的提示词工程可以**并行**

**核心原则**：信息对称是核心——AI 结果不好 = 信息不够，不是 AI 的问题

### 5.2 AI 内容生产管线

```
信息源 → 采集清洗 → 结构化 → AI 生成 → 人工审核 → 发布分发 → 效果追踪
```

- 多源可并行采集；不同主题可并行生成；多渠道可并行发布
- **人工审核不可跳过**：事实核查、品质审核、合规检查、品牌一致性

### 5.3 知识工程流程

```
数据采集 → 信息整理 → 知识提炼 → 系统化 → 验证 → 部署应用
```

- **严格串行**：每一步依赖前一步产出，不能跳步
- 关键规则：采集时"宁多勿少，先不判断"；提炼时"多源交叉验证"；验证时"找反例证伪"

## 六、NDHY 阶段门控设计

| Gate | 名称 | 审核人 | 关键检查项 | 决策选项 |
|------|------|--------|-----------|---------|
| G0 | 需求门 | Leader | 需求是否清晰？价值是否成立？ | Go / 回炉挖掘 |
| G1 | 方案门 | Leader + 老板 | 产品定义完整？体验方案合理？技术方案可行？ | Go / 调整 / Kill |
| G2 | 设计门 | Leader | 视觉完成？API 契约定义？数据模型设计？ | Go / 补齐设计 |
| G3 | 开发门 | 项目管理专家 | 代码审查通过？单元测试通过？ | Go / 打回修改 |
| G4 | 集成门 | 联调集成专家 | 前后端联调通过？端到端流程跑通？ | Go / 归因修复 |
| G5 | 发布门 | Leader + 老板 | 系统测试通过？发布策略就绪？回滚方案准备？ | Go / 延期 / Kill |
