本周行业最重大的趋势：**Harness Engineering（驭具工程）** 正式成为 AI Agent 领域的核心范式。

**关键事件**：

**核心概念**：

| 层级 | 类比 | 说明 |
|------|------|------|
| 模型 | CPU | 原始推理能力 |
| 上下文窗口 | 内存 | 有限的工作记忆 |
| Harness | 操作系统 | 管理上下文、工具、子Agent、记忆、验证 |
| Agent | 应用程序 | 运行在 Harness 之上 |

**Harness 的 6 大组件**（parallel.ai 总结，OpenAI/Anthropic 认可）：
1. **工具集成层** — 连接 API、数据库、代码执行环境
2. **记忆与状态管理** — 多层记忆（工作上下文/会话状态/长期记忆）
3. **上下文工程与提示管理** — 动态策展每次调用的信息（非静态模板）
4. **规划与分解** — 引导模型按结构化任务序列执行
5. **验证与防护** — 校验、格式化、安全过滤、自修正循环
6. **模块化与可扩展** — 组件可插拔替换

### 1.2 HumanLayer 的实战洞察（重要）

来源：humanlayer.dev，3月11日

**核心论点**：Agent 失败不是模型问题，是配置问题。

关键实践：
- **Sub-agents 是最强大的杠杆**：在复杂棕地企业代码库中，子 Agent 作为「上下文防火墙」，确保离散任务在隔离的上下文窗口中运行，避免中间噪声累积到父线程
- **Skills（技能模块）作为渐进式知识披露机制** — 他们称之为 "Instruction Modules"
- **Hooks 用于自动化集成和确定性控制流**
- **模型与 Harness 的耦合问题**：Opus 4.6 在 Claude Code 中排名 #33，但在不同 Harness 中排名 #5 — 模型可能对其训练时的 Harness 过拟合

### 1.3 Epsilla 的 30-60-90 天路线图

来源：epsilla.com，3月12日

**「AGENTS.md 谬误」**：一个 markdown 文件不是 Harness。它会随着系统演进快速腐化。规则必须是结构化的、可验证的、可循环的。

**更强的模型需要更严格的约束**（反直觉）：Agent 自主性越高，运行环境约束需要越严格，而非越宽松。

**功能正确性验证缺失**：当前 Harness 实践过度关注内部质量（代码一致性、lint、文档），但代码库可以从工程角度完美"干净"却在业务功能上灾难性失败。

### 🔑 3-5 条核心经验

**① 我们的架构方向完全正确**

**② Sub-agents 作为"上下文防火墙"是关键模式**

AI 设计工具已从"辅助画图"进化到"约束内生成"。体验设计专家需要跟进。
