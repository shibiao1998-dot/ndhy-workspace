## 一、AI Agent 组织管理

### 1.1 多 Agent 协作模式

| 模式 | 适用场景 | 关键挑战 |
|------|---------|---------|
| **层级式** | 复杂项目（4+ 专家） | 信息传递损耗、决策延迟 |
| **扁平式** | 独立任务并行 | 协调成本高、依赖管理难 |
| **混合式** | 中等复杂度（2-3 专家） | 需要清晰的交接标准 |
| **对抗式** | 关键质量节点（审查/验收） | 需要明确的裁决机制 |

### 1.2 最新实践洞察

**核心发现**：
1. **上下文注入质量 = 产出质量**：执行者的表现上限由指令包质量决定
2. **增量写入优于批量输出**：每完成一个模块就写入磁盘，避免超时导致全损
3. **文件即记忆**：ACP 模式下，文件是跨会话维持状态的最可靠方式
4. **对抗式协作提升质量**：关键节点引入质疑→回应→再判循环，比单向放行更有效

### 1.3 NDHY 当前实践对齐

| 实践 | 行业趋势 | NDHY 状态 | 建议 |
|------|---------|----------|------|
| 三文档门控 | 需求→设计→架构分离 | ✅ 已实施 | 坚持执行 |
| 模块化串行开发 | 小步快跑、逐模块验收 | ✅ 已实施 | 可增加并行 Track（设计/后端） |
| 对抗式协作 | 关键节点引入质疑 | ✅ 已实施 | 可扩展到技术选型 |
| 心跳巡查机制 | 主动监控子 Agent 状态 | ✅ 已实施 | 保持 5 分钟巡查频率 |

## 二、提示词/上下文工程

### 2.1 Context Engineering 核心原则

**渐进式披露（Progressive Disclosure）**：
- **设定（宪法）**：始终加载，零信息损耗
- **技能（操作手册）**：按需加载，匹配任务类型
- **References（深度细节）**：按需递归，避免堆上下文

**NDHY 实践**：已在 SOUL.md/AGENTS.md 中实现，符合行业最佳实践。

### 3.2 AI 生产线实践

**核心原则**：
1. **标准化输入**：需求文档→产品文档→架构文档，环环相扣
2. **自动化验证**：每个模块开发完成后自动 test/lint/类型检查
3. **质量门禁**：形式检查 + 内容检查，两层过滤
4. **可追溯性**：每个产出物关联上游输入和验收标准

**NDHY 实践**：三文档门控 + 模块化开发 + 两层质量检查，符合行业最佳实践。

## 七、MEMORY.md 追加建议

**建议追加内容**（需老板确认）：

```markdown
## 组织进化原则（2026-03-18 调研结论）

1. **指令包质量 = 产出质量**：执行专家的产出上限由指令包质量决定。上下文给少了执行者会瞎猜，给多了执行者会迷路——精准注入是项目管理最核心的能力。

2. **增量写入铁律**：所有 ACP 开发任务必须每步增量写入文件，不允许最后一次性大输出。每完成一个模块就写入磁盘，避免超时导致全损。

3. **对抗式协作扩展**：关键流程节点必须引入对抗环节（需求确认、技术选型、代码审查、测试验收），单向产出→放行不够，必须有质疑→回应→再判循环。

4. **文件即记忆（ACP 模式）**：项目状态、进度、决策全部写入文件，不依赖会话记忆。文件是跨会话维持状态的最可靠方式。

5. **技能优先于直觉**：所有子 Agent 开工前必须扫一遍适用技能表和 available_skills，有匹配的按技能规范执行。配了技能就必须用，不是装饰品。
```
