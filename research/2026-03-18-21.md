## 一、AI Agent 组织管理

### 核心发现

#### 1. LobeHub - Agent 作为工作单位
- **项目**: https://github.com/lobehub/lobehub
- **核心创新**: 将 Agent 作为工作单位 (Agents as the Unit of Work)
- **关键能力**:
  - 多 Agent 协作基础设施
  - 一键 MCP 插件安装
  - MCP 市场
  - 分支对话 (Branching Conversations)
  - 人工与 Agent 共同进化
- **启示**: 我们的 ACP-First 架构与 LobeHub 理念高度一致，但他们在 Agent 市场化和插件生态上更超前

#### 2. TeamClaw - YAML 可编程编排引擎
- **项目**: https://github.com/Avalon-467/Teamclaw
- **核心创新**: Auto-OASIS 核心 - 通过 YAML 配置定义专家协作流程
- **关键能力**:
  - 支持顺序、并行、复杂可编程结构
  - 后端分离执行模式
  - OpenAI 兼容 API 端点
  - 多用户 + 多会话隔离
- **示例配置**:
```yaml
version: 1
repeat: true
plan:
  - expert: "Creative Expert"
  - expert: "Critical Expert"
  - parallel:
      - expert: "Technical Expert"
      - expert: "Design Expert"
```
- **启示**: 我们的 leader-workflow 技能中的"执行拓扑选择"可以用 YAML 配置化，降低调度复杂度

#### 3. Agent Teams Playbook - 6 步协作流程
- **项目**: https://github.com/SameerVerse2/agent-teams-playbook
- **核心方法**: 6 步流程管理多 Agent 协作
- **5 大场景**: 任务自动化、问题解决、决策支持、内容生成、数据分析
- **启示**: 我们的项目执行流程 (project-execution skill) 已有 8 步 SOP，可对比优化

#### 4. 自主 Agent 最新趋势
- **Sigil Open Framework**: 解决自主 Agent 采用的最后障碍 - 责任归属 (Liability)
- **Goosetown**: 轻量级自主 Agent 管理环境
- **Spore Protocol**: 去中心化 AI Agent 群体协议 - 自组织、自复制网络
- **趋势判断**: 自主 Agent 正从"任务执行"向"责任可追溯"演进

## 二、提示词/上下文工程

### 核心发现

#### 1. PRD-Driven Context Engineering - 文档即记忆基础设施
- **项目**: https://github.com/mattgierhart/PRD-driven-context-engineering
- **核心理念**: 
  - 将文档转化为人类+AI 共享的记忆基础设施
  - Source of Truth Artifacts (SoT) - 带唯一 ID 的可引用决策节点
  - Knowledge Graph: BR-xxx, UJ-xxx, API-xxx 跨文件链接
- **关键转变**:
  | 传统敏捷 | PRD 驱动上下文工程 |
  |---------|------------------|
  | Sprints | Context Windows (按认知容量定范围) |
  | User Stories | Prompts (工程化加载上下文) |
  | Tribal Knowledge | Source of Truth |
  | Standups | Documentation Hooks (事件驱动) |
  | 项目管理 | 上下文治理 |
- **黄金规则**: If it isn't part of the memory infrastructure, it isn't true
- **启示**: 我们的 MEMORY.md + PROJECT.md 体系与 SoT 理念一致，但缺少唯一 ID 引用系统

#### 2. Langfuse - 开源 LLM 工程平台
- **项目**: https://github.com/langfuse/langfuse
- **核心能力**:
  - LLM 可观测性 (追踪、调试)
  - 提示词管理 (版本控制、协作迭代)
  - 评估系统 (LLM-as-a-judge、用户反馈、手动标注)
  - 数据集 (测试集、基准测试)
  - Playground (测试提示词和模型配置)
- **集成**: OpenTelemetry、LangChain、OpenAI SDK、LiteLLM
- **启示**: 我们的质量检查依赖人工，可引入 Langfuse 做自动化评估

#### 3. Context Engineering 工具链
- **AI-Tracker**: 通过 Model Context Protocol 优化大语言模型性能
- **Agentic Context Engineering**: 动态构建 JSONL playbook，每轮交互增强洞察
- **Copilot Memory Store**: 本地 JSON 数据存储用于上下文工程
- **趋势**: 上下文管理正从"手动维护"向"自动化版本控制"演进

## 六、对 NDHY AI Agent Team 的建议

### 3-5 条核心经验

#### 1. 引入"Source of Truth Artifacts (SoT)"唯一 ID 系统
- **现状**: MEMORY.md 和 PROJECT.md 无唯一标识，难以跨文档引用
- **建议**: 为关键决策、需求、架构决定分配唯一 ID (如 DEC-001、REQ-001、ARCH-001)
- **价值**: 建立 Knowledge Graph，支持人类和 AI 共同查询
- **执行专家**: 📐 产品定义专家 + 🏛️ 技术架构专家

#### 2. 引入 Langfuse 做自动化评估
- **现状**: 质量检查依赖人工审查，效率低
- **建议**: 部署 Langfuse，对子 Agent 产出进行自动化评估 (LLM-as-a-judge)
- **价值**: 将质量检查从"人工全审"升级为"自动化初筛 + 人工复审"
- **执行专家**: 🔧 Skill 开发专家 + 🧪 测试专家

#### 3. 将 leader-workflow 的执行拓扑 YAML 配置化
- **现状**: 执行拓扑由 Leader 手动判断
- **建议**: 借鉴 TeamClaw 的 YAML 编排引擎，将常见协作模式配置化
- **示例**:
```yaml
task_type: frontend_module
topology: serial
experts:
  - role: 体验设计专家
    output: UX 文档
  - role: UI 设计专家
    input: UX 文档
    output: 视觉原型
  - role: 前端开发专家
    input: 视觉原型 + API 契约
    output: 页面代码
```
- **价值**: 降低调度认知负荷，提高复用性
- **执行专家**: 🔧 Skill 开发专家

#### 4. 建立"AI 工具评估矩阵"
- **现状**: 27 位专家使用工具无统一评估标准
- **建议**: 每季度审计一次工具栈，评估维度：
  - 架构透明度
  - 与现有工作流集成度
  - 成本效益比
  - 社区活跃度
- **价值**: 避免工具碎片化，保持技术栈统一
- **执行专家**: 🔬 技术调研专家

#### 5. 教育产品差异化方向 - 教师端 AI 助手
- **市场空白**: 现有教育 AI 聚焦学生端，教师端工具稀缺
- **建议方向**:
  - AI 备课助手 (基于课标自动生成教案)
  - AI 批改助手 (主观题智能批改 + 反馈)
  - 学情分析看板 (基于作业/考试数据)
- **价值**: 避开学生端红海，切入教师端蓝海
- **执行专家**: 🎯 需求分析专家 + 📐 产品定义专家
