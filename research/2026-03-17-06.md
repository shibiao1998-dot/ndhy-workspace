## 一、本周核心发现

### 🔥 1. "Harness Engineering" 成为行业新范式

**这是本周最重要的发现，直接关系到我们组织的核心架构。**

LangChain 创始团队 Vivek Trivedy 于 3月15日发布了里程碑文章《The Anatomy of an Agent Harness》，正式定义了 **Harness Engineering** 概念：

> **Agent = Model + Harness**
> 如果你不是模型，你就是 Harness。

Harness 是围绕模型的所有代码、配置和执行逻辑的总和，包括：
- **System Prompts**（我们的 SOUL.md / STANDARDS.md）
- **Tools/Skills/MCPs + 描述**（我们的技能系统）
- **Bundled Infrastructure**（文件系统、沙箱、浏览器）
- **Orchestration Logic**（子Agent生成、handoffs、模型路由）
- **Hooks/Middleware**（确定性执行：压缩、续接、lint检查）

HumanLayer 团队跟进发文《Skill Issue: Harness Engineering for Coding Agents》，进一步阐述：
- Harness Engineering 是 **Context Engineering 的子集**
- 关键杠杆：system prompt、tools/MCPs、context、**sub-agents**（作为"上下文防火墙"）
- **Skills 作为知识渐进式披露**的关键工具（他们称之为"Instruction Modules"）
- 子 Agent 是解决复杂问题的最强杠杆——作为"context firewall"隔离中间噪音

**与 NDHY 团队的关系**：
- 我们的 SOUL.md + STANDARDS.md + Skills 体系 **恰好是 Harness Engineering 的实践**
- 我们的"渐进式披露"原则（设定始终加载、技能按需加载）被行业验证为正确方向
- 子 Agent 作为"上下文防火墙"的理论基础，验证了我们的 ACP-First 架构

### 🔥 2. Prompt Engineering → Context Design 演进

SDG Group 发布报告《The Evolution of Prompt Engineering to Context Design in 2026》，核心观点：

- **Prompt Engineer → Context Designer/Architect** 角色转变
- 传统提示词工程关注单次交互的指令优化
- Context Engineering 关注**动态信息生态系统**的编排：多源数据流、实时更新、用户偏好、历史消息
- 2026年重点：帮助 AI Agent 更好地**理解上下文**并高效自动化任务

Red Hat Developer 也发文对比了 Big Prompts vs Small Prompts 的策略，结论是小而精准的分步提示 + 上下文管理优于大一统提示。

**与 NDHY 团队的关系**：
- 验证了我们"技能 = 操作手册"的模式——按需加载比全量注入更有效
- 我们的指令包 7 层结构本质上就是 Context Engineering 的实践

#### Forge Orchestrator 深度分析（最值得研究）

Forge 的核心创新：**Contract Builder**——在任何 Agent 写代码前先生成接口契约，确保前后端一致。

| 痛点 | Forge 解法 |
|------|-----------|
| 一次只能提示一件事 | 任务分解为**依赖图**，独立任务**并行执行** |
| 并行 Agent 产出不兼容 | **Contract Builder** 先生成绑定的API和类型契约 |
| AI 改动破坏其他代码 | 每个 Agent 工作在**隔离的 git worktree** |
| 手动审查 AI 产出 | **多门审查流水线**：build > lint > test > LLM review > 契约合规 > merge check |
| 上下文在长会话中丢失 | 每个 Agent 是**全新 Claude 会话** + 聚焦提示 + 相关契约 |
| 合并手动且易出错 | 自动 **rebase + fast-forward merge** + 自动创建 **PR** |
| 无法控制 AI 花费 | **每流水线预算限制** + 实时成本追踪 |

## 二、对组织有价值的 5 条核心经验

### ① Harness Engineering 理论验证了我们的架构设计
我们的 SOUL.md + STANDARDS.md + Skills + ACP 架构 = 业界刚命名的 Harness Engineering。不需要大改，需要**有意识地用这个框架审视和优化**现有设计。

### ② "接口契约优先"是并行 Agent 开发的关键
Forge 的 Contract Builder 证明：并行 Agent 开发的最大风险不是代码质量，是**接口不一致**。我们的 API 设计专家 + 前端开发 + 后端开发协作流程应借鉴此思路。

### ③ Sub-Agent 作为 "Context Firewall" 的设计模式
子 Agent 不仅是执行者，更是**上下文防火墙**——隔离中间噪音，确保主编排线程保持清晰。这验证了我们的 ACP-First + subagent 策略。

### ④ Skills = "Instruction Modules" = 渐进式知识披露
行业开始将 Skills 视为不是工具，而是**知识的渐进式披露机制**。我们的技能系统走在了前面，但可以进一步优化技能的加载时机和信息密度。

### ⑤ Google Stitch 值得关注作为设计提效工具
免费、可导出 Figma 和 React 代码、持续升级中。可能成为我们体验设计环节的加速器。
