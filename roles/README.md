## 选角决策树

收到任务后，按以下逻辑匹配角色：

```
任务是什么？
│
├─ 需求模糊，需要挖掘和澄清？
│   └─→ 🎯 方寻真 · 需求挖掘专家 (方寻真-requirement-analyst/)
│
├─ 需要了解用户是谁 / 用户画像 / 用户旅程 / 用户需求洞察？
│   └─→ 🔍 聂听澜 · 用户研究专家 (聂听澜-user-researcher/)
│
├─ 市场规模 / 竞争格局全景 / 趋势时机 / 市场可行性 / 商业可行性初判？
│   └─→ 📊 顾衡野 · 市场评估专家 (顾衡野-market-assessor/)
│
├─ 单个竞品深度拆解 / 产品策略分析 / 功能对标 / 技术路线分析？
│   └─→ 🔭 沈鉴微 · 竞品调研专家 (沈鉴微-competitive-analyst/)
│
├─ 战略方向选择 / 竞争定位 / 价值主张 / 做什么不做什么 / 凭什么赢？
│   └─→ 🧭 韩定川 · 战略定位专家 (韩定川-strategic-positioner/)
│
├─ 产品到底是什么 / 产品形态 / 核心场景 / 功能范围 / 产品规则 / MVP定义？
│   └─→ 📐 凌器之 · 产品定义专家 (凌器之-product-definer/)
│
├─ 产品定义已完成，需要设计用户体验 / 信息架构 / 任务流程 / 交互模式 / 体验语言？
│   └─→ 🎨 沈映川 · 体验设计专家 (沈映川-experience-designer/)
│
├─ 方案已定，需要拆解执行流程 / 定义角色 / 设计协作？
│   └─→ 🔩 程织枢 · 流程设计专家 (程织枢-process-designer/)
│
├─ 需要创造AI专家角色设定 / 匹配专家池 / 组建项目团队？
│   └─→ 🧬 殷铸魂 · 角色创造专家 (殷铸魂-role-creator/)
│
├─ 创建/改进 OpenClaw Skill？
│   └─→ 🔧 贺匠心 · Skill 开发专家 (贺匠心-skill-developer/)
│
├─ 系统架构设计 / 技术选型 / 模块边界 / AI集成方案 / 非功能性设计？
│   └─→ 🏛️ 骆骨架 · 技术架构专家 (骆骨架-technical-architect/)
│
├─ 开发 Web 应用 / API / 前后端代码？
│   └─→ 🌐 林栈桥 · 全栈开发专家 (林栈桥-fullstack-developer/)
│
├─ 写文档 / 方案设计 / 知识整理？
│   └─→ 📝 苏墨言 · 技术文档专家 (苏墨言-technical-writer/)
│
├─ 审查代码 / 评审架构 / 检查质量？
│   └─→ 🔍 严守正 · 代码审查专家 (严守正-code-reviewer/)
│
├─ 技术选型 / 可行性评估 / 技术方案调研？
│   └─→ 🔬 陆知远 · 技术调研专家 (陆知远-researcher/)
│
├─ 多角色项目执行调度 / 指令包组装 / 全局上下文管理？
│   └─→ 🏗️ 纪统辰 · 项目管理专家 (纪统辰-project-manager/)
│
└─ 以上都不匹配？
    ├─ 偏执行/构建 → 🌐 林栈桥
    ├─ 偏分析/输出文字 → 📝 苏墨言
    └─ 偏信息收集 → 🔬 陆知远
```

### ⚠️ 易混淆角色辨析

| 场景 | 选谁 | 理由 |
|------|------|------|
| "这个市场有多大？谁在做？值不值得入场？" | 📊 顾衡野 | 宏观市场格局判断 |
| "这个竞品到底怎么做的？它为什么这么设计？" | 🔭 沈鉴微 | 单个竞品深度拆解 |
| "我们到底做什么方向？跟别人有什么不同？凭什么赢？" | 🧭 韩定川 | 战略方向选择与竞争定位 |
| "先做一轮市场摸底 + 竞品拆解" | 📊 顾衡野（格局）→ 🔭 沈鉴微（深拆 Top 竞品） | 先全景后聚焦 |
| "用户是谁？他们需要什么？" | 🔍 聂听澜 | 微观用户视角 |
| "这个项目能不能做？"（全面评估） | 📊 顾衡野 ∥ 🔍 聂听澜（并行）→ 联合输出 | Phase 0 双视角 |
| "市场可以做，但具体做什么方向？怎么定位？" | 🧭 韩定川 | 在市场可行性确认后做方向决策 |
| "差异化应该从哪个方向切入？" | 🧭 韩定川（主）← 📊 顾衡野（市场数据支撑） | 定位决策由韩定川做，顾衡野提供市场事实 |
| "这个产品到底是什么？长什么样？包含哪些功能？" | 📐 凌器之 | 战略方向已定，需要定义具体产品形态 |
| "MVP 应该包含哪些功能？" | 📐 凌器之 | 功能范围界定和优先级分层 |
| "定位有了，接下来怎么把它变成具体的产品？" | 📐 凌器之 | 战略定位→产品定义的衔接 |
| "用户用起来什么感觉？信息怎么组织？任务怎么走？" | 🎨 沈映川 | 产品定义已完成，需要设计使用体验 |
| "页面布局怎么安排？交互用什么模式？" | 🎨 沈映川 | 体验层的界面框架和交互设计 |
| "产品'说话'的风格是什么？按钮文案怎么写？" | 🎨 沈映川 | 体验语言和文案设计 |
| "系统怎么分层？模块怎么切？用什么技术？" | 🏛️ 骆骨架 | 系统架构设计和技术选型 |
| "这个技术能不能用？有没有替代方案？"（单项调研） | 🔬 陆知远 | 单项技术深度调研 |
| "整个系统的技术方案怎么定？"（系统级） | 🏛️ 骆骨架 | 系统级架构设计和选型决策 |
| "产品定义有了，技术上怎么实现？" | 🏛️ 骆骨架 | 产品定义→技术架构的衔接 |

### 🔑 韩定川 vs 凌器之 边界说明

两者是 Phase 2 内部的上下游关系，核心区别在于：

| 维度 | 🧭 韩定川（战略定位） | 📐 凌器之（产品定义） |
|------|----------------------|----------------------|
| **Phase 2 位置** | 起手式（第一棒） | 承接棒（第二棒） |
| **核心问题** | "走哪条路？跟谁竞争？凭什么赢？" | "这条路上造什么车？拉什么人？怎么跑？" |
| **工作性质** | 方向判断 + 战略决策 | 产品翻译 + 具象定义 |
| **边界划定** | 战略边界（做什么领域/不做什么领域） | 产品边界（与现有产品/外部系统/未来版本关系） |
| **输出去向** | → 凌器之（产品定义） | → 沈映川（体验设计）+ 产品架构师 + 项目管理 |
| **约束关系** | 韩定川的"不做清单"是凌器之的红线 | 凌器之的功能范围和规则约束体验设计和架构设计 |

### 🔑 凌器之 vs 沈映川 边界说明

两者是 Phase 2 内部的上下游关系（第二棒→第三棒），核心区别在于：

| 维度 | 📐 凌器之（产品定义） | 🎨 沈映川（体验设计） |
|------|----------------------|----------------------|
| **Phase 2 位置** | 第二棒 | 第三棒 |
| **核心问题** | "这个产品是什么？有什么功能？在什么场景下被谁使用？" | "用户用起来到底什么感觉？每一步怎么走、看到什么、感受到什么？" |
| **工作性质** | 产品翻译（战略→产品形态） | 体验翻译（产品形态→使用体验） |
| **场景相关** | 产品场景拆解：定义"什么场景下使用什么功能" | 任务流程设计：设计"在这个场景里用户具体怎么操作" |
| **流程相关** | 核心任务定义（抽象级别：做什么） | 任务流+用户流（具体步骤级别：怎么做） |
| **输出去向** | → 沈映川（体验设计）+ 产品架构师 + 项目管理 | → 视觉设计师 + 前端开发 + 项目管理 |
| **约束关系** | 凌器之的功能范围和产品规则是沈映川的设计边界 | 沈映川的交互规范和设计规范约束视觉设计和前端实现 |

### 🔑 方寻真 vs 凌器之 边界说明

两者分别在 Phase 1 和 Phase 2，视角不同：

| 维度 | 🎯 方寻真（需求挖掘） | 📐 凌器之（产品定义） |
|------|----------------------|----------------------|
| **Phase** | Phase 1（信息采集与需求整合） | Phase 2（方案设计） |
| **核心问题** | "用户到底想要什么？" | "产品到底是什么东西？" |
| **视角** | 需求视角：从用户出发 | 产品视角：从战略定位出发 |
| **场景相关** | 用户故事（用户想完成什么） | 产品场景（产品在什么情境下被使用） |
| **功能相关** | 需求优先级排序（用户需求的轻重缓急） | 功能范围界定（产品包含/不包含什么能力） |
| **输出去向** | → 韩定川（战略定位的事实输入） | → 产品架构师 + 交互设计（产品设计的具体指导） |

### 🔑 顾衡野 vs 韩定川 边界说明

两者的工作有上下游关系，核心区别在于：

| 维度 | 📊 顾衡野（市场评估） | 🧭 韩定川（战略定位） |
|------|----------------------|----------------------|
| **Phase** | Phase 0（信息采集） | Phase 2（方向决策） |
| **核心问题** | "市场成不成立？" | "我们做什么、不做什么、凭什么赢？" |
| **工作性质** | 信息采集 + 事实分析 | 方向判断 + 战略决策 |
| **差异化相关** | 绘制竞争地图 + 评估空白区域价值 → 输出初步定位建议 | 在竞争地图基础上做定位决策 → 输出完整竞争定位 + 价值主张 + 边界 |
| **输出去向** | → 方寻真（Phase 1 整合） | → 凌器之（产品定义，Phase 2 第二棒） |
| **约束力** | 提供信息和建议，不约束下游 | 输出的边界划定直接约束架构设计 |

### 🔑 骆骨架 vs 林栈桥 边界说明

两者是架构层与实现层的上下游关系：

| 维度 | 🏛️ 骆骨架（技术架构） | 🌐 林栈桥（全栈开发） |
|------|----------------------|----------------------|
| **核心问题** | "系统怎么分层？模块怎么切？用什么技术？" | "代码怎么写？功能怎么实现？" |
| **工作性质** | 方案设计 + 技术决策 | 代码实现 + 功能交付 |
| **技术选型** | 做系统级选型决策（框架/语言/数据库/架构模式） | 在选型框架内做实现层决策（组件库/工具库/代码模式） |
| **输出** | 架构方案文档 + ADR + 技术约束 | 可运行的代码 + 测试 + 文档 |
| **约束关系** | 骆骨架的架构约束是林栈桥的设计边界 | 林栈桥发现架构约束不可行时回调骆骨架调整 |

### 🔑 骆骨架 vs 陆知远 边界说明

两者在技术选型维度有交集，核心区别在于层级：

| 维度 | 🏛️ 骆骨架（技术架构） | 🔬 陆知远（技术调研） |
|------|----------------------|----------------------|
| **核心问题** | "系统整体用什么技术方案？" | "这个具体技术能不能用？怎么用？" |
| **工作性质** | 系统级选型决策（多技术组合的整体方案） | 单项技术深度调研（可行性/性能/风险） |
| **输出** | 技术选型决策记录（含整体方案） | 技术调研报告（单项技术的深度分析） |
| **约束关系** | 骆骨架委派陆知远做深度调研，然后做系统级决策 | 陆知远输出调研结论供骆骨架决策参考 |

## 组合使用

复杂需求可以拆解后给不同角色：

| 场景 | 角色组合 |
|------|----------|
| 从方案到执行团队 | 🔩 程织枢 → 🧬 殷铸魂 → 🔧 贺匠心 |
| 开发新 Skill 并写文档 | 🔧 贺匠心 → 📝 苏墨言 |
| 新功能开发 + 代码审查 | 🌐 林栈桥 → 🔍 严守正 |
| 技术选型 + 原型开发 | 🔬 陆知远 → 🌐 林栈桥 |
| Skill 改进 + 质量检查 | 🔧 贺匠心 → 🔍 严守正 |
| 产品定义到技术架构到开发 | 📐 凌器之 → 🏛️ 骆骨架 → 🌐 林栈桥 |
| 单项调研 + 系统级选型 | 🔬 陆知远（深度调研）→ 🏛️ 骆骨架（系统级选型决策） |
| 完整项目执行 | 🔩 程织枢 → 🧬 殷铸魂 → 🏗️ 纪统辰（调度执行专家） |
| 从方案到交付（全链路） | 🎯 方寻真 → 🔩 程织枢 → 🧬 殷铸魂 → 🏗️ 纪统辰 → 各执行专家 |
| 新项目用户洞察驱动 | 🔍 聂听澜 → 🎯 方寻真 → 🔩 程织枢 → 🧬 殷铸魂 → 🏗️ 纪统辰 |
| 用户研究 + 市场评估（Phase 0 并行） | 🔍 聂听澜（用户视角）∥ 📊 顾衡野（市场视角）→ 联合输出 → 🎯 方寻真 |
| 市场评估 + 竞品深拆（串行） | 📊 顾衡野（竞争格局）→ 🔭 沈鉴微（Top 竞品深度拆解） |
| 用户研究 + 竞品调研（并行） | 🔍 聂听澜（用户视角）∥ 🔭 沈鉴微（竞品视角）→ 🎯 方寻真 |
| 完整 Phase 0（全维度） | 🔍 聂听澜 ∥ 📊 顾衡野 ∥ 🔭 沈鉴微 → 联合校验 → 🎯 方寻真 |
| **完整 Phase 0→2（需求到定位）** | 🔍 聂听澜 ∥ 📊 顾衡野 → 🎯 方寻真（整合）→ 🧭 韩定川（定位） |
