## 选角决策树

收到任务后，按以下逻辑匹配角色：

```
任务是什么？
│
├─ 需求模糊，需要挖掘和澄清？
│   └─→ 🎯 方寻真 · 需求挖掘专家 (方寻真-requirement-analyst/)
│
├─ 需要了解用户是谁 / 用户画像 / 用户旅程 / 用户需求洞察？
│   └─→ 🔍 聂听澜 · 用户研究专家 (聂听澜-user-researcher/)
│
├─ 市场规模 / 竞争格局全景 / 趋势时机 / 市场可行性 / 商业可行性初判？
│   └─→ 📊 顾衡野 · 市场评估专家 (顾衡野-market-assessor/)
│
├─ 单个竞品深度拆解 / 产品策略分析 / 功能对标 / 技术路线分析？
│   └─→ 🔭 沈鉴微 · 竞品调研专家 (沈鉴微-competitive-analyst/)
│
├─ 方案已定，需要拆解执行流程 / 定义角色 / 设计协作？
│   └─→ 🔩 程织枢 · 流程设计专家 (程织枢-process-designer/)
│
├─ 需要创造AI专家角色设定 / 匹配专家池 / 组建项目团队？
│   └─→ 🧬 殷铸魂 · 角色创造专家 (殷铸魂-role-creator/)
│
├─ 创建/改进 OpenClaw Skill？
│   └─→ 🔧 贺匠心 · Skill 开发专家 (贺匠心-skill-developer/)
│
├─ 开发 Web 应用 / API / 前后端代码？
│   └─→ 🌐 林栈桥 · 全栈开发专家 (林栈桥-fullstack-developer/)
│
├─ 写文档 / 方案设计 / 知识整理？
│   └─→ 📝 苏墨言 · 技术文档专家 (苏墨言-technical-writer/)
│
├─ 审查代码 / 评审架构 / 检查质量？
│   └─→ 🔍 严守正 · 代码审查专家 (严守正-code-reviewer/)
│
├─ 技术选型 / 可行性评估 / 技术方案调研？
│   └─→ 🔬 陆知远 · 技术调研专家 (陆知远-researcher/)
│
├─ 多角色项目执行调度 / 指令包组装 / 全局上下文管理？
│   └─→ 🏗️ 纪统辰 · 项目管理专家 (纪统辰-project-manager/)
│
└─ 以上都不匹配？
    ├─ 偏执行/构建 → 🌐 林栈桥
    ├─ 偏分析/输出文字 → 📝 苏墨言
    └─ 偏信息收集 → 🔬 陆知远
```

### ⚠️ 易混淆角色辨析

| 场景 | 选谁 | 理由 |
|------|------|------|
| "这个市场有多大？谁在做？值不值得入场？" | 📊 顾衡野 | 宏观市场格局判断 |
| "这个竞品到底怎么做的？它为什么这么设计？" | 🔭 沈鉴微 | 单个竞品深度拆解 |
| "先做一轮市场摸底 + 竞品拆解" | 📊 顾衡野（格局）→ 🔭 沈鉴微（深拆 Top 竞品） | 先全景后聚焦 |
| "���户是谁？他们需要什么？" | 🔍 聂听澜 | 微观用户视角 |
| "这个项目能不能做？"（全面评估） | 📊 顾衡野 ∥ 🔍 聂听澜（并行）→ 联合输出 | Phase 0 双视角 |

## 组合使用

复杂需求可以拆解后给不同角色：

| 场景 | 角色组合 |
|------|----------|
| 从方案到执行团队 | 🔩 程织枢 → 🧬 殷铸魂 → 🔧 贺匠心 |
| 开发新 Skill 并写文档 | 🔧 贺匠心 → 📝 苏墨言 |
| 新功能开发 + 代码审查 | 🌐 林栈桥 → 🔍 严守正 |
| 技术选型 + 原型开发 | 🔬 陆知远 → 🌐 林栈桥 |
| Skill 改进 + 质量检查 | 🔧 贺匠心 → 🔍 严守正 |
| 完整项目执行 | 🔩 程织枢 → 🧬 殷铸魂 → 🏗️ 纪统辰（调度执行专家） |
| 从方案到交付（全链路） | 🎯 方寻真 → 🔩 程织枢 → 🧬 殷铸魂 → 🏗️ 纪统辰 → 各执行专家 |
| 新项目用户洞察驱动 | 🔍 聂听澜 → 🎯 方寻真 → 🔩 程织枢 → 🧬 殷铸魂 → 🏗�� 纪统辰 |
| 用户研究 + 市场评估（Phase 0 并行） | 🔍 聂听澜（用户视角）∥ 📊 顾衡野（市场视角）→ 联合输出 → 🎯 方寻真 |
| 市场评估 + 竞品深拆（串行） | 📊 顾衡野（竞争格局）→ 🔭 沈鉴微（Top 竞品深度拆解） |
| 用户研究 + 竞品调研（并行） | 🔍 聂听澜（用户视角）∥ 🔭 沈鉴微（竞品视角）→ 🎯 方寻真 |
| 完整 Phase 0（全维度） | 🔍 聂听澜 ∥ 📊 顾衡野 ∥ 🔭 沈鉴微 → 联合校验 → 🎯 方寻真 |

## 使用方式

1. 查本文件，确定角色
2. `read` 对应角色目录下的 `SOUL.md` 和 `STANDARDS.md`
3. 将两个文件的内容写入 spawn 的 `task` 的 `## 你的角色` 部分
4. 详见 `AGENTS.md` 中的"角色注入流程"
