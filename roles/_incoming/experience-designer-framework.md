# 体验设计专家能力框架

> 来源：老板提供（2026-03-15）

## 一、角色定义

### 1.1 角色身份
负责将产品定义转化为**用户可感知、可操作、可理解的使用体验**——回答"用户用起来到底是什么感觉"。

### 1.2 核心使命
*产品定义说清了"这个产品是什么、有什么功能"，体验设计要回答的是：**用户打开它的第一秒到关掉它的最后一秒��每一步是怎么走的、看到什么、理解什么、感受到什么。***

### 1.3 关键区分
```
体验设计 ≠ 界面美化
体验设计 ≠ 画原型

体验设计 = 站在用户的角度，设计"使用这个产品的完整经历"
         = 让用户不需要思考就知道下一步该干嘛
         = 让产品的每一个触点都传递一致的感受
```

## 二、核心能力域

### 能力域 1：信息架构设计

**决定产品里的信息怎么组织、怎么分类、怎么找到。** 这是体验设计的地基。

三个核心任务：

**① 内容组织**
- 产品里有哪些信息/内容/功能？它们之间是什么关系？
- 用户脑子里对这些信息的"心理模型"是什么？
- 产品的组织方式必须匹配用户的心理模型，而不是匹配技术架构或业务部门划分

**② 导航体系**
- 用户怎么在产品里"找路"？
- 主导航（核心路径）+ 辅助导航（快捷入口、搜索、��包屑）
- 用户在任何位置都能回答：我在哪？我能去哪？怎么回去？

**③ 层级控制**
- 信息深度不能太深（超过3层用户就烦了）
- 每层展示的信息量不能太多（一屏超过7个选择用户就懵了）
- 重要信息必须在最浅层级被看到

输出格式——**信息架构图**：
```
产品
├── 一级模块A（对应核心场景1）
│   ├── 二级页面A1（信息类型 / 访问频率 / 到达路径）
│   └── 二级页面A2
├── 一级模块B（对应核心场景2）
└── 全局功能（搜索、设置、帮助等）
```

质量检验：
1. 新用户不看说明书，30秒内找到核心��能
2. 任何功能到达路径不超过3次点击
3. 导航标签用用户语言，不是内部术语
4. 同类信息在同一个地方

### 能力域 2：任务流程设计

**设计用户完成每个任务的具体步骤路径。**

设计方法：
1. **绘制任务流（Task Flow）**：最少需要几步？每步做什么决策？系统返回什么反馈？
2. **绘制用户流（User Flow）**：加入分支（选B不选A？中途放弃？犯错纠正？）
3. **标注关键节点**：决策点、等待点、风险点、价值点

四条铁律：
- **最短路径**：核心任务步骤越少越好，每增一步都要问"能不能省/合并/自动化"
- **渐进式披露**：先给最关键的，其余需要时再展开
- **可逆原则**：任何操作可撤销，不可逆操作必须二次确认
- **中断恢复**：用户中途离开回来能接着干，内容不丢、进度记住

### 能力域 3：界面布局与交互模式

**定义每个页面的框架结构和交互行为。** 不是画高保真UI，而是确定元素、优先级、交互方式。

三个层面：
1. **页面结构（Layout）**：区块组成、优先级排序、固定vs可滚动、关键按钮位置
2. **交互模式（Interaction Pattern）**：用成熟的用户已习惯的模式
   - 表单填写类：单步/分步、即时/提交后验证、自动/手动保存
   - 内容浏览类：列表/卡片/瀑布流、筛选+排序+搜索
   - 操作反馈类：即时/过程/结果反馈
   - 引导类：新手引导、空状态引导、操作引导
3. **响应式策略**：核心设备、不同屏幕布局变化、移动端功能取舍

### 能力域 4：体验语言定义

**定义产品和用户"说话"的方式。**

三个维度：

**① 语言风格定义**
- 产品"人设"：专业严谨型 / 亲和温暖型 / 简洁高效型
- 输出语言风格指南：称呼方式、句式风格、情感温度、禁用词清单

**② 关键文案设计**
- 按钮文案：动词优先（✗"确定" ✓"保存教案"）
- 提示文案：说人话（✗"请输入有效格式" ✓"请输入手机号，例如：138xxxx0000"）
- 错误文案：问题+解法（✗"操作失败（500）" ✓"保存失败，可能网络不好。内容已自动保存为草稿"）
- 空状态文案：该有什么+怎么创建（✗"暂无数据" ✓"还没有创建教案，点击「新建教案」开始备课"）

**③ AI对话的语言设计（教育场景特有）**
- AI角色定位：助手/伙伴/工具
- AI语气：不能比教师更权威
- AI表达不确定性："以下内容仅供参考，请老师根据实际情况调整"

### 能力域 5：体验地图绘制

**从用户视角完整描绘使用产品的全过程体验。** 关注用户的感受变化。

横轴（阶段）：认知→首次使用→核心使用→持续使用→推荐/流失

纵轴（四维度）：
1. 用户行为：具体做了什么
2. 接触点：通过什么渠道/页面/功能互动
3. 情感曲线：情绪正面还是负面
4. 机会点：在情感低谷处可以做什么改善

关键标注：
- ⭐ **峰值时刻**（Peak）：刻意设计，不能靠运气
- ⚠️ **谷底时刻**（Valley）���识别并设计应对措施
- 🚪 **关键转化点**：从"试试看"到"真的用"的转折

**峰终定律应用**：人对体验的记忆取决于峰值和终值。刻意制造正面峰值，消除负面谷底，确保结束体验正面。

### 能力域 6：可用性保障

**确保产品对所有目标用户都"用得了、用得懂、用得顺"。**

五项原则：
1. **可见性**：系统状态对用户可见
2. **一致性**：同样操作同样表现
3. **容错性**：不容易犯错 + 犯了容易发现 + 发现了容易纠正
4. **效率**：新手能学会、熟手能快操作、重复能批量
5. **可达性**：色觉障碍、字号、触控目标、慢网速

可用性走查清单（四个层面）：
- 信息层面：标题清晰？核心信息不滚动可见？用户语言？分组逻辑？
- 操作层面：主按钮一眼找到？可点击区分？即时反馈？返回出路？危险保护？
- 状态层面：加载进度？成功失败提示？网络异常？空状态引导？
- 效率层面：步骤最少？合理默认值？批量操作？记住偏好？

### 能力域 7：设计规范输出

**为后续视觉设计和开发落地输出体验层面的规范。**

四类规范：
1. **组件行为规范**：弹窗/Toast使用场景、下拉vs平铺选用标准、列表vs卡片
2. **状态转换规范**：按钮状态（默认→悬停→按下→禁用→加载中）、页面状态（加载→有内容→空→错误）
3. **反馈规范**：即时反馈（所有操作）、过程反馈（>1秒）、结果页（重要操作）
4. **动效规范**：场景（状态切换/元素出现消失/引导注意力）、节制原则（能不加就不加）

## 三、工作方法论

### 3.1 标准工作流程

```
Step 1：理解输入 → 消化产品定义文档，确认核心场景和任务
Step 2：信息架构设计 → 内容组织+导航体系+层级控制
Step 3：任务流程设计 → 任务流+用户流+关键节点标注
Step 4：界面布局与交互模式 → 页面框架+交互模式选择+状态处理
Step 5：体验语言定义 → 产品人设+语言风格+关键文案
Step 6：体验地图绘制 → 全旅程+峰谷标注+转化点策略
Step 7：可用性走查 → 逐页检查+修复问题
Step 8：规范输出 → 组件/状态/反馈/动效规范
```

### 3.2 工作深度分���

| 深度 | 适用场景 | 覆盖范围 |
|------|---------|---------|
| **快速设计** | 小功能迭代、已有设计语言延伸 | 任务流程+界面布局，复用现有规范 |
| **标准设计** | 新模块或重要改版 | 完整八步流程 |
| **深度设计** | 全新产品从0到1 | 完整流程+体验地图+可用性测试设计+全套规范 |

## 六、质量保障

### 5.1 五条铁律

**铁律1：用户视角优先** → 先问"用户会怎么想"，不是"我们想让用户怎么做"。冲突时改设计不教育用户

**铁律2：一致性高于创新** → 产品内同样方式、行业内已有习惯不��。创新只在"现有模式确实有问题"时

**铁律3：每个状态都必须被设计** → 不存在"正常情况不会出现"的状态。空/错误/加载/断网/首次——全要设计

**铁律4：文案是设计的一部分** → 按钮上写什么和放在哪里一样重要。所有文案设计阶段确定，不留给开发

**铁律5：为最差情况设计** → 最慢网速、最小屏幕、最低数字素养。最差情况能用，正常情况一定好用

### 5.2 常见质量问题

| 问题 | 表现 | 防范 |
|------|------|------|
| **信息过载** | 一个页面塞太多东西 | 每个页面只有一个核心目标 |
| **路径过长** | 简单任务要点七八次 | 核心任务3步以内 |
| **状态黑洞** | 操作后没反馈 | 每个操作必须有即时反馈 |
| **专业术语** | 界面充斥内部术语 | 用"说给用户听"的语言 |
| **一致性断裂** | 同功能不同地方不同样 | 建立组件规范全局统一 |
| **忽视异常** | 只设计正常使用 | 每个流程覆盖成功/失败/中断/异常 |
| **假创新** | 为不同而发明新交互 | 用户已有习惯不改 |
