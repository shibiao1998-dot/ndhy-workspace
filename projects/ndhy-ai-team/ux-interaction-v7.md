## 一、信息架构

### 1.1 全站导航结构

```
NDHY AI Org Dashboard
├── / Landing（S1 首次到达 · 访问频率：高 · 入口：直链/搜索）
├── /team 组织全景（S2 了解组织 · 访问频率：高 · 入口：导航/Landing 卡片）
├── /capabilities 能力矩阵（S3 评估能力 · 访问频率：中 · 入口：导航/Landing 卡片）
├── /activity 活动流（S4 验证活跃 · 访问频率：中 · 入口：导航/Landing 卡片）
└── /404 错误页（入口：无效 URL）
```

**导航体系**

| 导航类型 | 内容 | 位置 | 说明 |
|---------|------|------|------|
| 主导航 | 首页 · 团队 · 能力 · 活动 | 顶部 Navbar（固定） | 4 项，当前页高亮 |
| 快速导航 | 3 张导航卡片 | Landing 页底部 | 仅首页有，引导进入子页面 |
| 上下文导航 | 每页底部"继续探索" | 各子页面底部 | 推荐下一页，防止死胡同 |

**层级控制**

| 规则 | 本产品标准 |
|------|-----------|
| 最大信息层级 | 1 层（全部页面从主导航直达） |
| 每层选项数 | 4 个（首页/团队/能力/活动） |
| 核心功能所在层级 | 第 1 层（≤ 2 层 ✅） |
| 重要信息首屏可见 | ✅ 每页核心数据首屏展示 |

### 1.2 每页信息层级

**Landing `/`**
1. Hero Hook — 一句话定位 + 运行天数/任务数（跳动数字）
2. 核心数据卡片 — 专家数 / 技能数 / 模型数
3. 快速导航 — 3 张卡片引导进入子页面

**Team `/team`**
1. 页面标题 + 组织概述数据
2. 组织架构图 — 6 层可视化
3. 专家卡片网格 — 按层级分组的 30 张卡片

**Capabilities `/capabilities`**
1. 页面标题 + 能力总览数据
2. 技能矩阵 — 8 类技能分类展示
3. 模型概览 — 9 类模型 + 精选模型
4. 全链路覆盖图 — 需求→运营 6 阶段

**Activity `/activity`**
1. 页面标题 + 活跃度数据
2. 24h 活动时间线
3. Git 提交热力图 — 90 天
4. 最近提交列表

### 1.3 推荐浏览动线

**主动线**（典型用户路径）：
```
Landing → Team → Capabilities → Activity → 离开/回到 Landing
```

**快速路径**（目标明确的用户）：
```
Landing → 直接点击感兴趣的导航卡片 → 单页深度浏览 → 离开
```

**页面间引导设计**：每个子页面底部设置"继续探索"区域，推荐下一站。顺序遵循 S1→S2→S3→S4 的认知递进逻辑：先认识人（Team）→ 再看能力（Capabilities）→ 最后验证活跃（Activity）。

## 二、每页交互设计

### 2.1 Landing 页（`/`）

> 核心目标：5 秒内用数据 Hook 住访客，让他点进子页面。

**区块划分**

| 区块 | 组件 | 内容 | 占比 |
|------|------|------|------|
| A. Hero | HeroHook | 组织名 + tagline + 运行天数/累计任务数（跳动数字） | 首屏主体 |
| B. 数据卡片 | StatsCards | 3 张卡片：30 AI 专家 / 51 技能 / 142 模型 | 首屏下半 |
| C. 快速导航 | QuickNav | 3 张导航卡片：认识团队 / 能力矩阵 / 实时活动 | 第二屏 |

**交互行为**

| 元素 | 触发条件 | 行为 |
|------|---------|------|
| 运行天数/任务数 | 页面加载完成 | 从 0 跳动到实际值，时长 1.2s，缓动 easeOut |
| 数据卡片数字 | 滚动到可见（`client:visible`） | 从 0 跳动到实际值，时长 0.8s，三张卡片依次启动（间隔 150ms） |
| 数据卡片 hover | 鼠标进入 | 卡片轻微上浮（translateY -4px）+ 阴影加深，过渡 200ms |
| 导航卡片 | 点击 | 跳转到对应子页面。hover 时卡片上浮 + 箭头图标右移 4px |

**状态设计**

| 组件 | 正常态 | loading 态 | 空数据态 |
|------|--------|-----------|---------|
| HeroHook | 显示 tagline + 跳动数字 | 数字位置显示骨架占位条 | 显示 tagline，数字显示"—" |
| StatsCards | 3 张卡片 + 跳动数字 | 3 张骨架卡片（保持布局） | 卡片显示但数字为"—" |
| QuickNav | 3 张导航卡片 | 无需 loading（静态内容） | 永远显示（静态配置） |

**文案建议**

| 位置 | 文案 |
|------|------|
| Hero tagline | "一个人 + 30 个 AI 专家 = 一支完整产品团队" |
| Hero 数字标签 | "已运行 {n} 天" · "累计完成 {n} 个任务" |
| 数据卡片标签 | "AI 专家" · "专业技能" · "接入模型" |
| 导航卡片标题 | "认识团队" · "能力矩阵" · "实时活动" |
| 导航卡片描述 | "30 位专家如何分工协作" · "51 技能 × 142 模型的全栈覆盖" · "过去 24 小时发生了什么" |

### 2.2 Team 组织全景页（`/team`）

> 核心目标：让访客理解"30 个 AI 专家如何分层协作"。

**区块划分**

| 区块 | 组件 | 内容 |
|------|------|------|
| A. 页面标题 | SectionHeader | "组织全景" + 副标题 + 专家总数 |
| B. 组织架构图 | OrgChart（React） | 6 层组织结构可视化，每层显示层名 + 专家数 |
| C. 专家卡片网格 | LayerSection + ExpertGrid + ExpertCard | 按层级分组，每组标题 + 专家卡片网格 |

**交互行为**

| 元素 | 触发条件 | 行为 |
|------|---------|------|
| 组织架构图 · 层级条 | hover | 高亮当前层（其他层降低透明度），显示该层专家数和简述 |
| 组织架构图 · 层级条 | 点击 | 平滑滚动到页面下方该层的专家卡片区域 |
| 专家卡片 | 默认 | 显示 emoji + 名称 + 所属层级标签 |
| 专家卡片 | hover | 卡片翻转/展开效果，显示 capability（一句话职责描述）。卡片上浮 + 阴影加深 |
| 层级分组标题 | 滚动到可见 | 标题从左侧淡入，卡片依次从下方淡入（间隔 80ms） |

**状态设计**

| 组件 | 正常态 | loading 态 | 空数据态 |
|------|--------|-----------|---------|
| OrgChart | 6 层结构完整展示 | 6 个骨架条 | "组织数据加载中…" |
| ExpertCard | emoji + 名称 + 层级 | 骨架卡片（圆形 + 两行文字占位） | 不会出现（至少有 Leader） |
| ExpertGrid | 按层级分组的卡片网格 | 每组 4-6 个骨架卡片 | 某层为空时显示"该层级暂无专家" |

**文案建议**

| 位置 | 文案 |
|------|------|
| 页面标题 | "组织全景" |
| 页面副标题 | "{n} 位 AI 专家，覆盖产品研发全链路" |
| 层级分组标题 | 数据驱动，取 `org_layers[].name`，如"需求洞察层 · {n} 位专家" |
| 专家卡片 hover | 显示 `experts[].capability` 字段 |
| 架构图 hover | "{layer_name}：{expert_count} 位专家" |

### 2.3 Capabilities 能力矩阵页（`/capabilities`）

> 核心目标：让访客判断"这个团队的技术覆盖面够不够广"。

**区块划分**

| 区块 | 组件 | 内容 |

| A. 页面标题 | SectionHeader | "能力矩阵" + 总览数字（技能数/模型数） |

| D. 全链路覆盖图 | CapabilityChain（React） | 6 阶段横向流程图：需求→设计→开发→测试→部署→运营 |

**交互行为**

| 元素 | 触发条件 | 行为 |
|------|---------|------|
| 技能分类卡片 | 默认 | 折叠状态，显示分类图标 + 名称 + 技能数量 badge |

| 模型类型卡片 | hover | 显示该类型模型数量和简述 |

**状态设计**

| 组件 | 正常态 | loading 态 | 空数据态 |

| 位置 | 文案 |

## 三、全局交互规范

### 3.1 导航栏（Navbar）

| 行为 | 定义 |
|------|------|
| 定位方式 | 顶部固定（sticky），滚动时始终可见 |
| 当前页标识 | 当前页导航项下方显示高亮指示条 |
| 滚动变化 | 初始透明背景，滚动超过 60px 后添加背景色 + 底部分割线 + 轻微阴影 |
| 品牌标识 | 左侧显示组织名称，点击回到 Landing |
| 导航项顺序 | 首页 · 团队 · 能力 · 活动 |
| 导航项 hover | 文字颜色变化，过渡 150ms |

### 3.2 页脚（Footer）

| 内容 | 说明 |
|------|------|
| 数据更新时间 | "数据更新于 {_meta.generated_at 格式化}"，格式：YYYY-MM-DD HH:mm |
| 版权信息 | "NDHY AI Agent Team · 由 AI 组织自动生成" |
| 技术栈标识 | "Built with Astro + React"（可选，极简） |

### 3.3 页面切换

| 行为 | 定义 |
|------|------|
| 切换方式 | 标准浏览器导航（多页应用，非 SPA） |
| 过渡动画 | 无页面切换动画（SSG 多页，由浏览器原生处理） |
| 滚动恢复 | 导航到新页面时，页面从顶部开始 |
| 后退行为 | 浏览器后退按钮正常工作（标准多页行为） |

### 3.4 动效总规范

| 场景 | 时长 | 缓动 | 说明 |
|------|------|------|------|
| 数字跳动 | 800-1200ms | easeOut | Hero 区域 1200ms，卡片区域 800ms |
| 元素淡入 | 300ms | easeOut | 滚动触发的内容出现 |
| hover 效果 | 200ms | ease | 卡片上浮、颜色变化 |
| 手风琴展开/折叠 | 250ms | easeInOut | 技能分类展开 |
| 全链路节点亮起 | 每节点 200ms 间隔 | easeOut | 依次从左到右 |
| 时间线条目淡入 | 每条 100ms 间隔 | easeOut | 从上到下依次出现 |

**动效原则**：所有动效服务于信息传达，不做纯装饰性动画。`prefers-reduced-motion` 媒体查询时关闭所有动画，直接显示最终态。

### 3.5 404 页面

| 内容 | 说明 |
|------|------|
| 标题 | "页面走丢了" |
| 描述 | "你访问的页面不存在，可能是链接有误" |
| 主操作 | "回到首页"按钮 → 跳转 `/` |
| 辅助操作 | 4 个子页面链接列表（团队/能力/活动） |

## 四、用户浏览动线

### 4.1 四类典型用户路径

**路径 A — 好奇探索型**（技术从业者，从社交媒体来）
```
Landing（被数字 Hook）→ Team（看看都有谁）→ Capabilities（技术覆盖面）→ Activity（验证真实性）→ 带走认知离开
```
预计停留：3-5 分钟，4 页

**路径 B — 能力评估型**（投资人/合作方，带着目的来）
```
Landing（快速扫视）→ Capabilities（直奔能力评估）→ Team（看组织结构）→ 离开
```
预计停留：2-3 分钟，3 页

**路径 C — 活跃度验证型**（技术博主，找写作素材）
```
Landing（确认定位）→ Activity（看活动数据和 Git 提交）→ Team（补充组织信息）→ 离开
```
预计停留：2-4 分钟，3 页

**路径 D — 浅度访客**（随手点开，判断是否值得深看）
```
Landing（5 秒判断）→ 关闭 或 点一个导航卡片 → 看 30 秒 → 离开
```
预计停留：≤1 分钟，1-2 页

### 4.2 每页"下一步"引导设计

每个子页面底部（Footer 上方）设置**"继续探索"区域**，包含 2 张引导卡片，引向其他子页面：

| 当前页 | 推荐下一站（主） | 推荐下一站（辅） |
|--------|----------------|----------------|
| Landing | — | —（用 QuickNav 替代） |
| Team | "看看我们能做什么" → /capabilities | "查看实时活动" → /activity |
| Capabilities | "验证我们的活跃度" → /activity | "认识团队" → /team |
| Activity | "认识团队" → /team | "查看能力矩阵" → /capabilities |

**引导卡片交互**：与 Landing 的 QuickNav 卡片一致（hover 上浮 + 箭头右移），保持全站交互一致性。

### 4.3 浏览终点设计

用户完成所有页面浏览后（已在最后一个子页面），"继续探索"引导回到其他子页面形成闭环——没有"到此结束"的感觉，但也不强迫用户继续。页脚的数据更新时间戳是最终的视觉锚点，暗示"这是一个持续更新的活体系统"。

## 五、设计决策记录

| # | 决策点 | 选择 | 替代方案 | 理由 |
|---|--------|------|---------|------|
| 1 | 数字动画触发时机 | Hero 区 = 页面加载；卡片区 = 滚动可见 | 全部滚动可见 | Hero 是第一印象，必须立即生效；卡片区滚动触发增加发现感 |
| 2 | 专家卡片交互深度 | hover 展示 capability | 点击弹窗展示详情 | MVP 无 F7（专家详情），hover 展示一句话足够；V2 加详情弹窗 |
| 3 | 技能分类交互 | 手风琴（同时只展开一个） | 全部默认展开 | 51 个技能全部展开页面过长，手风琴控制信息密度 |
| 4 | 页面切换动画 | 无（标准多页导航） | 全站 SPA + 页面过渡 | SSG 架构，标准多页导航性能最优；过渡动画对信息获取无价值 |
| 5 | 导航栏滚动行为 | 固定 + 滚动后加背景 | 隐藏后上滑显示 | 仅 4 个页面，导航始终可见减少迷路概率 |
| 6 | 活动时间线交互 | 静态列表 + 淡入动画 | 可筛选/可展开 | MVP 最多 30 条数据，无需筛选；V2 按类型筛选 |
| 7 | 全链路覆盖图 | 水平流程图 + 依次亮起 | 圆形/网状关系图 | 线性流程图匹配"需求到交付"的线性心智模型 |
