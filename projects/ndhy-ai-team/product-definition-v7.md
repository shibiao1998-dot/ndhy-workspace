## 一、产品概念

### 1.1 产品概念卡

**【产品名称】** NDHY AI Org Dashboard  
**【一句话定义】** 一个多页静态网站，把 NDHY AI Agent Team 的实时运行数据（专家、技能、活动、成本、代码提交）自动聚合展示，让外部访客在 3 分钟内看懂"一个人 + 30 个 AI 专家"的组织是怎么运转的。  
**【产品形态】** 多页静态网站（Astro + React），数据由 cron 定时生成的 data.json 驱动，部署在 GitHub Pages  
**【核心用户】**
- 主用户：技术从业者（想了解 AI Agent 多智能体协作的真实案例）
- 次要用户：产品经理（想看人机混合团队的实际运行效果）、AI 行业关注者（关注 Agent 生态动态）
- 利益相关者：潜在合作方 / 投资人（评估团队技术能力和运营成熟度）

**【核心场景】** 访客通过社交媒体/搜索引擎到达首页，被实时数据 Hook 住，按兴趣导航到组织全景/能力矩阵/活动流页面，带走"这个 AI 组织是真的在跑"的确信  
**【核心任务】** 用真实运行数据证明 AI 组织的可行性，替代传统营销叙事  
**【核心价值交付】**
- 用户付出：3-5 分钟浏览时间
- 用户获得：对 AI Agent 协作模式的具象理解 + 对 NDHY 团队能力的判断依据

### 1.2 五个核心问题

| 序号 | 问题 | 回答 |
|------|------|------|
| 1 | 它是什么（产品形态） | 多页静态网站，由 cron 驱动 data.json 自动更新，不是人工维护的营销页 |
| 2 | 给谁用（核心用户） | 技术从业者、产品经理、AI 行业关注者、潜在合作方/投资人 |
| 3 | 什么情况下用（核心场景） | 看到分享链接/搜到 NDHY → 想了解"AI 组织到底是什么" → 进入网站 |
| 4 | 用它干什么（核心任务） | 浏览实时数据，理解 AI 组织的架构、能力和运行状态 |
| 5 | 用完得到什么（核心价值） | "这个 AI 组织是真的在运转"的确信，以及对协作模式的具象认知 |

### 1.3 出租车司机测试

> "哦，就是一个能自动更新的网站，展示一个全是 AI 的团队每天在干什么、有多少人、花了多少钱、写了多少代码。"

## 二、场景定义

### 2.2 场景故事

#### 场景 S1：首次到达 — 被数据 Hook 住

**谁**：张明，某大厂后端工程师，关注 AI Agent 生态  
**何时**：周二午休刷 Twitter，看到一条"我们公司除了我全是 AI"的帖子  
**何地**：手机浏览器  
**因为什么**：好奇点开链接，想看看是不是又一个 PPT 团队  
**做了什么**：
1. 页面加载，看到一句话 Hook + 跳动的实时数字："已运行 247 天，完成 12,580 个任务"
2. 往下看到 3 张核心数据卡片（30 专家 / 51 技能 / 今日 ¥X.XX）
3. 心想"数字看起来不像编的"，点击"认识团队"卡片  

**得到了什么**：从"随便看看"变成"有点意思"，主动进入了子页面

> **场景边界**  
> - 起点：访客到达首页（URL 加载完成）  
> - 终点：点击任一导航卡片，或关闭页面  
> - 排除项：不含 SEO 优化过程、不含社交媒体传播链路

#### 场景 S2：了解组织 — 这群 AI 都是谁

**谁**：张明（续 S1），想看看这 30 个"专家"到底怎么分工  
**何时**：紧接着 S1，还在午休  
**何地**：同上  
**因为什么**：点了"认识团队"进入 `/team` 页面  
**做了什么**：
1. 看到组织架构图：6 个层级，从 Leader 到各领域专家
2. 点击某个专家卡片（如"🖥️ 前端开发专家"），看到它的职责、技能等级、负责项目
3. 浏览了 4-5 张卡片，理解了"产品、设计、开发、测试、运维"的完整分工  

**得到了什么**：理解了 AI 组织的分工模式——"原来是用不同 prompt/设定造出不同角色的 Agent"

> **场景边界**  
> - 起点：进入 `/team` 页面  
> - 终点：理解组织结构后导航到其他页面或离开  
> - 排除项：不含专家的详细技能执行历史、不含专家间的实时通信记录

#### 场景 S3：评估能力 — 能做什么、用什么做

**谁**：李芳，某 AI 创业公司 CTO，正在寻找技术合作方  
**何时**：周五下午，同事转发了 NDHY 的链接  
**何地**：笔记本电脑，Chrome 浏览器  
**因为什么**：想评估这个团队的技术深度，能不能做复杂项目  
**做了什么**：
1. 从首页直接点击"能力矩阵"进入 `/capabilities`
2. 看到 51 个技能的分类展示（产品/设计/开发/测试/运维/AI）
3. 看到 142 个模型的接入列表（文本/图像/视频/语音/嵌入）
4. 看到全链路覆盖图：需求→设计→开发→测试→部署  

**得到了什么**：判断出"这个团队的技术栈覆盖面够广，不是只会聊天的 chatbot"

> **场景边界**  
> - 起点：进入 `/capabilities` 页面  
> - 终点：对能力覆盖形成判断后离开或转到其他页面  
> - 排除项：不含技能的实际调用次数统计、不含模型的性能基准测试数据

#### 场景 S4：验证活跃 — 这组织是不是真的在跑

**谁**：王磊，技术博主，想写一篇关于 AI Agent 团队的文章  
**何时**：周六上午，在收集素材  
**何地**：笔记本电脑  
**因为什么**：看了首页数字后想验证"这些数字是不是每天更新的"  
**做了什么**：
1. 进入 `/activity` 页面
2. 看到时间线：最近 24 小时内的组织活动（如"14:32 🖥️ 前端开发专家完成组件重构"）
3. 看到 Git 提交热力图：最近 90 天的提交频率分布
4. 点击某天查看具体提交记录  

**得到了什么**：确认"数据是真实的，这个组织每天都在活动"，决定写到文章里

> **场景边界**  
> - 起点：进入 `/activity` 页面  
> - 终点：确认活跃度真实性后离开  
> - 排除项：不含完整的 Git diff 内容、不含活动的详细执行日志

### 3.3 关于成本透明的专项决策

**为什么 MVP 不含成本透明（F6）**：

成本数据是一把双刃剑：
- ✅ **正面**：公开成本体现透明和自信，是差异化亮点
- ❌ **风险**：外部访客可能对数字产生误读（"才花这么少？不靠谱" 或 "花这么多？烧钱"），且成本数据涉及商业敏感性

**建议路径**：MVP 上线后先用首页的"累计任务数"间接传递效率信息。收集用户反馈后，在 V2 以"效率指标"而非"原始成本"的方式呈现（如"每任务平均成本 ¥X.XX"）。

## 四、产品规则

### 4.1 准入规则

| 序号 | 规则 | 适用对象 | 违反处理 |
|------|------|---------|---------|
| R1 | 网站公开访问，无需注册登录 | 所有访客 | N/A |
| R2 | data.json 必须由 cron 脚本生成，禁止人工编辑 | 数据维护者 | 人工编辑的数据不被部署流程接受 |

### 4.2 流转规则

| 序号 | 规则 | 流转路径 | 触发条件 |
|------|------|---------|---------|
| R3 | 数据更新流转：workspace 变更 → Python 扫描 → data.json → git push → GitHub Pages 部署 | 单向自动 | cron 定时触发（每日 1 次） |
| R4 | 页面导航流转：首页 → 子页面（Team/Capabilities/Activity），子页面间可互跳 | 用户主动点击 | 点击导航/卡片 |

### 4.3 限制规则

| 序号 | 规则 | 限制内容 | 上限值 | 超限处理 |
|------|------|---------|-------|---------|
| R5 | 活动流展示时间窗口 | 仅展示最近 24 小时活动 | 24h | 超过 24h 的自动不显示 |
| R6 | Git 热力图时间范围 | 展示最近 90 天提交 | 90 天 | 更早数据不加载 |
| R7 | 首页数字更新延迟 | 数据最大滞后 | 24 小时 | 页面底部标注"数据更新于 YYYY-MM-DD HH:mm" |
| R8 | 单页内容量上限 | 组织全景页单次加载专家数 | 40 个 | 超过时分组折叠 |

### 4.4 异常规则

| 序号 | 异常场景 | 处理方式 | 用户感知 |
|------|---------|---------|---------|
| E1 | data.json 生成失败（Python 脚本异常） | 保留上一次有效 data.json，不部署空数据 | 用户看到的数据停留在上次成功更新（页面底部时间戳可识别） |
| E2 | GitHub Pages 部署失败 | 自动重试 1 次；失败则保持当前版本在线 | 用户无感知 |
| E3 | 访客访问不存在的页面（404） | 自定义 404 页面，引导回首页 | 看到友好的 404 提示 + 首页链接 |
| E4 | workspace 当天无活动（活动流为空） | 显示"今日暂无活动"+ 展示最近一次活动 | 不显示空白页面 |
| E5 | 数据量异常（专家数量突变、任务数归零） | 数据扫描脚本内置合理性校验，异常数据不覆盖 | 用户无感知 |

| workspace 数据（EXPERTS.md、skills/、memory/） | **上游数据源** | V7 只读消费，不写入不修改 workspace 数据 |
| GitHub 仓库（git log） | **上游数据源** | 只读提取提交历史 |

### 6.2 与外部系统的关系

| 外部系统 | 对接需求 | 对接深度 | 依赖程度 |
|---------|---------|---------|---------|
| GitHub Pages | 静态页面托管 | 中度（CI/CD 部署） | 强依赖（不可用则网站不可访问）；替代方案：Docker 自部署 |
| GitHub API（git log） | 提取提交历史 | 轻度（定时读取） | 弱依赖（不可用则热力图暂停更新，其他功能正常） |
| cron 宿主机 | 执行 Python 扫描脚本 | 中度 | 强依赖（不可用则数据停止更新）；替代方案：手动触发 |

### 6.3 与未来演进的关系

| 维度 | 说明 |
|------|------|
| **本版定位** | 过渡版（MVP）— 验证"用实时数据替代营销叙事"这个核心假设 |
| **预留接口** | ① data.json schema 预留扩展字段（`_meta`、`version`） ② 页面组件化设计，支持新页面快速扩展 ③ 导航结构支持动态添加页面 |
| **刻意不预留** | ① 不预留用户系统（除非未来明确需要互动功能） ② 不预留实时推送架构（与静态站点定位矛盾，未来需要时做架构切换而非渐进扩展） |
| **V7→V8 演进方向** | ① 成本透明模块上线（基于 MVP 反馈决定展示粒度） ② 项目 Showcase（展示 AI 组织完成的具体项目） ③ 时间旅行（查看组织任意历史时间点的状态） |
| **演进触发条件** | MVP 运行 1 个月，核心指标达标 + 用户反馈收集完成 |
