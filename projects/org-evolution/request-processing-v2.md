# 需求处理流程 V2 — 设计方案

> 目标：让老板的角色从"全程纠偏"变为"审批门控 + 验收交付"
> 设计依据：Superpowers 框架分析 + Context Engineering 调研 + Agent 协作方案 + 一周实战教训
> 设计者：Leader（发发）
> 日期：2026-03-18

---

## 一、问题诊断

### 当前痛点
老板花大量时间纠偏 Leader，而不是做产品决策。

### 根因链
```
Leader 收到需求
  → 习惯性"我理解了，直接干"
    → 跳过需求挖掘（信息不够就决策）
    → 跳过能力扫描（用错专家）
    → 跳过流程设计（各干各的，没有协作）
      → 产出偏差
        → 老板纠偏
          → Leader 记录教训
            → 下次依然跳（规则靠自觉，自觉不可靠）
```

### 核心矛盾
**规则已经够多了，缺的是执行机制。** MEMORY.md 有 20+ 条规则，AGENTS.md 有宪法，但没有任何东西阻止我跳步。

---

## 二、设计原则

1. **可见性**：每个环节产出可见的制品（artifact），老板一眼看到哪步做了、哪步没做
2. **不可跳过**：门控是硬性的，不通过不能进下一步——通过"必须输出制品"来强制
3. **分级处理**：不是所有需求都走全流程，但分级本身必须经过老板确认
4. **Agent 互查**：关键节点有第二个 Agent 质疑，不依赖 Leader 单点判断
5. **低摩擦**：对老板来说，审批成本要低——看制品 + 一句话确认/否决

---

## 三、需求分级

Leader 收到需求后，**第一件事是分级**，分级结果必须告知老板确认。

| 级别 | 判断标准 | 示例 | 必经门控 |
|------|----------|------|----------|
| **⚡ 快速** | 单专家、有先例、边界明确、≤1 天 | "把这个文档翻译成英文"、"跑一下安全审计"、"查个技术问题" | G0 → G3 → G5 |
| **🔧 标准** | 2-3 专家、有依赖、需要设计 | "做一份竞品分析报告"、"设计一个 API 契约"、"优化某个页面" | G0 → G1 → G2 → G3 → G5 |
| **🏗️ 复杂** | 4+ 专家、跨领域、无先例、需要多阶段 | "做一个产品官网"、"开发一个新功能模块"、"搭建设计系统" | G0 → G1 → G1.5 → G2 → G3 → G4 → G5 |

**防误判机制**：Leader 倾向于把"标准"判成"快速"。对策——
- 涉及 2+ 专业领域 → 自动升级为"标准"
- 涉及代码开发 → 最低"标准"
- 涉及用户体验/视觉 → 最低"标准"
- 老板说"这个很重要" → 自动"复杂"

## 四、门控体系

### G0：需求接收门（所有级别必经）

**触发**：老板发来任何需求

**Leader 必须输出**：

```
📋 需求接收卡

【需求原文】：{老板说的原话}
【我的理解】：{一句话概括}
【分级判断】：⚡快速 / 🔧标准 / 🏗️复杂
【分级理由】：{为什么是这个级别}
【涉及领域】：{列出专业领域}
【对应专家】：{从 EXPERTS.md 查}
【缺口】：{有/无，缺什么}
【下一步】：{根据级别，下一个 Gate 是什么}

→ 等待老板确认分级
```

**老板的动作**：看卡 → 确认/调整分级 → 一句话即可

**通过条件**：老板确认分级

---

### G1：需求理解门（标准 + 复杂）

**触发**：G0 通过

**执行**：spawn 🎯 需求分析专家，与老板深度对话

**产出制品**：结构化需求文档（由需求分析专家产出）

**质量门**：
- [ ] 需求文档包含：目标、场景、用户、期望、边界、验收标准
- [ ] 老板确认需求文档准确

**通过条件**：老板确认需求文档

---

### G1.5：对抗审查门（仅复杂级别）

**触发**：G1 通过

**执行**：spawn 📐 产品定义专家，阅读需求文档后提出质疑

**对抗规则**：
- 产品定义专家必须提出 ≥3 个质疑点（不是走过场）
- 质疑维度：可行性、范围蔓延风险、优先级合理性、遗漏场景
- 需求分析专家回应质疑
- 最多 2 轮对抗，Leader 整合分歧点

**产出制品**：对抗记录（质疑 → 回应 → 结论）

**通过条件**：关键分歧已解决，或标注为"需老板裁决"

---

### G2：方案设计门（标准 + 复杂）

**触发**：G1（标准）或 G1.5（复杂）通过

**Leader 必须做的事**（不是可选的）：

**Step 2a — 能力覆盖扫描**：
```
📋 能力覆盖表

| 所需能力 | 对应专家 | 状态 |
|----------|---------|------|
| {领域1} | {专家名} | ✅ 就位 / ❌ 缺失 |
| {领域2} | {专家名} | ✅ 就位 / ❌ 缺失 |
...

【缺口处理】：{无缺口 / 需创建 XX 专家}
```

**Step 2b — 协作流程设计**：
- **标准级**：Leader 直接设计简单流程（串行/并行）
- **复杂级**：spawn 🔩 流程设计专家，产出完整协作流程图

**Step 2c — 指令包预览**：
对每个要 spawn 的专家，预览指令包的 7 层摘要（不是完整指令包，是摘要）

**产出制品**：执行方案（能力覆盖表 + 协作流程 + 指令包摘要）

**通过条件**：老板确认执行方案

---

### G3：执行审批门（所有级别必经）

**触发**：
- 快速级：G0 通过后直接到这里
- 标准/复杂：G2 通过后到这里

**Leader 输出**：

```
📋 执行审批卡

【任务概要】：{做什么}
【执行模式】：subagent / ACP / 混合
【要 spawn 的专家】：

| 专家 | 任务 | 模式 | 预估时间 |
|------|------|------|----------|
| XX | XX | subagent/ACP | XX min |

【对抗环节】：{有/无，哪些节点}
【风险】：{已识别的风险}
【预估总耗时】：XX 分钟

→ 老板确认后开始执行
```

**老板的动作**：看卡 → 批准/调整 → 一句话

**通过条件**：老板批准

---

### G4：执行监控门（复杂级别 + 标准级中的 ACP 任务）

**触发**：G3 通过，任务开始执行

**Leader 必须做的事**：
- 每 5 分钟巡查一次（写入 HEARTBEAT.md）
- 检查文件产出、build 状态
- 发现偏差 → sessions_send 纠偏
- 10 分钟无产出 → 终止 + 拆分

**产出制品**：巡查日志（在 memory/当日.md 中记录）

**异常升级**：
- 连续 2 次巡查无产出 → 通知老板 + 拆分方案
- 任务完成但质量有疑问 → 进入 G5 前告知老板

---

### G5：交付验收门（所有级别必经）

**触发**：执行完成

**验收流程按级别**：

| 级别 | 验收流程 |
|------|---------|
| ⚡ 快速 | Leader 形式检查 → 直接交付老板 |
| 🔧 标准 | Leader 内容检查 → 交付老板 |
| 🏗️ 复杂 | 专家走查链（对抗审查）→ Leader 终审 → 交付老板 |

**复杂级走查链**：
- 代码类：代码审查专家 → 测试专家 → Leader
- UI 类：UI 设计专家 → 体验设计专家 → 产品视角 Leader
- 文档类：相关领域专家交叉审查 → Leader

**产出制品**：交付报告

```
📋 交付报告

【需求回顾】：{原始需求一句话}
【完成情况】：{做了什么}
【验收标准对照】：
  - [ ] AC1：XX → ✅/❌
  - [ ] AC2：XX → ✅/❌
【产出物位置】：{文件路径}
【遗留问题】：{有/无}
【建议下一步】：{有/无}
```

**通过条件**：老板验收

## 五、Agent 互查机制

### 互查原则
- 不是所有环节都需要互查（成本太高）
- 只在**偏差代价最高的节点**引入互查
- 互查 Agent 拿到上游产出 + 质疑 brief，不是从零做

### 互查节点

| 节点 | 触发条件 | 互查对 | 互查方式 |
|------|----------|--------|---------|
| 需求确认 | 复杂级 G1.5 | 🎯 需求分析 → 📐 产品定义质疑 | 产品定义读需求文档，提 ≥3 个质疑 |
| 技术方案 | 涉及架构决策 | 🔬 技术调研 → 🏛️ 技术架构质疑 | 架构师审可行性/成本/风险 |
| 代码交付 | 所有代码类任务 | 🖥️ 开发 → 🔍 代码审查 | 审查专家独立审查 + 反馈 |
| UI 交付 | 所有 UI 类任务 | 🖥️ 开发 → 🖌️ UI + 🎨 体验走查 | 浏览器实际走查，不是读代码 |

### 互查执行方式

```
Step 1: 上游专家完成产出 → 写入共享文件
Step 2: Leader spawn 互查专家，task 包含：
  - 上游产出路径
  - 质疑 brief（你需要检查什么、从什么角度）
  - 输出格式（质疑清单：问题 + 严重程度 + 建议）
Step 3: 互查专家产出质疑清单
Step 4: Leader 判断：
  - 全部 🟢 → 通过
  - 有 🟡 → Leader 自行修正或记录 TODO
  - 有 🔴 → 回退上游修复（最多 2 轮）
```

## 六、上下文工程化 — 指令包质量保证

### 问题
指令包质量参差不齐，有时关键信息缺失导致专家产出偏差。

### 解法：指令包 7 层检查模板

每次组装指令包时，Leader 必须逐层检查：

```
📋 指令包检查单（内部使用，不发给老板）

□ L1 项目上下文：是否包含项目名、目标、当前阶段？
□ L2 上游产出物：是否附上了前序专家的产出文件路径？内容是否最新？
□ L3 任务说明：任务边界是否清晰？"做什么"和"不做什么"都写了吗？
□ L4 验收标准：AC 是否 SMART 化？能否客观判断通过/不通过？
□ L5 约束与禁区：有没有容易踩的坑？前序任务踩过的坑写了吗？
□ L6 必须使用的技能：查了 STANDARDS.md 吗？适用技能都列出了吗？
□ L7 执行模式：ACP 还是 subagent？ACP 三条铁律写了吗？
```

### 上下文完整性的自检信号

如果组装指令包时发现：
- L2 上游产出物为空 → 说明前序步骤被跳过了，**停下来，回退**
- L4 验收标准写不出来 → 说明需求还不够清晰，**回退到 G1**
- L6 找不到适用技能 → 考虑是否需要创建新技能

## 七、老板角色变化

### Before（当前）
```
老板 → 发需求 → 发现 Leader 跳步 → 纠偏 → Leader 调整 → 再发现问题 → 再纠偏...
参与度：80%（大部分时间在纠偏）
```

### After（V2）
```
老板 → 发需求
  → 看需求接收卡 → 确认分级（10 秒）
  → [标准/复杂] 看需求文档 → 确认（1 分钟）
  → 看执行方案 → 批准（30 秒）
  → [等待执行，不需要盯]
  → 看交付报告 → 验收（按产出复杂度）
参与度：20%（只在门控点决策）
```

### 老板的 4 个决策点

| 决策点 | 老板看什么 | 老板做什么 | 耗时 |
|--------|-----------|-----------|------|
| G0 分级确认 | 需求接收卡 | 确认/调整分级 | ~10s |
| G1 需求确认 | 需求文档 | 确认准确性 | ~1-3min |
| G3 执行审批 | 执行审批卡 | 批准/调整 | ~30s |
| G5 交付验收 | 交付报告 + 产出物 | 验收 | 视复杂度 |

## 八、实施计划

### Phase 1：立即生效（今天）
- 将此方案写入 AGENTS.md 作为宪法级流程
- 创建 `leader-preflight` Skill：封装 G0-G3 的模板和检查逻辑
- 下一个需求开始严格执行

### Phase 2：一周内
- 实战 3-5 个需求，观察门控效果
- 收集老板反馈：哪些门控有价值、哪些是多余的
- 调整分级标准和门控细节

### Phase 3：两周内
- 基于实战数据优化
- 评估 Agent 互查的实际 Token 成本和质量提升
- 考虑是否需要自动化某些门控（如指令包完整性自动检查）

## 九、风险与对策

| 风险 | 对策 |
|------|------|
| Leader 在 G0 分级时故意低判 | 防误判规则已设计（2+ 领域自动升级） |
| 门控过多导致简单任务变慢 | 快速级只有 3 个门控（G0→G3→G5），最快 1 分钟走完 |
| Agent 互查增加 Token 成本 | 只在偏差代价最高的节点互查，不是每步都查 |
| 老板审批变成瓶颈 | 审批卡设计为"扫一眼即可判断"，不需要深度阅读 |
| 新流程初期 Leader 不适应 | Phase 2 观察期允许试错，老板随时可以叫停或调整 |
