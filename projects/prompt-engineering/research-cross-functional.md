## 1. 产品↔开发

### 沟通内容

| 阶段 | 内容 | 方向 |
|------|------|------|
| 需求评审 | PRD/用户故事、验收标准（AC）、优先级 | 产品→开发 |
| 可行性反馈 | 技术约束、工期评估、替代方案建议 | 开发→产品 |
| 变更管理 | 需求变更请求、影响分析、重新排期 | 双向 |
| 澄清 | 边界 case 确认、业务规则细化 | 开发→产品→开发 |

### 沟通时机

- **需求评审**：Sprint Planning 前 1-2 天，确保开发有时间消化
- **可行性反馈**：评审中即时反馈 + 评审后 24 小时内书面确认
- **变更管理**：变更发生时立刻通知，不等到下一个计划会
- **持续澄清**：开发过程中随时（但应先查已有文档）

### 沟通方式

| 方式 | 适用场景 |
|------|---------|
| 需求评审会 | 正式评审，所有相关开发参加，走读 PRD + AC |
| 异步文档评审 | 评审前预读，评审后补充问题 |
| 即时消息 | 快速澄清、非阻塞性问题 |
| 变更日志 | 所有需求变更必须书面记录，包含原因、影响范围、审批人 |

### 常见问题

1. **需求模糊**：AC 不够具体，不同开发者理解不一致 → 评审时未暴露，开发后才发现
2. **可行性评估滞后**：开发没参加需求讨论，评审时才发现技术不可行 → 推倒重来
3. **变更无管控**：需求口头变更，不记录不评估影响 → scope creep（需求蔓延）
4. **过度评审**：每个细节都开会讨论 → 评审疲劳，真正重要的点反而被忽略
5. **信息断层**：产品和开发使用不同的术语体系 → 理解偏差

### 最佳实践

1. **Definition of Ready（DoR）**：需求进入开发前必须满足的条件清单（AC 完整、UI 稿就绪、技术方案确认）
2. **Three Amigos 会议**：产品 + 开发 + 测试三方在开发前对齐每个用户故事，确保理解一致
3. **需求 Spike**：对不确定需求先做时间盒调研（timeboxed spike），再承诺工期
4. **变更影响矩阵**：变更前评估对已完成/进行中工作的影响，量化成本后再决策
5. **Living Document**：PRD 不是一次性文档，随开发进展持续更新，保持单一事实源（SSOT）

## 2. 产品↔测试

### 沟通内容

| 阶段 | 内容 | 方向 |
|------|------|------|
| 验收标准对齐 | AC 的可测性审查、边界条件补充 | 双向 |
| 测试策略 | 测试范围、优先级、环境要求 | 测试→产品确认 |
| UAT（用户验收测试） | 验收场景、通过标准、签收流程 | 产品主导 |
| 缺陷分级 | 缺陷严重性 vs 优先级判定、修复排期 | 双向 |
| 发布决策 | 残留缺陷风险评估、Go/No-Go | 双向 |

### 沟通时机

- **AC 对齐**：需求评审阶段（不是开发完了才写测试用例）
- **测试策略**：Sprint 开始时确定本轮测试重点
- **缺陷分级**：缺陷报告时即判定，争议时 24 小时内三方会议裁决
- **UAT**：功能开发完 + 内部测试通过后，发布前
- **Go/No-Go**：发布前 1 天

### 沟通方式

| 方式 | 适用场景 |
|------|---------|
| AC 走读 | 需求评审中嵌入，测试视角审查 AC |
| 缺陷管理工具 | Jira/Linear 等，结构化记录缺陷状态 |
| UAT 检查清单 | 结构化验收场景 + 通过/不通过标记 |
| Bug Triage 会议 | 定期（每日或隔日）缺陷分级和排期 |

### 常见问题

1. **AC 不可测**："用户体验要好"这类模糊 AC 无法转化为测试用例
2. **验收标准后补**：开发完了才定 AC → 测试只能基于猜测写用例
3. **缺陷优先级争议**：开发认为是低优先级，产品认为影响发布 → 拉锯
4. **UAT 走过场**：产品没时间认真验收，点两下就签收 → 上线后出问题
5. **测试左移不到位**：测试只在开发后介入 → 发现问题成本高

### 最佳实践

1. **AC SMART 化**：AC 必须具体（Specific）、可衡量（Measurable）、明确（Achievable/Agreed）、相关（Relevant）、有边界（Time-bound/Testable）
2. **测试左移（Shift Left）**：测试在需求阶段就介入，用测试视角补充遗漏场景
3. **缺陷分级标准**：团队统一的分级定义（P0 阻断/P1 严重/P2 一般/P3 建议），减少争议
4. **UAT 脚本化**：提前准备 UAT 场景和数据，不是临场发挥
5. **探索性测试**：除了按 AC 测试，鼓励基于经验的探索性测试发现意外缺陷

## 3. UX/UI↔产品

### 沟通内容

| 阶段 | 内容 | 方向 |
|------|------|------|
| 需求对齐 | 用户画像、使用场景、核心任务流 | 产品→UX |
| 设计方向 | 设计理念、竞品参考、约束条件 | 双向 |
| 设计评审 | 交互方案、信息架构、用户旅程 | UX→产品评审 |
| 设计决策 | 多方案抉择、优先级取舍 | 双向 |
| 用户反馈 | 可用性测试结果、数据反馈 | UX→产品 |

### 沟通时机

- **需求对齐**：产品定义阶段（PRD 草稿完成后、正式评审前）
- **设计方向**：开始画稿前，确保方向正确再投入
- **设计评审**：低保真→高保真→终稿，至少 2 轮
- **设计变更**：需求变更时同步通知 UX，评估设计影响

### 沟通方式

| 方式 | 适用场景 |
|------|---------|
| 设计评审会 | 关键节点的正式评审（低保真、高保真） |
| 设计看板 | Figma/Sketch 在线协作，异步评论 |
| 用户故事映射 | 对齐用户旅程和功能优先级 |
| 原型演示 | 可交互原型的走读和反馈 |

### 常见问题

1. **设计方向错误**：产品没传达清楚目标用户和场景 → UX 设计偏离
2. **过度设计**：UX 追求完美体验，但超出当前技术/时间约束
3. **美学 vs 功能冲突**：UX 优化体验 vs 产品要求塞更多功能
4. **评审反馈碎片化**：多人评审意见矛盾 → UX 无所适从
5. **需求变更不通知 UX**：产品调整需求但忘记同步设计侧

### 最佳实践

1. **设计 Brief**：产品提供结构化设计任务书（目标用户、核心场景、约束条件、成功标准）
2. **渐进式评审**：先对齐方向（概念草图）→ 再细化方案（低保真）→ 最后打磨细节（高保真）
3. **设计决策记录（DDR）**：记录为什么选 A 不选 B，避免反复翻烧饼
4. **产品和设计的术语表**：统一概念命名，减少理解偏差
5. **评审规则**：一人汇总评审意见，冲突时由产品基于业务目标裁决

## 4. UX/UI↔开发

### 沟通内容

| 阶段 | 内容 | 方向 |
|------|------|------|
| Design Handoff | 设计稿、标注、切图、交互说明 | UX→开发 |
| 技术约束反馈 | 实现难度评估、替代方案 | 开发→UX |
| Design Token | 颜色、字体、间距、圆角等设计变量 | UX→开发 |
| 设计走查 | 实现与设计稿的逐屏对比 | UX 主导，开发配合 |
| 组件复用 | 组件库同步、新组件需求 | 双向 |

### 沟通时机

- **Design Handoff**：设计评审通过后、开发开始前
- **技术约束**：Handoff 时或更早（设计阶段就应沟通可行性）
- **Design Token**：项目初期定义 → 后续增量更新
- **设计走查**：每个功能开发完成、提测前

### 沟通方式

| 方式 | 适用场景 |
|------|---------|
| 设计工具标注 | Figma/Zeplin 自动标注 + 手动补充说明 |
| Handoff 文档 | 交互细节、动画参数、异常状态说明 |
| Design Token 系统 | 代码化的设计变量（CSS Variables / Theme） |
| 走查 Checklist | 逐屏对比，标记差异和偏差等级 |
| 组件文档 | Storybook 等组件库文档 |

### 常见问题

1. **标注不全**：异常状态（空状态、加载中、错误）没有设计稿 → 开发自己发挥
2. **设计与实现偏差**：间距差 2px、颜色色值不对 → 视觉不一致
3. **响应式遗漏**：只给了桌面稿，移动端适配靠猜
4. **Token 不同步**：设计改了颜色但 Token 没更新 → 开发用旧值
5. **设计走查太晚**：上线前才走查，发现大量偏差已来不及改

### 最佳实践

1. **全状态设计**：每个页面/组件必须包含 5 种状态（默认、加载中、空、错误、成功）
2. **Design Token 系统**：用 Token 管理设计变量，代码直接引用 Token，不硬编码
3. **组件化设计**：设计稿基于组件库，开发基于组件库 → 保持同源
4. **设计走查前置**：开发阶段就分批走查（不等全部完成），小步快速修正
5. **交互动画规范**：动画类型、时长、缓动函数写入 Handoff 文档，不靠口述
6. **开发预审设计**：复杂交互设计定稿前，让前端评估实现成本，避免返工

## 5. UX↔视觉

### 沟通内容

| 阶段 | 内容 | 方向 |
|------|------|------|
| 交互稿交接 | 线框图、交互逻辑、信息层级 | UX→视觉 |
| 视觉方向 | 品牌调性、视觉参考、情绪板 | 双向 |
| 视觉稿评审 | 视觉方案、配色、排版、图标 | 视觉→UX 评审 |
| 一致性审查 | 视觉规范的跨页面一致性 | UX 主导 |

### 沟通时机

- **交互稿交接**：交互评审通过后立即
- **视觉方向**：项目初期确定，后续功能复用
- **视觉评审**：视觉初稿 → 修改 → 终稿，通常 2-3 轮
- **一致性审查**：视觉终稿确定后，跨页面统一审查

### 沟通方式

| 方式 | 适用场景 |
|------|---------|
| 标注交互稿 | 线框图 + 交互说明（hover、click、状态转换） |
| 情绪板（Mood Board） | 视觉方向探索阶段 |
| 视觉规范文档 | 字体、颜色、间距、组件视觉规范 |
| 走查比对 | 交互意图 vs 视觉表达的对齐检查 |

### 常见问题

1. **交互意图丢失**：视觉美化过程中改变了交互设计的信息层级
2. **视觉一致性差**：不同页面的按钮、卡片样式不统一
3. **图标/插画风格不统一**：多人设计导致风格混杂
4. **交互稿不够清晰**：视觉设计师不确定某些区域的交互行为
5. **视觉设计过度**：视觉效果炫但增加了交互复杂度

### 最佳实践

1. **设计系统（Design System）**：统一的视觉组件库 + 使用规范
2. **交互稿标注规范**：交互稿上必须标注状态转换、数据变化、边界说明
3. **视觉 QA Checklist**：检查一致性、可访问性（对比度、字号）、品牌合规
4. **原子设计方法（Atomic Design）**：从 Token → 原子 → 分子 → 组织 → 模板 → 页面，层层构建
5. **UX-视觉同步评审**：交互和视觉在同一个评审会议中讨论，减少信息来回

## 6. 测试↔开发

### 沟通内容

| 阶段 | 内容 | 方向 |
|------|------|------|
| 缺陷报告 | 复现步骤、环境信息、预期/实际结果 | 测试→开发 |
| 缺陷确认 | 确认/拒绝/需更多信息 | 开发→测试 |
| 修复通知 | 修复说明、影响范围、回归建议 | 开发→测试 |
| 验证闭环 | 修复验证结果、是否关闭 | 测试→开发 |
| 测试环境 | 部署状态、数据准备、环境问题 | 开发→测试 |

### 沟通时机

- **缺陷报告**：发现后立即记录（P0/P1 需额外即时通知）
- **缺陷确认**：24 小时内响应（P0 2 小时内）
- **修复通知**：代码合并后主动通知测试可以验证
- **验证闭环**：修复版本部署后 24 小时内验证

### 沟通方式

| 方式 | 适用场景 |
|------|---------|
| 缺陷追踪系统 | Jira/Linear/GitHub Issues — 结构化记录 |
| 即时消息 | P0/P1 缺陷的紧急通知 |
| Bug Bash | 集中式缺陷发现活动 |
| 状态看板 | 缺陷流转状态可视化 |

### 常见问题

1. **缺陷报告质量差**：复现步骤不清、缺少环境信息 → 开发无法复现
2. **"无法复现"拒绝**：开发简单标记"无法复现"而不深入排查
3. **修复不通知**：开发修完了但没告诉测试 → 缺陷长期挂起
4. **回归引入新缺陷**：修复 A 导致 B 出问题，缺少回归测试
5. **测试环境不稳定**：环境问题导致假阳性缺陷 → 浪费沟通成本

### 最佳实践

1. **缺陷报告模板**：强制包含复现步骤、环境、预期/实际结果、截图/录屏
2. **状态机管理**：New → Confirmed → In Progress → Fixed → Verified → Closed（每个状态有明确的负责人和时限）
3. **回归测试清单**：每次修复附带回归建议，测试执行对应回归
4. **每日 Bug Triage**：开发和测试每日对齐缺陷优先级和进展
5. **测试环境自动化**：环境部署和数据准备自动化，减少人工干预导致的环境问题

## 7. 架构↔各角色

### 沟通内容

| 对象 | 内容 | 方向 |
|------|------|------|
| 产品 | 技术约束、技术能力边界、技术债影响 | 架构→产品 |
| 开发 | 架构决策（ADR）、技术规范、模式指导 | 架构→开发 |
| 测试 | 非功能需求（性能、安全）的测试策略 | 架构→测试 |
| UX | 技术限制对交互的影响（动画性能、实时性） | 架构→UX |
| 全员 | 技术债评估和偿还计划 | 架构↔全员 |

### 沟通时机

- **架构决策**：重大技术选型前（不是选完了通知）
- **ADR 发布**：决策确定后 48 小时内发布书面 ADR
- **技术债**：每个 Sprint 分配固定比例（建议 15-20%）
- **技术约束通知**：需求评审阶段就介入，不等开发阶段

### 沟通方式

| 方式 | 适用场景 |
|------|---------|
| ADR（Architecture Decision Record） | 结构化记录：背景、选项、决策、后果 |
| 技术规范文档 | 编码规范、API 规范、模式指导 |
| 架构评审会 | 重大技术方案的评审 |
| 技术 Spike 报告 | 原型验证的结论报告 |
| 技术雷达 | 技术栈评估和演进路线 |

### 常见问题

1. **象牙塔架构**：架构师不了解实际开发痛点，做出不切实际的方案
2. **架构决策不透明**：开发不知道为什么选了 A 不选 B → 不信服，暗中违反
3. **技术债无人管**：一直加功能不还债 → 系统越来越难维护
4. **过度架构**：YAGNI（You Aren't Gonna Need It） — 为不存在的需求提前设计
5. **架构传达断层**：决策做了但没有效传达到每个开发者

### 最佳实践

1. **ADR 标准化**：所有重大架构决策必须有 ADR，包含 Context → Decision → Consequences
2. **嵌入式架构师**：架构师参与日常开发，不是远离代码的指挥者
3. **技术债看板**：可视化技术债，每个 Sprint 分配固定比例处理
4. **架构适应性函数（Fitness Function）**：用自动化测试验证架构约束（如依赖方向、性能基线）
5. **轻量级架构评审**：不是所有改动都需要正式评审，定义评审阈值（如影响 3 个以上模块）

## 8. 代码审查↔开发

### 沟通内容

| 阶段 | 内容 | 方向 |
|------|------|------|
| PR 提交 | 变更说明、关联 Issue、自检清单 | 开发→审查者 |
| 审查反馈 | 代码问题、改进建议、分级（阻断/建议/夸赞） | 审查者→开发 |
| 修改回复 | 修改说明、讨论澄清、不同意的理由 | 开发→审查者 |
| 再审通过 | 确认修复、Approve | 审查者→开发 |

### 沟通时机

- **PR 提交**：功能完成 + 自测通过 + CI 绿灯后提交
- **审查响应**：提交后 4 小时内开始审查（工作时间）
- **修改回复**：收到反馈后 24 小时内回复或修改
- **审查轮次**：目标 ≤2 轮完成审查（超过 3 轮说明问题）

### 沟通方式

| 方式 | 适用场景 |
|------|---------|
| PR 评论 | GitHub/GitLab 行内评论和通用评论 |
| 审查清单 | 结构化的审查维度（正确性/安全/性能/可读性） |
| 反馈分级 | 🔴 Must Fix / 🟡 Suggestion / 🟢 Nit / 💚 Praise |
| 结对审查 | 复杂变更的同步讨论 |

### 常见问题

1. **审查瓶颈**：少数人负责审查 → 成为瓶颈，PR 积压
2. **反馈模糊**："这里不太好"但不说为什么、怎么改
3. **求全责备**：每个 Nit 都当 Must Fix → 开发反感，审查变成对立
4. **橡皮图章**：走过场，LGTM 不看代码 → 失去审查价值
5. **审查范围蔓延**：PR 太大（500+ 行），审查质量急剧下降

### 最佳实践

1. **小 PR 原则**：每个 PR ≤300 行变更，超过就拆分
2. **反馈分级标注**：明确标注阻断/建议/挑剔/夸赞，让开发知道优先级
3. **PR 描述模板**：强制包含变更原因、方案说明、测试说明、自检清单
4. **审查轮换**：定期轮换审查者，避免瓶颈 + 促进知识共享
5. **Conventional Comments**：使用约定格式（`suggestion:`, `issue:`, `question:`, `praise:`）
6. **配对审查**：复杂改动用同步结对审查代替来回评论，效率更高

## 9. 项目管理↔全员

### 沟通内容

| 场景 | 内容 | 频率 |
|------|------|------|
| 站会（Daily Standup） | 昨天做了什么、今天做什么、有什么阻塞 | 每日 |
| 周会（Weekly Sync） | 本周进展、下周计划、风险预警 | 每周 |
| 阻塞升级 | 阻塞描述、已尝试方案、需要的帮助 | 随时 |
| Sprint 回顾 | 什么做得好、什么需改进、行动项 | 每 Sprint |
| 风险看板 | 项目风险识别、评估、应对措施 | 持续更新 |

### 沟通时机

- **站会**：每日固定时间，≤15 分钟
- **阻塞升级**：阻塞 2 小时未解决 → 立刻升级，不等站会
- **风险预警**：发现风险时立刻登记，不等周会
- **Sprint 回顾**：Sprint 结束后 24 小时内

### 沟通方式

| 方式 | 适用场景 |
|------|---------|
| 异步站会 | 在协作工具中文字更新状态（适合远程/AI 团队） |
| 看板 | 可视化任务流转状态 |
| 阻塞升级流程 | 结构化的升级模板（阻塞描述→影响→已尝试→需要帮助） |
| Sprint 回顾模板 | Start-Stop-Continue 或 Sailboat 等回顾方法 |
| 项目仪表盘 | 燃尽图、速率、缺陷趋势等可视化 |

### 常见问题

1. **站会变汇报会**：向 PM 汇报而不是团队互相同步 → 失去价值
2. **阻塞升级不及时**：成员卡住了但不说 → PM 毫不知情
3. **会议过多**：各种对齐会议挤占执行时间
4. **进度不透明**：PM 不知道真实进度，靠问才知道
5. **回顾没行动**：回顾提出改进但没人跟进 → 同样问题反复出现

### 最佳实践

1. **异步优先**：能异步解决的不开会，站会用文字更新代替视频通话
2. **阻塞 SLA**：定义阻塞升级时限（2 小时自助 → 4 小时 PM 介入 → 8 小时管理层介入）
3. **信息辐射器（Information Radiator）**：看板/仪表盘始终可见，不需要问就能知道进度
4. **回顾行动项追踪**：每个回顾行动项有 Owner 和 Due Date，下次回顾先检查上次行动项
5. **会议有准入标准**：不是所有人都需要参加所有会议，按 RACI 矩阵决定参会人

## 10. 常见反模式

### 10.1 甩过墙（Over the Wall）

| 维度 | 说明 |
|------|------|
| **表现** | 产出物做完直接扔给下游，不管对方能不能接住 |
| **典型场景** | 产品写完 PRD 直接给开发，不做评审；设计出完稿直接给前端，没有 Handoff |
| **根因** | 角色间缺乏 Overlap，只关注自己环节的输出 |
| **危害** | 信息损耗大、理解偏差大、返工率高 |
| **破解** | 交接时双方 Walkthrough（走读）；DoR / DoD 门槛；交接 Checklist |

### 10.2 信息孤岛（Information Silo）

| 维度 | 说明 |
|------|------|
| **表现** | 关键信息只存在于个人/小组中，不共享 |
| **典型场景** | 开发知道技术约束但没告诉产品；测试发现的问题模式没反馈给 UX |
| **根因** | 沟通渠道不畅、没有信息共享机制 |
| **危害** | 决策依据不全、重复踩坑、团队割裂 |
| **破解** | 单一事实源（SSOT）；跨角色评审会议；共享知识库；工作透明化 |

### 10.3 过度会议（Meeting Overload）

| 维度 | 说明 |
|------|------|
| **表现** | 太多同步会议，执行时间被挤占 |
| **典型场景** | 需求会 + 设计评审 + 技术评审 + 站会 + 周会 + 回顾 → 一天 4 小时在开会 |
| **根因** | 异步协作能力弱、信任度低（不开会不放心）、会议无准入标准 |
| **危害** | 执行时间碎片化、参会者疲劳、会议质量下降 |
| **破解** | 异步优先原则；会议准入标准（RACI）；会议时长限制；No-Meeting Day |

### 10.4 验收不对齐（Acceptance Misalignment）

| 维度 | 说明 |
|------|------|
| **表现** | 交付物"完成"了但不符合期望 |
| **典型场景** | 开发说"做完了"但产品说"不是这样的"；测试覆盖了 AC 但用户体验很差 |
| **根因** | AC 模糊、DoD 不统一、缺乏过程中对齐 |
| **危害** | 返工、拉锯、信任下降 |
| **破解** | Definition of Done 团队统一；AC 三方对齐（Three Amigos）；过程中 Demo 而非交付后验收 |

### 10.5 其他高频反模式

| 反模式 | 表现 | 破解 |
|--------|------|------|
| **HiPPO（最高薪者意见优先）** | 老板/资深者的意见自动胜出，不论证据 | 数据驱动决策、ADR 记录决策依据 |
| **Gold Plating（镀金）** | 加入用户没要求的功能/精度 | 严格按 AC 交付，超出范围走变更流程 |
| **Bike Shedding（帕金森琐碎定律）** | 简单问题讨论很久，重要问题快速通过 | 时间盒讨论，重要议题先行 |
| **Hero Culture（英雄文化）** | 依赖个人能力而非流程 | 知识共享、流程标准化、Bus Factor > 1 |
| **Silent Disagreement（沉默反对）** | 不同意但不说，执行时打折扣 | 安全发言环境、明确反对是义务不是挑衅 |

## 11. AI Agent 团队适配建议

> 以下基于上述调研，针对 NDHY AI Agent Team（1 人 + 26 AI 专家）的特殊性提出适配思考。

### 11.1 AI 团队 vs 人类团队的关键差异

| 维度 | 人类团队 | AI Agent 团队 |
|------|---------|---------------|
| 沟通渠道 | 面对面、视频、IM 多种混合 | 全部通过文本指令和文档 |
| 上下文连续性 | 人脑持续记忆（有遗忘） | 每次会话从零开始（靠文件记忆） |
| 并行度 | 受限于人数和注意力 | 可高度并行，但需依赖管理 |
| 模糊理解 | 人可以"意会"模糊指令 | AI 需要精确指令，模糊→偏差 |
| 情绪 | 需要考虑沟通方式和情绪 | 不需要，但需要考虑上下文限制 |
| 质量方差 | 人类有稳定性但有状态波动 | AI 质量受指令质量和上下文质量决定 |

### 11.2 适配原则

1. **文档即沟通**：所有"会议"场景转化为结构化文档交接。评审 = 文档走读 + 结构化评论。
2. **上下文即信息对称**：每次任务 Spawn 时必须提供完整上下文（SSOT），不能假设 Agent "还记得"。
3. **契约即验收标准**：模糊的 AC 在人类团队可能靠沟通补全，在 AI 团队会直接导致偏差。AC 必须精确到可自动验证。
4. **流转即交接协议**：每对角色间的交接需要定义明确的交接物格式和完整性检查清单。
5. **异步是天然优势**：AI Agent 没有"开会疲劳"，但过度串行会浪费并行优势。

### 11.3 对流程设计的建议

| 人类团队实践 | AI 团队适配 |
|-------------|-------------|
| Three Amigos 会议 | 三角色顺序评审 + 结构化反馈汇总 |
| 站会 | 项目管理专家定期读取各任务状态 |
| 设计评审会 | 设计稿 + 评审 Prompt → 结构化评审意见 |
| Bug Triage 会议 | 自动化缺陷分级规则 + 人工裁决争议 |
| 结对审查 | 独立审查 + 对话式讨论（审查者↔开发者 Agent 交互） |
| Sprint 回顾 | Leader 记忆系统中的模式识别和规则晋升 |
