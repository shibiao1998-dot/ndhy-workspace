> 流程设计专家产出 | 基于标准开发流裁剪 | 60分版本优先

## 一、项目现状评估

### 已完成

| 产出物 | 状态 | 位置 |
|--------|------|------|
| 维度内容（12类89维度，434KB） | ✅ 已完成 | `dimensions/` |

### 原型系统已有能力
- 维度加载器（自动解析 Markdown）
- 维度路由器（关键词匹配 → 任务类型 → 维度选择）
- 组装引擎（三级优先级排序 + 长度控制）
- 引擎适配器（Claude/Midjourney/Suno 三套格式）
- CLI 入口（`python main.py --task "..." --engine xxx`）

### 缺什么（从 CLI 到可用系统的差距）

| 缺口 | 影响 |
|------|------|
| 无 Web 界面 | DJ 团队不可能用命令行 |
| 路由靠关键词硬匹配 | 准确度有限，复杂场景失效 |
| 无优先级标注机制 | DJ 子项目2 未覆盖 |
| 无业务线扩展框架 | DJ 子项目4 未覆盖 |
| 无用户引导和交互流程 | 用户不知道怎么用 |
| 无数据持久化 | 生成的提示词无法保存/管理 |

## 二、60分版本目标定义

> DJ 原话："步步为营，先做60分"

**60分 = 以下5件事全部可用**：

1. ✅ **Web 界面**：用户打开浏览器就能用（不需要命令行）
2. ✅ **场景输入 → 维度路由 → 提示词输出**：核心链路跑通
3. ✅ **优先级排序**：维度分级（一级/二级/三级），用户可调整
4. ✅ **引擎选择**：用户选择目标引擎，输出自动适配
5. ✅ **结果可复制**：生成的提示词可一键复制使用

**60分不做**（留给后续迭代）：
- ❌ 用户账户体系
- ❌ 历史记录持久化到数据库
- ❌ AI 驱动的智能路由（先用增强版关键词匹配）
- ❌ 多业务线切换（先做好 AI 设计师/策划这一条线）
- ❌ 复杂的维度编辑界面

## 五、协作拓扑

### 依赖图

```
                    ┌──── M2 体验设计 ────────────────┐
                    │                                  │
M1 产品定义 ──→ ────┤                                  ├──→ M7 前端开发 ──┐
                    │                                  │                  │
                    └──── M3 技术架构 ──→ ┌─ M4 数据库设计 ─┐              │
                                         │               │              │
                                         └─ M5 API 设计 ──┘──→ M6 后端开发 ──→ S1 联调 ──→ S2 测试 ──→ D3 部署
                                                          │                                    
                                                          └──→ M7 前端开发 ──┘
```

### 并行策略

| 并行组 | 节点 | 协调要点 |
|--------|------|----------|
| Phase 2 | 体验设计 ∥ 技术架构 | 共享产品定义输入；体验设计不依赖技术细节 |
| Phase 3 | 数据库设计 ∥ API 设计 | 共享技术架构约束；字段语义需对齐 |
| Phase 4 | 后端开发 ∥ 前端开发 | 基于同一份 API 契约；前端可先用 Mock |

### 提前并行机会

- **前端开发可在 API 设计完成后立即启动**（不必等数据库设计和后端完成），使用 Mock 数据
- **体验设计与技术架构可完全并行**（60分版本体验设计不依赖技术选型细节）

### 回溯规则

| 触发条件 | 反馈给 | 从哪里重做 |
|----------|--------|-----------|
| 技术架构发现产品定义不可行 | 产品定义专家 | Phase 1 |
| API 设计发现架构约束不合理 | 技术架构专家 | Phase 2（技术架构） |
| 前端发现体验方案无法实现 | 体验设计专家 | Phase 2（体验设计） |
| 联调发现契约与实现不一致 | 后端/前端开发专家 | Phase 4 |
| 测试发现功能缺陷 | 对应开发专家 | Phase 4 |

## 六、执行流程

### Phase 1：产品定义（串行）

```
Step 1.1 → ⬡M1 产品定义锁定
```

**Step 1.1：产品定义**
- 执行角色：📐 产品定义专家
- 输入：
  - DJ 4个子项目定义（PROJECT.md）
  - 验收标准（5条）
  - 现有 CLI 系统能力（`system/README.md`）
  - 维度内容结构（`dimensions/` 目录）
  - DJ 会议分析（`meeting-analysis.md`）
- 输出：产品定义文档（`product-definition.md`）
  - 产品概念（一句话说清楚）
  - 核心场景拆解（3-5个）
  - MVP 功能清单（60分版本）
  - 产品规则与约束
  - 成功标准
- 完成标准：
  - [x] 覆盖 DJ 全部4个子项目
  - [x] MVP 功能清单明确且可实现
  - [x] 60分边界清晰（做什么/不做什么）
- 回溯规则：若 DJ 需求理解有偏差 → 重新分析会议原文

---

### Phase 2：方案设计（并行）

```
Step 2.1（体验设计）∥ Step 2.2（技术架构）→ ⬡M2 方案定稿
```

**Step 2.1：体验设计**
- 执行角色：🎨 体验设计专家
- 输入：
  - M1 产品定义文档
  - 现有 CLI 系统交互模式
- 输出：体验设计方案（`experience-design.md`）
  - 信息架构（页面结构）
  - 核心任务流程（用户操作步骤）
  - 页面布局方案（每个页面的区块和内容）
  - 交互模式（输入、选择、输出的交互方式）
  - 状态定义（loading/empty/error/success）
- 完成标准：
  - [x] 覆盖产品定义的所有核心场景
  - [x] 用户从"打开页面"到"拿到提示词"的完整路径清晰
  - [x] 每个页面的信息层级和操作流程明确
- 并行说明：与 Step 2.2 并行
- 回溯规则：若与技术架构冲突 → 双方协商调整

**Step 2.2：技术架构**
- 执行角色：🏛️ 技术架构专家
- 输入：
  - M1 产品定义文档
  - 现有 CLI 系统代码（`system/` 目录）
  - 技术约束：Python 现有代码可复用、零/低外部依赖偏好
- 输出：技术架构方案（`technical-architecture.md`）
  - 系统分层（前端/后端/数据层）
  - 技术选型决策（Web 框架、前端方案）
  - 现有代码复用策略
  - 模块边界定义
  - 部署架构（轻量级，适合内部使用）
- 完成标准：
  - [x] 技术选型有理有据且贴合60分原则
  - [x] 现有 Python 代码的复用路径清晰
  - [x] 部署方案简单可行
- 并行说明：与 Step 2.1 并行
- 回溯规则：若技术选型无法支撑产品定义 → 反馈产品定义专家调整

---

### Phase 3：详细设计（并行）

```
Step 3.1（数据库设计）∥ Step 3.2（API 设计）→ ⬡M3 契约冻结
```

**Step 3.1：数据库设计**
- 执行角色：🗄️ 数据库设计专家
- 输入：
  - M3 技术架构方案
  - M1 产品定义文档
- 输出：数据模型（`database-design.md`）
  - 实体定义（60分版本预计很轻量）
  - 表结构 + 字段字典
  - 索引策略
- 完成标准：
  - [x] 覆盖产品定义中需要持久化的数据
  - [x] 字段命名清晰一致
  - [x] 与 API 设计字段语义对齐
- 并行说明：与 Step 3.2 并行；字段语义需与 API 设计对齐
- 回溯规则：若架构约束不支持 → 反馈技术架构专家

**Step 3.2：API 设计**
- 执行角色：📜 API 设计专家
- 输入：
  - M3 技术架构方案
  - M1 产品定义文档
  - M2 体验设计方案（了解前端需要哪些数据）
- 输出：接口契约（`api-design.md`）
  - 接口清单（RESTful）
  - 请求/响应 Schema
  - 错误模型
  - 核心接口 Mock 示例
- 完成标准：
  - [x] 覆盖体验设计方案中所有前端需要的数据交互
  - [x] 请求/响应 Schema 完整可用
  - [x] 前端开发可基于此独立开发
- 并行说明：与 Step 3.1 并行
- 回溯规则：若架构约束不支持 → 反馈技术架构专家

```
Step 4.1（后端开发）∥ Step 4.2（前端开发）→ ⬡M4 开发完成
```

- 完成标准：
  - [x] 所有 API 接口可调用且返回正确
  - [x] 现有维度文件能正常加载和路由
  - [x] 单元测试通过
  - [x] 代码可独立运行
- 并行说明：与 Step 4.2 并行，基于同一份 API 契约
- 回溯规则：若契约不可实现 → 反馈 API 设计专家

**Step 4.2：前端开发**

  - 页面组件
  - 状态管理

  - 全态处理（loading/error/empty/success）
- 完成标准：
  - [x] 覆盖体验设计方案的所有页面和交互
  - [x] 使用 Mock 数据可独立运行
  - [x] 核心交互流程（输入场景 → 选择引擎 → 获取提示词 → 复制）完整
- 并行说明：与 Step 4.1 并行，先用 Mock 开发，后端就绪后联调
- 回溯规则：若体验方案无法实现 → 反馈体验设计专家

| 后端+前端 → 联调 | 可运行代码 | 代码 + 启动说明 | 各自单元测试通过 |
| 联调 → 测试 | 联调通过的系统 | 可运行系统 + 联调报告 | 核心场景端到端跑通 |
| 测试 → 部署 | 测试通过的代码 | 代码 + 测试报告 | 无 P0/P1 缺陷 |

## 十、关键决策点

### 技术选型建议（供技术架构专家参考）

基于60分原则和现有资源，建议技术架构专家重点评估：

1. **后端**：现有 Python 代码 → FastAPI / Flask 封装成 API（复用最大化）
2. **前端**：轻量方案（React/Vue SPA 或 Streamlit/Gradio 极简方案）
3. **数据库**：60分版本可能 SQLite 就够（轻量、零配置）
4. **部署**：Docker 单容器 或 直接 Python 进程

> ⚠️ 以上仅为建议，技术选型最终由技术架构专家决策。

### 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 现有维度内容质量参差 | 生成的提示词质量不均 | 先跑通链路，后续迭代优化内容 |
| 关键词路由不够准确 | 维度选择偏差 | 60分版本允许用户手动调整维度选择 |
| 前后端联调耗时超预期 | 交付延迟 | 严格契约冻结，前端先用 Mock 开发 |
| 60分边界蔓延 | 范围膨胀 | 产品定义明确写"不做清单"，Phase 2 后锁定 |
