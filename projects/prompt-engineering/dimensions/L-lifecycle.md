## L01 项目备选池标准

**维度定义**：项目进入备选池的条件、备选池项目的信息要求（初衷、价值、可行性前置分析）。

**质量作用**：备选池准入——明确什么项目可以进池，避免"拍脑袋立项"。

### 具体信息内容

#### 备选池准入条件

一个项目/需求要进入备选池，必须满足以下**全部前置条件**：

| 编号 | 准入条件 | 说明 | 检查方法 |
|------|---------|------|---------|
| L01-1 | **有明确的发起人** | 需求来源清晰（DJ 战略指令 / 业务线负责人 / 用户反馈汇总 / 数据洞察），非匿名需求 | 检查需求来源字段是否填写 |
| L01-2 | **有初步的价值假设** | 用 1-2 句话说清"做这个能带来什么价值"（用户价值 / 商业价值 / 战略价值至少占一项） | 检查价值假设字段是否有实质内容（非"提升体验"等笼统表述） |
| L01-3 | **不触碰战略红线** | 对照 A02 战略红线清单检查（不做大模型研发 / 不做脱离场景的炫技 / 不违背以学习者为中心） | A02 红线逐条对照 |
| L01-4 | **有初步的可行性判断** | 至少给出"技术上大概能不能做""资源大概够不够"的初步判断（不需要详细分析） | 检查可行性初判字段 |
| L01-5 | **与至少一个战略重点相关** | 对照 A01 战略核心方向，需求至少与一个方向有关联 | A01 对齐检查 |

#### 备选池项目信息卡模板

每个进入备选池的项目，必须填写以下信息卡：

```markdown
### 备选项目信息卡

**基本信息**
- 项目名称：[简洁明确的名称]
- 发起人：[谁提出的]
- 发起日期：[YYYY-MM-DD]
- 需求来源类型：[DJ 战略指令 / 业务线需求 / 用户反馈 / 数据洞察 / 竞品参考 / 其他]

**初衷与价值**
- 项目初衷：[为什么要做这个？解决什么问题？]
- 目标用户：[粗略描述，不需要详细画像]
- 价值假设：[做了能带来什么价值？用户价值 / 商业价值 / 战略价值]
- 战略对齐：[对齐 A01 中的哪个战略重点？]

**可行性初判**
- 技术可行性：[高 / 中 / 低 / 未知] + [一句话说明]
- 资源需求估算：[大 / 中 / 小] + [一句话说明]
- 时间预期：[紧急(1月内) / 正常(1-3月) / 长期(3月+)]

**优先级初判**
- 紧迫性：[高 / 中 / 低] + [为什么？]
- 影响范围：[大(全平台) / 中(单产品线) / 小(单功能)]
```

#### 华渔典型的备选池项目类型

结合华渔的产品线，备选池项目通常属于以下类型：

| 项目类型 | 说明 | 典型案例 | 评估侧重 |
|---------|------|---------|---------|
| **产品功能迭代** | 现有产品（101 PPT、未来实验室等）的新功能或功能优化 | 101 PPT 新增某学科工具 | 用户价值 + 技术可行性 |
| **新产品孵化** | 全新产品或产品线 | EDA 平台 | 战略价值 + 市场验证 |
| **平台升级** | 底层平台或架构升级 | AI 生产中心升级 | 技术价值 + 长期 ROI |
| **ToG 项目** | 政府/国家级平台项目 | 国家智慧教育平台升级 | 战略价值 + 合规要求 |
| **国际化项目** | 面向海外市场的产品/内容适配 | Mynd.ai ActivSuite | 商业价值 + 本地化 |
| **内部工具** | 提升内部效率的工具（如提示词工程系统） | AI 设计师提示词工程 | 效率提升 + 复用性 |

**数据来源**：项目管理流程文档。

**更新机制**：流程优化时更新准入条件。每季度复盘备选池质量。

## L02 前置分析标准

**维度定义**：前置分析的内容要求（原始需求、项目初衷、项目定义、项目价值分析、可行性分析），每项包含分析深度、产出物格式、质量标准。

**质量作用**：前置分析完整性——确保立项前的分析全面深入，不带着模糊需求进入执行。

### 具体信息内容

#### 前置分析五要素

| 编号 | 分析内容 | 分析深度要求 | 产出物格式 | 质量标准 |
|------|---------|------------|-----------|---------|
| L02-1 | **原始需求** | 完整记录需求来源、背景、触发事件。引用需求发起人的原话。 | 结构化文档（背景→问题→期望→约束） | 每个核心判断可追溯到证据来源 |
| L02-2 | **项目初衷** | 回答"为什么做这个？"——不是"做什么"，是"为什么"。追溯到用户痛点或战略需要。 | 初衷声明（1 段话）+ 背景分析（1 页） | 初衷必须关联到"用户痛点"或"战略方向"，不能是"因为竞品有" |
| L02-3 | **项目定义** | 明确项目的范围边界（做什么 / 不做什么）、项目类型（实验/工具/平台）、成功标准 | 项目定义文档（范围→边界→类型→成功标准） | 范围边界清晰，"不做什么"与"做什么"同样明确 |
| L02-4 | **项目价值分析** | 分别分析用户价值、商业价值、战略价值，每种价值给出证据和量化预期 | 三维价值分析表 | 至少一种价值有数据支撑或可验证假设 |
| L02-5 | **可行性分析** | 技术可行性 + 资源匹配度 + 时间线合理性 + 风险识别 | 可行性报告（引用 F01-F05 维度） | 技术可行性须有技术团队确认；资源匹配须对照 F01 资源现状 |

#### 前置分析的三级深度

根据项目规模选择不同深度：

| 深度级别 | 适用场景 | 工作量 | 各要素要求 |
|---------|---------|--------|-----------|
| **轻量级** | 小功能迭代、bug 修复级改动 | 1-2 小时 | L02-1 填写需求卡 + L02-3 简述范围 + L02-5 口头确认可行性 |
| **标准级** | 常规产品功能、季度迭代 | 1-3 天 | 五要素全部完成，每项 1-2 页 |
| **深度级** | 新产品/新平台、重大战略项目 | 1-2 周 | 五要素全部完成，含用户调研、技术 PoC、竞品深度分析 |

#### 华渔场景的特别注意事项

- **101 PPT 功能迭代**：通常用标准级。必须引用 101 PPT 的三类用户分层（依赖资源型/修改整合型/自主设计型），明确新功能主要服务哪一层
- **未来实验室新实验**：通常用标准级。必须确认学科知识准确性（引用 J04 学科知识库）和实验安全要求（引用 J05 实验教学方法库）
- **EDA 平台功能**：通常用深度级。需要考虑多语言、多文化适配和全球教育标准差异
- **国家级平台项目**：必须用深度级。需要额外的合规性分析（引用 F04 法规与合规要求清单）
- **提示词工程类项目**：DJ 强调"步步为营"——用标准级启动，快速迭代中持续补充

**数据来源**：前置分析事务定义、项目管理流程。

**更新机制**：流程优化时更新。每季度复盘前置分析质量。

## L03 立项评估标准

**维度定义**：立项评估的判断标准（必要性、可行性、战略对齐度、资源匹配度），每项包含评估方法、通过标准、案例。

**质量作用**：立项决策依据——科学评估是否立项，避免"拍脑袋立项"或"因为能做就做"。

### 具体信息内容

#### 立项评估四维矩阵

| 评估维度 | 权重 | 评估方法 | 通过标准（最低分） | 否决线 |
|---------|------|---------|-----------------|--------|
| **必要性**（为什么必须做？） | 30% | 回答三个问题：①用户痛点有多痛？②不做会怎样？③有没有替代方案？ | 60/100 | 如果"不做也没关系"，直接否决 |
| **可行性**（能不能做出来？） | 25% | 技术可行性（技术团队评审）+ 资源匹配度（对照 F01）+ 时间线合理性 | 60/100 | 技术上明确不可行，直接否决 |
| **战略对齐度**（方向对不对？） | 25% | 对照 A01 战略核心方向 + A02 红线检查 | 70/100 | 触碰红线，直接否决 |
| **ROI 预估**（值不值得做？） | 20% | 预估投入（人力×时间×成本）vs 预估产出（用户价值 + 商业价值 + 战略价值） | 50/100 | ROI < 0.5 需要额外论证 |

#### 立项决策树

```
项目进入立项评估
    │
    ├─ A02 红线检查 → 触碰红线 → ❌ 直接否决
    │
    ├─ 必要性评估 → "不做也没关系" → ❌ 否决（或降级为"观察"）
    │
    ├─ 技术可行性 → 明确不可行 → ❌ 否决（或转为"技术预研"）
    │
    ├─ 四维评分计算
    │   ├─ 总分 ≥ 75 → ✅ 立项
    │   ├─ 60 ≤ 总分 < 75 → 🟡 有条件立项（补充分析 / 缩小范围 / 降低规格）
    │   └─ 总分 < 60 → ❌ 不立项（回到备选池继续观察）
    │
    └─ DJ 战略指令类项目 → 跳过必要性评估 → 聚焦可行性和 ROI 评估
```

#### 不同项目类型的评估侧重

| 项目类型 | 必要性侧重 | 可行性侧重 | 战略对齐侧重 | ROI 侧重 |
|---------|-----------|-----------|-------------|---------|
| 产品功能迭代 | 用户痛点证据 | 技术实现复杂度 | 与产品路线图一致 | 用户价值/开发成本 |
| 新产品孵化 | 市场空白验证 | 团队能力 + 技术 PoC | 与公司战略方向一致 | 长期市场空间 |
| ToG 项目 | 政策趋势 + 中标概率 | 交付能力 + 合规性 | 国家级平台战略 | 合同金额/交付成本 |
| 国际化项目 | 目标市场需求 | 本地化能力 | Mynd.ai SaaS 转型 | 海外收入增长 |
| 内部工具 | 效率提升幅度 | 开发成本 | AI+战略 | 人效提升/投入成本 |

#### 立项评估案例

**案例 1：提示词工程系统（本项目）**

| 维度 | 评估 | 得分 |
|------|------|------|
| 必要性 | DJ 亲自发起，明确指出"AI 产出差是因为信息不对称"。不做 = AI 设计师/策划能力无法提升。无替代方案。 | 95/100 |
| 可行性 | 信息收集和结构化是已知技术，不涉及新技术攻关。需要的是企业私域知识整理，主要消耗是人力。 | 80/100 |
| 战略对齐 | 直接对齐"AI+"战略。提示词工程是 AI 应用层的基础设施。 | 95/100 |
| ROI | 如果 AI 设计师/策划能产出可用方案，可显著降低人力成本（参考 AI 生产中心已节约近 1 亿元） | 85/100 |
| **总分** | **加权 = 95×0.3 + 80×0.25 + 95×0.25 + 85×0.2** | **89.25 → ✅ 立项** |

**数据来源**：立项评估流程文档。

**更新机制**：流程优化时更新评估标准。每半年复盘立项决策质量（回顾哪些立项是对的/错的）。

## L04 核心价值分析要求

**维度定义**：核心价值分析在项目生命周期中的位置、与其他环节的关系。标准已在 A 类和 I01 详细定义。

**质量作用**：核心价值定位——明确核心价值在全生命周期中的作用，确保核心价值贯穿始终。

### 具体信息内容

#### 核心价值在生命周期中的位置

```
备选池 → 前置分析 → 立项 → 【核心价值分析】→ 目标用户分析 → 理想化状态 → 目标拆解 → 执行
                                    ↑
                          最关键的转折点
                    从"为什么做"转向"做成什么样"
```

#### 核心价值分析的输入输出

| 输入 | 输出 | 关联维度 |
|------|------|---------|
| 前置分析结果（L02） | 核心价值声明 | E14 核心价值推导案方法 |
| 战略方向（A01） | 衍生价值清单 | E15 核心价值编写标准 |
| 用户痛点证据（B04） | 价值验证方案 | E13 核心价值验证方法库 |
| 竞品差异化分析（C04） | 差异化价值定位 | I01 核心价值质量评分卡 |

#### 核心价值分析的质量标准

1. **结构化**：核心价值必须按 E15 编写标准结构化表述（主谓宾清晰、定量可衡量）
2. **有证据**：核心价值必须有推导过程（E14），推导过程中的每个判断有证据支撑
3. **可验证**：核心价值必须配有验证方案（E13），说明如何证明"价值确实存在"
4. **可评分**：核心价值必须通过 I01 评分卡评分，总分 ≥ 70 方可通过

#### 华渔场景的核心价值分析要点

- **教育产品**：核心价值必须回答"学习者/教师的什么行为或能力发生了什么变化"
  - ✅ "让高中化学教师能在 5 分钟内准备一个 3D 虚拟实验课件"
  - ❌ "提升教学体验"
- **工具产品**：核心价值必须回答"用户在什么场景下能比以前更快/更好地完成什么任务"
  - ✅ "依赖资源型教师无需设计能力即可完成符合课标的课件"
  - ❌ "提高备课效率"
- **平台产品**：核心价值必须回答"平台连接了什么供给和什么需求，创造了什么网络效应"

**数据来源**：A 类维度（战略价值观）、I01（评分卡）、E13-E15（方法论）。

**更新机制**：已在其他维度覆盖，本维度负责维护"位置关系"和"华渔场景要点"。每季度复盘。

## L05 目标用户分析要求

**维度定义**：目标用户分析在项目生命周期中的位置、与其他环节的关系。标准已在 B 类和 I02 详细定义。

**质量作用**：目标用户定位——明确目标用户分析在全生命周期中的作用。

### 具体信息内容

#### 目标用户分析在生命周期中的位置

```
备选池 → 前置分析 → 立项 → 核心价值分析 → 【目标用户分析】→ 理想化状态 → 目标拆解 → 执行
                                                  ↑
                                       从价值到人
                                 "这个价值具体服务于谁？"
```

#### 目标用户分析的输入输出

| 输入 | 输出 | 关联维度 |
|------|------|---------|
| 核心价值声明（L04） | 用户细分矩阵 | B03 用户细分维度库 |
| 用户原声（B01） | 用户画像（每类用户） | E04 目标用户分析方法 |
| 用户行为数据（B02） | 用户场景地图 | B06 用户使用场景库 |
| 用户痛点证据链（B04） | 各类用户的痛点优先级 | I02 目标用户质量评分卡 |

#### 目标用户分析的质量标准

1. **分层而非笼统**：不能只说"教师"，必须区分不同类型的教师（参考 101 PPT 三分类）
2. **有证据的画像**：用户画像基于 B01 用户原声和 B02 行为数据，不能凭空想象
3. **痛点有优先级**：不是列出所有痛点，而是按影响范围和紧迫性排序
4. **场景具体完整**：每个关键场景包含 Who/When/Where/Why/What/How/干扰因素/期望结果（E06 标准）
5. **可评分**：通过 I02 评分卡评分，总分 ≥ 70 方可通过

#### 华渔产品的典型用户分层维度

| 产品线 | 核心用户 | 推荐分层维度 | 参考案例 |
|--------|---------|------------|---------|
| 101 PPT | 教师 | 信息化能力（依赖资源/修改整合/自主设计） | 101 PPT 白皮书 |
| 未来实验室 | 教师 + 学生 | 学段（初中/高中）× 学科 × 实验类型 | — |
| EDA 平台 | 全球教育者 | 国家/语言 × 教育阶段 × 技术接受度 | — |
| 网龙多学 | 企业培训师/学员 | 企业规模 × 行业 × 学习目标 | — |
| 智慧教室 | 教师 + 教育管理者 | 角色（教师/管理者）× 学校类型 | — |

**数据来源**：B 类维度（用户真实性）、I02（评分卡）、E04（方法论）。

**更新机制**：已在其他维度覆盖，本维度负责维护"位置关系"和"华渔用户分层参考"。每季度复盘。

## L06 理想化状态定义与案例

**维度定义**：理想化状态的定义（项目最终想达到的完美状态，不考虑资源约束），包含定义方法、描述标准、历史案例、常见误区。

**质量作用**：理想化状态清晰——明确长期愿景，为目标拆解提供"北极星"。

### 具体信息内容

#### 什么是理想化状态

**定义**：项目在"不考虑任何资源、时间、技术约束"的前提下，最终想要达到的完美终态。它是方向灯塔，不是执行目标。

**与其他目标层级的关系**：

```
理想化状态（永远在前方的灯塔）
    ↓ 缩放到 3-12 个月可达
中短期目标（可衡量、可实现）
    ↓ 缩放到 1-3 个月可交付
阶段结项目标（可验收、可交付）
    ↓ 缩放到月度
月度计划（可执行的任务集）
    ↓ 缩放到周
周版本（可发布/可展示的增量）
```

#### 理想化状态描述标准

好的理想化状态描述应该满足：

| 标准 | 说明 | ✅ 好的例子 | ❌ 差的例子 |
|------|------|-----------|-----------|
| **画面感** | 读者能在脑中想象出那个场景 | "一位高中化学教师打开 101 PPT，说出'酸碱滴定实验'，系统自动生成一个完整的 3D 交互实验课件，学生戴上 VR 头盔就能动手做实验" | "教师能高效备课" |
| **用户视角** | 从用户的行为和体验出发，而非从系统功能出发 | "学生在虚拟实验室中亲手操作滴定管，看到指示剂颜色逐渐变化，AI 助教在旁边实时指导" | "系统支持 3D 实验仿真和 AI 辅助" |
| **超越当前约束** | 不被现有技术/资源限制，敢于想象 | "任何老师，任何地方，打开手机就能给学生上一堂身临其境的实验课" | "在学校电脑上使用 3D 实验" |
| **可分解性** | 虽然是理想状态，但可以被拆解为多个可逐步实现的部分 | 上述例子中："语音生成课件""3D 交互实验""VR 支持""AI 助教"都是可独立拆解的 | "让教育变得更好" |

#### 理想化状态定义方法

**三步法**：

1. **想象完美用户体验**：如果没有任何限制，用户的体验会是什么样？
   - 技巧：用"一天日记法"——描述目标用户使用产品的完美一天
2. **对齐 DJ 教育理念**：这个理想化状态是否体现 3E（Effective/Efficient/Engaging）？
   - Effective：学习效果是否真正提升？
   - Efficient：教学效率是否真正提高？
   - Engaging：学习体验是否真正有趣？
3. **检查独特性**：这个理想化状态是否利用了华渔的独特资源（3D/VR 技术、游戏 DNA、全球装机量）？

#### 华渔各产品线的理想化状态参考

**101 教育 PPT 理想化状态**

> 任何学科的教师，无论信息化能力高低，打开 101 PPT 就能在 10 分钟内完成一堂高质量课件——系统根据课标自动推荐最佳资源组合，3D/VR 素材一键插入，AI 自动适配学生认知水平。备课不再是负担，而是创造性的快乐。

**未来实验室理想化状态**

> 每个学生都有一个私人虚拟实验室——可以做任何实验，没有安全风险，没有耗材成本，AI 实验助教随时在旁指导。实验数据自动记录和分析，学生的实验报告由 AI 辅助生成，教师一眼看到每个学生的操作问题和理解盲区。

**AI 设计师/策划提示词工程理想化状态（本项目）**

> AI 设计师/策划拿到一个新需求后，自动加载所有相关维度信息（用户洞察、竞品情报、历史案例、设计标准、专业知识），产出的设计方案质量接近资深设计师水平。人的角色从"亲自做设计"变为"审核和补充 AI 遗漏的维度"。提示词工程系统成为华渔设计能力的知识基础设施。

**EDA 教育元宇宙平台理想化状态**

> 全球任何国家的教师和学生，进入 EDA 就能获得符合本地教育标准的优质学习资源。教师可以跨国协作设计课程，学生可以在沉浸式环境中学习任何学科。教育公平不再是口号——一个非洲乡村的学生和北京的学生拥有同等质量的学习体验。

#### 常见误区

| 误区 | 表现 | 纠正 |
|------|------|------|
| **与现实目标混淆** | 理想化状态写成"Q2 完成 XX 功能" | 理想化状态不受时间约束，是终极方向 |
| **过于抽象** | "让教育变得更好""提升用户体验" | 需要有画面感，能想象出具体场景 |
| **脱离用户** | 从技术角度描述，不提用户 | 必须从用户视角描述体验 |
| **没有独特性** | 任何教育公司都能写出一样的理想化状态 | 必须体现华渔的独特资源和理念 |

**数据来源**：项目规划经验、DJ 教育理念（调研报告 §3.2、§4.2）。

**更新机制**：新项目启动时创建理想化状态。每半年复盘，检查理想化状态是否需要更新。

#### 各层级拆解原则与粒度标准

##### 原则 1：理想化状态 → 中短期目标（价值优先拆解）

**拆解方法**：
1. 将理想化状态分解为 3-5 个"价值切面"（独立的价值维度）
2. 用 **ICE 评分**（Impact 影响力 × Confidence 置信度 × Ease 容易度）排序
3. 选择 ICE 最高的 1-2 个切面作为中短期目标

**粒度标准**：
- 时间跨度：3-12 个月
- 数量：每个理想化状态拆出 3-5 个中短期目标
- 每个目标必须是 SMART 的（Specific / Measurable / Achievable / Relevant / Time-bound）

**华渔案例**：

理想化状态："AI 设计师/策划拿到新需求后，自动加载相关维度信息，产出接近资深设计师水平的方案"

| 价值切面 | 中短期目标 | ICE 评分 | 时间 |
|---------|-----------|---------|------|
| 信息收集 | 完成 12 类 89+ 个维度的信息填充，覆盖率 ≥ 80% | I:9 C:8 E:7 = 504 | 3 个月 |
| 信息排序 | 建立按需加载机制，不同任务自动推荐维度组合 | I:8 C:7 E:6 = 336 | 6 个月 |
| AI 引擎适配 | 针对 3+ 主流大模型完成提示词格式适配 | I:7 C:6 E:7 = 294 | 6 个月 |
| 质量闭环 | 建立"AI 产出→人审核→反馈→维度更新"的迭代闭环 | I:8 C:5 E:5 = 200 | 9 个月 |

→ DJ "步步为营"原则：先做"信息收集"（ICE 最高），快速迭代。

##### 原则 2：中短期目标 → 阶段结项目标（可交付分解）

**拆解方法**：
1. 将中短期目标分解为 2-4 个可独立交付的里程碑
2. 每个里程碑必须有"可展示的交付物"和"明确的验收标准"
3. 里程碑之间有依赖关系时，标注依赖链

**粒度标准**：
- 时间跨度：1-3 个月
- 数量：每个中短期目标拆出 2-4 个阶段
- 每个阶段必须有可交付、可验收的产出物

**华渔案例**：

中短期目标："完成 12 类 89+ 个维度的信息填充，覆盖率 ≥ 80%"

| 阶段 | 阶段结项目标 | 交付物 | 验收标准 | 时间 |
|------|------------|--------|---------|------|
| 阶段 1 | 维度框架搭建 + A/K 类填充 | 维度文档 v0.1 | 12 类维度结构定义完成，A/K 类信息填充完成 | 2 周 |
| 阶段 2 | B/C/D/E 类填充 | 维度文档 v0.5 | 核心设计方法类维度填充完成，覆盖率 ≥ 50% | 4 周 |
| 阶段 3 | F/G/H/I/J/L 类填充 | 维度文档 v0.8 | 全部维度框架填充完成，覆盖率 ≥ 80% | 6 周 |
| 阶段 4 | 私域信息补充 + 质量审核 | 维度文档 v1.0 | 私域信息填充 ≥ 50%，通过 DJ 审核 | 4 周 |

##### 原则 3：阶段结项目标 → 月度计划（任务化分解）

**拆解方法**：
1. 将阶段目标分解为具体的任务项（Task）
2. 每个任务有明确的负责人、预估工时、完成标准
3. 任务之间的依赖关系明确标注

**粒度标准**：
- 时间跨度：约 4 周
- 单个任务预估工时：2-40 小时（超过 40 小时的继续拆分）
- 每个任务有明确的"完成定义"（Definition of Done）

**模板**：

```markdown
### 月度计划：[YYYY-MM]

**对齐的阶段目标**：[阶段X 的目标描述]

| 任务ID | 任务描述 | 负责人 | 预估工时 | 依赖 | 完成标准 | 状态 |
|--------|---------|--------|---------|------|---------|------|
| T01 | [具体任务] | [人/AI] | [X小时] | [无/T0X] | [具体完成标准] | 🔴/🟡/🟢 |
```

##### 原则 4：月度计划 → 周版本（增量发布）

**拆解方法**：
1. 将月度任务按周分配，确保每周有可展示/可发布的增量
2. 每周结束时有"周版本"——不必完美，但必须是可见的进展
3. 遵循 DJ "步步为营"原则：每周都有可见的成果，而非最后一周突击交付

**粒度标准**：
- 时间跨度：1 周（5 个工作日）
- 每周必须有至少 1 个可展示的增量
- 周版本需要有简短的 changelog（做了什么、变化了什么、下周做什么）

**模板**：

```markdown
### 周版本：[YYYY-WXX]（[MM-DD] ~ [MM-DD]）

**本周交付增量**：
- [增量1描述 + 验收结果]
- [增量2描述 + 验收结果]

**本周发现/阻塞**：
- [如有]

**下周计划**：
- [计划1]
- [计划2]
```

#### 拆解质量自检清单

每次拆解完成后，用以下清单自检：

| 检查项 | 检查方法 | 通过标准 |
|--------|---------|---------|
| **纵向对齐** | 从周版本向上追溯，是否能连到理想化状态？ | 每个周版本至少对应一个月度任务→一个阶段目标→一个中短期目标→理想化状态的某个切面 |
| **横向覆盖** | 同一层级的多个目标合在一起，是否覆盖了上层目标？ | 阶段目标合起来 = 中短期目标的 100%；中短期目标合起来 ≥ 理想化状态的 70% |
| **粒度适当** | 每个层级的目标是否在合理的粒度范围内？ | 中短期(3-12月) / 阶段(1-3月) / 月度(4周) / 周版本(1周) |
| **可验收** | 每个目标是否有明确的验收标准？ | 验收标准可以回答"做到了/没做到"，不能模糊 |
| **步步为营** | 是否每周都有可见的增量？ | 不存在"前3周准备，第4周才出成果"的情况 |

**数据来源**：项目规划经验、DJ "步步为营"理念、通用产品管理方法论（ICE/SMART/增量交付）。

**更新机制**：方法优化时更新。每个大项目结项后复盘拆解质量，更新案例。

## L08 中短期目标定义标准

**维度定义**：中短期目标的时间范围（3-12 个月）、定义标准（可衡量、可实现、与理想化状态对齐），包含编写模板、案例。

**质量作用**：中短期目标质量——目标明确可执行，是连接理想化状态和阶段执行的关键桥梁。

### 具体信息内容

#### 中短期目标定义标准（SMART+V）

在经典 SMART 基础上增加 V（Value-aligned，价值对齐）：

| 标准 | 定义 | 检查方法 |
|------|------|---------|
| **S** — Specific（具体） | 目标描述具体到"谁、做什么、达到什么程度" | 能否回答"对谁/做什么/多好"？ |
| **M** — Measurable（可衡量） | 有量化指标或可观察的完成标准 | 能否用数据判断"达到了/没达到"？ |
| **A** — Achievable（可实现） | 在现有资源（对照 F01）和技术能力内可实现 | 技术团队确认可行 + 资源匹配？ |
| **R** — Relevant（相关） | 与理想化状态的某个切面直接相关 | 完成这个目标后，离理想化状态更近了吗？ |
| **T** — Time-bound（有时限） | 明确的截止时间（3-12 个月） | 有明确的截止日期？ |
| **V** — Value-aligned（价值对齐） | 对齐 A01 战略方向和 DJ 教育理念 | 通过 A01 对齐检查？ |

## L09 阶段结项目标定义标准

**维度定义**：阶段结项目标的时间范围（1-3 个月）、定义标准（可交付、可验收、与中短期目标对齐），包含编写模板、验收标准、案例。

**质量作用**：阶段目标质量——目标可验收，是项目管理的核心颗粒度。

### 具体信息内容

#### 阶段结项目标定义标准

| 标准 | 定义 | 检查方法 |
|------|------|---------|
| **可交付** | 阶段结束时必须有具体的交付物（文档/代码/原型/报告） | 交付物列表是否明确？ |
| **可验收** | 每个交付物有明确的验收标准，可以判断"通过/不通过" | 验收标准是否具体到"是/否"判断？ |
| **对齐上层** | 与对应的中短期目标直接关联，完成后中短期目标进度可量化推进 | 完成此阶段后，中短期目标的衡量指标提升了吗？ |
| **自包含** | 阶段目标可以独立验收，不依赖后续阶段才能体现价值 | 如果后续阶段取消，此阶段的产出是否仍然有用？ |

## L10 目标对齐检查机制

**维度定义**：如何检查周版本→月度计划→阶段结项目标→中短期目标→理想化状态的对齐性，包含检查方法、对齐标准、偏离处理。

**质量作用**：目标一致性——确保所有目标层层对齐，不偏航。这是 DJ "步步为营"理念的制度保障。

### 具体信息内容

#### 对齐检查的时机

| 检查时机 | 检查内容 | 检查方法 | 责任人 |
|---------|---------|---------|--------|
| **每周五** | 周版本 ↔ 月度计划对齐 | 周版本 changelog 是否覆盖了月度计划中本周任务 | 项目执行者 |
| **每月末** | 月度计划 ↔ 阶段目标对齐 | 本月完成的任务合起来是否推进了阶段目标 | 项目管理者 |
| **阶段结项** | 阶段目标 ↔ 中短期目标对齐 | 阶段交付物合起来是否推进了中短期目标的衡量指标 | 项目负责人 |
| **每季度** | 中短期目标 ↔ 理想化状态对齐 | 中短期目标的进展是否让我们离理想化状态更近 | Leader / DJ |
| **战略调整时** | 理想化状态 ↔ 战略方向对齐 | 理想化状态是否仍然对齐最新的战略方向（A01） | DJ |

#### 对齐检查方法

**方法 1：自下而上追溯法**

从最底层（周版本）开始，逐层向上追问：

```
周版本的每个增量
  → "这个增量服务于哪个月度任务？"
    → "这个月度任务服务于哪个阶段目标？"
      → "这个阶段目标服务于哪个中短期目标？"
        → "这个中短期目标服务于理想化状态的哪个切面？"
```

如果任何一层追溯断裂（找不到上层对应），说明存在对齐问题。

**方法 2：自上而下覆盖法**

从最顶层（理想化状态）开始，逐层向下检查覆盖度：

```
理想化状态的每个切面
  → "有对应的中短期目标吗？"（覆盖度 ≥ 70%）
    → "每个中短期目标有对应的阶段目标吗？"（覆盖度 100%）
      → "每个阶段目标有对应的月度任务吗？"（覆盖度 100%）
        → "每个月度任务有对应的周版本增量吗？"（覆盖度 ≥ 80%）
```

如果覆盖度不足，说明存在"空白地带"——有目标但没有执行计划。

**方法 3：偏离度量化**

每次检查时，用以下公式估算偏离度：

```
偏离度 = (计划外工作量 / 总工作量) × 100%

- 偏离度 < 10%：✅ 健康（正常的灵活调整）
- 10% ≤ 偏离度 < 30%：🟡 需关注（是否有新的紧急需求插入？）
- 偏离度 ≥ 30%：🔴 需干预（目标可能需要调整）
```

#### 偏离处理决策树

```
发现对齐偏离
    │
    ├─ 偏离原因：外部变化（战略调整、市场变化、DJ 新指令）
    │   → 调整目标（自上而下更新）
    │   → 记录变更原因
    │
    ├─ 偏离原因：执行偏离（做了不在计划中的事）
    │   ├─ 偏离的工作有价值吗？
    │   │   ├─ 有 → 补录到计划中，调整上层目标覆盖
    │   │   └─ 没有 → 停止偏离工作，回归计划
    │   └─ 分析偏离原因（目标不清晰？任务分配不当？紧急插入？）
    │
    ├─ 偏离原因：目标本身不合理（过高/过低/方向错误）
    │   → 重新评估目标（回到 L06-L08 调整）
    │   → 记录教训到 G 类维度
    │
    └─ 偏离原因：资源不足（人力/技术/时间）
        → 调整目标范围（砍功能/延期/申请资源）
        → 更新 F01 资源现状快照
```

#### 华渔场景的特别注意事项

- **DJ "步步为营"原则**：对齐检查的核心不是"目标做对了吗"，而是"每一步都有可见的进展吗"。如果阶段目标对齐但连续两周没有可见增量，需要干预
- **提示词工程项目特殊性**：维度填充是知识型工作，进展不像代码那样容易衡量——建议用"填充覆盖率"作为周版本的可量化增量
- **多产品线协调**：如果同一个中短期目标涉及多个产品线（如 101 PPT + 未来实验室），需要额外检查跨产品线的对齐一致性

**更新机制**：每半年优化检查方法。每次重大偏离后复盘，更新偏离处理决策树。
