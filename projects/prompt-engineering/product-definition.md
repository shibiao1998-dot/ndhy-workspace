## 一、产品概念

**一句话定义**：提示词工程系统是一个 Web 工具，让华渔设计师/策划输入工作场景后，自动从 89 个预置维度中路由、排序、组装出针对特定 AI 引擎优化的高质量提示词，一键复制即用。

**核心隐喻**：它是"AI 的信息弹药库调度系统"——弹药（维度内容）已经备好，系统负责根据战场（场景）选择弹药、按重要性装填、适配武器（引擎），交给士兵（用户）一键发射。

**DJ 定义对齐**：
- 本质 = 信息对称（为 AI 提供最全面最准确的信息）
- 系统职责 = 帮用户完成"信息收集→排序筛选→引擎适配"这三步，不让人手动翻 434KB 维度文件

## 二、核心场景

### 场景 1：设计师接到新设计任务，需要给 AI 写提示词

| 维度 | 内容 |
|------|------|
| **用户** | 华渔设计师（如杜建强团队），日常使用 Claude/Midjourney 做设计 |
| **触发** | 收到需求"设计一个初中化学虚拟实验的交互方案" |
| **痛点** | 以前写提示词靠个人经验，想到什么写什么，漏掉战略对齐、用户研究、竞品差异化等关键维度，导致 AI 产出平庸 |
| **目标** | 拿到一份覆盖所有相关维度、按优先级排好序、适配目标引擎的提示词 |
| **操作流程** | ① 打开 Web 页面 → ② 输入任务描述"设计一个初中化学虚拟实验交互方案" → ③ 系统自动匹配任务类型"原型设计"并选出相关维度 → ④ 用户确认/调整维度选择和优先级 → ⑤ 选择目标引擎（Claude） → ⑥ 点击"生成" → ⑦ 查看生成的提示词 → ⑧ 一键复制，粘贴到 Claude 使用 |
| **成功标志** | 从输入到拿到提示词 < 2 分钟；生成的提示词覆盖了用户自己想不到的维度 |

### 场景 2：策划要给不同 AI 引擎分别准备提示词

| 维度 | 内容 |
|------|------|
| **用户** | 华渔策划（如金伟华团队），需要同时用 Claude 做方案 + Midjourney 出效果图 |
| **触发** | 同一个项目需要文字方案（Claude）和视觉方案（Midjourney），两个引擎的提示词格式完全不同 |
| **痛点** | 手动改格式：Claude 要结构化长文本，Midjourney 要精简关键词。每次改都可能丢信息 |
| **目标** | 同一个任务一键切换引擎，自动适配输出格式和长度 |
| **操作流程** | ① 输入任务描述 → ② 生成 Claude 版提示词并复制 → ③ 切换引擎到 Midjourney → ④ 系统自动重新组装（精简关键词、4K 字符以内） → ⑤ 复制 Midjourney 版提示词 |
| **成功标志** | 切换引擎后 < 3 秒出结果；两个版本信息一致但格式各自最优 |

### 场景 3：DJ 要求复盘提示词质量改进

| 维度 | 内容 |
|------|------|
| **用户** | DJ 本人或其指定的复盘负责人 |
| **触发** | DJ 原话："要看提示词本身的进步：原来怎么写→现在怎么写→结果为什么变好了" |
| **痛点** | 不知道提示词覆盖了哪些维度、遗漏了哪些、每次用了多少信息 |
| **目标** | 看到每次生成的提示词覆盖了哪些维度、维度覆盖率是多少 |
| **操作流程** | ① 生成提示词后 → ② 页面下方显示"维度覆盖报告"：已用维度列表、各级维度数量、覆盖率百分比、缺失的必须维度警告 |
| **成功标志** | 用户能直观看到"这次用了 23/89 个维度，覆盖率 87%（按权重）" |

### 场景 4：用户觉得自动路由不准，想手动调整

| 维度 | 内容 |
|------|------|
| **用户** | 任何使用系统的华渔员工 |
| **触发** | 系统自动匹配了"产品策划"类型，但用户觉得应该是"设计方法论编写" |
| **痛点** | 关键词匹配有时不准确（系统已知限制），需要人工纠偏手段 |
| **目标** | 能手动切换任务类型、手动增删维度 |
| **操作流程** | ① 系统自动匹配后显示"匹配结果：产品策划（置信度 65%）" → ② 用户点击"切换类型"选择"设计方法论编写" → ③ 维度列表自动更新 → ④ 用户在维度列表中手动勾选/取消某些维度 → ⑤ 重新生成 |
| **成功标志** | 切换类型后维度列表即时刷新；手动调整不丢失已有输入 |

### 场景 5：用户想微调维度优先级

| 维度 | 内容 |
|------|------|
| **用户** | 对维度内容有了解的华渔资深设计师/策划 |
| **触发** | 默认优先级不完全匹配当前任务——比如本次任务竞品分析特别重要，想把 C 类从二级提升到一级 |
| **痛点** | 固定优先级无法适配所有场景的差异化需求 |
| **目标** | 手动调整维度级别（一级/二级/三级），调整后重新组装 |
| **操作流程** | ① 维度列表中每个维度显示当前级别（一级🔴/二级🟡/三级🟢） → ② 用户拖拽或点击切换维度级别 → ③ 点击重新生成 → ④ 组装引擎按新优先级重排 |
| **成功标志** | 优先级调整后输出可见变化（高优维度内容排前，低优被截断的维度重新出现） |

## 四、产品规则与约束

### 4.1 维度路由规则

| 规则 | 具体定义 | 触发条件 |
|------|----------|----------|
| **R1 关键词匹配路由** | 用户输入的任务描述与 `TASK_KEYWORDS` 中的关键词列表做包含匹配，命中数最多的类型胜出 | 每次输入任务描述或修改后 |
| **R2 置信度显示** | 匹配置信度 = 命中关键词数 / 该类型总关键词数，最低 0.5（有命中），无命中时回退"通用设计任务"（0.3） | 路由完成后 |
| **R3 手动覆盖优先** | 用户手动选择的任务类型覆盖自动匹配结果，置信度显示为 100% | 用户点击切换 |
| **R4 路由回退** | 无关键词命中时，使用"通用设计任务"路由（必须维度：A+B+D+K） | 匹配得分全部为 0 |

### 4.2 维度优先级规则

| 规则 | 具体定义 |
|------|----------|
| **P1 三级优先级体系** | 一级（必须/required）= 缺失会警告；二级（建议/recommended）= 空间允许时加载；三级（可选/optional）= 充裕时加载 |
| **P2 默认优先级来源** | 每个任务类型在 `TASK_ROUTES` 中预定义了各维度的默认级别 |
| **P3 用户可调整** | 用户可将任意维度在三级之间切换。但取消"必须"级维度时显示警告（不阻止） |
| **P4 类别默认优先级** | A（战略）、K（术语）、B（用户）、D（场景）= 一级；C（竞品）、E（方法）、F（可行性）、I（质量）、G（历史）、H（规范）= 二级；J（专业知识）、L（生命周期）= 三级 |

### 4.3 组装规则

| 规则 | 具体定义 |
|------|----------|
| **A1 组装顺序** | 系统指令头 → 一级维度内容（按类别字母序） → 二级维度内容 → 三级维度内容 → 反向约束段 |
| **A2 长度控制** | 先满足全部一级维度；剩余空间填二级维度（按字符数从小到大）；仍有空间填三级维度。超过引擎 max_tokens 时截断低优先级维度（整个维度不拆分） |
| **A3 反向约束提取** | 从维度内容中自动提取 ❌、"不要"、"禁止"、"红线" 标记的内容，独立汇总为"约束与禁忌"段落 |
| **A4 维度整体性** | 单个维度要么完整包含，要么完全跳过。不做维度内部截断（保证信息完整性） |

### 4.4 引擎适配规则

| 引擎 | 格式 | 最大长度 | 适配策略 |
|------|------|----------|----------|
| Claude / GPT / DeepSeek | 结构化 Markdown | 180K / 120K / 60K 字符 | 完整维度内容，层级标题，代码块格式的具体案例 |
| GPT-4 / GPT-4o | 结构化 Markdown | 120K 字符 | 同 Claude，但空间更紧凑时优先保留一级+二级 |
| DeepSeek | 结构化 Markdown | 60K 字符 | 空间有限，只保留一级维度的核心内容 |
| Midjourney / DALL-E | 精简关键词 | 4K 字符 | 提取关键词、风格词、情感词，去掉所有详细说明 |
| Suno | 简洁描述 | 2K 字符 | 提取情感、氛围、节奏描述，去掉技术细节 |

### 4.5 系统行为规则

| 规则 | 具体定义 |
|------|----------|
| **B1 无状态** | 每次页面加载是全新会话。不读取、不存储任何用户历史数据 |
| **B2 维度文件热加载** | 后端启动时扫描 `dimensions/` 目录并缓存维度索引。提供 `POST /api/reload` 接口手动触发重新扫描（维度专家更新文件后使用） |
| **B3 容错处理** | 维度文件中标记 `⚠️ 待填充` 的维度仍然加载，但在维度列表中灰色显示并标注"内容未完成" |
| **B4 错误提示** | 所有用户可感知的错误（路由失败、生成失败）显示具体原因和建议操作，不显示技术堆栈 |
| **B5 单一业务线** | 系统只服务"AI 设计师/策划"场景。维度内容来自 `dimensions/` 目录下的 A-L 类文件 |

## 六、后端 API 概要

> 供技术架构专家和 API 设计专家参考的接口概要（详细 Schema 由 API 设计专家定义）

| # | 接口 | 方法 | 功能 | 关键入参 | 关键返回 |
|---|------|------|------|----------|----------|
| 1 | `/api/dimensions` | GET | 获取维度索引 | — | `[{id, name, category, char_count, status}]` |
| 2 | `/api/task-types` | GET | 获取任务类型列表 | — | `[{key, name, description}]` |
| 3 | `/api/engines` | GET | 获取引擎列表 | — | `[{key, name, type, max_tokens}]` |
| 4 | `/api/route` | POST | 执行维度路由 | `{task, task_type?}` | `{task_type, confidence, required[], recommended[], optional[], total_chars}` |
| 5 | `/api/generate` | POST | 生成提示词 | `{task, engine, dimensions[], priorities{}}` | `{prompt, stats{total_chars, dimensions_used, coverage}}` |
| 6 | `/api/reload` | POST | 重新加载维度文件 | — | `{dimensions_count, categories_count}` |

**技术基础**：现有 Python CLI 系统（`system/` 目录）已实现维度加载、路由、组装、适配四个核心模块，零外部依赖。后端需将这些模块封装为 HTTP API。

### 8.3 DJ 复盘标准对照

> DJ："要看提示词本身的进步：原来怎么写→现在怎么写→结果为什么变好了"

| 对比维度 | 原来（手写提示词） | 现在（系统生成） | 为什么变好了 |
|----------|-------------------|-----------------|-------------|
| 维度覆盖 | 凭个人经验，3-5 个维度 | 系统路由，15-30 个维度 | 信息更对称，AI 收到的信息量翻 3-6 倍 |
| 优先级 | 无排序，想到什么写什么 | 三级排序，重要信息在前 | AI 在长文本中优先读到关键信息 |
| 引擎适配 | 一份提示词走天下 | 按引擎自动调格式和长度 | Midjourney 不再收到 10 万字长文，Claude 不再缺少关键上下文 |
| 可复现 | 每次写法不同 | 相同输入 = 相同输出 | 质量稳定可复盘，而非依赖个人状态 |
| 反向约束 | 容易遗漏"不要什么" | 系统自动提取并单独汇总 | AI 不再做被明确禁止的事（如"不做大模型研发"相关内容） |

## 九、关键约束与风险

### 约束

| 约束 | 来源 | 影响 |
|------|------|------|
| 零/低外部依赖 | 现有系统设计决策 | 后端纯 Python 标准库，前端可选轻量方案 |
| 维度内容只读 | 60分边界（X5） | 维度更新流程：专家编辑 Markdown 文件 → 调用 /api/reload |
| 华渔内网部署 | DJ 团队使用环境 | 不需要公网域名/HTTPS/CDN |
| 单业务线 | 60分边界（X4） | config.py 中其他业务线路由配置保留但不暴露 |

### 风险

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| 关键词路由不够准确 | 高 | 用户需频繁手动切换 | F3 手动切换 + F5 维度勾选作为补偿；v2 引入 AI 路由 |
| 维度内容质量参差 | 中 | 部分维度内容太笼统影响提示词质量 | F11 覆盖报告暴露问题；B3 灰色标注未完成维度 |
| 434KB 维度内容传输慢 | 低 | 首次加载时间长 | F13 只传元信息不传全文；生成时后端读取全文 |
| 60分边界蔓延 | 中 | 开发过程中加需求 | 本文档"❌ 不做"清单为硬约束，Phase 2 后锁定 |
