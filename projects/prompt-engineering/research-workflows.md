## 1. 软件/产品开发标准工作流程

### 1.1 瀑布模型（Waterfall Model）

> **来源**：Winston W. Royce（1970）原始论文；ProjectManager.com Waterfall Methodology Guide

**核心特征**：线性顺序、每个阶段完成后才进入下一阶段、不可回退（理论上）。

#### 标准阶段（严格串行）

```
需求分析 → 系统设计 → 实现（编码）→ 测试验证 → 部署 → 维护
```

| 阶段 | 具体步骤 | 准入条件 | 准出条件 | NDHY 对应角色 |
|------|---------|---------|---------|--------------|
| **1. 需求分析** | ①收集利益相关者需求 ②定义项目范围 ③市场调研 ④组建团队 ⑤启动会 | 项目立项 | 需求文档签字确认 | 🎯需求分析专家、📊市场评估专家、🔍用户研究专家 |
| **2. 系统设计** | ①逻辑设计（方案脑暴和理论推演）②物理设计（转化为具体规格）③WBS 任务分解 ④创建时间表和预算 | 需求文档已确认 | 设计文档完成、计划评审通过 | 🧭战略定位专家、📐产品定义专家、🏛️技术架构专家、🗄️数据库设计专家、📜API设计专家 |
| **3. 实现（编码）** | ①分配任务 ②单元开发 ③单元测试 ④资源管理和负载均衡 ⑤向利益相关者汇报进度 | 设计文档完成 | 代码完成、单元测试通过 | ⚙️后端开发专家、🖥️前端开发专家 |
| **4. 测试验证** | ①集成测试 ②系统测试 ③Bug 修复 ④部署到生产环境 | 代码开发完成 | 测试报告通过、无阻断缺陷 | 🔗联调集成专家、🧪测试专家、🔍代码审查专家 |
| **5. 部署** | ①发布到生产环境 ②合同收尾 ③存档模板 ④总结文档 | 测试通过 | 系统上线运行 | 🚀部署运维专家、📦发布策略专家 |
| **6. 维护** | ①Bug 修复 ②功能增强 ③用户支持 | 系统已上线 | 持续进行 | 🚀部署运维专家、📈增长运营专家 |

**⚠️ 串并行规则**：
- **严格串行**：每个阶段必须完成后才能进入下一阶段
- **不可并行**：理论上不允许阶段重叠
- **不可回退**：一旦进入下一阶段，不应回到上一阶段（实际中常需要，这是瀑布模型的主要缺陷）

**适用场景**：需求明确且不变的项目（如政府采购、合规系统、硬件集成项目）

### 1.2 Scrum 框架

> **来源**：Scrum Guide 2020（Ken Schwaber & Jeff Sutherland 官方版本）

**核心特征**：迭代增量、经验主义（透明-检视-适应）、自组织团队。

#### Scrum 角色

| Scrum 角色 | 职责 | NDHY 对应 |
|-----------|------|----------|
| **Product Owner** | 最大化产品价值、管理 Product Backlog、排列优先级 | Leader（目标定义）+ 老板（最终决策） |
| **Scrum Master** | 促进 Scrum 落地、移除障碍、教练团队 | 🏗️项目管理专家 |
| **Developers** | 创建每个 Sprint 的可用增量 | 所有执行类专家 |

#### Sprint 周期（标准 2-4 周）

```
Product Backlog 梳理
        ↓
Sprint Planning（Sprint 开始）
        ↓
Daily Scrum（每天）→ 开发工作
        ↓
Sprint Review（Sprint 结束，展示增量）
        ↓
Sprint Retrospective（回顾改进）
        ↓
下一个 Sprint Planning
```

#### 5 个 Scrum 事件

| 事件 | 时间 | 参与者 | 目的 | 产出 |
|------|------|--------|------|------|
| **Sprint Planning** | Sprint 开头，4-8h（2周Sprint） | 全 Scrum Team | 确定 Sprint Goal + 选择 Backlog Items | Sprint Backlog |
| **Daily Scrum** | 每天 15min | Developers | 检视进展、调整当日计划 | 当日行动计划 |
| **Sprint Review** | Sprint 末尾，2-4h | Scrum Team + 利益相关者 | 检视增量、收集反馈、调整 Backlog | 更新的 Product Backlog |
| **Sprint Retrospective** | Sprint Review 后，1.5-3h | Scrum Team | 回顾改进、计划下一步优化 | 改进行动项 |
| **Product Backlog Refinement** | 持续活动（Sprint 内约 10% 时间） | Product Owner + Developers | 细化未来 Sprint 的 Backlog Items | 细化后的 Backlog |

**⚠️ 串并行规则**：
- Sprint **内部**：设计、开发、测试**可以并行**（但对同一 feature 仍有先后顺序）
- Sprint **之间**：严格串行（上一个 Sprint 完成后才开始下一个）
- **关键约束**：Sprint Goal 一旦设定，在 Sprint 期间不应更改
- **定义完成（DoD）**：增量必须满足 DoD 才算完成，不能"假完成"

### 1.3 看板（Kanban）

> **来源**：Atlassian Kanban Guide；David J. Anderson（2010）《Kanban: Successful Evolutionary Change for Your Technology Business》

**核心特征**：持续流动、WIP 限制、可视化、没有固定迭代周期。

#### 看板核心实践

| 实践 | 说明 |
|------|------|
| **可视化工作流** | 所有工作用卡片展示在看板上 |
| **限制在制品（WIP）** | 每列设置最大卡片数，防止过载 |
| **管理流动** | 跟踪前置时间（Lead Time）和周期时间（Cycle Time） |
| **明确流程策略** | 每列的进入和退出标准要清晰 |
| **实施反馈循环** | 定期检视和改进 |
| **协作改进** | 团队共同优化流程 |

#### 典型看板列

```
Backlog → Selected → 分析 → 开发 → 测试 → 部署 → Done
         [WIP:3]  [WIP:2] [WIP:3] [WIP:2] [WIP:1]
```

**⚠️ 串并行规则**：
- **单个工作项**：必须按列顺序流动（串行）
- **不同工作项**：可以在不同列同时处理（并行）
- **WIP 限制**是关键：某列达到 WIP 上限时，上游必须停止推送
- **拉动制（Pull System）**：下游拉动上游，不是上游推动下游

### 1.4 敏捷开发（Agile）总体原则

> **来源**：Agile Manifesto（2001）；Scrum Guide 2020

#### 敏捷宣言四大价值观
1. 个体和互动 > 流程和工具
2. 可工作的软件 > 详尽的文档
3. 客户协作 > 合同谈判
4. 响应变化 > 遵循计划

#### 敏捷中的串并行原则

| 必须串行 | 可以并行 |
|---------|---------|
| 需求理解 → 方案设计（没理解清楚不能设计） | 不同功能模块的开发（无依赖时） |
| 架构设计 → 详细设计（架构方向未定不能做细节） | 前端 ∥ 后端（基于约定的 API 契约） |
| API 契约设计 → 前后端开发（没契约不能联调） | 单元测试 ∥ 文档编写 |
| 开发 → 代码审查（没代码不能审查） | 多 Sprint Team 各自的 Sprint |
| 代码审查 → 合并（未审查不能合并） | 测试用例设计 ∥ 开发进行中 |
| 联调通过 → 系统测试（联调不通不能测） | UI 设计 ∥ 技术架构（面向同一产品定义） |

### 1.5 前端/后端/设计/测试 的正确协作顺序

> **来源**：行业最佳实践综合（Google Design Sprint, Spotify Squad Model, Atlassian DACI）

#### 标准产品开发协作流

```
Phase 0: 信息采集
  用户研究 ∥ 市场评估 ∥ 竞品分析
        ↓（联合输出）
Phase 1: 需求整合
  需求分析（整合 Phase 0 输出）
        ↓
Phase 2: 方案设计
  战略定位
        ↓
  产品定义
        ↓
  体验设计（UX）→ 视觉设计（UI）  ← 注意：这两步必须串行
        ∥                              ↓
  技术架构设计 ──────────────→ 设计交接（Design Handoff）
        ↓
  数据库设计 ∥ API 设计 ← 注意：这两步可以并行，但需要字段语义对齐
        ↓
Phase 3: 开发实现
  后端开发 ∥ 前端开发 ← 基于 API 契约可并行
        ↓
  代码审查（各自独立 → 交叉审查）
        ↓
Phase 4: 集成验证
  联调集成（前后端对齐）
        ↓
  系统测试（功能/边界/异常/回归）
        ↓
Phase 5: 发布上线
  发布策略制定 ∥ 部署准备
        ↓
  灰度发布 → 全量发布
        ↓
Phase 6: 运营增长
  增长运营 + 数据分析 + 迭代优化
```

#### ⚡ 关键依赖关系（绝对不能违反的顺序）

| 序号 | 依赖规则 | 原因 |
|------|---------|------|
| 1 | **需求理解 → 方案设计** | 不知道做什么就不能决定怎么做 |
| 2 | **产品定义 → 体验设计** | 不知道产品是什么就不能设计用户体验 |
| 3 | **体验设计 → 视觉设计** | 不知道交互骨架就不能填视觉血肉 |
| 4 | **技术架构 → 数据库设计/API 设计** | 架构方向未定就不能做细节设计 |
| 5 | **API 契约 → 前后端开发** | 没有契约就不能保证前后端一致 |
| 6 | **数据模型 → 后端开发** | 没有数据模型后端没法写 Repository 层 |
| 7 | **前后端各自开发完成 → 联调集成** | 模块没做完不能集成 |
| 8 | **联调通过 → 系统测试** | 模块间都不通，测功能没意义 |
| 9 | **系统测试通过 → 发布部署** | 质量不达标不能上线 |
| 10 | **战略定位 → 产品定义** | 方向未定不能做具体产品定义 |

## 2. 设计行业标准工作流程

### 2.1 UI/UX 设计标准流程

> **来源**：Smashing Magazine "The Real-Life UX Design Process"（Marcin Treder）；Design Council Double Diamond Model（2004）；IDEO Design Thinking

#### Double Diamond 模型

```
   发散          收敛          发散          收敛
    ↓             ↓             ↓             ↓
  Discover → Define → Develop → Deliver
  （发现）   （定义）   （开发）   （交付）
```

- **Diamond 1（做对的事）**：理解问题 → 定义问题
- **Diamond 2（把事做对）**：探索方案 → 交付方案

#### 完整 UX 设计流程（7 步）

| 步骤 | 具体活动 | 串/并行 | 准入条件 | 准出条件 | NDHY 对应角色 |
|------|---------|--------|---------|---------|--------------|
| **1. 用户研究** | 用户访谈、画像构建、旅程地图、需求推演 | 可与市场调研并行 | 项目启动 | 用户洞察报告 | 🔍用户研究专家 |
| **2. 需求定义** | 功能需求整理、优先级排序、范围界定 | 必须在研究后 | 用户洞察完成 | 需求文档 | 🎯需求分析专家 |
| **3. 信息架构** | 站点地图、导航结构、内容分层、标签系统 | 必须在需求定义后 | 需求和产品定义完成 | 信息架构图 | 🎨体验设计专家 |
| **4. 线框图/原型** | 低保真线框 → 交互原型 → 用户测试 | 串行（逐步细化） | 信息架构完成 | 交互原型 + 测试反馈 | 🎨体验设计专家 |
| **5. 视觉设计** | 配色系统、字体排印、组件视觉、品牌视觉 | 必须在线框图后 | 交互原型确认 | 视觉规范文档 | 🎨视觉设计专家 |
| **6. 设计交接** | Design Token 输出、组件规格标注、切图、动效说明 | 必须在视觉完成后 | 视觉稿确认 | 设计交付包 | 🎨视觉设计专家 |
| **7. 开发协作** | 组件走查、还原度检查、设计 QA | 与开发并行 | 开发已启动 | 还原度通过 | 🎨体验设计专家 + 🎨视觉设计专家 |

**⚠️ 关键串行约束**：
- **用户研究 → 信息架构**（不了解用户就不能组织信息）
- **信息架构 → 线框图**（不知道信息结构就不能画页面）
- **线框图 → 视觉设计**（不知道布局就不能做视觉）
- **视觉设计 → 设计交接**（视觉未定就不能输出规范）

**可并行的部分**：
- 用户研究 ∥ 竞品分析（都是信息采集阶段）
- 视觉设计 ∥ 技术架构设计（各自独立，但同步进行）
- 不同页面/模块的设计（无强依赖时可并行）

### 2.2 品牌设计标准流程

> **来源**：AIGA (American Institute of Graphic Arts) 品牌设计标准流程；Brand New (Under Consideration)

```
品牌调研 → 品牌策略 → 品牌概念 → 视觉识别系统 → 品牌应用 → 品牌管理
```

| 步骤 | 具体活动 | 依赖 |
|------|---------|------|
| **1. 品牌调研** | 市场分析、竞品品牌分析、目标用户洞察、企业现状评估 | 项目立项 |
| **2. 品牌策略** | 品牌定位、价值主张、品牌个性、品牌故事 | 依赖调研结论 |
| **3. 品牌概念** | 核心视觉概念、Moodboard、概念方向提案 | 依赖品牌策略 |
| **4. 视觉识别系统** | Logo、色彩系统、字体系统、图形元素、设计语言 | 依赖概念确认 |
| **5. 品牌应用** | 名片、网站、APP、包装、环境导视 | 依赖识别系统 |
| **6. 品牌管理** | 品牌手册、使用规范、培训 | 依赖应用设计 |

**全部严格串行**，每步依赖上一步的产出。

### 2.3 设计与开发的交接规范（Design Handoff）

> **来源**：InVision Design Handoff Best Practices；Figma Dev Mode Documentation

#### 交接清单

| 交接物 | 内容 | 责任方 | 消费方 |
|--------|------|--------|--------|
| **Design Tokens** | 颜色变量、间距、圆角、阴影、字号 | 视觉设计专家 | 前端开发专家 |
| **组件规格** | 各状态视觉（default/hover/active/disabled/error）| 视觉设计专家 | 前端开发专家 |
| **交互规范** | 状态转换规则、动画时序、响应式断点 | 体验设计专家 | 前端开发专家 |
| **文案规范** | 按钮文案、提示语、错误文案、空状态文案 | 体验设计专家 | 前端开发专家 |
| **标注文档** | 尺寸、间距、字号、颜色代码 | 视觉设计专家 | 前端开发专家 |
| **切图资源** | SVG 图标、位图资源、动效文件 | 视觉设计专家 | 前端开发专家 |

**交接不是一次性事件，而是持续协作过程。**

## 3. 提示词工程 / AI 内容生产工作流程

### 3.1 提示词工程最佳实践流程

> **来源**：OpenAI Prompt Engineering Guide；Anthropic Prompt Design Documentation；Google DeepMind PALM Prompting Techniques

#### 6 步提示词工程流程

```
目标定义 → 信息收集 → 提示词设计 → 测试迭代 → 优化固化 → 部署监控
```

| 步骤 | 具体活动 | 关键原则 | NDHY 对应角色 |
|------|---------|---------|--------------|
| **1. 目标定义** | ①明确 AI 要完成的任务 ②定义输出格式和质量标准 ③确定评估指标 | 先定义成功标准再动手 | 🎯需求分析专家 + Leader |
| **2. 信息收集** | ①收集领域知识 ②整理上下文信息 ③准备示例数据 ④识别约束条件 | 信息对称是核心——AI 结果不好 = 信息不够 | 🔬技术调研专家 + 领域专家 |
| **3. 提示词设计** | ①选择提示词策略（Zero-shot/Few-shot/CoT/ReAct）②结构化提示词组件 ③设计系统提示词 + 用户提示词 | 结构清晰、指令明确、角色设定 | 📝技术文档专家 |
| **4. 测试迭代** | ①边界测试 ②对抗测试 ③多模型对比 ④评估输出质量 | 每次只改一个变量 | 🧪测试专家 |
| **5. 优化固化** | ①提炼最佳模板 ②版本化管理 ③建立提示词库 | 可复用 > 一次性 | 🔧Skill开发专家 |
| **6. 部署监控** | ①生产环境部署 ②输出质量监控 ③用户反馈收集 ④持续优化 | 提示词不是写完就完了 | 🚀部署运维专家 + 📈数据分析专家 |

**⚠️ 串并行规则**：
- 步骤 1→2→3 **严格串行**（不知道目标就不能收集信息，不知道信息就不能设计提示词）
- 步骤 3↔4 **快速迭代循环**（设计→测试→调整→再测试）
- 步骤 5→6 **串行**（先固化再部署）
- 不同业务线的提示词工程可以**并行**

### 3.2 AI 内容生产管线（Content Pipeline）

> **来源**：Content Science "AI Content Operations Framework"；GPT-based Content Production 行业实践

```
信息源 → 采集清洗 → 结构化 → AI 生成 → 人工审核 → 发布分发 → 效果追踪
```

| 阶段 | 具体步骤 | 串/并行 | 质量关卡 |
|------|---------|--------|---------|
| **1. 信息源管理** | 确定权威信息源、建立采集规则、设置更新频率 | 项目初始化时完成 | 来源可靠性评级 |
| **2. 采集清洗** | 爬取/API获取原始数据、去重、格式标准化、错误过滤 | 多源可并行采集 | 数据完整性检查 |
| **3. 结构化** | 标注分类、实体识别、关系提取、知识图谱构建 | 必须在清洗后 | 结构化覆盖率 |
| **4. AI 生成** | 用结构化数据+提示词生成内容、多轮迭代 | 不同主题可并行 | 生成质量评分 |
| **5. 人工审核** | 事实核查、品质审核、合规检查、品牌一致性 | 必须在生成后 | 审核通过率 |
| **6. 发布分发** | 多渠道发布、格式适配、SEO优化 | 多渠道可并行 | 发布成功率 |
| **7. 效果追踪** | 阅读量、互动率、转化率、用户反馈 | 发布后持续 | KPI 达标率 |

### 3.3 信息系统化工作流程（知识工程）

> **来源**：DIKW 模型（Data-Information-Knowledge-Wisdom）；知识管理标准 ISO 30401:2018

```
数据采集 → 信息整理 → 知识提炼 → 系统化 → 验证 → 部署应用
```

| 步骤 | 活动 | 产出 | 关键规则 |
|------|------|------|---------|
| **1. 数据采集** | 多源采集、全面覆盖 | 原始数据集 | 宁多勿少，先不判断 |
| **2. 信息整理** | 去重、分类、标注、关联 | 结构化信息库 | 统一格式和分类标准 |
| **3. 知识提炼** | 提取模式、总结规律、识别因果 | 知识条目 | 多源交叉验证 |
| **4. 系统化** | 构建知识体系、建立索引、设计层级 | 知识体系文档 | 体系完整性检查 |
| **5. 验证** | 专家审核、实际测试、边界验证 | 验证报告 | 找反例证伪 |
| **6. 部署应用** | 整合到工作流、培训、持续更新 | 可用的知识系统 | 使用反馈闭环 |

**严格串行**：每一步依赖前一步的产出，不能跳步。

## 4. 多团队协作的标准编排模式

### 4.1 依赖管理与关键路径

> **来源**：PMI PMBOK Guide 7th Edition；Eliyahu Goldratt《Critical Chain》（关键链理论）

#### 关键路径法（CPM）

**定义**：项目中最长的任务链，决定项目最短完成时间。关键路径上的任何延迟都会导致项目整体延迟。

**识别步骤**：
1. 列出所有任务
2. 确定每个任务的前置依赖
3. 估算每个任务的持续时间
4. 画出网络图
5. 找出最长路径 = 关键路径
6. 关键路径上的任务 = 零浮动（不能延迟）

#### 四种依赖类型

| 依赖类型 | 含义 | 示例 |
|---------|------|------|
| **FS（完成-开始）** | A 完成后 B 才能开始 | 设计完成 → 开发开始 |
| **SS（开始-开始）** | A 开始后 B 可以开始 | 需求分析开始 → 技术调研可以同步开始 |
| **FF（完成-完成）** | A 完成时 B 也应完成 | 开发完成 → 文档也应同步完成 |
| **SF（开始-完成）** | A 开始时 B 应完成（少见）| 新系统上线 → 旧系统应下线 |

#### NDHY AI Agent Team 中的关键路径示例

对于典型产品开发项目：

```
需求分析 → 战略定位 → 产品定义 → 体验设计 → 视觉设计 → 前端开发 → 联调 → 测试 → 发布
    ↑           ↑              ↑                              ↑
  这条是关键路径，任何延迟都导致项目延期
```

并行支线（不在关键路径上，有浮动时间）：
```
                    产品定义 → 技术架构 → 数据库设计 → 后端开发 ─┐
                                  ↓                           ↓
                              API 设计 ─────────────────→ 联调
```

### 4.2 并行条件判断规则

> **来源**：PMI PMBOK；SAFe (Scaled Agile Framework)；Spotify Squad Model

#### ✅ 可以并行的条件（全部满足才可并行）

| 条件 | 说明 | 示例 |
|------|------|------|
| **无数据依赖** | B 不需要 A 的产出作为输入 | 前端 ∥ 后端（基于约定的 API 契约） |
| **无资源冲突** | A 和 B 不争夺相同资源 | 不同专家各自执行 |
| **有明确接口** | 并行双方约定好交互接口 | API 契约已定义 |
| **失败独立** | A 失败不会导致 B 也失败 | 不同模块独立部署 |
| **可独立验收** | A 和 B 各自有独立的验收标准 | 各自有单元测试 |

#### ❌ 必须串行的条件（任一成立就必须串行）

| 条件 | 说明 | 示例 |
|------|------|------|
| **产出-输入依赖** | B 需要 A 的产出 | 视觉设计需要体验设计的线框图 |
| **决策依赖** | B 的执行取决于 A 的决策结果 | 技术选型结果决定开发方式 |
| **共享状态** | A 和 B 修改同一数据/资源 | 同一代码文件的并发修改 |
| **质量门控** | B 的开始需要 A 通过质量检查 | 代码审查通过才能合并 |
| **认知依赖** | 做 B 之前需要从 A 中获取理解 | 需求理解后才能做方案设计 |

#### 🔄 有条件并行（最容易犯错的地方）

| 场景 | 条件 | 正确做法 |
|------|------|---------|
| 体验设计 ∥ 技术架构 | 产品定义已完成 | 两者基于同一产品定义各自工作，通过对齐会议保持一致 |
| 数据库设计 ∥ API 设计 | 技术架构方向已定 | 并行执行，但需要**字段语义对齐**环节 |
| 前端 ∥ 后端 | API 契约已定义 | 基于契约各自开发，前端可用 Mock 数据 |
| 多个 Sprint Team 并行 | 各 Team 的范围无重叠 | 通过 Sprint 同步会议避免冲突 |

### 4.3 阶段门控（Stage Gate）

> **来源**：Robert G. Cooper（1990）"Stage-Gate Process"；PDMA (Product Development and Management Association)

#### Stage Gate 模型

```
创意 →[Gate 1]→ 范围界定 →[Gate 2]→ 商业论证 →[Gate 3]→ 开发 →[Gate 4]→ 测试验证 →[Gate 5]→ 发布
```

每个 Gate 是决策点：**Go / Kill / Hold / Recycle**

#### NDHY AI Agent Team 阶段门控设计

| Gate | 名称 | 准入（谁审核） | 关键检查项 | 决策选项 |
|------|------|--------------|-----------|---------|
| **G0** | 需求门 | Leader | 需求是否清晰？价值是否成立？ | Go / 回炉需求挖掘 |
| **G1** | 方案门 | Leader + 老板 | 产品定义是否完整？体验方案是否合理？技术方案是否可行？ | Go / 调整方案 / Kill |
| **G2** | 设计门 | Leader | 视觉设计是否完成？API 契约是否定义？数据模型是否设计？ | Go / 补齐设计 |
| **G3** | 开发门 | 项目管理专家 | 代码是否通过审查？单元测试是否通过？ | Go / 打回修改 |
| **G4** | 集成门 | 联调集成专家 | 前后端联调是否通过？端到端流程是否跑通？ | Go / 归因修复 |
| **G5** | 发布门 | Leader + 老板 | 系统测试是否通过？发布策略是否就绪？回滚方案是否准备？ | Go / 延期 / Kill |

### 4.4 常见反模式（Anti-Patterns）

> **来源**：Martin Fowler "Refactoring"；Fred Brooks "The Mythical Man-Month"；行业经验总结

| 反模式 | 描述 | 危害 | 正确做法 |
|--------|------|------|---------|
| **🚫 跳过需求直接开发** | 觉得需求很清楚就直接开干 | 做出来不是用户想要的 | 即使看起来清楚，也要过需求验证门 |
| **🚫 没设计就写代码** | 直接开始编码，边写边想 | 架构混乱、返工严重 | 先出架构方案和 API 契约 |
| **🚫 视觉设计先于体验设计** | 先做好看的再考虑好用的 | 好看但不好用 | 必须先完成信息架构和交互设计 |
| **🚫 全部并行** | 把所有任务同时开始 | 互相等待、频繁返工 | 按依赖关系排序，只并行无依赖的 |
| **🚫 全部串行** | 所有任务排成一条线 | 项目周期过长 | 识别无依赖的任务并行执行 |
| **🚫 没有 API 契约就前后端并行** | 前端和后端各做各的 | 联调时发现接口对不上 | 先定义 API 契约，再并行开发 |
| **🚫 跳过代码审查** | 写完就合并 | 质量隐患、技术债务 | 必须经过至少一轮代码审查 |
| **🚫 联调前不做单元测试** | 模块没测试就拿去联调 | 联调问题难以归因 | 各模块先通过自己的测试 |
| **🚫 边开发边改需求** | 需求频繁变更但不走变更流程 | 范围蔓延、永远做不完 | 需求变更走正式流程，评估影响 |
| **🚫 Leader 替代专家做决策** | Leader 拍脑袋决定技术方案 | 外行指挥内行 | 专业的事交给专业的角色 |
| **🚫 没有回滚方案就发布** | 默认发布一定成功 | 出问题后手忙脚乱 | 发布前必须准备回滚方案 |
| **🚫 测试只测正常路径** | 只测 Happy Path | 上线后各种边界异常 | 必须覆盖异常、边界、回归场景 |

## 5. 教育科技行业特有的工作流程

### 5.1 教育内容开发流程（ADDIE 模型）

> **来源**：InstructionalDesign.org ADDIE Model；Florida State University 原始 ISD 模型（1975）

ADDIE 是教育内容开发的**行业标准模型**，被全球教育机构广泛采用。

```
分析（Analysis）→ 设计（Design）→ 开发（Development）→ 实施（Implementation）→ 评估（Evaluation）
```

| 阶段 | 具体步骤 | 产出 | NDHY 对应角色 |
|------|---------|------|--------------|
| **1. 分析** | ①明确学习者特征 ②确定学习目标 ③分析学习约束 ④确定交付方式 ⑤确定项目时间表 | 需求分析报告 | 🔍用户研究专家 + 🎯需求分析专家 |
| **2. 设计** | ①编写学习目标（按认知/情感/动作域）②设计评估工具 ③选择教学策略 ④创建故事板 ⑤设计用户界面和体验 ⑥创建原型 | 教学设计文档 + 故事板 + 原型 | 📐产品定义专家 + 🎨体验设计专家 + 🎨视觉设计专家 |
| **3. 开发** | ①创建内容资产 ②程序开发 ③技术集成 ④内部测试和 Debug | 可运行的学习产品 | ⚙️后端开发专家 + 🖥️前端开发专家 + 📝技术文档专家 |
| **4. 实施** | ①培训引导者/教师 ②准备学习者 ③部署工具和平台 ④确保技术就绪 | 已部署的学习系统 | 📦发布策略专家 + 🚀部署运维专家 |
| **5. 评估** | ①形成性评估（过程中持续进行）②终结性评估（使用效果的量化评估）③收集用户反馈 | 评估报告 + 改进建议 | 📈数据分析专家 + 🔄迭代优化专家 |

**⚠️ 关键特性**：
- **每个步骤的产出都是下一步的输入**（严格串行）
- **评估是贯穿全程的**（形成性评估在每个阶段都有）
- **快速原型法**是 ADDIE 的改进版——在 Design 阶段就开始原型验证，获取早期反馈

### 5.2 教育产品设计流程

> **来源**：ISTE (International Society for Technology in Education) Standards；Learning Engineering 实践

教育产品与普通产品的**关键区别**：
1. **双用户群**：教师（使用者）+ 学生（学习者）
2. **教学目标驱动**：功能设计必须服务于教学目标
3. **学习科学约束**：需要符合认知负荷理论、间隔重复等教育原理
4. **教学日历约束**：开发和发布节奏受学期/假期影响

#### 教育产品全链路设计流程

```
Phase 0: 教育需求调研
  教学痛点调研 ∥ 课程标准分析 ∥ 教师工作流观察
           ↓
Phase 1: 教学设计
  学习目标定义 → 教学策略选择 → 评估方式设计
           ↓
Phase 2: 产品设计
  产品定义（功能范围）→ 体验设计 → 视觉设计
           ↓                  ↓
  技术架构 → 数据模型 + API 设计
           ↓
Phase 3: 内容与开发
  教学内容制作 ∥ 前后端开发（并行）
           ↓
Phase 4: 质量验证
  内容审核（学科专家）∥ 技术测试（测试专家）
           ↓
  联调集成 → 系统测试
           ↓
Phase 5: 试点部署
  种子学校试点 → 教师培训 → 学生使用
           ↓
  收集使用数据 + 教学效果评估
           ↓
Phase 6: 迭代优化
  基于数据优化 → 扩大推广
```

**⚠️ 教育行业特殊约束**：
- **发布窗口**：新学期开始前 2-4 周是最佳发布窗口
- **避免期中发布**：期中考试前后避免重大更新
- **教师培训前置**：发布前必须完成教师培训
- **内容审核不可跳过**：教育内容必须经过学科专家审核

### 5.3 课件/学习材料制作流程

> **来源**：Quality Matters Rubric（高等教育质量标准）；SCORM 标准

```
教学目标确认 → 内容大纲 → 素材收集 → 内容制作 → 技术封装 → 审核测试 → 发布
```

| 步骤 | 具体活动 | 串/并行 | 质量标准 |
|------|---------|--------|---------|
| **1. 教学目标确认** | 确认每个课件的学习目标（可衡量） | 必须最先完成 | 目标可衡量、可评估 |
| **2. 内容大纲** | 知识点拆解、教学顺序安排、时长规划 | 依赖目标确认 | 知识点覆盖完整 |
| **3. 素材收集** | 文本、图片、视频、动画、3D模型、VR素材 | 多类素材可并行 | 素材质量和版权 |
| **4. 内容制作** | 排版、动画制作、视频录制、交互设计 | 不同课件可并行 | 内容准确性 |
| **5. 技术封装** | SCORM/xAPI 封装、平台适配、响应式处理 | 依赖内容完成 | 技术规范合规 |
| **6. 审核测试** | 学科审核 + 技术测试 + 用户体验测试 | 三类审核可并行 | 全部通过 |
| **7. 发布** | 上传到 LMS、配置权限、通知用户 | 依赖审核通过 | 发布检查清单 |

## 7. Leader 常犯错误速查表

> 这是本调研最核心的输出——直接针对 Leader 调度决策的实操指南。

### 7.1 顺序错误速查

| 错误 | 正确做法 | 违反的依赖 |
|------|---------|-----------|
| 产品定义还没做就开始体验设计 | 产品定义 → 体验设计 | 产出-输入依赖 |
| 体验设计还没做就开始视觉设计 | 体验设计 → 视觉设计 | 产出-输入依赖 |
| 技术架构还没做就开始数据库设计 | 技术架构 → 数据库设计 | 决策依赖 |
| API 契约还没定义就开始前后端开发 | API 设计 → 前后端开发 | 接口依赖 |
| 前后端还没联调就开始系统测试 | 联调集成 → 系统测试 | 质量门控依赖 |
| 代码审查没通过就合并代码 | 代码审查 → 合并 | 质量门控依赖 |
| 需求还没清楚就开始方案设计 | 需求分析 → 方案设计 | 认知依赖 |
| 没有回滚方案就发布上线 | 部署准备（含回滚）→ 发布 | 安全依赖 |

### 7.2 并行错误速查

| 错误 | 原因 | 正确做法 |
|------|------|---------|
| 所有 Phase 0 调研做完才开始 Phase 1 | 过度串行 | 用户研究、市场评估、竞品分析可并行 |
| 所有设计做完才开始所有开发 | 过度串行 | 模块级可以"滚动交付"——A 模块设计完就开始 A 的开发 |
| 前端和后端串行开发 | 过度串行 | 有 API 契约后可并行，前端用 Mock |
| 体验设计和技术架构串行 | 过度串行 | 产品定义完成后两者可并行 |
| 体验设计和视觉设计并行 | 过度并行 | 必须串行——视觉需要体验设计的交互骨架 |
| 数据库设计和后端开发并行 | 过度并行 | 后端的 Repository 层依赖数据模型 |
| 联调和系统测试并行 | 过度并行 | 必须串行——联调不通测不了功能 |

### 7.3 Leader 调度决策流程图

```
收到任务
    ↓
【1】需求是否清晰？
    ├─ 否 → spawn 需求分析专家深挖 → 等确认 → 继续
    └─ 是 ↓
【2】判断规模
    ├─ 简单（单专家）→ 直接 spawn 对应专家
    ├─ 中等（2+专家）→ 步骤 3
    └─ 重大（组织级）→ 步骤 3，Leader 全程主导
        ↓
【3】spawn 流程设计专家
    - 输入：需求包 + 产出物列表
    - 产出：角色清单 + 协作拓扑 + 执行流程
        ↓
【4】spawn 项目管理专家
    - 输入：流程设计产出 + 验收标准
    - 由项目管理专家负责：任务拆解、排期、依赖管理、调度执行
        ↓
【5】项目管理专家按流程调度执行
    - 遵循依赖关系
    - 串并行按规则判断
    - 每个阶段设 Gate 检查
        ↓
【6】Leader 在里程碑节点验收
    - S级全审，A级审关键点，B/C级抽检
    - 有问题才介入
```

### 7.4 三条铁律

1. **不理解就不设计，不设计就不开发，不测试就不发布。**
   - 每一步都依赖前一步的产出，不能跳步。

2. **有契约才能并行，无契约必须串行。**
   - 前后端并行的前提是 API 契约已定义。
   - 数据库设计和 API 设计并行的前提是技术架构方向已定。

3. **专业的事交给专业的角色，Leader 只管"做什么"和"做到什么标准"。**
   - 执行顺序由流程设计专家设计。
   - 任务拆解由项目管理专家执行。
   - Leader 不拍脑袋决定执行顺序。
