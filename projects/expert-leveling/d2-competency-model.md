## 一、通用维度总览

通用维度占总评估权重的 **70%**，适用于所有 27 位专家，不分领域。

| # | 维度 | 权重 | 一句话定义 |
|---|------|------|-----------|
| C1 | 产出完整度 | 20% | 交付物覆盖了多少该覆盖的内容，遗漏了什么 |
| C2 | 任务复杂度承接 | 15% | 能独立处理多复杂的任务——从明确指令到模糊需求到未定义问题 |
| C3 | 自主决策质量 | 15% | 在没有上游明确指定的地方，自己做的选择是否合理、有据、经得起审查 |
| C4 | 产出可消费性 | 10% | 下游（其他专家/Leader/老板）拿到产出后能否直接使用，还是需要二次加工 |
| C5 | 影响力半径 | 10% | 产出的影响范围——仅限当前任务，还是被其他项目/专家/组织复用 |

**为什么是这 5 个维度：**
- C1（完整度）和 C4（可消费性）覆盖"做没做完、能不能用"——产出的基本面
- C2（复杂度）和 C3（决策质量）覆盖"能做多难的活、做的选择好不好"——能力的天花板
- C5（影响力）覆盖"产出的价值辐射范围"——区分高段位专家的关键维度

**为什么不是 6 个：** 曾考虑加入"协作贡献"维度，但 AI 专家通过 spawn 独立执行，协作主要体现为"产出被他人引用"（已被 C5 覆盖）和"对上下游的适配"（已被 C4 覆盖）。独立维度会导致重复计分。

## 二、各维度 P0-P12 分级标准

### C1 产出完整度（20%）

> 定义：交付物对任务需求的覆盖程度，包括主路径、边界情况、异常处理、自检证据。

| 等级 | 行为描述（可检验） |
|------|-------------------|
| **P0** | 产出仅覆盖任务指令中显式提及的主路径，覆盖度 ≤60%；存在 ≥2 处关键遗漏需他人补全后才可用 |
| **P1** | 产出覆盖指令显式提及的全部要求，覆盖度 ≥80%；无关键遗漏，但指令未提及的边界情况未处理 |
| **P2** | 产出覆盖指令要求 + 常见边界情况（如空值、异常输入、权限缺失），无需他人审查即可直接使用 |
| **P3** | 产出附带自检清单，逐项列出已覆盖的主路径/边界/异常，覆盖度经自检达同领域标准 ≥90%；附下游消费说明 |
| **P4** | 面对模糊任务，产出的覆盖范围由自主拆解决定（非上游指定），拆解后的覆盖度与明确指令下差异 ≤20% |
| **P5** | 产出中对"为什么覆盖这些、不覆盖那些"有显式说明，边界判断有依据（引用需求文档/行业惯例/数据） |
| **P6** | 产出不仅覆盖当前任务需求，还预判并覆盖了下一步可能需要的扩展点（≥1 个），且预判被后续任务验证为有效 |
| **P7** | 产出中包含可复用的完整度检查标准/模板，该标准被本领域 ≥2 次任务采用 |
| **P8** | 产出的完整度标准覆盖 ≥2 个领域的交叉地带（如前后端接口覆盖、设计与开发的状态覆盖） |
| **P9** | 产出定义了端到端链路的完整度体系——从输入到交付的每个环节都有完整度检查点，且体系经 ≥1 个完整项目验证 |
| **P10** | 该完整度体系被全组织采纳为默认检查流程，采纳后产出遗漏率较采纳前下降 ≥30% |
| **P11** | 完整度方法论超越本组织，具备行业通用性——可被其他 AI Agent 组织直接复用 |
| **P12** | 提出全新的完整度范式（在此之前行业无同类方法），被行业作为标准参考 |

### C2 任务复杂度承接（15%）

> 定义：能独立处理的任务复杂度上限——从结构化指令到模糊需求到未定义问题。

| 等级 | 行为描述（可检验） |
|------|-------------------|
| **P0** | 仅能处理结构化指令（输入/输出/步骤均明确），遇到指令歧义时停滞或产出偏离 |
| **P1** | 能处理结构化指令且产出稳定，对指令中的小幅歧义（如用词不精确）能正确推断意图 |
| **P2** | 能处理包含 1-2 个未说明边界条件的任务，自主补全常见场景的处理方式 |
| **P3** | 能处理标准复杂度任务（主路径 + 边界 + 异常均需自主覆盖），产出质量稳定 |
| **P4** | 能处理模糊任务——仅给出目标和约束，自主完成任务拆解、步骤规划、产出交付 |
| **P5** | 能处理含多个决策点的复杂任务，每个决策点有替代方案对比和选择理由 |
| **P6** | 能处理需要前瞻性判断的任务——当前需求之外，识别并处理潜在的未来需求 |
| **P7** | 能处理需要制定标准的任务——从多次具体实践中抽象出可复用的规则/模板/规范 |
| **P8** | 能处理跨 ≥2 个专业领域的交叉任务，识别领域间的矛盾并产出整合方案 |
| **P9** | 能处理端到端体系构建任务——产出包含 ≥3 个协调运转的子系统，覆盖完整业务链路 |
| **P10** | 能处理组织级变革任务——产出改变组织的默认工作方式，可测量效率/质量提升 ≥30% |
| **P11** | 能处理行业级命题——产出方法论/框架可被其他组织直接复用 |
| **P12** | 能处理开创性命题——产出定义新范式，之前行业无同类方法 |

### C3 自主决策质量（15%）

> 定义：在任务中需要自主做选择的地方（上游未指定），决策的合理性、可追溯性和风险预判。

| 等级 | 行为描述（可检验） |
|------|-------------------|
| **P0** | 未做任何自主决策——完全按指令执行；遇到需要选择的地方跳过或随机选择，无说明 |
| **P1** | 对简单二选一做出选择（如命名、格式），选择结果合理但无理由说明 |
| **P2** | 对常见场景的决策（如异常处理策略、默认值选择）做出合理选择，选择符合行业惯例 |
| **P3** | 所有自主决策均附一句话理由，理由可追溯到指令要求或行业惯例 |
| **P4** | 面对模糊任务中的关键决策点，能列出 ≥2 个选项并选择其一，附选择依据 |
| **P5** | 关键决策附完整的替代方案对比（≥2 方案的优劣分析），选择理由引用数据/标准/先例 |
| **P6** | 决策中包含风险预判——标注所选方案的潜在风险和缓解措施，或标注"如果 X 条件变化则需重新评估" |
| **P7** | 从多次决策实践中提炼出可复用的决策框架/标准，被 ≥2 次任务采用 |
| **P8** | 决策覆盖跨领域权衡——在 ≥2 个领域的约束间找到平衡点，各方约束均被显式纳入考量 |
| **P9** | 构建系统性的决策体系——覆盖完整链路中各环节的决策标准，形成可查阅的决策手册 |
| **P10** | 决策体系被全组织采纳，采纳后决策一致性提升（同类问题不同专家的决策趋同率 ≥80%） |
| **P11** | 决策方法论具备行业通用性，可被其他组织复用 |
| **P12** | 提出全新的决策范式，行业之前无同类方法 |

### C4 产出可消费性（10%）

> 定义：下游消费者（其他专家/Leader/老板）拿到产出后的可用程度——能否直接消费，还是需要二次加工。

| 等级 | 行为描述（可检验） |
|------|-------------------|
| **P0** | 产出需要下游 ≥2 处补充/修正后才可使用；格式或结构不符合下游预期 |
| **P1** | 产出格式正确、结构完整，下游可读懂，但部分内容需要下游自行推断或补全 |
| **P2** | 产出可直接使用，无需下游补全；包含必要的上下文说明（如术语解释、前置条件） |
| **P3** | 产出附下游消费指南——说明"谁会用、怎么用、注意什么"，下游零推断成本 |
| **P4** | 产出的结构和组织方式适配下游的消费习惯（如给开发看的文档有代码示例，给老板看的有摘要） |
| **P5** | 产出中标注了下游可能的疑问点并预先解答；关键决策附上下文，下游不需要追问 |
| **P6** | 产出预留了下游扩展的接口/接入点，下游在产出基础上继续工作时无需重构 |
| **P7** | 产出包含面向下游的标准化接口规范/模板，被 ≥2 个下游消费者采用 |
| **P8** | 产出同时适配 ≥2 类不同下游消费者（如同一份技术方案同时服务开发和测试），各方均无需二次加工 |
| **P9** | 构建端到端的产出交接体系，链路上每个节点的输入/输出格式标准化，全链路交接零损耗 |
| **P10** | 交接体系被全组织采纳为默认流程，采纳后跨专家协作的返工率下降 ≥30% |
| **P11** | 产出交接方法论具备行业通用性 |
| **P12** | 提出全新的产出交接范式，行业之前无同类方法 |

### C5 影响力半径（10%）

> 定义：产出的价值辐射范围——从仅完成当前任务，到被跨项目/跨领域/跨组织复用。

| 等级 | 行为描述（可检验） |
|------|-------------------|
| **P0** | 产出仅服务当前任务，且不完整（需他人补全） |
| **P1** | 产出完整服务当前任务，但仅限本任务使用，无复用价值 |
| **P2** | 产出中包含可独立使用的片段（如一个工具函数、一个分析模板），但未被实际复用 |
| **P3** | 产出被同一项目内的 ≥1 个后续任务引用或复用 |
| **P4** | 产出被 ≥1 个其他专家作为输入引用（非同任务链的主动引用） |
| **P5** | 产出被其他项目的 ≥1 个任务引用或复用 |
| **P6** | 产出被 ≥2 个不同专家/项目直接作为输入引用 |
| **P7** | 产出从"交付物"进化为"可复用资产"——标准/模板/规范被 ≥2 个项目采用 |
| **P8** | 产出影响 ≥2 个专业领域的工作方式 |
| **P9** | 产出构成覆盖完整业务链路的体系，影响 ≥3 个子系统 |
| **P10** | 产出改变全组织的默认实践，全员遵循 |
| **P11** | 产出超越组织边界，具备行业通用性 |
| **P12** | 产出开创行业新范式 |

## 三、评估打分方法

### 3.1 打分规则

**每个维度独立打分，分数 = 该维度上专家稳定达到的等级编号（P0-P12 → 0-12 分）。**

判定规则：
- **"稳定达到"的定义**：最近 5 次该维度有效评估中，≥3 次达到该级别标准 = 稳定达到
- **不足 5 次记录时**：取全部记录，≥60% 达到该级别标准 = 稳定达到
- **仅 1 次记录时**：按该次实际表现打分，标注"单次评估，待积累更多数据"

### 3.2 从维度分数到等级判断

**步骤 1：计算加权总分**

```
加权总分 = C1分数×0.20 + C2分数×0.15 + C3分数×0.15 + C4分数×0.10 + C5分数×0.10
```

注：以上为通用维度部分（70%）。领域专属维度（30%）在 D3 中定义，最终公式为：

```
综合分数 = 通用加权分 + 领域加权分
```

**步骤 2：确定等级**

- **等级 = 综合分数向下取整**（如综合分数 4.6 → P4）
- **附加约束（地板规则）**：任何单一维度的分数不得低于目标等级 -2。即：若目标为 P5，则每个维度 ≥P3。违反此规则时，等级 = min(各维度分数) + 2

### 3.3 维度间不均衡处理

**短板约束机制：**

| 情况 | 处理规则 |
|------|---------|
| 最高维度与最低维度差距 ≤2 级 | 正常，按加权总分定级 |
| 差距 3 级 | 等级判定 = 加权总分，但评估记录中标注"短板警告：[维度名] 偏低"，下次评估重点关注 |
| 差距 ≥4 级 | 触发**地板规则**：等级 = min(各维度分数) + 2，不论加权总分多高。评估记录中标注"短板限制：[维度名] 拖低定级，需针对性提升" |

**设计理由**：AI 专家的通用能力应均衡发展。某维度极强但另一维度极弱 = 产出存在系统性盲区，不应以高分维度掩盖低分维度。地板规则确保等级反映的是"全面可靠的产出能力"而非"偏科的最高水平"。

| 维度 | 本次表现等级 | 判定依据（≤50 字） |

| C4 可消费性 | P? | [具体证据] |

（D3 定义后补充）

## 五、体系自检

| 检查项 | 结果 |
|--------|------|
| 每个维度每级描述是否可检验（无"较好""基本掌握"等模糊词） | ✅ 全部使用"产出具备什么特征""能做到什么"描述 |
| 维度定义是否适用于全部 27 位专家（跨领域通用） | ✅ 5 个维度均描述产出的通用属性，不涉及领域专属技能 |
| 各维度的 P 级描述是否与 D1 等级定义对齐 | ✅ P0-P3=执行者特征、P4-P6=解决者特征、P7-P9=引领者特征、P10-P12=奠基者特征 |
| 评估方法是否 Leader（AI）可直接操作 | ✅ 规则化打分 + 公式化汇总 + 模板化记录，无"视情况而定" |
| 维度间不均衡是否有明确处理规则 | ✅ 3 级差距警告、≥4 级地板规则 |
| 评估记录模板是否 token 高效 | ✅ 单次记录约 300-500 字，汇总约 500-800 字 |
| 是否侵入领域专属维度（30%）的设计范围 | ✅ 未侵入，已预留接口 |
| 是否侵入晋升流程（Phase 2）的设计范围 | ✅ 仅定义"怎么评"，未涉及"怎么升" |

---

_本文档定义了"用什么尺子量、每级量出来什么样、怎么打分、怎么记录"。领域专属维度（30%）在 D3 中展开，晋升流程在 Phase 2 中设计。_
