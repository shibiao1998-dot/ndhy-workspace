## 一、评估触发机制

### 1.1 定期评审

- **周期**：每自然月最后一周
- **纳入范围**：当月有 ≥3 次任务产出记录的专家。不足 3 次的跳过本期，累积到下月
- **执行者**：Leader
- **操作**：遍历符合条件的专家，按第二节步骤逐一执行综合评估

### 1.2 里程碑触发

**触发条件（满足任一即触发）**：
1. 专家在单个项目中完成 ≥3 个关键任务（指令包中标注为"关键"的任务）
2. 专家首次承接跨段复杂度的任务且产出通过质量检查（如当前 P3 的专家承接了 P4+ 复杂度的任务）
3. 专家产出被 ≥2 个其他专家/项目引用复用（影响力半径跃迁信号）

**触发者**：Leader 在质量检查环节自动识别。不依赖专家自我触发。

### 1.3 优先级与冲突处理

| 场景 | 处理规则 |
|------|---------|
| 里程碑触发发生在定期评审前 ≤7 天 | 合并到定期评审一起做，不单独评 |
| 里程碑触发发生在定期评审后立即 | 正常执行里程碑评估（刚做完定期评审不影响） |
| 同一专家在一个月内触发多次里程碑评估 | 只执行第一次触发的评估，后续触发的记录归入下月定期评审 |
| 定期评审和里程碑评估结论不一致 | 以数据更充分的那次为准——比较两次评估涵盖的产出记录数，取记录数多的结论 |

## 二、单次评估操作步骤

> Leader 逐步执行以下清单。每步有明确的输入、操作和输出。

### Step 1：收集产出记录

**输入**：专家名称、评估周期起止日期

**操作**：
1. 读取 `projects/expert-leveling/eval-records/[专家英文ID]/` 目录下该周期内的所有单次评估记录
2. 统计有效记录数 N
3. 若 N < 3 且为定期评审 → 输出"记录不足，跳过本期"，终止
4. 若 N < 1 → 输出"无记录，跳过"，终止

**输出**：有效记录列表、记录数 N

### Step 2：按通用维度（D2）逐维度打分

**输入**：Step 1 的有效记录列表

**操作**：对 C1-C5 每个维度，按 D2 §3.1 打分规则判定稳定水平——

| 记录数 | 判定规则 |
|--------|---------|
| ≥5 次 | 最近 5 次中 ≥3 次达到该级别 = 稳定达到 |
| 3-4 次 | ≥60% 达到该级别 = 稳定达到 |
| 1-2 次 | 按实际表现打分，标注"数据不足，待积累" |

逐维度记录：维度名 → 各次评分 → 稳定水平 → 趋势（↑/→/↓）

**输出**：C1-C5 各维度稳定水平表

### Step 3：按领域专属维度（D3）打分

**输入**：Step 1 的有效记录列表 + 该专家的 D3 领域专属标准卡片

**操作**：
1. 确认该专家的领域专属维度数量（1 或 2）和各维度权重
2. 对每个领域维度，按与 Step 2 相同的稳定水平判定规则打分
3. 若该专家尚无 D3 标准卡片 → 标注"领域标准未定义，仅以通用维度评估"，领域加权分按 0 计算，总权重归一化到通用维度

**输出**：D1（及 D2 如有）各维度稳定水平表

### Step 4：汇总计算综合等级

**输入**：Step 2 + Step 3 的维度稳定水平

**操作**：

**4a. 计算加权总分**

```
综合分数 = C1×0.20 + C2×0.15 + C3×0.15 + C4×0.10 + C5×0.10 + 领域维度加权分
```

领域加权分规则：
- 1 个专属维度：D1×0.30
- 2 个专属维度：D1×权重 + D2×权重（权重之和 = 0.30）
- 无专属标准时：通用维度权重按比例放大至 100%（即各维度分数 ÷ 0.70）

**4b. 向下取整**

等级 = floor(综合分数)

**4c. 地板规则检查**

检查所有维度（通用 + 领域）的分数，取最低值 min_score：
- 若 min_score < 等级 - 2 → 触发地板规则，等级 = min_score + 2
- 若 min_score ≥ 等级 - 2 → 通过

**4d. 短板标记**

- 最高维度与最低维度差距 = 3 → 标注"短板警告：[维度名]偏低"
- 差距 ≥ 4 → 地板规则已自动处理，标注"短板限制：[维度名]拖低定级"

**输出**：综合分数、判定等级、地板规则检查结果、短板标记

### Step 5：与当前等级对比，判断是否触发晋升

**输入**：Step 4 的判定等级 + 专家当前等级（读取 SOUL.md 中的等级标识）

**操作**：

| 对比结果 | 判定 | 后续动作 |
|---------|------|---------|
| 判定等级 > 当前等级 | **建议晋升** | 进入第三节晋升审批流程 |
| 判定等级 = 当前等级 | **维持** | 记录本次评估结果，归档 |
| 判定等级 < 当前等级 | **标注观察** | 不立即降级。标注"表现下滑，连续 2 次评估低于当前等级则启动降级评审"。降级评审流程与晋升审批流程相同（Leader 提案 → 老板决策） |

**跨段晋升额外检查**：若判定等级跨段（如 P3→P4、P6→P7、P9→P10），必须在评估报告中附加 D1 §三 中对应的段间跃迁验证——逐条对照跃迁标准，全部满足才可提议跨段晋升。

**输出**：晋升/维持/观察 判定

### Step 6：生成评估报告

**输入**：Step 1-5 全部输出

**操作**：按 D2 §4.3 汇总评估模板填写，存入 `projects/expert-leveling/eval-records/[专家英文ID]/[YYYY-MM]-summary.md`

**模板**：

```markdown
## 等级汇总评估

- 专家：[emoji + 角色名]
- 评估周期：[起止日期]
- 评估次数：[N 次]
- 当前等级：P?
- 触发方式：[定期评审 / 里程碑触发]

### 各维度稳定水平

| 维度 | 近期评分 | 稳定水平 | 趋势 |
|------|---------|---------|------|
| C1 产出完整度 | P?, P?, ... | P? | ↑/→/↓ |
| C2 复杂度承接 | ... | P? | ... |
| C3 决策质量 | ... | P? | ... |
| C4 可消费性 | ... | P? | ... |
| C5 影响力半径 | ... | P? | ... |
| D1 [领域维度名] | ... | P? | ... |
| D2 [领域维度名]（如有） | ... | P? | ... |

### 综合等级计算

- 通用加权分：[计算过程]
- 领域加权分：[计算过程]
- 综合分数：[X.XX]
- 地板规则检查：[通过 / 触发（说明）]
- 短板标记：[无 / 警告 / 限制]
- **判定等级：P?**

### 结论

- 对比当前等级：[晋升 / 维持 / 观察]
- 跨段验证：[不适用 / 通过 / 未通过（说明）]
- 后续动作：[进入晋升审批 / 归档 / 列入观察清单]
```

## 三、晋升审批流程

### 3.1 Leader 生成晋升提案

当 Step 5 判定"建议晋升"时，Leader 按以下格式生成晋升提案：

```markdown
## 晋升提案

### 基本信息
- 专家：[emoji + 角色名]
- 当前等级：P? → 提议等级：P?
- 是否跨段：[是（[段名]→[段名]）/ 否]
- 评估周期：[起止日期]
- 评估记录数：[N 次]

### 晋升理由（≤200 字）
[用老板能理解的语言概括：这个专家做了什么、达到了什么水平、为什么该升级]

### 关键证据（≤3 条）
1. [最能说明等级提升的产出事实，附项目名和任务名]
2. [第二条证据]
3. [第三条证据]（可选）

### 综合评分摘要
| 维度 | 稳定水平 |
|------|---------|
| C1-C5 + D维度 | P? |
| 综合分数 | X.XX → P? |

### 晋升后变化
- 设定更新：[SOUL.md 增加等级标识 / STANDARDS.md 增加新标准条目]
- 具体变化：[等级提升后设定中新增/修改什么内容]

### Leader 建议
[通过 / 暂缓（附理由）]
```

**关键要求**：
- "晋升理由"必须用非专业人士能理解的语言，不写"C2 维度稳定 P5"，写"连续 3 次独立拆解模糊任务并高质量交付"
- "关键证据"必须指向具体的项目和任务，不写"多次表现优异"
- 整个提案控制在 **500 字以内**

### 3.2 老板审批

**提交方式**：Leader 将晋升提案发送给老板（私聊消息）

**老板的决策选项**：
- ✅ **批准**：进入 3.3 执行步骤
- ❌ **驳回**：进入 3.4 驳回处理
- 🟡 **追问**：老板要求补充信息 → Leader 补充后重新提交

### 3.3 批准后执行步骤

Leader 按以下清单逐步执行：

1. **更新 SOUL.md**：在专家 SOUL.md 的身份章节增加等级标识，格式：`**当前等级**：P[N]（[等级名称]）`
2. **更新 STANDARDS.md**（跨段晋升时）：按 D1 §4.4 的说明，增加新段位对应的标准章节。例如 P3→P4 需增加"模糊任务处理规范"章节
3. **归档晋升记录**：将晋升提案 + 老板批准记录存入 `projects/expert-leveling/eval-records/[专家英文ID]/promotions/[YYYY-MM-DD]-P[旧]toP[新].md`
4. **记录到每日记忆**：在 `memory/YYYY-MM-DD.md` 中记录晋升事件
5. **通知老板确认**：执行完毕后私聊老板确认"[专家名] 已从 P[旧] 晋升至 P[新]，设定已更新"

### 3.4 驳回后处理

1. **记录驳回理由**：在评估报告中附加"老板驳回，理由：[原文]"
2. **标注改进方向**：根据驳回理由提炼改进点，写入评估报告的"后续动作"
3. **冷却期**：被驳回的专家在 **下一个完整评估周期**（即下个月）之前不再提交晋升提案
4. **归档**：驳回的提案存入 `eval-records/[专家英文ID]/rejected/[YYYY-MM-DD].md`

## 四、产出记录规范

### 4.1 记录时机

**每次任务完成后**，Leader 在质量检查环节同步完成产出评估记录。不单独设评估环节——评估是质量检查的一部分。

### 4.2 记录存放位置

```
projects/expert-leveling/eval-records/
  ├── [专家英文ID]/
  │   ├── [YYYY-MM-DD]-[任务简称].md    # 单次评估记录
  │   ├── [YYYY-MM]-summary.md           # 月度汇总评估
  │   ├── promotions/                     # 晋升记录
  │   │   └── [YYYY-MM-DD]-P?toP?.md
  │   └── rejected/                       # 驳回记录
  │       └── [YYYY-MM-DD].md
  └── ...
```

### 4.3 单次评估记录模板

> 与 D2 §4.2 对齐，Leader 每次任务质量检查后填写。

```markdown
## 评估记录

- 专家：[emoji + 角色名]
- 任务：[任务简述，≤30 字]
- 日期：[YYYY-MM-DD]
- 项目：[项目名]

### 通用维度评分

| 维度 | 本次表现等级 | 判定依据（≤50 字） |
|------|------------|-------------------|

| C2 复杂度承接 | P? | [具体证据] |
| C3 决策质量 | P? | [具体证据] |
| C4 可消费性 | P? | [具体证据] |
| C5 影响力半径 | P? | [具体证据] |

### 领域维度评分

| 维度 | 本次表现等级 | 判定依据（≤50 字） |
|------|------------|-------------------|
| D1 [维度名] | P? | [具体证据] |
| D2 [维度名]（如有） | P? | [具体证据] |

### 短板标记
- 维度间差距 ≥3 级：[是/否]，涉及维度：[...]
```

## 五、特殊场景处理

### 5.1 新创建的专家

| 阶段 | 规则 |
|------|------|
| 创建时 | SOUL.md 中标注 `**当前等级**：P0（新手）`。无评估记录 |
| 首次被调用 | Leader 正常记录产出评估。标注"首次任务" |
| 何时可参与晋升评估 | 累积 ≥3 次评估记录后，纳入下一次定期评审或符合里程碑触发条件时评估 |
| 首次评估的特殊处理 | 评估报告标注"初始评估，数据样本量小，结论待后续验证"。如判定 ≥P2，仍需提交老板审批（跳级幅度大，需确认） |

### 5.2 长期未被调用的专家

**定义**："长期未调用"= 连续 **60 天**无任务产出记录。

| 阶段 | 规则 |
|------|------|
| 30 天无记录 | Leader 在月度评审中标注"低活跃度"，不做其他处理 |
| 60 天无记录 | 标注"休眠状态"。等级保持不变，不参与评审。下次被调用时按正常流程恢复评估 |
| 恢复后 | 首次任务产出按正常流程记录评估。不因休眠降级——等级只能通过评估证据变更，不因时间流逝自动调整 |

**设计理由**：AI 专家是无状态的，不存在"技能退化"。休眠只说明该领域近期没有需求，不代表专家能力下降。等级锚定在设定文件（SOUL.md/STANDARDS.md），设定不变则能力不变。

### 5.3 同一周期内多次触发晋升条件

| 场景 | 处理规则 |
|------|---------|
| 一次评估判定跳升 ≥2 级（如 P2→P5） | **不允许跳级 ≥2 级**。最多晋升至当前等级 +1（段内）或 +1 跨段。理由：跳级过大缺乏中间验证数据，风险高。超出部分在下一评估周期处理 |
| 定期评审 + 里程碑在同月都触发晋升 | 执行一次晋升（取先触发的），另一次记录归档，可作为下月评估的提前证据 |
| 晋升被老板批准后，同月又触发新的里程碑 | 记录归档。新等级需至少积累 **3 次新等级下的评估记录**后，才可再次触发晋升评估。防止连续快速晋升导致等级通胀 |

**等级通胀防范**：每次晋升后设置 **最少观察期**——在新等级下积累 ≥3 次评估记录后才可再次晋升。这意味着即使表现极好，两次晋升之间至少间隔 3 次任务。

## 七、体系自检

| 检查项 | 结果 |
|--------|------|
| 评估步骤是否 Leader 可逐步执行（无"视情况而定"） | ✅ 6 步清单，每步有输入/操作/输出 |
| 打分方法是否与 D2 §3 严格一致 | ✅ 使用相同的稳定水平判定规则、地板规则、短板机制 |
| 综合分数计算是否与 D3 §4.1 严格一致 | ✅ 使用相同的加权公式和向下取整规则 |
| 维度名称是否与 D2/D3 一致（C1-C5, D1-D2） | ✅ 未创造新维度名 |
| 晋升提案是否老板可快速理解 | ✅ ≤500 字，用非专业语言，有关键证据 |
| 产出记录模板是否与 D2 §4.2 对齐 | ✅ 直接复用 D2 模板，增加领域维度行 |
| 特殊场景是否有明确规则（无模糊地带） | ✅ 新专家/休眠/多次触发均有具体数字门槛 |
| 是否有等级通胀防范机制 | ✅ 跳级限制 + 最少观察期 |
| 是否适配 AI 无状态特性 | ✅ 全部基于外部记录，不依赖专家自我汇报 |
