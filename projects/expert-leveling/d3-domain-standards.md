## 一、领域专属维度设计原则

### 1.1 什么算"领域专属"

**判定三条件（必须同时满足）**：
1. **C1-C5 不可覆盖**：该能力维度无法被通用维度的任何一个充分评估（否则重复计分）
2. **领域间有差异**：前端开发和需求分析在此维度上的表现形式/评估标准截然不同（否则应归入通用维度）
3. **产出可检验**：能从产出物中直接观测到该维度的水平差异（适配 AI 无状态评估）

**与通用维度的边界**：

| 通用维度覆盖的 | 领域专属维度覆盖的 |
|---------------|-------------------|
| 产出覆盖了多少（C1） | 产出的**专业深度和技艺水准**——同样完整的产出，专业质量可以天差地别 |
| 能做多复杂的任务（C2） | 该领域**独有的复杂性类型**——每个领域的"难"不一样 |
| 决策质量（C3） | 该领域**独有的评判标准**——什么算"好"由领域定义 |
| 下游可消费（C4） | 该领域**独有的交付形态**——代码要能跑、设计要能还原 |
| 影响力范围（C5） | 该领域**独有的专业贡献形式**——标准制定的内容不同 |

**核心区分**：通用维度量"做了多少、做到多难、影响多广"，领域专属维度量"在这个领域做得多专业"。一个 P4 前端开发专家和 P4 需求分析专家在通用维度上可比（都能自主拆解模糊任务），但"代码质量"和"需求洞察深度"是各自领域才有的专业标尺。

### 1.2 每个领域的专属维度数量与权重

- 每个领域 **1-2 个专属维度**（控制评估复杂度）
- 1 个维度时权重 = 30%；2 个维度时默认 15%+15%（可按领域特性调整为 20%+10%，需附理由）
- 所有领域的专属维度权重总和 = 30%，与通用维度（70%）不重叠

### 1.3 专属维度命名规范

格式：`D[序号] [领域缩写]-[维度名]`
示例：`D1 FE-代码工程质量`、`D1 RA-需求洞察深度`

## 二、通用模板

Leader 或组织设计专家为任意新领域填写专属标准时，直接复制以下模板并填写。

### 2.1 领域专属标准卡片模板

```markdown
## [emoji] [角色名] 领域专属标准

### 基本信息
- 角色：[角色名]
- 英文 ID：[english-id]
- 专属维度数量：[1 或 2]

### 维度 D1：[维度名称]
- **权重**：[15% 或 30%，取决于维度数量]
- **定义**：[一句话，说明这个维度量什么。≤40 字]
- **与通用维度的边界**：[说明为什么 C1-C5 覆盖不了。≤50 字]

#### 分级标准

| 等级 | 行为描述（可检验） |
|------|-------------------|
| P0 | [该维度上最低水平的产出特征] |
| P1 | [...] |
| P2 | [...] |
| P3 | [...] |
| P4 | [...] |
| P5 | [...] |
| P6 | [...] |
| P7 | [...] |
| P8 | [...] |
| P9 | [...] |
| P10 | [...] |
| P11 | [...] |
| P12 | [...] |

#### 评估方法
- **评估素材**：[从产出物中看什么来判断等级]
- **评估步骤**：[1-2-3 步骤，Leader 可直接操作]
- **易混点**：[容易和通用维度搞混的地方，提醒区分]

### 维度 D2：[维度名称]（如只有 1 个维度则删除本节）
（同上结构）
```

### 2.2 填写指南——怎么写才"可检验、不模糊"

| 规则 | ✅ 正确 | ❌ 错误 |
|------|---------|---------|
| **用产出特征描述，不用能力形容词** | "代码通过 ESLint 零 error" | "代码质量较高" |
| **用可计数指标** | "覆盖 ≥4 种组件状态" | "覆盖了大多数状态" |
| **每级只描述该级新增的特征** | "P3 新增：附自检清单" | "P3：做得比 P2 更好" |
| **相邻等级有明确跃迁点** | P2→P3 的差异是"从无自检到有自检" | P2 和 P3 描述高度相似 |
| **段间跃迁呼应 D1** | P4 描述体现"自主拆解"（D1 中级段特征） | P4 仍在描述"执行指令" |
| **不与 C1-C5 重复** | "组件状态覆盖"（专业技艺） | "产出覆盖度高"（已被 C1 覆盖） |

**自检三问**（每条描述写完后过一遍）：
1. Leader（AI）看到一份产出物，能否不借助主观判断就判定达到/未达到此级？
2. 这条描述是否在 C1-C5 中已经有等价表述？（有 → 删）
3. 拿掉这条描述，该领域和其他领域的评估还有区别吗？（无区别 → 这就是领域专属的核心）

## 三、示例领域标准

### 3.1 🖥️ 前端开发专家（开发类代表）

**专属维度数量**：2（代码是开发类最核心的产出形态，需要拆细）

#### 维度 D1 FE-代码工程质量（15%）
- **定义**：代码产出在类型安全、规范一致、状态完备、可维护性方面的技艺水准
- **与通用维度的边界**：C1 量"覆盖了多少功能"，D1 量"代码本身写得多专业"——功能完整但代码烂 vs 代码精良

| 等级 | 行为描述（可检验） |
|------|-------------------|
| **P0** | 代码有语法/类型错误，无法通过 lint 检查；组件仅覆盖 happy path |
| **P1** | 代码通过 lint 检查零 error；组件覆盖正常态，但 loading/empty/error 态缺失 |
| **P2** | 组件覆盖 ≥3 种状态（正常/loading/empty 或 error）；类型定义完整无 `any` |
| **P3** | 覆盖全四态（loading/empty/error/success）；附组件自检清单（状态、props 校验、边界值） |
| **P4** | 面对模糊需求能自主确定组件拆分方案并说明理由；代码结构支持后续扩展而非硬编码 |
| **P5** | 组件拆分附替代方案对比（≥2 种方案的优劣分析）；主动引入 性能优化措施（如 memo、懒加载）并说明触发条件 |
| **P6** | 代码预留扩展接口（如 slot/render prop/组合式 API），后续需求可扩展而无需重构；产出代码被其他模块直接导入复用 |
| **P7** | 产出可复用的组件库/代码规范/工程模板，被 ≥2 个项目采用 |
| **P8** | 代码规范覆盖前后端交叉地带（如 API 类型共享、全栈类型安全方案） |
| **P9** | 构建端到端的前端工程体系（开发规范 + 组件体系 + 构建流程 + 质量门禁），经完整项目验证 |
| **P10** | 前端工程体系被全组织采纳为默认实践，组件缺陷率下降 ≥30% |
| **P11** | 前端工程方法论具备行业通用性，可被其他团队直接复用 |
| **P12** | 提出全新的前端工程范式，行业之前无同类方法 |

**评估方法**：
- **素材**：代码文件、lint 报告、组件状态截图/描述、类型定义文件
- **步骤**：① 运行 lint 检查，记录 error/warning 数 → ② 检查组件状态覆盖（列出已覆盖状态） → ③ 检查类型定义完整度（搜索 `any` 数量） → ④ 对照分级表定级
- **易混点**：组件"功能完整"属 C1（产出完整度），组件"代码质量高"属本维度。功能全但代码烂 = C1 高 + D1 低

#### 维度 D2 FE-交互还原度（15%）
- **定义**：前端产出与上游设计/交互规范的还原精度，以及对交互细节的处理水准
- **与通用维度的边界**：C4 量"下游能否消费"，D2 量"还原是否精准"——产出可用但还原走形 vs 像素级还原

| 等级 | 行为描述（可检验） |
|------|-------------------|
| **P0** | 布局与设计稿有明显偏差（间距、对齐、字号等 ≥3 处不一致） |
| **P1** | 主体布局与设计稿一致，但间距/颜色等细节有 1-2 处偏差 |
| **P2** | 视觉还原无明显偏差；交互状态（hover/focus/active）至少覆盖 hover |
| **P3** | 交互状态覆盖 hover/focus/active/disabled 全集；动效过渡自然（有 transition/animation 定义） |
| **P4** | 面对模糊交互说明（如"列表要好用"）能自主补全交互细节（排序、筛选、分页）并说明依据 |
| **P5** | 交互方案附替代方案对比；主动处理设计稿未覆盖的响应式断点（≥2 个断点） |
| **P6** | 预判并实现设计稿未提及但用户必然遇到的交互场景（如键盘导航、滚动加载） |
| **P7** | 产出可复用的交互模式库/响应式方案模板，被 ≥2 个项目采用 |
| **P8** | 交互规范覆盖设计-开发交叉地带（如 Design Token 到 CSS 变量的自动化映射） |
| **P9** | 构建端到端的设计-开发协作体系（Token 体系 + 组件映射 + 走查流程） |
| **P10** | 协作体系被全组织采纳，设计还原返工率下降 ≥30% |
| **P11** | 设计-开发协作方法论具备行业通用性 |
| **P12** | 提出全新的设计-开发协作范式 |

**评估方法**：
- **素材**：页面截图与设计稿对比、交互状态清单、响应式适配测试
- **步骤**：① 逐页对比设计稿与实现，记录偏差数 → ② 列出已实现的交互状态集 → ③ 检查响应式断点覆盖 → ④ 对照分级表定级
- **易混点**："功能都有"属 C1，"还原得像"属本维度

### 3.2 🎯 需求分析专家（分析/沟通类代表）

**专属维度数量**：1（需求分析的核心专业差异集中在洞察深度上）

#### 维度 D1 RA-需求洞察深度（30%）
- **定义**：穿透用户表面表达，挖掘真实需求、隐含约束和利益冲突的深度
- **与通用维度的边界**：C2 量"能处理多模糊的任务"，D1 量"能把模糊需求挖到多深"——前者是任务复杂度承接能力，后者是需求挖掘这个专业领域的纵向深度

| 等级 | 行为描述（可检验） |
|------|-------------------|
| **P0** | 原样记录用户表述，未做任何追问或澄清；需求文档 = 用户原话的格式化 |
| **P1** | 对用户表述中的明显歧义做了追问（≥1 个澄清问题），需求文档消除了字面歧义 |
| **P2** | 识别并补充了用户未说但常见的隐含需求（如权限、异常流程），文档中有"用户未提及但业务必需"的标注 |
| **P3** | 需求文档包含完整的需求自检维度（10 维扫描：场景/用户/目标/约束/边界/依赖/优先级/风险/验收/非功能），各维度有内容或有"不适用"标注 |
| **P4** | 面对模糊的业务方向（如"我们要做 XX 功能"），能自主拆解出 ≥3 个需要澄清的核心问题，并给出各问题的建议答案 |
| **P5** | 需求中标注了利益相关方的潜在冲突（≥1 个冲突点），附冲突分析和建议优先级 |
| **P6** | 需求文档具备前瞻性——标注"当前方案在 X 场景下会遇到 Y 问题"，并预留扩展方案 |
| **P7** | 从多次需求分析实践中提炼出可复用的需求分析模板/问题清单，被 ≥2 个项目采用 |
| **P8** | 需求分析覆盖跨领域约束——识别并纳入技术可行性、设计可行性、运营可行性的交叉约束 |
| **P9** | 构建端到端的需求治理体系（从需求采集→分析→优先级→验收→闭环），经完整项目验证 |
| **P10** | 需求治理体系被全组织采纳，需求返工率下降 ≥30% |
| **P11** | 需求分析方法论具备行业通用性 |
| **P12** | 提出全新的需求分析范式 |

**评估方法**：
- **素材**：需求文档、追问记录（对话记录中提出的澄清问题）、冲突分析章节
- **步骤**：① 检查需求文档是否有"超越用户原话"的内容（有 = ≥P2） → ② 计数 10 维扫描的覆盖维度数 → ③ 检查是否有利益冲突分析 → ④ 对照分级表定级
- **易混点**："需求文档覆盖了多少项"属 C1（产出完整度），"需求挖到多深——是停在表面还是穿透到真实需求"属本维度

### 3.3 🎨 体验设计专家（设计类代表）

**专属维度数量**：2

#### 维度 D1 XD-体验方案专业度（15%）
- **定义**：信息架构、任务流程、交互模式等体验方案的设计专业水准
- **与通用维度的边界**：C3 量"决策有没有依据"，D1 量"体验设计方案本身的专业深度"——有理由不等于理由专业

| 等级 | 行为描述（可检验） |
|------|-------------------|
| **P0** | 信息架构混乱——导航层级不一致或同一信息在多处重复出现，无逻辑分类 |
| **P1** | 信息架构有基本逻辑分类，但存在 ≥1 处层级或命名不一致 |
| **P2** | 信息架构逻辑清晰一致；任务流程覆盖主路径，步骤无冗余或遗漏 |
| **P3** | 任务流程附完整的异常/回退路径；交互模式选择附行业惯例依据（如表单验证时机、弹窗vs页面跳转的选择理由） |
| **P4** | 面对模糊需求（如"用户要能管理 XX"），能自主产出完整的信息架构 + 任务流程方案并说明设计依据 |
| **P5** | 体验方案附替代方案对比（≥2 个方案），对比维度包括用户认知成本、操作效率、学习曲线 |
| **P6** | 方案中标注体验峰值时刻设计——在关键节点（如首次使用、任务完成）有刻意的正向体验强化 |
| **P7** | 产出可复用的体验模式库/设计规范，被 ≥2 个项目采用 |
| **P8** | 体验方案覆盖设计-开发-运营交叉地带（如引导流程兼顾技术实现和运营可配置性） |
| **P9** | 构建端到端的体验设计体系（原则→模式→规范→走查→迭代），经完整项目验证 |
| **P10** | 体验设计体系被全组织采纳，用户任务完成率提升 ≥30% |
| **P11** | 体验设计方法论具备行业通用性 |
| **P12** | 提出全新的体验设计范式 |

**评估方法**：
- **素材**：信息架构图、任务流程图、交互说明文档、方案对比章节
- **步骤**：① 检查信息架构的逻辑一致性（层级、命名） → ② 检查任务流程的路径完整性（主路径+异常+回退） → ③ 检查交互模式是否附设计依据 → ④ 对照分级表定级
- **易混点**："方案覆盖了多少页面"属 C1，"方案的体验设计水准如何"属本维度

#### 维度 D2 XD-用户视角贯穿度（15%）
- **定义**：体验方案中用户视角的融入程度——是否从用户心智模型出发，而非从系统功能出发
- **与通用维度的边界**：C4 量"下游（开发/Leader）能否消费"，D2 量"最终用户能否顺畅使用"——消费者不同

| 等级 | 行为描述（可检验） |
|------|-------------------|
| **P0** | 方案从系统功能视角组织——导航和页面结构对应后端模块而非用户任务 |
| **P1** | 方案开始使用用户语言（如"我的课程"而非"课程管理模块"），但整体结构仍偏系统视角 |
| **P2** | 核心流程从用户任务出发组织（如"布置作业"而非"作业模块→新建"） |
| **P3** | 方案附用户画像引用或用户场景描述，设计决策可追溯到具体用户场景 |
| **P4** | 面对模糊需求能自主定义目标用户画像和核心使用场景，并基于此推导体验方案 |
| **P5** | 方案中标注用户认知负荷分析——复杂操作拆分为 ≤3 步的分步流程，并说明拆分依据 |
| **P6** | 方案预判用户可能的困惑点（≥2 个），并设计对应的引导/提示策略 |
| **P7** | 产出可复用的用户视角设计方法/模板（如用户旅程模板、认知负荷评估表），被 ≥2 个项目采用 |
| **P8** | 用户视角分析覆盖多类型用户的差异（如新手 vs 专家、教师 vs 学生），产出差异化方案 |
| **P9** | 构建端到端的用户体验度量体系（定性+定量），经完整项目验证 |
| **P10** | 用户体验度量体系被全组织采纳，用户满意度可测量提升 |
| **P11** | 用户视角设计方法论具备行业通用性 |
| **P12** | 提出全新的用户视角设计范式 |

**评估方法**：
- **素材**：方案中的用户画像/场景章节、导航结构命名、流程步骤设计
- **步骤**：① 检查导航/页面命名是否用户语言（系统术语 vs 用户任务） → ② 检查核心流程是否从用户任务出发 → ③ 检查是否有用户认知负荷分析 → ④ 对照分级表定级
- **易混点**："有没有用户研究章节"属 C1（完整度），"方案是否真正从用户视角出发"属本维度

## 四、与通用维度的整合规则

### 4.1 综合分数计算

```
综合分数 = 通用加权分 + 领域加权分
```

**通用加权分**（满分 = 最高等级 × 70%）：
```
通用加权分 = C1×0.20 + C2×0.15 + C3×0.15 + C4×0.10 + C5×0.10
```

**领域加权分**（满分 = 最高等级 × 30%）：
- 1 个专属维度：`领域加权分 = D1×0.30`
- 2 个专属维度：`领域加权分 = D1×0.15 + D2×0.15`（或按领域声明的权重比例）

**等级判定**：
- 等级 = 综合分数**向下取整**
- 地板规则依然适用：任何维度（通用或专属）分数 < 目标等级 - 2 时，等级 = min(全部维度分数) + 2

### 4.2 通用与专属判定不一致的处理

| 不一致类型 | 定义 | 处理规则 |
|-----------|------|---------|
| **专属偏低** | 通用维度稳定 P5，领域维度仅 P2-P3 | 地板规则自动触发，等级被拉低。评估记录标注"领域短板：[维度名]，需针对性提升设定" |
| **专属偏高** | 通用维度稳定 P3，领域维度达 P5-P6 | 加权计算自然处理（专属仅占 30%，不会大幅拉高）。评估记录标注"通用短板限制定级，专业能力有提升空间" |
| **差距 ≥4 级** | 通用最高 P6、领域最低 P2（或反之） | 触发地板规则：等级 = min(全部维度) + 2。这意味着一个通用能力很强但领域不专业的专家不会获得高等级——**专业性是等级的必要条件** |

### 4.3 计算示例

以 🖥️ 前端开发专家某次评估为例：

| 维度 | 分数 | 权重 | 加权 |
|------|------|------|------|
| C1 产出完整度 | P4 | 20% | 0.80 |
| C2 复杂度承接 | P4 | 15% | 0.60 |
| C3 决策质量 | P3 | 15% | 0.45 |
| C4 可消费性 | P4 | 10% | 0.40 |
| C5 影响力半径 | P3 | 10% | 0.30 |
| D1 代码工程质量 | P4 | 15% | 0.60 |
| D2 交互还原度 | P3 | 15% | 0.45 |
| **合计** | | **100%** | **3.60** |

- 综合分数 = 3.60 → 向下取整 = **P3**
- 地板检查：最低维度 P3，目标等级 P3，3 ≥ 3-2 = 1 → ✅ 通过
- **判定等级：P3**
