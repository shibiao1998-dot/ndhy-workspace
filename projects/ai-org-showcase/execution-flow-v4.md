## 一、方案理解摘要

### 最终交付物
一个**生产环境就绪的产品官网**，技术架构现代化，视觉品质对标 Stripe/Linear/Anthropic，在移动端和桌面端均有世界级表现，品牌叙事经过专家论证。

### 质量标杆
- **对照评测法**：与 Stripe/Linear/Anthropic 对比不输
- **专家评测法**：叙事力、视觉精度、动效品质、技术实现 4 维度均 ≥ 8/10

### 关键约束
1. 每个专家产出 ≤ 5000 字（上下文窗口约束）
2. 分阶段交付，老板每阶段验收后才推进
3. 叙事分析是前置依赖——老板要先看报告再定内容方向
4. V3 文档可作为基线复用
5. 前端开发通过 ACP（Claude Code）执行

### V3 审查暴露的核心问题（新流程必须解决）

| 来源 | 核心问题 | V4 流程中的解决节点 |
|------|---------|-------------------|
| 前端审查 6.1/10 | 单文件架构不可维护、响应式仅 5/10、无障碍 5/10 | Phase 2 技术架构 + Phase 3 开发 |
| 视觉审查 7.2/10 | 灰阶只有 5 级、40+ 裸色值、间距系统执行不严 | Phase 2 视觉设计升级 |
| UX 审查 6.4/10 | 滚动叙事精度不够、动画类型单一、移动端未独立设计 | Phase 2 体验设计升级 |
| UI 审查 6.3/10 | 25+ 处裸间距、组件状态覆盖不足、移动端拓扑图不可读 | Phase 2 UI 规范升级 |
| 产品审查 7.3/10 | 叙事结构 9/10（保留）、CTA 路径 5/10（需改进）、数据回填缺失 | Phase 1 叙事分析 + Phase 3 精细打磨 |

## 三、Phase 划分与专家分工

### Phase 0：叙事分析与内容定向（前置门控）

> **目标**：让老板看到叙事分析报告，决定 V4 的内容方向。
> **为什么单独设 Phase**：老板明确要求"先看叙事专家分析再决定"，这是后续所有工作的方向锚点。

#### 执行步骤

| Step | 执行角色 | 任务 | 输入 | 输出 | 完成标准 |
|------|---------|------|------|------|---------|
| 0.1 | ✨ 叙事策略专家 | 分析 V3 七幕叙事结构的优劣势，对标 Stripe/Linear/Cognition，输出叙事策略建议 | content-v3.md、review-worldclass-product.md（叙事结构 9/10 部分）、experience-design-v3.md（信息架构） | **叙事分析报告**（≤5000字）：V3 叙事评价 + 改进/保留建议 + 论证依据 | 报告回答了"V3 叙事保留/调整/重做"并给出专业论证 |

#### 并行说明
- 无并行。这是整个流程的第一步。

#### 🔴 老板验收点 #1
**验收内容**：叙事分析报告
**验收方式**：阅读报告，决定以下选项之一：
- A）V3 叙事结构基本保留，在细节上调整
- B）叙事结构需要较大调整（叙事专家进一步出调整方案）
- C）叙事结构推翻重来（重大风险，需评估对整体周期的影响）

**老板决策后**，后续 Phase 的内容文案工作方向才确定。

> **目标**：完成从内容到技术架构的完整设计体系，为开发提供精确输入。
> **关键原则**：V3 文档可复用，升级而非重写。

#### 执行步骤

| Step | 执行角色 | 任务 | 输入 | 输出 | 完成标准 |
|------|---------|------|------|------|---------|
| 1.1a | 📝 技术文档专家 | 基于老板对叙事分析报告的决策，升级内容文案：解决 CTA 路径弱（5/10）、14 处[待回填]、区块三数据回填 | content-v3.md、叙事分析报告、老板决策、review-worldclass-product.md | **content-v4.md**（≤5000字）：升级版内容文案，含所有数据回填 | CTA 路径清晰、无占位符、数据真实 |
| 1.1b | 🏛️ 技术架构专家 | 前端技术选型和架构方案设计 | requirement-document-v4.md、review-worldclass-frontend.md（架构问题）、V3 index.html（了解现状） | **tech-architecture-v4.md**（≤5000字）：框架选型、目录结构、组件拆分策略、构建部署方案、性能策略 | 方案能解决 V3 的单文件/不可维护/响应式 5/10 问题 |
| 1.2 | 🎨 体验设计专家 | 升级交互体验方案：解决 V3 滚动叙事精度不够（6/10）、动画类型单一、移动端未独立设计 | content-v4.md、experience-design-v3.md、review-worldclass-ux.md | **experience-design-v4.md**（≤5000字）：信息架构升级、滚动叙事设计、动效规范、移动端独立方案 | 滚动叙事有"电影感"节奏设计、移动端非桌面缩放版 |

| 1.3b | 🖌️ UI 设计专家 | 升级 UI 规范：解决间距裸值、组件状态不全、移动端拓扑图不可读 | experience-design-v4.md、visual-spec-v4.md、review-worldclass-ui.md | **ui-design-v4.md**（≤5000字）：间距系统执行规范、组件库（含全状态）、响应式断点策略、移动端拓扑图方案 | 所有间距/圆角/阴影使用 Token、组件覆盖 5 种状态 |

#### 依赖关系
```
Phase 0 完成（老板决策）
        │
        ├──→ Step 1.1a（内容文案）  ──→ Step 1.2（体验设计）
        │                                      │
        └──→ Step 1.1b（技术架构）[并行]        ├──→ Step 1.3a（视觉设计）──→ Step 1.3b（UI 设计）
                                               │
                                               └──→ 1.3a 和 1.3b 串行（UI 消费视觉 Token）
```
- **可并行**：Step 1.1a（内容）和 Step 1.1b（技术架构），输入独立
- **必须串行**：内容 → 体验 → 视觉 → UI（每一步的输出是下一步的输入）

#### 🔴 多专家交叉审查（Phase 1 质量门）

Phase 1 所有设计产出完成后，**启动交叉走查**：

| 审查者 | 审查对象 | 审查焦点 |
|--------|---------|---------|
| ✨ 叙事策略专家 | content-v4.md + experience-design-v4.md | 叙事传达是否完整、品牌调性是否一致 |
| 🎨 体验设计专家 | visual-spec-v4.md + ui-design-v4.md | 视觉规范是否支持交互设计意图 |
| 🎨 视觉设计专家 | ui-design-v4.md | UI 组件是否严格遵循 Token 系统 |
| 🏛️ 技术架构专家 | experience-design-v4.md + ui-design-v4.md | 设计方案的技术可实现性 |

**走查产出**：各审查者输出审查意见（≤1000 字/份），发现问题由原作者修正。

**走查标准**：全部审查无 🔴 阻断问题后，Phase 1 才算完成。

#### 🔴 老板验收点 #2
**验收内容**：完整的设计体系文档包
**验收方式**：
1. 阅读 content-v4.md —— 内容文案是否满意
2. 阅读 experience-design-v4.md 中的信息架构和滚动叙事设计
3. 了解技术选型方向（不做技术决策，只确认专家选了什么）
4. 确认移动端方案（拓扑图怎么展示）

**老板确认后进入 Phase 2。**

### Phase 2：核心开发

> **目标**：基于设计体系产出可在浏览器预览的完整页面，包含所有区块内容和基础交互。

#### 执行步骤

| Step | 执行角色 | 任务 | 输入 | 输出 | 完成标准 |
|------|---------|------|------|------|---------|
| 2.1 | 🖥️ 前端开发专家（via ACP Claude Code） | 项目初始化 + 全区块页面实现：搭建框架骨架、实现 7 个区块的组件、接入 Design Token、响应式布局、基础动画 | tech-architecture-v4.md、content-v4.md、experience-design-v4.md、visual-spec-v4.md、ui-design-v4.md | **website/ 可预览前端工程** | 7 区块全可见、邮箱链接可用、三断点（375/768/1440）可正常浏览、Design Token 零裸值 |
| 2.2 | 🖥️ 前端开发专家（via ACP Claude Code） | 动效实现：滚动驱动叙事、进入/离开动画、微交互、移动端动效适配 | experience-design-v4.md 中的动效规范 + Step 2.1 的代码 | 动效全实现的页面 | 所有动画 60fps、滚动过渡有节奏感、移动端动效是适配版而非砍掉版 |
| 2.3 | 🖥️ 前端开发专家（via ACP Claude Code） | 无障碍 + SEO：语义化 HTML（main/h1/heading 层级）、aria 属性、meta 标签、SSR/SSG 配置 | WCAG 2.1 AA 标准 + Lighthouse 基线 | 无障碍+SEO 就绪的代码 | Lighthouse Accessibility ≥ 90、SEO ≥ 90 |

#### 依赖关系
```
Phase 1 完成 + 老板验收
        │
        └──→ Step 2.1（骨架+全区块）──→ Step 2.2（动效）──→ Step 2.3（无障碍+SEO）
```
- 2.1 → 2.2 → 2.3 **严格串行**，每步基于上一步代码增量开发。
- 但注意：2.1 内部可以按区块并行开发（由 Claude Code 自主管理）。

#### 🔴 多专家交叉审查（Phase 2 质量门）

开发完成后，**启动双层审查**：

**第一层：技术审查**

| 审查者 | 审查焦点 |
|--------|---------|
| 🔍 代码审查专家 | 代码质量、架构一致性、Design Token 使用一致性、无死代码/重复声明 |
| 🏛️ 技术架构专家 | 架构方案执行一致性、组件拆分合理性、性能策略落地 |

**第二层：设计审查**

| 审查者 | 审查焦点 |
|--------|---------|
| 🎨 体验设计专家 | 滚动叙事节奏、动效精度、交互反馈一致性 |
| 🎨 视觉设计专家 | 视觉规范执行精度、Token 使用一致性、色彩/间距/排版 |
| 🖌️ UI 设计专家 | 组件状态覆盖、间距系统执行、移动端 UI 适配 |
| ✨ 叙事策略专家 | 叙事传达效果、品牌调性在实现中是否保持 |

**审查产出**：每位审查者输出审查意见（≤2000 字/份），标注 🔴 阻断 / 🟡 建议。
**回溯规则**：
- 🔴 阻断问题 → 前端开发专家修复 → 相关审查者复审
- 🟡 建议 → 记录，Phase 3 精细打磨中解决
- 最多 2 轮审查-修复循环，仍有 🔴 问题则升级给 Leader

#### 🔴 老板验收点 #3
**验收内容**：可浏览的完整页面（预览链接）
**验收方式**：
1. **手机实测**：在自己手机上打开，完整浏览 7 个区块
2. **电脑实测**：在电脑上打开，完整浏览
3. **邮箱测试**：点击联系邮箱链接，确认能打开邮件客户端
4. **感性评测**：和 Stripe.com / Linear.app / Anthropic.com 分别对比浏览
5. **反馈**：告诉我哪里满意、哪里不满意

**老板确认后进入 Phase 3。**

### Phase 3：精细打磨与上线

> **目标**：解决所有遗留问题，达到生产环境就绪标准，完成部署上线。

#### 依赖关系
```
Phase 2 完成 + 老板验收

        └──→ Step 3.1（反馈修复）──→ Step 3.2（测试）──→ Step 3.3（部署）
```

## 六、回溯规则

| 发现问题的阶段 | 问题类型 | 回溯到 | 处理方式 |
|--------------|---------|--------|---------|
| Phase 1 走查 | 内容方向偏离老板决策 | Step 1.1a | 技术文档专家修正文案 |
| Phase 1 走查 | 技术方案不可行 | Step 1.1b | 技术架构专家修正方案 |
| Phase 1 走查 | 视觉/UI 不支持交互意图 | Step 1.3a/1.3b | 视觉/UI 专家修正 |
| Phase 2 审查 | 代码架构偏离方案 | Step 2.1 | 前端开发专家重构相关组件 |
| Phase 2 审查 | 动效精度不达标 | Step 2.2 | 前端开发专家精调动效参数 |
| Phase 2 审查 | 视觉还原偏差 | Step 2.1 | 前端开发专家按视觉审查修正 |
| Phase 3 测试 | Lighthouse 不达标 | Step 2.3/3.1 | 前端开发专家性能优化 |
| Phase 3 老板反馈 | 整体不满意 | Phase 1 | 需重新评估（升级给 Leader） |

**最多回溯深度**：任何审查-修复循环最多 2 轮。2 轮后仍有 🔴 问题 → 升级给 Leader 重新评估任务拆分或专家能力。

## 七、关键成功因素

1. **叙事分析质量决定全局方向**：Phase 0 的叙事分析报告是后续一切的锚点。如果叙事方向错了，后续所有设计和开发都会偏
2. **指令包精度决定开发质量**：Phase 2 前端开发的指令包必须包含 5 份完整的设计文档（content/experience/visual/ui/architecture），任何一份缺失都会导致开发者"瞎猜"
3. **多专家走查是"超越所有网站"的保障**：V3 的问题不是某个专家能力不够，是缺乏交叉审查。单人视角不可能覆盖所有维度
4. **移动端不是附属品**：V3 的致命伤是移动端是桌面缩放版。V4 的体验设计和 UI 设计必须有独立的移动端方案
5. **Design Token 零裸值是铁律**：V3 有 40+ 裸色值、25+ 裸间距，V4 必须在设计阶段就定义完整 Token 系统，开发阶段严格执行，审查阶段逐项检查

## 八、时间节奏建议

| Phase | 估计工作量 | 并行优化后 | 说明 |
|-------|-----------|-----------|------|
| Phase 0 | 1 专家 × 1 轮 | 1 轮 | 叙事分析 + 等老板决策 |
| Phase 1 | 5 专家 × 多轮 | 2-3 轮 | 内容和架构并行，设计链串行，走查 1 轮 |
| Phase 2 | 1 专家（开发）+ 6 专家（审查） | 3-4 轮 | 开发 3 步串行 + 审查最多 2 轮修复 |
| Phase 3 | 3 专家串行 | 2-3 轮 | 修复→测试→部署 + 终审 |

**总计约 8-11 轮调度**（不含老板验收等待时间）。
