## 第一部分：核心架构与设计逻辑

### 1.1 为什么需要这些维度？

**DJ会议核心：**

> "提示词工程的本质是什么？第一，信息对称...为AI提供更准确，更详细，更结构化的信息...第二，相信AI可以做到了，有能力了...我要的是提示词，我要的是为AI提供最全面的信息，保证它能够把后续工作做好的信息。"

**核心假设：**

1. **AI有基础能力**，但提供的维度越多、越具体、越高质量 → AI的产出就越好
2. **信息越充分**，AI就能从更多角度思考，避免遗漏
3. **人的价值**是不断发现"AI没想到的维度"，告诉AI"你还可以从这个角度看"

### 1.2 什么是高质量的设计？

#### **1. 真实性（30%）- 基于证据而非想象**

```Plain
✅ 好的表现：
- "用户访谈#45中，15位高中生明确表示..."
- "数据显示，87%的用户在XX步骤流失..."
- "竞品A的用户评价中，200条差评集中在..."

❌ 差的表现：
- "用户应该会喜欢..."
- "一般来说，高中生都..."
- "我觉得这样设计会..."

检查方法：每个关键判断能否追溯到证据？
```

#### **2. 独特性（25%）- 有差异化价值**

```Plain
✅ 好的表现：
- 基于我方独特资源（如"化学专用3D技术+教师团队"）
- 竞品无法快速复制
- 差异化是用户真正在乎的

❌ 差的表现：
- "竞品有排行榜，我们也做排行榜"
- "为了不同而不同"（用户不在乎的差异化）
- "抄竞品+微创新"

检查方法：我们的独特价值是什么？基于什么资源？
```

#### **3. 可行性（25%）- 能做出来**

```Plain
✅ 好的表现：
- 技术能力边界内，或明确攻关计划
- 资源与方案规模匹配
- 有历史类似项目参考

❌ 差的表现：
- "理想中的3D交互"（技术不成熟）
- "需要50人团队"（只有10人）
- "类似马斯克的Neuralink"（不切实际）

检查方法：我们真的能做出来吗？需要什么代价？
```

#### **4. 价值观一致性（20%）- 符合企业理念**

```Plain
✅ 好的表现：
- 体现"尊重用户""真实学习"理念
- 符合战略方向和教育理念
- 不触碰红线（如"不做大模型研发"）

❌ 差的表现：
- "用积分排行榜驱动学习"（违背教育理念）
- "方向偏离战略重点"
- "触碰价值观禁忌"

检查方法：符合DJ的教育理念吗？符合战略方向吗？
```

### 1.3 维度使用原则

#### **原则1：按需加载，而非全部加载**

```Plain
不同任务加载不同维度组合：

核心价值编写：
- 必需：A类全部 + K类(术语) + M类(核心价值专项) + I01(评分卡)
- 强建议：B类(用户) + C类(竞品) + G类(案例)
- 可选：其他

设计方法论编写：
- 必需：A-E类 + K类 + I03(评分卡)
- 强建议：F类(可行性) + G类(案例) + J类(专业知识)

原型设计：
- 必需：E11(方法) + H类(规范) + I04(评分卡)
- 强建议：G类(案例) + D07(边界情况)
```

#### **原则2：信息必须具体可用**

```Plain
❌ 不好的维度信息：
"竞品的优势"（太笼统）

✅ 好的维度信息：
"竞品A的核心优势：AI智能推荐（用户好评率92%，证据：应用市场评价）
 我们的劣势：无AI推荐（用户抱怨"题目重复"占差评35%）
 我们的对策：人工+规则推荐（成本低，但个性化不足）"
```

#### **原则3：时效性明确管理**

```Plain
每个维度必须标注：
- 更新频率（实时/每月/每季度/按需）
- 最后更新时间
- 有效期（如有）
- 过期提醒机制
```

#### **原则4：持续进化，而非静态知识库**

```Plain
维度库不是一次性建设，而是持续进化：
- 每个项目结束复盘 → 发现新维度
- 每次质量问题 → 补充缺失维度
- 每季度清理 → 删除无效维度
- 每半年优化 → 调整维度颗粒度
```

## 第二部分：维度总览

**总维度数：99个**

```Plain
【A类】战略与价值观维度（8个）
    → 确保方向正确、价值观一致
    
【B类】用户真实性维度（12个）
    → 确保基于真实用户而非想象
    
【C类】竞品与差异化维度（6个）
    → 确保有独特价值、理解竞争格局
    
【D类】场景与需求维度（7个）
    → 确保需求真实、场景完整
    
【E类】设计方法与技巧维度（16个）
    → 提供高质量的设计方法
    
【F类】可行性与资源维度（5个）
    → 确保能做出来、资源匹配
    
【G类】历史经验与案例维度（5个）
    → 从成败中学习、避免重复错误
    
【H类】设计标准与规范维度（7个）
    → 确保设计符合标准、可交付
    
【I类】质量判断与检查维度（8个）
    → 可量化的质量标准、支持AI自检
    
【J类】特殊领域专有知识维度（9个）
    → 教育、化学等专业知识 + 学习目标库
    
【K类】专有名词与定义维度（6个）
    → 术语标准化、保证沟通准确
    
【L类】项目全生命周期维度（10个）
    → 从备选池到周版本的完整执行链路
```

## 第三部分：维度迭代机制

### 3.1 维度补充触发

```Plain
【被动补充】（AI发现信息缺口）
- AI执行任务时发现某个维度信息缺失
- AI标注"该信息影响质量但库中没有"
→ 人补充后，添加到对应维度

【主动补充】（人发现遗漏）
- 人审核时发现AI未考虑某个重要因素
- 新项目涉及全新领域/方法
→ 人主动添加新维度

【定期补充】（项目复盘）
- 每个项目结束后复盘
- 识别"如果当时有XX信息，质量会更高"
→ 将XX信息补充为新维度
```

### 3.2 维度质量管理

```Plain
【有效性检查】（每季度）
- 统计每个维度的使用频率
- 识别"从未使用"的维度 → 考虑删除或合并
- 识别"高频使用"的维度 → 重点维护和细化

【准确性检查】（每月）
- 检查信息是否过时（如战略调整、竞品变化）
- 更新时效性信息
- 标注信息有效期

【完整性检查】（项目复盘）
- 质量问题是否因维度缺失导致
- 补充新维度
- 调整维度颗粒度（太粗则拆分，太细则合并）
```

### 3.3 维度版本管理

```Plain
【版本记录格式】
v1.0 (2025-01)：初始版本，135个核心维度

v1.1 (2025-02)：
  - 新增：XX维度（因XX项目发现缺失）
  - 修改：YY维度（细化为更具体的信息）
  - 删除：ZZ维度（3个月未使用）
  - 优化：AA维度（根据反馈调整定义）

【每次变更必须记录】
- 变更内容
- 变更原因（哪个项目/问题触发的）
- 预期效果（这个变更如何提升质量）
- 实际效果（下次复盘时验证）
```
