# 官网 V5 — 执行流程

> 🔩 流程设计专家 · 2026-03-16
> 输入：V5 需求文档、V4 执行流程、33 项修复清单

---

## 一、方案理解

**V5 ≠ 从零构建。** V4 已完成完整设计体系 + 前端代码 + 老板批准原型。V5 的本质是：

| 工作类型 | 内容 | 体量 |
|---------|------|------|
| 修复 | 33 项（7 阻断 + 10 老板反馈 + 16 建议） | 主体工作量 |
| 内容更新 | 3 项组织进化成果写入官网 | 轻量 |
| 新增功能 | N1 专家等级徽章（拓扑图+悬停卡片） | 中等 |

**铁律**：修复优先于新增。先修 33 项，再做 N1。

---

## 二、Phase 划分

### Phase 1：设计补丁 + 内容更新（并行）

> **目标**：为修复和新功能提供精确设计输入，同时完成内容更新

| Step | 角色 | 任务 | 输出 | 完成标准 |
|------|------|------|------|---------|
| 1.1 | 🎨 体验设计专家 | 补齐动效精度规范：GSAP scrub/pin 参数、高潮爆发序列 5 步时序、pathDraw 路径描边规格 | 动效补丁规范（≤3000 字） | R1/R2/R3 有精确到帧/参数的实现规格，前端可直接消费 |
| 1.2 | 🎨 视觉设计专家 + 🖌️ UI 设计专家 | ① 视觉：补充 N1 等级徽章视觉规范（颜色/大小/样式）② UI：补充 5 态交互规范（R6）、等级徽章在卡片中的布局 | 设计补丁规范（≤3000 字） | 等级徽章有完整视觉定义，5 态有精确 Token 值 |
| 1.3 | 📖 叙事策略专家 | 3 项内容更新文案：C1 双层审查描述、C2 "31 个 AI 专家"确认、C3 巡查机制卡片文案 + S2 区块⑤底文确认 | 内容更新文案（≤1500 字） | 文案无句号，调性一致，可直接替换 |

**并行关系**：1.1 / 1.2 / 1.3 **完全并行**——三者输入独立（均基于已有 V4 设计体系 + V5 需求文档），输出服务不同修复项，无交叉依赖。

**预估时间**：1 轮调度（3 个 subagent 并行 spawn）

---

### Phase 2：核心修复（ACP 开发）

> **目标**：修复 33 项 + 内容替换 + N1 新功能实现
> **执行方式**：ACP（Claude Code），Leader 每 5 分钟巡查

分两批串行开发，每批内部由 ACP 自主管理顺序：

| Step | 任务 | 范围 | 完成标准 |
|------|------|------|---------|
| 2.1 | **P0 阻断 + 老板反馈修复** | R1-R7（7 阻断）+ B1-B10（10 老板反馈），去重后约 14 项独立修复 + Phase 1 内容更新文案替换（C1/C2/C3） | 7 阻断全部关闭；老板 10 项反馈逐项落实；内容更新到位；零裸色值/零硬编码尺寸 |
| 2.2 | **P2 建议修复 + N1 新功能** | 前 8 项建议（S6/S7/S12/S13/S16/S18/S19/S21）+ N1 专家等级徽章 + 剩余 8 项建议尽量修复 | ≥ 6 项 P2 建议完成；N1 徽章在拓扑图 + 悬停卡片可见；focus-visible + 键盘导航可用 |

**串行原因**：2.2 基于 2.1 的代码增量开发，2.1 修复的动画系统（R1/R2/R3）是 2.2 部分建议项的前置。

**Leader 巡查协议**：
- 每 5 分钟检查文件产出（目录变化）
- 验证 Token 使用一致性（抽查 TSX 文件无裸值）
- 检查 build 状态
- 连续 2 次（10 分钟）无输出 → 终止并拆分任务重新派发

**预估时间**：2-3 轮调度（2.1 约 30-45 分钟，2.2 约 20-30 分钟）

---

### Phase 3：审查 + 测试 + 上线

> **目标**：确保修复质量，达到生产就绪，完成部署

| Step | 角色 | 任务 | 完成标准 |
|------|------|------|---------|
| 3.1 | 🔍 代码审查专家 | 代码质量审查：Token 一致性、动画实现精度、无死代码、事件监听清理 | 无 🔴 阻断，放行决策 |
| 3.2 | 🎨 体验 + 🎨 视觉 + 🖌️ UI（并行走查） | 体验：动效精度、滚动叙事节奏；视觉：Token 执行一致性；UI：5 态覆盖、响应式断点 | 各维度无 🔴 阻断 |
| 3.3 | 🖥️ 前端（ACP） | 修复 3.1 + 3.2 的 🔴 阻断项（如有） | 所有 🔴 关闭 |
| 3.4 | 🧪 测试专家 | 跨浏览器 + 多设备 + Lighthouse + 33 项修复逐项验证 | Lighthouse 四项 ≥ 90；33 项修复零遗漏 |
| 3.5 | 🚀 部署运维专家 | 生产环境更新部署 | URL 可访问、HTTPS 正常 |

**并行/串行**：
- 3.1 和 3.2 **并行**（代码审查和设计走查互不依赖）
- 3.3 **串行等待** 3.1 + 3.2 完成
- 3.4 → 3.5 **严格串行**

**回溯规则**：审查-修复最多 2 轮。2 轮后仍有 🔴 → 升级 Leader。

**🔴 老板验收**：3.5 完成后老板实测（手机 + 电脑 + 邮箱链接 + 对标对比）

**预估时间**：2-3 轮调度

## 四、关键成功因素

1. **指令包精度**：Phase 2 ACP 指令包必须包含 V4 全套设计文档 + Phase 1 补丁 + 33 项修复清单（带优先级），缺一项前端就会瞎猜
2. **修复顺序**：P0 阻断 → 老板反馈 → P2 建议 → N1 新功能，严格按优先级
3. **ACP 巡查不可跳过**：Leader 每 5 分钟检查文件产出和 Token 使用，偏差立即纠偏
4. **不改方向**：V5 是精度补齐，不是重新设计。任何"不如我们改一下视觉方向"的冲动 → 打回
