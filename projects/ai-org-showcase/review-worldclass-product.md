## 维度 1：价值主张 — 7/10

### 3 秒测试

访客首屏看到的是：

```
2026 年 3 月 10 日
一个人启动了一个项目。
6 天后，
他拥有了一个 28 人的产品团队。
他没有招过一个人。
```

**判定**：3 秒内能制造强烈的认知冲突（"怎么可能？"），但**不能**让访客理解"这是什么"。访客需要滚动到区块二（揭幕震撼）才能看到"AI 组织"这个核心概念。

### 对标分析

| 标杆 | 首屏价值主张 | 秒懂程度 |
|------|-------------|---------|
| **Stripe** | "Financial infrastructure for the internet" | 3 秒内知道它是什么 |
| **Linear** | "Linear is a purpose-built tool for planning and building products" | 3 秒内知道它干什么 |
| **Notion** | "Your wiki, docs, & projects. Together." | 3 秒内知道它解决什么 |
| **NDHY** | "一个人…28 人团队…没招过一个人" | 3 秒内只有悬念，没有答案 |

这是一种**刻意的设计选择**——design-v3.md 明确说了"不出现 AI、Agent、组织任何一个词"，用事实制造认知冲突。这在 Cognition.ai（Devin 的官网）上验证过是有效策略。

### 问题

1. **悬念策略有效但有风险**：如果访客不往下滚（跳出），他们带走的认知是零——"有个人搞了个团队"，不知道是什么。Stripe 和 Linear 即使用户秒退也能带走一个印象。
2. **首屏动画等待时间过长**：打字机 + 延迟序列总计约 5.5 秒才展示完"他没有招过一个人"——这是悬念揭晓前的最低信息量。移动端用户耐心更短。
3. **区块二的揭幕文案虽好但位置太深**："一个人类。28 个 AI 专家。一个完整的产品组织"——这才是价值主张的一句话定义。但它在整个首屏之后。

### 改进建议

- **不改变悬念策略的前提下**，考虑在首屏最底部（滚动箭头上方）加一行极淡的提示词，如 `AI 组织` 或 `The AI Organization`——不破坏悬念，但给秒退用户留下一个关键词种子。
- **缩短 Hook 动画至 3.5 秒**。当前 5.5 秒在移动端偏长。Devin 的首屏动画控制在 3 秒内。
- 跳过机制（ESC/点击/触摸）已实现，✅ 这是正确的安全网。

## 维度 2：叙事结构 — 9/10

### 七幕结构评估

| 区块 | 叙事功能 | 评价 |
|------|---------|------|
| 1. 悬念开场 | 制造认知冲突 → "不可能" | ✅ 极致留白，电影级开场。文案零废字。 |
| 2. 揭幕震撼 | 回答"怎么做到的" → 组织拓扑 | ✅ 从悬念到震撼的转折精准。28 个节点逐个亮起的视觉叙事配合文案揭幕。 |
| 3. 运转揭秘 | "你正在看的网站就是证据" | ✅ 自证策略 = S 级叙事。"真实过程，0 行人类代码"是全站最有力的一句话。 |
| 4. 认知落差 | "散装 AI vs AI 组织"对比 | ✅ 对比结构清晰，左暗右亮的视觉处理与内容语义一致。标签系统精炼有力。 |
| 5. 深层架构 | 三张剖面图 = 权力/记忆/进化 | ✅ 从"what"到"why"的深入恰到好处。三个维度覆盖了最核心的差异化要素。 |
| 6. 更大图景 | 行业宣言："答案是——组织" | ✅ 宣言式文案+首尾呼应完成度极高。"散兵赢不了战争"是全站最佳金句。 |
| 7. 行动号召 | CTA | 🟡 见维度 4 详述。相比前 6 个区块，CTA 明显弱了一截。 |

### 对标判定

- **情绪曲线**：悬念（好奇）→ 震撼（惊叹）→ 沉浸（理解）→ 对比（认同）→ 深入（信服）→ 共鸣（认可）→ 行动。与 design-v3.md 设计完全一致，执行到位。
- **Stripe 级别的自证叙事**：区块三"你正在看的这个网站"是全站核心策略的执行。这个策略比 Stripe 的"API 代码演示"更极端——Stripe 展示的是产品能力的片段，NDHY 展示的是产品能力的完整闭环。
- **Linear 级别的克制**：全站无感叹号、无"最好""颠覆""革命"等膨胀词汇。事实陈述 + 节制语气 = 高级感。

### 扣分点

- **区块三和区块五的叙事节奏略有重叠**：区块三的时间线步骤描述和区块五的架构剖面都在解释"组织如何运转"，只是维度不同（过程 vs 结构）。连续阅读时会有轻微的"还在讲同一个事"的感觉。Stripe 的解决方案是在中间插入一个完全不同质感的区块（如客户 logo 或数据 showcase）来重置阅读节奏。

## 维度 3：内容密度 — 8/10

### 信息量评估

| 区块 | 字数（约） | 密度判断 | 说明 |
|------|-----------|---------|------|
| 1. Hook | ~40 字 | ✅ 恰到好处 | 极致留白，每个字都在工作 |
| 2. Reveal | ~60 字 + 28 节点 | ✅ 恰到好处 | 文字极少，视觉（拓扑图）承担信息量 |
| 3. Process | ~800 字（含展开） | 🟡 略冗余 | 8 步展开详情文案模式重复（"不是 X。是 Y。"出现 4 次） |
| 4. Contrast | ~400 字 | ✅ 恰到好处 | 左右对比，扫描式阅读友好 |
| 5. Architecture | ~300 字 | ✅ 恰到好处 | 三个剖面均精炼 |
| 6. Vision | ~250 字 | ✅ 恰到好处 | 宣言体，节奏感好 |
| 7. CTA | ~40 字 | 🟡 偏少 | 见维度 4 |

### 冗余点

1. **区块三的展开详情**：8 个步骤的展开文案结构过于统一——"[角色]接到/接手任务 → 解释做了什么 → 产出：[待回填]"。建议差异化每个步骤的叙述方式，或精简为 3-4 个关键步骤的深度展开，其余保持折叠卡片即可。
2. **区块五"剖面 2"和"剖面 3"的底文高度相似**：
   - 剖面 2："每一次纠正 → 最高优先级信号 → 立即记录 → 永不重犯"
   - 剖面 3："不靠'下次注意'。靠写入记忆文件的硬规则驱动行为改变。"
   - 这两句说的是同一件事，建议合并或区分角度。

### 不足点

- **缺少社会证明（Social Proof）**：全站零客户引用、零使用数据、零第三方背书。对比 Stripe（"百万企业使用"）、Linear（客户 logo 墙）、Notion（用户数据）。这可以理解——NDHY 处于早期阶段——但至少可以放 GitHub star 数、项目完成数量等可量化的自有数据。
- **区块三的 `[待回填]` 占位符**：当前 HTML 中通过 JS 隐藏了含 `[待回填]` 的元素。这意味着实际渲染时这些数据点不可见，步骤卡片的 `.step-meta` 行整个被隐藏。**上线前必须回填**。

## 维度 4：CTA 转化路径 — 5/10

### 当前状态

| CTA | 动作 | 目标 |
|-----|------|------|
| 主按钮 | "跟我们聊聊" | `mailto:shibiao1998@gmail.com` |
| 次按钮 | "在 GitHub 上关注" | GitHub 仓库 |

### 问题诊断

1. **"跟我们聊聊"指向 mailto = 高摩擦转化**。
   - 对标：Stripe 用 "Start now"（注册）、Linear 用 "Get started" → 产品内、Notion 用 "Get Notion free"（零摩擦）。
   - mailto 在 2026 年是最高摩擦的转化方式之一——要打开邮件客户端、写邮件、等回复。跳出率极高。
   - 个人 Gmail 地址（`shibiao1998@gmail.com`）进一步降低信任感。

2. **缺少中间转化路径**。当前是二元选择：发邮件 or 看 GitHub。没有"轻量级参与"选项，如：
   - 订阅 newsletter
   - 加入 Discord / 微信群
   - 查看文档 / 演示视频
   - 在线体验 demo

3. **CTA 副标题"所有源码、角色设定、技能代码，全部公开"**——这是一个很好的信任信号，但它暗示的行动是"去 GitHub 看"而不是"联系我们"。主按钮和副标题的引导方向不一致。

4. **全站仅在最底部有 CTA**。7 个区块的沉浸式阅读后才到行动号召。对比 Stripe/Linear 在首屏和中段都有 CTA 入口。

### 改进建议

| 优先级 | 建议 |
|--------|------|
| P0 | 将 `mailto:` 替换为表单页面或 Calendly 预约链接。即使暂时没有产品可注册，也应该用专业的联系方式。 |
| P0 | 使用品牌域名邮箱（如 `hello@ndhy-ai-team.com`）而非个人 Gmail。 |
| P1 | 在区块三或区块四后插入一个轻量 CTA（如"在 GitHub 上看完整源码"按钮），不必等到最后。 |
| P2 | 增加一个低摩擦转化路径：微信群/Discord 社区入口、Newsletter 订阅等。 |
| P2 | 主按钮文案从"跟我们聊聊"改为更具行动指向性的表述，如"预约演示"或"加入社区"。 |

### 硬伤

1. **`[待回填]` 占位符大量存在**：区块三的 8 个步骤中，产出时间、产出大小均为 `[待回填]`。虽然 JS 隐藏了含占位符的元素，但这意味着：
   - 步骤卡片的 `.step-meta` 行整个不可见——用户看不到产出文件名
   - 展开详情中的"产出：[待回填] 字的结构化内容文案"等也被隐藏

## 维度 7：差异化 — 9/10

### 竞品官网对比

| 维度 | CrewAI | AutoGen | MetaGPT | NDHY |
|------|--------|---------|---------|------|
| **定位** | Multi-agent framework | Multi-agent conversation framework | Multi-agent framework for software company | AI 组织（不是框架） |
| **首页策略** | 功能列表 + 客户 logo + 用例 | 文档驱动，技术先行 | GitHub README 风格 | 电影式叙事 + 自证 |
| **叙事角度** | "Build AI agent teams" | "Enable next-gen LLM apps" | "Collaborative agents for complex tasks" | "一个人 + AI 组织 = 完整产品公司" |
| **情感连接** | 低（工具感） | 低（学术感） | 中（开源社区感） | **高（人性化叙事）** |
| **自证力度** | 无（展示 Demo） | 无（展示代码） | 无（展示论文） | **极强（网站本身就是产品）** |

### 差异化维度分析

1. **品类差异化 ✅**："不是工具。不是框架。不是 Prompt 模板。是一支有灵魂、有纪律、有记忆、会进化的 AI 组织。"——这句话一笔划清了与所有 multi-agent framework 的品类边界。CrewAI 说"Build"（你来搭），NDHY 说"是"（它已经存在并运转）。

2. **叙事差异化 ✅**：竞品官网全部是 Builder 视角（"用我的工具搭建你的 agent"），NDHY 是 Organization 视角（"这个组织已经在运转，你看到的就是证据"）。这是根本性的叙事差异。

3. **价值层次差异化 ✅**：竞品在卖能力（"能做什么"），NDHY 在卖范式（"AI 组织化是未来"）。区块六的宣言把差异化从产品层面拉到了行业认知层面——"2026 年，AI 行业的真正问题不是模型够不够强，而是怎么把强大的模型组织成可靠的生产力。"

4. **人格差异化 ✅**：竞品官网是机构化语言，NDHY 有一个真实的人类故事（"一个人启动了一个项目"）。这种 founder-led narrative 在 Linear（Karri Saarinen 的设计哲学）和 Anthropic（Dario Amodei 的安全信条）上已被验证有效。

### 扣分点

- **缺少技术深度层**：差异化在叙事和情感层面是世界级的，但缺少技术层面的差异化展示。竞品用户（开发者/技术决策者）可能会问："组织化"具体在技术上是怎么实现的？SOUL.md 的格式是什么？记忆系统的存储机制是什么？区块五的三张剖面图是概念级的，不是技术级的。对于技术受众，可能需要一个"Deep Dive"入口链接到更详细的技术文档。

## 总结：差距地图

### 🟢 世界级（9-10 分）

- **叙事结构**：七幕电影式叙事，情绪曲线控制精准，首尾呼应闭环完整。比 95% 的技术产品官网更有故事性。
- **差异化**：品类定义（"不是框架，是组织"）、自证策略（"网站本身就是证据"）、宣言式收束（"答案是——组织"）三层差异化构成护城河。

### 🟡 良好但有改进空间（6-8 分）

- **价值主张**：悬念策略有效但有跳出风险。建议在不破坏叙事的前提下增加一个极轻量的品类提示。
- **内容密度**：整体克制精炼，区块三展开详情稍有模式化重复。
- **数据准确性**：`[待回填]` 占位符是上线阻断项。"28"的口径需统一。
- **SEO**：框架在但缺少 OG 图片、`<h1>`、JSON-LD 三个关键件。

### 🔴 短板（5 分及以下）
