# 流程设计全流程

## 输入与输出

**输入**：产品方案(PRD)、体验设计方案、技术/实现方案、项目约束条件。

**输出**（全部 5 项）：
1. 专家角色清单 — 每个角色含完整 7 维度定义
2. 角色协作拓扑图 — 依赖、并行、交接关系
3. 执行流程图 — Phase/Step 结构，标注每步由谁执行
4. 交接规范 — 每个角色的输出格式要求
5. 回溯反馈规则 — 出问题时谁找谁、怎么回退

---

## 核心流程：7 步工作方法论

每次流程设计严格遵循 7 步。**不可跳过 Step 1（理解方案）和 Step 7（自检）**。

### Step 1：理解方案

**目标**：提炼交付物、质量标准和约束。

操作：
1. 完整阅读 PRD / 设计方案 / 技术方案
2. 提炼：最终要交付什么？质量标准是什么？有哪些约束（时间、技术、人力）？
3. 识别方案类型，为后续步骤选择合适的领域角色池

**关键**：不理解方案就动手拆解 = 方向错误。这一步决定后续所有工作的质量。

### Step 2：拆解产出物

**目标**：识别所有需要生产的产出物。

操作：
1. 列出**最终交付物**：用户/客户最终拿到的东西
2. 反推**中间产物**：为做出最终交付物需要先产出的
3. 识别**支撑产物**：保障质量和可持续性需要的（文档、监控、自动化流程等）

产出物三层分类参考：

| 方案类型 | 最终交付物 | 中间产物（部分） |
|----------|-----------|----------------|
| SaaS产品 | 可访问的Web应用 | 数据库、API、前端页面、后端服务 |
| 在线课程 | 可观看的课程包 | 脚本、PPT、视频录制、剪辑成片、字幕 |
| 行业白皮书 | 可发布的PDF报告 | 数据采集、分析图表、文案初稿、排版设计 |
| 营销活动 | 活动落地页+传播素材 | 文案、平面设计、H5页面、短视频、投放计划 |
| AI训练数据集 | 标注好的数据集 | 数据采集、清洗、标注规范、标注执行、质检 |

**关键**：支撑产物最容易遗漏——测试、文档、部署配置、监控往往在方案中未明确写出，但缺了项目就跑不起来。

### Step 3：识别工作类型

**目标**：把每个产出物拆解为具体的工作类型。

遵循**单一职责原则**：

| 原则 | 说明 | 举例 |
|------|------|------|
| 专业独立 | 需要一种独立的专业能力 | "写代码"和"测代码"是两种不同专业 |
| 可独立执行 | 给定输入后可独立完成 | 前端拿到 API 文档后可独立开发 |
| 输出明确 | 有明确的可检验输出物 | "可运行的API接口"而非"差不多写完了" |
| 粒度适中 | 不能太粗也不能太细 | "前端开发"合适，"写CSS的人"太细 |

**粒度判断标准**：

太粗的信号：
- 需要在 2 种以上完全不同的专业技能间切换
- 输出物服务于完全不同的下游
- 去掉一部分剩下的仍然完整

太细的信号：
- 输出物不能独立被下游使用
- 工作量极小不足以形成独立单元
- 两个角色需要逐句逐行级别实时协同

操作：
1. 为每个产出物列出需要的专业能力
2. 合并相同专业能力的工作
3. 用粒度标准检查，太粗的拆、太细的合

### Step 4：定义专家角色

**目标**：为每个工作类型定义完整的角色。

每个角色必须包含 **7 个维度**：

```
1. 角色名称 — 简洁明确
2. 一句话定位 — 大白话说清楚干什么
3. 核心问题 — 这个角色存在是为了回答什么
4. 输入 — 开始工作需要什么
5. 输出 — 必须交付什么
6. 核心能力 — 需要什么专业能力
7. 边界 — 不做什么
```

**额外要求**：
- 每个角色的输出格式**必须能直接作为下游输入**
- 每个角色必须附带**验收标准**："做完 XX 且满足 YY 条件"
- **必须包含"项目管理专家"角色**（无论方案多简单，这是固定角色）

操作：
1. 从 Step 3 的工作类型映射为角色
2. 按 7 维度逐一定义
3. 加入项目管理专家角色
4. 根据方案类型，从领域角色池中选取参考
5. 检查：是否有遗漏？是否有冗余？

→ 领域角色池参考，按方案涉及的领域读取对应文件：
- 软件开发类：`references/domain-software.md`
- 多媒体内容生产类：`references/domain-multimedia.md`
- AIGC 内容生产类：`references/domain-aigc.md`
- 文档/知识类：`references/domain-document.md`
- 教育/培训类：`references/domain-education.md`
- 数据工程类：`references/domain-data-engineering.md`

### Step 5：设计协作关系

**目标**：定义角色间的依赖、并行、交接和回溯关系。

回答 4 个关键问题：

**Q1 谁先谁后（依赖关系）**
- 画出依赖图：哪个角色的输出是另一个角色的输入？
- 找出关键路径

**Q2 哪些可以并行**
- 并行条件（三个同时满足才可并行）：
  - 输入来自同一上游
  - 不需要实时交互
  - 输出分别服务不同下游

**Q3 怎么交接**
- 每个交接点定义：
  - 交付格式（什么文件/什么格式）
  - 可交接条件（满足什么标准才算可以交）
  - 下游确认方式（收到后怎么确认可用）

**Q4 出了问题怎么回退**
- 每种问题类型：
  - 反馈给谁
  - 从哪一步重新开始
  - 影响范围评估

### Step 6：编排执行流程

**目标**：产出可执行的 Phase/Step 结构。

输出格式，每步包含：

```
Phase X：[阶段名称]
  Step X.1：[步骤名称]
    - 执行角色：XXX
    - 输入：XXX
    - 输出：XXX
    - 并行说明：可与 Step X.2 并行 / 需等待 Step Y.Z 完成
    - 完成标准：XXX
    - 回溯规则：若不合格 → 反馈给 XXX → 从 Step Y.Z 重做
```

操作：
1. 按依赖关系排定顺序
2. 标注可并行的步骤
3. 为每步设定完成标准
4. 设定回溯规则

### Step 7：自检

**目标**：确保设计没有漏洞。

逐条检查以下问题，任何一条答"否"就需要修补：

- [ ] 每个交付物是否都有完整的生产链路？（从原始输入到最终产出）
- [ ] 每个角色的输出是否都有下游消费者？（无孤岛输出）
- [ ] 是否存在"孤岛角色"？（输入输出都不与其他角色关联）
- [ ] 项目管理专家是否已包含？
- [ ] 关键路径上是否有瓶颈？是否合理？
- [ ] 每个角色是否都有明确的验收标准？
- [ ] 交接规范是否每个节点都定义了？
- [ ] 回溯路径是否覆盖了主要风险点？
- [ ] 是否存在角色边界模糊（两个角色职责重叠）？
- [ ] 涉及物理世界操作的部分是否标注为"外部依赖"？

---

## 6 条关键约束（强制遵守）

1. **按需生成，不预设** — 不因"通常需要 XX 角色"就默认加入，只有方案明确需要时才纳入
2. **项目管理必须在** — 无论方案多简单，项目管理专家是固定角色，不可省略
3. **角色可跨领域组合** — 一个方案可同时需要多领域角色，不受单一领域限制
4. **只纳入计算机可执行的工作** — 涉及物理世界操作（拍摄、寄送、面对面会议等）标注为"外部依赖"
5. **每个角色必须有验收标准** — "做完 XX 且满足 YY 条件"，不可只有输出没有标准
6. **宁可多一个角色也不要模糊边界** — 不确定两个工作是否该合并时，单独设角色

---

## 难点提醒

1. **领域知识广度要求高** — 不熟悉的领域，先读对应领域角色池参考文件，必要时搜索补充
2. **粒度判断是最难的** — 没有标准答案，严格按粒度判断标准执行，宁细勿粗
3. **与项目管理专家的交接** — 你的输出必须是项目管理能直接用的：角色清单、流程图、交接规范缺一不可
