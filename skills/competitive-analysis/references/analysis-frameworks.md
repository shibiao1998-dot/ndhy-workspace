# 分析框架工具箱

7 个分析框架，按需选用。每个框架包含：适用场景、操作步骤、输出格式、注意事项。

---

## 1. 功能矩阵

**适用场景**：功能层面对比，回答"各家有没有/做得好不好"。

**操作步骤**：
1. 列出所有需要对比的功能项（纵轴）
2. 列出所有竞品（横轴）
3. 逐项评估，使用统一评分标准

**评分标准**（建议）：
- ✅ 完善：功能完整、体验好
- ⚠️ 基础：有但简陋或有明显缺陷
- ❌ 无：不具备该功能
- 🔜 规划中：有明确信号表明在开发（标注来源）

**输出格式**：

| 功能项 | 竞品A | 竞品B | 竞品C |
|--------|-------|-------|-------|
| 功能1 | ✅ | ⚠️ | ❌ |
| 功能2 | ✅ | ✅ | ✅ |

**注意**：
- 功能项粒度要合适——太粗（"AI 功能"）没意义，太细（"按钮颜色"）也没意义
- 不只是打勾，关键功能要附一句"怎么做的"
- 空白列（全 ❌）= 市场空白信号

---

## 2. 定位地图

**适用场景**：战略层面对比，回答"各家在抢什么位置"，找空白地带。

**操作步骤**：
1. 选择两个最有区分度的维度作为坐标轴
2. 维度选择建议：价格高低、专业度/通用度、企业/个人、技术深度/易用性、垂直/平台
3. 将每个竞品标注在坐标中
4. 识别空白区域和拥挤区域

**输出格式**（用文字描述坐标位置）：
```
横轴：[维度1] 低 ←→ 高
纵轴：[维度2] 低 ←→ 高

- 竞品A：右上（高价格、高专业度）
- 竞品B：左下（低价格、低专业度）
- 空白区域：右下（高专业度、低价格）→ 可能的差异化空间
```

**注意**：
- 维度选择决定了分析质量——选没有区分度的维度（如"功能多 vs 功能少"）产出价值低
- 空白不等于机会——可能是因为那个位置不可行
- 同一组竞品可以画多张定位地图（不同维度组合）

## 3. SWOT 分析

**适用场景**：全面评估单个竞品，识别其优劣势和外部机会威胁。

**操作步骤**：
1. 分析竞品的内部因素（优势/劣势）
2. 分析竞品面临的外部因素（机会/威胁）
3. 交叉分析：SO（用优势抓机会）、ST（用优势应对威胁）、WO（改进劣势抓机会）、WT（劣势+威胁=最大风险）

**输出格式**：

| | 正面 | 负面 |
|--|------|------|
| **内部** | **优势 (S)**：<br>1. …<br>2. … | **劣势 (W)**：<br>1. …<br>2. … |
| **外部** | **机会 (O)**：<br>1. …<br>2. … | **威胁 (T)**：<br>1. …<br>2. … |

**注意**：
- 优势和劣势必须具体——"技术强"不是优势，"自研XX引擎，处理速度比竞品快3倍"才是
- 标注每条的来源：`[事实]` 还是 `[推测]`
- SWOT 最大的价值在交叉分析，不是四个象限本身

---

## 4. 价值曲线

**适用场景**：寻找差异化方向，回答"应该在哪些要素上加大投入、在哪些上减少"。

**操作步骤**：
1. 列出行业的关键竞争要素（横轴），6-10 个
2. 对每个竞品评估各要素的投入程度（纵轴），用 1-5 分
3. 画出每个竞品的"曲线"
4. 寻找独特的"凹凸"形态——与众不同的曲线 = 差异化成功

**输出格式**：

| 竞争要素 | 竞品A | 竞品B | 竞品C | 行业平均 |
|----------|-------|-------|-------|----------|
| 价格竞争力 | 2 | 5 | 3 | 3.3 |
| 功能丰富度 | 5 | 2 | 4 | 3.7 |
| 易用性 | 3 | 5 | 3 | 3.7 |
| 客户服务 | 4 | 3 | 2 | 3.0 |
| … | … | … | … | … |

**注意**：
- 竞争要素选择是关键——要选客户真正在意的，不是行业自嗨的
- 差异化的核心是"取舍"：在某些要素上远超平均，在另一些上低于平均
- 如果所有竞品的曲线形状相似 = 行业同质化严重 = 差异化机会大

## 5. 用户旅程对比

**适用场景**：体验层面对比，回答"完成同一件事，各家的体验差在哪"。

**操作步骤**：
1. 定义一个核心用户任务（如"从注册到完成第一次XX"）
2. 在每个竞品中实际走一遍完整流程
3. 记录每一步的操作、时长、摩擦点
4. 横向对比同一步骤在不同竞品中的体验差异

**输出格式**：

| 步骤 | 竞品A | 竞品B | 竞品C |
|------|-------|-------|-------|
| 注册 | 邮箱即可，30秒 | 需手机验证，2分钟 | 需企业邮箱，被拒 |
| 引导 | 3步引导，清晰 | 无引导，直接进后台 | 视频教程，5分钟 |
| 首次使用 | … | … | … |
| 核心操作 | … | … | … |

**注意**：
- 选择最能体现产品价值的核心任务，不是边缘功能
- 记录主观感受和客观数据（步骤数、耗时）
- 摩擦点是金矿——竞品的摩擦点 = 我们的优化方向

---

## 6. 商业模式画布

**适用场景**：商业层面对比，回答"各家怎么赚钱、靠什么活着"。

**操作步骤**：
拆解竞品的 9 个商业模块：

1. **客户细分**：服务谁？
2. **价值主张**：提供什么独特价值？
3. **渠道通路**：通过什么渠道触达客户？
4. **客户关系**：怎么维护客户关系？
5. **收入来源**：怎么赚钱？
6. **核心资源**：最关键的资源是什么？
7. **关键活动**：最重要的事是什么？
8. **重要合作**：跟谁合作？
9. **成本结构**：主要成本在哪？

**输出格式**：

| 模块 | 竞品A | 竞品B |
|------|-------|-------|
| 客户细分 | … | … |
| 价值主张 | … | … |
| … | … | … |

**注意**：
- 很多信息需要推断 `[推测]`，诚实标注
- 重点关注"收入来源"和"成本结构"——这决定了竞品能不能活下去
- 商业模式画布最适合对比"看起来相似但赚钱方式不同"的竞品

---

## 7. 技术栈对比

**适用场景**：技术层面对比，回答"各家的技术选型和架构思路"。

**操作步骤**：
1. 从公开信息推断技术栈（招聘信息、GitHub、技术博客、BuiltWith）
2. 对比关键技术选型
3. 分析技术选型背后的战略意图

**输出格式**：

| 技术维度 | 竞品A | 竞品B | 竞品C |
|----------|-------|-------|-------|
| 前端框架 | React | Vue | Angular |
| 后端语言 | Go | Python | Java |
| 数据库 | PostgreSQL | MongoDB | MySQL |
| 云服务 | AWS | 自建 | 阿里云 |
| AI/ML | 自研模型 | 调用 OpenAI | 无 |
| 部署方式 | SaaS only | SaaS + 私有化 | 私有化为主 |

**信息来源优先级**：
1. 官方技术博客 / 工程团队分享
2. GitHub 开源项目
3. 招聘 JD 中的技术要求
4. BuiltWith / Wappalyzer 等工具检测
5. 推断（需标注 `[推测]`）

**注意**：
- 技术栈信息大部分是推断，务必标注确信度
- 关注技术选型背后的战略含义：选 Go = 追求性能；选 Python = 追求开发速度；支持私有化 = 面向大客户
- 技术栈对比对非技术读者价值有限——只在技术决策场景下深入展开

## 框架组合建议

不同调研目的推荐的框架组合：

| 调研目的 | 推荐框架组合 |
|----------|-------------|
| **A. 市场验证型** | 定位地图 + 商业模式画布 + SWOT |
| **B. 方案参考型** | 功能矩阵 + 用户旅程对比 + 技术栈对比 |
| **C. 竞争定位型** | 功能矩阵 + 定位地图 + 价值曲线 |
| **D. 趋势洞察型** | 价值曲线 + 商业模式画布 + 定位地图 |
