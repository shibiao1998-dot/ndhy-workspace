## 一、串并行判断树

Leader 或项目管理专家在排执行顺序时，对每一对任务按此判断树决策：

```
任务 A 和任务 B 需要排序
        ↓
B 是否需要 A 的产出作为输入？
  ├─ 是 → 必须串行（FS 依赖）
  └─ 否 ↓
B 的执行是否取决于 A 的决策结果？
  ├─ 是 → 必须串行（决策依赖）
  └─ 否 ↓
A 和 B 是否修改同一数据/资源？
  ├─ 是 → 必须串行（共享状态）
  └─ 否 ↓
B 的开始是否需要 A 通过质量检查？
  ├─ 是 → 必须串行（质量门控）
  └─ 否 ↓
做 B 是否需要先从 A 获取理解？
  ├─ 是 → 必须串行（认知依赖）
  └─ 否 ↓
A 和 B 有明确的接口约定吗？
  ├─ 是 → 可以并行 ✅
  └─ 否 → 先定义接口，再并行
```

### 速查表：NDHY 常见任务对的串并行结论

| 任务 A | 任务 B | 结论 | 原因 |
|--------|--------|------|------|
| 用户研究 | 市场评估 | ✅ 并行 | 无依赖，各自独立 |
| 用户研究 | 竞品调研 | ✅ 并行 | 无依赖，各自独立 |
| 需求分析 | 方案设计 | ❌ 串行 | 认知依赖——不知道做什么不能设计怎么做 |
| 战略定位 | 产品定义 | ❌ 串行 | 产出-输入——方向未定不能做产品定义 |
| 产品定义 | 体验设计 | ❌ 串行 | 产出-输入——不知道产品是什么不能设计体验 |
| 产品定义 | 技术架构 | ✅ 并行 | 两者基于同一产品定义各自工作 |
| 体验设计 | 技术架构 | ✅ 并行 | 产品定义完成后各自独立，通过对齐保持一致 |
| 体验设计 | 视觉设计 | ❌ 串行 | 产出-输入——视觉需要交互骨架 |
| 技术架构 | 数据库设计 | ❌ 串行 | 决策依赖——架构方向决定数据库选型 |
| 技术架构 | API 设计 | ❌ 串行 | 决策依赖——架构方向决定 API 风格 |
| 数据库设计 | API 设计 | ✅ 并行 | 技术架构已定后各自独立，但需字段语义对齐 |
| API 设计 | 前端开发 | ❌ 串行 | 接口依赖——没契约前端没法写 |
| API 设计 | 后端开发 | ❌ 串行 | 接口依赖——没契约后端没法写 |
| 前端开发 | 后端开发 | ✅ 并行 | 基于 API 契约各自开发，前端用 Mock |
| 前后端开发 | 代码审查 | ❌ 串行 | 质量门控——没代码不能审查 |
| 代码审查 | 联调集成 | ❌ 串行 | 质量门控——审查没过不能合并 |
| 联调集成 | 系统测试 | ❌ 串行 | 质量门控——联调不通测不了功能 |
| 系统测试 | 发布部署 | ❌ 串行 | 质量门控——质量不达标不能上线 |
| 发布策略 | 部署准备 | ✅ 并行 | 各管各的领域 |

## 二、项目类型选择指南

### 选择决策表

```
收到新项目
    ↓
需求是否完全确定且不允许变更？
  ├─ 是 → 瀑布模型
  └─ 否 ↓
是否需要迭代探索、有用户反馈循环？
  ├─ 是 → Scrum（2-4 周 Sprint）
  └─ 否 ↓
是持续流动的工作（没有明确起止点）？
  ├─ 是 → 看板（Kanban）
  └─ 否 ↓
是紧急修复/热补丁？
  ├─ 是 → 看板（Kanban）
  └─ 否 → 默认 Scrum
```

### 三种模型对比速查

| 维度 | 瀑布 | Scrum | 看板 |
|------|------|-------|------|
| 迭代方式 | 无迭代，一次做完 | 固定周期（2-4 周 Sprint） | 持续流动，无固定周期 |
| 需求变更 | 不允许（理论上） | Sprint 间可调整 | 随时可调整优先级 |
| 串并行 | 严格串行 | Sprint 内可并行 | 不同工作项可并行 |
| 适用 NDHY 项目 | 需求完全锁定的项目 | 产品开发类（默认选择） | 技能开发、知识库维护、持续优化 |
| 关键约束 | 阶段不可回退 | Sprint Goal 不变 | WIP 限制 |

### NDHY 项目类型映射

| 项目类型 | 推荐模型 | 理由 |
|---------|---------|------|
| 新产品开发 | Scrum | 需要迭代探索，有用户反馈 |
| 技能体系建设 | 看板 | 持续流动，优先级随时调整 |
| 知识库构建 | 看板 | 持续积累，无固定周期 |
| 组织流程优化 | Scrum（短 Sprint） | 小步迭代验证 |
| 紧急修复 | 看板 | 快速响应，不等 Sprint |
| 合规性项目 | 瀑布 | 需求固定，流程严格 |

## 三、常犯错误防范清单

### 3.1 调度前必查（每次 spawn 前过一遍）

| # | 检查项 | 不通过的后果 |
|---|--------|-------------|
| 1 | 需求是否已通过验证门？ | 做出来不是用户要的 |
| 2 | 前置依赖的产出是否已完成？ | 下游任务无法正确执行 |
| 3 | API 契约/数据模型是否已定义？（如涉及开发） | 联调时接口对不上 |
| 4 | 验收标准是否具体可验证？ | 产出质量无法判断 |
| 5 | 执行者是否拥有足够的上下文？ | 信息不对称导致偏差 |

### 3.2 串并行决策必查

| # | 检查项 | 违反的后果 |
|---|--------|-----------|
| 1 | 并行的两个任务是否真的无数据依赖？ | 并行后发现需要返工 |
| 2 | 并行任务之间是否有明确的接口约定？ | 各做各的，集成时对不上 |
| 3 | 是否错误地串行了可以并行的任务？ | 项目周期不必要地延长 |
| 4 | 有条件并行时，前提条件是否已满足？ | 假并行，实际在等依赖 |

### 3.3 阶段门控必查

| Gate | 必须检查 | 放行条件 |
|------|---------|---------|
| G0 需求门 | 需求清晰？价值成立？ | Leader 确认 |
| G1 方案门 | 产品定义完整？体验合理？技术可行？ | Leader + 老板确认 |
| G2 设计门 | 视觉完成？API 契约定义？数据模型设计？ | Leader 确认 |
| G3 开发门 | 代码审查通过？单元测试通过？ | 项目管理专家确认 |
| G4 集成门 | 联调通过？端到端跑通？ | 联调集成专家确认 |
| G5 发布门 | 系统测试通过？发布策略就绪？回滚方案？ | Leader + 老板确认 |

### 3.4 教育科技项目额外检查

| # | 检查项 | 说明 |
|---|--------|------|
| 1 | 发布时间是否在学期窗口内？ | 新学期前 2-4 周是最佳窗口 |
| 2 | 是否安排了教师培训？ | 发布前必须完成 |
| 3 | 教学内容是否经过学科专家审核？ | 不可跳过 |
| 4 | 是否考虑了双用户群？ | 教师和学生的需求可能冲突 |
| 5 | 是否符合课程标准？ | 教育产品的硬约束 |

## 四、关键路径识别方法

### 典型产品开发关键路径

```
需求分析 → 战略定位 → 产品定义 → 体验设计 → 视觉设计 → 前端开发 → 联调 → 测试 → 发布
```

这条路径上的任何延迟都导致项目整体延期。

### 并行支线（有浮动时间）

```
产品定义 → 技术架构 → 数据库设计 → 后端开发 → 联调（汇入关键路径）
                   ↓
               API 设计 → 联调（汇入关键路径）
```

### 识别步骤

1. 列出所有任务及其依赖关系
2. 估算每个任务的持续时间
3. 找出最长路径 = 关键路径
4. 关键路径上的任务 = 零浮动（不能延迟）
5. 非关键路径的任务有浮动时间，可以灵活调整

## 五、Leader 调度决策流程图

```
收到任务
    ↓
【1】需求是否清晰？
    ├─ 否 → spawn 需求分析专家深挖 → 等确认 → 继续
    └─ 是 ↓
【2】判断规模
    ├─ 简单（单专家）→ 直接 spawn 对应专家
    ├─ 中等（2+ 专家）→ 步骤 3
    └─ 重大（组织级）→ 步骤 3，Leader 全程主导
        ↓
【3】spawn 流程设计专家
    - 输入：需求包 + 产出物列表
    - 产出：角色清单 + 协作拓扑 + 执行流程
        ↓
【4】spawn 项目管理专家
    - 输入：流程设计产出 + 验收标准
    - 由项目管理专家负责全部调度执行
        ↓
【5】项目管理专家按流程调度
    - 遵循依赖关系（用本文件的判断树）
    - 每个阶段设 Gate 检查
        ↓
【6】Leader 在里程碑节点验收
    - S 级全审，A 级审关键点，B/C 级抽检
```
