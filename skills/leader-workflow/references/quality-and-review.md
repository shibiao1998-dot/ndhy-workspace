# 质量验收、改动决策与自省

## 质量验收（分级验收）

**硬规则：代码类产出必须先过自动验证（test/lint/类型检查），未通过不进入人工审查。**

| 级别 | 标准 | 验收方式 |
|------|------|---------|
| **S级** | 交付给老板的最终产出 | Leader 全审 |
| **A级** | 关键技术决策、架构方案、组织变更 | Leader 审关键点 + 抽查 |
| **B级** | 常规开发、文档整理 | 项目管理专家验收，Leader 抽检 |
| **C级** | 内部工具、辅助任务 | 项目管理专家验收即可 |

**权力下放后的风险控制**：项目管理专家拥有 B/C 级任务的验收权，但 Leader 保留抽检权和任何时候的否决权。

> 📖 代码审查详细流程见 Skill: code-quality-gate

## 异常介入决策（异常驱动介入）

**原则：正常流程不需要 Leader，只有出问题时才介入。**

| 信号 | 动作 |
|------|------|
| 按时交付、质量达标 | 不介入，记录结果 |
| 超时但有进展 | 追问一次，给延期空间 |
| 连续失败/质量差 | 接管协调，分析原因 |
| 跨层依赖/方向分歧 | 主动介入收敛 |
| 涉及老板决策权限 | 上报，给选项+推荐 |

> 📖 具体恢复操作（追问→重启→上报 3 阶段）见 Skill: heartbeat-ops

## 改动决策护栏

### 关卡一：反漂移检查
- 这个改动是解决真实问题，还是"为了显得聪明"？
- 改完后效果可验证吗？不可验证的改动不做
- 优先级排序：**稳定 > 可解释 > 复用 > 扩展 > 新奇**
- 如果去掉这个改动，系统会变差吗？不会 → 不做

### 关卡二：值不值得做

任意一个答"是"就值得做，全"否"就别做：
1. **高频吗？** 影响的操作每天都在发生？
2. **止损吗？** 能防止已发生的失败再次发生？
3. **减负吗？** 能减少人类需要亲自介入的次数？
4. **提效吗？** 能让 Agent 完成同样任务消耗更少资源？

向人类交付时使用结构化格式：标题 + 成果摘要 + 验收结果逐条 + 产出物路径 + 备注。

> 📖 详见 Skill: relay-protocol

## 自省（重大任务完成后）

- **达标了吗？** 对照验收标准，结果是否真正达到预期？
- **哪里能更好？** 过程中有没有低效、返工、信息缺失的环节？
- **这是一个 pattern 吗？** 如果同类问题反复出现，提炼规则写入 `MEMORY.md`
- **工具和流程需要更新吗？** 观察到的 Agent 行为模式，是否暗示某个技能或流程该迭代？
