# deployment-ops -- 部署运维执行工具箱

将通过开发、联调和测试的交付物安全、可控、可观测地部署到目标环境，并建立发布、监控、告警、回滚和运行反馈闭环。

> 能部署不等于可上线。能上线不等于可稳定运行。没有回滚和监控的发布不是合格发布。

## 9 步标准流程

```
Step 1: 理解发布目标、范围与依赖
  |  消化发布范围、目标环境、变更清单、上游交付结论
  |  -> 提取: 发布目标、变更模块、目标环境、依赖服务、已知风险
  |  -> 确认: 联调结论、测试结论、代码审查结论是否齐备
  |  -> 信息不足 -> 要求上游补充（不猜测、不假设）
  |
Step 2: 校验发布前提与环境一致性
  |  -> 校验构建产物完整性（前端包/后端镜像/数据库迁移脚本）
  |  -> 校验环境一致性（开发→预发→生产配置对齐）
  |  -> 校验外部依赖可达性（第三方API/数据库/消息队列/缓存）
  |  -> 详细方法 -> references/prerequisites.md
  |
Step 3: 校验配置、密钥与权限
  |  -> 校验环境变量、配置文件差异
  |  -> 校验密钥/证书有效性和过期时间
  |  -> 校验部署账号权限、网络策略、防火墙规则
  |  -> 校验数据库连接、第三方服务凭证
  |  -> 详细方法 -> references/prerequisites.md 配置密钥权限
  |
Step 4: 制定发布步骤与发布窗口
  |  -> 设计发布步骤（顺序、依赖、并行度）
  |  -> 确定发布窗口（低峰期、维护窗口）
  |  -> 设计 War Room 模式（人员、通道、决策链）
  |  -> 设计灰度/蓝绿/滚动策略（如适用）
  |  -> 详细方法 -> references/release-execution.md
  |
Step 5: 制定回滚方案与故障恢复
  |  -> 制定每个变更点的回滚步骤
  |  -> 定义回滚触发条件和决策阈值
  |  -> 制定数据回滚策略（如涉及schema变更）
  |  -> 制定紧急发布流程（hotfix路径）
  |  -> 详细方法 -> references/rollback-recovery.md
  |
Step 6: 校验可观测性（健康检查+日志+监控+告警）
  |  -> 校验健康检查端点和探针配置
  |  -> 校验日志格式、级别和采集链路
  |  -> 校验核心指标监控（延迟/错误率/吞吐/饱和度）
  |  -> 校验告警规则和通知渠道
  |  -> 制定发布观察计划（观察窗口/观察指标/升级条件）
  |  -> 详细方法 -> references/observability.md
  |
Step 7: 评估发布风险与汇总 readiness
  |  -> 逐项评估发布风险（影响面/概率/可控度）
  |  -> 汇总发布就绪状态（见 §发布就绪判断框架）
  |  -> 输出发布决策建议（放行/带风险发布/阻塞）
  |  -> 详细方法 -> references/checklists.md
  |
Step 8: 执行发布观察与异常跟踪
  |  -> 按观察计划监控关键指标
  |  -> 记录异常事件（时间/现象/影响/处置）
  |  -> 触发回滚决策（见 §回滚决策树）
  |  -> 确认发布稳定后关闭观察窗口
  |
Step 9: 输出发布结论与运行反馈回流
     -> 7 个标准文档（见下方）
     -> 生成运行反馈回流清单（问题→归属专家→建议措施）
     -> 过发布检查清单（见下方速查）
     -> 按模板组织 -> references/templates.md
```

## 发布深度速查

| 深度 | 适用场景 | 覆盖步骤 | 输出物 |
|------|----------|----------|--------|
| **快速发布** | 配置变更/小补丁/单服务热修 | Step 1-3 重点 + 5 简版 + 7 概述 | 发布检查清单 + 简版回滚方案 |
| **标准发布** | 常规迭代/多模块上线 | 完整 9 步 | 完整交付包（7 文档） |
| **深度发布** | 全新产品首发/大规模架构变更/数据迁移/多环境灰度 | 完整 9 步 + 多方案对比 + 演练验证 | 完整交付包 + 演练报告 + 应急预案 |

判断不清时向上确认。

## 5 层深度边界速查

| 层级 | 主责 | 核心产出 | 边界说明 |
|------|------|---------|---------|
| **需求与设计定义层** | 项目管理/API/数据库/架构专家 | 需求定义、契约、数据模型 | 部署专家只消费，不定义 |
| **实现层** | 前端/后端开发专家 | 代码、构建产物 | 部署专家只校验产物，不写代码 |
| **审查与验证层** | 代码审查/联调/测试专家 | 审查结论、测试报告 | 部署专家消费结论作为发布前提 |
| **发布与运行保障层** | **部署运维专家（核心）** | 发布方案、回滚、监控、观察 | 本层全部职责 |
| **运行反馈层** | 部署运维专家（输出）→ 各专家 | 运行反馈、问题回流 | 提供反馈，不吞掉后续修复职责 |

**铁律**：向下只给运行约束和反馈，不越界写代码。向上只消费已确认的交付结论，不替上游补质量。

## 发布就绪判断框架

```
发布就绪评估
  |
  |- 前提检查（全部必须通过）
  |   |- 构建产物完整且版本正确？
  |   |- 联调结论为"通过"？
  |   |- 测试结论为"可发布"？
  |   |- 代码审查无阻断项？
  |   +- 全部 YES -> 继续 | 任一 NO -> ⛔ 阻塞，回流上游
  |
  |- 环境与配置检查（全部必须通过）
  |   |- 环境一致性已确认？
  |   |- 配置/密钥/权限齐备？
  |   |- 外部依赖可达？
  |   +- 全部 YES -> 继续 | 任一 NO -> ⛔ 阻塞，补齐后重检
  |
  |- 保障能力检查（按项评估）
  |   |- 回滚方案已制定且可执行？
  |   |- 健康检查已覆盖？
  |   |- 监控告警已配置？
  |   |- 发布观察计划已制定？
  |   +- 全部 YES -> ✅ 具备发布条件
  |   +- 部分 NO -> ⚠️ 带风险发布（须明示风险 + PM审批）
  |   +- 关键项 NO -> ⛔ 阻塞
  |
  -> 输出: ✅ 放行 | ⚠️ 带风险发布 | ⛔ 阻塞
```

### 判断规则

| 结论 | 条件 | 要求 |
|------|------|------|
| ✅ **放行** | 前提+环境+保障全部通过 | 直接发布 |
| ⚠️ **带风险发布** | 前提+环境通过，保障有非关键缺口 | 明示风险 + PM审批 + 加强观察 |
| ⛔ **阻塞** | 前提或环境未通过，或保障有关键缺口 | 回流上游/补齐后重新评估 |

## 回滚决策树

```
发布后检测到异常
  |
  |- 异常影响面评估
  |   |- 核心业务功能不可用？ -> 立即回滚
  |   |- 错误率超过阈值（如 >5%）？ -> 立即回滚
  |   |- 响应延迟严重恶化（如 P99 >3x 基线）？ -> 立即回滚
  |   |- 数据一致性受损？ -> 立即回滚 + 紧急数据修复
  |   |
  |   |- 非核心功能异常？
  |   |   |- 有快速修复方案（<30min）？ -> 尝试 hotfix
  |   |   |- 无快速修复 -> 评估降级/回滚
  |   |
  |   |- 性能轻微下降但在可接受范围？
  |   |   |- 趋势恶化？ -> 准备回滚
  |   |   +- 趋势稳定 -> 继续观察 + 记录
  |   |
  |   +- 仅告警无用户影响？ -> 记录 + 继续观察
  |
  |- 回滚执行
  |   |- 纯服务回滚（无schema变更）-> 镜像/容器版本回退
  |   |- 含 schema 变更 -> 执行 down migration + 服务回退
  |   |- 含数据变更 -> 数据修复脚本 + 服务回退
  |   +- 回滚后验证 -> 健康检查 + 核心链路验证
  |
  +- 回滚后复盘 -> 记录原因 + 回流相关专家
```

### 回滚触发阈值（参考值，按项目调整）

| 指标 | 立即回滚 | 准备回滚 | 继续观察 |
|------|---------|---------|---------|
| 错误率 | >5% | 2%-5% | <2% |
| P99 延迟 | >3x 基线 | 2x-3x 基线 | <2x 基线 |
| 核心功能 | 不可用 | 降级 | 正常 |
| 健康检查 | 持续失败 | 间歇失败 | 正常 |

## 反模式速查

| # | ❌ 错误做法 | ✅ 正确做法 |
|---|------------|------------|
| 1 | 手工上线步骤不可重复，每次临时拼 | 发布步骤文档化、可审查、可重复执行 |
| 2 | 只关注"能部署成功"，不关注能回滚 | 每次发布必须附带经过验证的回滚方案 |
| 3 | 没有监控和告警就直接上线 | 上线前确认监控覆盖、告警规则和通知渠道 |
| 4 | 环境变量和权限临时在生产现场补 | 环境配置提前对齐，生产变更走审批流程 |
| 5 | 发布前不检查关键依赖和外部服务 | 发布前逐项校验依赖可达性和版本兼容性 |
| 6 | 把"测试通过"等同于"上线就安全" | 测试结论是前提之一，还需校验环境、配置、可观测性 |
| 7 | 上线后没有观察窗口和应急预案 | 制定观察计划，明确观察指标、窗口和升级条件 |
| 8 | 发现运行问题却没有回流机制 | 运行问题按归属分类，回流对应专家并跟踪闭环 |
| 9 | 用"先发再说"掩盖环境和配置债务 | 环境和配置问题必须发布前解决，不带入生产 |

-> 详细防范与案例 [references/checklists.md](references/checklists.md) 反模式详细防范

## 发布检查清单速查

| # | 检验项 | 通过条件 |
|---|--------|---------|
| 1 | **构建产物完整** | 前端包/后端镜像/迁移脚本版本正确、hash校验通过 |
| 2 | **上游结论齐备** | 联调通过 + 测试可发布 + 审查无阻断项 |
| 3 | **环境一致** | 开发→预发→生产配置差异已确认并对齐 |
| 4 | **配置密钥齐备** | 环境变量/密钥/证书/权限全部就位且有效 |
| 5 | **外部依赖可达** | 第三方API/数据库/消息队列/缓存连通性已验证 |
| 6 | **发布步骤明确** | 步骤文档化、顺序明确、可重复执行 |
| 7 | **回滚方案就绪** | 每个变更点有回滚步骤、触发条件已定义 |
| 8 | **健康检查覆盖** | 存活探针/就绪探针已配置、端点已验证 |
| 9 | **监控告警就绪** | 核心指标有监控、告警规则和通知渠道已配置 |
| 10 | **日志可采集** | 日志格式规范、级别合理、采集链路畅通 |
| 11 | **观察计划制定** | 观察窗口/指标/阈值/升级条件已明确 |
| 12 | **风险已评估** | 已知风险已列出、影响已评估、应对已准备 |

## 协作接口

### 上游输入

| 来源 | 接收内容 | 使用方式 |
|------|---------|---------|
| 项目管理专家 | 发布目标、范围、窗口、优先级 | 确定发布范围和时间约束 |
| 后端开发专家 | 构建产物、服务依赖、健康检查、配置项 | 校验发布前提和运行依赖 |
| 前端开发专家 | 前端构建产物、环境变量、CDN配置 | 校验前端部署前提 |
| 联调集成专家 | 联调结论、关键链路验证结果 | 作为发布前提判断依据 |
| 测试专家 | 测试结论、已知缺陷、质量门槛 | 作为发布前提判断依据 |
| 代码审查专家 | 审查结论、阻断项 | 作为发布前提判断依据 |

### 下游输出

| 接收方 | 传递内容 | 约束关系 |
|--------|---------|---------|
| 项目管理专家 | 发布readiness、风险评估、阻塞项 | 供发布决策 |
| 后端开发专家 | 运行异常回流、健康检查缺口 | 在运行反馈内修复 |
| 前端开发专家 | 前端运行问题回流、资源部署问题 | 在运行反馈内修复 |
| 联调集成专家 | 环境链路问题回流 | 区分环境问题和业务问题 |
| 测试专家 | 生产环境新发现的质量问题 | 补充回归测试 |

### 升级规则

| 场景 | 升级给 |
|------|--------|
| 发布目标/范围/优先级不清 | 项目管理专家 |
| 服务依赖/健康检查/日志缺口 | 后端开发专家 |
| 前端资源/环境变量/客户端监控缺口 | 前端开发专家 |
| 关键链路未验证/集成问题 | 联调集成专家 |
| 质量门槛/已知缺陷评估 | 测试专家 |
| 系统性基础设施/架构限制 | 技术架构专家 |
| 发布阻塞/风险取舍/延期决策 | 项目管理专家 |

## 触发时首要动作

无论如何触发，第一步永远是：

> "发布目标是什么？变更了哪些模块？目标环境是什么？联调和测试结论是否齐备？有没有已知风险和约束？"

-> 判断发布深度 -> 消化输入 -> 按 9 步流程执行
