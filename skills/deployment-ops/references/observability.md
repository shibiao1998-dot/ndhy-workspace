# 可观测性设计

## 概览

可观测性设计覆盖四大领域：
1. **健康检查** — 服务存活性和就绪性探测
2. **日志规范** — 日志格式、级别和采集
3. **监控与告警** — 指标采集、面板和告警规则
4. **发布观察计划** — 发布后系统观察窗口和响应

---

## 一、健康检查

### 1.1 健康检查类型

| 类型 | 用途 | 检查内容 | 失败后果 |
|------|------|---------|---------|
| **存活探针（Liveness）** | 检测服务是否活着 | 进程存活、主线程响应 | 重启容器/进程 |
| **就绪探针（Readiness）** | 检测服务是否能接收流量 | 数据库连接、缓存连接、依赖服务 | 从负载均衡摘除 |
| **启动探针（Startup）** | 检测服务是否启动完成 | 初始化完成、预热完成 | 延迟启动其他探针 |

### 1.2 健康检查设计规范

```
健康检查端点设计：

GET /health/live    → 存活检查（轻量，只检查进程）
  返回: { "status": "ok" } | HTTP 200
  失败: HTTP 503
  耗时: <100ms

GET /health/ready   → 就绪检查（检查依赖）
  返回: {
    "status": "ok|degraded|unhealthy",
    "checks": {
      "database": { "status": "ok", "latency_ms": 5 },
      "cache": { "status": "ok", "latency_ms": 2 },
      "queue": { "status": "ok", "latency_ms": 3 }
    }
  }
  失败: HTTP 503（任一关键依赖不可用）
  耗时: <500ms
```

### 1.3 健康检查配置参考

| 参数 | 存活探针 | 就绪探针 | 启动探针 |
|------|---------|---------|---------|
| 间隔 | 10s | 5s | 5s |
| 超时 | 3s | 5s | 5s |
| 失败阈值 | 3次 | 2次 | 30次 |
| 成功阈值 | 1次 | 1次 | 1次 |
| 初始延迟 | 15s | 5s | 0s |

### 1.4 健康检查常见问题

| 问题 | 原因 | 解决 |
|------|------|------|
| 就绪检查误判 | 检查了非关键依赖 | 区分关键/非关键依赖 |
| 存活检查太重 | 存活探针检查了数据库 | 存活只检查进程，就绪检查依赖 |
| 启动超时 | 初始化时间过长 | 增加启动探针超时/失败阈值 |
| 健康检查雪崩 | 大量实例同时检查同一依赖 | 添加随机抖动（jitter） |

## 二、日志规范

### 2.1 日志级别使用规范

| 级别 | 使用场景 | 示例 | 生产配置 |
|------|---------|------|---------|
| **ERROR** | 需要立即关注的错误 | 数据库连接失败、支付异常 | ✅ 开启 |
| **WARN** | 潜在问题但当前可处理 | 重试成功、降级生效、配额接近上限 | ✅ 开启 |
| **INFO** | 关键业务事件 | 用户登录、订单创建、发布部署 | ✅ 开启 |
| **DEBUG** | 调试信息 | 函数调用参数、中间状态 | ❌ 生产关闭 |
| **TRACE** | 极细粒度跟踪 | 每次循环、每个SQL | ❌ 生产关闭 |

### 2.2 结构化日志格式

```json
{
  "timestamp": "2024-01-15T10:30:00.123Z",
  "level": "ERROR",
  "service": "order-service",
  "version": "v1.2.3",
  "trace_id": "abc123",
  "span_id": "def456",
  "message": "Failed to process order",
  "error": {
    "type": "DatabaseError",
    "message": "Connection timeout",
    "stack": "..."
  },
  "context": {
    "order_id": "ORD-001",
    "user_id": "USR-001",
    "action": "create_order"
  }
}
```

### 2.3 日志检查清单

| 检查项 | 要求 | 原因 |
|--------|------|------|
| 格式统一 | JSON 结构化日志 | 便于采集和查询 |
| 时间戳 | ISO 8601 + UTC | 跨时区一致性 |
| Trace ID | 请求级追踪ID | 链路追踪 |
| 服务标识 | 服务名+版本 | 快速定位问题服务 |
| 脱敏 | 不记录密码/密钥/完整身份证号 | 安全合规 |
| 级别合理 | 不把所有日志都标 ERROR | 减少噪音 |
| 采集验证 | 日志能被采集系统正确解析 | 确保可查询 |

## 三、监控与告警

### 3.1 四个黄金指标（Four Golden Signals）

| 指标 | 含义 | 典型监控方式 | 告警示例 |
|------|------|-------------|---------|
| **延迟（Latency）** | 请求处理时间 | P50/P95/P99 延迟分布 | P99 > 2s 持续5分钟 |
| **错误率（Errors）** | 请求失败比例 | 5xx/4xx 比例 | 5xx率 > 1% 持续3分钟 |
| **吞吐量（Traffic）** | 请求量/QPS | 每秒请求数趋势 | QPS 突降 > 50% |
| **饱和度（Saturation）** | 资源使用程度 | CPU/内存/磁盘/连接池 | CPU > 80% 持续10分钟 |

### 3.2 监控层次

| 层次 | 监控内容 | 工具示例 |
|------|---------|---------|
| **基础设施** | CPU/内存/磁盘/网络 | Prometheus Node Exporter |
| **运行时** | GC/线程/连接池 | 语言运行时指标 |
| **应用** | 四个黄金指标 | APM / 自定义指标 |
| **业务** | 订单量/注册量/活跃用户 | 业务埋点 |
| **依赖** | 数据库/缓存/第三方延迟和错误率 | 客户端指标 |

### 3.3 告警设计原则

| 原则 | 说明 | 反例 |
|------|------|------|
| **可行动** | 收到告警后有明确的下一步 | "CPU使用率50%"（然后呢？） |
| **低噪音** | 减少误报和无意义告警 | 每5分钟一条"连接数正常"的通知 |
| **分级** | 不同级别不同通知渠道和响应要求 | 所有告警都发到同一群 |
| **有上下文** | 告警信息包含足够诊断上下文 | "服务异常"（哪个服务？什么异常？） |
| **有抑制** | 同类告警合并，避免告警风暴 | 同一问题5分钟内发了100条通知 |

### 3.4 告警级别与响应

| 级别 | 定义 | 通知渠道 | 响应时间 |
|------|------|---------|---------|
| **P0 紧急** | 核心服务不可用 | 电话 + 短信 + IM | 立即 |
| **P1 严重** | 核心功能降级 / 性能严重下降 | 短信 + IM | 15分钟 |
| **P2 警告** | 非核心功能异常 / 资源接近阈值 | IM | 1小时 |
| **P3 通知** | 信息性告警 / 趋势提醒 | IM（低优先级频道） | 下个工作日 |

### 3.5 核心告警规则参考

| 告警名 | 条件 | 级别 | 说明 |
|--------|------|------|------|
| 服务不可用 | 健康检查连续失败 >3次 | P0 | 服务可能已宕机 |
| 高错误率 | 5xx率 >5% 持续3分钟 | P0 | 大量请求失败 |
| 错误率上升 | 5xx率 >1% 持续5分钟 | P1 | 错误开始增多 |
| 高延迟 | P99 >3s 持续5分钟 | P1 | 响应严重变慢 |
| CPU 过高 | CPU >80% 持续10分钟 | P2 | 资源可能不足 |
| 内存过高 | 内存 >85% 持续10分钟 | P2 | 可能内存泄漏 |
| 磁盘即将满 | 磁盘使用 >90% | P2 | 需要扩容或清理 |
| 证书即将过期 | 证书有效期 <30天 | P2 | 需要续期 |
| QPS 突变 | QPS 变化 >50%（与历史同期比） | P2 | 可能有异常流量 |

## 四、发布观察计划

### 4.1 观察阶段定义

| 阶段 | 时间范围 | 关注重点 | 典型问题 |
|------|---------|---------|---------|
| **即时观察** | 0 - 30 min | 服务启动、健康检查、核心功能 | 启动失败、配置错误、接口不通 |
| **短期观察** | 30 min - 4 h | 业务指标、性能趋势、错误模式 | 性能退化、间歇性错误、内存缓慢上涨 |
| **中期观察** | 4 h - 24 h | 稳定性、资源趋势、边缘场景 | 内存泄漏、慢查询累积、定时任务异常 |
| **长期观察** | 24 h - 72 h | 全周期覆盖、周期性问题 | 高峰期性能、定时任务、数据增长影响 |

### 4.2 各阶段观察指标

#### 即时观察（0-30min）— 必须全部通过

| 指标 | 检查方式 | 通过标准 |
|------|---------|---------|
| 健康检查 | 存活/就绪探针 | 全部 200 OK |
| 核心接口 | Smoke Test | 关键接口返回正确 |
| 错误率 | 监控面板 | ≤ 基线水平 |
| 启动日志 | 日志查看 | 无 ERROR 级别日志 |

#### 短期观察（30min-4h）— 关注趋势

| 指标 | 检查方式 | 关注点 |
|------|---------|--------|
| 错误率趋势 | 监控面板 | 是否在上升 |
| 延迟趋势 | P50/P95/P99 | 是否在恶化 |
| 业务量 | 业务指标面板 | 是否与预期一致 |
| 资源使用 | CPU/内存趋势 | 是否异常增长 |

#### 中期观察（4h-24h）— 关注积累

| 指标 | 检查方式 | 关注点 |
|------|---------|--------|
| 内存趋势 | 资源监控 | 是否持续增长（泄漏） |
| 连接池 | 应用指标 | 是否持续增长（泄漏） |
| 慢查询 | 数据库监控 | 是否出现新的慢查询 |
| 定时任务 | 任务执行日志 | 是否正常执行 |
| 磁盘使用 | 资源监控 | 日志/数据增长是否正常 |

#### 长期观察（24h-72h）— 关注全周期

| 指标 | 检查方式 | 关注点 |
|------|---------|--------|
| 高峰期表现 | 对比历史高峰 | 性能是否可接受 |
| 周期性任务 | 任务日志 | 每日/每周任务是否正常 |
| 错误聚合 | 错误日志分析 | 是否有新的错误模式 |
| 用户反馈 | 反馈渠道 | 是否有相关投诉 |

### 4.3 观察结束标准

```
可以关闭观察窗口的条件（全部满足）：
  |- 所有健康检查持续通过
  |- 错误率在基线范围内
  |- 性能指标在基线范围内
  |- 资源使用趋势正常（无泄漏迹象）
  |- 无未解决的告警
  |- 已覆盖至少一个完整业务周期
  +- 无用户反馈相关问题
```

### 4.4 观察期间异常处理

```
观察期间发现异常
  |
  |- 在"立即回滚"阈值内？ → 触发回滚
  |
  |- 在"准备回滚"范围内？
  |   |- 有快速修复方案？ → 评估 hotfix vs 回滚
  |   +- 无快速修复 → 评估影响面，决定回滚/降级/继续
  |
  |- 在"继续观察"范围内？
  |   |- 记录异常详情
  |   |- 分析可能原因
  |   |- 调整告警阈值（如需要）
  |   +- 继续监控趋势
  |
  +- 所有异常记录到发布后观察日志，供复盘使用
```
