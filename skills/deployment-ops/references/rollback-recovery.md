# 回滚与恢复

## 概览

回滚与恢复覆盖四大领域：
1. **回滚方案设计** — 针对每种变更类型的回滚策略
2. **回滚触发条件** — 何时决定回滚及决策阈值
3. **故障恢复** — 非回滚场景的故障处理
4. **紧急发布流程** — Hotfix 路径和简化流程

---

## 一、回滚方案设计

### 1.1 回滚方案设计原则

| 原则 | 说明 |
|------|------|
| **预先制定** | 回滚方案在发布前制定，不是出问题后临时想 |
| **经过验证** | 回滚步骤在预发环境至少演练一次 |
| **独立可执行** | 回滚不依赖"出问题的那个人还在" |
| **有时限** | 回滚预期耗时明确，超时有升级机制 |
| **完整性** | 覆盖服务回滚、数据回滚、配置回滚 |

### 1.2 按变更类型的回滚策略

#### 纯服务变更（无schema/数据变更）

```
回滚方式: 镜像/容器版本回退
  |
  |- 回退后端服务到前一版本
  |   |- 容器编排: 更新镜像版本 → 滚动更新
  |   |- 传统部署: 切换到前一版本包 → 重启
  |   +- 蓝绿部署: 流量切回蓝环境
  |
  |- 回退前端资源到前一版本
  |   |- CDN: 切换资源指向前一版本
  |   +- 静态服务: 回退到前一版本目录
  |
  +- 验证: 健康检查通过 + 核心链路正常
```

**回滚耗时预期**：1-5分钟（取决于部署方式）

#### 含数据库 Schema 变更

```
回滚方式: down migration + 服务回退
  |
  |- 评估 down migration 可行性
  |   |- 新增列（无数据）→ DROP COLUMN（低风险）
  |   |- 新增表（无数据）→ DROP TABLE（低风险）
  |   |- 列类型变更 → down migration（中风险，可能数据截断）
  |   |- 删除列（已有数据）→ ⚠️ 需从备份恢复（高风险）
  |   +- 大量数据变更 → ⚠️ 需从备份恢复（高风险）
  |
  |- 执行 down migration
  |- 回退服务版本
  +- 验证: schema 正确 + 数据完整 + 服务正常
```

**关键规则**：
- 涉及数据删除的 migration 必须先备份
- down migration 必须在预发环境验证过
- 高风险 schema 变更建议采用"扩展-迁移-收缩"三阶段策略

#### 含数据变更（数据回填/修复）

```
回滚方式: 数据修复脚本 + 服务回退
  |
  |- 数据回填类
  |   |- 回填了新字段 → 清空新字段值（如可接受）
  |   +- 修改了存量数据 → 从变更前备份恢复受影响数据
  |
  |- 数据修复类
  |   |- 有明确的反向操作 → 执行反向修复脚本
  |   +- 无法反向 → 从备份恢复 + 评估数据损失
  |
  +- 验证: 数据一致性校验 + 业务验证
```

**数据回滚红线**：
- 数据变更前必须有备份
- 数据变更脚本必须支持断点续传（大量数据场景）
- 数据回滚脚本必须提前准备，不是出问题后现写

### 1.3 回滚方案文档要素

每个回滚方案必须包含：

| 要素 | 内容 |
|------|------|
| **触发条件** | 什么情况下触发回滚 |
| **决策人** | 谁有权决定回滚 |
| **步骤** | 具体执行步骤（可直接执行） |
| **预期耗时** | 回滚需要多长时间 |
| **验证方式** | 回滚后如何确认恢复成功 |
| **通知机制** | 回滚后通知谁、通知什么 |
| **数据影响** | 回滚是否有数据损失，损失范围 |

## 二、回滚触发条件

### 2.1 立即回滚条件（任一满足即触发）

| 条件 | 判断依据 | 响应时间 |
|------|---------|---------|
| 核心业务功能完全不可用 | 健康检查持续失败 / 用户无法完成核心操作 | 立即 |
| 错误率突破高阈值 | 错误率 >5%（基线的10倍以上） | 5分钟内 |
| 数据一致性受损 | 发现数据丢失/损坏/不一致 | 立即 |
| 安全漏洞暴露 | 发布引入了安全漏洞 | 立即 |
| 级联故障 | 新版本导致依赖服务连锁故障 | 立即 |

### 2.2 准备回滚条件（需评估后决策）

| 条件 | 判断依据 | 观察时间 | 决策方式 |
|------|---------|---------|---------|
| 错误率升高但未达阈值 | 错误率 2%-5% | 15分钟 | 趋势判断 |
| 性能显著下降 | P99 >2x 基线 | 15分钟 | 趋势判断 |
| 部分功能降级 | 非核心功能异常 | 30分钟 | 影响面评估 |
| 资源使用异常增长 | CPU/内存持续攀升 | 30分钟 | 趋势判断 |
| 告警频繁触发 | 同类告警反复出现 | 15分钟 | 原因分析 |

### 2.3 不回滚条件（继续观察）

| 条件 | 处理方式 |
|------|---------|
| 仅告警无用户影响 | 记录 + 继续观察 + 分析原因 |
| 性能轻微波动且趋势稳定 | 记录 + 继续观察 |
| 已知可接受的问题（发布前已评估） | 按预案处理 |
| 误报告警 | 确认误报 + 调整告警规则 |

### 2.4 回滚决策流程

```
检测到异常
  |
  |- 是否满足"立即回滚"条件？
  |   +- YES → 立即执行回滚，同时通知所有相关方
  |
  |- 是否满足"准备回滚"条件？
  |   +- YES → 进入观察评估
  |       |- 启动倒计时（15-30分钟）
  |       |- 收集指标数据
  |       |- 趋势恶化？ → 执行回滚
  |       |- 趋势稳定？ → 继续观察，延长倒计时
  |       +- 趋势好转？ → 取消回滚准备，继续观察
  |
  +- 不满足以上条件 → 记录 + 继续正常观察
```

## 三、故障恢复

### 3.1 非回滚的故障处理

有些故障不需要完整回滚，可以通过其他方式恢复：

| 故障类型 | 恢复方式 | 适用条件 |
|---------|---------|---------|
| **配置错误** | 修正配置 + 重启/热加载 | 问题定位明确，修正成本低 |
| **资源不足** | 扩容（增加实例/提升规格） | 功能正常，只是容量不够 |
| **依赖服务异常** | 等待恢复 / 启用降级 | 本服务无bug，问题在依赖端 |
| **缓存问题** | 清除缓存 + 预热 | 缓存脏数据/过期导致 |
| **连接池耗尽** | 重启服务 / 调整池大小 | 连接泄漏或配置不合理 |

### 3.2 故障分级与响应

| 级别 | 定义 | 响应时间 | 处理方式 |
|------|------|---------|---------|
| **P0 灾难** | 全站不可用 / 数据大规模损坏 | 立即 | 全员响应 + 立即回滚 |
| **P1 严重** | 核心功能不可用 / 部分数据受损 | 5分钟 | 值班响应 + 评估回滚 |
| **P2 一般** | 非核心功能异常 / 性能下降 | 30分钟 | 值班处理 + 评估修复 |
| **P3 轻微** | 日志错误 / 非功能性问题 | 下个工作日 | 记录 + 排期修复 |

### 3.3 故障复盘模板

每次故障（P0/P1）必须进行复盘：

```markdown
## 故障复盘

### 基本信息
- 故障时间：
- 发现时间：
- 恢复时间：
- 影响范围：
- 影响时长：

### 时间线
| 时间 | 事件 | 操作人 |
|------|------|--------|

### 根因分析
- 直接原因：
- 根本原因：

### 处置过程
- 做了什么：
- 效果如何：

### 改进措施
| 措施 | 负责人 | 截止日期 | 类型 |
|------|--------|---------|------|
| {措施} | {人} | {日期} | 修复/预防/检测 |

### 经验教训
- 做得好的：
- 需要改进的：
```

## 四、紧急发布流程

### 4.1 紧急发布触发条件

| 条件 | 说明 |
|------|------|
| P0/P1 线上故障 | 需要通过代码修复解决 |
| 安全漏洞 | 被利用风险高，需紧急修补 |
| 数据损坏 | 需要通过发布修复脚本来恢复 |

### 4.2 紧急发布简化流程

```
常规流程 9 步 → 紧急发布简化为 5 步：

Step 1: 确认故障 + 确认修复方案
  -> 明确修复范围（最小化变更）
  
Step 2: 快速修复 + 快速审查
  -> 代码修复 → 至少1人审查 → 构建

Step 3: 预发验证
  -> 在预发环境验证修复点（不做全量回归）
  
Step 4: 最小化发布检查 + 发布
  -> 构建产物 ✅ + 回滚方案 ✅ + 核心监控 ✅ → 发布

Step 5: 加强观察 + 事后补齐
  -> 加强观察窗口 + 事后补齐完整审查和测试
```

### 4.3 紧急发布红线

即使是紧急发布，以下仍不可省略：

| 不可省略 | 原因 |
|---------|------|
| 回滚方案 | 紧急修复也可能引入新问题 |
| 预发验证 | 至少验证修复点有效且无明显回归 |
| 代码审查（至少1人） | 紧急时更容易犯错 |
| 发布后观察 | 确认修复生效且无副作用 |
| 事后补齐 | 防止紧急流程变成常态 |

### 4.4 紧急发布审批

| 场景 | 审批权限 |
|------|---------|
| P0 故障（全站不可用） | 值班发布指挥可直接执行，事后报备PM |
| P1 故障（核心功能异常） | 需PM审批后执行 |
| 安全漏洞 | 需PM + 安全负责人审批 |

## 回滚演练

### 建议演练频率

| 场景 | 频率 | 方式 |
|------|------|------|
| 服务回滚 | 每月一次 | 在预发环境执行完整回滚流程 |
| 数据库回滚 | 每季度一次 | 在预发环境验证 down migration |
| 全站恢复 | 每半年一次 | 模拟全站故障的完整恢复流程 |

### 演练后检查

- [ ] 回滚步骤是否可执行？
- [ ] 回滚耗时是否在预期内？
- [ ] 回滚后验证是否通过？
- [ ] 回滚文档是否需要更新？
- [ ] 是否发现新的风险点？
