# 发布执行设计

## 概览

发布执行设计覆盖三大领域：
1. **发布步骤设计** — 从构建到上线的完整步骤编排
2. **发布窗口** — 时间选择、维护窗口和时间约束
3. **War Room 模式** — 发布协调、通讯和决策机制

---

## 一、发布步骤设计

### 1.1 步骤编排原则

| 原则 | 说明 | 反例 |
|------|------|------|
| **可重复** | 每个步骤可精确重复执行 | 口头说"把那个服务重启一下" |
| **有序** | 步骤间依赖关系明确，顺序不可随意调整 | 先启动应用再迁移数据库 |
| **可验证** | 每个步骤完成后有验证手段 | 部署完不检查就继续下一步 |
| **可回退** | 关键步骤标记回滚点 | 整个流程走完才发现需要回滚 |
| **有时限** | 每个步骤有预期耗时和超时阈值 | 迁移跑了2小时没人看 |

### 1.2 标准发布步骤模板

```
Phase 0: 发布准备（发布窗口前）
  |- 确认发布前提已通过
  |- 确认 War Room 人员就位
  |- 确认通讯渠道畅通
  |- 确认回滚方案已就位
  +- 宣布开始发布

Phase 1: 数据库变更（如适用）
  |- 备份当前数据库（生产必备）
  |- 执行数据库迁移脚本
  |- 验证迁移结果（schema对比/数据抽检）
  +- 回滚点 RP1：迁移失败 → 执行 down migration → 恢复备份

Phase 2: 后端服务部署
  |- 部署新版本后端服务（灰度/蓝绿/滚动）
  |- 等待服务启动完成（健康检查通过）
  |- 验证后端核心接口可用
  +- 回滚点 RP2：后端异常 → 回退到旧版本镜像

Phase 3: 前端资源部署
  |- 部署新版本前端资源到 CDN/静态服务
  |- 验证资源可访问（版本号/hash校验）
  |- 验证前后端集成（关键页面+核心操作）
  +- 回滚点 RP3：前端异常 → 切回旧版本资源

Phase 4: 配置生效与切流
  |- 更新配置（如有配置变更）
  |- 切流（从旧版本切到新版本）
  |- 验证新版本接收流量正常
  +- 回滚点 RP4：切流异常 → 切回旧版本

Phase 5: 发布后验证
  |- 执行核心链路验证（Smoke Test）
  |- 检查健康检查状态
  |- 检查监控指标基线
  |- 检查日志无异常错误
  +- 验证通过 → 进入观察阶段

Phase 6: 观察与确认
  |- 按观察计划监控指标
  |- 观察窗口内无异常 → 宣布发布成功
  +- 出现异常 -> 按回滚决策树处理
```

### 1.3 发布策略选择

| 策略 | 适用场景 | 优势 | 风险 | 回滚速度 |
|------|---------|------|------|---------|
| **全量发布** | 小变更/配置变更/紧急修复 | 简单快速 | 影响面大 | 需全量回退 |
| **灰度发布** | 功能发布/用户可感知变更 | 可控风险 | 流程较长 | 调整灰度比例 |
| **蓝绿部署** | 需要快速回退的重要发布 | 秒级回退 | 需双倍资源 | 秒级切换 |
| **滚动更新** | 无状态服务的常规更新 | 零停机 | 新旧版本共存 | 停止滚动+回退 |
| **金丝雀发布** | 高风险变更/不确定性大 | 最小影响 | 流程复杂 | 移除金丝雀实例 |

**选择决策**：
```
变更风险高？
  |- YES -> 用户可感知？
  |         |- YES -> 灰度发布（按用户/流量比例）
  |         +- NO  -> 金丝雀发布（先放少量实例）
  +- NO  -> 需要快速回退？
            |- YES -> 蓝绿部署
            +- NO  -> 滚动更新
```

### 1.4 数据库迁移特别注意

| 场景 | 风险等级 | 处理方式 |
|------|---------|---------|
| 新增表/新增列 | 低 | 先迁移后部署，向后兼容 |
| 修改列类型 | 高 | 双写阶段 → 数据迁移 → 切换 → 清理 |
| 删除列/删除表 | 高 | 先部署不使用该列的代码 → 观察 → 再删除列 |
| 大表添加索引 | 中 | 使用 `CREATE INDEX CONCURRENTLY`，避免锁表 |
| 数据回填 | 中 | 分批执行，监控性能影响 |

## 二、发布窗口设计

### 2.1 发布窗口选择原则

| 原则 | 说明 |
|------|------|
| **低峰期** | 选择用户访问量最低的时段 |
| **留出缓冲** | 窗口时长 = 预计发布时间 × 2（含回滚时间） |
| **避开关键期** | 不在业务高峰期、月末结算、大促前发布 |
| **工作时间** | 优先在工作时间发布（便于快速响应） |
| **避免周五** | 周五下午发布 = 周末加班的前奏 |

### 2.2 发布窗口模板

| 窗口类型 | 时间建议 | 适用场景 |
|---------|---------|---------|
| **常规发布窗口** | 工作日 10:00-12:00 或 14:00-16:00 | 标准迭代发布 |
| **维护窗口** | 工作日 22:00-02:00 或 周末 | 需要停机的大变更 |
| **紧急发布窗口** | 随时（但需审批） | 线上事故热修复 |

### 2.3 发布冻结期

| 场景 | 冻结规则 | 例外条件 |
|------|---------|---------|
| 大促/活动期 | 活动前3天至活动结束 | 仅限P0故障hotfix |
| 月末结算 | 月末最后2个工作日 | 仅限P0故障hotfix |
| 长假前 | 假期前1天至假期结束 | 仅限P0故障hotfix |
| 全公司重要事件 | 事件期间 | 仅限P0故障hotfix |

## 三、War Room 模式

### 3.1 何时启用 War Room

| 场景 | 是否启用 |
|------|---------|
| 快速发布（配置变更/小补丁） | ❌ 不需要 |
| 标准发布（常规迭代） | ⚠️ 简化版（通讯渠道+值班人员） |
| 深度发布（大变更/首发/迁移） | ✅ 完整 War Room |
| 线上事故恢复 | ✅ 紧急 War Room |

### 3.2 War Room 角色

| 角色 | 职责 | 必须在场 |
|------|------|---------|
| **发布指挥** | 整体协调、进度控制、最终决策 | ✅ |
| **后端执行** | 后端服务部署和验证 | ✅ |
| **前端执行** | 前端资源部署和验证 | 视变更 |
| **DBA** | 数据库迁移执行和验证 | 视变更 |
| **测试确认** | 核心链路验证 | ✅ |
| **监控值班** | 监控指标和告警 | ✅ |

### 3.3 War Room 通讯规范

| 事项 | 格式 |
|------|------|
| **步骤开始** | `[HH:MM] 🟡 开始执行: {步骤描述}` |
| **步骤完成** | `[HH:MM] ✅ 完成: {步骤描述}，验证结果: {结果}` |
| **步骤失败** | `[HH:MM] ❌ 失败: {步骤描述}，原因: {原因}` |
| **决策点** | `[HH:MM] ⚠️ 需要决策: {问题描述}，选项: A/B` |
| **回滚触发** | `[HH:MM] 🔴 触发回滚: {原因}，执行回滚方案 RP{N}` |
| **发布结论** | `[HH:MM] 📋 发布结论: ✅成功 / ❌已回滚` |

### 3.4 决策授权

| 决策类型 | 授权人 | 时限 |
|---------|--------|------|
| 继续下一步骤 | 发布指挥 | 即时 |
| 暂停观察 | 发布指挥 | 即时 |
| 触发回滚 | 发布指挥 | 5分钟内 |
| 取消回滚（继续观察） | 发布指挥 + PM | 10分钟内 |
| 放弃本次发布 | PM | 不限 |

## 紧急发布（Hotfix）流程

```
线上故障 → 确认需要 Hotfix
  |
  |- 评估: 故障影响、修复范围、风险
  |
  |- 简化流程:
  |   |- 代码修复 → 快速审查（至少1人）
  |   |- 构建 → 预发环境验证核心修复点
  |   |- 最小化发布检查（构建产物+回滚方案+核心监控）
  |   +- 发布执行 + 加强观察
  |
  |- 事后补齐:
  |   |- 完整代码审查
  |   |- 补充测试用例
  |   |- 更新发布文档
  |   +- 故障复盘
```

**Hotfix 红线**：
- 即使是紧急修复，也必须有回滚方案
- 即使是紧急修复，也必须在预发环境验证
- 即使是紧急修复，也必须有人审查代码
- 事后必须补齐完整流程（不能"急的时候跳过，不急了就忘了"）
