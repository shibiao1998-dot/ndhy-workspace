# 教育行业运维特别指南

## 概览

教育行业运维有其独特约束和要求，本文件作为 deployment-ops 技能的行业适配补充，覆盖：
1. **发布窗口约束** — 教育行业的特殊时间限制
2. **学校环境** — 校园网络和设备环境特点
3. **数据备份合规** — 教育数据的特殊保护要求
4. **灰度策略** — 适合教育场景的灰度方案

---

## 一、发布窗口约束

### 1.1 教育行业时间禁区

| 时段 | 原因 | 发布限制 | 例外 |
|------|------|---------|------|
| **考试期间** | 高并发、零容错 | ⛔ 完全冻结 | 仅P0故障hotfix |
| **开学第一周** | 注册/选课高峰 | ⛔ 完全冻结 | 仅P0故障hotfix |
| **期末最后两周** | 成绩录入/查询高峰 | ⚠️ 仅低风险变更 | 需校方确认 |
| **招生季** | 报名/录取关键期 | ⛔ 招生模块冻结 | 其他模块可正常发布 |
| **上课时间（8:00-17:00）** | 师生在线使用 | ⚠️ 避免影响核心教学功能 | 非教学模块可灰度 |

### 1.2 推荐发布窗口

| 窗口 | 时间 | 适用场景 |
|------|------|---------|
| **常规窗口** | 工作日 19:00-22:00 | 标准迭代（课后时段） |
| **维护窗口** | 周六 22:00-周日 06:00 | 大变更/数据迁移 |
| **寒暑假窗口** | 假期工作日 | 大版本升级/架构变更 |
| **紧急窗口** | 随时（需审批） | P0故障hotfix |

### 1.3 学期日历同步

**每学期开始前**，运维必须同步以下日历：
- 开学/放假日期
- 考试周日期
- 招生时间节点
- 大型活动日期（运动会、校庆等）
- 教务系统关键操作期（选课、成绩录入等）

## 二、学校环境特点

### 2.1 校园网络特点

| 特点 | 影响 | 应对 |
|------|------|------|
| **带宽限制** | CDN/大文件下载受限 | 资源压缩、按需加载、校内CDN节点 |
| **防火墙严格** | 外部服务可能不通 | 提前确认白名单、使用国内服务 |
| **高峰并发集中** | 上课/下课时段流量暴涨 | 自动扩缩容、预热、限流策略 |
| **多校区网络隔离** | 校区间访问受限 | 确认跨校区连通性 |
| **Wi-Fi不稳定** | 移动端体验差 | 断线重连、数据本地缓存 |

### 2.2 终端设备多样性

| 设备类型 | 使用场景 | 注意事项 |
|---------|---------|---------|
| 教室多媒体一体机 | 教学 | 系统版本老旧、浏览器版本低 |
| 学生平板/笔记本 | 学习 | 型号多样、系统版本分散 |
| 教师办公电脑 | 管理 | Windows为主、可能有安全软件干扰 |
| 手机（家长端） | 家校沟通 | 微信内嵌浏览器、Android碎片化 |

### 2.3 学校环境部署注意

| 场景 | 注意事项 |
|------|---------|
| **校内服务器** | 物理环境（机房温度/UPS）、远程维护通道、假期断电风险 |
| **教育云** | 教育网专线、等保要求、数据不出境 |
| **混合部署** | 校内+云端的网络连通性、数据同步延迟 |

## 三、数据备份与合规

### 3.1 教育数据分类

| 数据类型 | 敏感度 | 备份要求 | 保留期限 |
|---------|--------|---------|---------|
| **学生个人信息** | 🔴 极高 | 加密备份 + 异地存储 | 按法规（通常毕业后6年） |
| **成绩/学籍** | 🔴 极高 | 每日备份 + 操作审计 | 永久归档 |
| **教学资源** | 🟡 中 | 定期备份 | 按课程周期 |
| **系统配置** | 🟢 低 | 变更前备份 | 3个月 |
| **日志数据** | 🟡 中 | 自动轮转 | 6个月（审计日志1年以上） |

### 3.2 发布前数据备份要求

| 变更类型 | 备份范围 | 备份方式 | 恢复验证 |
|---------|---------|---------|---------|
| Schema变更 | 受影响表 + 关联表 | 全量备份 | 必须验证可恢复 |
| 数据迁移 | 迁移涉及的所有数据 | 全量备份 | 必须验证可恢复 |
| 常规发布 | 核心业务表 | 增量备份 | 抽检验证 |
| 配置变更 | 配置快照 | 版本控制 | 不需要 |

### 3.3 合规要求

| 要求 | 说明 | 落地方式 |
|------|------|---------|
| **数据不出境** | 学生数据不得存储在境外服务器 | 使用国内云服务、确认CDN节点 |
| **等保合规** | 教育系统通常需要等保二级以上 | 按等保要求配置安全策略 |
| **操作审计** | 关键数据操作必须有审计日志 | 确保审计日志采集完整 |
| **数据脱敏** | 非生产环境不得使用真实学生数据 | 测试数据脱敏/造假 |
| **家长知情** | 收集学生信息需家长同意 | 隐私政策更新需同步通知 |

## 四、教育场景灰度策略

### 4.1 灰度维度

| 维度 | 适用场景 | 示例 |
|------|---------|------|
| **按学校** | 多校部署 | 先在1-2所试点校发布 |
| **按年级** | 功能差异大 | 先在非毕业班发布 |
| **按角色** | 不同角色差异 | 先放教师端，再放学生端 |
| **按校区** | 多校区 | 先在分校区，再到主校区 |
| **按功能** | Feature Flag | 新功能灰度开放 |

### 4.2 推荐灰度流程

```
教育产品灰度发布：

Phase 1: 内部验证（1-2天）
  |- 内部测试环境完整验证
  +- 产品团队内部试用

Phase 2: 试点学校（3-5天）
  |- 选择1-2所试点学校
  |- 选择非核心时段发布
  |- 重点收集教师反馈
  +- 监控性能和异常

Phase 3: 扩大范围（3-5天）
  |- 扩展到更多学校（30-50%）
  |- 覆盖不同类型学校（小学/中学/高中）
  +- 监控全量指标

Phase 4: 全量发布
  |- 确认前几阶段无重大问题
  |- 选择合适窗口全量发布
  +- 加强观察至少72小时
```

### 4.3 灰度回退策略

| 阶段 | 回退方式 | 影响范围 |
|------|---------|---------|
| Phase 1 | 直接回退 | 仅内部 |
| Phase 2 | 试点校回退 | 少量学校 |
| Phase 3 | 按学校逐步回退 | 部分学校 |
| Phase 4 | 全量回退 | 所有学校 |

## 五、教育行业发布检查附加项

在常规发布检查清单基础上，教育项目需额外检查：

| # | 检查项 | 通过标准 |
|---|--------|---------|
| E1 | 发布窗口避开教育时间禁区 | 不在考试/开学/招生关键期 |
| E2 | 学校网络连通性 | 校园网可正常访问新版本 |
| E3 | 浏览器兼容性 | 支持学校常见浏览器版本 |
| E4 | 数据备份完成 | 学生数据/成绩数据已备份 |
| E5 | 家长端兼容 | 微信内嵌浏览器/主流手机可正常使用 |
| E6 | 离线/弱网降级 | Wi-Fi不稳定时核心功能可用 |
| E7 | 灰度策略确定 | 已选择灰度维度和试点范围 |
| E8 | 学校通知 | 已通知受影响学校（如需要） |
| E9 | 数据合规 | 无数据出境风险、等保配置正确 |

## 六、教育行业典型风险场景

| 场景 | 风险 | 应对 |
|------|------|------|
| **考试系统发布后卡顿** | 影响考试公平性 | 冻结期制度 + 压测 + 扩缩容预案 |
| **成绩系统数据丢失** | 极严重后果 | 加密备份 + 操作审计 + 即时回滚 |
| **招生数据泄露** | 法律/声誉风险 | 数据加密 + 权限最小化 + 审计 |
| **假期断电导致服务中断** | 学校机房无人值守 | UPS + 云端备份 + 远程监控 |
| **大班额并发** | 上课高峰崩溃 | 限流 + 缓存 + 自动扩缩容 |
