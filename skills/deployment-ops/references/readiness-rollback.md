# 发布就绪判断框架 & 回滚决策树

## 发布就绪判断框架

```
发布就绪评估
  |
  |- 前提检查（全部必须通过）
  |   |- 构建产物完整且版本正确？
  |   |- 联调结论为"通过"？
  |   |- 测试结论为"可发布"？
  |   |- 代码审查无阻断项？
  |   +- 全部 YES -> 继续 | 任一 NO -> ⛔ 阻塞，回流上游
  |
  |- 环境与配置检查（全部必须通过）
  |   |- 环境一致性已确认？
  |   |- 配置/密钥/权限齐备？
  |   |- 外部依赖可达？
  |   +- 全部 YES -> 继续 | 任一 NO -> ⛔ 阻塞，补齐后重检
  |
  |- 保障能力检查（按项评估）
  |   |- 回滚方案已制定且可执行？
  |   |- 健康检查已覆盖？
  |   |- 监控告警已配置？
  |   |- 发布观察计划已制定？
  |   +- 全部 YES -> ✅ 具备发布条件
  |   +- 部分 NO -> ⚠️ 带风险发布（须明示风险 + PM审批）
  |   +- 关键项 NO -> ⛔ 阻塞
  |
  -> 输出: ✅ 放行 | ⚠️ 带风险发布 | ⛔ 阻塞
```

### 判断规则

| 结论 | 条件 | 要求 |
|------|------|------|
| ✅ **放行** | 前提+环境+保障全部通过 | 直接发布 |
| ⚠️ **带风险发布** | 前提+环境通过，保障有非关键缺口 | 明示风险 + PM审批 + 加强观察 |
| ⛔ **阻塞** | 前提或环境未通过，或保障有关键缺口 | 回流上游/补齐后重新评估 |

## 回滚决策树

```
发布后检测到异常
  |
  |- 异常影响面评估
  |   |- 核心业务功能不可用？ -> 立即回滚
  |   |- 错误率超过阈值（如 >5%）？ -> 立即回滚
  |   |- 响应延迟严重恶化（如 P99 >3x 基线）？ -> 立即回滚
  |   |- 数据一致性受损？ -> 立即回滚 + 紧急数据修复
  |   |
  |   |- 非核心功能异常？
  |   |   |- 有快速修复方案（<30min）？ -> 尝试 hotfix
  |   |   |- 无快速修复 -> 评估降级/回滚
  |   |
  |   |- 性能轻微下降但在可接受范围？
  |   |   |- 趋势恶化？ -> 准备回滚
  |   |   +- 趋势稳定 -> 继续观察 + 记录
  |   |
  |   +- 仅告警无用户影响？ -> 记录 + 继续观察
  |
  |- 回滚执行
  |   |- 纯服务回滚（无schema变更）-> 镜像/容器版本回退
  |   |- 含 schema 变更 -> 执行 down migration + 服务回退
  |   |- 含数据变更 -> 数据修复脚本 + 服务回退
  |   +- 回滚后验证 -> 健康检查 + 核心链路验证
  |
  +- 回滚后复盘 -> 记录原因 + 回流相关专家
```

### 回滚触发阈值（参考值，按项目调整）

| 指标 | 立即回滚 | 准备回滚 | 继续观察 |
|------|---------|---------|---------|
| 错误率 | >5% | 2%-5% | <2% |
| P99 延迟 | >3x 基线 | 2x-3x 基线 | <2x 基线 |
| 核心功能 | 不可用 | 降级 | 正常 |
| 健康检查 | 持续失败 | 间歇失败 | 正常 |
