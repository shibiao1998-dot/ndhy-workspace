# 发布前提校验

## 概览

发布前提校验覆盖三大领域：
1. **发布前提** — 上游交付结论和构建产物完整性
2. **环境一致性** — 跨环境配置对齐和基础设施就绪
3. **配置/密钥/权限** — 环境变量、凭证和访问控制

---

## 一、发布前提校验

### 1.1 上游交付结论检查

发布前必须确认以下上游结论：

| 结论来源 | 要求状态 | 检查方式 | 缺失时处理 |
|---------|---------|---------|-----------|
| 联调结论 | "通过" | 要求联调专家提供结论文档 | 回流联调专家，不猜测 |
| 测试结论 | "可发布" | 要求测试专家提供结论文档 | 回流测试专家，不假设 |
| 代码审查 | "无阻断项" | 要求审查专家确认 | 回流审查专家 |
| 已知缺陷评估 | 已评估且可接受 | 测试专家提供缺陷清单和评估 | 同步PM确认可接受范围 |

**铁律**：
- 上游结论必须是明确的文档/结论，不接受口头"应该没问题"
- 缺少任何一项上游结论 → 发布前提不满足 → 不进入后续步骤
- 不替上游做质量判断（如"测试没说，应该是通过了"）

### 1.2 构建产物完整性检查

```
构建产物检查清单：
  |- 前端构建产物
  |   |- 构建版本是否与发布版本一致？
  |   |- 产物 hash 是否可校验？
  |   |- 是否已推送到目标 CDN/存储？
  |   +- 静态资源路径是否正确？
  |
  |- 后端构建产物
  |   |- 镜像/包版本是否与发布版本一致？
  |   |- 镜像是否已推送到目标镜像库？
  |   +- 依赖版本是否锁定（lock file）？
  |
  |- 数据库迁移（如适用）
  |   |- 迁移脚本版本是否与代码版本匹配？
  |   |- up/down 迁移是否都存在？
  |   +- 迁移脚本是否在预发环境验证过？
  |
  +- 配置文件
      |- 配置模板是否与代码版本匹配？
      +- 新增配置项是否已在目标环境配置？
```

### 1.3 依赖服务可达性检查

| 检查项 | 检查方式 | 通过标准 | 失败处理 |
|--------|---------|---------|---------|
| 数据库连通性 | 连接测试 | 可连接且权限正确 | 检查网络策略/凭证 |
| 缓存服务 | ping/连接测试 | 可连接且读写正常 | 检查网络策略/配置 |
| 消息队列 | 生产/消费测试 | 可收发消息 | 检查网络策略/权限 |
| 第三方API | 健康检查/ping | 返回正常响应 | 确认服务状态/网络 |
| 内部微服务 | 健康检查端点 | 返回 healthy | 确认服务部署状态 |

## 二、环境一致性校验

### 2.1 环境对齐原则

**核心原则**：生产环境不应出现"惊喜"。开发→预发→生产的差异必须是已知的、有意的、已记录的。

### 2.2 环境对比维度

| 维度 | 对比内容 | 常见差异原因 | 处理方式 |
|------|---------|-------------|---------|
| **运行时版本** | 语言版本/框架版本/系统版本 | 环境搭建时间不同 | 对齐或记录差异风险 |
| **依赖版本** | 第三方包/内部SDK版本 | lock file 未同步 | 使用相同 lock file |
| **配置值** | 环境变量/配置文件 | 环境特定配置 | 逐项确认差异合理性 |
| **基础设施** | 实例规格/存储/网络 | 成本/规模差异 | 记录性能影响风险 |
| **数据规模** | 数据量/数据分布 | 测试数据 vs 生产数据 | 评估性能影响 |
| **网络拓扑** | 防火墙/代理/DNS | 环境隔离策略 | 逐项确认连通性 |

### 2.3 环境一致性检查流程

```
Step 1: 收集环境配置快照
  |  开发环境配置 + 预发环境配置 + 生产环境配置
  |
Step 2: 逐维度对比
  |  标记：✅ 一致 | ⚠️ 有差异（已知可接受） | ❌ 有差异（需处理）
  |
Step 3: 差异分类
  |  |- 环境特定差异（如数据库连接串）→ ✅ 确认值正确即可
  |  |- 版本差异 → ⚠️ 评估兼容性风险
  |  |- 配置缺失 → ❌ 必须补齐
  |  +- 功能差异（如生产启用了开发未启用的特性）→ ⚠️ 评估影响
  |
Step 4: 输出环境一致性报告
     标记所有差异项 + 处理状态 + 风险评估
```

### 2.4 常见环境不一致陷阱

| 陷阱 | 表现 | 防范 |
|------|------|------|
| **"开发能跑"假象** | 开发环境宽松，生产环境严格（权限/网络/资源） | 预发环境尽量模拟生产约束 |
| **配置漂移** | 生产环境被手工修改后，与配置管理不一致 | 配置变更走版本控制 |
| **依赖隐式升级** | 未锁定版本导致不同环境拉到不同版本 | 使用 lock file + 固定版本 |
| **时区差异** | 环境时区不同导致定时任务/日志时间混乱 | 统一使用 UTC |
| **DNS/域名差异** | 开发用 localhost，生产用域名 | 配置化 + 环境变量 |

## 三、配置/密钥/权限检查

### 3.1 环境变量检查

```
环境变量检查流程：
  |- 列出所有需要的环境变量（从代码中提取或从配置模板获取）
  |- 逐个确认目标环境是否已配置
  |- 检查变量值是否正确（格式/范围/引用）
  |- 检查新增变量是否已在所有环境配置
  |- 检查废弃变量是否已清理
  +- 输出环境变量对比表
```

**分类管理**：

| 类别 | 示例 | 敏感度 | 管理方式 |
|------|------|--------|---------|
| **业务配置** | 功能开关、分页大小 | 低 | 配置中心/环境变量 |
| **服务连接** | 数据库URL、API地址 | 中 | 配置中心 + 环境隔离 |
| **凭证密钥** | API Key、数据库密码 | 高 | 密钥管理服务（Vault等） |
| **证书** | TLS证书、签名证书 | 高 | 证书管理 + 自动轮转 |

### 3.2 密钥与证书检查

| 检查项 | 检查内容 | 通过标准 | 失败处理 |
|--------|---------|---------|---------|
| API Key | 是否存在、是否有效 | 调用验证通过 | 联系服务提供方 |
| 数据库密码 | 是否正确、权限是否足够 | 连接测试通过 | 确认凭证/权限 |
| TLS证书 | 是否有效、是否过期 | 距过期 >30天 | 续期或更换 |
| JWT密钥 | 是否配置、是否与签名端一致 | 签名验证通过 | 对齐配置 |
| SSH密钥 | 是否部署、权限是否正确 | ssh连接测试通过 | 部署密钥 |

**密钥安全红线**：
- ❌ 密钥不得硬编码在代码中
- ❌ 密钥不得出现在日志中
- ❌ 密钥不得在不加密的通道传输
- ❌ 不同环境不得共用同一密钥
- ✅ 密钥通过密钥管理服务或加密环境变量管理

### 3.3 权限检查

| 权限类型 | 检查内容 | 最小权限原则 |
|---------|---------|-------------|
| **部署账号** | 能否推送镜像、重启服务、执行迁移 | 只授予发布所需权限 |
| **数据库账号** | 读写权限、DDL权限（迁移用） | 应用账号只读写，迁移账号有DDL |
| **云服务权限** | 资源操作权限（如K8s RBAC） | 按namespace/资源类型限制 |
| **第三方服务** | API调用权限、速率限制 | 确认配额充足 |

### 3.4 配置变更审计

每次发布涉及的配置变更必须记录：

| 变更项 | 变更前 | 变更后 | 变更原因 | 审批人 |
|--------|--------|--------|---------|--------|
| {配置项} | {旧值} | {新值} | {原因} | {审批人} |

## 前提校验决策树

```
开始前提校验
  |
  |- 上游结论齐备？
  |   |- NO -> ⛔ 回流上游，列出缺失项
  |   +- YES -> 继续
  |
  |- 构建产物完整？
  |   |- NO -> ⛔ 回流构建，列出缺失项
  |   +- YES -> 继续
  |
  |- 环境一致性确认？
  |   |- 有阻塞差异 -> ⛔ 修复后重检
  |   |- 有非阻塞差异 -> ⚠️ 记录风险，继续
  |   +- 全部一致 -> 继续
  |
  |- 配置/密钥/权限齐备？
  |   |- 有缺失 -> ⛔ 补齐后重检
  |   +- 全部就位 -> 继续
  |
  |- 依赖服务可达？
  |   |- 有不可达 -> ⛔ 排查后重检
  |   +- 全部可达 -> ✅ 前提校验通过
```
