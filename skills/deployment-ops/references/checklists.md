## 二、反模式详细防范

### 反模式 1：手工上线不可重复

**表现**：每次发布都是临时拼凑命令，依赖"那个人还记得步骤"。

**危害**：
- 换人就出错
- 无法审查和改进
- 每次发布都是一次冒险

**防范**：
- 发布步骤必须文档化，可供任何人执行
- 关键步骤脚本化
- 定期审查和更新发布文档

### 反模式 2：只关注能部署，不关注能回滚

**表现**：精力全放在"怎么上去"，回滚方案一笔带过甚至没有。

**危害**：
- 出问题后手忙脚乱
- 回滚耗时远超预期
- 可能造成不可逆的数据损失

**防范**：
- 回滚方案与发布方案同等重要，同时制定
- 回滚方案必须在预发环境验证
- 涉及数据变更的回滚方案必须有数据恢复步骤

### 反模式 3：没有监控和告警就上线

**表现**："先上线，监控后面再补"。

**危害**：
- 问题发生后无法及时发现
- 发现问题后无法快速定位
- 用户先于运维发现问题

**防范**：
- 监控和告警是发布前提，不是发布后的加分项
- 至少覆盖四个黄金指标
- 告警通知渠道必须验证过

### 反模式 4：环境配置临时现场补

**表现**：发布时才发现生产缺配置，现场临时加。

**危害**：
- 配置值可能错误
- 绕过了审批流程
- 配置管理失控

**防范**：
- 环境配置提前对齐，走版本控制
- 生产环境配置变更必须走审批
- 发布前逐项确认配置完整性

### 反模式 5：发布前不检查依赖

**表现**：假设依赖服务一定正常，不做连通性检查。

**危害**：
- 发布后才发现依赖不通
- 问题归因困难（是新代码问题还是依赖问题？）
- 浪费发布窗口

**防范**：
- 发布前逐项校验依赖可达性
- 记录依赖服务基线状态
- 区分"依赖问题"和"代码问题"

### 反模式 6：测试通过等于上线安全

**表现**：测试说通过了，所以直接上线，不再检查其他。

**危害**：
- 测试环境与生产环境不一致
- 测试未覆盖的运行时问题
- 忽略了运维层面的风险

**防范**：
- 测试结论是发布前提之一，不是唯一前提
- 还需校验环境、配置、可观测性、回滚能力
- 区分"测试可接受"和"运行可接受"

### 反模式 7：上线后没有观察窗口

**表现**：发布完就散场，没人盯指标。

**危害**：
- 问题延迟发现
- 错过最佳回滚时机
- 小问题累积成大故障

**防范**：
- 每次发布必须有明确的观察计划
- 观察窗口内有人值守
- 定义观察结束标准

### 反模式 8：运行问题不回流

**表现**：发现问题后自己扛，不通知相关专家。

**危害**：
- 问题根因未解决
- 同类问题反复出现
- 运维压力持续增大

**防范**：
- 运行问题按归属分类回流
- 建立回流闭环（反馈→确认→修复→验证）
- 定期汇总运行反馈

### 反模式 9：先发再说掩盖债务

**表现**：明知环境/配置有问题，但"先上去再说"。

**危害**：
- 技术债务持续累积
- 运行稳定性逐步恶化
- 紧急情况下没有余量

**防范**：
- 环境和配置问题必须发布前解决
- 不得已带风险发布时，必须明确记录债务和解决计划
- 债务有跟踪和解决时限

## 三、发布风险评估框架

### 3.1 风险评估维度

| 维度 | 高风险 | 中风险 | 低风险 |
|------|--------|--------|--------|
| **变更范围** | 多模块/核心流程 | 单模块/非核心流程 | 配置变更/文案修改 |
| **数据影响** | schema变更/数据迁移 | 数据回填/新增字段 | 无数据变更 |
| **用户感知** | 核心功能/关键路径 | 辅助功能 | 后台/不可见 |
| **回滚难度** | 涉及数据回滚 | 服务版本回退 | 配置回退 |
| **依赖变更** | 新增外部依赖 | 升级已有依赖 | 无依赖变更 |
| **首次发布** | 全新服务/全新环境 | 新模块但环境已有 | 常规迭代 |

### 3.2 风险等级判定

```
风险等级 = max(所有维度的风险等级)

P0 阻断: 任一维度为"高风险"且无有效应对
P1 严重: 任一维度为"高风险"但有应对方案
P2 一般: 最高为"中风险"
P3 轻微: 全部为"低风险"
```

### 3.3 风险处置矩阵

| 风险等级 | 处置要求 |
|---------|---------|
| **P0 阻断** | 解决后方可发布。回流上游或补齐缺失能力 |
| **P1 严重** | 制定应对方案 + PM审批 + 加强观察 |
| **P2 一般** | 记录风险 + 制定观察计划 |
| **P3 轻微** | 记录即可 |

## 四、Readiness 汇总方法

### 4.1 汇总流程

```
Step 1: 逐类别检查
  |  6大类检查清单逐项标记 ✅/❌/⚠️
  |
Step 2: 统计结果
  |  ✅ 通过项数 / 总项数
  |  ❌ 阻塞项列表
  |  ⚠️ 风险项列表
  |
Step 3: 风险评估
  |  评估所有风险项的影响和应对
  |
Step 4: 输出结论
  |  |- 全部 ✅ → ✅ 放行
  |  |- 有 ⚠️ 无 ❌ → ⚠️ 带风险发布（需PM审批）
  |  +- 有 ❌ → ⛔ 阻塞（列出阻塞项和解决建议）
```

### 4.2 Readiness 报告模板

```markdown
## 发布 Readiness 报告

### 总览
- 检查项: {N} 项
- 通过: {N} 项 ✅
- 风险: {N} 项 ⚠️
- 阻塞: {N} 项 ❌

### 结论: ✅ 放行 | ⚠️ 带风险发布 | ⛔ 阻塞

### 阻塞项（如有）
| # | 检查项 | 问题描述 | 解决建议 | 归属 |
|---|--------|---------|---------|------|

### 风险项（如有）
| # | 检查项 | 风险描述 | 应对措施 | 影响评估 |
|---|--------|---------|---------|---------|

### 建议
{具体建议}
```

## 五、发布后异常跟踪

### 5.1 异常记录格式

| 字段 | 内容 |
|------|------|
| **时间** | 发现时间（精确到分钟） |
| **现象** | 具体表现（指标/日志/用户反馈） |
| **影响面** | 影响的功能/用户范围 |
| **严重度** | P0/P1/P2/P3 |
| **处置** | 采取的措施 |
| **结果** | 处置效果 |
| **根因** | 初步判断的原因 |
| **归属** | 应回流给哪个专家 |

### 5.2 异常跟踪闭环

```
发现异常 → 记录 → 初步处置 → 归因
  |
  |- 运维层问题（配置/环境/资源）→ 运维处理
  |- 后端代码问题 → 回流后端专家
  |- 前端资源问题 → 回流前端专家
  |- 数据问题 → 回流数据库专家
  |- 集成问题 → 回流联调专家
  +- 归因不清 → 协同分析
```

## 六、运行反馈回流清单

### 6.1 回流清单格式

| # | 问题描述 | 发现时间 | 严重度 | 归属专家 | 建议措施 | 跟踪状态 |
|---|---------|---------|--------|---------|---------|---------|
| 1 | {描述} | {时间} | P0-P3 | {专家} | {措施} | 待处理/处理中/已解决 |

### 6.2 回流分类规则

| 问题类别 | 回流给 | 示例 |
|---------|--------|------|
| 服务依赖/健康检查 | 后端开发专家 | 健康检查端点缺失、服务启动慢 |
| 日志格式/级别 | 后端开发专家 | 日志缺少trace_id、ERROR级别滥用 |
| 静态资源/CDN | 前端开发专家 | 资源404、缓存策略不当 |
| 前端监控 | 前端开发专家 | 前端错误未上报、性能数据缺失 |
| 链路集成 | 联调集成专家 | 环境间链路不通、接口不兼容 |
| 数据异常 | 数据库设计专家 | 数据不一致、慢查询 |
| 测试覆盖 | 测试专家 | 生产出现测试未覆盖的场景 |
| 架构瓶颈 | 技术架构专家 | 系统性性能瓶颈、容量规划 |
| 发布/排期 | 项目管理专家 | 发布阻塞、范围调整、延期 |

### 6.3 回流闭环跟踪

```
回流闭环：
  发布运维 → 提交回流清单（问题+建议）
    → 归属专家确认接收
    → 归属专家处理
    → 发布运维验证修复效果
    → 关闭跟踪项
```

**规则**：
- 每个回流项必须有归属专家
- 归属专家必须在 2 个工作日内确认
- 超过 5 个工作日未处理的回流项 → 升级给项目管理专家
