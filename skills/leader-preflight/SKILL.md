# Leader 预检技能（需求处理门控 V2）

> 收到任何需求时，Leader 必须按此技能执行预检流程。没有制品 = 门控没过 = 不能进下一步。

## 总览

```
需求进入 → G0 分级 → [按级别走对应门控] → 执行 → G5 交付
```

| 级别 | 门控路径 | 判断标准 |
|------|---------|---------|
| ⚡ 快速 | G0 → G3 → G5 | 单专家、有先例、边界明确 |
| 🔧 标准 | G0 → G1 → G2 → G3 → G5 | 2-3 专家、有依赖 |
| 🏗️ 复杂 | G0 → G1 → G1.5 → G2 → G3 → G4 → G5 | 4+ 专家、跨领域 |

**防误判自动升级**：
- 涉及 2+ 专业领域 → 自动升"标准"
- 涉及代码开发 → 最低"标准"
- 涉及 UI/UX → 最低"标准"
- 老板说"很重要"或"好好做" → 自动"复杂"

## G0 需求接收卡（所有级别必经）

收到需求后，**第一件事**输出此卡片给老板：

```
📋 需求接收卡

【需求原文】：{老板说的原话，不改写}
【我的理解】：{一句话概括核心意图}
【分级判断】：⚡快速 / 🔧标准 / 🏗️复杂
【分级理由】：{为什么是这个级别，引用防误判规则}
【涉及领域】：{列出专业领域}
【对应专家】：{从 EXPERTS.md 查，列表}
【能力缺口】：{有/无。有则说明缺什么}
【下一步】：{根据级别，下一个 Gate 是什么}

→ 等待老板确认分级后继续
```

**禁止行为**：
- ❌ 跳过此卡直接开干
- ❌ 在卡片中就开始方案设计
- ❌ 把"标准"判成"快速"来省事

## G1 需求理解（标准 + 复杂）

**执行**：spawn 🎯 需求分析专家，与老板深度对话挖掘真实需求。

**产出**：结构化需求文档，必须包含：
- [ ] 核心目标（做了之后要达成什么）
- [ ] 用户场景（谁在什么情况下用）
- [ ] 期望产出（最终交付物是什么样的）
- [ ] 边界（不做什么）
- [ ] 验收标准（怎么判断做完了，必须 SMART 化）
- [ ] 参考/标杆（有没有类似的东西可以参考）

**通过条件**：老板确认需求文档准确

---

## G1.5 对抗审查（仅复杂级别）

**执行**：spawn 📐 产品定义专家，阅读 G1 需求文档后提出质疑。

**规则**：
- 必须提出 ≥3 个质疑点
- 质疑维度：可行性、范围蔓延风险、优先级、遗漏场景、隐含假设
- 需求分析专家回应
- 最多 2 轮，Leader 整合分歧

**产出**：对抗记录（质疑点 → 回应 → 结论/待裁决）

## G4 执行监控（复杂级 + ACP 任务）

Leader 在执行阶段的必做动作：

1. spawn 后立即写入 HEARTBEAT.md
2. ACP 任务：定期读 `streamLogPath` 检查进度
3. 每 5 分钟巡查一次文件产出
4. 10 分钟无产出 → sessions_send 追问 → 无响应则终止
5. 发现偏差 → sessions_send 纠偏

## 指令包 7 层检查单

组装每个 spawn 的 task 时，逐层自检：

- [ ] **L1 项目上下文**：项目名、目标、当前阶段——写了吗？
- [ ] **L2 上游产出物**：前序专家的产出文件路径——附了吗？内容最新吗？
- [ ] **L3 任务说明**："做什么"和"不做什么"——都写清楚了吗？
- [ ] **L4 验收标准**：能客观判断通过/不通过吗？SMART 化了吗？
- [ ] **L5 约束与禁区**：容易踩的坑、前序踩过的坑——写了吗？
- [ ] **L6 必须使用的技能**：查了 STANDARDS.md 吗？适用技能都列了吗？
- [ ] **L7 执行模式**：ACP 还是 subagent？ACP 的三条铁律 + streamTo 写了吗？

**自检信号**：
- L2 为空 → 前序步骤被跳过，**停下来回退**
- L4 写不出来 → 需求不够清晰，**回退到 G1**
- L6 找不到技能 → 考虑是否需要创建新技能
