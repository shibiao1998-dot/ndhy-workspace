# 原型评审流程

> 原型做出来不是终点——通过评审拿到反馈、迭代修改、最终确认才是目标。

## 评审流程

```
原型完成 → 自检通过 → 提交评审 → PM 浏览器预览 → 收集反馈 → 迭代修改 → 确认交付
```

### 评审前准备

| 准备项 | 说明 |
|--------|------|
| 自检通过 | 对照 checklist 全部 ✅ |
| 预览方式 | 单文件 HTML → 直接发文件；React → 部署到预览环境或本地启动 |
| 状态说明 | 附带状态切换操作说明（原型右下角的控制按钮） |
| 已知限制 | 列出原型简化了什么（不影响评审的简化项） |

### 评审关注点

| 评审维度 | 评审者关注什么 | 示例反馈 |
|---------|--------------|---------|
| 布局结构 | 信息层级是否正确、区块划分是否合理 | "统计卡片应该放在最上面" |
| 视觉效果 | 颜色/字号/间距是否协调 | "标题和正文的对比度不够" |
| 响应式 | 移动端布局是否可用 | "手机上卡片太挤了" |
| 全状态 | 异常状态是否被考虑 | "空状态没有引导按钮" |
| 交互感受 | hover/点击反馈是否自然 | "按钮 hover 效果不够明显" |
| 内容真实感 | 假数据是否像真的 | "用户名不要用 test1/test2" |

### 反馈记录模板

```markdown
## 原型评审反馈 - [页面名称]

**评审日期**：YYYY-MM-DD
**评审者**：PM / 老板

### 修改项

| # | 类型 | 描述 | 优先级 | 状态 |
|---|------|------|--------|------|
| 1 | 布局 | 统计卡片移到页面顶部 | P0 | ⬜ |
| 2 | 视觉 | 标题字号增大到 24px | P1 | ⬜ |
| 3 | 状态 | 空状态增加"创建"引导按钮 | P0 | ⬜ |
| 4 | 交互 | 卡片 hover 增加阴影效果 | P2 | ⬜ |

### 确认项（无需修改）
- ✅ 整体配色方案 OK
- ✅ 导航结构 OK
- ✅ 移动端基本可用
```

## 迭代规则

| 规则 | 说明 |
|------|------|
| **最多 3 轮** | 原型评审不超过 3 轮迭代，超过说明设计方案有根本问题 |
| **P0 必改** | 影响核心功能/体验的问题必须修改 |
| **P2 可延后** | 细节优化可以推到前端实现阶段处理 |
| **记录变更** | 每轮修改的内容都记录在反馈文档中 |
| **截图对比** | 修改前后截图对比，方便确认 |

## 交付给前端开发

### 交付物清单

| 交付物 | 格式 | 用途 |
|--------|------|------|
| 原型文件 | HTML / React | 视觉参考和交互参考 |
| Design Tokens | CSS / JSON | 前端直接使用 |
| 评审反馈（已确认） | Markdown | 最终确认的设计方案 |
| 状态说明 | Markdown | 全部状态列表和预期表现 |
| 响应式方案 | Markdown | 各断点的布局规则 |

### 交接说明模板

```markdown
## 原型交付说明 - [页面名称]

### 原型文件
- 路径：`prototypes/dashboard.html`
- 打开方式：浏览器直接打开

### Design Tokens
- 路径：`tokens/tokens.css`
- 使用方式：`<link>` 引入或复制到项目

### 布局要点
1. 栅格：Desktop 12列（间距24px），Mobile 4列单列
2. 最大宽度：1280px，居中
3. Sidebar：Desktop 展开(240px) / Mobile 汉堡菜单

### 状态覆盖
- default：正常数据展示
- loading：骨架屏（保持布局一致）
- empty：居中图标+说明+创建按钮
- error：错误说明+重试按钮

### 已知简化（原型与生产代码的差异）
- 假数据硬编码，前端需要接入真实 API
- 点击事件用 alert() 模拟，前端替换为路由跳转
- 表单无验证逻辑，前端需要实现
- 动画效果简化，前端可以增强

### 注意事项
- 所有颜色/间距/圆角使用 CSS 变量，不要硬编码
- 移动端 320px 宽度已测试通过
- 焦点顺序已验证（Tab 遍历顺序与视觉一致）
```
