# 技术调研 — 信息源分级与验证

## 信息源可靠性分级

### Tier 1：高可靠（可直接引用）

| 信息源 | 说明 |
|--------|------|
| 官方文档 | 框架/工具的官方文档站 |
| 官方 GitHub 仓库 | README、Issues、Release Notes、Discussions |
| 官方基准测试 | 官方发布的 benchmark 数据 |
| 权威技术标准 | RFC、W3C、ECMA、IEEE 等标准文档 |
| 一手实验数据 | 自己跑的 benchmark、自己的测试结果 |

### Tier 2：较可靠（需交叉验证）

| 信息源 | 说明 | 注意 |
|--------|------|------|
| 知名技术博客 | 个人技术博客（知名开发者） | 检查发布日期 |
| 技术媒体 | InfoQ、ThoughtWorks Radar、Hacker News | 可能有偏见 |
| Stack Overflow | 高赞回答 | 检查是否过时 |
| 第三方 benchmark | 独立测试机构或社区的基准测试 | 验证测试条件 |
| 知名公司技术博客 | Netflix、Uber、Airbnb 等工程博客 | 场景可能不适用 |

### Tier 3：低可靠（仅作参考）

| 信息源 | 说明 | 风险 |
|--------|------|------|
| 通用搜索结果 | 一般的网页搜索结果 | 质量参差不齐 |
| 社交媒体 | Twitter/X、Reddit 讨论 | 情绪化、碎片化 |
| 营销内容 | 供应商白皮书、产品宣传页 | 有明显利益倾向 |
| AI 生成内容 | ChatGPT 等 AI 的回答 | 可能有幻觉 |
| 过期文档 | 超过 2 年的技术文档 | 技术迭代快，可能过时 |

## 信息验证规则

### 核心规则

1. **双源验证**: 关键结论至少需要两个独立的 Tier 1 或 Tier 2 来源佐证
2. **时效性标注**: 所有引用标注获取日期，超过 1 年的标注"⚠️ 可能过时"
3. **区分事实和观点**: 引用时明确标注"[事实]"还是"[观点]"
4. **来源透明**: 每个数据点都要能追溯到具体 URL 或文档位置

### 验证检查清单

| # | 检查项 | 说明 |
|---|--------|------|
| 1 | 信息是否有明确来源？ | 不接受"据说"、"普遍认为" |
| 2 | 来源是否为 Tier 1 或 Tier 2？ | Tier 3 只作补充 |
| 3 | 信息发布日期是否在 1 年内？ | 超过标注时效风险 |
| 4 | 是否有第二个独立来源验证？ | 关键结论必须双源 |
| 5 | 信息是否有利益倾向？ | 供应商信息要对冲 |
| 6 | 数据的测试条件是否说明？ | benchmark 必须附条件 |

## 信息采集工具匹配

| 信息类型 | 推荐工具 | 说明 |
|---------|---------|------|
| 官方文档 | `web_fetch` / `search-layer` | 直接抓取 |
| GitHub 仓库 | `github` CLI | Star、Issues、Release 等 |
| 基准测试 | `search-layer` + `web_fetch` | 搜索 + 抓取详情 |
| 技术博客 | `content-extract` / `web_fetch` | 提取结构化内容 |
| 竞品分析 | `agent-browser` | 网页深度调研 |
| 飞书文档 | `feishu-fetch-doc` | 内部文档获取 |
