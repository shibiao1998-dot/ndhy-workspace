# 技术文档 — 6步标准工作流

## 总览

```
Step 1: 明确文档目标
  |  收到任务 → 理解文档用途和读者
  |  -> 确定: 文档类型(API/架构/用户指南/README/变更日志)
  |  -> 确定: 目标读者(开发者/决策者/最终用户/运维)
  |  -> 确定: 深度级别(L1/L2/L3)
  |  -> 收集: 源材料(代码/方案文档/会议记录/口头说明)
  |
Step 2: 设计文档结构
  |  -> 选择结构模板(详见 templates.md)
  |  -> 列出大纲(一级/二级标题)
  |  -> 确认信息流: 读者从哪里进来、想找什么、怎么找到
  |  -> 大纲确认后再写内容(不边写边想)
  |
Step 3: 填充内容
  |  -> 按大纲逐节撰写
  |  -> 关键概念配示例(抽象描述 + 具体示例)
  |  -> 善用表格、列表、代码块、流程图
  |  -> 信息密度优先: 精简无信息量的句子
  |
Step 4: 交叉引用与去重
  |  -> 检查信息是否只在一处维护
  |  -> 重复内容改为链接引用
  |  -> 确保内外部链接有效
  |
Step 5: 质量审查
  |  -> 过审查清单(详见 checklists.md)
  |  -> 检查: 结构、语言、完整性、准确性、可维护性
  |  -> 模拟读者视角: 30秒能否找到关键信息
  |
Step 6: 输出与交付
     -> 确认格式兼容(Markdown / 飞书 / GitHub)
     -> 标注更新日期和维护责任
     -> 汇报: 文档用途 + 路径 + 结构概览 + 关键内容摘要
```

## 文档类型决策树

```
收到文档任务
  |- 代码接口相关 → API 文档
  |- 系统设计/决策相关 → 架构文档
  |- 操作步骤/怎么用 → 用户指南
  |- 项目介绍/快速了解 → README
  |- 版本变更/更新记录 → 变更日志
  +- 混合类型 → 拆分为多个独立文档
```

## 读者画像与文档风格匹配

| 读者类型 | 关注点 | 文档风格 | 结构偏好 |
|---------|--------|---------|---------|
| **开发者** | 怎么用、API细节、示例代码 | 技术精确、代码多、废话少 | 快速参考+详细API |
| **决策者** | 为什么选这个、有什么风险 | 结论先行、图表多、技术少 | 摘要+方案对比+结论 |
| **最终用户** | 怎么操作、常见问题 | 友好、步骤清晰、配截图 | 分步指南+FAQ |
| **运维人员** | 怎么部署、怎么排错 | 精确命令、配置说明 | 操作手册+故障排查 |
| **新成员** | 全貌了解、快速上手 | 循序渐进、背景说明 | 概述→入门→深入 |

## 工作模式决策树

```
收到文档任务
  |- 单文档补写/更新 → L1 独立完成
  |- 模块级文档(API/用户指南) → L2 标准流程
  +- 文档体系规划 → L3 深度规划
      |- 涉及架构决策 → 源材料找架构专家
      |- 涉及产品功能 → 源材料找产品专家
      +- 涉及代码实现 → 源材料找开发专家
```

## 协作接口

### 上游输入

| 来源 | 接收内容 | 使用方式 |
|------|---------|---------|
| 技术架构专家 | 架构方案、技术决策 | 架构文档的源材料 |
| 后端/前端开发专家 | API实现、代码注释 | API文档的源材料 |
| API设计专家 | OpenAPI契约、设计说明 | API文档的规范输入 |
| 产品定义专家 | PRD、功能说明 | 用户指南的需求来源 |
| 项目管理专家 | 版本计划、里程碑 | 变更日志的信息来源 |

### 下游输出

| 接收方 | 传递内容 | 约束关系 |
|--------|---------|---------|
| 开发团队 | API文档、架构文档 | 实现参考 |
| 测试专家 | 功能说明文档 | 测试用例参考 |
| 用户/客户 | 用户指南、README | 产品使用参考 |
| 新成员 | 项目文档、上手指南 | 快速融入团队 |
