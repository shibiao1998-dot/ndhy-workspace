# 技术文档 — 写作标准与规范

## 核心写作原则

### 1. 信息密度优先

| 原则 | 反面 | 正面 |
|------|------|------|
| 精简 | "为了能够实现这个功能，我们需要..." | "实现此功能需要..." |
| 直达 | "下面我们将介绍如何配置..." | 直接开始配置步骤 |
| 无冗余 | "众所周知，数据库是..." | 跳过读者已知的背景 |

### 2. 结构清晰

- **标题层级**: 最多 3 级（H1/H2/H3），超过说明结构需要拆分
- **段落长度**: 单段不超过 5 行，长段必须拆分
- **列表优先**: 并列信息用列表而非长句
- **表格替代**: 多维对比信息用表格而非文字描述

### 3. 示例驱动

**规则**: 每个关键概念 = 一段描述 + 一个具体示例

```
❌ 纯抽象: "API 支持分页查询，使用 offset 和 limit 参数"
✅ 示例驱动:
   API 支持分页查询：
   GET /users?offset=20&limit=10
   返回第 21-30 条记录。
```

### 4. 视觉友好

| 元素 | 使用场景 | 说明 |
|------|---------|------|
| 代码块 | 命令、API、配置、代码 | 用语言标注（```json、```bash等） |
| 表格 | 参数说明、对比、枚举值 | 替代冗长的文字列表 |
| 列表 | 步骤、特性、要点 | 有序列表=步骤，无序列表=要点 |
| 引用块 | 注意事项、重要提示 | `> ⚠️ 注意:` 或 `> 💡 提示:` |
| Mermaid 图 | 流程、架构、序列 | 替代纯文字描述的复杂流程 |

## 命名与格式规范

### 文件命名

- 小写英文 + 连字符: `api-design-guide.md`
- 不使用空格、下划线、中文
- 名称反映内容: `database-migration.md` 而非 `doc1.md`

### Markdown 格式

- 编码: UTF-8，无 BOM
- 换行: LF（Unix 风格）
- 标题: `#` 后加空格，标题上方空一行
- 列表: `-` 后加空格，嵌套缩进 2 空格
- 代码块: 标注语言 ` ```json `
- 链接: 相对路径优先 `[参考](./references/xxx.md)`

### 术语一致性

| 规则 | 说明 |
|------|------|
| 统一术语 | 同一概念全文使用同一名称 |
| 首次出现定义 | 专业术语首次出现时给出定义或解释 |
| 缩写规范 | 首次出现: "应用程序接口 (API)"，后续用 "API" |
| 中英混排 | 英文单词前后加空格: "使用 React 框架" |

## 面向受众的风格差异

### 开发者文档

- 直接、精确、代码多
- 避免解释基础概念
- 提供可运行的完整示例
- 错误处理和边界情况不可省略

### 决策者文档

- 结论先行，细节后置
- 图表 > 文字，数据 > 描述
- 聚焦"为什么"和"什么影响"
- 减少技术细节

### 用户文档

- 友好、耐心、步骤清晰
- 每步配截图或示例
- 避免技术黑话
- FAQ 覆盖常见困惑

## 可维护性规范

| 规则 | 做法 |
|------|------|
| 单一信息源 | 每条信息只在一处维护，其他地方用链接 |
| 更新日期 | 文档头部标注最后更新日期 |
| 维护责任 | 明确文档的维护者是谁 |
| 版本标注 | 与产品版本关联的文档标注适用版本 |
| 废弃标注 | 过时内容标注 `[已废弃]` 而非直接删除 |
