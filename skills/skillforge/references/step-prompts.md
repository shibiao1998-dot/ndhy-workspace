# 7 步生成 Prompt 模板

> 系统角色：你是一位专业的 Agent Skills 架构师。你设计生产级 Agent Skills，遵循 Manus Skills 生态和 Agent Skills 开放标准的最佳实践。

## 核心设计原则（贯穿全流程）

1. **简洁至上** — Context window 是公共资源。只提供 AI 模型不已知的信息
2. **Description 是触发器** — 决定 Skill 是否被选中。必须包含 WHAT + WHEN
3. **渐进式披露** — SKILL.md 是主入口（< 500 行）。支持文件放 scripts/、references/、templates/，按需加载
4. **代码示例 > 文字说明**
5. **反面案例必不可少** — 用 ❌/✅ 对比格式展示什么不该做
6. **祈使语气** — "执行"而非"你应该执行"
7. **不加辅助文件** — 不创建 README.md、CHANGELOG.md

---

## Step 1：需求深度挖掘（5 维框架）

### 目标
输出结构化需求文档，覆盖 5 个维度。

### 产出模板

```markdown
## 1. 核心定位
- **名称**：{hyphen-case 格式}
- **一句话定位**：{20 字以内}
- **目标用户画像**：{谁会用这个 Skill}
- **核心价值主张**：{解决什么问题，为什么现有方案不够}

## 2. 功能边界
### 核心功能（3-7 个）
| # | 功能 | 输入 | 处理 | 输出 |
|---|------|------|------|------|
| 1 | {功能名} | {输入描述} | {处理逻辑} | {输出描述} |

### 扩展功能
- {可选的增强功能}

### 明确排除
- {不做什么，边界在哪}

## 3. 使用场景（至少 5 个）
| # | 用户原始请求 | 期望行为 | 输出格式 |
|---|-------------|----------|----------|
| 1 | "{用户会怎么说}" | {Agent 应该怎么做} | {产出什么} |

## 4. 知识缺口分析（最关键）
### AI 已知（不写进 SKILL.md）
- {AI 模型已经知道的知识，列举具体项}

### AI 不知道（SKILL.md 核心内容）
- {AI 不知道但执行此 Skill 必须知道的知识}

### AI 易犯错（需要反面案例 ❌/✅）
- {AI 容易做错的事情，需要用对比示例纠正}

## 5. 依赖和约束
- **外部工具依赖**：{需要什么工具/CLI/API}
- **文件格式要求**：{输入输出的格式约束}
- **环境约束**：{OS、运行时、权限要求}
- **性能约束**：{超时、大小限制等}
```

### 执行要点
- 知识缺口分析是整个流程最关键的一步，投入最多精力
- 每个核心功能必须用"输入→处理→输出"三元组描述，不留模糊
- 使用场景要用用户的真实语言，不是技术术语
- 明确排除和 AI 已知同样重要——防止 Skill 膨胀

---

## Step 2：架构决策引擎（5 大决策）

### 目标
做出 5 个关键架构决策，决定 Skill 的整体形态。

### 产出模板

```markdown
## 决策 1：结构模式
**选择**：{工作流型 / 任务型 / 指南型 / 能力型}
**理由**：{为什么选这个模式}

模式说明：
- 工作流型：严格的步骤序列，适合流程化任务（如部署、发布）
- 任务型：目标驱动，多条路径可达，适合创建类任务（如生成代码）
- 指南型：规则和最佳实践集合，适合规范类 Skill（如代码风格）
- 能力型：赋予新能力，适合工具集成类 Skill（如 API 调用）

## 决策 2：自由度级别
**选择**：{高 / 中 / 低}
**理由**：{为什么选这个级别}

级别说明：
- 高自由度：文本指导为主，多种方案都可接受
- 中自由度：伪代码或参数化脚本，有推荐模式但允许变化
- 低自由度：具体脚本，少量参数，适合易出错的关键操作

## 决策 3：资源文件规划
| 路径 | 用途 | 预估行数 |
|------|------|----------|
| references/{文件名}.md | {内容描述} | {行数} |
| scripts/{文件名}.py | {功能描述} | {行数} |
| templates/{文件名} | {模板描述} | {行数} |

## 决策 4：渐进式披露策略
### SKILL.md（< 500 行）包含：
- {列出 SKILL.md 中保留的内容}

### references/ 包含：
- {列出拆到 references/ 的内容及加载触发条件}

### scripts/ 包含：
- {列出拆到 scripts/ 的内容}

### 加载触发条件
| 文件 | 何时加载 |
|------|----------|
| references/{文件} | {触发条件描述} |

## 决策 5：质量保证方案
### 验证检查清单
- [ ] {检查项 1}
- [ ] {检查项 2}

### 常见错误
- {错误 1}：{如何避免}

### 输出标准
- {标准 1}
```

### 执行要点
- 结构模式决定了 SKILL.md 的整体组织方式
- 自由度级别决定了指令的粒度——越易出错越要低自由度
- 资源文件规划必须在此步完成，不要在写 SKILL.md 时临时决定
- 渐进式披露策略中，每个 references 文件必须标注加载触发条件

## Step 3：元数据精炼

### 目标
生成最精准的 YAML frontmatter，尤其是 description。

### 执行流程

**3.1 生成 3 个候选 description**

每个候选必须：
- 30-200 词
- 包含 WHAT（做什么）+ WHEN（何时用）
- 列举具体触发场景

**3.2 三维度自评打分**

| 候选 | 触发精准度 (1-5) | 能力覆盖度 (1-5) | 信息密度 (1-5) | 总分 |
|------|------------------|------------------|----------------|------|
| A | | | | |
| B | | | | |
| C | | | | |

评分标准：
- **触发精准度**：用户说了相关请求时，这个 description 能否准确触发？会不会误触发？
- **能力覆盖度**：description 是否覆盖了 Skill 的所有核心能力？
- **信息密度**：每个词都有信息量吗？有没有废话？

**3.3 选最高分，输出最终 frontmatter**

```yaml
---
name: {hyphen-case}
description: "{最高分的 description}"
---
```

### 反面案例

❌ 错误的 description：
```yaml
description: "A tool for working with data"
```
问题：太模糊，不知道做什么，不知道何时触发。

❌ 错误的 description：
```yaml
description: "This skill helps you to process CSV files and it can also do JSON and XML and it supports many formats"
```
问题：啰嗦，"helps you to"是废话，能力边界不清晰。

✅ 正确的 description：
```yaml
description: "Parse, transform, and validate CSV/JSON/XML data files. Use when converting between data formats, cleaning malformed records, validating against schemas, or batch-processing tabular data. Handles files up to 500MB."
```

## Step 4：SKILL.md 主体生成

### 目标
按架构决策生成 SKILL.md 正文（< 500 行）。

### 内容组织顺序

```markdown
# {Skill 名称}

{概述：2-3 句话说明这个 Skill 是什么、解决什么问题}
{与相关 Skill 的关系说明（如果有）}

## {核心工作流 / 核心规则}
{根据架构决策的结构模式组织}

## {详细规则 / 步骤}
{只包含 AI 不知道的内容}

## {代码示例}
{❌/✅ 对比格式}

## {边界处理}
{异常情况、边界条件}

## {输出格式}
{期望的输出标准}

## {验证清单}
- [ ] {检查项}
```

### 写作原则
- **只写 AI 不知道的** — 反复问自己"AI 已经知道这个吗？"
- **不重复 description** — description 已说过的不要在正文重复
- **超 100 行的内容移到 references/** — 在 SKILL.md 中用链接引用，标注何时加载
- **代码示例优先** — 能用代码说清楚的不用文字
- **祈使语气** — "执行 xxx" 不是 "你应该执行 xxx"
- **每个 references 文件必须有加载说明** — 格式：`详见 [references/xxx.md](references/xxx.md)`，附加载触发条件

### 反面案例

❌ 正文中重复 description 内容：
```markdown
# My Skill

## When to Use This Skill
Use this skill when you need to process CSV files...
```
问题："When to Use"应在 description 中，正文加载时 Skill 已被触发，无需再说。

✅ 正文直接进入核心内容：
```markdown
# My Skill

CSV/JSON/XML 数据处理引擎。支持格式转换、清洗和校验。

## 处理流程
1. 检测输入文件编码...
```

## Step 5：质量审计与优化

### 目标
用 10 维度评分卡对 Skill 进行审计，低于 8 分的维度自动修复。

### 执行流程

**5.1 逐维度评分**

详见 [references/quality-audit.md](../quality-audit.md) 中的完整评分标准。

| 维度 | 评分 (1-10) | 问题描述 | 修复方案 |
|------|-------------|----------|----------|
| 1. Description 触发精准度 | | | |
| 2. 知识增量 | | | |
| 3. 代码示例质量 | | | |
| 4. 反面案例覆盖 | | | |
| 5. 结构清晰度 | | | |
| 6. 渐进式披露 | | | |
| 7. 语气一致性 | | | |
| 8. 边界处理 | | | |
| 9. 可操作性 | | | |
| 10. 完整性 | | | |

**5.2 自动修复**

对所有 < 8 分的维度：
1. 标注具体问题
2. 提出修复方案
3. 执行修复
4. 重新评分确认 ≥ 8

**5.3 输出优化后的完整版 SKILL.md**

## Step 6：配套资源生成

### 目标
按 Step 2 的资源文件规划，生成所有配套文件。

### 执行要点
- 严格按 Step 2 规划的文件列表生成，不多不少
- references/ 文件：长文件（> 100 行）加目录索引
- scripts/ 文件：包含错误处理，不留硬编码
- templates/ 文件：包含占位符说明
- 每个文件开头注释说明用途和加载条件

---

## Step 7：最终组装与交付

### 目标
完成最终验证，确保 Skill 可交付。

### 验证清单

**格式验证：**
- [ ] YAML frontmatter 格式正确（只有 name + description）
- [ ] name 为 hyphen-case
- [ ] SKILL.md < 500 行
- [ ] 目录结构符合规范（SKILL.md + scripts/ + references/ + templates/）
- [ ] 无 README.md、CHANGELOG.md 等辅助文件
- [ ] UTF-8 编码

**内容验证：**
- [ ] description 包含 WHAT + WHEN，30-200 词
- [ ] 概述 2-3 句
- [ ] 至少 1 组 ❌/✅ 反面案例
- [ ] 所有 references 文件有加载说明和触发条件
- [ ] 不含 AI 已知的常识性内容
- [ ] 祈使语气，无"你应该"

**最佳实践验证：**
- [ ] 知识缺口分析已完成且反映在内容中
- [ ] 10 维度质量审计全部 ≥ 8 分
- [ ] 渐进式披露合理（SKILL.md 精简，详细内容在 references/）
- [ ] 引用文件保持一层深度（不嵌套引用）

### 交付物
```
skill-name/
├── SKILL.md
├── references/
│   └── *.md
├── scripts/
│   └── *.py / *.sh（如有）
└── templates/
    └── *（如有）
```
