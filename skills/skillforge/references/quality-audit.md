# 10 维度质量审计评分卡

> 每个维度 1-10 分。所有维度必须 ≥ 8 分方可交付。低于 8 分的维度必须修复后重新评分。

---

## 维度 1：Description 触发精准度

**评估对象**：YAML frontmatter 中的 description 字段

| 分数 | 标准 |
|------|------|
| 1-3 | description 模糊、过短（< 30 词）、缺少 WHEN 触发条件 |
| 4-5 | 包含 WHAT 但 WHEN 不够具体，可能误触发或漏触发 |
| 6-7 | WHAT + WHEN 都有，但触发场景不够全面，或措辞不够精准 |
| 8-9 | WHAT + WHEN 精准覆盖核心场景，误触发概率低，30-200 词 |
| 10 | 完美触发——所有目标场景都能命中，非目标场景不误触发，措辞简洁有力 |

### 检查要点
- [ ] 包含"做什么"（WHAT）
- [ ] 包含"何时用"（WHEN）+ 具体触发短语
- [ ] 30-200 词范围内
- [ ] 不含废话（"helps you"、"a tool for"）
- [ ] 不与其他 Skill 的 description 冲突

---

## 维度 2：知识增量（只含 AI 不知道的）

**评估对象**：SKILL.md 正文内容

| 分数 | 标准 |
|------|------|
| 1-3 | 大量常识性内容（AI 已知），知识增量 < 30% |
| 4-5 | 部分常识混杂，知识增量 30-50% |
| 6-7 | 少量常识残留，知识增量 50-80% |
| 8-9 | 几乎全是增量知识，偶有可删减内容 |
| 10 | 每一行都是 AI 不知道的、执行任务必须的知识 |

### 检查要点
- [ ] 不解释编程语言基础语法
- [ ] 不解释通用工具的基本用法
- [ ] 不重复 description 中已说的内容
- [ ] 专注于：专有流程、内部 API、特定配置、隐含陷阱

---

## 维度 3：代码示例质量

**评估对象**：SKILL.md 及 references/ 中的代码示例

| 分数 | 标准 |
|------|------|
| 1-3 | 无代码示例，或示例有语法错误 |
| 4-5 | 有示例但不完整，缺少上下文，难以直接使用 |
| 6-7 | 示例完整可运行，但缺少 ❌/✅ 对比 |
| 8-9 | 示例完整、有 ❌/✅ 对比、覆盖主要场景 |
| 10 | 示例精准、对比鲜明、覆盖核心和边界场景，可直接复制使用 |

### 检查要点
- [ ] 至少 1 组代码示例
- [ ] 示例可直接运行（无占位符需替换）
- [ ] 有 ❌/✅ 对比展示常见错误
- [ ] 覆盖正常和异常场景

---

## 维度 4：反面案例覆盖（❌/✅ 格式）

**评估对象**：整个 Skill 中的反面案例

| 分数 | 标准 |
|------|------|
| 1-3 | 无反面案例 |
| 4-5 | 有 1-2 个但不够典型，或格式不统一 |
| 6-7 | 覆盖主要易错点，格式统一使用 ❌/✅ |
| 8-9 | 覆盖全面，每个核心功能的常见错误都有对比 |
| 10 | 反面案例精准命中 AI 最容易犯的错误，对比鲜明，学习价值高 |

### 检查要点
- [ ] 使用 ❌/✅ 格式
- [ ] 覆盖"AI 易犯错"分析中识别的所有项
- [ ] 错误示例真的是 AI 常犯的错误（不是人造的稻草人）
- [ ] 正确示例可直接替代错误示例使用

---

## 维度 5：结构清晰度

**评估对象**：SKILL.md 整体结构

| 分数 | 标准 |
|------|------|
| 1-3 | 无结构，大段文字堆砌 |
| 4-5 | 有标题但层次不清，内容归属混乱 |
| 6-7 | 结构合理，但个别部分归属不当 |
| 8-9 | 结构清晰，层次分明，信息密度均匀 |
| 10 | 结构完美——一眼扫过标题就能理解全貌，每部分独立且完整 |

### 检查要点
- [ ] 标题层次不超过 3 级（#, ##, ###）
- [ ] 每个章节职责单一
- [ ] 信息不重复出现在多个章节
- [ ] 有合理的信息流动顺序（概述→核心→细节→边界）

---

## 维度 6：渐进式披露（< 500 行）

**评估对象**：SKILL.md 行数和内容分布

| 分数 | 标准 |
|------|------|
| 1-3 | SKILL.md > 500 行，未拆分 |
| 4-5 | SKILL.md < 500 行但内容臃肿，该拆未拆 |
| 6-7 | 已拆分但 references 文件缺少加载说明 |
| 8-9 | SKILL.md 精简，references 拆分合理，有加载触发条件 |
| 10 | 完美三级披露——metadata/SKILL.md/references 各司其职，按需加载 |

### 检查要点
- [ ] SKILL.md < 500 行
- [ ] 超 100 行的详细内容已拆到 references/
- [ ] 每个 references 文件在 SKILL.md 中有链接和加载说明
- [ ] 引用深度 ≤ 1 层（references 文件不再引用其他 references）
- [ ] 长 references 文件（> 100 行）有目录索引

---

## 维度 7：语气一致性（祈使语气）

**评估对象**：SKILL.md 正文的写作语气

| 分数 | 标准 |
|------|------|
| 1-3 | 大量"你应该"、"建议"、"可以考虑"等委婉语 |
| 4-5 | 混合使用祈使语气和委婉语 |
| 6-7 | 大部分使用祈使语气，偶有不一致 |
| 8-9 | 全篇祈使语气，直接、清晰 |
| 10 | 完美祈使——每条指令都是明确的动作指示，零冗余 |

### 检查要点
- [ ] 无"你应该"、"你可以"、"建议"
- [ ] 使用"执行"、"检查"、"读取"、"输出"等动词开头
- [ ] 无被动语态（"应该被执行"→"执行"）
- [ ] 无客套话（"请注意"→ 直接说要点）

---

## 维度 8：边界处理

**评估对象**：异常情况和边界条件的处理说明

| 分数 | 标准 |
|------|------|
| 1-3 | 无边界处理说明 |
| 4-5 | 提到了部分边界但处理方式不明确 |
| 6-7 | 主要边界场景有处理说明 |
| 8-9 | 全面覆盖边界，每个异常有明确处理步骤 |
| 10 | 边界处理完备——包含降级方案、错误恢复、超时处理 |

### 检查要点
- [ ] 输入异常处理（缺失、格式错误、超大）
- [ ] 外部依赖不可用时的降级方案
- [ ] 超时和重试策略
- [ ] 明确说明"不处理什么"（与功能边界呼应）

---

## 维度 9：可操作性

**评估对象**：Agent 拿到 Skill 后能否直接执行

| 分数 | 标准 |
|------|------|
| 1-3 | 大量模糊指令，需要猜测具体操作 |
| 4-5 | 部分步骤可执行，部分需要额外信息 |
| 6-7 | 大部分步骤可直接执行，少量需推断 |
| 8-9 | 几乎所有步骤都是可直接执行的具体操作 |
| 10 | 完美可操作——任何 Agent 拿到就能执行，零猜测 |

### 检查要点
- [ ] 每个步骤都是具体的动作（不是"处理数据"而是"用 jq 提取 .results 字段"）
- [ ] 命令、路径、参数都有具体值或明确的参数化方式
- [ ] 工具依赖有安装说明
- [ ] 前置条件明确列出

---

## 维度 10：完整性

**评估对象**：Skill 作为整体是否覆盖所有必要信息

| 分数 | 标准 |
|------|------|
| 1-3 | 缺少关键信息，无法完成核心功能 |
| 4-5 | 核心功能可用但有明显遗漏 |
| 6-7 | 覆盖核心功能，少量次要场景缺失 |
| 8-9 | 全面覆盖核心和常见场景 |
| 10 | 完备——核心功能、常见场景、边界处理、验证清单一应俱全 |

### 检查要点
- [ ] 所有核心功能都有说明
- [ ] 使用场景覆盖率 ≥ 80%
- [ ] 有验证清单（用户可逐条检查）
- [ ] 有输出格式说明
- [ ] references 文件完整，无断链
