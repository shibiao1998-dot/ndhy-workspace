# 实现质量检查清单 + 反模式详细防范 + 联调交接指南

---

## 实现质量检查清单

### 契约一致性

- [ ] **URL 路径**：实现的路由与 API 契约完全一致
- [ ] **HTTP Method**：GET/POST/PUT/DELETE 使用正确
- [ ] **请求参数**：名称、类型、必填/可选与契约一致
- [ ] **响应结构**：字段名、类型、嵌套关系与契约一致
- [ ] **错误码**：实现的错误码在契约中有定义
- [ ] **HTTP 状态码**：200/201/400/401/403/404/500 使用正确
- [ ] **分页格式**：分页参数和响应格式与通用规范一致

### 分层架构

- [ ] **Handler 层**：只做协议适配，不含业务逻辑
- [ ] **Service 层**：包含所有业务规则，不依赖 HTTP 框架
- [ ] **Repository 层**：只做数据访问，不含业务判断
- [ ] **调用方向**：严格 Handler → Service → Repository
- [ ] **数据对象**：DTO / Command / Entity 各层分离

### 输入校验

- [ ] **必填校验**：所有必填字段有校验
- [ ] **格式校验**：邮箱、手机、日期等有格式校验
- [ ] **范围校验**：数值、长度在合理范围内
- [ ] **业务校验**：唯一性、存在性、状态合法性有检查
- [ ] **校验顺序**：格式 → 权限 → 业务

### 权限控制

- [ ] **认证**：需认证接口有认证中间件保护
- [ ] **授权**：需授权接口有角色/权限检查
- [ ] **数据权限**：涉及数据隔离的查询有数据权限过滤
- [ ] **错误码**：未认证 401、未授权 403
- [ ] **无泄漏**：权限错误不泄漏敏感信息

### 错误处理

- [ ] **异常分层**：系统异常/业务异常/参数异常有区分
- [ ] **全局处理**：有全局异常处理器统一转换
- [ ] **错误码**：每个业务异常有明确的错误码
- [ ] **无吞异常**：没有空的 catch 块
- [ ] **不暴露内部**：系统错误不返回堆栈/SQL

### 事务

- [ ] **边界明确**：事务声明在 Service 方法级别
- [ ] **范围最小**：事务只包含必须原子的操作
- [ ] **无外部调用**：事务内不做 HTTP/MQ 调用
- [ ] **幂等设计**：关键操作支持安全重试

### 日志可观测

- [ ] **请求日志**：每个请求有 requestId、方法、路径、状态码、耗时
- [ ] **业务日志**：关键业务操作有日志
- [ ] **异常日志**：异常有错误日志含上下文
- [ ] **脱敏**：敏感数据已脱敏
- [ ] **健康检查**：有 /health 端点

### 测试

- [ ] **核心逻辑有测试**：Service 层关键方法有单元测试
- [ ] **异常路径有测试**：业务异常场景有测试
- [ ] **边界条件有测试**：空值、边界值、特殊字符有测试
- [ ] **测试可独立运行**：不依赖外部服务和执行顺序
- [ ] **测试支撑完备**：有 Fixture/Builder/Mock 配置

### 代码质量

- [ ] **命名自解释**：变量名/方法名/类名无需注释即可理解
- [ ] **无魔法值**：常量和枚举替代硬编码
- [ ] **无超长方法**：handler ≤ 30 行，service 方法 ≤ 50 行
- [ ] **无重复代码**：同一逻辑不出现在 3+ 处
- [ ] **无深层嵌套**：if 嵌套不超过 3 层

## 反模式详细防范

### 1. 胖 Controller

**症状**：Controller/Handler 方法超过 30 行，包含业务判断、数据库查询、复杂计算。

**危害**：
- 业务逻辑无法复用（只能通过 HTTP 触发）
- 无法单元测试（依赖 HTTP 上下文）
- 代码膨胀后无法维护

**修复**：
- Handler 只做：解析请求 → 调用 Service → 返回响应
- 所有业务逻辑下沉到 Service 层
- 一个 Handler 方法只调用一个 Service 方法

### 2. 职责混乱

**症状**：Service 中直接写 SQL；Repository 中做 if/else 业务判断；Handler 中做数据库查询。

**危害**：
- 层级边界模糊，维护困难
- 换数据库/换协议影响面巨大
- 测试难度倍增

**修复**：
- 每层职责一句话说清
- 严格单向调用：Handler → Service → Repository
- 跨层只通过接口

### 3. 绕过校验和权限

**症状**：先写业务逻辑，校验和权限"后面再加"——然后永远不加。

**危害**：
- 安全漏洞
- 数据被篡改
- 越权访问

**修复**：
- 校验和权限是 Definition of Done 的一部分
- 没有校验和权限的代码不算完成
- 在 code review 中作为必查项

### 4. if/else 堆砌状态逻辑

**症状**：状态转换用多层 if/else 嵌套，新增状态就加更多 if。

**危害**：
- 状态转换不可枚举
- 非法状态转换没有检测
- 代码膨胀，无法测试

**修复**：
- 定义状态枚举和合法转换矩阵
- 用状态机或策略模式处理
- 非法转换抛出明确异常

### 5. 数据库字段泄漏为 API 响应

**症状**：数据库 Entity 直接作为 API 响应返回。

**危害**：
- 数据库改字段 → API 响应结构变化 → 前端崩溃
- 暴露内部字段（is_deleted、internal_note）
- 暴露敏感数据

**修复**：
- 通过 DTO/VO 做映射
- 数据库结构变更不穿透到 API
- 只暴露契约定义的字段

### 6. 前端适配污染长期实现

**症状**：前端说"给我加个字段"、"把格式改成这样"，直接在 Service 层加逻辑。

**危害**：
- Service 层掺杂展示逻辑
- 多客户端（Web/App/小程序）需求冲突
- 实现越来越臃肿

**修复**：
- 前端适配放在 Handler 层的 DTO 映射
- Service 层只输出业务结果
- 不同客户端的格式差异在 Handler 处理

### 7. 无错误码、无日志、无异常分层

**症状**：catch 到异常直接 `return error`；日志只有 `log.info("error")`；返回给前端只有 "操作失败"。

**危害**：
- 前端无法区分不同错误做不同处理
- 运维无法定位问题
- 用户只看到"操作失败"不知道为什么

**修复**：
- 定义三层异常体系
- 每个业务异常有明确错误码
- 结构化日志包含请求上下文

### 8. 没有测试默认正确

**症状**：手动调了一下接口能通，就说"实现完成"。

**危害**：
- 回归风险（改 A 坏 B）
- 边界条件未覆盖
- 重构不敢动

**修复**：
- 核心 Service 方法必须有单元测试
- 边界条件和异常路径必须有测试
- 没有测试 ≠ 已完成

### 9. 复制粘贴逻辑

**症状**：同样的校验逻辑、同样的错误处理、同样的数据转换出现在多个地方。

**危害**：
- 改一处忘改其他处
- 逻辑不一致
- 代码膨胀

**修复**：
- 提取公共方法/工具类
- 提取公共 Service
- DRY 原则

### 10. 实现层偷偷改契约或模型

**症状**：实现中发现契约定义不方便，直接改了返回格式；发现表结构不合适，直接加了字段。

**危害**：
- 前端/测试基于旧契约工作 → 联调爆炸
- 数据模型不一致 → 隐性 bug
- 上游设计专家不知道变化

**修复**：
- 发现问题 → 记录问题 → 回流上游讨论
- 不私自改契约或模型
- 改动必须走正式变更流程

## 联调交接指南

### 联调前准备

| 准备项 | 说明 | 检查 |
|--------|------|------|
| 接口实现完成 | 所有约定接口已实现且自测通过 | ✅/❌ |
| Mock 数据就绪 | 核心接口有示例请求和响应 | ✅/❌ |
| 环境可访问 | 联调环境部署且前端可访问 | ✅/❌ |
| 认证方式明确 | 测试账号/Token 获取方式已说明 | ✅/❌ |
| 错误码文档 | 错误码含义和触发条件已说明 | ✅/❌ |
| 已知限制说明 | 未完成功能和已知问题已列出 | ✅/❌ |

### 联调说明内容

1. **接口清单**：哪些接口可联调、哪些还在开发中
2. **环境信息**：联调地址、数据库状态
3. **认证方式**：如何获取测试 token
4. **Mock 数据**：核心接口的请求和响应示例
5. **错误码速查**：常见错误码和前端处理建议
6. **注意事项**：特殊约束、已知问题、临时限制
7. **验收检查点**：联调通过的标准

### 交接给测试专家

在 test-notes.md 中提供：

1. **关键业务路径**：测试优先覆盖的主流程
2. **边界条件**：需要特别测试的边界场景
3. **已知限制**：当前实现的已知问题
4. **测试数据**：如何准备和清理测试数据
5. **环境要求**：测试环境的配置要求
6. **覆盖范围**：已有单元测试覆盖的范围（避免重复测试）
