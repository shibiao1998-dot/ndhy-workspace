# integration-testing — 联调集成执行工具箱

将多模块、多专家的独立产出放到真实集成链路中进行连接、验证、归因和回流，确保系统不是"局部正确"而是"整体可跑通、可进入验证与交付阶段"。

> 局部能跑 ≠ 整体可用。联调的目标是收敛真实问题，不是制造扯皮。归因必须准确，回归必须闭环。

## 8 步标准流程

```
Step 1: 理解集成边界与关键流程
  |  消化 API 契约、前后端实现说明、环境配置、关键流程清单
  |  -> 确认: 联调范围、关键链路、涉及模块、验收标准、已知限制
  |  -> 信息不足 -> 要求补充（不猜测、不假设、不硬调）
  |
Step 2: 校验联调前提
  |  -> 环境是否就绪（数据库/服务/网关/依赖服务均可达）
  |  -> 配置是否正确（环境变量/端点/密钥/特性开关）
  |  -> 权限是否到位（API认证/角色权限/跨服务授权）
  |  -> mock与真实数据是否明确（哪些已对接真实、哪些仍是mock）
  |  -> 版本是否匹配（前后端版本/契约版本/数据库schema版本）
  |  -> 前提不满足 -> 停止联调 -> 列出缺口 -> 回流对应专家
  |  -> 详细方法 -> references/prerequisite-validation.md
  |
Step 3: 核对契约与实现一致性
  |  -> 契约 vs 后端实现：路径/方法/参数/响应结构/错误码/分页
  |  -> 契约 vs 前端消费：请求构造/响应解析/字段映射/状态处理
  |  -> 前后端对齐：字段名/类型/枚举值/必填可选/默认值/空值语义
  |  -> 错误语义对齐：错误码含义/错误消息展示/降级行为
  |  -> 状态码对齐：HTTP状态码/业务状态码/前端状态映射
  |  -> 详细方法 -> references/contract-alignment.md
  |
Step 4: 跑通端到端关键流程
  |  -> 按关键流程清单逐条执行（正常路径优先）
  |  -> 记录每步：请求 -> 后端处理 -> 响应 -> 前端展示 -> 用户反馈
  |  -> 验证异常路径：参数错误/权限不足/资源不存在/服务超时
  |  -> 验证边界路径：空列表/首次使用/并发操作/大数据量
  |  -> 详细方法 -> references/flow-and-timing.md §端到端验证
  |
Step 5: 识别不一致与时序问题
  |  -> 字段不一致：名称/类型/枚举值/默认值/空值处理不匹配
  |  -> 状态不一致：后端状态变更未反映到前端/状态流转顺序错误
  |  -> 错误语义不一致：错误码解释不同/错误消息不匹配/降级行为不同
  |  -> 时序问题：请求顺序依赖/异步回调丢失/轮询时机不当/竞态条件
  |  -> 对每个问题记录：现象 + 期望 + 实际 + 复现路径
  |  -> 详细方法 -> references/flow-and-timing.md §时序与异步诊断
  |
Step 6: 归因集成问题并回流
  |  -> 按归因决策树定位问题根因层（契约/前端/后端/数据/环境/流程）
  |  -> 为每个问题标注：归因层 + 责任方 + 修复建议 + 优先级
  |  -> 回流到对应专家（不笼统甩锅，附复现路径和证据）
  |  -> 归因不清 -> 标记待定 -> 建议拉多方核对
  |  -> 归因决策树 -> 本文 §归因决策树
  |  -> 详细方法 -> references/triage-and-attribution.md
  |
Step 7: 回归验证
  |  -> 根据 Step 6 的修复回流，生成回归清单
  |  -> 修复到位后逐项回归验证
  |  -> 验证不通过 -> 重新归因 -> 再次回流
  |  -> 全部通过 -> 更新联调状态
  |  -> 详细方法 -> references/checklists.md §回归验证
  |
Step 8: 输出联调结论 + 就绪判断
     -> 汇总联调状态（关键流程通过率/残余问题/风险项）
     -> 按就绪判断标准评估是否可进��下一阶段
     -> 输出 6 个标准文档（见下方）
     -> 过联调检查清单（见下方）
     -> 按模板组织 -> references/templates.md
```

## 联调深度速查

| 深度 | 适用场景 | 覆盖步骤 | 输出物 |
|------|----------|----------|--------|
| **快速验证** | 单接口/单流程/小改动后回归 | Step 1-4 重点 + 5-6 扫描 | 流程验证报告 + 问题归因 |
| **标准联调** | 功能模块联调/迭代联调 | 完整 8 步 | 完整交付包（6 文档） |
| **深度联调** | 全新产品首次联调/跨系统对接/核心链路 | 完整 8 步 + 多轮回归 + 压力路径 | 完整交付包 + 风险专项报告 |

判断不清时，按标准联调执行。

## 5 层深度边界速查

| 层级 | 主责角色 | 联调专家的关系 |
|------|---------|---------------|
| **设计定义层** | 产品/API/数据库/架构专家 | 消费其契约和约束作为联调基准，不替代其定义 |
| **单模块实现层** | 后端/前端开发专家 | 验证其实现是否符合契约，不替代其开发修复 |
| **集成闭环层** | **联调集成专家（核心）** | 主责：契约对齐、一致性校验、流程跑通、归因回流、回归闭环 |
| **完整验证层** | 测试专家 | 提供集成层结论和回归清单，不替代完整验收 |
| **发布运行层** | 运维专家 | 提出环境和配置问题，不替代部署运维 |

**铁律**：联调专家主责第 3 层。消费第 1-2 层输入，为第 4-5 层提供集成结论和风险信息，不替代其职责。

## 归因决策树

```
发现集成问题
  |
  |- 环境/配置/权限/依赖不可达？
  |   -> 归因: 环境层 -> 回流: 运维专家 / 相关配置负责人
  |
  |- 契约本身有歧义或前后端理解不一致？
  |   -> 归因: 契约层 -> 回流: API 设计专家澄清
  |
  |- 后端返回与契约不符？
  |   |- 响应结构/字段/类型不匹配？
  |   |   -> 归因: 后端实现层 -> 回流: 后端开发专家
  |   |- 错误码/状态码与契约不一致？
  |   |   -> 归因: 后端实现层 -> 回流: 后端开发专家
  |   +- 业务逻辑/状态流转不正确？
  |       -> 归因: 后端实现层 -> 回流: 后端开发专家
  |
  |- 前端请求构造或响应消费不正确？
  |   |- 请求参数/路径/方法与契约不符？
  |   |   -> 归因: 前端实现层 -> 回流: 前端开发专家
  |   |- 响应字段解析/展示/状态映射错误？
  |   |   -> 归因: 前端实现层 -> 回流: 前端开发专家
  |   +- 错误处理/降级行为不符合预期？
  |       -> 归因: 前端实现层 -> 回流: 前端开发专家
  |
  |- 数据层语义不一致？
  |   |- 字段含义/默认值/枚举定义与契约不匹配？
  |   |   -> 归因: 数据层 -> 回流: 数据库设计专家
  |   +- 持久化状态与接口状态不同步？
  |       -> 归因: 数据层 + 后端层 -> 回流: 数据库 + 后端专家
  |
  |- 时序/异步/并发问题？
  |   |- 请求顺序依赖未保障？
  |   |   -> 归因: 调用方（前端/后端）-> 回流对应专家
  |   |- 异步回调/Webhook 未正确处理？
  |   |   -> 归因: 后端实现层 -> 回流: 后端开发专家
  |   +- 竞态条件导致数据不一致？
  |       -> 归因: 后端实现层 -> 回流: 后端开发专家（可能需架构专家）
  |
  +- 归因不清 / 涉及多方？
      -> 标记: 待定 -> 建议拉 API + 前端 + 后端联合核对
      -> 升级: 项目管理专家协调
```

### 归因铁律

- **有证据才归因**：每个归因必须附复现路径和实际 vs 期望的对比
- **不笼统甩锅**：不说"接口有问题"，要说"POST /api/orders 返回的 status 字段值为 'CREATED'，契约定义为 'created'（大小写不一致）"
- **多方问题分别归因**：一个问题涉及多方时，拆分为多个归因项分别回流
- **归因不清就标记**：不确定时标记"待定"，建议多方核对，不强行定责

## 就绪判断标准

### 可进入下一阶段（测试/交付）的条件

| # | 条件 | 判断方式 |
|---|------|---------|
| 1 | **关键流程全部跑通** | 关键流程清单中所有正常路径验证通过 |
| 2 | **契约消费一致** | 前后端实际请求/响应与契约定义一致 |
| 3 | **状态与错误反馈一致** | 状态流转正确，错误码/消息/降级行为前后端对齐 |
| 4 | **环境前提满足** | 无 mock 冒充真实、无临时绕过、配置正确 |
| 5 | **核心问题已归因并回流** | 无 P0 问题残留，P1 已有明确修复计划 |
| 6 | **回归验证通过** | 所有已修复问题的回归验证通过 |
| 7 | **无阻断性残余风险** | 无已知但未处理的阻断性问题 |

### 不可宣称"联调完成"的情况

- 仍有 P0 问题未修复
- 关键流程未全部跑通
- mock 跑通被当作真实联调通过
- 异常路径和错误反馈未验证
- 回归验证未完成
- 环境前提存在临时绕过

## 反模式速查

| # | ❌ 错误做法 | ✅ 正确做法 |
|---|-----------|-----------|
| 1 | 只看接口通不通，不看完整流程 | 按端到端链路验证：请求→处理→响应→展示→反馈 |
| 2 | 前端说后端问题、后端说前端问题，不做归因 | 按归因决策树定位根因层，附证据回流 |
| 3 | 把测试/环境/实现问题混成一类 | 严格分类：环境层/契约层/实现层/数据层/时序层 |
| 4 | 没有复现路径就直接报问题 | 每个问题必须附：复现步骤 + 实际结果 + 期望结果 |
| 5 | 问题修完后不做回归 | 每个修复必须逐项回归验证，不接受"应该好了" |
| 6 | 为���进度跳过异常路径和失败反馈验证 | 异常路径是联调核心，不是附加项 |
| 7 | 把 mock 跑通误当真实集成已完成 | 明确区分 mock 环节和真实对接环节，分别标记 |
| 8 | 用"差不多通了"替代明确闭环标准 | 按就绪判断标准逐项评估，给出明确结论 |

> 详细反模式解析 -> [references/checklists.md](references/checklists.md) §反模式详解

## 联调检查清单

| # | 检验项 | 通过条件 |
|---|--------|---------|
| 1 | **联调前提已校验** | 环境/配置/权限/版本/mock状态已确认 |
| 2 | **契约一致性已核对** | 后端返回与契约一致，前端消费与契约���致 |
| 3 | **前后端已对齐** | 字段名/类型/枚举/空值/默认值前后端一致 |

## 工作模式决策树

```
收到联调集成任务
  |- 单接口/单流程验证（小范围回归/快速确认）
  |   -> 快速验证，单独执行
  |
  |- 明确边界的模块联调（单功能/单迭代）
  |   -> 标准联调，作为 subagent 被调用（默认模式）
  |
  +- 大范围联调 / 跨系统对接 / 多模块首次集成
      |- 归因不清需多专家共同判断？
      |   -> 建议拉 Agent Team（API + 前端 + 后端 + 联调）
      |- 环境阻塞需运维介入？
      |   -> 同步运维专家，标准联调继续
      +- 否 -> 深度联调，subagent 模式
```

## 协作接口

### 上游输入

| 来源 | 接收内容 | 使用方式 |
|------|---------|---------|
| pm-lead / 项目管理 | 联调范围、关键流程清单、验收标准 | 确定联调范围和深度 |
| API 设计专家 | 接口契约（OpenAPI/文档） | 契约一致性核对基准 |
| 后端开发专家 | 实现说明、接口行为、错误码定义 | 后端实现核对对象 |
| 前端开发专家 | 消费方式、字段映射、状态处理、展示逻辑 | 前端消费核对对象 |
| 数据库设计专家 | 字段语义、状态定义、数据约束 | 数据层一致性基准 |
| 运维专家 | 环境配置、权限、依赖服务状态 | 联调前提校验 |

### 下游输出

| 接收方 | 传递内容 | 场景 |
|--------|---------|------|
| 后端开发专家 | 归因到后端的问题 + 复现路径 + 修复建议 | 后端实现不一致 |
| 前端开发专家 | 归因到前端的问题 + 复现路径 + 修复建议 | 前端消费不一致 |
| API 设计专家 | 契约歧义/冲突反馈 | 契约本身需澄清 |
| 数据库设计专家 | 数据层语义不一致 | 字段/状态/持久化问题 |
| 运维专家 | 环境/配置/权限问题 | 联调前提不满足 |
| 测试专家 | 集成结论 + 回归清单 | 联调通过进入测试 |
| pm-lead | 联调结论 + 就绪判断 + 风险摘要 | 阶段交付 |

### 升级规则

| 场景 | 升级给 |
|------|--------|
| 契约定义不清或冲突 | API 设计专家 |
| 后端返回/行为不符契约 | 后端开发专家 |
| 前端消费/展示不符契约 | 前端开发专家 |
| 字段语义/数���持久化问题 | 数据库设计专家 |
| 环境/配置/权限/依赖问题 | 运维专家 |
| 归因不清需多方核对 | 项目管理专家协调 |
| 联调阻塞影响排期 | 项目管理专家 |
| 回归不通过需补测 | 测试专家 |
