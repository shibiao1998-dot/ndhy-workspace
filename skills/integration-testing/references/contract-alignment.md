# 契约一致性对齐

系统性核对 API 契约与前后端实际实现的一致性，覆盖字段、类型、状态、错误语义等维度。

> 契约是集成的基准线。如果契约本身有歧义，先回流 API 专家澄清，不自行解释。

---

## 契约 vs 后端实现一致性

### 核对维度

| 维度 | 核对内容 | 常见不一致 |
|------|---------|-----------|
| **路径与方法** | URL 路径、HTTP 方法、路径参数 | 路径拼写不一致、PUT vs PATCH 混用 |
| **请求参数** | Query/Body/Header 参数名、类型、必填性 | 参数名大小写不一致、必填标记缺失 |
| **响应结构** | 响应体字段名、类型、嵌套层级 | 字段名驼峰vs下划线、嵌套层级多一层 |
| **响应状态码** | HTTP 状态码与业务语义匹配 | 创建成功返回 200 而非 201、删除返回 200 而非 204 |
| **错误响应** | 错误码值、错误消息格式、错误响应结构 | 错误码命名不一致、缺少错误响应体 |
| **分页** | 分页参数名、响应中分页信息格式 | page/pageSize vs offset/limit、total 字段缺失 |
| **空值处理** | null vs 空字符串 vs 不返回该字段 | 契约没说清，后端选了不同策略 |
| **日期格式** | 时间戳 vs ISO 8601 vs 自定义格式 | 前端期望 ISO，后端返回时间戳 |

### 核对方法

```
对每个接口执行:
  1. 读取契约定义（OpenAPI/文档）
  2. 用真实请求调用后端接口
  3. 逐字段对比:
     - 路径是否匹配
     - 请求参数名/类型/必填��是否匹配
     - 响应字段名/类型/结构是否匹配
     - 错误码/状态码是否匹配
     - 分页格式是否匹配
  4. 记录所有不一致项（附实际值 vs 契约定义）
  5. 归因: 后端实��偏离契约 or 契约本身需更新
```

## 契约 vs 前端消费一致性

### 核对维度

| 维度 | 核对内容 | 常见不一致 |
|------|---------|-----------|
| **请求构造** | 前端发��的请求是否符合契约 | 多传/少传参数、参数名拼写错误 |
| **响应解析** | 前端解析响应的字段映射 | 取错字段名、嵌套路径错误 |
| **字段使用** | 前端使用了哪些字段、忽略了哪些 | 使用了契约未定义的字段、忽略了必要字段 |
| **类型处理** | 前端对字段类型的假设 | 把数字当字符串处理、把数组当对象处理 |
| **空值处理** | 前端对 null/undefined/空数组的处理 | 未处理 null 导致页面崩溃 |
| **错误处理** | 前端对错误响应的处理逻辑 | 未处理特定错误码、错误消息未展示 |
| **状态映射** | 前端把后端状态码映射为 UI 状态 | 状态映射遗漏、未知状态未处理 |

### 核对方法

```
对每个接口执行:
  1. 读取前端调用代码（API 层/store 层）
  2. 对比请求构造与契约:
     - URL/方法/参数是否匹配
     - Header 是否正确（Content-Type/Authorization）
  3. 对比响应消费与契约:
     - 字段名是否正确引用
     - 嵌套路径是否正确
     - 类型假设是否正确
  4. 检查错误处理:
     - 是否处理了契约定义的所有错误码
     - 错误消息是否正确展示
  5. 记录所有不一致项
  6. 归因: 前端消费偏离契约 or 契约表述不清
```

## 前后端直接对齐

当前后端都已实现但未通过契约中间层对比时，直接对比前后端的实际行为。

### 字段与状态一致性

| 对齐点 | 检查内容 | 不一致后果 | 排查方法 |
|--------|---------|-----------|---------|
| **字段名** | 前端读取的字段名 vs 后端返回的字段名 | 数据取不到/取错 | 对比前端代码和实际响应 |
| **字段类型** | 前端期望的类型 vs 后端实际返回的类型 | 类型转换错误/显示异常 | 检查实际响应的 JSON 类型 |
| **枚举值** | 前端枚举映射 vs 后端返回的枚举值 | 状态显示错误/未知状态 | 对比前后端枚举定义 |
| **必填/可选** | 前端是否处理了字段可能不存在的情况 | null 错误/页面崩溃 | 测试字段缺失场景 |
| **默认值** | 前端默认值 vs 后端默认值 | 初始状态不一致 | 创建资源后检查初始状态 |
| **空值语义** | null vs "" vs undefined vs [] | 行为不一致 | 测试各种空值场景 |

### 错误语义一致性

| 对齐点 | 检查内容 | 不一致后果 | 排查方法 |
|--------|---------|-----------|---------|
| **错误码含义** | 前端对错误码的理解 vs 后端发出错误码的意图 | 错误提示不准确 | 对比前后端错误码映射表 |
| **错误消息** | 前端展示的消息 vs 后端返回的消息 | 用户看到不友好/不准确的提示 | 触发各种错误检查展示 |
| **HTTP 状态码** | 前端对状态码的处理 vs 后端返回的状态码 | 错误分类错误 | 检查前端的状态码处理逻辑 |
| **降级行为** | 前端的降级策略 vs 后端的��常返回 | 降级不触发/降级行为错误 | 模拟后端异常检查前端行为 |
| **重试策略** | 前端重试条件 vs 后端的幂等性 | 重复提交/数据不一致 | 检查可重试错误码和幂等性 |

## 状态流转一致性

### 检查方法

```
对每个有状态的资源:
  1. 绘制后端状态机（从代码/文档提取合法状态转换）
  2. 绘制前端状态映射（从 UI 代码提取状态展示逻辑）
  3. 对比:
     - 前端是否覆盖了后端所有可能的状态值
     - 前端状态转换 UI 是否与后端状态机一致
     - 前端是否处理了"意外状态"（后端返回前端不认识的状态）
  4. 测试每个状态转换路径:
     - 操作 -> 状态变更 -> 前��展示更新
     - 验证前端是否正确轮询/监听状态变化
```

### 常见状态不一致

| 问题 | 表现 | 根因 |
|------|------|------|
| 状态值不匹配 | 后端返回 `PENDING`，前端只识别 `pending` | 大小写/命名风格不一致 |
| 状态遗漏 | 后端新增了状态，前端未处理 | 契约更新后前端未同步 |
| 状态展示错误 | 后端是 `PROCESSING`，前端显示"已完成" | 状态映射表有误 |
| 状态流转不同步 | 后端已变更状态，前端仍显示旧状态 | 缺少状态变更通知/轮询 |
| 并发状态冲突 | 两个操作同时改同一资源的状态 | 后端缺少并发控制 |

## 契约歧义处理

当发现契约本身有歧义时：

1. **不自行解释**：联调专家不替 API 专家定义契约语义
2. **记录歧义**：在 `contract-mismatch-report.md` 的"契约本身歧义"段记录
3. **回流 API 专家**：请求澄清，附前后端各自的理解
4. **暂停受影响的验证**：等待澄清后再继续该接口的联调
5. **不阻塞其他接口**：无歧义的接口继续联调

### 常见契约歧义

| 歧义类型 | 示例 | 处理 |
|---------|------|------|
| 空值语义未定义 | `name` 字段是否可以是 null？ | 回流 API 专家补充空值规则 |
| 部分更新语义 | PATCH 时未传的字段是"不改"还是"置空"？ | 回流 API 专家定义部分更新语义 |
| 枚举值不完整 | 状态字段只列了 3 种，实际可能有 5 种 | 回流 API 专家补全枚举定义 |
| 错误码重叠 | 两个不同场景返回相同错误码 | 回流 API 专家拆分错误码 |
| 分页行为未说明 | 超出最后一页返回空数组还是报错？ | 回流 API 专家补充边界行为 |
