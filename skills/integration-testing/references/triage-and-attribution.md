# 问题归因与回流

系统性归因集成问题的方法、归因决策树详解、回流路径规范和多方问题处理原则。

> 归因的目标是让问题到达能修复它的人手里，附带足够的证据和上下文。不是定罪，是修复。

---

## 归因方法论

### 归因六层模型

每个集成问题归因到以下六层之一：

| 层级 | 含义 | 典型问题 | 回流对象 |
|------|------|---------|---------|
| **环境层** | 配置/网络/权限/依赖服务导致 | 服务不可达/配置错误/权限缺失 | 运维专家 |
| **契约层** | 契约本身有歧义或缺陷 | 字段语义不清/枚举不全/行为未定义 | API 设计专家 |
| **后端实现层** | 后端实现偏离契约或逻辑错误 | 返回结构不匹配/错误码错误/状态流转错误 | 后端开发专家 |
| **前端实现层** | 前端消费偏离契约或逻辑错误 | 字段取错/状态映射错/错误处理缺失 | 前端开发专家 |
| **数据层** | 数据模型/初始数据/持久化导致 | 字段默认值不对/枚举定义不一致/数据状态错误 | 数据库设计专家 |
| **时序层** | 调用顺序/异步/并发导致 | 竞态条件/回调丢失/请求顺序错 | 前端或后端（视具体场景） |

### 归因三步法

```
Step 1: 确认现象
  - 精确描述问题: 什么操作 -> 期望什么 -> 实际什么
  - 记录复现路径: 具体步骤，可重复执行
  - 收集证据: 请求/响应截图、日志、数据库状态

Step 2: 定位问题层
  - 按归因决策树逐层排查
  - 从最容易排除的层开始（环境 -> 契约 -> 实现）
  - 确定问题出在哪一层

Step 3: 确定责任方并回流
  - 根据问题层确定回流对象
  - 附上: 复现路径 + 证据 + 期望vs实际 + 修复建议
  - 归因不清时标记待定，建议多方核对
```

## 归因决策树（详解版）

### 第一关: 是否环境问题?

```
问题现象
  |- 服务是否可达?（HTTP 请求能到达后端?）
  |   |- 不可达 -> 环境层（网络/部署/网关）
  |   +- 可达 -> 继续
  |
  |- 配置是否正确?（端点/密钥/环境变量）
  |   |- 配置错误 -> 环境层（配置管理）
  |   +- 配置正确 -> 继续
  |
  |- 权限是否到位?（认证/授权）
  |   |- 权限缺失 -> 环境层（权限配置）
  |   +- 权限正常 -> 排除环境层，进入第二关
```

### 第二关: 是否契约问题?

```
已排除环境层
  |- 契约对该场景有明确定义吗?
  |   |- 没有定义 -> 契约层（契约缺失）
  |   +- 有定义 -> 继续
  |
  |- 契约定义是否有歧义?（前后端理解是否一致?）
  |   |- 有歧义 -> 契约层（回流 API 专家澄清）
  |   +- 无歧义 -> 排除契约层，进入第三关
```

### 第三关: 是后端还是前端的问题?

```
已排除环境层和契约层
  |- 后端返回是否符合契约?
  |   |- 不符合 -> 后端实现层
  |   |   |- 响应结构不匹配 -> 后端字段/类型问题
  |   |   |- 错误码不正确 -> 后端错误处理问题
  |   |   |- 业务逻辑不正确 -> 后端逻辑问题
  |   |   +- 状态流转不正确 -> 后端状态机问题
  |   |
  |   +- 符合 -> 继续检查前端
  |
  |- 前端请求是否符合契约?
  |   |- 不符合 -> 前端实现层（请求构造问题）
  |   +- 符合 -> 继续
  |
  |- 前端对响应的处理是否正确?
  |   |- 不正确 -> 前端实现层（响应消费问题）
  |   +- 正确 -> 进入第四关（数据/时序层）
```

### 第四关: 数据层还是时序层?

```
已排除环境/契约/明显实现问题
  |- 是否与数据初始状态/持久化有关?
  |   |- 是 -> 数据层
  |   |   |- 字段默认值不一致 -> 数据库设计专家
  |   |   |- 枚举值定义不匹配 -> 数据库 + API 专家
  |   |   +- 数据状态异常 -> 数据库 + 后端专家
  |   +- 否 -> 继续
  |
  |- 是否与操作顺序/时机有关?
  |   |- 是 -> 时序层
  |   |   |- 前端发请求顺序不对 -> 前端专家
  |   |   |- 后端处理时序不对 -> 后端专家
  |   |   +- 异步通知/回调问题 -> 后端专家
  |   +- 否 -> 归因不清，标记待定
```

## 回流规范

### 回流信息模板

每次回流给对应专家时，必须包含：

```markdown
## 集成问题回流

| 项目 | 内容 |
|------|------|
| 问题编号 | {如 P0-1} |
| 归因层 | {如：后端实现层} |
| 优先级 | P0 / P1 / P2 |

### 问题描述
{一句话描述}

### 复现路径
1. {步骤1}
2. {步骤2}
3. {步骤3}

### 期望行为
{契约定义/正确行为}

### 实际行为
{当前实际表现}

### 证据
- 请求: {URL/Method/Body}
- 响应: {状态码/响应体}
- 日志: {关键日志行}（如有）
- 数据: {数据库状态}（如有）

### 修复建议
{建议的修复方向，不替代实现}

### 回归要求
{修复后联调专家需要验证的内容}
```

### 回流路径速查

| 问题类型 | 回流对象 | 附带什么 |
|---------|---------|---------|
| 后端返回与契约不符 | 后端开发专家 | 契约定义 + 实际响应 + 对比 |
| 前端消费与契约不符 | 前端开发专家 | 契约定义 + 前端代码引用 + 期望行为 |
| 契约有歧义 | API 设计专家 | 歧义点 + 前后端各自理解 |
| 数据层不一致 | 数据库设计专家 | 字段定义 + 实际数据 + 不一致点 |
| 环境/配置问题 | 运维专家 | 具体配置项 + 期望值 + 实际值 |
| 归因不清 | 项目管理专家 | 问题描述 + 已排查的层 + 建议拉谁 |

## 多方问题处理

当一个问题涉及多方时：

1. **拆分归因**：把一个问题拆成多个子项，各自归因
   - 例：前端传参格式错（前端问题）+ 后端未校验参数（后端问题）
2. **分别回流**：每个子项分别回流对应专家
3. **标注依赖**：如果修复有依赖关系，标明修复顺序
   - 例："先由 API 专家澄清契约，然后后端按澄清结果修改"
4. **升级协调**：如果多方需要共同讨论，升级项目管理专家协调

## 问题优先级定义

| 级别 | 含义 | 判断标准 | 处理要求 |
|------|------|---------|---------|
| **P0 阻断** | 必须修复才能继续联调 | 关键流程无法跑通/数据损坏/安全风险 | 立即修复 + 回归验证 |
| **P1 需修复** | 应修复，不影响其他流程继续联调 | 非关键路径失败/错误信息不准/降级异常 | 限期修复 + 回归验证 |
| **P2 建议** | 改了更好，不改不阻断 | 体验不佳/提示不友好/非标准处理 | 记录，由开发自行决定 |

### 优先级判断规则

- 影响关键业务流程正常路径 -> P0
- 影响异常路径/错误处理/降级行为 -> P1
- 影响体验/规范性/一致性但功能可用 -> P2
- 不确定时宁可高估一级
