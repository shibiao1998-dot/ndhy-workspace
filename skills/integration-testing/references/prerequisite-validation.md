# 联调前提校验

联调开始前的系统性检查，确保联调环境、配置、权限、依赖和版本全部就绪。前提不满足时不启动联调，避免把环境问题和实现问题混在一起排查。

> 前提校验的目标：确保联调失败时，问题一定出在实现层或契约层，而不是环境层。

---

## 前提校验清单

### 1. 环境可达性检查

| 检查项 | 验证方法 | 通过条件 | 失败处理 |
|--------|---------|---------|---------|
| 数据库连接 | 执行简单查询（如 `SELECT 1`） | 连接成功且响应 < 1s | 回流运维 / 检查连接配置 |
| 后端服务 | 调用健康检查端点（如 `/health`） | 返回 200 且服务状态正常 | 回流运维 / 检查部署状态 |
| 前端应用 | 访问前端入口页 | 页面正常加载 | 回流前端 / 检查构建部署 |
| API 网关 | 调用网关健康检查 | 网关可达且路由正常 | 回流运维 / 检查网关配置 |
| 依赖服务 | 逐一检查外部依赖（缓存/消息队列/文件存储等） | 各依赖服务可达 | 回流运维 / 确认哪些可用 mock 替代 |
| 网络连通性 | 前端→网关→后端→数据库 链路可达 | 端到端网络通畅 | 回流运维 / 检查防火墙和安全组 |

### 2. 配置正确性检查

| 检查项 | 验证方法 | 通过条件 | 常见问题 |
|--------|---------|---------|---------|
| 环境变量 | 检查后端/前端的环境变量配置 | 与联调环境匹配（非 dev/prod） | 混用环境配置 |
| API 端点 | 前端调用的 API 基础 URL | 指向正确的联调环境后端 | 前端仍指向 mock 或本地 |
| 数据库连接串 | 后端数据库配置 | 指向联调环境数据库 | 连接到开发库或生产库 |
| 密钥/凭证 | API 密钥、JWT 密钥、第三方凭证 | 联调环境专用凭证已配置 | 使用过期/错误的密钥 |
| 特性开关 | Feature flags 状态 | 联调相关功能已启用 | 功能被开关关闭 |
| 跨域配置 | CORS 设置 | 允许前端域名访问 | 跨域请求被拒绝 |

### 3. 权限检查

| 检查项 | 验证方法 | 通过条件 | 常见问题 |
|--------|---------|---------|---------|
| 认证机制 | 用测试账号获取 token | 认证流程正常，token 可获取 | 认证服务未部署/配置错误 |
| 角色权限 | 用不同角色账号调用受限接口 | 权限控制符合预期 | 角色未配置/权限表未初始化 |
| 跨服务授权 | 服务间调用权限 | 服务间认证正常 | 服务间密钥未配置 |
| 测试数据 | 联调所需的基础数据 | 必要的种子数据已就位 | 数据库为空/缺少基础数据 |

### 4. mock 状态确认

| 检查项 | 验证方法 | 通过条件 | 风险 |
|--------|---------|---------|------|
| mock 清单 | 列出所有仍在使用 mock 的环节 | 清单明确，每个 mock 有理由 | mock 冒充真实对接 |
| mock 一致性 | mock 返回结构是否与真实契约一致 | mock 数据格式符合契约定义 | mock 和真实接口结构不同 |
| mock 标记 | 联调报告中 mock 环节是否明确标记 | 所有 mock 环节已标记 | 混淆 mock 通过和真实通过 |
| 去 mock 计划 | 每个 mock 环节的去 mock 时间和条件 | 有明确的去 mock 计划 | 永久依赖 mock |

### 5. 版本匹配检查

| 检查项 | 验证方法 | 通过条件 | 常见问题 |
|--------|---------|---------|---------|
| 前端版本 | 检查前端构建版本/commit hash | 与联调计划版本一致 | 前端未部署最新版本 |
| 后端版本 | 检查后端部署版本 | 与联调计划版本一致 | 后端仍是旧版本 |
| 契约版本 | 对比当前契约文档版本 | 前后端基于同一版本契约开发 | 基于不同版本契约 |
| 数据库 schema | 检查数据库迁移状态 | 所有迁移已执行 | 缺少最新迁移 |
| 依赖版本 | 检查关键依赖库版本 | 无已知不兼容版本 | 依赖库版本冲突 |

## 前提校验流程

```
开始前提校验
  |
  |- 1. 环境可达性 -> 全部通过?
  |   |- 否 -> 记录不可达项 -> 回流运维 -> 停止联调
  |   +- 是 -> 继续
  |
  |- 2. 配置正确性 -> 全部正确?
  |   |- 否 -> 记录配置问题 -> 回流对应负责人 -> 停止联调
  |   +- 是 -> 继续
  |
  |- 3. 权限检查 -> 权限到位?
  |   |- 否 -> 记录权限缺口 -> 回流运维/后端 -> 停止联调
  |   +- 是 -> 继续
  |
  |- 4. mock 状态 -> 已明确标记?
  |   |- 否 -> 要求明确 mock 清单 -> 标记后继续
  |   +- 是 -> 继续
  |
  +- 5. 版本匹配 -> 版本一致?
      |- 否 -> 记录版本差异 -> 要求对齐 -> 停止联调
      +- 是 -> 前提校验通过 -> 进入联调
```

---

## 前提不满足时的处理原则

1. **不绕过**：前提不满足时不启动联调，避免浪费排查时间
2. **明确归因**：每个不满足项标明责任方和回流对象
3. **区分等级**：
   - **阻断**：环境不可达、版本严重不匹配 -> 必须修复后才能开始
   - **可接受**：部分非关键依赖使用 mock -> 明确标记后可开始，联调结论中注明
4. **记录留痕**：所有前提检查结果写入 `integration-checklist.md`

## 常见前提陷阱

| 陷阱 | 表现 | 防范 |
|------|------|------|
| **环境漂移** | 联调环境被其他任务改动 | 联调开始前重新验证环境状态 |
| **配置混用** | 后端用联调配置、前端仍指向 mock | 配置检查覆盖前后端双方 |
| **隐式依赖** | 某个服务依赖另一个未部署的服务 | 依赖链完整性检查 |
| **数据污染** | 上次联调残留数据影响本次结果 | 联调前确认数据初始状态 |
| **版本错位** | 前端基于 v2 契约，后端仍是 v1 | 版本匹配检查 |
