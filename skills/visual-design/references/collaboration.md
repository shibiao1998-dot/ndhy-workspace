# 视觉设计 — 协作接口与边界

## 上游输入

| 来源 | 接收内容 | 使用方式 |
|------|---------|---------|
| **体验设计专家** | 信息架构、交互模式、组件清单、体验语言、响应式策略 | **主要输入**，视觉设计在体验框架内展开 |
| 产品定义专家 | 品牌调性、产品定位 | 色彩/字体选择的品牌依据 |
| 用户研究专家 | 用户特征（年龄、设备、数字素养） | 可达性和适配策略的依据 |

→ 信息不足时有权要求上游补充

## 下游输出

| 接收方 | 传递内容 | 约束关系 |
|--------|---------|---------|
| **前端开发专家** | 设计 Token 文件 + 组件视觉规范 + 响应式规则 | 前端必须使用 Token，不允许硬编码 |
| 体验设计专家 | 视觉可达性反馈 | 如视觉约束导致体验方案需调整 |
| 项目管理专家 | 视觉规范完成度、Token 就绪状态 | 排期参考 |

## 与体验设计的边界划分

**核心原则**：体验设计回答"怎么交互"，视觉设计回答"长什么样"。

| 维度 | 体验设计（UX） | 视觉设计（Visual） |
|------|--------------|------------------|
| **信息架构** | ✅ 定义结构和层级 | ❌ 不改变结构 |
| **交互模式** | ✅ 定义交互行为（点击/滑动/拖拽） | ❌ 不改变交互逻辑 |
| **页面布局** | ✅ 定义区块组成和优先级 | 🔄 在布局框架内定义视觉处理 |
| **组件行为** | ✅ 定义组件何时出现、如何响应 | 🔄 定义组件视觉样式 |
| **体验语言** | ✅ 定义文案风格、产品人设 | ❌ 不改变文案内容 |
| **状态设计** | ✅ 定义哪些状态需要处理 | 🔄 定义每个状态的视觉表现 |
| **色彩体系** | ❌ | ✅ 品牌色/语义色/中性色全部 |
| **字体排印** | ❌ | ✅ 字体/字号/行高/字距 |
| **间距系统** | ❌ | ✅ 间距阶梯/网格/留白 |
| **组件外观** | ❌ | ✅ 圆角/阴影/色彩/尺寸 |
| **图标插画** | ❌ | ✅ 风格/规格/使用规范 |
| **动效参数** | 🔄 定义"何时需要动效" | ✅ 定义动效的缓动/时长/类型 |
| **设计 Token** | ❌ | ✅ Token 架构/命名/输出 |
| **响应式视觉** | 🔄 定义断点策略和内容取舍 | ✅ 定义字号/间距/网格的断点变化 |

**✅** = 完全负责 | **🔄** = 协作（有交叉但各有侧重） | **❌** = 不涉及

### 交叉区域的协作规则

1. **页面布局**：体验设计定义区块和优先级 → 视觉设计在此框架内处理视觉层次（字号、色彩、间距、阴影）
2. **组件设计**：体验设计定义组件行为和状态 → 视觉设计定义每个状态的外观参数
3. **动效设计**：体验设计定义"哪里需要动效" → 视觉设计定义"动效的具体参数"
4. **响应式**：体验设计定义断点策略和移动端取舍 → 视觉设计定义视觉元素在各断点的具体变化

### 禁止事项

- ❌ 视觉设计不改变信息架构（页面结构、导航层级）
- ❌ 视觉设计不改变交互逻辑（点击行为、流程步骤）
- ❌ 视觉设计不改变文案内容（按钮文字、提示语）
- ❌ 体验设计不指定具体色值、字号数值、阴影参数
- ❌ 体验设计不定义图标风格、插画风格
