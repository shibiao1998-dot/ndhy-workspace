## 质量检查清单（7 项）

逐条对照检查。每项给出"通过/不通过"判定，不通过必须说明原因。

### 1. 身份是否清晰

**检查标准**：能否用一句话说清这个角色干什么。

**通过**：
- 一句话定位明确、具体，不用读第二遍
- 包含专业领域和在项目中的位置
- 读完身份部分，知道这个角色和其他角色的区别

**不通过**：
- 定位模糊，如"负责相关工作"
- 和另一个角色的身份可以互换
- 需要读完整个设定才知道角色干什么

**检查方法**：遮住设定的其他部分，只看身份声明，问自己"我知道这个角色要做什么了吗？"

### 2. 认知是否具体

**检查标准**：有没有具体的专业知识和判断标准。

**通过**：
- 技术栈/方法论/工具具体到名称和版本
- 质量判断标准是可检验的（"响应时间 < 200ms"而非"快"）
- 思维模型描述了从输入到输出的思考路径

**不通过**：
- 只说"熟悉 XX 技术"，不说具体哪些
- 判断标准是主观的（"高质量"、"优秀"）
- 没有思维模型，不知道角色怎么思考

**检查方法**：找 3 个认知描述，问"AI 能据此做出具体判断吗？"

### 3. 行为是否可执行

**检查标准**：工作流程是否具体到每步都可执行。

**通过**：
- 每条规则是具体的行动指令
- 异常处理是"情况→动作"对应关系
- AI 读完能直接执行，不需要额外解释

**不通过**：
- 出现"尽力做好"、"高质量"等模糊描述
- 异常处理是"灵活处理"、"视情况而定"
- 工作流程缺步骤，从 A 到 C 中间缺 B

**检查方法**：逐条念出来，问"AI 能据此写出一段代码/文档/操作吗？"

### 4. 输入输出是否对得上

**检查标准**：与上下游角色的格式是否一致。

**通过**：
- 输入格式与上游角色的输出格式匹配
- 输出格式能被下游角色直接使用
- 必须/可选分明，不完整输入有处理方案

**不通过**：
- 上游输出 Markdown，本角色输入要求 JSON
- 本角色输出的东西下游不知道怎么用
- 没有说明缺输入时怎么办

**检查方法**：画一条线：上游输出 → 本角色输入 → 本角色输出 → 下游输入，每个箭头格式是否兼容？

### 5. 协作是否闭环

**检查标准**：上下游关系和反馈路径是否明确。

**通过**：
- 上游、下游、并行角色都明确列出
- 每个协作方向有具体的交互方式
- 依赖问题有明确的反馈路径（反馈给谁、怎么反馈）

**不通过**：
- 只说"和其他角色配合"，不说和谁
- 没有反馈路径，出了问题不知道找谁
- 缺并行角色的协调方式

**检查方法**：问"如果上游交付不合格，这个角色知道找谁吗？怎么找？"

### 6. 边界是否明确

**检查标准**：不该做的和越界动作是否列清楚。

**通过**：
- "不做什么"具体列出，不是空泛的"不做其他事"
- 越界标准动作是"停止+反馈"，不是"灵活处理"
- 覆盖了主要的越界风险场景

**不通过**：
- 没有边界定义
- 边界模糊，如"不做与专业无关的事"
- 越界后没有标准动作

**检查方法**：想 3 个边界场景，看设定中是否有对应的处理方式。

### 7. 语气是否一致

**检查标准**：整份设定的口吻是否统一。

**通过**：
- SOUL.md 全程第二人称（"你…"）
- STANDARDS.md 风格统一（条目式、精炼）
- 性格特质和工作风格呼应，不自相矛盾

**不通过**：
- SOUL.md 一会儿"你"一会儿"该角色"
- 性格特质说"谨慎"但工作风格说"快速迭代不纠结"
- 前后语气风格不一致

**检查方法**：通读全文，标注人称和语气变化点。

## 反模式清单（5 项）

以下 5 种写法是设定中的"毒药"——看到就消灭。

### 反模式 1：空洞赞美

**毒药**："你是一个优秀的 XX 专家"

**为什么有毒**：AI 不知道"优秀"是什么意思。这句话等于没说。

**修正**：改为具体定位 + 核心能力 + 方法

| 毒药 | 解药 |
|------|------|
| "你是一个优秀的前端开发专家" | "你专注于 React 框架构建 Web 应用，用组件化思维把复杂界面拆成可复用的积木" |
| "你是一个经验丰富的项目经理" | "你负责将需求拆解为可执行的任务、通过 spawn 调度子 Agent、对照验收标准验收交付物" |

### 反模式 2：无标准的"做好"

**毒药**："请尽你所能做好"

**为什么有毒**：没有标准就没有质量底线。AI 不知道"好"的标准在哪。

**修正**：改为具体的交付标准

| 毒药 | 解药 |
|------|------|
| "请尽力写出好的代码" | "代码必须：(1) 通过 ESLint 零 error (2) 包含 TypeScript 类型定义 (3) 处理 loading/error 状态" |
| "请写出优质的文档" | "文档必须：(1) 包含使用示例 (2) 覆盖所有公开 API (3) 格式符合 Markdown 规范" |

### 反模式 3：模糊的"灵活处理"

**毒药**："遇到问题灵活处理"

**为什么有毒**：AI 理解的"灵活"可能是你最不想看到的操作。没有明确指令，AI 会编造解决方案。

**修正**：改为具体情况对应具体动作

| 毒药 | 解药 |
|------|------|
| "遇到不确定的需求灵活处理" | "需求不明确时：停止执行，列出不明确的点，请求项目管理澄清" |
| "API 接口有问题时自行解决" | "API 返回格式与文档不符时：(1) 记录差异 (2) 按文档格式做 mock (3) 反馈 API 设计专家" |

### 反模式 4：不教只说"应该了解"

**毒药**："你应该了解 XX 知识"

**为什么有毒**：AI 的知识可能过时、不完整、或与项目规范不同。"应该了解"等于假设 AI 已经知道了。

**修正**：直接写进认知框架，或指明确切来源

| 毒药 | 解药 |
|------|------|
| "你应该了解 React 最佳实践" | 直接在专业标准中写："组件设计遵循单一职责、Props 接口清晰、支持组合模式"，或指明："React 最佳实践参考 Skill `react-best-practices`" |
| "你应该了解公司的编码规范" | 直接在行为规则中写明规范内容，或指明参考文件路径 |

### 反模式 5：缺失输入输出格式

**毒药**：没有定义输入输出格式

**为什么有毒**：上下游交接鸡同鸭讲。上游不知道给什么格式，下游不知道拿到的是什么。

**修正**：精确定义结构和格式

| 毒药 | 解药 |
|------|------|
| "接收设计稿开始工作" | "接收：UI 设计稿(Figma链接/图片, 必须) + API 文档(Markdown/Swagger, 必须) + 交互说明(文字, 可选)" |
| "输出完成的代码" | "输出：React 组件代码(TypeScript) + 组件说明文档(Markdown) + 自测报告(四态测试结果)" |

## 检查执行流程

创造每个角色设定后，按以下顺序执行检查：

```
Step 1：通读全文，标注直觉上"觉得不对"的地方
    ↓
Step 2：逐条过 7 项质量检查清单
    ↓
Step 3：全文搜索 5 项反模式关键词
    ├─ 搜索："优秀"、"出色"、"经验丰富" → 反模式 1
    ├─ 搜索："尽力"、"做好"、"尽可能" → 反模式 2
    ├─ 搜索："灵活"、"视情况"、"酌情" → 反模式 3
    ├─ 搜索："应该了解"、"需要知道" → 反模式 4
    └─ 检查：是否定义了输入/输出格式 → 反模式 5
    ↓
Step 4：检查双文件映射是否正确
    ├─ SOUL.md 是否只包含 A + B核心？
    └─ STANDARDS.md 是否包含 B标准 + C + D + E + F + G？
    ↓
Step 5：模拟下游消费
    ├─ Skills 专家能据此配技能吗？
    ├─ 项目管理能据此排期吗？
    └─ 角色本身能据此行动吗？
    ↓
全部通过 → 交付
任何不通过 → 标注问题 → 修正 → 重新检查
```
