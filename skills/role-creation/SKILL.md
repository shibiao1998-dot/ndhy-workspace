# 专家角色创造全流程

## 输入与输出

**输入**：
- 专家角色需求清单（来自流程设计专家，含 7 维度定义）
- 角色协作拓扑图
- 已有专家池（路径：`roles/` 目录）

**输出**（全部 5 项）：
1. 匹配报告 — 哪些角色从池中复用、哪些需要新建
2. 新角色的 SOUL.md + STANDARDS.md
3. Skills 需求说明（每个角色）
4. 组建完成的项目团队清单
5. 新角色入池记录

---

## 核心流程：5 步工作法

### Step 1：池中匹配

**目标**：遍历需求清单，逐个与已有专家池对比。

**操作**：
1. 读取 `roles/README.md` 获取当前专家池索引
2. 遍历需求清单中每个角色需求
3. 对每个需求，按 3 个维度判断匹配：

| 匹配维度 | 判断标准 | 示例 |
|----------|---------|------|
| 专业领域一致 | 池中角色的专业领域与需求角色完全重合 | 需要"React前端开发" vs 池中"前端开发专家（React）" → 一致 |
| 能力范围覆盖 | 池中角色的核心能力完全覆盖需求角色所需能力 | 需要"组件开发+状态管理+API联调" vs 池中角色能力含这三项 → 覆盖 |
| 输入输出兼容 | 池中角色的输入输出格式与本项目上下游角色兼容 | 池中角色输出"React组件+文档"，下游角色需要"前端代码+组件说明" → 兼容 |

**关键规则**：
- **三维度全部满足 = 完全匹配**，直接调用
- **任一维度不满足 = 不匹配**，新建
- **不存在"部分匹配后修改"** — 已有角色的设定是资产，不改

**输出格式**：

```
## 匹配报告

### 完全匹配（直接调用）
| 需求角色 | 匹配池中角色 | 匹配依据 |
|----------|-------------|---------|
| [需求名] | [池中角色名] | 领域一致 + 能力覆盖 + 输入输出兼容 |

### 不匹配（需要新建）
| 需求角色 | 不匹配原因 |
|----------|-----------|
| [需求名] | [具体哪个维度不满足，为什么] |
```

### Step 2：创造专家设定

**目标**：为每个不匹配的角色编写完整设定。

**操作**：
1. 读取角色需求的 7 维度定义（来自流程设计专家）
2. 按 7 模块结构编写设定 → 详见 `references/seven-modules.md`
3. 将 7 模块映射到双文件：
   - **SOUL.md** ← 模块 A（身份声明）+ 模块 B 核心部分（信条、性格特质、工作风格）
   - **STANDARDS.md** ← 模块 B 标准部分（专业标准）+ C（行为规则）+ D（输入规范）+ E（输出规范）+ F（协作规则）+ G（边界禁区）
4. 逐条过质量检查清单 → 详见 `references/quality-gate.md`
5. 扫描反模式 → 详见 `references/quality-gate.md`

**双文件映射规则**：

```
SOUL.md 结构：
  # [emoji] [角色名] — [一句话定位]
  ## 身份        ← 模块A：角色名称、定位、专业领域、项目位置
  ## 信条        ← 模块B核心：1-2句话的专业信念
  ## 性格特质    ← 模块B核心：3-5个带标签的特质描述
  ## 工作风格    ← 模块B核心：4-6条工作习惯

STANDARDS.md 结构：
  # [emoji] [角色名] — 专业标准
  ## 专业标准    ← 模块B标准：具体的专业知识和判断标准
  ## 工作原则    ← 模块C精髓：核心行为准则（5-7条）
  ## 产出规范    ← 模块D+E：输入格式、输出格式、质量自检清单
  ## 适用技能    ← 表格：技能名 + 使用场景
  ## 协作规范    ← 模块F：上游/下游/并行角色及交互方式
  ## 边界与禁区  ← 模块G：不做什么 + 越界标准动作
```

**SOUL.md 写法要求**：
- 身份部分：第二人称（"你叫XX"），语气像在告诉角色它是谁
- 信条：一段话，浓缩角色的专业灵魂，读完就知道这个角色的核心追求
- 性格特质：每条用**加粗标签**开头 + 具体描述，体现人格而非功能
- 工作风格：**先动词**，每条描述一个可观察的行为习惯

**STANDARDS.md 写法要求**：
- 专业标准：每条用**加粗标签**开头 + 具体的专业能力描述
- 工作原则：编号列表，每条是一个可执行的行为准则
- 产出规范：明确输入（必须/可选）、输出（格式+内容）、质量自检清单
- 协作规范：用上游/下游/并行结构，明确交互方式
- 边界禁区：不做什么 + 越界时的标准动作（停止并反馈）

### Step 3：提交 Skills 需求

**目标**：为每个角色（包括匹配复用的）梳理所需 Skills。

**操作**：
1. 分析角色的工作流程，识别需要哪些工具支持
2. 按以下格式输出

**输出格式**：

```
角色：[角色名]
所需 Skills 类型：
  - 类型：[工具类型，如"代码编辑器"/"搜索引擎"/"文档生成"]
    用途：[角色用这个工具做什么]
    具体要求：[对工具的具体要求]
```

**原则**：只说"需要什么"，不说"怎么配"。配 Skills 是 Skill 专家（贺匠心）的事。

**常见 Skills 类型参考**：

| 类型 | 典型用途 |
|------|---------|
| 代码开发 | 编写、调试、运行代码 |
| 文档读写 | 读取/创建飞书文档、Markdown |
| 搜索调研 | Web 搜索、信息收集 |
| 数据处理 | 数据分析、表格操作 |
| 设计工具 | UI 设计、图片生成 |
| 项目管理 | 任务跟踪、进度管理 |
| 质量检查 | 代码审查、测试执行 |
| 沟通协作 | 消息读取、通知发送 |

### Step 4：接收 Skills 配置结果

**目标**：确认所有角色的 Skills 已就绪。

**操作**：
1. 接收贺匠心（Skill 专家）的配置结果
2. 检查每个角色是否拥有所需的 Skills
3. 确认完毕，进入下一步

**判断标准**：角色设定（SOUL.md + STANDARDS.md）+ 适配 Skills = 可上岗。缺任何一项不算就绪。

### Step 5：组建项目团队

**目标**：交付完整团队清单 + 新角色入池。

**操作**：
1. 汇总所有角色（复用 + 新建），输出项目团队清单
2. 将新建角色的设定文件存入 `roles/[角色名]/`
3. 新角色入池 → 详见 `references/expert-pool.md`
4. 交付给项目管理

**项目团队清单格式**：

```
## 项目团队

| 角色 | 来源 | 设定路径 | 状态 |
|------|------|---------|------|
| [角色名] | 池中复用 / 新建 | roles/[路径] | 就绪 |

### 协作关系
[简述关键依赖和并行关系]

### 关键路径
[标注关键路径上的角色和步骤]
```

## 8 条设计原则速查

| # | 原则 | 一句话 |
|---|------|-------|
| 1 | 灵魂与肉体分离 | 设定（SOUL.md + STANDARDS.md）和 Skills 独立，互不耦合 |
| 2 | 只创造不修改 | 已有角色不改，不匹配就新建 |
| 3 | 无状态释放 | 执行层专家任务结束即释放，设定必须自包含 |
| 4 | 设定即资产 | 每个设定入池后可被未来项目复用 |
| 5 | 精确优于通用 | 宁可新建精准角色也不用万能模糊角色 |
| 6 | 下游可消费 | 设定写完必须验证下游看得懂 |
| 7 | 三方分工 | 编剧（殷铸魂）创设定、选角导演（程织枢）定需求、道具师（贺匠心）配技能 |
| 8 | 双文件映射 | 7 模块 → SOUL.md + STANDARDS.md |

## 完整工作示例

**场景**：项目需要 6 个角色 — 项目管理 / 教学设计 / 脚本撰写 / AI视频生成 / 视觉设计 / 内容审核。

**Step 1 匹配**：
- 遍历 `roles/` 目录，逐个比对
- 结果：项目管理、脚本撰写、视觉设计、内容审核 → 池中已有，完全匹配
- 教学设计、AI视频生成 → 不匹配，需新建

**Step 2 创造**（以"教学设计专家"为例）：

1. 读取流程设计专家给出的 7 维度定义
2. 按 7 模块展开：
   - A 身份声明：教学设计专家，专注于在线课程的教学结构和学习路径设计
   - B 认知框架：教学设计理论（ADDIE、布鲁姆分类）、学习心理学、成人学习特点
   - C 行为规则：分析学习目标→设计课程大纲→拆分知识点→设计评估方式→自检
   - D 输入规范：课程主题(必须) + 目标受众(必须) + 知识范围(必须) + 时长约束(可选)
   - E 输出规范：课程大纲 + 每课时知识点拆解 + 学习目标(可验证) + 评估设计
   - F 协作规则：上游项目管理，下游脚本撰写，并行视觉设计
   - G 边界禁区：不写脚本内容、不做视觉设计、不负责视频制作
3. 映射到 SOUL.md + STANDARDS.md
4. 过 7 项质量检查 → 全部通过
5. 扫描反模式 → 无命中

**Step 3 提 Skills 需求**：
```
角色：教学设计专家
所需 Skills 类型：
  - 类型：搜索调研
    用途：查找目标领域的知识结构和教学最佳实践
    具体要求：支持学术类和专业类内容搜索
  - 类型：文档读写
    用途：输出课程大纲和教学设计文档
    具体要求：支持结构化 Markdown 输出
```

**Step 4 确认**：Skill 专家确认配置完毕。

**Step 5 组建**：
- 输出 6 人团队清单（4 复用 + 2 新建）
- 新建的教学设计专家、AI视频生成专家 → 入池，标签化
