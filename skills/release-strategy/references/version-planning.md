# 版本规划与发布类型

## 一、版本体系设计

版本号是一套"沟通语言"——让所有人一眼看出这次更新"有多大变化"。

### 语义化版本规则

```
主版本号.次版本号.修订号
例：2.3.1
```

| 变化类型 | 示例 | 含义 | 用户影响 |
|----------|------|------|---------|
| **主版本号变化** | 1.0 → 2.0 | 大改版，功能或体验有重大变化 | 可能不兼容旧版，需重新学习部分功能 |
| **次版本号变化** | 2.3 → 2.4 | 新增功能或明显优化 | 向下兼容，用户可能需了解新功能 |
| **修订号变化** | 2.3.1 → 2.3.2 | 修bug、小优化 | 不影响使用习惯，用户甚至可以不知道 |

### 版本号示例

- `1.0 → 2.0`：整个产品从Web端重构为全平台版本
- `2.3 → 2.4`：新增AI教案生成功能
- `2.3.1 → 2.3.2`：修复某个页面的显示问题

## 二、发布类型

| 发布类型 | 内容 | 节奏 | 风险等级 |
|----------|------|------|---------|
| **大版本发布** | 全新产品/重大改版 | 每学期或每年1次 | 极高 |
| **功能发布** | 新功能上线 | 每2-4周1次 | 中高 |
| **优化发布** | 体验优化、性能改进 | 每1-2周1次 | 中低 |
| **修复发布** | 紧急bug修复 | 随时 | 低（但紧急） |

### 各类型的发布要求

**大版本发布**
- 完整灰度流程（5阶段）
- 完整培训方案
- 全渠道沟通
- 数据迁移方案（如涉及）
- War Room 模式发布
- 至少2周观察期
- 深度复盘

**功能发布**
- 标准灰度流程（可跳过内部灰度）
- 定向培训（针对使用该功能的用户）
- 重点渠道沟通
- 1周观察期
- 标准复盘

**优化发布**
- 简化灰度（内部验证 + 小比例灰度）
- 轻量沟通（产品内更新日志）
- 2-3天观察期

**修复发布**
- 快速验证后发布
- 如修复的是用户已感知的问题，在社群通知"已修复"
- 后台修复无需通知

## 三、教育行业发布日历

教育产品的发布窗口不是你想发就能发的，必须看教育行业的"日历"。

### 春季学期（2月-6月）

| 月份 | 时段 | 发布策略 | 风险等级 |
|------|------|---------|---------|
| 2月上旬 | 寒假尾声 | **黄金窗口** — 大版本发布最后机会 | 🟢 低 |
| 2月中下旬 | 开学前2周 | **冻结期** — 仅允许P0级bug修复 | 🔴 极高 |
| 3月 | 开学后1个月 | **高度谨慎** — 仅修复和小优化 | 🟠 高 |
| 4月-5月上旬 | 学期中段 | **常规窗口** — 功能迭代主要时段 | 🟡 中 |
| 5月中下旬 | 考试周前2周 | **冻结期** — 禁止任何发布 | 🔴 极高 |
| 6月 | 期末/学期末 | **谨慎期** — 收尾发布，为暑假准备 | 🟠 高 |

### 暑假（7月-8月）

| 月份 | 时段 | 发布策略 | 风险等级 |
|------|------|---------|---------|
| 7月-8月 | 暑假 | **最佳窗口** — 大版本发布、系统升级、数据迁移 | 🟢 低 |

### 秋季学期（9月-1月）

| 月份 | 时段 | 发布策略 | 风险等级 |
|------|------|---------|---------|
| 8月下旬-9月上旬 | 开学前2周 | **冻结期** — 仅允许P0级bug修复 | 🔴 极高 |
| 9月中下旬 | 开学后1个月 | **高度谨慎** — 仅修复和小优化 | 🟠 高 |
| 10月-11月上旬 | 学期中段 | **常规窗口** — 功能迭代主要时段 | 🟡 中 |
| 11月中下旬-12月上旬 | 考试周前2周 | **冻结期** — 禁止任何发布 | 🔴 极高 |
| 12月中下旬 | 期末/年末 | **谨慎期** — 收尾发布，为寒假准备 | 🟠 高 |
| 1月 | 寒假 | **黄金窗口** — 大版本发布 | 🟢 低 |

### 关键约束

- 一个学期约 18-20 周教学时间
- 能做发布的窗口约 8-10 周
- 放在互联网公司可以每天发布的节奏，在教育行业可能一个月只能发一次
- **每一次发布窗口都要珍惜——发就要发好，不要浪费窗口**

## 四、版本规划的节奏感

### 规划原则

- **每个版本都要有"主题"** — 这个版本解决什么核心问题
- **不要把所有东西塞到一个版本** — 功能太多 = 风险太大 = 出问题找不到原因
- **每个版本的范围要可控** — 能在2-4周内完成的量
- **每个版本都要有"发布价值"** — 不值得发布的就别发，攒一攒一起发

### 版本优先级排序框架

| 优先级 | 类别 | 示例 |
|--------|------|------|
| **必须发** | 影响核心使用的bug、合规性要求、安全漏洞 | 登录失败、数据丢失、安全补丁 |
| **应该发** | 用户呼声最高的功能、有明确商业价值的功能 | 教师强烈要求的备课工具、付费转化关键功能 |
| **可以发** | 锦上添花的优化、内部效率提升 | UI微调、管理后台优化 |
| **先不发** | 没验证过需求的新功能、技术团队的"理想方案" | 概念阶段的创新功能 |

### 版本规划决策

```
是否必须发？ → 是 → 排入最近版本
  ↓ 否
是否应该发？ → 是 → 排入下个迭代
  ↓ 否
是否可以发？ → 是 → 积累到足够量后统一发
  ↓ 否
暂时搁置，放入需求池等待验证
```
