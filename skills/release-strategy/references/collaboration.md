# 发布策略 — 协作接口

## 上游输入

| 来源 | 接收内容 | 重点关注 |
|------|---------|---------|
| 产品定义文档 | 产品定位、功能范围、目标用户 | 这次发布的核心价值主张 |
| 体验设计文档 | 用户流程、交互方案 | 用户上手难度、学习成本 |
| 技术架构文档 | 系统架构、发布能力、性能指标 | 技术层面的灰度能力 |
| 项目计划 | 开发进度、里程碑 | 什么时候能发、测试时间 |
| 运营数据 | 现有用户情况、使用数据 | 使用习惯、迁移风险 |

## 下游输出

| 交付物 | 接收方 | 用途 |
|--------|--------|------|
| 发布策略文档 | 产品/技术/运营团队 | 全团队对齐发布计划和职责 |
| 试点方案 | 运营/客户成功团队 | 执行试点计划 |
| 培训方案 | 培训/客户成功团队 | 设计和执行用户培训 |
| 沟通计划 | 市场/运营团队 | 发布相关内外部沟通 |
| 发布检查清单 | 全团队 | 发布前逐项确认 |
| 发布复盘报告 | 管理层/全团队 | 经验沉淀、持续改进 |

## 升级规则

| 场景 | 升级给 |
|------|--------|
| 发布内容/范围变更 | 产品定义专家 |
| 技术发布能力/灰度基础设施 | 技术架构专家 |
| 发布执行中的技术问题 | 后端开发/运维专家 |
| 用户反馈涉及体验重设计 | 体验设计专家 |
| 发布延期/风险升级 | 项目管理专家 |
