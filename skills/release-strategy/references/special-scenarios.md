# 特殊场景发布策略

教育产品会遇到一些特殊的发布场景，需要专门的策略。

## 一、全新产品首次发布（从0到1）

> 这是风险最高、也最重要的一次发布。

### 核心挑战

- 用户对产品一无所知，从"知道"到"愿意用"的路很长
- 没有历史数据做参考，所有预判都可能是错的
- 系统从未承受过真实用户使用，各种问题会冒出来

### 4阶段发布策略

#### 阶段一：内部打磨（发布前4-2周）

```
做什么：公司内部全员使用，当"第一批用户"
关键点：
  → 尤其让非技术人员参与——他们的视角最接近真实用户
  → 目标：消除最明显的"硬伤"
产出：一堆要改的bug和体验问题
```

#### 阶段二：定向邀请（3-5所种子学校）

```
做什么：带着产品上门服务（不是"上线让人来用"）
关键点：
  → 产品团队要驻校或高频远程支持
  → 目的不是推广，是"在真实环境中验证产品"
产出：一堆要改的问题 + 第一批使用案例
```

#### 阶段三：小范围试点（10-20所学校）

```
做什么：基于种子学校经验，优化产品和培训方案
关键点：
  → 开始测试"可复制的推广方案"——不依赖产品团队驻校也能用起来
  → 形成第一批"成功案例"和"种子教师"
产出：可复制的推广方案 + 成功案例
```

#### 阶段四：扩大推广

```
做什么：基于试点成功经验，逐步扩大
关键点：
  → 每扩大一批，都要验证"方案是否仍然有效"
  → 不要着急追数量
  → 100所学校用得好 > 1000所学校用不动
```

## 二、大版本升级（从1.0到2.0）

### 核心挑战

- 已有用户对旧版本有使用习惯和情感依赖
- 新版本再好，"和以前不一样"就会引发抵触
- 数据迁移和功能映射是技术难题

### 3阶段升级策略

#### 阶段一：提前预热（发布前1-2个月）

- 让用户知道"新版本要来了"
- 展示新版本亮点——不是功能列表，而是"能帮你解决什么问题"
- 招募"内测体验官"——让部分用户提前用，既收集反馈又创造期待

#### 阶段二：平滑过渡

- 新旧版本**并行运行**一段时间（建议4-8周）
- 提供"一键切换"——用户可以随时回到旧版本
- 在旧版本中温和引导——"试试新版本？"而不是"您必须升级"
- 当 **90%+活跃用户已迁移**到新版本，再关闭旧版本

#### 阶段三：关闭旧版本

- 提前至少**1个月**通知旧版本关闭日期
- 提供**数据导出功能**——让用户有"带走自己东西"的安全感
- 关闭前最后一周**每天提醒**
- 关闭后保留**只读访问权限**一段时间（让用户还能查看历史数据）

## 三、AI功能发布

> AI功能的发布有其特殊性——用户对AI的期望管理尤为重要。

### 特殊挑战

| 挑战 | 说明 |
|------|------|
| 期望过高 | 以为AI什么都能做、什么都做得对 |
| 期望过低 | "AI不靠谱，不如自己来" |
| 输出不确定 | 同样输入不一定得到同样输出 |
| 教育容错低 | AI生成内容有知识性错误后果严重 |

### 4大策略

#### 1. 期望管理

```
发布前就说清楚"AI能做什么"和"AI不能做什么"

✗ "AI帮你自动生成完美教案"
✓ "AI帮你生成教案初稿，你来把关和调整"

定位：人类的辅助工具，不是万能助手
```

#### 2. 渐进暴露

```
不要一上来就推出全部AI功能

推荐顺序（按风险从低到高）：
  1. "AI帮你总结知识点"（风险低，验证信任）
  2. "AI帮你出测验题"（需要教师审核）
  3. "AI帮你做教学设计"（需要教师深度参与）

先推最成熟、最安全的——让用户建立信任后再推更复杂的
```

#### 3. 安全兜底

- AI生成内容默认标注"AI生成，请核实"
- 提供"一键反馈"——用户发现问题能方便反馈
- 后台监控AI输出质量——发现质量下降时主动干预
- 敏感场景（出试卷、写评语）增加人工审核环节

#### 4. 效果展示

- 用"前后对比"让用户感受价值——"以前备课花2小时，现在30分钟"
- 用"同行案例"增强信心——"李老师用AI生成了20份差异化练习"
- 提供"试用空间"——让教师可以无压力试用，错了也没关系

## 四、多端/多平台同步发布

### 挑战

- 各端开发进度不一定一致
- APP端需要应用商店审核，时间不可控
- 不同端的用户群体可能不同
- 各端的功能范围可能不同

### 发布策略

#### 基本原则

- 核心功能各端同步发布——不能出现"Web端有但APP端没有"的割裂感
- 非核心功能可以分端分批上线
- 如果APP端审核可能延迟，Web端可以先发但要做好沟通

#### APP审核策略

| 动作 | 时间要求 |
|------|---------|
| 提交审核 | 至少提前1周 |
| 大版本提交 | 至少提前2周 |
| 准备被拒备选方案 | 与提交同步 |

> iOS和Android审核标准不同，分别准备。

#### 版本同步策略

- 制定"功能矩阵表"——每个功能在每个端的支持情况
- 某端暂不支持某功能时，在产品内明确说明
- 各端更新日志分别编写——侧重点不同

## 五、面向政府/教育局的发布

### 特殊要求

| 要求 | 说明 |
|------|------|
| 明确验收标准 | 必须全部通过才算交付 |
| 前置条件 | 可能需要"等保测评""安全审计"等 |
| 决策链条长 | 发布时间可能受政府流程影响 |
| 稳定性优先 | 对"稳定性"的要求远高于"新功能" |
| 不接受灰度 | 通常要么上线要么不上线 |

### 策略调整

1. **充分预演** — 发布前模拟验收流程
2. **完整交付文档** — 不只是产品，还有培训方案、运维方案等
3. **专人对接** — 安排专人对接政府侧项目负责人
4. **充足缓冲** — 政府流程的时间经常超出预期
5. **保障期** — 发布后通常要求至少1-3个月的驻场或高频支持

## 六、教育行业特别提醒

### 教师是最特殊的用户群体

- 时间被课表严格切割——培训必须见缝插针
- 信息化水平参差不齐——不能假设"基本操作都会"
- 对"额外负担"极其敏感——产品让他们觉得"又多了个事"就死了
- 对"同行推荐"的信任远大于"厂商宣传"
- 一旦形成使用习惯，忠诚度很高——但前提是你先帮他们养成习惯

### 不要低估"关键人物"的影响力

| 关键人物 | 影响力 |
|----------|--------|
| 校长的支持 | = 全校推行的通行证 |
| 教研组长的认可 | = 整个学科组的采纳 |
| 信息化老师的配合 | = 技术问题的本地化解决 |

> 关键人物的沟通不能群发通知——必须一对一、面对面。

### 教育行业的口碑效应

- 教育圈子不大——一个区域的学校间消息传得很快
- "XX学校用了这个系统很好用"能帮你打开一片市场
- "XX学校被这个系统坑了"也能让你失去一片市场
- **首批学校的体验一定要做到极致——他们是你的"活广告"**
- 宁可首批少推几所学校但服务好，也不要铺开后服务跟不上

### AI功能在教育行业的特殊敏感性

- 家长对"AI教孩子"天然有疑虑——发布沟通需特别谨慎
- 教育管理部门对AI的态度还在变化中——紧跟政策，不冒进
- "AI替代教师"是教师群体的敏感神经——永远定位为"AI辅助教师"
- AI生成内容的准确性在教育场景中是底线——一个知识点的错误都不能容忍
- **发布AI功能时，先展示"如何帮教师减负"，而不是"AI多厉害"**
