# 发布策略 — 教育发布日历 & 灰度阶段

## 教育发布日历速查

| 时段 | 策略 | 允许的发布类型 |
|------|------|---------------|
| **寒暑假** | 大版本发布的黄金窗口 — 用户少、有时间培训、开学直接用新版 | 所有类型 |
| **开学前2周** | 冻结期 — 系统必须绝对稳定 | 仅P0级bug修复 |
| **开学后1个月** | 高度谨慎期 — 密切监控系统指标 | 仅修复和小优化 |
| **学期中段** | 常规发布窗口 — 功能迭代主要时段 | 按正常节奏灰度发布 |
| **考试周前2周** | 冻结期 — 出问题无法交代 | 禁止任何发布 |
| **期末/年末** | 谨慎期 — 收尾性发布，为假期大版本做准备 | 技术优化、收尾发布 |

> 教育行业一个学期约18-20周教学时间，能做发布的窗口可能只有8-10周。每次发布窗口都要珍惜。

## 灰度阶段速查

| 阶段 | 范围 | 目的 | 时长 | 通过标准 |
|------|------|------|------|---------|
| **1. 内部灰度** | 公司内部团队 | 发现最明显的bug和体验问题 | 3-5个工作日 | 无P0/P1级bug，核心流程走通 |
| **2. 种子用户** | 3-5所试点校，每校5-10位教师 | 验证真实场景可用性 | 1-2周 | 使用成功率>95%，满意度>70% |
| **3. 试点校全量** | 试点学校所有目标用户 | 验证规模稳定性和培训方案 | 2-4周 | 系统稳定，培训方案验证有效 |
| **4. 扩大灰度** | 10-30所学校，覆盖不同类型 | 验证方案普适性 | 2-4周 | 方案无需单校特殊适配 |
| **5. 全量发布** | 所有目标用户 | 全面铺开 | 分批开放 | 第1天30%→第2天50%→第3天100% |

**开门标准**：P0/P1全修复 + 成功率达标 + 性能正常 + 无方向性问题 + 负责人确认
**关门标准**：P0级bug无法短期修复 / 成功率低于阈值 / 性能或稳定性问题 / 方向性负面反馈 / 外部环境变化
