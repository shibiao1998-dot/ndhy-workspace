# 发布效果评估与复盘

> 发布是否成功，不是看"有没有出事故"，而是看"产品价值有没有传递到用户手中"。

## 一、发布成功的4层定义

很多团队对"发布成功"的定义是"没出bug"——这远远不够。

| 层次 | 名称 | 含义 | 判断标准 |
|------|------|------|---------|
| **第一层** | 技术成功 | 系统稳定、没有严重故障 | 性能指标正常，可用率达标 — **这只是及格线** |
| **第二层** | 到达成功 | 目标用户知道了这次发布 | 通知到达率、更新率达标 |
| **第三层** | 采纳成功 | 用户开始使用新功能/新产品 | 核心功能使用率达标，用户能独立操作 |
| **第四层** | 价值成功 | 新功能真的帮到了用户 | 效率提升、满意度提升、业务指标改善 |

> **目标：每次发布至少评估到第三层，重要发布必须评估到第四层。**

---

## 二、发布效果评估指标体系

### 技术指标

| 指标 | 说明 | 参考目标 |
|------|------|---------|
| 发布过程耗时 | 从开始到完成 | 不超过预定时间的120% |
| 回退次数 | 0次最好 | 0 |
| 发布后24h bug数量 | 按等级分类 | P0=0, P1≤2, P2≤5 |
| 系统可用率 | 系统正常运行比例 | ≥ 99.9% |
| 性能指标变化 | 核心接口响应时间 | 不超过发布前的120% |

### 到达指标

| 指标 | 说明 | 参考目标 |
|------|------|---------|
| 通知触达率 | 多少用户收到了发布通知 | ≥ 90% |
| 应用更新率 | 多少用户更新到新版本 | 1周内 ≥ 70% |
| 培训参与率 | 多少用户参加了培训 | ≥ 80% |

### 采纳指标

| 指标 | 说明 | 参考目标 |
|------|------|---------|
| 新功能首次使用率 | 多少人"试了一下" | ≥ 40% |
| 新功能持续使用率 | 多少人"一直在用" | ≥ 25% |
| 核心操作完成率 | 用户用得顺不顺 | ≥ 90% |
| 客服工单量 | 使用过程中的问题量 | 不超过预期的150% |

### 价值指标

| 指标 | 说明 | 参考目标 |
|------|------|---------|
| 用户满意度变化 | 发布前 vs 发布后 | 正向变化 |
| 效率提升数据 | 完成同样任务的时间变化 | 可量化的提升 |
| 留存率变化 | 发布后用户留存 | 不低于发布前 |
| 业务目标达成情况 | 和发布前设定的目标对比 | 达成或超出 |

## 三、复盘机制

每次重要发布都必须做复盘——不是"走过场"，而是"真的改进"。

### 复盘时间

| 发布类型 | 复盘时间 |
|----------|---------|
| 大版本发布 | 发布后2周进行 |
| 功能发布 | 发布后1周进行 |
| 紧急发布/事故 | 问题解决后48小时内进行 |

### 复盘参与者

- 产品、技术、测试、运营、客服——所有参与发布的角色
- 管理层列席听（了解问题即可，不要主导复盘）

### 复盘6模块框架

#### 模块一：发布概况
- 发布了什么？什么时间发布的？
- 预定计划 vs 实际执行的差异
- 是否有回退？是否有紧急修复？

#### 模块二：做得好的（继续保持）
- 哪些环节执行得好？
- 哪些提前准备发挥了作用？
- 有没有值得推广的好做法？

#### 模块三：做得不好的（需要改进）
- 哪些环节出了问题？
- 问题的根本原因是什么？（不要停留在表面）
- 需要改进的具体行动是什么？
- **每个改进项必须有责任人和完成时间**

#### 模块四：数据回顾
- 发布效果的各项指标数据
- 和预设目标的对比
- 数据揭示的趋势和洞察

#### 模块五：用户声音
- 收到了哪些用户反馈？
- 正面反馈和负面反馈分别是什么？
- 有没有"出乎意料"的反馈？

#### 模块六：行动项
- 列出所有需要跟进的事项
- 每项有明确的负责人、完成时间、验收标准
- 下次复盘时回顾这些行动项的完成情况

### 复盘铁律

1. **对事不对人** — 不是追责会，是改进会
2. **问五个为什么** — 不要停在表面原因
   ```
   "为什么上线后出了bug？"→"因为测试没覆盖到"
   →"为什么没覆盖到？"→"因为测试用例没有这个场景"
   →"为什么没有？"→"因为需求文档里没写清楚边界条件"
   →"为什么没写清楚？"→"因为评审时没有人提这个问题"
   → 根因：需求评审的检查清单需要完善
   ```
3. **改进项必须落地** — 复盘不输出行动项 = 白开

## 四、经验沉淀

复盘的成果要变成"组织记忆"，不能每次都从零开始。

### 需要沉淀的4类内容

#### 1. 发布检查清单的迭代

```
每次复盘后如果发现检查清单漏了什么，就加上。
检查清单是一个"活文档"，越来越完善。
```

#### 2. 常见问题库

```
"这个问题我们遇到过，当时是这么解决的"
新人来了能快速学习历史经验。

格式建议：
  问题描述 → 根因分析 → 解决方案 → 预防措施
```

#### 3. 发布模板的优化

```
通知模板、培训模板、FAQ模板
每次发布不用从头写，在模板上改就行。
```

#### 4. 指标基线的建立

```
"我们的功能发布，平均首次使用率是40%，好的能到60%"
有了基线，才能判断每次发布的效果是好是坏。

建议建立的基线：
  → 功能发布：首次使用率、持续使用率
  → 培训：参与率、完成率、转化率
  → 技术：发布耗时、bug数量、回退次数
  → 满意度：发布后NPS变化
```
