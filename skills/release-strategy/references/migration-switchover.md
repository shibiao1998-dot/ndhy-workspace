# 数据迁移与系统切换

> 用户从旧系统切到新系统，是发布过程中风险最高的环节。如果处理不好，一堆"历史数据找不到了"的投诉会淹没你。

## 一、3种切换策略

### 策略一：并行运行（最安全，成本最高）

```
方式：新旧系统同时运行一段时间，用户可自由选择
适用：旧系统有大量存量用户且数据重要的场景
建议并行期：4-8周

优点：
  → 用户可以按自己的节奏迁移
  → 出了问题随时回到旧系统
  → 给用户适应新系统的缓冲时间

风险：
  → 用户不愿意迁移（"旧的能用为什么要换"）
  → 维护两套系统成本高
  → 数据在两个系统间可能不同步

应对措施：
  → 在旧系统中温和引导"试试新版本？"
  → 设定旧系统关闭的明确日期并反复提醒
  → 当90%+活跃用户已迁移后关闭旧系统
```

### 策略二：分批切换（平衡方案）

```
方式：按学校/区域分批从旧系统切到新系统
适用：大多数场景
建议间隔：每批之间间隔1-2周

优点：
  → 每批切换用户量有限，风险可控
  → 前一批的经验可以优化后续批次
  → 切换过程有序可控

风险：
  → 需要维护切换状态管理
  → 不同批次用户可能体验不一致

应对措施：
  → 第一批选最容易的（用户量少、配合度高）
  → 每批切换后收集反馈、优化方案
  → 对外说"分阶段升级"而非"还没轮到你"
```

### 策略三：一刀切（风险最高，速度最快）

```
方式：某个时间点统一从旧系统切到新系统
适用：旧系统用户量很少，或新旧系统差异不大
必要条件：一定要选在假期 + 一定要有回退方案

优点：
  → 所有用户同时使用新系统，无版本割裂
  → 不需要维护两套系统
  → 切换完就完了

风险：
  → 出了问题影响所有人
  → 回退难度大

应对措施：
  → 选在假期凌晨执行
  → 回退方案必须在30分钟内可执行
  → 切换后第一天全员高度警戒
```

### 切换策略选择决策树

```
旧系统有大量活跃用户？
  ├─ 是 → 数据迁移复杂度高？
  │        ├─ 是 → 并行运行
  │        └─ 否 → 分批切换
  └─ 否 → 一刀切（假期执行 + 回退方案）
```

## 二、数据迁移7步流程

### Step 1：数据盘点

```
目标：搞清楚旧系统到底有多少数据、什么类型

盘点内容：
  → 数据总量（记录数、存储大小）
  → 数据类型分类（用户数据、业务数据、配置数据、历史数据）
  → 数据质量状况（有多少脏数据、缺失数据）
  → 数据归属（哪些数据属于用户、哪些属于系统）

产出：数据盘点报告
```

### Step 2：映射规则

```
目标：定义旧数据的每个字段对应新系统的哪个字段

关键问题：
  → 哪些数据必须迁移？（通常不是全部）
  → 字段名称/格式变化的对应关系是什么？
  → 有些数据在新系统里没有对应的地方怎么办？
  → 数据类型变化（如旧系统是文本，新系统是枚举）怎么转换？

产出：字段映射表
```

### Step 3：试迁移

```
目标：先用一小部分数据做迁移测试

执行方式：
  → 选取有代表性的数据子集（不同类型、不同量级）
  → 在测试环境执行迁移
  → 记录迁移过程中的所有问题

关注点：
  → 迁移速度（全量迁移需要多长时间）
  → 数据转换是否正确
  → 边界情况处理（空值、超长文本、特殊字符）
```

### Step 4：验证

```
目标：迁移后逐项对比，确认数据没有丢、没有错

验证方法：
  → 数量校验：迁移前后记录数是否一致
  → 抽样校验：随机抽取N条记录逐字段对比
  → 完整性校验：关键业务数据是否完整
  → 关联性校验：数据之间的关联关系是否保持

验证通过标准：
  → 数量一致率 100%
  → 抽样准确率 > 99.9%
  → 无关键数据丢失
```

### Step 5：正式迁移

```
目标：确认无误后全量迁移

执行要点：
  → 选在系统使用低谷期执行（凌晨或假期）
  → 迁移过程中暂停旧系统的数据写入
  → 全程记录日志
  → 预留足够时间（实际耗时通常是测试的1.5-2倍）
```

### Step 6：二次验证

```
目标：全量迁移后再次核对

验证内容：
  → 重复 Step 4 的所有验证
  → 增加：全量数据的统计对比
  → 增加：核心业务流程的端到端验证
  → 增加：性能验证（大数据量下查询是否正常）
```

### Step 7：用户确认

```
目标：让用户检查自己的数据是否完整

执行方式：
  → 给用户提供"数据核对"入口
  → 重点用户（数据量大的）安排专人协助核对
  → 设置反馈渠道——"数据有问题请联系我们"
  → 对于发现的问题，24小时内响应修复
```

### 数据迁移铁律

- **旧系统数据在确认迁移成功之前不能删除**
- **迁移过程要有完整日志**，能追溯每一条数据的去向
- **给用户提供"数据有问题"的反馈渠道** — 迁移不可能100%完美
- **准备好人工修复方案** — 总会有一些边角数据需要手工处理

## 三、系统切换时的用户体验管理

### 切换前

| 时间节点 | 动作 |
|----------|------|
| 提前2周 | 告知用户切换计划和时间 |
| 提前1周 | 说清楚影响——切换期间哪些功能不能用、什么时候恢复 |
| 提前3天 | 提供操作指南——用户需要做什么（下载新APP、更新浏览器等） |
| 提前1天 | 建议用户备份——给用户"导出历史数据"的能力 |

### 切换日

- 如需停服，选在影响最小的时段（凌晨或假期）
- 展示友好的维护页面——"正在升级，预计XX时间恢复"（不要显示错误信息）
- 实时更新进度——通过公告或社群告知"当前进展"

### 切换后

| 时间节点 | 动作 |
|----------|------|
| 完成即刻 | 通知"已完成"——告诉用户可以使用了 |
| 第1天 | 提供"新旧对照指南"——旧系统的XX功能现在在新系统的哪里 |
| 第1周 | 强化客服支持——加倍客服人手 |
| 持续 | 留好反馈渠道——"如果发现数据有问题，请通过XX联系我们" |
