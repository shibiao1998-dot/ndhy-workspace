# 试点与灰度策略

## 一、为什么必须做试点

- 办公室里想象的场景和真实场景总有差距
- 测试环境跑得好的功能，到真实环境可能出问题
- 10个用户没问题，1000个一起用可能就有问题
- **试点 = 花小代价发现大问题**

### 不做试点的后果

- 一次性推给5000所学校 → 发现bug → 5000所学校同时炸锅 → 客服瘫痪
- 新功能推给所有教师 → 教师不会用 → 大量投诉 → 被校长/教育局投诉
- **教育行业容错空间特别小——"影响了教学"这件事，任何理由都不好使**

## 二、试点学校的选择方法

选试点学校不是随便选的，也不是只选"关系好的学校"。

### 选择维度

| 维度 | 要考虑的点 |
|------|-----------|
| **代表性** | 学���类型要有差异——城市/农村、重点/普通、大校/小校，不能全是最好的学校 |
| **信息化基础** | 至少选一些信息化基础弱的学校——它们的反馈才能暴露真正的兼容性问题 |
| **配合意愿** | 试点学校要愿意配合——能及时反馈问题、愿意填写反馈表、允许观察 |
| **影响可控** | 万一出问题影响范围��控——不要拿"教育局重点关注的标杆校"做第一批试点 |
| **反馈能力** | 学校里要有"关键联系人"能说清楚问题，而不是只会说"不好用" |

### 试点学校数量建议

| 场景 | 建议数量 | 说明 |
|------|---------|------|
| 全新产品首次试点 | 3-5所 | 太多管不过来 |
| 重要新功能试点 | 5-10所 | 覆盖不同类型 |
| 小功能优化 | 不需要专门试点 | 直接灰度发布 |

## 三、灰度发布5阶段详解

### 第一阶段：内部灰度

```
范围：公司内部团队使用
目的：发现最明显的bug和体验问题
时长：3-5个工作日

通过标准：
  ✅ 无P0/P1级bug
  ✅ 核心流程走通
  ✅ 基本性能正常

关键动作：
  → 让非技术人员（销售、客服、HR等）也参与——他们的视角最接近真实用户
  → 每天收集反馈，当日修复当日验证
  → 记录所有发现的问题，分级处理
```

### 第二阶段：种子用户灰度

```
范围：3-5所试点学校，每校选5-10位愿意尝鲜的教师
目的：验证真实场景下的可用性
时长：1-2周

通过标准：
  ✅ 核心功能使用成功率 > 95%
  ✅ 无严重bug
  ✅ 用户满意度 > 70%

关���动作：
  → 提供专属支持通道（微信群、专线电话）
  → 产品团队跟踪每位种子用户的使用情况
  → 每周收集结构化反馈
  → 关注"用户卡在哪里"——这是优化产品和培训方案的关键输入
```

### 第三阶段：试点校全量

```
范围：试点学校的所有目标用户
目的：验证规模使用的稳定性和培训方案有效性
时长：2-4周

通过标准：
  ✅ 系统稳定
  ✅ 培训方案验证有效
  ✅ 形成可复制的推广方案

关键动作：
  → 执行标准培训方案（而非产品团队手把手教）
  → 测试"可复制性"——不依赖产品团队驻校也能用起来
  → 观察不同类型用户的使用差异
  → 收集培训方案的改进建议
```

### 第四阶段：扩大灰度

```
范围：扩大到10-30所学校，覆盖不同类型
目的：验证方案的普适性
时长：2-4周

通过标准：
  ✅ 方案不需要针对单校做特殊适配
  ✅ 系统在更大并发下仍然稳定
  ✅ 客服工单量在可控范围内

关键动作：
  → 确认培训方案在不同类型学校都有效
  → 关注"边缘场景"——特殊的网络环境、设备、使用习惯
  → 建立"问题→解决方案"的标准流程
  → 培养第一批种子教师
```

### 第五阶段：全量发布

```
范围：所有目标用户
注意：即使全量发布，也建议分批——

推荐节奏：
  第1天：开放 30% 用户
  第2天：开放 50% 用户
  第3天：开放 100% 用户

关键动作：
  → 全量发布不等于"完事了"——进入观察期
  → 密切监控各项指标
  → 客服加强值班
  → 准备好应对可能的问题爆发
```

## 四、开门/关门标准详解

每个灰度阶段就像一扇门——满足条件就"开门"进入下一阶段，出了问题就"关门"退回上一阶段。

### 开门标准（进入下一阶段）

- ✅ 当前阶段的P0/P1级bug全部修复
- ✅ 核心功能的成功率达标
- ✅ 系统性能指标正常
- ✅ 用户反馈中没有方向性问题（不需要推翻重做的问题）
- ✅ 负责人明确签字确认

### 关门标准（暂停或回退）

- ❌ 出现P0级bug且短时间内无法修复
- ❌ 核心功能使用成功率低于阈值
- ❌ 系统出现性能或稳定性问题
- ❌ 收到方向性负面反馈（"这个功能根本不该这么做"）
- ❌ 外部环境变化（如突然进入考试周）

### 关门后的动作

1. 不是"停在那里等"，而是**立刻拉问题清单、排优先级、限时修复**
2. 修复完毕 + 重新验证通过后，才能重新"开门"
3. **如果回退了，要通知已经在用的用户**——不能悄悄回退

## 五、灰度期间数据监控

不能只凭感觉判断"试���效果好不好"，要看数据。

### 必须监控的核心指标

**功能维度**
| 指标 | 说明 | 关注阈值 |
|------|------|---------|
| 核心功能使用率 | 有多少人真的在用？ | 试点阶段 > 60% |
| 核心功能完成率 | 开始用的人，有多少顺利完成？ | > 90% |
| 核心功能出错率 | 使用过程中出了多少错？ | < 5% |

**体验维度**
| 指标 | 说明 | 关注阈值 |
|------|------|---------|
| 用户操作路径 | 是否和设计路径一致？有没有卡住？ | 路径偏离率 < 20% |
| 平均完成时长 | 做一件事花了多长时间？ | 不超过预期的1.5倍 |
| 求助率 | 多少人需要找客服/看帮助？ | < 15% |

**稳定性维度**
| 指标 | 说明 | 关注阈值 |
|------|------|---------|
| 页面加载时间 | 核心页面加载速度 | < 3�� |
| 接口错误率 | API调用失败比例 | < 1% |
| 系统可用率 | 系统正常运行时间比例 | > 99.9% |

**满意度维度**
| 指标 | 说明 | 关注阈值 |
|------|------|---------|
| 用户满意度评分 | 综合满意度 | > 4.0/5.0 |
| 主动反馈内容 | 表扬/吐槽分别统计 | 正面反馈占比 > 60% |
| 继续使用意愿 | 用户是否愿意继续使用 | > 80% |
