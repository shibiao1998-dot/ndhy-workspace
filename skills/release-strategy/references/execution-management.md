# 发布执行管理

> 发布日就是"打仗"的日子。必须有明确的流程、清晰的分工、充足的预案。

## 一、发布检查清单（6大类完整版）

### 1. 产品准备

- [ ] 功能开发全部完成，无遗留项
- [ ] 所有P0/P1级bug已修复
- [ ] 已知的P2/P3级bug已评估影响，确认可以带着上线
- [ ] 产品验收通过（产品经理签字确认）
- [ ] 在预发布环境完成最终验证

### 2. 技术准备

- [ ] 数据库变��已准备并验证
- [ ] 回滚方案已确认可执行
- [ ] 灰度发布的配置已就绪
- [ ] 监控告警已部署
- [ ] 性能压测通过
- [ ] 安全检查通过

### 3. 用户准备

- [ ] 培训内容已准备（视频/文档/指南）
- [ ] 帮助中心已更新
- [ ] FAQ已准备并发给客服团队
- [ ] 用户通知已准备（文案已审核）

### 4. 运营准备

- [ ] 客服团队已培训，了解新功能和常见问题
- [ ] 客服排班已调整（发布后第一周加强）
- [ ] 反馈收集渠道已准备
- [ ] 应急联系人名单已确认

### 5. 沟通准备

- [ ] 内部发布邮件/通知已准备
- [ ] 外部发布公告已准备
- [ ] 更新日志已撰写
- [ ] 重要客户的1对1通知已安排

### 6. 应急准备

- [ ] 回退方案已确认，参与人已知晓
- [ ] 应急响应团队已就位
- [ ] 紧急联系人电话清单已发给所有相关人员
- [ ] 回退的触发条件已明确定义
- [ ] 如需停服，维护页面已准备

## 二、发布日流程（War Room 模式）

大版本发布或重要功能发布，发布当天要进入"作战室"模式——所有关键角色集中，实时协同，直到发布稳定。

### War Room 角色分工

| 角色 | 职责 |
|------|------|
| **发布指挥官**（通常是产品负责人） | 全局决策——发不发、继续还是回退、对外怎么说。所有"做不做"的决定由这个人拍板 |
| **技术发布负责人** | 执行部署操作、监控系统状态，出技术问题第一时间排查 |
| **测试负责人** | 发布后立刻做线上验证（冒烟测试），确认核心流程是否正常 |
| **客服负责人** | 监控用户反馈渠道，出现用户投诉立即上报 |
| **运营负责人** | 执行对外通知、更新公告，监控业务数据指标 |

### 发布日标准时间线（以凌晨发布为例）

```
T-2小时  │ 所有角色就位，最终确认检查清单
T-1小时  │ 技术团队开始部署前准备
T 时刻   │ 开始部署
T+15分钟 │ 部署完成，技术团队确认系统启动正常
T+30分钟 │ 测试团队完成线上冒烟测试，报告结果
T+1小时  │ 如果灰度发布，开放第一批用户（1-5%��
T+2小时  │ 检查第一批用户的使用数据和反馈
T+4小时  │ 无异常则扩大灰度比例
...      │ 逐步扩大直到全量

任何环节出现问题：
  → 立即暂停扩大
  → 评估是否需要回退
  → 发布指挥官做最终决定
```

## 三、发布后观察期

发布完成不等于发布结束。发布后至少要有一个"观察期"。

### 观察期时长

| 发布类型 | 观察期 |
|----------|--------|
| 大版本发布 | 至少2周 |
| 功能发布 | 至少1周 |
| 修复/优化发布 | 至少2-3天 |

### 观察期各阶段动作

**第1天（高度警戒）**
- 每小时查看一次关键指标
- 客服团队加强值班
- 技术团队保持随时响应状态
- 任何异常立即上报

**第2-3天（持续关注）**
- 每天查看一次关键指标
- 收集并分类用户反馈
- 评估是否需要发布补丁

**第4-7天（常规监控）**
- 关注使用数据的趋势变化
- 整理用户反馈，区分"bug"和"需求"
- 评估培训方案是否需要调整

**第2周及以后（复盘阶段）**
- 汇总发布数据
- 输出发布复盘报告
- 沉淀经验教训

## 四、回退（回滚）决策

回退是发布的"安全绳"——什么时候拉、怎么拉、谁来拉，必须提前定义清楚。

### 必须回退的情况（无需讨论，直接回退）

| 情况 | 说明 |
|------|------|
| 核心功能完全不可用 | 登录失败、数据丢失等 |
| 系统性能严���下降 | 影响大面积用户 |
| 数据安全问题 | 数据泄露或篡改风险 |
| 错误率突然飙升 | 超过预设阈值 |

### 需要评估的情况

| 情况 | 评估要点 |
|------|---------|
| 非核心功能出现bug | 评估影响范围和修复时间 |
| 性能有所下降但可接受 | 评估是否会恶化 |
| 部分用户反馈问题但不普遍 | 评估影响面 |

### 回退决策原则

1. **修复时间预估 > 2小时且影响面 > 10%用户** → 优先回退
2. **不确定问题会不会扩大** → 优先回退
3. **回退不丢人，带病运行才丢人**
4. **宁可多回退一次，也不要硬撑导致更大事故**

### 回退后的动作

1. 通知用户：不推卸、不含糊 — "为了确保您的使用体验，我们决定先回到稳定版本"
2. 给出预计重新发布的时间 — 哪怕只是"预计本周内"
3. 彻底排查问题、修复
4. 重新走发布流程

## 五、紧急发布（Hotfix）流程

当生产环境出现严重问题，需要紧急修复并发布。

### 触发条件

| 类型 | 说明 |
|------|------|
| P0级bug | 影响核心功能、影响大量用户 |
| 安全漏洞 | 被发现或被利用的安全问题 |
| 数据问题 | 数据丢失或错乱的风险 |

### 紧急发布特殊规则

**可以跳过的环节**（部分常规流程）
**不能跳过的环节**：
- ✅ 代码审查（至少一人审查）
- ✅ 基本测试（核心流程跑通）
- ✅ 回退方案（必须有）
- ✅ 有人"盯着"（不能发完就走）

### 紧急发布审批

```
不需要走常规审批链，但至少需要：
  → 技术负责人确认修复方案
  → 产品负责人确认影响范围
  → 值班负责人批准发布

审批方式：
  → 可以口头进行（电话/即时通讯）
  → 事后必须补文字记录

紧急发布后，下一个工作日必须补齐所有跳过的流程
```
