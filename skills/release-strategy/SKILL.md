# release-strategy -- 发布策略执行工具箱

设计产品从"开发完成"到"用户成功用起来"的全过程策略——确保产品发得出去、用得起来、稳得下来。

> 代码上了服务器不等于发布成功，用户真正用起来了才算成功。
> 发布 = 上线 + 触达 + 教会 + 驱动使用。

## 9 步标准流程

```
Step 1: 理解发布背景
  |  → 发布内容：全新产品/新功能/优化/修复？
  |  → 发布目标：用户增长/体验提升/客户交付？
  |  → 上游文档：产品定义/体验设计/技术架构/项目计划是否齐全？
  |  → 约束条件：时间窗口/预算/团队资源？
  |  → 已知风险有哪些？
  |  → 信息不足 → 要求上游补充（不猜测、不假设）
  |
Step 2: 制定发布策略
  |  → 确定发布类型和规模 → references/version-planning.md
  |  → 选择发布窗口（结合教育行业日历，见下方速查）
  |  → 制定灰度/试点方案 → references/pilot-grayscale.md
  |  → 设定各阶段"开门/关门"标准
  |  → 确认：这个节奏，团队和用户都跟得上吗？
  |
Step 3: 设计用户准备方案
  |  → 培训内容和形式设计（三层蛋糕 + 6种形式）
  |  → 培训对象分层（管理员/领导/教师/学生）
  |  → 种子教师策略
  |  → 帮助文档和FAQ准备
  |  → 详细方法 → references/user-training.md
  |
Step 4: 制定沟通计划
  |  → 内部沟通计划（各团队对齐时间表）
  |  → 外部沟通计划（用户通知/更新日志/宣传配合）
  |  → 变更管理策略（处理用户抵触情绪）
  |  → 详细方法 → references/communication-change.md
  |
Step 5: 准备数据迁移方案（如需要）
  |  → 数据盘点和映射规则
  |  → 选择切换策略（并行/分批/一刀切）
  |  → 迁移测试和验证方案
  |  → 用户告知和确认方案
  |  → 详细方法 → references/migration-switchover.md
  |
Step 6: 制定发布执行计划
  |  → 发布检查清单（6大类，见下方速查）
  |  → 发布日流程和角色分工（War Room 模式）
  |  → 回退方案和触发条件
  |  → 应急预案 + 紧急发布流程
  |  → 详细方法 → references/execution-management.md
  |
Step 7: 设定效果评估方案
  |  → 定义"发布成功"的4层标准（技术/到达/采纳/价值）
  |  → 设定各阶段监控指标
  |  → 规划复盘时间和形式
  |  → 详细方法 → references/evaluation-retrospective.md
  |
Step 8: 执行与监控
  |  → 按计划执行发布
  |  → 实时监控各项指标
  |  → 处理发布过程中的问题
  |  → 根据数据决定"继续推进"还是"暂停/回退"
  |
Step 9: 复盘与沉淀
     → 执行复盘会议（6模块框架）
     → 输出复盘报告
     → 更新检查清单、模板、知识库
     → 跟踪改进项落地
     → 详细方法 → references/evaluation-retrospective.md
```

## 发布深度速查

| 深度 | 适用场景 | 覆盖步骤 | 输出物 |
|------|----------|----------|--------|
| **轻量发布** | bug修复、小优化 | Step 1 + 6（简化清单）+ 发布通知 | 简化检查清单 + 更新日志 |
| **标准发布** | 新功能上线 | 完整 9 步 | 灰度方案 + 培训方案 + 沟通计划 + 检查清单 + 复盘 |
| **重大发布** | 全新产品/大版本/政府交付 | 完整 9 步 + 试点 + 迁移 + 应急预案 | 完整交付包（7文档）+ 深度复盘 |

判断不清时向上确认。

## 铁律速查

| # | 铁律 | 核心要求 |
|---|------|---------|
| 1 | **没有回退方案的发布 = 赌博** | 任何发布必须有经过验证的回退方案，回退决策权和触发条件提前明确 |
| 2 | **用户不会自己发现新功能** | 发布 = 上线 + 触达 + 教会 + 驱动使用，缺一不可 |
| 3 | **灰度是保护伞不是麻烦事** | 跳过灰度 = 把所有用户当小白鼠，"时间紧"不是跳过灰度的理由 |
| 4 | **发布日就是作战日** | 关键角色集中在线、实时响应，没有指挥官的发布 = 混乱的发布 |
| 5 | **用户的抵触是正常的** | 给"为什么"、给过渡期、给同伴示范、给支持、给认可 |
| 6 | **复盘不做等于白做** | 不复盘 = 同样的坑继续踩，改进项必须有责任人和完成时间 |

## 教育发布日历速查

| 时段 | 策略 | 允许的发布类型 |
|------|------|---------------|
| **寒暑假** | 大版本发布的黄金窗口 — 用户少、有时间培训、开学直接用新版 | 所有类型 |
| **开学前2周** | 冻结期 — 系统必须绝对稳定 | 仅P0级bug修复 |
| **开学后1个月** | 高度谨慎期 — 密切监控系统指标 | 仅修复和小优化 |
| **学期中段** | 常规发布窗口 — 功能迭代主要时段 | 按正常节奏灰度发布 |
| **考试周前2周** | 冻结期 — 出问题无法交代 | 禁止任何发布 |
| **期末/年末** | 谨慎期 — 收尾性发布，为假期大版本做准备 | 技术优化、收尾发布 |

> 教育行业一个学期约18-20周教学时间，能做发布的窗口可能只有8-10周。每次发布窗口都要珍惜。

## 灰度阶段速查

| 阶段 | 范围 | 目的 | 时长 | 通过标准 |
|------|------|------|------|---------|
| **1. 内部灰度** | 公司内部团队 | 发现最明显的bug和体验问题 | 3-5个工作日 | 无P0/P1级bug，核心流程走通 |
| **2. 种子用户** | 3-5所试点校，每校5-10位教师 | 验证真实场景可用性 | 1-2周 | 使用成功率>95%，满意度>70% |
| **3. 试点校全量** | 试点学校所有目标用户 | 验证规模稳定性和培训方案 | 2-4周 | 系统稳定，培训方案验证有效 |
| **4. 扩大灰度** | 10-30所学校，覆盖不同类型 | 验证方案普适性 | 2-4周 | 方案无需单校特殊适配 |
| **5. 全量发布** | 所有目标用户 | 全面铺开 | 分批开放 | 第1天30%→第2天50%→第3天100% |

**开门标准**：P0/P1全修复 + 成功率达标 + 性能正常 + 无方向性问题 + 负责人确认
**关门标准**：P0级bug无法短期修复 / 成功率低于阈值 / 性能或稳定性问题 / 方向性负面反馈 / 外部环境变化

## 反模式速查

| # | 反模式 | 错误做法 | 正确做法 |
|---|--------|---------|---------|
| 1 | **发布无节奏** | 想发就发，用户不知道什么时候变化 | 建立固定发布节奏，让用户能预期 |
| 2 | **灰度走过场** | 灰度从1%直接跳100%，中间不观察 | 每阶段至少观察足够时间，看数据再决定 |
| 3 | **培训缺失** | 功能发了但没人会用，使用率低 | 培训和功能一样重要，不能砍 |
| 4 | **沟通脱节** | 客服不知道发了什么，用户问了答不上来 | 内部沟通前置，发布前至少1周对齐客服 |
| 5 | **回退混乱** | 出了问题不知道该不该回退、谁来决定 | 回退标准和决策人在发布前就确定 |
| 6 | **数据迁移出错** | 用户历史数据丢了或乱了 | 迁移前充分测试，迁移后让用户确认 |
| 7 | **复盘不做** | 同样的问题反复出现 | 复盘是强制环节，不是"有时间就做" |
| 8 | **期望管理失败** | 用户期望太高，实际体验落差大 | 发布宣传实事求是，先管好期望再超越期望 |

## 发布检查清单速查

### 产品准备
- [ ] 功能开发全部完成，无遗留项
- [ ] 所有P0/P1级bug已修复
- [ ] 已知P2/P3级bug已评估影响
- [ ] 产品验收通过
- [ ] 预发布环境完成最终验证

### 技术准备
- [ ] 数据库变更已准备并验证
- [ ] 回滚方案已确认可执行
- [ ] 灰度配置已就绪
- [ ] 监控告警已部署
- [ ] 性能压测通过
- [ ] 安全检查通过

### 用户准备
- [ ] 培训内容已准备（视频/文档/指南）
- [ ] 帮助中心已更新
- [ ] FAQ已准备并发给客服团队
- [ ] 用户通知已准备（文案已审核）

### 运营准备
- [ ] 客服团队已培训
- [ ] 客服排班已调整（发布后首周加强）
- [ ] 反馈收集渠道已准备
- [ ] 应急联系人名单已确认

### 沟通准备
- [ ] 内部发布通知已准备
- [ ] 外部发布公告已准备
- [ ] 更新日志已撰写
- [ ] 重要客户1对1通知已安排

### 应急准备
- [ ] 回退方案已确认，参与人已知晓
- [ ] 应急响应团队已就位
- [ ] 紧急联系人电话清单已发放
- [ ] 回退触发条件已明确定义
- [ ] 如需停服，维护页面已准备

> 完整检查清单详见 [references/templates.md](references/templates.md)

### 上游输入

| 来源 | 接收内容 | 重点关注 |
|------|---------|---------|
| 产品定义文档 | 产品定位、功能范围、目标用户 | 这次发布的核心价值主张 |

## 工作模式决策树

```
收到发布策略任务
  |- 轻量任务（更新日志/单次沟通/简单通知）
  |   → 单独执行
  |
  |- 明确边界的专项任务（试点方案/培训方案/迁移方案）
  |   → 作为 subagent 被调用（默认模式）
  |
  +- 大发布 / 全新产品 / 多团队协同
      |- 涉及技术部署/数据迁移/多端同步？
      |   → Agent Team（拉技术/运维/测试专家协作）
      +- 否 → subagent 模式 + 输出后交相关专家执行
```

## 触发时首要动作

无论如何触发，第一步永远是：

> 1. 确认"发的是什么" — 功能范围是否清晰，核心功能是否通过测试
> 2. 确认"发给谁" — 目标用户群体，是否有需要特别关注的客户
> 3. 确认"什么时候能发" — 技术Ready时间，是否在合适的发布窗口
> 4. 确认"风险在哪里" — 最可能出什么问题，回退方案是否就绪
> 5. 确认"怎么判断成功" — 量化标准，监控指标，埋点是否就绪
> 6. 判断发布深度 → 按9步流程执行

**教育行业项目**：额外参考教育发布日历速查 + [references/special-scenarios.md](references/special-scenarios.md) 了解行业特殊约束。
