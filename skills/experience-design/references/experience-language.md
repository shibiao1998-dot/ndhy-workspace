## 语言风格定义方法

### 产品人设定义

产品和用户"说话"的方式，决定了用户对产品的感知。先定义产品人设，再推导具体文案规范。

#### 步骤 1：选择人设类型

| 人设类型 | 特征 | 适用场景 | 典型产品 |
|---------|------|---------|---------|
| **专业严谨型** | 精确、规范、正式 | 金融、医疗、政务 | 银行 App、医疗系统 |
| **亲和温暖型** | 温柔、鼓励、有温度 | 教育、健康、社区 | 学习 App、健康管理 |
| **简洁高效型** | 简短、直接、无废话 | 工具、效率、B2B | 项目管理、代码工具 |
| **活泼趣味型** | 轻松、幽默、有个性 | 社交、娱乐、年轻用户 | 社交 App、游戏 |

#### 步骤 2：定义语言维度

| 维度 | 定义范围 | 示例 |
|------|---------|------|
| **称呼方式** | "你" / "您" / 不使用人称 | 教育产品常用"你"（亲和） |
| **句式风格** | 短句/口语化/书面正式 | 工具类产品：短句为主，不超过 20 字 |
| **情感温度** | 1（冷静克制）— 5（热情友好） | 教育产品：3-4 分 |
| **禁用词** | 不允许出现的表达 | "报错"→"出了点问题"，"非法"→"不符合要求" |
| **语气词** | 是否使用语气词 | "哦"、"呢"、"吧"——亲和型可用，严谨型避免 |

#### 步骤 3：输出语言风格指南

使用 `references/templates.md` 中的**体验语言规范模板**输出。

### 语言一致性检查

定义好风格后，用以下清单检查全产品一致性：

- [ ] 所有页面称呼方式统一
- [ ] 按钮文案风格一致（都用动词 or 都用名词）
- [ ] 错误提示语气统一
- [ ] 不同开发者写的文案风格无明显差异
- [ ] 禁用词清单在全产品范围内执行

## 文案设计规范

### 铁律：文案是设计的一部分

> 按钮上写什么和放在哪里一样重要。所有文案在设计阶段确定，不留给开发。

### 五类关键文案

#### 1. 按钮文案

**原则：** 动词优先，说明点击后会发生什么。

| ❌ 不要 | ✅ 应该 | 原因 |
|--------|--------|------|
| "确定" | "保存教案" | 用户知道点了会怎样 |
| "提交" | "发布课程" | 具体动作 + 对象 |
| "下一步" | "下一步：填写教学目标" | 预告下一步内容 |
| "是" / "否" | "删除" / "保留" | 明确选择结果 |
| "OK" | "知道了" / "开始使用" | 使用用户语言 |

**按钮文案规则：**
- 主操作：动词 + 对象（"保存教案"、"创建课程"）
- 取消类：统一用"取消"
- 确认类：用具体动词（"删除"而非"确定"）
- 长度：2-6 个字

#### 2. 提示文案

**原则：** 说人话，给例子。

| 场景 | ❌ 不要 | ✅ 应该 |
|------|--------|--------|
| 输入提示（placeholder） | "请输入" | "请输入课程名称，如：七年级英语第一单元" |
| 帮助提示 | "格式要求：XXX" | "支持 jpg、png 图片，单张不超过 5MB" |
| 操作说明 | "请按照规范操作" | "拖动卡片可以调整顺序" |

**提示文案规则：**
- 输入框 placeholder：说明要填什么 + 给一个例子
- 字段下方帮助文字：说明格式要求或限制
- 操作引导：说明怎么做，而非告知规则

#### 3. 错误文案

**原则：** 问题 + 解法，不甩锅给用户。

| 场景 | ❌ 不要 | ✅ 应该 |
|------|--------|--------|
| 格式错误 | "请输入有效格式" | "请输入手机号，例如：138xxxx0000" |
| 网络错误 | "操作失败（500）" | "保存失败，可能网络不好。内容已自动保存为草稿" |
| 权限错误 | "无权限" | "你还没有编辑权限，可以联系管理员开通" |
| 系统错误 | "系统异常" | "出了点问题，请稍后重试。如持续出现请联系客服" |

**错误文案公式：**
```
[发生了什么，用大白话] + [你可以这样解决 / 我们已经这样处理]
```

**错误文案规则：**
- 不用技术术语（"500"、"timeout"、"参数非法"）
- 不甩锅（"你的操作有误" → "信息填写不完整"）
- 说明系统已做了什么保护（"内容已自动保存"）
- 给出下一步行动建议

#### 4. 空状态文案

**原则：** 该有什么 + 怎么创建。

| 场景 | ❌ 不要 | ✅ 应该 |
|------|--------|--------|
| 列表为空 | "暂无数据" | "还没有创建教案，点击「新建教案」开始备课" |
| 搜索无结果 | "未找到结果" | "没有找到「XXX」，试试其他关键词？" |
| 收藏为空 | "暂无收藏" | "还没有收藏内容，浏览课程时点 ⭐ 即可收藏" |

**空状态文案公式：**
```
[这里应该有什么（现在没有）] + [怎么创建/添加第一条]
```

#### 5. 加载/过程文案

| 场景 | ❌ 不要 | ✅ 应该 |
|------|--------|--------|
| 页面加载 | "加载中..." | "正在获取课程数据..." |
| AI 生成 | "处理中..." | "AI 正在生成教案建议，大约需要 10 秒..." |
| 文件上传 | "上传中" | "正在上传「教案.docx」（3/5MB）..." |

## AI 对话语言设计

> 适用于产品中包含 AI 对话/AI 助手功能的场景。

### AI 角色定位

| 定位 | 特征 | 适用场景 | 语言风格 |
|------|------|---------|---------|
| **助手** | 被动响应，执行指令 | 工具型 AI | "好的，已为你生成..." |
| **伙伴** | 主动建议，协作共创 | 创作型 AI | "我有个建议，你觉得..." |
| **工具** | 无人格，纯功能 | 嵌入式 AI | 无对话，直接输出结果 |

### AI 语言设计原则

#### 原则 1：不能比核心用户更权威

```
❌ "这节课应该用任务型教学法"
✅ "根据课程内容，任务型教学法可能比较适合，仅供参考"
```

#### 原则 2：表达不确定性

```
❌ "这份教案的教学目标是..."
✅ "以下是基于课程标准的教学目标建议，请老师根据班级实际情况调整："
```

#### 原则 3：引导而非替代

```
❌ 直接生成完整教案，无需用户参与
✅ 生成框架 → 用户确认/修改 → 细化内容 → 用户最终定稿
```

#### 原则 4：适度的人格化

```
❌ "亲~我帮你看看哦~"（过度亲昵）
❌ "查询结果如下：（序号123）"（完全机器感）
✅ "我整理了以下内容，你看看是否符合需求："（适度友好）
```

### AI 对话文案模板

| 场景 | 模板 |
|------|------|
| AI 生成开始 | "正在为你[做什么]，大约需要 [X] 秒..." |
| AI 生成完成 | "已为你生成[什么]，你可以直接编辑或重新生成" |
| AI 建议 | "基于[依据]，建议[什么]。这只是参考，请根据实际情况调整" |
| AI 不确定 | "关于这个问题，我的理解是[什么]，但建议你[怎么验证]" |
| AI 出错 | "抱歉，这次没能生成满意的结果。你可以[试试什么]，或重新描述需求" |
| AI 引导输入 | "想要更好的结果，可以补充[什么信息]" |

### AI 交互边界

| 边界 | 做法 |
|------|------|
| AI 不能做什么 | 明确告知（"目前不支持直接修改已发布的课程"） |
| AI 可能出错 | 提前说明（"AI 生成内容仅供参考，请核实后使用"） |
| 用户可以干预 | 随时可编辑、重新生成、忽略建议 |
| 数据使用说明 | "你的课程内容仅用于本次生成，不会被用于其他用途" |
