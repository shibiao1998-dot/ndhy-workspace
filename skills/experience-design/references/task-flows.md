## 任务流设计方法

### 什么是任务流（Task Flow）

任务流是用户完成单一任务的**最优路径**——最少步骤、无分支、假设一切正常。

### 设计步骤

1. **明确任务**：用户要完成什么？（从产品定义的场景清单获取）
2. **确定起点**：用户从哪里开始？（入口在哪）
3. **列出步骤**：完成任务最少需要几步？每步做什么？
4. **标注反馈**：每步系统返回什么反馈？
5. **确定终点**：怎么算完成？用户看到什么？

### 任务流模板

```
任务：[任务名称]
触发：[什么情况下用户会做这个任务]
入口：[从哪个页面/按钮开始]

[入口] → [步骤1：做什么] → [步骤2：做什么] → [步骤3：做什么] → [完成：看到什么]
            反馈：___          反馈：___          反馈：___         反馈：___
```

### 步骤精简检查

对每一步问：
- **能省吗？** 这步不做，任务能完成吗？→ 能省就省
- **能合并吗？** 和前/后步骤能合成一步吗？→ 能合就合
- **能自动化吗？** 系统��自动完成吗？→ 能自动就自动
- **能延后吗？** 必须现在做还是以后也行？→ 能延后就延后（渐进式）

## 用户流设计方法

### 什么是用户流（User Flow）

用户流是在任务流基础上加入**所有分支可能**——用户选A还是选B？中途放弃怎么办？犯错了怎么纠正？

### 设计步骤

1. **以任务流为基础**：先画好最优路径
2. **加入决策分支**：用户可能的选择点
3. **加入异常分支**：出错、中断、放弃、超时
4. **加入恢复路径**：每个异常分支如何回到主路径
5. **加入替代路径**：达成同一目标的其他方式

### 用���流模板

```
[入口]
  │
  ▼
[步骤1] ──── 异常A → [处理方案] → 回到步骤1
  │
  ▼
[决策点] ── 选择A → [步骤2A] → [步骤3]
  │                               │
  └──── 选择B → [步骤2B] ─────────┘
                                   │
                                   ▼
                              [步骤3]
                                   │
                                   ▼ 
                        [完成] / [中途放弃 → 保存草稿]
```

### 分支类型清单

| 分支类型 | 说明 | 必须设计 |
|---------|------|---------|
| **决策分支** | 用户主动选择不同路径 | ✅ 是 |
| **条件分支** | 系统根据状态自动走不同路径 | ✅ 是 |
| **异常分支** | 操作失败、数据错误、网络中断 | ✅ 是 |
| **中断分支** | 用户中途离开/被打断 | ✅ 是 |
| **放弃分支** | 用户主动取消/退出 | ✅ 是 |
| **纠错分支** | 用户发现之前步骤有误想返回修改 | ✅ 是 |

## 关键节点标注指南

### 四类关键节点

在任务流/用户流中，以下节点必须标注：

#### 📌 决策点（Decision Point）

用户在此需要做出选择。

**设计要点：**
- 选项数量 ≤ 3 个（超过提供推荐/默认值）
- 每个选项的后果可预见（用户知道选了会怎样）
- 提供"不确定时的推荐"
- 可以反悔（选了之后能改）

**标注格式：**
```
📌 决策点：[决策描述]
   选项A：___ → 导致 ___
   选项B：___ → 导致 ___
   默认/推荐：___
   可否反悔：是/否
```

#### ⏳ 等待点（Waiting Point）

系统需要处理时间，用户需要等待。

**设计要点：**
- < 1 秒：无需反馈（但需视觉变化确认操作生效）
- 1-5 秒：显示加载指示器
- 5-30 秒：显示进度条 + 说明正在做什么
- \> 30 秒：允许后台处理 + 完成后通知

**标注格式：**
```
⏳ 等待点：[等待原因]
   预计耗时：___
   反馈方式：___
   可否取消：___
```

#### ⚠️ 风险点（Risk Point）

操作不可逆或影响重大。

**设计要点：**
- 不可逆操作必须二次确认
- 确认弹窗说明后果（"删除后无法恢复"）
- 提供"软删除"优先于真删除
- 高风险操作按钮用警示色，远离常规按钮

**标注格式：**
```
⚠️ 风险点：[风险描述]
   后果：___
   保护措施：___（二次确认 / 软删除 / 冷却期）
   恢复方案：___
```

#### 🎯 价值点（Value Point）

用户在此获得价值/达成目标。

**设计要点：**
- 给予明确的正面反馈（成功提示、数据展示）
- 可以是制造峰值时刻的机会
- 引导下一步行动（"教案已保存，是否预览？"）

**标注格式：**
```
🎯 价值点：[用户获得了什么]
   反馈方式：___
   下一步引导：___
```

## 任务流四条铁律详解

### 铁律 1：最短路径

> 核心任务步骤越少越好，每增一步都要问"能不能省/合并/自动化"。

**检查方法：**
- 统计核心任务的步骤数
- 与同类竞品对比：我们的步骤数 ≤ 竞品
- 每步审查：省/合并/自动化/延后

**具体手段：**

| 手段 | 做法 | 示例 |
|------|------|------|
| **省略** | 非必要步骤直接去掉 | 注册后不强制填写个人资料 |
| **合并** | 相关步骤合为一步 | 选择模板 + 填写标题合为一步 |
| **自动化** | 系统自动完成 | 自动检测设备类型，无需用户选择 |
| **延后** | 当前不需要的信息稍后再收集 | 先创建课程，后续再补充详细信息 |
| **智能默认** | 预填合理默认值 | 日期默认今天，年级默认上次选择 |

**量化标准：**
- P0 核心任务：≤ 3 步
- P1 重要任务：≤ 5 步
- P2 次要任务：≤ 7 步

### 铁律 2：渐进式披露

> 先给最关键的，其余需要时再展开。

**核心思想：** 不要一次把所有选项/信息/功能摆在用户面前。按需展示。

**三层披露：**

| 层级 | 展示时机 | 内容 |
|------|---------|------|
| **第一层** | 默认展示 | 完成任务必须的信息和操作 |
| **第二层** | 用户主动展开 | "高级选项"、"更多设置" |
| **第三层** | 特定条件触发 | 帮助文档、专家功能 |

**❌ 错误：** 创建��案页面同时展示：标题、年级、科目、教材版本、课时、目标、重难点、导入活动、练习题、作业、反思…
**✅ 正确：** 第一步只要标题+年级+科目，其余分步完成或"展开更多"

### 铁律 3：可逆原则

> 任何操作可撤销，不可逆操作必须二次确认。

**分级处理：**

| 操作类型 | 可逆性 | 设计策略 |
|---------|--------|---------|
| 输入/编辑 | 完全可逆 | 支持 Ctrl+Z / 撤销按钮 |
| 状态变更 | 可逆 | 提供"撤回"/"恢复"操作 |
| 删除 | 软可逆 | 移入回收站，30 天后彻底删除 |
| 发布/提交 | 部分可逆 | 支持"撤回发布"或"重新编辑" |
| 不可逆操作 | 不可逆 | 二次确认 + 说明后果 + 冷却期 |

**二次确认设计要点：**
- 弹窗标题说明将要发生什么（"确认删除这份教案？"）
- 正文说明后果（"删除后可在回收站找回，30 天后彻底删除"）
- 确认按钮用具体动词（"删除"而非"确定"）
- 危险操作确认按钮用警示色
- 特别危险的操作：要求输入确认文字

### 铁律 4：中断恢复

> 用户中途离开回来能接着干，内容不丢、进度记住。

**设计策略：**

| 场景 | 保存策略 | 恢复策略 |
|------|---------|---------|
| 表单填写中途离开 | 自动保存草稿（每 30 秒或字段失焦时） | 下次打开提示"有未完成的草稿，是否继续？" |
| 多步流程中途退出 | 记录当前步骤和已填数据 | 下次进入跳到上次步骤 |
| 网络中断 | 本地缓存 + 联网后自动同步 | 恢复网络后提示"已自动保存" |
| 浏览器崩溃 | 定时本地备份 | 重新打开后提示恢复 |

**实现要点：**
- 自动保存必须有视觉反馈（"已自动保存"小字提示）
- 恢复选择权给用户："继续上次"或"重新开始"
- 草稿有过期策略：默认保留 30 天

## 流程图格式规范

### 文本流程图格式

在文档中使用 ASCII 格式绘制流程图：

#### 简单线性流程
```
[入口] → [步骤1] → [步骤2] → [步骤3] → [完成]
```

#### 带分支的流程
```
[入口]
  │
  ▼
[步骤1]
  │
  ▼
[决策点？] ── 是 → [步骤2A] ─┐
  │                           │
  └── 否 → [步骤2B] ──────────┘
                              │
                              ▼
                         [步骤3]
                              │
                              ▼
                          [完成]
```

#### 带异常处理的流程
```
[入口] → [步骤1] → [步骤2] → [完成]
              │          │
              ↓          ↓
         [异常A]    [异常B]
         [处理A]    [处理B]
              │          │
              └────┬─────┘
                   ↓
              [回到主路径]
```

### 流程图标注规范

| 标注 | 含义 | 示例 |
|------|------|------|
| `📌` | 决策点 | 📌 选择模板类型 |
| `⏳` | 等待点 | ⏳ AI 生成中（约 5 秒） |
| `⚠️` | 风险点 | ⚠️ 删除教案（不可逆） |
| `🎯` | 价值点 | 🎯 教案创建成功 |
| `→` | 正常流转 | 步骤1 → 步骤2 |
| `↓` | 异常分支 | 步骤1 ↓ 异常处理 |
| `[  ]` | 步骤/状态 | [填写表单] |
| `?` | 判断/条件 | [数据合法?] |

### 流程图质量自检

- [ ] 每个流程有明确的起点和终点
- [ ] 所有决策分支都有标注
- [ ] 异常路径都有处理方案和回到主路径的方式
- [ ] 关键节点（决策/等待/风险/价值）全部标注
- [ ] 步骤描述用动词开头（"填写"、"选择"、"确认"）
- [ ] 步骤数量符合四条铁律的量化标准
