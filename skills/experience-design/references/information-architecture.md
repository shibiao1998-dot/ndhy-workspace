## 信息架构设计方法

### 核心原则

信息架构是体验设计的地基。组织方式必须匹配**用户的心理模型**，而非技术架构或业务部门划分。

### 三个核心任务

#### 1. 内容组织

**步骤：**
1. **盘点信息**：列出产品中所有信息/内容/功能
2. **分析关系**：梳理信息之间的层级、并列、关联关系
3. **对齐心理模型**：用户脑子里怎么分类这些信息？
4. **组织分类**：按用户心理模型而非业务逻辑组织

**常用组织方式：**

| 方式 | 适用场景 | 示例 |
|------|---------|------|
| 按任务/场景 | 用户目标驱动的产品 | "备课"、"上课"、"改作业" |
| 按内容类型 | 内容消费型产品 | "视频"、"文档"、"题库" |
| 按时间 | 流程性/时序性内容 | "待处理"、"进行中"、"已完成" |
| 按受众 | 多角色产品 | "教师空间"、"学生空间" |

**❌ 错误做法：** 按内部部门划分（"教研中心"、"运营模块"）
**✅ 正确做法：** 按用户任务划分（"备课"、"授课"、"评估"）

#### 2. 卡片分类法（可选验证）

当不确定用户心��模型时，用卡片分类法验证：
1. 列出所有功能/信息条目（写成卡片）
2. 请目标用户按自己的理解分组
3. 统计分组结果，提取用户的自然分类逻辑
4. 以此为基础设计信息架构

#### 3. 信息架构图格式

```
产品
├── 一级模块A（对应核心场景1）
│   ├── 二级页面A1（信息类型 / 访问频率：高/中/低 / 到达路径：主导航直达）
│   ├── 二级页面A2（信息类型 / 访问频率 / 到达路径���
│   └── 二级页面A3
├── 一级模块B（对应核心场景2）
│   ├── 二级页面B1
│   └── 二级页面B2
├── 一级模块C（对应核心场景3）
└── 全局功能
    ├── 搜索（全局入口）
    ├── 通知（全局入口）
    ├── 设置（全局入口）
    └── 帮助（全局入口）
```

**标注要素：**
- 每个一级模块标注对应的核心场景
- 每个二级页面标注：信息类型、访问频率、到达路径
- 全局功能单独列出

## 导航体系设计

### 导航三件套

| 导航类型 | 作用 | 形式 | 设计要点 |
|---------|------|------|---------|
| **主导航** | 核心路径入口 | 顶部栏 / 侧边栏 / 底部 Tab | 项目数 3-7 个，标签用用户语言 |
| **辅助导航** | 快捷到达 | 搜索 / 快捷入口 / 最近使用 | 高频功能快速触达 |
| **位置导航** | 定位感知 | 面包屑 / 步骤条 / 高亮当前 | 用户随时知道"我在哪" |

### 导航设计三问

每个页面的用户必须能回答：

| # | 问题 | 设计手段 |
|---|------|---------|
| 1 | **我在哪？** | 面包屑、当前导航高亮、页面标题 |
| 2 | **我能去哪？** | 导航菜单、相关链接、操作按钮 |
| 3 | **怎么回去？** | 返回按钮、面包屑、浏览器后退 |

### 主导航选型

| 类型 | 适用场景 | 优势 | 劣势 |
|------|---------|------|------|
| **顶部导航** | 模块 ≤ 7 个，桌面端为主 | 空间利用好 | 移动端不友好 |
| **侧边导航** | 模块多、层级深、管理后台 | 可展开/折叠 | 占用水平空间 |
| **底部 Tab** | 移动端、模块 3-5 个 | 拇指可达 | 最多 5 个 |
| **汉堡菜单** | 辅助功能、低频入口 | 节省空间 | 隐藏降低发现率 |

### 导航标签命名规则

| ❌ 不要 | ✅ 应该 | 原因 |
|--------|--------|------|
| 内部术语："资源中心" | 用户语言："备课素材" | 用户不知道"资源中心"是什么 |
| 抽象名词："管理" | 具体动作："我的课程" | 用户需要预判点击后看到什么 |
| 缩写："AI 辅助" | 完整描述："AI 帮你备课" | 清晰优于简短 |

## 层级控制规则

### 三条硬规则

| # | 规则 | 标准 | 原因 |
|---|------|------|------|
| 1 | **信息深度限制** | ≤ 3 层（主流功能 ≤ 2 层） | 超过 3 层用户开始迷路 |
| 2 | **每层选项限制** | 一屏 ≤ 7 个选项 | 超过 7 个选择困难（米勒法则） |
| 3 | **重要信息位置** | 最浅层级可见 | 重要的东西不能藏 |

### 层级过深的解决方案

当内容天然层级多时：

| 方案 | 做法 | 适用场景 |
|------|------|---------|
| **扁平化** | 合并相似层级 | 中间层级区分度低 |
| **快捷入口** | 高频内容提供跳板 | 深层内容访问频率高 |
| **搜索替代** | 搜索 + 筛选代替层层点击 | 内容量大、用户目标明确 |
| **智能推荐** | 基于上下文推荐 | 用户可能需要但不会主动找 |

### 每层选项过多的解决方案

| 方案 | 做法 | 适用场景 |
|------|------|---------|
| **分组** | 用标题/分割线将选项分组 | 选项有自然分类 |
| **渐进披露** | 先展示常用，其余"更多" | 部分选项使用频率低 |
| **搜索/筛选** | 提供搜索框 | 选项 > 20 个 |

## 质量检验标准

### 信息架构四检验

每次完成信息架构设计后，逐条检验：

| # | 检验标准 | 检验方法 | 通过标准 |
|---|---------|---------|---------|
| 1 | **30 秒找到核心功能** | 新用户测试：不看说明书，30 秒内能否找到核心功能 | 所有核心功能通过 |
| 2 | **3 次点击到达** | 统计每个功能的最短点击路径 | 任何功能到达 ≤ 3 次点击 |
| 3 | **用户语言** | 审查所有导航标签和分类名称 | 无内部术语，全部使用用户能理解的词 |
| 4 | **同类信息归位** | 检查是否有同类信息分散在不同位置 | 同类信息在同一个地方 |

### 自检清单

- [ ] 一级模块数量 ≤ 7 个
- [ ] 每个一级模块对应至少一个核心场景
- [ ] 最大层级深度 ≤ 3 层
- [ ] 核心功能 ≤ 2 层可达
- [ ] 每层选项 ≤ 7 个
- [ ] 导航标签使用用户语言
- [ ] 全局功能（搜索、设置、帮助）有明确位置
- [ ] 面包屑或位置指示覆盖所有深层页面
- [ ] 分类方式基于用户心理模型（非业务/技术架构）
- [ ] 信息架构图标注完整（模块-场景对应、访问频率、到达路径）
