# experience-design — 体验设计执行工具箱

将产品定义转化为用户可感知的使用体验。回答"用户打开它的第一秒到关掉它的最后一秒，每一步怎么走、看到什么、理解什么、感受到什么"。

> 体验设计 ≠ 界面美化 ≠ 画原型。体验设计 = 站在用户角度，设计"使用产品的完整经历"。

## 8 步标准流程

```
Step 1：理解输入
  │  消化产品定义文档，确认核心场景和任务
  │  → 提取：产品概念、核心场景、功能范围、产品规则、成功标准
  │  → 信息不足时，有权要求上游补充
  │
Step 2：信息架构设计
  │  → 内容组织（按用户心理模型，非技术架构）
  │  → 导航体系（主导航 + 辅助导航 + 位置导航）
  │  → 层级控制（≤ 3 层深度，每层 ≤ 7 选项）
  │  → 产出【信息架构图 + 导航结构表】
  │  → 过信息架构四检验（见下方速查）
  │  → 详细方法 → references/information-architecture.md
  │
Step 3：任务流程设计
  │  → 绘制任务流（最优路径、最少步骤）
  │  → 绘制用户流（加入分支：决策/异常/中断/放弃）
  │  → 标注关键节点（📌决策 ⏳等待 ⚠️风险 🎯价值）
  │  → 遵守任务流四铁律（见下方速查）
  │  → 详细方法 → references/task-flows.md
  │
Step 4：界面布局与交互模式
  │  → 每个关键页面定义框架结构（区块 + 优先级 + 固定/滚动）
  │  → 选择交互模式（表单/浏览/反馈/引导，用成熟模式）
  │  → 处理所有页面状态（加载/有内容/空/错误）
  │  → 定义响应式策略（核心设备 + 断点 + 移动端取舍）
  │  → 交互模式库 → references/interaction-patterns.md
  │
Step 5：体验语言定义
  │  → 定义产品人设（类型 + 称呼 + 句式 + 情感温度）
  │  → 设计关键文案（按钮/提示/错误/空状态/加载）
  │  → AI 对话语言设计（如适用）
  │  → 产出【语言风格指南 + 文案模板 + 禁用词清单】
  │  → 详细方法 → references/experience-language.md
  │
Step 6：体验地图绘制
  │  → 横轴：认知 → 首次使用 → 核心使用 → 持续使用 → 推荐/流失
  │  → 纵轴：行为 + 接触点 + 情感曲线 + 机会点
  │  → 标注：⭐ 峰值 / ⚠️ 谷底 / 🚪 转化点
  │  → 应用峰终定律：刻意制造正面峰值 + 消除负面谷底 + 确保终值正面
  │  → 绘制指南 → references/checklists.md §体验地图绘制指南
  │
Step 7：可用性走查
  │  → 逐页检查四个层面（信息/操作/状态/效率）
  │  → 问题记录 + 修复方案
  │  → 可达性检查（色觉/字号/触控/弱网）
  │  → 走查清单 → references/checklists.md §可用性走查清单
  │
Step 8：规范输出
     → 组件行为规范（弹窗/Toast/下拉/卡片使用场景）
     → 状态转换规范（按钮状态链 + 页面状态链）
     → 反馈规范（即时/过程/结果三层）
     → 动效规范（场景 + 时长 + 节制原则）
     → 按 7 章节模板组织文档 → references/templates.md
```

## 设计深度速查

| 深度 | 适用场景 | 覆盖范围 | 输出物 |
|------|----------|----------|--------|
| **快速设计** | 小功能迭代、已有设计语言延伸 | 任务流程 + 界面布局，复用现有规范 | 体验设计文档（轻量版） |

## 7 大能力域速查

### 域 1：信息架构设计

**核心问题**：产品里的信息怎么组织、怎么分类、怎么找到？

三个核心任务：
- **内容组织**：按用户心理模型分类，非技术架构或业务部门
- **导航体系**：用户在任何位置能回答"我在哪？能去哪？怎么回去？"
- **层级控制**：深度 ≤ 3 层 / 每层 ≤ 7 选项 / 重要信息在最浅层

→ 详细方法 [references/information-architecture.md](references/information-architecture.md)

### 域 2：任务流程设计

**核心问题**：用户完成每个任务的具体步骤路径是什么？

设计方法：
1. 绘制**任务流**（最少步骤、无分支）
2. 绘制**用户流**（加入决策/异常/中断/放弃/纠错分支）
3. 标注**关键节点**（📌决策 ⏳等待 ⚠️风险 🎯价值）

→ 详细方法 [references/task-flows.md](references/task-flows.md)

### 域 3：界面布局与交互模式

**核心问题**：每个页面的结构和交互行为是什么？

三个层面：
- **页面结构**：区块组成、优先级排序、固定/可滚动
- **交互模式**：表单（单步/分步）、浏览（列表/卡片）、反馈（Toast/结果页）、引导（新手/空状态）
- **响应式策略**：核心设备、断点、移动端取舍

→ 交互模式库 [references/interaction-patterns.md](references/interaction-patterns.md)

### 域 4：体验语言定义

**核心问题**：产品和用户"说话"的方式是什么？

三个维度：
- **语言风格**：产品人设 + 称呼 + 句式 + 情感温度 + 禁用词
- **关键文案**：按钮（动词优先）/ 提示（说人话+给例子）/ 错误（问题+解法）/ 空状态（该有什么+怎么创建）
- **AI 对话语言**：角色定位 + 不比用户更权威 + 表达不确定性

→ 详细方法 [references/experience-language.md](references/experience-language.md)

### 域 5：体验地图绘制

**核心问题**：用户从认知到推荐的全过程感受如何？

关键标注：
- ⭐ **峰值时刻**：刻意设计（超预期/成就感/惊喜/效率飞跃），不靠运气
- ⚠️ **谷底时刻**：识别并设计应对措施
- 🚪 **关键转化点**：从"试试看"到"真的用"的转折

**峰终定律**：人对体验的记忆取决于峰值和终值 → 刻意制造正面峰值 + 确保结束体验正面

→ 绘制指南 [references/checklists.md](references/checklists.md) §体验地图绘制指南

### 域 6：可用性保障

**核心问题**：产品对所有目标用户都"用得了、用得懂、用得顺"吗？

五项原则：可见性 / 一致性 / 容错性 / 效率 / 可达性

走查四层面：信息层面 / 操作层面 / 状态层面 / 效率层面

→ 走查清单 [references/checklists.md](references/checklists.md) §可用性走查清单

### 域 7：设计规范输出

**核心问题**：为视觉设计和开发落地输出什么规范？

四类规范：
- **组件行为**：弹窗/Toast 使用场景、下拉 vs 平铺选用标准
- **状态转换**：按钮状态链（默认→悬停→按下→禁用→加载中）、页面状态链（加载→有内容→空→错误）
- **反馈规范**：即时（所有操作）/ 过程（>1秒）/ 结果（重要操作）
- **动效规范**：节制原则——能不加就不加

## 5 条铁律

| # | 铁律 | 核心要求 | 违反后果 |
|---|------|----------|----------|
| 1 | **用户视角优先** | 先问"用户会怎么想"，冲突时改设计不教育用户 | 用户不买账 |
| 2 | **一致性高于创新** | 产品内同样方式，行业已有习惯不改 | 用户困惑 |
| 3 | **每个状态都必须被设计** | 空/错误/加载/断网/首次——全要设计 | 体验断裂 |
| 4 | **文案是设计的一部分** | 所有文案设计阶段确定，不留给开发 | 体验不完整 |
| 5 | **为最差情况设计** | 最慢网速、最小屏幕、最低数字素养 | 部分用户用不了 |

## 任务流 4 铁律速查

| # | 铁律 | 核心要求 | 量化标准 |
|---|------|----------|----------|
| 1 | **最短路径** | 每增一步都要问"能不能省/合并/自动化" | P0 ≤ 3 步，P1 ≤ 5 步 |
| 2 | **渐进式披露** | 先给最关键的，其余需要时再展开 | 默认只展示完成任务必须的信息 |
| 3 | **可逆原则** | 任何操作可撤销，不可逆操作必须二次确认 | 删除→回收站，危险→确认弹窗 |
| 4 | **中断恢复** | 中途离开回来能接着干，内容不丢、进度记住 | 自动保存 + 恢复提示 |

→ 详细方法 [references/task-flows.md](references/task-flows.md) §任务流四条铁律详解

## 信息架构 4 检验速查

| # | 检验标准 | 通过条件 |
|---|---------|---------|
| 1 | **30 秒找到核心功能** | 新用户不看说明书能找到 |
| 2 | **3 次点击到达** | 任何功能最短路径 ≤ 3 次点击 |
| 3 | **用户语言** | 导航标签无内部术语 |
| 4 | **同类信息归位** | 同类信息在同一个地方 |

→ 详细方法 [references/information-architecture.md](references/information-architecture.md) §质量检验标准

## 7 个质量问题（❌/✅ 对比）

| # | 问题 | ❌ 错误 | ✅ 正确 |
|---|------|--------|--------|
| 1 | **信息过载** | 首页放 15 个功能入口 | 每个页面只有一个核心目标 |
| 2 | **路径过长** | 创建教案要 8 步 | 核心任务 3 步以内 |
| 3 | **状态黑洞** | 点了保存啥反应没有 | 每个操作必须有即时反馈 |
| 4 | **专业术语** | "资源库管理"、"模块配置" | 用"说给用户听"的语言 |
| 5 | **一致性断裂** | 删除有时确认有时不确认 | 建立组件规范全局统一 |
| 6 | **忽视异常** | 断网时页面空白 | 每个流程覆盖成功/失败/中断/异常 |
| 7 | **假创新** | 自创下拉选择器样式 | 用户已有习惯不改 |

→ 详细防范措施 [references/checklists.md](references/checklists.md) §7 个质量问题防范

## 协作接口

### 上游输入

| 来源 | 接收内容 | 使用方式 |
|------|---------|---------|
| 产品定义专家 | 产品定义文档（概念、场景、功能、规则、标准） | **主要输入**，体验设计在产品定义框架内展开 |
| 战略定位专家 | 差异化要求 | 交互方案必须体现核心差异 |
| 用户研究专家 | 用户画像、行为模式 | 信息架构和任务流程的用户依据 |

→ 信息不足时有权要求上游补充

### 下游输出

| 接收方 | 传递内容 | 约束关系 |
|--------|---------|---------|
| 视觉设计师 / 前端开发 | 体验设计文档全套 | 视觉和开发必须遵循体验规范 |
| 产品架构师 | 信息架构、任务流程 | 技术架构需支撑体验设计 |
| 项目管理专家 | 页面清单、优先级 | 排期参考 |

## 触发时首要动作

无论如何触发，第一步永远是：

> "产品定义告诉了我们什么？核心场景是什么、核心任务是什么、用户是谁、约束在哪？"

→ 判断设计深度 → 消化产品定义输入 → 按 8 步流程执行
