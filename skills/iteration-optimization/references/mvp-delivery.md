# MVP 与渐进式交付

## 核心理念

不要一步到位，分步验证。

- MVP不是"做一个很粗糙的东西"，而是"只做最核心的部分，把它做好"
- 目的是**学习**，不是交差
- 用最小的投入验证一个想法是否可行

---

## MVP 拆解方法

### 拆解三问

```
问1：这个功能要解决的核心问题是什么？
  → 从用户角度定义问题，不是从功能角度

问2：解决这个问题的最小验证是什么？
  → 验证的是"用户是否需要"，而不是"能不能做完美"

问3：最小验证需要什么？
  → 列出必须有的能力，砍掉所有"最好有"的
```

### 拆解示例

**完整想法**："做一个AI自动生成教案的功能"

```
问1：核心问题是什么？
→ 教师备课太耗时

问2：最小验证是什么？
→ 验证"教师是否愿意使用AI生成的教案" 
→ 而不是验证"AI能不能生成完美教案"

问3：最小验证需要什么？

  必须有：
  ✅ 支持1-2个高频学科（语文和数学）
  ✅ 输入课题，AI生成教案框架（目标+环节+时间分配）
  ✅ 教师可以在线编辑修改
  ✅ 可以导出为Word

  不需要（留给后续）：
  ❌ 其他学科支持
  ❌ 自动匹配课标和教材
  ❌ 素材自动推荐
  ❌ 协作和分享功能
  ❌ 历史教案管理
```

### MVP 范围检验

MVP方案确定后，用以下问题检验：

| # | 检验问题 | 不通过的后果 |
|---|---------|-------------|
| 1 | 这个MVP能验证核心假设吗？ | 做了也不知道对不对 |
| 2 | 去掉任何一项，还能验证吗？ | 如果能，说明还可以更精简 |
| 3 | 用户能感知到价值吗？ | MVP也要让用户觉得有用 |
| 4 | 2-3周内能交付吗？ | 超过3周就不够"M" |
| 5 | 失败了能承受吗？ | 投入太大就不是MVP了 |

## 渐进式交付节奏

### 四阶段模型

```
V1 (MVP)
  │  验证需求是否真实存在
  │  关注：有多少人愿意试用？试用后觉得有帮助吗？
  │  周期：2-3周
  │
  │  ◆ 决策点：需求真实吗？继续/调整/放弃？
  │
V1.5 (快速修补)
  │  修复MVP暴露的关键问题
  │  关注：V1中用户遇到的最大障碍是什么？
  │  周期：1-2周
  │
  │  ◆ 决策点：核心体验可接受吗？继续/调整/放弃？
  │
V2 (核心增强)
  │  把核心体验做到及格线以上
  │  关注：留存率达标？核心流程顺畅？
  │  周期：3-4周
  │
  │  ◆ 决策点：核心数据达标吗？可以拓展了吗？
  │
V2.5 / V3 (拓展)
     覆盖更多场景和学科
     前提：V2核心数据已达标
     关注：能否复制到更多场景？通用性如何？
     周期：根据拓展范围定
```

### 每个阶段的决策要素

| 阶段 | 核心问题 | 达标信号 | 不达标行动 |
|------|---------|---------|-----------|
| V1→V1.5 | 需求真实吗？ | 试用率>预期，正面反馈>负面 | 深挖原因，调整方向或止损 |
| V1.5→V2 | 核心体验可接受？ | 关键阻碍已解决，使用频率稳定 | 继续修补或重新评估需求 |
| V2→V2.5 | 核心数据达标？ | 留存、完成率达到设定阈值 | 先优化核心，不急于拓展 |
| V2.5→V3 | 可以规模化？ | 拓展后数据不降级 | 控制拓展范围，逐步验证 |

---

## 决策点设计

每个阶段之间都有一个**决策点**，这是防止沉没成本陷阱的关键。

### 决策三选一

```
1. 继续投入 → 数据达标，方向正确，进入下一阶段
2. 调整方向 → 需求存在但实现方式需要改变
3. 止损放弃 → 需求不成立或ROI不划算
```

### 决策纪律

- **不要因为"已经做了V1就一定要做V2"** — 沉没成本不是继续的理由
- **每个决策点都要重新评估** — 环境可能已经变了
- **决策要有数据支撑** — "我觉得应该继续"不够，要拿出数据
- **决策要记录** — 无论继续还是止损，记录理由，方便未来回顾

## MVP 常见误区

| 误区 | 表现 | 正确做法 |
|------|------|---------|
| MVP = 粗糙版 | "先凑合做一个，后面再优化" | 核心部分做好，非核心不做 |
| MVP = 功能砍半 | "把完整方案每项都做一半" | 砍需求项不砍质量 |
| MVP = 原型 | "做个Demo就算MVP" | MVP要能真实使用和验证 |
| MVP = 最终版删减版 | "先想最终版，再删功能" | 从核心问题出发，而非从完整方案出发 |
| 一个产品只有一次MVP | "MVP做完就不用了" | 每个大功能都可以用MVP思维拆解 |

## 渐进交付的范围控制

### 每轮交付的范围原则

```
一轮交付 = 1个可验证的价值增量

验证标准：
→ 用户能感知到变化
→ 可以收集到反馈或数据
→ 可以独立判断是否成功

反例：
→ "做了后端但前端还没开始" — 用户感知不到
→ "功能做了但没发布" — 收集不到数据
→ "做了3个功能但没法分别评估" — 无法判断哪个成功
```

### 交付块拆解原则

- 每块 ≤ 1周工作量
- 可独立发布、独立验证
- 有明确的验收标准
- 块之间的依赖关系清晰
