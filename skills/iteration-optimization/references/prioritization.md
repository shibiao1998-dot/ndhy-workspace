# 优先级排序框架详解

## ABCD 四级分类

### A类：🩸 止血

**定义**：不修就会持续伤害用户或业务的问题。

**典型场景**：
- 核心功能崩溃 / 频繁闪退
- 数据丢失或损坏
- 严重的体验bug（流程走不通）
- 安全漏洞
- 合规性硬伤

**判断标准**：
- 正在造成用户流失？ → A类
- 正在造成数据损失？ → A类
- 正在违反法规/政策？ → A类

**处理原则**：
- 立刻做，放下手里一切来做
- 不需要走完整的排期流程
- 修完立即发布，不等版本窗口

---

### B类：💪 强化核心

**定义**：让产品核心价值变得更强的改进。

**典型场景**：
- 核心功能的效率提升
- 核心流程的步骤简化
- 关键转化节点的优化
- 核心体验的打磨（不是边缘体验）

**判断标准**：
- 做了之后，核心功能的用户满意度会提升？ → B类
- 做了之后，核心流程的完成率会提升？ → B类
- 做了之后，用户留存会直接受益？ → B类

**处理原则**：
- 优先做，这是迭代的主战场
- 占每轮迭代 50-70% 的工作量
- B类需求应该占据每轮迭代的绝对主力

---

### C类：🌱 拓展边界

**定义**：为产品增加新能力或覆盖新场景。

**典型场景**：
- 新功能模块
- 新用户角色支持
- 新终端/平台适配
- 新场景覆盖

**判断标准**：
- 这是核心体验之外的新领域？ → C类
- 需要新建而非优化？ → C类
- 服务的是新用户群或新场景？ → C类

**处理原则**：
- 谨慎做，确认核心已经足够好再拓展
- 核心功能满意度没达标时，不做C类
- 做C类前先用MVP验证需求真实性

---

### D类：✨ 锦上添花

**定义**：有了更好，没有也不影响核心的改进。

**典型场景**：
- 动效美化
- 非核心流程的体验优化
- 视觉微调
- 边缘场景的便利性提升

**判断标准**：
- 去掉这个改进，核心体验不受影响？ → D类
- 用户不会因为没有它而不满？ → D类

**处理原则**：
- 资源富余时做，或与其他需求顺便做
- 绝不挤占B类的资源
- 如果需要砍需求，D类第一个砍

---

### 常见分类错误

| 错误 | 表现 | 后果 |
|------|------|------|
| D当B做 | 花大量资源打磨边缘体验 | 核心问题无人管 |
| 总想做C | "再加一个功能就好了" | 核心越来越弱，产品臃肿 |
| A压着不做 | "先做完这个版本再修" | 用户跑了 |
| B降级为D | "核心优化不紧急" | 产品竞争力持续下降 |

## ICE 评分法

### 三个维度

#### I = Impact（影响力）
做了之后对核心目标的影响有多大？

| 分数 | 描述 | 判断依据 |
|------|------|---------|
| 3 高 | 直接影响核心指标 | 做了之后核心数据会有可见变化 |
| 2 中 | 间接影响或影响次要指标 | 对核心有帮助但不是直接推动 |
| 1 低 | 影响有限 | 做了锦上添花，不做也没事 |

#### C = Confidence（信心）
我们有多确定它能产生预期效果？

| 分数 | 描述 | 判断依据 |
|------|------|---------|
| 3 高 | 有数据/案例支持 | 有A/B测试数据、竞品验证、用户调研结果 |
| 2 中 | 有合理推测无直接证据 | 逻辑上说得通但没有直接数据 |
| 1 低 | 主要靠直觉 | "我觉得应该是这样" |

#### E = Ease（容易度）
实现它需要多少资源和时间？

| 分数 | 描述 | 判断依据 |
|------|------|---------|
| 3 高 | 1-2天搞定 | 改改配置、调调参数、小的UI调整 |
| 2 中 | 1-2周搞定 | 一个完整的需求块 |
| 1 低 | 需要大量资源或跨多个迭代 | 大功能、需要多团队协作 |

### 计算与使用

**ICE 分 = I × C × E**

| 分数区间 | 优先级建议 |
|----------|-----------|
| 18-27 | 🔴 立即做 |
| 8-17 | 🟡 排入本轮 |
| 1-7 | 🟢 排入待办池 |

### 使用注意事项

1. **分数是辅助工具，不是圣旨**
   - ICE分高但直觉不对 → 停下来想想为什么
   - 两个分数接近 → 考虑依赖关系，先做前置的

2. **不要过度精确**
   - 分出大致的高/中/低就够了
   - 不要花30分钟给一个需求精确打分

3. **团队共识比分数重要**
   - 有争议的需求讨论清楚比打分有用
   - ICE是促进讨论的工具，不是终结讨论的武器

## 砍需求原则

### 核心规则：宁可少做做好，不要多做做烂

### 产能计算
```
每轮迭代有效需求量 = 团队可用人天 × 0.7

为什么打7折？
→ 10% 给突发和紧急问题
→ 10% 给技术债务和内部优化
→ 10% 给沟通协调和不可预期的复杂度
```

### 砍需求流程

```
1. 所有需求按 ABCD 分类
2. 每类内部按 ICE 排序
3. 按 A→B→C→D 顺序累加工作量
4. 超出有效产能的部分，从底部砍掉
5. 砍掉的放入"待办池"，下轮重新评估
```

### 砍需求的三条铁律

**铁律一：从底部砍，不要"每个都压缩一点"**
- ❌ "这个需求我们简化一下做"——简化后的半成品往往比不做更差
- ✅ 要么做完整版，要么不做，放到下一轮

**铁律二：警惕"顺手做"**
- "这个很小，顺手就做了" → 这是最大的谎言
- 10个"顺手做的"加起来 = 1个大需求
- 把"顺手做"也算进工作量

**铁律三：砍掉 ≠ 否决**
- 砍掉的需求进"待办池"而非"垃圾桶"
- 下轮重新评估，环境变了可能优先级就变了
- 记录砍掉的原因，方便下次决策

## 范围控制实操

### 迭代开始后的需求变更

```
新需求插入请求
├─ A类止血 → 允许插入，不需要换出（但记录对计划的影响）
└─ 非A类
    ├─ 评估紧急度
    │   ├─ 本轮必须做 → 走"换入换出"：插入这个，拿掉一个同等工作量的
    │   └─ 可以等 → 放入待办池，下轮评估
    └─ 任何情况都要记录变更原因
```

### "老板临时加的"怎么办？
- 帮老板理解：加需求 = 延期或砍别的
- 给出具体影响："加这个需要3天，意味着B3要推到下轮"
- 让老板做选择，而不是闷头硬塞

### 范围蔓延预警信号
- 迭代过了一半，需求数量比启动时多了20%以上
- 团队成员在做"计划外"的事情
- 频繁出现"顺便做一下"
- 每日站会的"阻塞"变多

→ 出现以上信号，立即启动范围审查
