# 五阶段迭代循环详解

## 循环模型

```
┌──→ ① 审视现状 ──→ ② 确定方向 ──→ ③ 设计方案 ─┐
│                                                │
│    ⑤ 复盘学习 ←── ④ 执行验证 ←─────────────────┘
│         │
└─────────┘（进入下一轮）
```

> 每一阶段都有明确的输入、输出和关键动作。跳过任何一个阶段都会留下隐患。

---

## 阶段①：审视现状（我们在哪？）

### 输入
- 上一轮迭代的效果数据
- 用户反馈汇总
- 市场/竞品变化
- 业务目标与路线图
- 上轮复盘的改进项

### 关键动作
1. **回顾上轮目标达成度** — 用数据说话，不凭感觉
2. **扫描预期外的问题或发现** — 意外往往比计划更有价值
3. **检查外部环境变化** — 用户需求变了？竞品有新动作？政策有调整？
4. **定位当前最大瓶颈** — 产品的"天花板"在哪里？

### 动作清单
- [ ] 拉取核心数据指标，对比上轮基线
- [ ] 整理用户反馈 top 5 问题
- [ ] 扫描竞品最近动态
- [ ] 检查业务/政策环境变化
- [ ] 回顾上轮复盘改进项执行情况

### 输出
**现状诊断报告**（简明扼要，不写论文），包含：
- 上轮达成度总结（1-2句话）
- 当前核心瓶颈（1-2个）
- 环境变化要点
- 上轮改进项执行回顾

### 常见陷阱
- ❌ 跳过数据直接拍脑袋 → 数据是诊断的基础
- ❌ 只看内部不看外部 → 环境变化可能比内部改进更重要
- ❌ 写长篇分析报告 → 诊断报告一页纸足够

---

## 阶段②：确定方向（我们去哪？）

### 输入
- 现状诊断报告
- 产品路线图
- 资源约束（人力、时间、技术）

### 关键动作
1. **锁定核心目标** — 只能有1个，不能是"什么都想做"
2. **论证优先级** — 为什么是这个目标而不是别的？
3. **定义成功标准** — 怎么知道这轮迭代"做对了"？
4. **明确排除项** — 本轮不做什么，写在纸上避免范围蔓延

### 核心目标的质量检验

好的目标 vs 坏的目标：

| 坏目标 | 好目标 |
|--------|--------|
| "优化用户体验" | "注册后首周留存从40%提升到55%" |
| "增加新功能" | "教师备课时间减少30%" |
| "提升性能" | "核心页面加载时间从3s降到1.5s" |
| "做AI功能" | "验证教师是否愿意使用AI生成的教案框架" |

**检验标准**：
- 一句话说清楚？ ✅
- 有可量化的成功指标？ ✅
- 排除项明确？ ✅
- 团队成员复述一致？ ✅

### 输出
**迭代目标声明**（一页纸以内），模板见 [templates.md](templates.md)

### 常见陷阱
- ❌ 核心目标超过1个 → 等于没有目标
- ❌ 目标不可量化 → 等于没有验收标准
- ❌ 没有排除项 → 等于给范围蔓延开门

---

## 阶段③：设计方案（怎么去？）

### 输入
- 迭代目标声明
- 技术约束
- 时间约束

### 关键动作
1. **方案对比** — 实现目标的方案有哪些？各有什么利弊？
2. **MVP拆解** — 能不能用最小成本先验证假设？
3. **拆成小块** — 每块最好不超过1周工作量，可独立交付
4. **定验收标准** — 什么样算"做完了"，什么样算"做好了"
5. **排优先级** — 用ABCD分类 + ICE评分

### 方案设计原则
- **先验证假设再投入资源**：不确定需求真实性时，先做最小验证
- **可独立交付**：每个需求块交付后，用户应能感知到变化
- **前置依赖先做**：有依赖关系的需求，先做前置项
- **预留缓冲**：需求量 = 产能 × 0.7

### 输出
**迭代方案**，包含：
- 需求清单（按ABCD分级，附ICE分和验收标准）
- 交付计划（时间线 + 验证节点）
- 风险和依赖关系
- MVP拆解（如适用）

模板见 [templates.md](templates.md)

### 常见陷阱
- ❌ 方案没有备选 → 至少考虑2种方案
- ❌ 拆得太粗（超过1周一块） → 交付风险增大
- ❌ 验收标准模糊 → "做好了"没有客观判断依据
- ❌ 没留缓冲 → 必然延期或质量下降

---

## 阶段④：执行验证（走起来）

### 输入
- 迭代方案

### 关键动作
1. **按优先级逐个交付** — 不要并行太多，集中力量打歼灭战
2. **小块验证** — 每完成一块就做验证，不攒到最后
3. **及时止损** — 发现方向不对，果断调整
4. **记录过程** — 问题和发现为复盘积累素材

### 执行纪律
- **每完成一个需求块**：对照验收标准检查 → 通过才进入下一个
- **发现阻塞**：立即升级，不等不拖
- **新需求插入**：走换入换出流程，A类例外
- **执行中发现假设有误**：暂停，重新评估方向

### 验证节奏
```
日维度：每日站会确认进展和阻塞
块维度：每个需求块完成时验证
周维度：周回顾检查整体进度和方向
```

### 输出
- 交付物（已发布或可发布状态）
- 执行过程记录（问题、发现、调整）

### 常见陷阱
- ❌ 并行太多导致都做不完 → 集中力量，逐个击破
- ❌ 验证攒到最后 → 问题越晚发现修复成本越高
- ❌ 发现不对还硬做 → 沉没成本不是继续的理由
- ❌ 不记录过程 → 复盘时"记不住了"

---

## 阶段⑤：复盘学习（学到了什么？）

### 输入
- 执行结果（数据）
- 效果数据（对照成功标准）
- 执行过程记录

### 关键动作
1. **目标达成度** — 用数据对照成功标准，差距在哪？
2. **做对了什么** — 可复用的好做法
3. **做错了什么** — 下次要避免的坑
4. **意外发现** — 计划之外的收获
5. **继续/停止/开始** — 形成下一轮的行动指引

### 复盘纪律
- **时间**：控制在1小时以内，超时说明讨论太多细节
- **态度**：聚焦"我们能控制的部分"，不甩锅
- **结论**：必须具体到可执行的动作，不说"加强沟通"
- **闭环**：每次复盘先回顾上轮改进项是否执行

### 复盘时机
| 时机 | 目的 |
|------|------|
| 每轮迭代结束后 | 必做，不能省 |
| 重大功能上线1-2周后 | 等数据稳定再看 |
| 重大问题/事故后 | 越快越好 |
| 季度/学期节点 | 大复盘，看全局 |

### 复盘常见陷阱

| 陷阱 | 表现 | 正确做法 |
|------|------|---------|
| 甩锅大会 | "是XX没配合好" | 聚焦我们能控制的部分 |
| 表功大会 | "这轮做了很多事" | 从结果出发，不从工作量出发 |
| 结论太空 | "下次加强沟通" | 具体到可执行动作 |
| 没有闭环 | 上次说的改进没跟踪 | 每次先回顾上轮改进项 |

### 输出
**迭代复盘记录** → 直接输入下一轮的"审视现状"

模板见 [templates.md](templates.md)
