# 教育行业迭代特别指南

教育产品的迭代有其特殊性：教学节奏、教师习惯、政策驱动、效果验证周期和多角色复杂性都需要特别考量。

---

## ① 教学节奏决定迭代节奏

教育产品的迭代不能只看"产品自己的节奏"，必须对齐"教学节奏"。

### 学期节奏与迭代策略

| 时间段 | 教学状态 | 迭代策略 |
|--------|---------|---------|
| **学期初**（9月/2月） | 教师最愿意尝试新工具 | 🟢 黄金窗口：大版本尽量在学期初前就绪 |
| **学期中**（10-12月/3-5月） | 教师最忙，不愿改变习惯 | 🟡 小优化为主，不做大变更，不破坏已有习惯 |
| **考试季**（期中/期末前） | 教师压力最大 | 🔴 冻结变更，只做紧急修复 |
| **假期**（寒暑假） | 有时间但不在教学状态 | 🟢 大迭代、技术重构、架构升级 |

### 迭代日历（示例）

```
8月中    大版本发布，新学期功能就绪
         → 学期初上线的功能必须稳定，教师没时间陪你调试

9月      跟踪新功能使用情况，快速修复问题
         → 密切关注数据，对新功能做热修复

10-11月  根据反馈做小迭代优化
         → 不动核心交互，只做体验微调

12月     冻结大变更，稳定运行
         → 为期末考试季做准备

1月      期末考试期间，只做紧急修复
         → 最小化干扰

寒假     大迭代、技术重构、架构升级
         → 可以做培训和引导
         → 但不要假设教师假期一定会用产品

（下学期类推）
```

### 教育迭代节奏的铁律

1. **学期初前大版本必须就绪且稳定**
2. **学期中不做破坏性变更**
3. **考试季冻结发布**
4. **假期是大迭代的窗口**

## ② 教师的"切换成本"极高

教师不像消费者——他们不会随便切换工具。

### 为什么切换成本高？

- 教师的工具切换涉及"教学流程重组"，成本远高于一般用户
- 一旦投入时间（上传课件、建班级），迁移成本极高
- 教师的工具选择往往绑定整个教学流程

### 好处与风险

| 方面 | 影响 |
|------|------|
| 好处 | 留存基础好，教师不会轻易走 |
| 风险 | 教师也不会轻易来——首次使用门槛必须极低 |

### 迭代启示

1. **改版保持核心交互一致**
   - 不要让教师觉得"这还是那个产品吗？"
   - UI可以美化，但核心操作路径不能大改

2. **功能迁移提供平滑过渡**
   - 旧数据、旧习惯不能一刀切
   - 新旧版本并行一段时间

3. **新功能要有引导期**
   - 不是上线就完了，要教会教师用
   - 提供使用引导、教程、示例

4. **大改版给足过渡时间**
   - 提前通知（至少2周）
   - 提供明确的切换指引
   - 考虑在假期发布大改版

## ③ "政策需求"的特殊处理

教育行业经常出现"上面要求做"的需求。

### 政策需求分类

| 类型 | 定义 | 处理原则 |
|------|------|---------|
| **硬性要求** | 不做会影响合规、准入、招标 | 直接纳入迭代，不讨论优先级，争取合理时间 |
| **软性要求** | "最好有"但不是硬性门槛 | 和其他需求一起评估优先级 |

### 政策需求处理原则

1. **先确认"硬度"** — 是必须做还是"最好做"？找到政策原文确认
2. **找"既满足政策又对用户有价值"的实现方式** — 别浪费一次改进的机会
3. **方案留灵活性** — 政策需求来得急、变得快，实现方案要能调整
4. **记录政策来源** — 方便后续追溯和变更管理

### 政策需求的常见陷阱

- ❌ 二手传话当政策 → 找到原文确认
- ❌ 过度解读政策 → 做到合规就够了，不要镀金
- ❌ 只为应付政策 → 找到对用户也有价值的实现方式
- ❌ 政策变了但实现没跟上 → 建立政策变更追踪机制

## ④ 教育产品的"效果验证"周期更长

### 一般互联网 vs 教育产品

| 维度 | 一般互联网 | 教育产品 |
|------|-----------|---------|
| 改了按钮颜色 | 第二天看转化率 | — |
| 改了教学工具 | — | 教师适应→实际教学→一段时间→看效果变化 |
| 验证周期 | 天到周 | 周到学期 |

### 三层验证指标

| 层级 | 指标类型 | 时间尺度 | 示例 |
|------|---------|---------|------|
| 短期 | 使用行为指标 | 天~周 | 有没有用、用了多少、怎么用的 |
| 中期 | 教学过程指标 | 周~月 | 备课时间变化、教学方式变化 |
| 长期 | 教学效果指标 | 月~学期 | 学生表现、教师满意度 |

### 验证策略

```
短期指标（使用行为）
  → 和一般互联网产品类似，可以快速验证
  → 用来判断"教师愿不愿意用"

中期指标（教学过程）
  → 需要几周到一个月
  → 用来判断"教师能不能用好"

长期指标（教学效果）
  → 可能需要一个学期
  → 用来判断"对教学真的有帮助吗"

注意：
→ 不要用一般互联网的"快速验证"标准要求教育产品
→ 迭代决策不能只看短期数据——有些改进需要时间
→ 但也不能以"长期才能见效"为借口不做短期验证
```

## ⑤ 多角色、多层级的迭代复杂性

### 教育产品的角色生态

| 角色 | 定位 | 核心诉求 |
|------|------|---------|
| 教师 | 直接使用者 | 简单好用、节省时间 |
| 学生 | 最终受益者 | 有趣、能学到东西 |
| 学校管理者 | 决策者/采购者 | 数据丰富、管理可控 |
| 教育局 | 监管者/推动者 | 合规、可监管 |
| 家长 | 关注者 | 孩子学到东西、安全 |

### 角色诉求冲突

| 冲突 | 表现 |
|------|------|
| 教师 vs 管理者 | "简单好用" ↔ "数据丰富" |
| 学生 vs 家长 | "有趣" ↔ "学到东西" |
| 教育局 vs 教师 | "合规" ↔ "灵活" |
| 管理者 vs 教师 | "统一标准" ↔ "个性化教学" |

### 迭代中的多角色策略

1. **每轮明确"主要服务角色"** — 不要试图同时满足所有人

2. **默认优先级**（可根据产品阶段调整）：
   ```
   教师日常使用 > 学生学习体验 > 管理数据需求 > 行政合规要求
   ```

3. **为什么教师优先？**
   - 教师不用 → 其他角色的需求都是空谈
   - 教师是产品的"引擎"，引擎不转一切不转

4. **管理者需求不能完全忽视**
   - 管理者影响采购决策
   - 但不能为了管理者牺牲教师体验
   - 找到"管理者看得到数据，但不增加教师负担"的方案

5. **角色冲突处理流程**
   ```
   发现角色诉求冲突
   ├─ 能否找到同时满足的方案？ → 优先选这个
   ├─ 不能同时满足
   │   ├─ 参照默认优先级做取舍
   │   └─ 特殊情况（如政策硬性要求）可调整优先级
   └─ 记录取舍理由，方便后续复盘
   ```

## 教育迭代检查清单（补充）

在通用迭代检查清单基础上，教育项目额外检查：

| # | 检查项 | 通过条件 |
|---|--------|---------|
| E1 | 发布时机对齐教学节奏 | 不在考试季发大版本 |
| E2 | 核心交互一致性 | 改版后教师不会"找不到东西" |
| E3 | 迁移方案平滑 | 旧数据可迁移，有过渡期 |
| E4 | 政策需求已确认硬度 | 有原文依据，非二手传话 |
| E5 | 验证指标分层设计 | 短/中/长期指标都有 |
| E6 | 本轮主要服务角色明确 | 写清楚本轮优先满足谁 |
| E7 | 引导方案已准备 | 新功能有使用引导/教程 |
