# 迭代优化 — 决策场景速查

## 做 vs 不做

```
这个改进解决的是真实问题还是假设问题？
├─ 假设问题 → 先做最小验证
└─ 真实问题
    ├─ 影响多少用户？ → <5%低优 / 5-30%中优评估ROI / >30%高优
    ├─ 不做后果？ → 用户流失→尽快做 / 体验差但不走→排期 / 几乎无影响→可不做
    └─ 做的成本？ → 几天→倾向做 / 几周→仔细权衡 / 几个月→充分论证才做
```

## 优化现有 vs 新功能

```
核心功能满意度达标？ + 核心流程完成率达标？ + 留存率达标？
├─ 任一未达标 → 优先优化现有（漏水的桶别再开口）
└─ 全部达标 → 可以考虑新功能

例外：战略性必须有 / 成本极低可快速验证 / 优化已进入边际递减
```

## 止损信号（出现2个以上认真考虑）

1. 连续3轮优化后核心指标仍未达最低预期
2. 用户调研显示需求本身不强烈（当初误判）
3. 实现成本远超预期且看不到收敛
4. 市场/政策环境已变，方向不再有价值
5. 团队投入大量精力但士气持续走低

## 技术债处理

- 每轮预留 **15-20%** 资源给技术优化
- 技术团队反映"历史代码有问题" → 不无视，一起评估是否先重构
- 重大重构当独立项目规划，不要"顺便做"
