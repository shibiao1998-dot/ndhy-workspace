# 改进信号捕捉

从四个来源发现"该优化什么"，区分"噪音"和"信号"，找到真正需要改的地方。

> 信号捕捉是迭代优化的燃料。没有信号输入的迭代 = 闭门造车。

---

## 信号来源 ①：数据指标

### 看什么

| 数据类型 | 关注点 | 示例 |
|----------|--------|------|
| 核心漏斗转化率 | 哪一步流失最严重？ | 注册→激活→留存→付费 |
| 功能使用频率 | 哪些功能没人用？ | 某功能日活跃<1%用户 |
| 用户行为路径 | 实际用法 vs 设计预期 | 用户绕过了我们设计的主流程 |
| 留存曲线 | 在哪个节点开始流失？ | 第3天留存断崖下跌 |
| 错误与异常日志 | 用户遇到了什么技术问题？ | 某页面500错误率突增 |

### 怎么看

1. **看趋势不看绝对值** — 数据是涨了还是跌了比"数字是多少"更重要
2. **按用户群体切分** — 新用户和老用户的问题可能完全不同
3. **关注"反常"** — 数据突然好或突然差都值得挖

### 数据信号的局限

⚠️ 数据会告诉你"是什么"和"有多少"，但不会告诉你"为什么"。

处理原则：
- 发现数据异常 → 必须通过其他渠道追问原因
- 数据是起点不是终点 → 数据发现问题 + 反馈解释原因 = 完整信号
- 不要只盯一个指标 → 多指标交叉验证

### 数据信号处理流程

```
发现数据异常
├─ 确认数据准确性（排除埋点错误、口径变化）
├─ 切分维度定位问题范围（用户群/功能/时段）
├─ 关联其他渠道追问原因（反馈/调研/日志）
└─ 形成改进假设 → 纳入迭代评估
```

## 信号来源 ②：用户反馈

### 收集渠道

| 渠道 | 特点 | 注意事项 |
|------|------|---------|
| 客服/技术支持汇总 | 量大、频率高 | 注意去重和分类 |
| 用户调研和访谈 | 深度高 | 样本代表性 |
| 产品内反馈入口 | 实时、场景化 | 可能有情绪偏差 |
| 社群/社交媒体 | 真实、无引导 | 声量不等于普遍性 |
| 销售/运营转述 | 来自付费决策者 | 可能有中间传话失真 |

### 处理原则

**原则一：听问题不听方案**
- 用户说："我希望有一个按钮一键导出所有数据"
- 真实问题："导出数据太麻烦"
- 方案不一定是加按钮，可能有更好的路径

**原则二：频率 > 音量**
- 1个用户提出 = 可能是个案
- 10个用户提出类似问题 = 这是信号
- 不要被"声音最大"的用户牵着走

**原则三：关注沉默用户**
- 不反馈的人可能已经放弃了你的产品
- 流失用户访谈往往比活跃用户反馈更有价值
- 留意"突然不用了"的用户群

**原则四（教育行业）：区分"个人偏好"和"教学需要"**
- 教师说"我希望字更大" → 可能是个人偏好
- 教师说"我上课时看不清" → 这是教学需要
- 追问使用场景可以区分

### 反馈信号处理流程

```
收到用户反馈
├─ 分类：功能缺失 / 体验问题 / bug / 建议 / 抱怨
├─ 追问：真实问题是什么？使用场景是什么？
├─ 计数：同类问题出现了多少次？
├─ 判断：个案 or 信号？
│   ├─ 个案（<3次） → 记录，暂不行动
│   └─ 信号（≥3次） → 纳入迭代评估
└─ 沉默检测：最近有没有用户群体突然沉默？
```

## 信号来源 ③：竞品动态

### 关注什么

| 维度 | 问题 |
|------|------|
| 功能动态 | 竞品做了什么新功能/大改动？为什么？ |
| 用户声音 | 用户在竞品社区讨论什么？抱怨什么？ |
| 策略变化 | 竞品的定价和市场策略有什么变化？ |

### 态度原则

1. **竞品做了 ≠ 我们要做**
   - 先判断：它做得对吗？适合我们的用户吗？
   - 有些竞品的"创新"是错误的，不要跟着犯错

2. **竞品没做 ≠ 我们不该做**
   - 竞品的空白可能是我们的差异化机会
   - 关注竞品解决不好的问题

3. **学思路不抄功能**
   - 思考竞品为什么这么做，解决了什么问题
   - 找到更适合我们用户的实现方式

### 竞品信号处理流程

```
发现竞品动态
├─ 判断：这个动态与我们的目标相关吗？
│   ├─ 不相关 → 记录，不行动
│   └─ 相关
│       ├─ 竞品做得好 → 学思路，评估是否纳入迭代
│       ├─ 竞品做得不好 → 记录用户抱怨点，可能是我们的机会
│       └─ 竞品的空白 → 评估是否是差异化方向
└─ 更新竞品认知地图
```

## 信号来源 ④：业务和市场环境

### 关注什么

| 维度 | 示例 |
|------|------|
| 政策变化 | 课标改了、考试改了、评价方式改了（教育行业尤其重要） |
| 技术变化 | 新技术成熟了，以前做不到的现在能做了 |
| 合作伙伴诉求 | 渠道商、集成商有新需求 |
| 公司战略调整 | 商业方向、目标市场变了 |

### 处理原则

1. **政策变化**
   - 先确认"硬度"：必须做 vs 最好做
   - 硬性要求不讨论优先级，直接纳入
   - 软性要求和其他需求一起评估
   - 尽量找"既满足政策又对用户有价值"的实现方式

2. **技术变化**
   - 不要为了用新技术而用新技术
   - 关注："这个技术能让我们更好地解决用户问题吗？"
   - 技术成熟度要评估，不做小白鼠

3. **战略调整**
   - 战略调整会影响迭代方向的全局优先级
   - 及时同步到迭代规划中
   - 避免"战略已经转向但迭代还在做老方向"

### 环境信号处理流程

```
发现环境变化
├─ 判断影响范围
│   ├─ 影响当前迭代 → 评估是否需要调整目标
│   ├─ 影响下轮迭代 → 纳入下轮审视现状
│   └─ 长期影响 → 更新产品路线图
├─ 判断紧急度
│   ├─ 硬性要求（政策/合规） → 立即响应
│   └─ 软性变化 → 按正常优先级评估
└─ 记录信号来源和判断依据
```

## 信号综合评估

当多个来源的信号指向同一个方向时，信号强度叠加：

| 来源数量 | 信号强度 | 行动建议 |
|----------|---------|---------|
| 单一来源 | 弱 | 记录观察，不急于行动 |
| 2个来源交叉验证 | 中 | 纳入下轮迭代评估 |
| 3个以上来源一致 | 强 | 优先纳入本轮/下轮迭代 |
| 数据+反馈+环境三重确认 | 极强 | 必须尽快响应 |

> 最有价值的改进方向往往是"数据发现了问题 + 用户反馈解释了原因 + 环境变化提供了契机"三者叠加的方向。
