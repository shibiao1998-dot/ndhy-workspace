# 关键决策场景

迭代过程中的四个关键岔路口：做vs不做、优化vs新功能、何时止损、技术债处理。

---

## 场景一：做还是不做？

### 决策树

```
这个改进解决的是真实问题还是假设问题？
│
├─ 假设问题
│   └─ 先做最小验证，确认问题真实存在再决策
│
└─ 真实问题
    │
    ├─ 影响多少用户？
    │   ├─ 极少数（<5%） → 低优先级
    │   │   └─ 除非这些用户极其重要（大客户、种子用户）
    │   ├─ 部分（5%-30%） → 中优先级，评估投入产出比
    │   └─ 大多数（>30%） → 高优先级，认真对待
    │
    ├─ 不做的后果？
    │   ├─ 用户会流失 → 尽快做
    │   ├─ 体验变差但不会走 → 排期做
    │   └─ 几乎无负面影响 → 可以不做
    │
    └─ 做的成本？
        ├─ 低（几天） → 倾向于做
        ├─ 中（几周） → 仔细权衡
        └─ 高（几个月） → 必须充分论证才做
```

### 快速判断法

三个"是"就值得做，三个"否"就别做：

| # | 问题 | 是/否 |
|---|------|-------|
| 1 | 有数据或反馈证明这是真实问题？ | |
| 2 | 影响面足够大或用户足够重要？ | |
| 3 | 投入产出比合理（ROI > 1）？ | |

## 场景二：优化现有 vs 做新功能

### 现状评估

先回答三个问题：

```
1. 核心功能的用户满意度达标了吗？
2. 核心流程的完成率达标了吗？
3. 用户留存率达标了吗？
```

### 决策逻辑

```
三个问题的答案：
├─ 任一项未达标
│   └─ 优先优化现有功能
│       理由：核心没做好就加新功能 = 在漏水的桶上再开口
│
└─ 全部达标
    └─ 可以考虑新功能
        但仍需评估：新功能是否服务于产品的核心价值？
```

### 为什么"核心优先"？

- 用户不会因为你有100个功能而留下来
- 但会因为1个核心功能好用而留下来
- 新功能的获客成本 > 核心优化的留存价值（大多数情况）

### 例外情况

| 例外 | 说明 |
|------|------|
| 战略性必须有 | 政策要求、核心客户硬性需求 |
| 成本极低可快速验证 | 1-2天能做完并上线验证 |
| 优化已进入边际递减 | 投入很多但提升很小，说明核心已经足够好 |
| 竞争逼迫 | 竞品的新功能正在分流我们的用户 |

### 判断是否进入"边际递减"

```
最近3轮迭代对核心功能的优化：
├─ 每轮都有显著提升 → 还没到边际递减，继续优化
├─ 提升幅度逐轮缩小 → 接近边际递减，可以考虑新功能
└─ 几乎看不到提升 → 已经进入边际递减，转向新功能或重新定义核心
```

## 场景三：何时止损？

### 5条止损信号

一个功能/方向做了一段时间效果不好，出现**2个以上信号**就该认真考虑止损：

| # | 止损信号 | 检测方式 |
|---|---------|---------|
| 1 | 连续3轮优化后核心指标仍未达最低预期 | 对比3轮数据趋势 |
| 2 | 用户调研显示需求本身不强烈（当初误判） | 用户访谈/调研结果 |
| 3 | 实现成本远超预期，且看不到收敛的趋势 | 实际投入 vs 预估对比 |
| 4 | 市场/政策环境已经变化，方向不再有价值 | 环境扫描 |
| 5 | 团队投入大量精力但士气持续走低 | 团队状态观察（信号不是原因） |

### 止损不代表失败

- "我们花了2个月验证了这个方向不可行" → 这本身是有价值的结论
- 比"花了12个月坚持一个错误方向"好得多
- 关键是把学到的东西记录下来

### 止损流程

```
Step 1: 整理数据和事实
  → 不要靠感觉决定止损，用数据说话

Step 2: 与相关人员沟通
  → 确保信息对称，不是突然宣布

Step 3: 做一次止损复盘
  → 重点是"学到了什么"，不是"谁的错"
  → 记录：为什么不可行？哪些投入可以复用？

Step 4: 释放资源
  → 释放到更有价值的方向

Step 5: 归档
  → 不要删除——未来可能有参考价值
  → 记录完整的决策链路
```

### 止损的心理障碍

| 障碍 | 表现 | 突破方式 |
|------|------|---------|
| 沉没成本 | "已经投了这么多，放弃太可惜" | 问"从现在起继续投入值不值得"，忽略过去 |
| 面子问题 | "是我推的方向，止损等于承认我错了" | 止损是能力，不是失败 |
| 乐观偏差 | "再给一轮机会就好了" | 设定明确的止损条件，提前约定 |
| 信息不对称 | "也许我们没看到的地方在变好" | 用数据和事实评估，不靠猜测 |

## 场景四：技术债务处理

### 什么是技术债务（给非技术人员的解释）

- 为了赶进度走了捷径，导致代码/架构不够好
- 短期看省了时间，长期看越来越拖慢开发速度
- 就像信用卡——先爽后痛，利息越滚越多

### 产品经理需要知道的

1. **技术债不是技术团队自己的事** — 它直接影响产品迭代速度
2. **适度的技术债是正常的** — 不可能每次都做到完美
3. **问题是不能只借不还** — 必须定期"还债"

### 技术债处理原则

```
1. 每轮迭代预留 15-20% 资源给技术优化（和团队约定好）

2. 技术团队说"这个需求很难做因为历史代码有问题"
   → 不要无视
   → 和技术负责人一起评估：是否需要先重构？

3. 重大重构 → 当作独立项目规划，不要想着"顺便做"

4. 规划迭代速度时，把技术债影响算进去
   → 系统越来越难改，迭代速度必然下降
   → 这不是团队变懒了，是债务在收利息
```

### 技术债分级处理

| 级别 | 特征 | 处理方式 |
|------|------|---------|
| 紧急 | 正在影响线上稳定性 | 当A类止血处理 |
| 重要 | 显著拖慢开发速度 | 安排在最近1-2轮迭代中 |
| 一般 | 有一定影响但可控 | 日常15-20%资源中处理 |
| 可接受 | 暂时不影响 | 记录，定期评估 |

### 技术债的可视化

让技术债变得"可见"是管理它的前提：

- 维护"技术债清单"，定期更新
- 每条技术债标注：影响范围、紧急程度、预估修复成本
- 让产品和技术共同看到这份清单
- 在迭代回顾中讨论技术债趋势（是在减少还是在增加？）
