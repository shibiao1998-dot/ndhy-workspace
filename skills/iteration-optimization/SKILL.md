# iteration-optimization -- 迭代优化执行工具箱

在"发布→反馈→改进→再发布"的循环中驱动产品持续进化，确保每一轮迭代都比上一轮更接近用户的真实需求。

> 迭代不是"做更多功能"，而是"让产品更好地解决问题"。不迭代也是一种选择——如果现状足够好，不要为了迭代而迭代。

## 五阶段迭代循环速查

```
┌──→ ① 审视现状 ──→ ② 确定方向 ──→ ③ 设计方案 ─┐
│                                                │
│    ⑤ 复盘学习 ←── ④ 执行验证 ←─────────────────┘
│         │
└─────────┘（进入下一轮）
```

| 阶段 | 核心问题 | 关键动作 | 输出 |
|------|---------|---------|------|
| ① 审视现状 | 我们在哪？ | 回顾上轮数据、收集反馈、扫描环境变化、定位瓶颈 | 现状诊断报告 |
| ② 确定方向 | 我们去哪？ | 锁定1个核心目标、定义成功标准、明确排除项 | 迭代目标声明 |
| ③ 设计方案 | 怎么去？ | 方案对比、MVP拆解、定验收标准、拆成≤1周的交付块 | 迭代方案 |
| ④ 执行验证 | 走起来 | 按优先级逐个交付、小块验证、及时止损、记录过程 | 交付物+执行记录 |
| ⑤ 复盘学习 | 学到了什么？ | 目标达成度、继续/停止/开始清单、经验沉淀 | 复盘记录 |

→ 每阶段详细输入/输出/关键动作 [references/iteration-cycle.md](references/iteration-cycle.md)

## 优先级框架速查

### ABCD 四级分类

| 级别 | 类型 | 定义 | 原则 |
|------|------|------|------|
| **A** | 🩸 止血 | 不修就持续伤害用户/业务（崩溃、数据丢失、严重bug） | 立刻做，放下一切 |
| **B** | 💪 强化核心 | 让产品核心价值变得更强（效率提升、流程简化） | 优先做，迭代主战场 |
| **C** | 🌱 拓展边界 | 增加新能力或覆盖新场景（新功能、新角色支持） | 谨慎做，核心够好再拓展 |
| **D** | ✨ 锦上添花 | 有了更好没有不影响核心（动效美化、边缘优化） | 资源富余时做或顺便做 |

**常见错误**：D当B做（花资源打磨边缘）、总想做C（"再加一个功能"）、A压着不做（"先做完这版再修"）

### ICE 评分法

| 维度 | 高(3) | 中(2) | 低(1) |
|------|-------|-------|-------|
| **I**mpact 影响力 | 直接影响核心指标 | 间接影响/次要指标 | 影响有限 |
| **C**onfidence 信心 | 有数据/案例支持 | 合理推测无直接证据 | 主要靠直觉 |
| **E**ase 容易度 | 1-2天搞定 | 1-2周搞定 | 大量资源/跨迭代 |

**ICE分 = I × C × E**（分数是辅助工具不是圣旨，直觉不对就停下想想为什么）

### 砍需求三原则

1. **产能×0.7**：每轮需求量 = 团队产能 × 0.7（留30%缓冲）
2. **从底部砍**：超出产能从底部砍，不要"每个都压缩一点"
3. **警惕"顺手做"**：10个"顺手做"= 1个大需求，砍掉的放"待办池"下轮重评

→ 详细方法与案例 [references/prioritization.md](references/prioritization.md)

## 迭代深度速查

| 深度 | 适用场景 | 覆盖阶段 | 输出物 |
|------|----------|----------|--------|
| **快速迭代** | 小功能优化 / 单点改进 / bug修复批次 | ①②③ 重点 + ④⑤ 简要 | 迭代目标 + 轻量方案 |
| **标准迭代** | 常规版本迭代 / 重要功能改版 | 完整五阶段 | 完整迭代交付包 |
| **深度迭代** | 产品大版本 / 方向性调整 / MVP验证 | 完整五阶段 + 多方案对比 + 信号深挖 | 完整交付包 + 决策记录 + 信号分析 |

判断不清时按标准迭代执行。

## 六条铁律

```
铁律1：每轮迭代只有一个核心目标
  → "提升留存+加新功能+优化性能+改UI" = 没有目标
  → "首周留存从40%提升到55%" = 有目标
  → 可以有次要任务，但必须服务于或至少不妨碍核心目标

铁律2：不发布就不算完成
  → 开发分支里的完美功能 = 价值为零
  → 灰度、小范围试点都算发布——不需要一次全量推

铁律3：数据验证不是可选项
  → 每个迭代目标必须有可量化的验收标准
  → "体验优化"不是验收标准，"完成率从60%→75%"才是
  → 无法量化时至少做定性验证（用户访谈）

铁律4：范围一旦锁定，轻易不加需求
  → 新增必须"换入换出"——加一个拿掉一个
  → 除非A类止血，否则新需求排到下一轮
  → 保护迭代范围 = 保护交付能力

铁律5：迭代节奏要稳定
  → 推荐2周一轮（教育产品可按教学节奏调整）
  → 稳定节奏让团队形成节拍感，效率自然提升
  → 可以有例外，但例外不能成为常态

铁律6：每轮必须复盘
  → 不复盘 = 不学习 = 原地踏步
  → 时间控制1小时以内，输出的改进项下轮跟踪
  → 连续3轮复盘同一问题 → 升级处理
```

## 决策场景速查

### 做 vs 不做

```
这个改进解决的是真实问题还是假设问题？
├─ 假设问题 → 先做最小验证
└─ 真实问题
    ├─ 影响多少用户？ → <5%低优 / 5-30%中优评估ROI / >30%高优
    ├─ 不做后果？ → 用户流失→尽快做 / 体验差但不走→排期 / 几乎无影响→可不做
    └─ 做的成本？ → 几天→倾向做 / 几周→仔细权衡 / 几个月→充分论证才做
```

### 优化现有 vs 新功能

```
核心功能满意度达标？ + 核心流程完成率达标？ + 留存率达标？
├─ 任一未达标 → 优先优化现有（漏水的桶别再开口）
└─ 全部达标 → 可以考虑新功能

例外：战略性必须有 / 成本极低可快速验证 / 优化已进入边际递减
```

### 止损信号（出现2个以上认真考虑）

1. 连续3轮优化后核心指标仍未达最低预期
2. 用户调研显示需求本身不强烈（当初误判）
3. 实现成本远超预期且看不到收敛
4. 市场/政策环境已变，方向不再有价值
5. 团队投入大量精力但士气持续走低

### 技术债处理

- 每轮预留 **15-20%** 资源给技术优化
- 技术团队反映"历史代码有问题" → 不无视，一起评估是否先重构
- 重大重构当独立项目规划，不要"顺便做"

→ 决策树详解与案例 [references/decision-scenarios.md](references/decision-scenarios.md)

## 反模式速查

| # | 反模式 | 表现 | 正确做法 |
|---|--------|------|---------|
| 1 | **需求蔓延** | 范围不断膨胀，"再加一个小需求" | 严格换入换出，对"小需求"保持警惕 |
| 2 | **伪迭代** | 做很多事但产品没有实质进步 | 每轮问"做完后用户感知到什么变化？" |
| 3 | **速度焦虑** | 追求更快更多，忽视有效性 | 区分"有效速度"和"表面速度" |
| 4 | **完美主义** | 迟迟不发布，"还差一点" | 设"足够好"标准而非"完美"标准 |
| 5 | **复盘缺失** | 不回头看直接冲下一轮 | 复盘是必做环节不是可选 |
| 6 | **数据失明** | 上线不看数据，全凭感觉 | 上线前定好看什么数据、什么时候看 |
| 7 | **反馈断路** | 用户声音传不到产品决策中 | 建立固定反馈收集传递机制 |
| 8 | **技术债累积** | 只做新功能不还债 | 每轮15-20%资源给技术优化 |
| 9 | **方向摇摆** | 今天做A明天改B | 路线图稳定大方向，执行期不轻易改 |
| 10 | **闭门造车** | 不做调研就决定方向 | 每轮至少1个方向来自用户反馈 |

## 迭代检查清单

### 迭代启动前

| # | 检查项 | 通过条件 |
|---|--------|---------|
| 1 | 核心目标明确 | 只有1个核心目标，可一句话说清 |
| 2 | 成功标准可量化 | 有具体数字或可验证的定性标准 |
| 3 | 排除项已明确 | 本轮不做什么写清楚了 |
| 4 | 需求已分级 | 全部需求经过ABCD分类+ICE评分 |
| 5 | 范围已锁定 | 需求量 ≤ 产能×0.7 |
| 6 | 验收标准已定 | 每个需求都有"什么样算做完、做好" |
| 7 | 上轮复盘项已跟踪 | 上轮改进项本轮有对应动作 |

### 迭代执行中

| # | 检查项 | 通过条件 |
|---|--------|---------|
| 8 | 按优先级交付 | A→B→C→D顺序，不跳级 |
| 9 | 小块验证 | 每完成一块做验证，不攒到最后 |
| 10 | 范围受控 | 新增需求走换入换出流程 |
| 11 | 过程有记录 | 执行中的问题和发现有记录 |

### 迭代结束后

| # | 检查项 | 通过条件 |
|---|--------|---------|
| 12 | 已发布 | 交付物已上线（灰度也算） |
| 13 | 数据已验证 | 看过效果数据，对照成功标准 |
| 14 | 复盘已完成 | 有复盘记录，含继续/停止/开始清单 |
| 15 | 经验已沉淀 | 可复用的方法论已记录 |

## 协作接口

### 上游输入

| 来源 | 接收内容 | 使用方式 |
|------|---------|---------|
| 产品定义专家 | 产品路线图、功能范围、成功标准 | 框定迭代的战略方向 |

### 下游输出

| 接收方 | 传递内容 | 约束关系 |

## 触发时首要动作

无论如何触发，第一步永远是：

> "当前产品处于什么阶段？上轮迭代效果如何？本轮核心要解决什么问题？资源和时间约束是什么？"

→ 判断迭代深度 → 消化输入 → 按五阶段循环执行

**教育行业项目**：额外读取 [references/education-iteration.md](references/education-iteration.md) 了解教学节奏和行业特殊约束。
