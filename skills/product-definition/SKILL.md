# product-definition — 产品定义执行工具箱

将战略定位转化为具体可执行的产品蓝图。回答"这条路上，我们造的是一辆什么车、拉什么人、跑多快、到哪里停"。

## 8 步标准流程

```
Step 1：消化输入
  │  完整理解战略定位文档，回溯 Phase 1 关键洞察
  │  → 提取：方向、客群、差异化、边界
  │  → 信息不足时，有权要求上游补充
  │
Step 2：概念定义
  │  → 回答五个核心问题：是什么？给谁？什么场景？干什么？得到什么？
  │  → 产出【产品概念卡】
  │  → 通过"出租车司机测试"：普通人听完能复述
  │
Step 3：场景拆解
  │  → 识别 3-5 个核心场景
  │  → 每个场景展开三层：核心场景 → 场景故事 → 场景边界
  │  → 遵守场景三铁律（见下方速查）
  │  → 详细方法 → references/analysis-frameworks.md §1
  │
Step 4：功能范围界定
  │  → 从场景推导功能，不凭空想象
  │  → 按三层模型分级：核心层 / 重要层 / 扩展层
  │  → 每个功能过四检验（见下方速查）
  │  → 核心层不超过 5 个功能
  │  → 详细方法 → references/analysis-frameworks.md §2
  │
Step 5：规则设计
  │  → 定义四类核心规则：准入 / 流转 / 限制 / 异常
  │  → 规则要少、透明、不矛盾
  │  → 每条规则考虑"被违反了怎么办"
  │  → 详细方法 → references/analysis-frameworks.md §3
  │
Step 6：成功标准
  │  → 定义两层标准：可用标准（及格线）+ 成功标准（优秀线）
  │  → 分短/中/长期，必须可量化或可观测
  │  → 区分虚荣指标和真实指标
  │  → 详细方法 → references/analysis-frameworks.md §4
  │
Step 7：边界确认
  │  → 明确三组关系：与现有产品 / 与外部系统 / 与未来演进
  │  → 详细方法 → references/analysis-frameworks.md §5
  │
Step 8：输出产品定义文档
     → 按模板组织 6 章节文档 → references/templates.md
     → 交付前跑质量自检清单 → references/checklists.md
     → 教育类产品额外读 → references/education-product.md
```

## 定义深度速查

| 深度 | 适用场景 | 覆盖范围 | 输出物 |
|------|----------|----------|--------|
| **轻量定义** | 已有产品的功能迭代 | 概念确认 + 功能范围 + 成功标准 | 产品定义文档（轻量版） |
| **标准定义** | 新产品模块或重要改版 | 完整 8 步流程 | 产品定义文档（标准版） |
| **深度定义** | 全新产品从 0 到 1 | 完整流程 + 多轮场景验证 + 边界深度展开 | 产品定义文档（完整版） |

判断不清时向上确认。

## 6 大能力域速查

### 域 1：产品概念定义

**核心问题**：用一段话说清楚这个产品是什么。

五问速查：

| # | 问题 | 要求 |
|---|------|------|
| 1 | **它是什么？** | 具体产品形态，不是抽象概念 |
| 2 | **给谁用？** | 细化到"拿着产品的那个人"：主用户/次要用户/利益相关者 |
| 3 | **什么情况下用？** | 触发时机、使用场所、使用时长 |
| 4 | **用它干什么？** | 最重要的 1-3 件事 |
| 5 | **用完得到什么？** | 用户付出后的具体收获 |

**关键原则**：
- 产出物 = 产品概念卡（模板 → [references/templates.md](references/templates.md) §产品概念卡）
- 质量标准 = "出租车司机测试"：不了解项目的人看完能说出"哦，就是一个___"

### 域 2：场景拆解

**核心问题**：用户到底怎么用这个产品？

三层拆解：

| 层级 | 内容 | 数量要求 |
|------|------|----------|
| **第一层：核心场景** | 最重要的使用情境，覆盖主线任务 | 3-5 个 |
| **第二层：场景故事** | 谁/何时/何地/为何/做什么/得到什么 | 每个核心场景展开 |
| **第三层：场景边界** | 起点/终点/排除项 | 每个场景明确 |

详细方法 → [references/analysis-frameworks.md](references/analysis-frameworks.md) §1

### 域 3：功能范围界定

**核心问题**：产品包含哪些功能，优先做什么？

三层模型：

| 层级 | 定义 | 数量限制 |
|------|------|----------|
| **核心层 Must Have** | 没有就不成立，支撑核心场景，MVP 必含 | ≤ 5 个 |
| **重要层 Should Have** | 有了明显更好，优化体验 | 第二版加入 |
| **扩展层 Could Have** | 锦上添花，差异化加分 | 资源充足时 |

详细方法 → [references/analysis-frameworks.md](references/analysis-frameworks.md) §2

### 域 4：产品规则设计

**核心问题**：产品里什么能做、什么不能做、做了会发生什么？

四类规则：

| 类型 | 回答的问题 | 示例 |
|------|-----------|------|
| **准入规则** | 谁能用？怎么进来？ | 注册条件、角色权限 |
| **流转规则** | 内容/数据怎么流动？ | 任务流转、状态变更 |
| **限制规则** | 使用上限在哪？ | 次数限制、容量限制 |
| **异常规则** | 出了意外怎么办？ | 超时处理、冲突解决 |

详细方法 → [references/analysis-frameworks.md](references/analysis-frameworks.md) §3

### 域 5：成功标准定义

**核心问题**：做出来后怎么判断好不好？

两层标准：

| 层级 | 定义 | 时间维度 |
|------|------|----------|
| **可用标准（及格线）** | 核心场景走通、关键任务完成 | 上线门槛 |
| **成功标准（优秀线）** | 用户持续用、关键指标达标 | 短期 1 月 / 中期 3 月 / 长期 6 月+ |

详细方法 → [references/analysis-frameworks.md](references/analysis-frameworks.md) §4

### 域 6：产品边界管理

**核心问题**：这个产品和外部世界的关系是什么？

三组关系：

| 关系 | 需明确的问题 |
|------|-------------|
| **与现有产品** | 替代？互补？数据打通？账号共用？功能重叠处理？迁移路径？ |
| **与外部系统** | 对接什么系统？对接深度？ |
| **与未来演进** | 本版是终态还是过渡？预留什么？刻意不预留什么？ |

详细方法 → [references/analysis-frameworks.md](references/analysis-frameworks.md) §5

## 4 条铁律

| # | 铁律 | 核心要求 | 违反后果 |
|---|------|----------|----------|
| 1 | **每个功能必须对应场景** | 说不出"谁在什么场景下用来干什么"的功能不能进范围 | 功能堆砌、资源浪费 |
| 2 | **产品概念必须一句话说清** | 说不清 = 没定义好 | 团队理解不一致 |
| 3 | **不能超出战略边界** | 战略定位说"不做"的不能偷偷加回来 | 战略失焦 |
| 4 | **成功标准必须在定义阶段确定** | 不能等做完再想怎么算成功 | 无法验收 |

## 功能四检验速查

每个功能逐一过四关，任何一关不通过则调整层级或移除：

| # | 检验问题 | 不通过 = |
|---|---------|---------|
| Q1 | 这个功能支撑哪个场景？ | 答不上来 → **不该做** |
| Q2 | 没有它用户能完成核心任务吗？ | 能完成 → **不是核心层** |
| Q3 | 在战略定位边界内吗？ | 不在 → **不能做** |
| Q4 | 实现成本和价值匹配吗？ | 不匹配 → **暂时不做** |

## 场景三铁律速查

| # | 铁律 | ❌ 错误 | ✅ 正确 |
|---|------|--------|--------|
| 1 | **真实情境，不是抽象流程** | "教师进行备课" | "周三晚上 8 点，李老师在家用笔记本准备明天 Unit 3 阅读课" |
| 2 | **必须有触发条件** | 无触发 = 可能是伪需求 | "因为明天要上新课，需要准备导入活动和练习题" |
| 3 | **必须有完成标志** | 无终点 = 无法衡量 | "备课完成：教案已保存，课件已导出到 U 盘" |

## 6 个质量问题（❌/✅ 对比）

❌ **概念模糊**："智慧教育解决方案"
✅ 通过出租车司机测试："一个给初中英语老师用的 AI 备课网页工具"

❌ **场景虚构**：想象中的完美场景，无真实依据
✅ 追问"见过真实用户这样做吗"，追溯用户研究数据

❌ **功能堆砌**：核心层塞了十几个功能
✅ 核心层 ≤ 5 个，每个过四检验，多余的降级

❌ **规则缺失**：只定义正常流程，没考虑异常
✅ 每个核心流程必须回答"出错了怎么办"

❌ **指标虚荣**：成功标准全是注册量、下载量
✅ 必须包含使用深度指标（活跃率、任务完成率、留存率）

❌ **边界模糊**：和现有产品功能重叠说不清
✅ 明确列出重叠部分的处理方式（替代/共存/打通）

详细防范措施 → [references/checklists.md](references/checklists.md)

## 协作接口

### 上游输入

| 来源 | 接收内容 | 使用方式 |
|------|---------|---------|
| 战略定位专家 | 战略定位文档（方向、客群、差异化、边界） | **主要输入**，所有定义在定位框架内展开 |
| 需求分析师 | 需求优先级排序、用户故事 | 场景拆解和功能推导的事实依据 |
| 用户研究专家 | 用户画像、行为模式、痛点细节 | 场景故事的真实性校验 |

→ 信息不足时有权要求上游补充

### 下游输出

| 接收方 | 传递内容 | 约束关系 |
|--------|---------|---------|
| 产品架构师 | 功能范围、产品规则、边界说明 | 架构必须支撑所有核心层功能 |
| 交互设计专家 | 场景定义、核心任务、用户流程 | 交互围绕场景展开，核心任务路径最短 |
| 项目管理专家 | MVP 功能清单、功能优先级、成功标准 | 排期以功能分层为依据，验收以成功标准为依据 |

### 回调机制

| 触发条件 | 动作 |
|---------|------|
| 架构师反馈功能技术不可行或成本过高 | 重新评估功能层级，必要时降级或替换 |
| 交互设计发现场景与真实使用脱节 | 修正场景定义，追溯用户研究数据 |
| 战略定位发生调整 | 重新审视产品概念和功能范围 |
| 开发中范围蔓延 | 用功能范围表和边界说明做"挡板"，溢出需求推回评估 |

## 触发时首要动作

无论如何触发，第一步永远是：

> "战略定位告诉了我们什么？方向是什么、服务谁、差异化在哪、边界在哪？"

→ 判断定义深度 → 消化战略定位输入 → 按 8 步流程执行
