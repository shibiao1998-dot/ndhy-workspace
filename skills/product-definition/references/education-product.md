# 教育行业产品定义特别指南

教育产品有其独特的复杂性。本指南针对教育行业产品定义中的 5 条特别规则，提供详细的操作指导。

---

## 规则 1：场景要贴合教育日历

### 核心要求

教育行业的使用场景与学期节奏强绑定，每个场景必须标注"在学期什么阶段发生"。

### 教育日历分段

```
学期初（开学前 2 周 到 开学后 2 周）
  教师：分班、排课、制定学期计划、了解新学生
  学生：适应新老师和课程、建立学习节奏
  管理者：检查开学准备、分配资源

学期中（第 3 周 到 期中）
  教师：日常备课、授课、作业批改、阶段测试
  学生：日常学习、做作业、准备测试
  管理者：教学巡查、教研活动

期中（期中考试前后各 1 周）
  教师：出题、阅卷、成绩分析、家长沟通
  学生：复习、考试
  管理者：成绩汇总、质量分析

学期后半（期中后 到 期末前 2 周）
  教师：教学调整、查缺补漏、准备期末
  学生：日常学习、纠错
  管理者：教学督导

期末（期末考试前后）
  教师：复习规划、出题、阅卷、写评语、写总结
  学生：复习、考试
  管理者：成绩分析、学期总结

寒暑假
  教师：培训、下学期准备、教研
  学生：假期作业、预习
  管理者：规划、采购、人事
```

### 场景标注格式

```
场景名称：___
教育日历阶段：学期初 / 学期中 / 期中 / 学期后半 / 期末 / 寒暑假
使用频率变化：
  - 高峰期：___ （使用频率上升）
  - 低谷期：___ （使用频率下降）
  - 不适用期：___ （完全不用）
```

### 常见错误

错误：假设产品全年均匀使用
正确：标注高峰/低谷/不适用期，针对高峰期做性能预案

错误：忽略"开学第一周"和"期末最后一周"的极端压力场景
正确：这两个时间点往往是使用强度最高的，专门评估

## 规则 2：功能设计要考虑"最差使用环境"

### 核心要求

核心功能在最差环境下也必须能用。中国教育市场，县镇学校是大盘。

### 最差环境参数

| 维度 | 最差情况 | 设计要求 |
|------|---------|---------|
| **网速** | 2G/3G 移动网络、校园 WiFi 带宽有限 | 核心功能在 500KB/s 下可用 |
| **设备** | 3-5 年前的安卓手机、老旧 Windows 电脑 | 兼容 Android 8+、Windows 7+、Chrome 70+ |
| **屏幕** | 5 寸手机屏、1366x768 笔记本 | 响应式设计，小屏可用 |
| **数字素养** | 教师刚会用微信发文件的水平 | 操作步骤不超过 3 步完成核心任务 |
| **使用时间** | 课间 10 分钟、下班后碎片时间 | 核心任务 5 分钟内完成 |

### 功能分级设计

```
A级（最差环境也能用）：核心层功能
  离线或弱网可用
  低配设备流畅
  操作极简

B级（一般环境能用）：重要层功能
  需要稳定网络
  中等设备即可
  操作相对简单

C级（好环境才好用）：扩展层功能
  需要好网络、好设备
  操作可以复杂
  体验有额外加分
```

### 设计检查

- [ ] 每个核心层功能标注了"最差环境可用性"
- [ ] 弱网降级方案已设计（非直接报错）
- [ ] 核心任务操作步骤不超过 3 步
- [ ] 关键页面在 3 秒内可交互（500KB/s 网络下）

## 规则 3：产品规则必须遵循教育伦理

### 核心要求

教育产品影响的是人（尤其是未成年人），规则设计必须有伦理底线。

### 三条伦理红线

#### 红线 1：AI 不能替代教师判断权

```
允许：AI 提供建议、推荐、初稿
禁止：AI 直接做出教育评价、分班决策、处分建议

具体要求：
  AI 生成的内容必须经教师确认后才能发给学生或家长
  AI 的评价性输出必须标注"仅供参考，请教师判断"
  教师必须能修改 AI 的所有输出
  不能有"一键发送 AI 结果"的功能设计
```

#### 红线 2：学生数据严格边界

```
允许：
  学习行为数据用于改进教学
  聚合后的班级或年级数据用于教学分析
  教师查看自己班级的学生数据

禁止：
  学生数据用于商业推荐或广告
  向家长暴露学生在其他维度的数据（如社交行为）
  跨学校或跨区域的学生个人数据共享
  用学生数据做排名并公开
  未经同意向第三方提供学生数据
```

#### 红线 3：不加剧教育焦虑

```
允许：
  展示学生进步趋势（纵向对比）
  诊断学习薄弱点并给出改进建议
  让教师了解整体学情

禁止：
  学生之间的横向排名
  "你的孩子落后同龄人 X%"类提示
  刺激焦虑的推送（"别人在学你在玩"）
  过度量化学生表现（精确到小数点的分数排名）
  用对比制造焦虑来驱动使用
```

### 规则审查清单

- [ ] AI 输出需教师确认后才能到达学生或家长
- [ ] 学生数据使用范围有明确限制
- [ ] 无横向排名功能
- [ ] 无焦虑驱动的推送或提示设计
- [ ] 数据存储符合个人信息保护法和儿童个人信息网络保护规定

## 规则 4：成功标准要包含教育效果指标

### 核心要求

教育产品不只看使用数据，要有"对教学和学习是否有实质帮助"的衡量。

### 教育效果指标类型

| 类型 | 指标示例 | 观测方式 |
|------|---------|---------|
| **教学效率** | 备课时间变化、作业批改时间变化 | 前后对比 |
| **教学质量** | 教案质量评分、课堂互动次数 | 教研评估 + 数据 |
| **学习效果** | 知识点掌握率、作业正确率变化 | 学情数据 |
| **教师满意度** | NPS、主动推荐意愿 | 问卷调研 |
| **持续使用** | 教师连续使用周数、学期内留存 | 行为数据 |

### 教育效果指标设计模板

```
指标名称：___
教育意义：这个指标为什么能说明"对教学或学习有帮助"？
计算方式：___
基准值：使用前 ___（如何获取基准）
目标值：使用后 ___
观测周期：___（注意：教育效果见效慢，至少一个学期）
数据来源：___
注意事项：___
```

### 常见错误

错误：只看使用数据（DAU、时长），不看教育效果
正确：使用数据是过程指标，教育效果是结果指标，两者都要

错误：期望上线 1 个月就看到教育效果
正确：教育效果至少一个学期才能初步观测，短期看过程指标

错误：用学生考试成绩作为唯一教育效果指标
正确：考试成绩受太多因素影响，应结合教学行为变化综合评估

## 规则 5：多角色场景必须分开定义

### 核心要求

教育产品通常涉及教师、学生、家长、管理者多个角色。同一功能在不同角色视角下是不同场景，必须分别定义。

### 多角色场景拆解方法

```
功能：___

教师视角：
  场景名称：___
  触发条件：___
  操作流程：___
  得到什么：___
  场景边界：___

学生视角：
  场景名称：___
  触发条件：___
  操作流程：___
  得到什么：___
  场景边界：___

家长视角：
  场景名称：___
  触发条件：___
  操作流程：___
  得到什么：___
  场景边界：___

管理者视角：
  场景名称：___
  触发条件：___
  操作流程：___
  得到什么：___
  场景边界：___
```

### 角色交叉点检查

不同角色的场景会在某些节点交叉，必须明确交叉处的规则：

| 交叉点 | 需明确的问题 |
|--------|-------------|
| 教师到学生 | 教师操作后，学生什么时候看到？看到什么？ |
| 学生到教师 | 学生提交后，教师怎么收到通知？以什么形式？ |
| 教师到家长 | 什么信息同步给家长？家长能做什么操作？ |
| 管理者到教师 | 管理者能看到什么？不能看到什么？ |

### 常见错误

错误：只从教师视角定义场景，学生或家长视角缺失
正确：主用户的场景定义完后，必须检查其他角色的视角

错误：不同角色共用一个场景描述
正确：即使功能相同，不同角色的触发条件、操作流程、获得价值都不同，分开写

错误：角色之间的信息流向不清
正确：画出角色交叉点，明确"谁的操作会影响谁，以什么方式"

## 教育产品定义附加检查清单

在总体质量检查清单基础上，教育类产品额外检查：

- [ ] 每个场景标注了教育日历阶段
- [ ] 核心功能在最差环境下可用
- [ ] AI 输出需教师确认才到达学生或家长
- [ ] 学生数据使用范围有明确限制
- [ ] 无横向排名和焦虑驱动设计
- [ ] 成功标准包含教育效果指标
- [ ] 多角色场景分别定义
- [ ] 角色交叉点规则明确
- [ ] 符合个人信息保护法和儿童个人信息网络保护规定
