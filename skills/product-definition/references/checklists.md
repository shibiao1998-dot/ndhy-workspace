# 产品定义 — 质量检查清单

## 一、总体质量检查清单

交付产品定义文档前，逐项检查：

### 产品概念（域 1）

- [ ] 产品概念卡完整填写
- [ ] 五个核心问题全部回答
- [ ] 一句话定义具体、清晰（非大词堆砌）
- [ ] 通过"出租车司机测试"——不了解项目的人能复述
- [ ] 核心用户细化到"拿着产品的那个人"，非泛指

### 场景定义（域 2）

- [ ] 核心场景 3-5 个，数量合理
- [ ] 每个场景有完整的场景故事（6W）
- [ ] 每个场景有明确的边界（起点/终点/排除项）
- [ ] 场景三铁律全部通过（真实/有触发/有完成标志）
- [ ] 场景来源于用户研究数据，非凭空想象

### 功能范围（域 3）

- [ ] 核心层不超过 5 个功能
- [ ] 每个功能通过四检验
- [ ] 每个功能对应至少一个场景
- [ ] 三层分级清晰（核心/重要/扩展）
- [ ] "不做"清单非空
- [ ] MVP 功能清单明确

### 产品规则（域 4）

- [ ] 四类规则全覆盖（准入/流转/限制/异常）
- [ ] 每条规则有存在理由
- [ ] 规则之间无矛盾
- [ ] 每条规则回答了"被违反了怎么办"
- [ ] 规则对用户透明

### 成功标准（域 5）

- [ ] 两层标准完整（可用标准 + 成功标准）
- [ ] 所有指标可量化或可观测
- [ ] 分短中长期
- [ ] 包含使用深度指标（非仅注册或下载量）
- [ ] 指标定义精确（含计算方式）

### 边界说明（域 6）

- [ ] 三组关系全覆盖（现有产品/外部系统/未来演进）
- [ ] 功能重叠处理方式明确
- [ ] 外部依赖风险已评估
- [ ] 本版本定位清晰（终态/过渡）

### 总体一致性

- [ ] 产品概念与战略定位一致
- [ ] 功能范围未超出战略边界
- [ ] 场景与用户研究数据吻合
- [ ] 成功标准与产品概念的价值承诺对应
- [ ] 文档内部无矛盾

## 二、6 个质量问题详细防范

### 问题 1：概念模糊

**表现**：看完产品概念还不知道这是什么东西。充斥"智慧""赋能""一站式"等大词。

**根本原因**：定义者试图用抽象词汇覆盖更大范围，实际是没想清楚。

**防范措施**：
1. **"出租车司机测试"** — 把一句话定义说给不了解项目的人听，对方能说出"哦，就是一个___"吗？不能则重写
2. **禁用词清单** — 以下词汇不能出现在一句话定义中：
   - 解决方案、平台化、赋能、生态、智慧、一站式、全方位、综合性
3. **具体化公式** — 产品形态 + 核心用户 + 核心场景 + 核心任务 + 核心价值

错误示例：
> "打造智慧教育一站式解决方案，全方位赋能教师教学。"

正确示例：
> "一个给初中英语老师用的 AI 备课网页工具，帮老师在晚上备课时 15 分钟内生成第二天上课要用的教案和课件。"

### 问题 2：场景虚构

**表现**：场景看起来合理但现实中不存在。往往是"如果用户这样做就好了"的理想化想象。

**根本原因**：跳过用户研究，直接想象用户行为。

**防范措施**：
1. **溯源检验** — 每个场景追问："见过真实用户这样做吗？"如果答案是"没有，但应该会"则属于高风险
2. **数据支撑** — 场景必须能追溯到以下至少一项：
   - 用户访谈记录
   - 行为观察数据
   - 竞品用户真实反馈
   - 行业公认使用模式
3. **反面检验** — 问："如果这个场景不存在，产品会怎样？"如果答案是"没什么影响"则是伪场景

错误示例：
> "教师打开系统，浏览AI推荐的跨学科融合教案，获得灵感后制作创新课件。"
> （实际：大多数教师赶着备明天的课，没时间浏览推荐内容）

正确示例：
> "周三晚上 8 点，李老师在家打开笔记本，要备明天第二节的 Unit 3 阅读课。她选了教材版本和章节，输入'需要导入活动和阅读理解练习'，3 分钟拿到教案初稿。"
> （来源：12 位初中英语教师访谈，8 位提到'晚上在家备第二天的课'）

### 问题 3：功能堆砌

**表现**：核心层塞了十几个功能，说什么都是"必须有"。

**根本原因**：没有做取舍的勇气，或者核心场景没界定清楚导致什么都像是必要的。

**防范措施**：
1. **硬上限** — 核心层不超过 5 个功能，这是红线
2. **四检验逐一过** — 每个功能必须通过 Q1-Q4，不通过的降级或移除
3. **场景推导法** — 功能从场景推导，不从"竞品有什么"推导
4. **减法思维** — 问："如果只能保留 3 个功能，保留哪 3 个？"以此为核心层起点

错误示例：
> 核心层：AI 备课、作业批改、学情分析、家校沟通、课程管理、资源库、考试出题、学生画像、教研协作、数据看板

正确示例：
> 核心层（3 个）：AI 生成教案、AI 生成课件、教案编辑保存
> 重要层：作业出题、学情简报
> 扩展层：教研分享、资源收藏

### 问题 4：规则缺失

**表现**：只定义了正常流程（happy path），没考虑异常情况。

**根本原因**：只考虑了"用户正确使用"的情况，忽略了现实世界的复杂性。

**防范措施**：
1. **异常清单法** — 每个核心流程必须回答以下问题：
   - 网络断了怎么办？
   - 用户操作到一半退出怎么办？
   - 两个用户同时操作同一内容怎么办？
   - 输入数据格式不对怎么办？
   - 第三方服务不可用怎么办？
2. **强制要求** — 四类规则必须全覆盖，异常规则不能为空
3. **违反检验** — 每条规则写完追问"如果被违反了会怎样"

错误示例：
> 规则：教师选择教材版本后，系统生成教案。
> （没说：如果该教材版本不支持呢？如果生成失败呢？如果网络超时呢？）

正确示例：
> 规则：教师选择教材版本后，系统在 30 秒内生成教案。
> 异常处理：
> - 教材版本不支持：提示"暂不支持此版本，预计 X 月上线"并引导选择其他版本
> - 生成超时（>30s）：显示进度条，60s 后提示"生成遇到问题，是否重试"
> - 生成失败：记录错误日志，提示"生成失败，已自动反馈给技术团队"

### 问题 5：指标虚荣

**表现**：成功标准全是注册用户数、下载量、访问量等虚荣指标。

**根本原因**：选了容易看起来漂亮的数字，而非真正反映产品价值的指标。

**防范措施**：
1. **真实指标必含** — 成功标准中必须包含以下至少 2 类：
   - 使用深度：核心功能使用率、任务完成率
   - 留存指标：次日/次周/次月留存率
   - 价值验证：用户主动推荐率（NPS）、付费转化率
2. **虚荣指标标记** — 注册量、下载量等可以记录但必须标注"参考指标"而非"成功标准"
3. **反问检验** — 每个指标追问："这个数字好看，能说明用户真的在用并且觉得有用吗？"

错误示例：
> 成功标准：上线 3 个月注册教师 10 万，日活 5 万

正确示例：
> 成功标准：
> - 短期（1 月）：核心功能（AI 生成教案）周使用率 > 60%
> - 中期（3 月）：教师次周留存率 > 40%，单次备课时间降至 20 分钟内
> - 长期（6 月）：NPS > 30，学校续费率 > 70%

### 问题 6：边界模糊

**表现**：新产品和现有产品功能重叠但说不清楚关系，用户不知道该用哪个。

**根本原因**：回避了"新旧产品关系"这个棘手问题，希望模糊处理。

**防范措施**：
1. **重叠清单** — 逐一列出功能重叠点，每个给出明确处理方式：
   - 新产品承接（旧产品下线该功能）
   - 保留在旧产品（新产品不做）
   - 两边都有但数据打通
2. **用户视角检验** — 问："一个同时使用新旧产品的用户，会困惑吗？"困惑则必须澄清
3. **迁移路径** — 如果是替代关系，必须有明确的迁移时间表和过渡方案

错误示例：
> "新产品是独立产品，与现有教务系统相互补充。"
> （没说：都有'课程管理'功能怎么办？数据互通吗？老师要维护两套？）

正确示例：
> 与教务系统的关系：互补，功能分工如下：
> - 课程基础信息（名称/班级/时间） -- 以教务系统为准，新产品只读同步
> - 备课内容（教案/课件） -- 新产品独有，教务系统无此功能
> - 学生名单 -- 从教务系统同步，新产品不维护
> - 数据互通方式：通过 API 单向同步（教务到新产品），每日凌晨全量 + 变更实时

## 三、功能四检验详解

### Q1：场景对应检验

**核心问题**：这个功能支撑哪个场景？

**操作方法**：
1. 拿出功能范围表
2. 每个功能逐一回答："这个功能在【场景名称】中被【核心用户】用来【做什么】"
3. 如果三个空都填不满则这个功能不该做

**判定标准**：

| 结果 | 判定 |
|------|------|
| 能明确对应 1-2 个核心场景 | 通过，继续 Q2 |
| 只能对应次要场景 | 降级到重要层或扩展层 |
| 对应不了任何场景 | 移除 |

### Q2：必要性检验

**核心问题**：没有这个功能，用户能完成核心任务吗？

**操作方法**：
1. 假设移除这个功能
2. 重新走一遍它对应的核心场景
3. 用户能否用替代方式完成任务？

**判定标准**：

| 结果 | 判定 |
|------|------|
| 完全无法完成核心任务 | 核心层（Must Have） |
| 能完成但体验明显变差 | 重要层（Should Have） |
| 能完成且影响不大 | 扩展层（Could Have）或移除 |

### Q3：边界检验

**核心问题**：这个功能在战略定位划定的边界内吗？

**操作方法**：
1. 翻出战略定位文档的边界章节
2. 对照"做什么""不做什么""暂不做"三个清单
3. 逐一检验

**判定标准**：

| 结果 | 判定 |
|------|------|
| 在"做什么"范围内 | 通过 |
| 在"不做什么"清单中 | 不能做，移除 |
| 在"暂不做"清单中 | 标记为 Deferred |
| 三个清单都没提到 | 需回溯确认，可能是战略定位遗漏或功能越界 |

### Q4：成本价值检验

**核心问题**：实现成本和带来的价值匹配吗？

**操作方法**：
1. 粗估实现成本（人天/技术难度/外部依赖）
2. 评估带来的价值（用户获益 x 受影响用户比例）
3. 比较性价比

**判定标准**：

| 结果 | 判定 |
|------|------|
| 价值明显大于成本 | 通过，保留当前层级 |
| 价值约等于成本 | 降级到扩展层 |
| 价值小于成本 | 暂时不做 |
| 无法评估（不确定性大） | 标记风险，建议先做最小验证 |
