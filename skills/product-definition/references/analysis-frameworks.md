# 产品定义 — 分析框架集

## 第1节 场景三层拆解方法

### 第一层：核心场景识别

从战略定位的客群和差异化出发，识别 3-5 个核心使用场景。

**识别方法**：
1. 列出核心用户的主要任务和目标
2. 每个任务对应一个核心场景
3. 按使用频率乘以重要程度排序，取 Top 3-5

**核心场景描述格式**：

```
场景名称：___
核心用户：___
触发条件：___（用户因为什么打开产品）
核心任务：___（用产品做什么）
完成标志：___（做到什么程度算完成）
使用频率：每天 / 每周 / 每月 / 事件驱动
```

### 第二层：场景故事展开

每个核心场景展开为具体的场景故事，回答 6W：

```
谁（Who）：具体角色，最好有名字
何时（When）：具体时间点或时间段
何地（Where）：具体使用环境
为何（Why）：触发原因，为什么这个时候打开产品
做什么（What）：
  1. 第一步___
  2. 第二步___
  3. ......
得到什么（Result）：离开产品时带走了什么
```

**场景故事质量标准**：
- 读完后能在脑海中"看到"这个画面
- 包含具体的名字、时间、地点（非抽象代号）
- 触发条件真实可信（来源于用户研究数据）

### 第三层：场景边界定义

每个场景必须划定边界：

```
进入条件：从哪里开始算"进入这个场景"

退出条件：到哪里算"完成这个场景"

明确排除：哪些情况不属于这个场景
  - ___
  - ___

与其他场景的关系：
  - 前置场景：___（需要先完成什么）
  - 后续场景：___（完成后通常接着做什么）
  - 并行场景：___（可能同时发生的场景）
```

### 场景三铁律

| 序号 | 铁律 | 检验方法 |
|------|------|---------|
| 1 | **真实情境，不是抽象流程** | 场景中有具体人名、时间、地点吗？能画面化吗？ |
| 2 | **必须有触发条件** | 能回答"用户因为什么才打开产品"吗？说不清则可能是伪需求 |
| 3 | **必须有完成标志** | 能回答"用户做到什么程度算用完了，带走了什么"吗？ |

**对比示例**：

错误写法（抽象流程）：
> "教师使用备课功能，选择教材章节，生成教案。"

正确写法（真实场景）：
> "周三晚上 8 点，李老师在家里打开笔记本电脑，想准备明天第二节课的 Unit 3 阅读部分。她在系统里选了人教版七年级下册第三单元，输入'导入活动 + 阅读理解练习'，3 分钟后拿到一份教案初稿，调整了两处后保存，导出 PPT 到 U 盘。整个过程 15 分钟，比之前手动写省了一个小时。"

## 第2节 功能三层模型

### 分层标准

```
核心层（Must Have）
  判断标准：没有这个功能，产品是否能成立？
  数量限制：不超过 5 个（超过说明定义不够聚焦）
  与 MVP 的关系：核心层 = MVP 功能清单
  来源：直接从核心场景的核心任务推导

重要层（Should Have）
  判断标准：有了明显更好，没有也能凑合用
  与版本的关系：第二版本（V2）加入
  来源：次要场景或核心场景的优化体验

扩展层（Could Have）
  判断标准：锦上添花，差异化加分
  与版本的关系：资源充足时迭代加入
  来源：竞品差异化、长尾需求
```

### 功能推导流程

```
核心场景 -> 场景中的核心任务 -> 完成任务需要什么功能 -> 核心层
次要场景 -> 优化体验的任务 -> 需要什么功能 -> 重要层
差异化需求 -> 加分项 -> 需要什么功能 -> 扩展层
```

**关键原则**：功能从场景推导，不凭空想象。

### 功能四检验

每个功能必须通过四个检验才能确定层级：

**Q1：场景对应检验**
```
这个功能支撑哪个场景？
  能明确回答 -> 通过
  答不上来 -> 不该做，移除
```

**Q2：必要性检验**
```
没有这个功能，用户能完成核心任务吗？
  不能完成 -> 核心层
  能完成但体验差 -> 重要层
  能完成且无明显影响 -> 扩展层或移除
```

**Q3：边界检验**
```
这个功能在战略定位划定的边界内吗？
  在边界内 -> 通过
  在"不做"清单中 -> 不能做，移除
  在"暂不做"清单中 -> 标记为 Deferred
```

**Q4：成本价值检验**
```
实现成本和带来的价值匹配吗？
  价值大于成本 -> 通过
  价值约等于成本 -> 降级到扩展层
  价值小于成本 -> 暂时不做
```

### 功能范围红线

- 核心层不超过 5 个功能，超过说明不够聚焦
- 每个功能必须对应至少一个场景（铁律 1）
- "不做"清单不能为空

## 第3节 四类规则设计

### 规则设计总原则

1. **规则要少** — 每条都有明确的存在理由
2. **规则要透明** — 用户能感知、能理解
3. **规则不矛盾** — 规则之间无冲突
4. **规则有后果** — 每条规则考虑"被违反了怎样"

### 四类规则详细设计

#### 类型 1：准入规则

定义谁能用、怎么进来、需要什么前提条件。

```
设计维度：
  用户身份：什么角色可以访问？
  准入条件：注册/审核/邀请/付费？
  前提要求：需要什么设备/环境/前置操作？
  违反处理：不满足条件时的提示和引导
```

#### 类型 2：流转规则

定义内容、数据、任务在产品里怎么流动。

```
设计维度：
  流转对象：什么在流动？（内容/数据/任务/状态）
  流转路径：从哪到哪？经过什么节点？
  触发条件：什么操作触发流转？
  状态变更：流转过程中状态如何变化？
  回退机制：能否撤销？回退到哪一步？
```

#### 类型 3：限制规则

定义使用限制和上限。

```
设计维度：
  限制对象：限制什么？（次数/容量/时间/频率）
  限制值：具体上限是多少？
  限制理由：为什么要限制？（技术/商业/体验/伦理）
  接近上限提醒：到 80% 时如何提示？
  超限处理：超过后怎么办？
```

#### 类型 4：异常规则

定义出了意外怎么处理。

```
设计维度：
  异常类型：可能出什么错？（网络/数据/操作/并发）
  检测方式：怎么发现异常？
  处理方式：自动重试/降级/提示用户/人工介入？
  用户感知：用户看到什么？能做什么？
  恢复路径：异常解除后如何恢复？
```

### 规则自检清单

- [ ] 每条规则有明确的存在理由
- [ ] 规则对用户透明
- [ ] 规则之间无矛盾
- [ ] 每条规则有"被违反了怎么办"
- [ ] 异常规则覆盖了核心流程的关键节点

## 第4节 成功标准设计

### 两层标准框架

```
第一层：可用标准（上线门槛 = 及格线）
  核心场景能走通吗？
  关键任务能完成吗？
  核心体验可接受吗？（不要求完美，要求可用）
  验证方式：内部测试 + 灰度验证

第二层：成功标准（业务目标 = 优秀线）
  用户愿意持续使用吗？
  关键业务指标达标吗？
  产品价值被验证了吗？
  验证方式：数据埋点 + 用户反馈
```

### 指标设计原则

**1. 必须可量化或可观测**

错误："教师觉得好用"
正确："教师次周留存率 > 40%"

错误："提升教学效率"
正确："教师备课时间从平均 2 小时降至 30 分钟"

**2. 分时间维度**

| 维度 | 时间范围 | 关注重点 |
|------|---------|---------|
| 短期 | 上线后 1 个月 | 基础使用数据：活跃率、核心功能使用率 |
| 中期 | 上线后 3 个月 | 留存和深度：留存率、任务完成率、使用频次 |
| 长期 | 上线后 6 个月+ | 业务价值：NPS、付费转化、口碑传播 |

**3. 区分虚荣指标和真实指标**

| 虚荣指标（参考但不作为成功标准） | 真实指标（作为成功标准） |
|-------------------------------|----------------------|
| 注册用户数 | 周活跃用户数（WAU） |
| 下载量 | 核心功能使用率 |
| 页面浏览量 | 任务完成率 |
| 访问时长（可能是迷路） | 任务完成时间（越短越好） |

### 指标模板

```
指标名称：___
定义：___（精确定义如何计算）
目标值：___
数据来源：___
观测频率：每日 / 每周 / 每月
达标判定：连续 __ 周大于等于目标值
预警线：低于 ___ 时启动分析
```

## 第5节 边界管理框架

### 三组关系详细展开

#### 关系 1：与现有产品

```
评估维度：
  关系类型：替代 / 互补 / 独立
  功能重叠：
    哪些功能重叠？
    重叠功能的处理方式：新产品承接 / 保留在旧产品 / 两边都有但数据打通
    过渡期策略：并行运行多久？何时切换？
  数据关系：
    数据是否打通？
    用户数据迁移方案
    历史数据处理
  账号关系：
    共用账号 / 独立账号 / SSO
    权限继承规则
  迁移路径：
    用户迁移方案
    迁移时间表
    回退方案
```

#### 关系 2：与外部系统

```
评估维度：
  需对接的系统清单
  对接深度：
    轻度：数据导入导出
    中度：API 实时同步
    深度：深度集成、功能嵌入
  依赖程度：
    强依赖：对方不可用则本产品不可用
    弱依赖：对方不可用则部分功能受限
    无依赖：可选增强
  风险评估：
    对方系统的稳定性
    接口变更风险
    替代方案
```

#### 关系 3：与未来演进

```
评估维度：
  本版本定位：
    终态：这就是最终形态
    过渡：这是迈向终态的一步
  预留设计：
    为什么要预留？
    预留到什么程度？
    预留的成本是否可控？
  刻意不预留：
    为什么不预留？（避免过度设计）
    如果未来需要，改造成本评估
  演进路线图：
    V1 到 V2 的关键演进方向
    演进触发条件
    演进约束（哪些不变）
```
