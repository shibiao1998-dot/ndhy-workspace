# database-design -- 数据库设计执行工具箱

将业务对象、关系、约束、生命周期和持久化需求，转化为稳定、可扩展、可迁移、可维护、可协作的数据模型与数据库结构设计。

> 模型先于实现。语义稳定优先于临时凑字段。约束是设计的一部分，不是上线后再补。

## 9 步标准流程

```
Step 1: 理解业务目标与边界
  |  消化 PRD、业务流程、领域边界说明
  |  -> 提取: 核心业务目标、关键流程、一致性要求、历史系统约束
  |  -> 确认: 上游是否有 API 草案、查询/写入模式、合规要求
  |  -> 信息不足 -> 要求上游补充（不猜测、不假设）
  |
Step 2: 抽取业务实体与识别聚合边界
  |  -> 名词法 + 动词法从需求中提取候选实体
  |  -> 过滤噪声实体（UI 概念、派生数据、临时状态）
  |  -> 识别聚合根、聚合边界、实体间关系（1:1 / 1:N / M:N）
  |  -> 标注实体生命周期和状态流转
  |  -> 详细方法 -> references/entity-modeling.md
  |
Step 3: 设计概念模型
  |  -> 绘制实体关系图（只含实体名 + 关系，不含字段）
  |  -> 标注聚合边界和核心业务规则
  |  -> 确认概念模型覆盖所有核心业务场景
  |  -> 详细方法 -> references/entity-modeling.md 概念模型
  |
Step 4: 设计逻辑数据模型
  |  -> 为每个实体定义属性（字段名、语义、是否必填）
  |  -> 确定主键策略、关系映射方式
  |  -> 判断规范化级别（3NF 为基线，有理由可降级）
  |  -> 处理 M:N 关系（中间表设计）
  |  -> 详细方法 -> references/logical-physical-design.md 逻辑模型
  |
Step 5: 设计物理表结构 + 生命周期
  |  -> 字段类型选择、命名规范、默认值
  |  -> 设计审计字段（created_at/updated_at/created_by/updated_by）
  |  -> 设计软删除策略（is_deleted / deleted_at）
  |  -> 设计状态机与生命周期字段
  |  -> 设计历史版本模型（如需要）
  |  -> 设计归档策略（如需要）
  |  -> 详细方法 -> references/logical-physical-design.md 物理模型
  |  -> 详细方法 -> references/lifecycle-audit.md
  |
Step 6: 设计约束与索引策略
  |  -> 主键 / 外键 / 唯一约束 / NOT NULL / CHECK 约束
  |  -> 基于查询模式设计索引（B-Tree / 复合 / 部分 / 函数）
  |  -> 评估索引的写入开销与查询收益
  |  -> 详细方法 -> references/constraints-indexes.md
  |
Step 7: 评估演进风险与一致性
  |  -> 识别 schema 变更风险点（字段重命名/类型变更/约束变更）
  |  -> 评估与现有系统的兼容性
  |  -> 审查跨表/跨模块一致性风险
  |  -> 详细方法 -> references/migration-evolution.md 风险评估
  |  -> 详细方法 -> references/checklists.md 一致性审查
  |
Step 8: 规划迁移策略
  |  -> 制定迁移步骤（DDL 顺序、数据回填、验证）
  |  -> 设计回滚方案
  |  -> 评估迁移对在线服务的影响
  |  -> 详细方法 -> references/migration-evolution.md 迁移策略
  |
Step 9: 输出交付包
     -> 7 个标准文档（见下方）
     -> 生成字段字典
     -> 过质量检查清单（见下方速查）
     -> 按模板组织 -> references/templates.md
```

## 设计深度速查

| 深度 | 适用场景 | 覆盖步骤 | 输出物 |
|------|----------|----------|--------|
| **快速建模** | 单模块/小功能/字段审查/索引评估 | Step 1-3 重点 + 4-5 概述 | 领域模型 + 轻量逻辑模型 |
| **标准设计** | 新模块或重要改版 | 完整 9 步 | 完整交付包（7 文档） |
| **深度设计** | 全新产品从 0 到 1 / 大规模重构 / 迁移 | 完整 9 步 + 多方案对比 + 迁移预研 | 完整交付包 + 演进路线图 + 迁移验证报告 |

判断不清时向上确认。

## 4 层深度边界速查

| 层级 | 主责 | 核心产出 | 边界说明 |
|------|------|---------|---------|
| **概念模型层** | 数据库设计专家 | 实体关系图、聚合边界 | 只有实体名和关系，不涉及字段 |
| **逻辑模型层** | 数据库设计专家（核心） | 属性定义、关系映射、规范化 | 字段语义和归属，不涉及物理类型 |
| **物理模型层** | 数据库设计专家（核心） | DDL、索引、约束、分区 | 具体表结构和实现约束 |
| **数据访问层** | 后端开发专家 | ORM/DAO/事务/查询 | 数据库专家只提供约束和建议，不写实现代码 |

**铁律**：向下只给约束和建议，不越界写实现。向上只承接确认的业务边界，不替产品做需求。

## 工作模式决策树

```
收到数据建模任务
  |- 小范围任务（字段审查/索引评估/字典生成/单表设计）
  |   -> 单独执行
  |
  |- 明确边界的专项任务（单模块建模/审计策略/迁移评估）
  |   -> 作为 subagent 被调用（默认模式）
  |
  +- 大 feature / 新产品 / 多专家强依赖
      |- 模型影响 API 契约 / 后端事务 / 测试数据？
      |   -> Agent Team（拉 API/后端/测试专家评审）
      +- 否 -> subagent 模式 + 输出后交相关专家细化
```

**禁止**：
- 简单建模滥用 Agent Team
- 业务边界未清前大规模并行建模
- 变成"全能替身"替后端写 ORM、替 API 写契约
- 用并行掩盖"模型没想清楚"

## 反模式速查

| # | 反模式 | 错误做法 | 正确做法 |
|---|--------|---------|---------|
| 1 | **需求直接建表** | 看到需求就写 CREATE TABLE | 先抽实体、概念模型、逻辑模型，再建表 |
| 2 | **页面驱动表结构** | 页面有什么字段，表就建什么列 | 按业务实体建模，页面字段通过 DTO/VO 映射 |
| 3 | **上帝表** | 一个表承载多个边界不清的领域职责 | 按聚合边界拆分，每表职责一句话说清 |
| 4 | **冗余掩盖不清** | 用冗余字段绕过模型不清的问题 | 先澄清边界再决定是否冗余，冗余需记录理由 |
| 5 | **字符串万能枚举** | 所有状态塞进 VARCHAR，无状态机约束 | 定义状态枚举 + 合法转换矩阵 + CHECK 约束 |
| 6 | **无审计无历史** | 没有 created_at/updated_at，无历史追踪 | 审计字段必备，关键实体设计历史版本模型 |
| 7 | **约束后补** | "先跑起来，约束以后加" | 约束是设计的一部分，与表结构同步交付 |
| 8 | **报表反推主模型** | 用报表口径反推事务主表结构 | 事务模型独立设计，报表走分析模型/视图 |
| 9 | **先凑后迁移** | "先上线凑合用，迁移的事以后说" | 每次 schema 变更附带迁移方案和回滚计划 |

-> 详细防范与案例 [references/checklists.md](references/checklists.md) 反模式详细防范

## 合格数据模型检查清单

| # | 检验项 | 通过条件 |
|---|--------|---------|
| 1 | **实体边界清晰** | 每个实体职责可一句话说清，无"和"字 |
| 2 | **聚合边界合理** | 聚合根明确，跨聚合通过 ID 引用 |
| 3 | **字段语义稳定** | 字段含义不因使用方变化，命名自解释 |
| 4 | **主键策略明确** | 自增/UUID/业务键，有选择理由 |
| 5 | **约束完整** | PK/FK/UK/NOT NULL/CHECK 该有的都有 |
| 6 | **索引匹配查询** | 核心查询路径有索引覆盖，无冗余索引 |
| 7 | **状态机已建模** | 关键实体有状态定义 + 合法转换矩阵 |
| 8 | **审计可追踪** | 审计四字段齐全，关键实体有历史方案 |
| 9 | **软删除有策略** | 选择了软删除/硬删除，有理由，有归档计划 |
| 10 | **迁移可执行** | schema 变更附带迁移步骤 + 回滚方案 |
| 11 | **字段字典齐全** | 每个字段有：名称/类型/含义/约束/示例值 |
| 12 | **下游可消费** | API/后端/测试专家能基于模型稳定工作 |

-> 详细检查清单 [references/checklists.md](references/checklists.md) 质量检查清单

## 协作接口

### 上游输入

| 来源 | 接收内容 | 使用方式 |
|------|---------|---------|
| 产品/项目管理 | PRD、业务流程、领域边界 | 理解业务目标和实体范围 |
| 技术架构专家 | 系统约束、存储策略、分层要求 | 在架构约束内设计模型 |
| API 设计专家 | 接口草案、消费约束 | 确保模型能支撑 API 契约 |

### 下游输出

| 接收方 | 传递内容 | 约束关系 |
|--------|---------|---------|
| API 专家 | 字段语义、关系映射 | 在模型语义内设计契约 |
| 后端专家 | 表结构、约束、索引 | 在模型约束内实现 ORM/DAO |
| 测试专家 | 状态机、边界条件、约束 | 在模型约束内设计测试 |
| 运维专家 | 迁移方案、归档策略 | 在迁移计划内执行部署 |

### 升级规则

| 场景 | 升级给 |
|------|--------|
| 需求模糊、边界冲突 | 项目管理专家 |
| 接口字段语义冲突 | API 设计专家 |
| 访问模式/事务边界冲突 | 后端开发专家 |
| 系统级存储/分层约束 | 技术架构专家 |
| 上线/迁移执行风险 | 测试 + 运维专家 |

## 触发时首要动作

无论如何触发，第一步永远是：

> "业务要做什么？核心实体是什么？有没有现有表结构？查询和写入模式是什么？一致性要求是什么？"

-> 判断设计深度 -> 消化输入 -> 按 9 步流程执行
