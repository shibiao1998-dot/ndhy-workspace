## 反模式详细防范

### 反模式 1：需求直接建表

❌ **错误**：看到需求"用户可以创建课程"，直接写 `CREATE TABLE courses (...)` 。

**后果**：遗漏实体关系，表结构碎片化，后续不断 ALTER TABLE 打补丁。

✅ **正确**：
1. 先抽实体（用户、课程、章节、学员）
2. 画概念模型（关系和聚合边界）
3. 设计逻辑模型（属性和规范化）
4. 最后写 DDL

### 反模式 2：页面驱动表结构

❌ **错误**：页面表单有 15 个字段，表就建 15 个列，一一对应。

**后果**：UI 改了表就要改，前端展示逻辑污染持久化层。

✅ **正确**：
- 按业务实体建模，不是按页面建表
- 页面字段通过 DTO/VO 映射，不等于数据库列
- 派生字段（如"已完成百分比"）不落库

### 反模式 3：上帝表

❌ **错误**：一个 `items` 表存课程、作业、考试、资料，用 `type` 字段区分。

约束无法生效，查询需要大量条件判断。

✅ **正确**：
- 不同领域概念用不同表
- 如有共性，用继承模式（公共表 + 扩展表）
- 每个表的字段对所有记录都有意义

### 反模式 4：冗余掩盖不清

❌ **错误**：不确定地址属于用户还是订单，两边都冗余存一份。

**后果**：数据不一致，修改一处另一处不同步。

✅ **正确**：
- 先确定地址的归属和生命周期
- 如果订单需要快照地址 → 明确为快照（不是冗余）
- 冗余必须有理由 + 同步策略 + 风险记录

### 反模式 5：字符串万能枚举

❌ **错误**：`status VARCHAR(32)` 无约束，应用层随意写入 `'actve'`、`'ACTIVE'`、`'1'`。

**后果**：脏数据，状态统计不准，排查困难。

✅ **正确**：
```sql
CONSTRAINT ck_status CHECK (status IN ('draft','active','completed','cancelled'))
```
- 在数据库层限制合法值
- 关键实体定义状态转换矩阵
- 应用层和数据库层双重校验

### 反模式 6：无审计无历史

❌ **错误**：表里没有 created_at、updated_at，无法追溯数据何时被谁修改。

**后果**：数据出问题无法排查，合规审计无法通过。

✅ **正确**：
- 所有表必须有审计四字段
- 关键实体（订单/合同/成绩）设计历史版本模型
- 状态变更单独记录到日志表

### 反模式 7：约束后补

❌ **错误**："先把表建出来让开发跑起来，约束以后加"。

**后果**：等想加约束时已有大量脏数据，加约束反而导致故障。

✅ **正确**：
- 约束与 DDL 同步交付
- CREATE TABLE 里就包含所有 CONSTRAINT
- 建完表立即建索引

### 反模式 8：报表反推主模型

❌ **错误**：老板要看"每日注册用户数"，于是在 users 表加 `daily_count` 字段。

**后果**：事务表被报表需求污染，统计逻辑耦合进主模型。

✅ **正确**：
- 事务模型和分析模型分离
- 报表数据走物化视图/分析表/数据仓库
- 主表不为报表需求加字段

### 反模式 9：先凑后迁移

❌ **错误**："先凑几个表把功能跑通，迁移的事上线后再说"。

债务累积，最终迁移成本远超一开始设计好的成本。

✅ **正确**：
- 每次 schema 变更都附带迁移方案
- 评估变更对在线服务的影响
- 设计回滚方案

## 一致性风险审查

### 审查维度

#### 1. 同表一致性

| 检查项 | 风险示例 | 处理方式 |
|--------|---------|---------|
| 字段间逻辑矛盾 | end_time < start_time | CHECK 约束 |
| 冗余字段不同步 | total ≠ sum(items) | 触发器或应用层同步 |
| 枚举值不一致 | 同含义不同写法 | CHECK 约束统一 |

#### 2. 跨表一致性

| 检查项 | 风险示例 | 处理方式 |
|--------|---------|---------|
| FK 悬挂引用 | 子表引用不存在的父记录 | 外键约束 |
| 状态不一致 | 订单已取消但子项仍是"进行中" | 应用层级联更新 |
| 汇总字段失真 | 课程学员数与实际不一致 | 定期校验任务 |

#### 3. 跨聚合一致性

| 检查项 | 风险示例 | 处理方式 |
|--------|---------|---------|
| 最终一致性延迟 | 支付完成但订单状态未更新 | 事件驱动 + 补偿 |
| 并发冲突 | 两人同时修改同一记录 | 乐观锁（version 字段） |

#### 4. 跨系统一致性

| 检查项 | 风险示例 | 处理方式 |
|--------|---------|---------|
| 数据同步延迟 | 主系统更新了但从系统未同步 | 同步监控 + 告警 |
| ID 映射错误 | 外部系统 ID 与本系统 ID 对不上 | ID 映射表 + 校验 |

### 一致性审查清单

| # | 检查项 | 状态 |
|---|--------|------|
| 1 | 所有 CHECK 约束覆盖字段间逻辑 | ✅/❌ |
| 2 | 外键关系无悬挂引用 | ✅/❌ |
| 3 | 冗余字段有同步策略 | ✅/❌ |
| 4 | 跨聚合操作有最终一致性方案 | ✅/❌ |
| 5 | 并发控制方案已设计 | ✅/❌ |
| 6 | 外部系统同步有监控 | ✅/❌ |

## 字段字典生成指南

### 字段字典的作用

字段字典是数据模型的"人话版翻译"。确保所有消费方（API/后端/测试/运维）对每个字段的含义、约束、取值理解一致。

### 字段字典必填项

| 维度 | 说明 | 示例 |
|------|------|------|
| **字段名** | 物理名 | `enrollment_status` |
| **类型** | 物理类型 | `VARCHAR(32)` |
| **必填** | NOT NULL？ | ✅ |
| **默认值** | 有无默认值 | `'pending'` |
| **含义** | 业务语义（中文） | "学生选课的状态" |
| **约束** | PK/FK/UK/CHECK | CHECK IN ('pending','active','completed') |
| **示例值** | 典型合法值 | `"active"` |
| **备注** | 补充说明 | "状态转换见状态机文档" |

### 生成步骤

1. **遍历所有表** — 按聚合分组
2. **逐字段填写** — 上述 8 个维度
3. **公共字段统一说明** — 审计四字段 + 软删除字段只写一次
4. **枚举值单独列表** — 每个枚举字段的所有合法值及含义
5. **交叉检查** — 同语义字段在不同表中命名/类型一致

### 字段字典质量检查

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | 所有字段都有记录 | 字典字段数 = DDL 字段数 |
| 2 | 含义可理解 | 非技术人员能看懂 |
| 3 | 类型匹配 DDL | 字典中的类型与 DDL 一致 |
| 4 | 约束匹配 DDL | 字典中的约束与 DDL 一致 |
| 5 | 枚举值完整 | 每个枚举字段列出所有合法值 |
| 6 | 跨表一致 | 同语义字段命名/类型一致 |
