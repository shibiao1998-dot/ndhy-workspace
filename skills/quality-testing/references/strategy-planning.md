# 测试策略制定 + 风险场景识别 + 测试计划设计

## 一、测试策略制定

### 策略制定流程

```
输入 -> 理解业务目标和验收标准
  |
  +-> 分析变更范围和影响面
  |
  +-> 识别质量风险（业务 + 技术 + 环境）
  |
  +-> 确定测试类型组合
  |
  +-> 确定测试深度和优先级
  |
  +-> 确定测试范围（测什么 + 不测什么）
  |
输出 -> 测试策略（写入 test-plan.md）
```

### 测试类型选择矩阵

| 测试类型 | 何时使用 | 关注点 |
|---------|---------|--------|
| **功能测试** | 所有新功能/改动功能 | 功能是否按需求正确工作 |
| **边界测试** | 有输入范围/状态转换/数量限制 | 边界值是否正确处理 |
| **异常测试** | 有错误输入/超时/并发/权限场景 | 异常是否被正确捕获和处理 |
| **兼容性测试** | 多角色/多端/多浏览器/多状态 | 不同条件组合下是否一致 |
| **回归测试** | 有缺陷修复/代码重构/依赖变更 | 修复是否有效 + 未引入新问题 |
| **探索性测试** | 深度测试/复杂交互/首次发布 | 发现文档化用例未覆盖的问题 |

### 测试深度判断规则

```
任务特征分析
  |- 小范围修复/配置变更/单接口
  |   -> 快速验证：聚焦变更点 + 核心路径回归
  |
  |- 新功能模块/重要改版/常规迭代
  |   -> 标准测试：完整 9 步流程
  |
  +- 全新产品/核心模块重构/发布前
      -> 深度测试：9 步 + 探索性 + 多轮回归
```

### 测试优先级排序原则

1. **P0 - 必须测试**：核心业务路径、涉及资金/数据/安全的功能
2. **P1 - 应该测试**：主要功能、高频使用场景、已知高风险区域
3. **P2 - 可以测试**：次要功能、低频场景、有变通方案的区域
4. **P3 - 按需测试**：优化类功能、极端边界、锦上添花特性

## 二、风险场景识别

### 风险识别维度

| 维度 | 关注点 | 典型高风险场景 |
|------|--------|--------------|
| **业务影响** | 用户/数据/收入影响 | 支付流程、数据持久化、权限控制 |
| **技术复杂度** | 实现难度/依赖关系 | 状态机流转、并发操作、第三方集成 |
| **变更范围** | 改动面积/影响面 | 核心模型重构、公共组件修改、API 变更 |
| **历史问题** | 历史缺陷/回归风险 | 曾经出过 P0 的模块、频繁修改的区域 |
| **数据敏感性** | 数据正确性/隐私/合规 | 用户个人信息、成绩数据、财务数据 |
| **环境依赖** | 外部服务/配置/网络 | 第三方 API、CDN、消息队列 |

### 风险场景识别方法

#### 方法 1：从需求反推风险

```
需求描述 -> 提取关键动词（创建/修改/删除/查询/导入/导出/授权）
  |
  +-> 每个动词问：如果失败会怎样？
  +-> 每个动词问：如果并发会怎样？
  +-> 每个动词问：如果输入异常会怎样？
  +-> 每个动词问：如果权限不足会怎样？
  |
  -> 产出：风险场景清单
```

#### 方法 2：从实现反推风险

```
实现说明 -> 识别关键技术组件
  |
  +-> 数据库操作：事务边界/并发冲突/数据一致性
  +-> API 调用：超时/重试/幂等/错误处理
  +-> 状态流转：非法转换/并发状态/回退
  +-> 权限控制：越权/绕过/提权
  +-> 缓存使用：一致性/过期/穿透
  |
  -> 产出：技术风险清单
```

#### 方法 3：从历史反推风险

```
历史缺陷记录 -> 按模块统计缺陷密度
  |
  +-> 高密度模块 -> 重点回归
  +-> P0/P1 历史 -> 同类风险检查
  +-> 频繁修改区域 -> 回归覆盖
  |
  -> 产出：历史风险清单
```

### 风险场景记录格式

| # | 风险场景 | 风险维度 | 风险等级 | 影响范围 | 测试策略 |
|---|---------|---------|---------|---------|---------|
| RS-001 | {场景描述} | 业务/技术/环境 | 高/中/低 | {影响说明} | {如何验证} |

### 风险等级判断矩阵

```
          影响程度
          低        中        高
发  高   中风险    高风险    高风险
生
概  中   低风险    中风险    高风险
率
    低   低风险    低风险    中风险
```

## 三、测试计划设计

### 测试计划核心要素

一个合格的测试计划必须回答 6 个问题：

1. **测什么**：测试范围（包含 + 排除）
2. **为什么测**：测试目标和验收标准
3. **怎么测**：测试类型、方法、工具
4. **先测什么**：优先级和执行顺序
5. **什么条件下测**：环境、数据、前置条件
6. **什么算通过**：通过标准和退出条件

### 测试范围确定规则

#### 纳入范围的条件（任一满足）

- 本次新增或修改的功能
- 与变更有直接依赖关系的功能
- 历史高风险区域
- 核心业务路径（无论是否变更）

#### 排除范围的条件（需明确记录）

- 未变更且无依赖关系的功能
- 已有独立自动化测试覆盖的部分（需确认自动化仍有效）
- 环境不具备测试条件的部分（需标注为未验证风险）
- 由其他阶段覆盖的部分（需说明由谁何时覆盖）

### 测试环境检查清单

| 检查项 | 说明 |
|--------|------|
| 环境可访问 | 测试环境已部署最新代码 |
| 数据已准备 | 测试数据已创建/导入 |
| 配置已确认 | 关键配置与预期一致 |
| 依赖已就绪 | 第三方服务/Mock 已可用 |
| 账号已准备 | 各角色测试账号已创建 |
| 工具已就绪 | 测试工具/抓包工具已配置 |

### 测试计划评审要点

在测试计划定稿前，自查以下问题：

1. 范围是否完整覆盖了需求和变更？
2. 排除的部分是否有充分理由？
3. 风险场景是否已识别并有对应策略？
4. 环境和数据是否已确认就绪？
5. 通过标准是否与项目验收标准一致？
6. 时间和资源是否合理？

### 测试入口条件（开始测试前必须满足）

| 条件 | 说明 |
|------|------|
| 需求明确 | 有可验证的验收标准 |
| 代码就绪 | 功能代码已提交并部署到测试环境 |
| 联调通过 | 核心链路联调已通过（非首次提测可例外） |
| 环境稳定 | 测试环境可正常访问和操作 |
| 数据就绪 | 测试数据已准备完毕 |

### 测试出口条件（结束测试的判断依据）

| 条件 | 标准 |
|------|------|
| 用例执行完成 | 所有计划用例已执行或有明确跳过理由 |
| P0 缺陷 | 全部关闭 |
| P1 缺陷 | 全部关闭或有明确延期计划 |
| 回归验证 | 已执行且达到通过标准 |
| 风险评估 | 已完成且结论明确 |
