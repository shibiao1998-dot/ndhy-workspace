# 真实世界测试教训

> **提炼日期**：2026-03-16
> **性质**：实战踩坑记录，非理论补充

---

## 教训 1：Mock 模式是测试的双刃剑

**踩坑场景**：提示词工程 v1 前端使用 `USE_MOCK = true` 硬编码，所有测试都在 Mock 数据下通过。前端 TypeScript 编译 0 错误、13 项功能全部实现，但从未在真实 API 环境运行过。

**后果**：
- Mock 数据维度总数 89（32 条示例），后端实际 99 个维度——数据量级差异未暴露
- 测试报告结论是"有条件发布"，条件之一是"切换 USE_MOCK=false 并完成端到端验证"
- 如果不做这步就发布，所有"前端通过"的结论都不可靠

**教训规则**：
- 🔴 凡是有 Mock 开关的项目，测试报告必须明确标注"Mock 模式"还是"真实 API 模式"
- 🔴 "Mock 模式通过"和"真实 API 通过"是两个不同的结论，不能等价
- 🟡 Mock 开关必须通过环境变量控制，不能硬编码在源码中
- 🟡 发布前必须有一个明确的"切换真实环境 + 端到端验证"步骤

## 教训 2：框架验证会偷偷覆盖你的契约

**踩坑场景**：后端使用 FastAPI + Pydantic，契约约定空 task 返回 HTTP 400 + 错误码 `INVALID_REQUEST_BODY`。实际 Pydantic 的 `validation_exception_handler` 先于业务逻辑触发，返回了 HTTP 422。

**后果**：
- HTTP 状态码与契约不一致（422 vs 400）
- 空 dimensions 应返回专用错误码 `EMPTY_DIMENSIONS`，实际返回了通用 `INVALID_REQUEST_BODY`
- 前端通过 error.code 判断，功能不受影响——但契约完整性被破坏

**教训规则**：
- 🔴 使用 Pydantic/Zod 等验证框架时，第一件事是确认其默认错误响应格式
- 🔴 写单元测试时必须包含"框架先于业务逻辑拦截"的场景
- 🟡 自定义 exception handler 要覆盖框架默认的 validation error handler
- 🟡 API 测试用例中加一类：发送能触发框架校验的请求，验证返回的状态码和错误码

## 教训 3：setInterval 有启动就必须有停止

**踩坑场景**：官网的 Pipeline 动画用 `setInterval` 每 400ms 执行 DOM 操作，用 `IntersectionObserver` 触发启动。但 observer 只 disconnect 了自身，`setInterval` 永远不会被清理。

**后果**：用户滚过 Pipeline 区域后，每 400ms 仍在对不可见的 DOM 做 classList 操作。单个 interval 不致命，但在长时间打开的页面上累积，浪费 CPU。

**教训规则**：
- 🔴 代码审查时看到 `setInterval` / `setTimeout`，立即搜索对应的 `clearInterval` / `clearTimeout`
- 🔴 用 IntersectionObserver 启动的动画，必须用同一个 observer 控制停止
- 🟡 测试时检查"离开视口后资源是否释放"——这是一个常被忽略的场景

## 教训 4：缓存策略不能一刀切

**踩坑场景**：Caddy 配置中 `@static` matcher 包含了 `*.html`，所有文件统一 `max-age=3600`。HTML 文件被缓存 1 小时，更新部署后用户看到旧版本。

**后果**：对于单文件站点（只有 index.html），一次部署更新最长需要 1 小时才能对所有用户生效。

**教训规则**：
- 🔴 HTML 文件和静态资源（CSS/JS/图片/字体）必须分开设置缓存策略
- 🟡 HTML：短缓存（如 60s + must-revalidate）
- 🟡 静态资源：可长缓存（如 24h + stale-while-revalidate）
- 🟡 部署配置审查时，逐个检查 cache matcher 的文件类型覆盖

## 教训 5：安全头不是配一次就完事

**踩坑场景**：`X-XSS-Protection: 1; mode=block` 写在 Caddyfile 中。这个头从 Chrome v78 起已被移除支持，保留无害但属于无效配置。更微妙的是，某些安全审计工具建议设为 `0`，因为旧浏览器的 XSS Auditor 本身有漏洞。

**教训规则**：
- 🟡 安全头配置来源不能是"上次项目的模板"——需要验证当前浏览器兼容性
- 🟡 定期审查安全头（每季度或每次大版本部署时）
- 🟡 CSP 中 `unsafe-inline` 如果是架构限制（单文件站 CSS/JS 全内联），必须在审查报告中标注为"已知权衡"

## 教训 6："有条件通过"的条件容易被遗忘

**踩坑场景**：两份报告都给出了"有条件通过/有条件发布"的结论。条件分别是：
- 官网：移动端性能保护、setInterval 清理、HTML 缓存策略——"建议部署前处理"
- 提示词工程：切换 USE_MOCK=false 端到端验证 + DJ 现场验收

**风险**：如果没有跟踪机制，这些"条件"会在发布后被遗忘，变成永久技术债。

**教训规则**：
- 🔴 每次"有条件通过"必须生成一个 follow-up issue 列表，有明确的负责人和截止时间
- 🟡 "建议部署前处理"不够强——要明确是"部署前必须完成"还是"可以部署后迭代"
- 🟡 项目管理专家收到"有条件通过"结论后，应将条件转化为具体任务并纳入跟踪

## 教训 7：60 分版本的质量底线要画清楚

**踩坑场景**：提示词工程明确是"60 分版本"，接受了"无前端自动化测试"的现状。测试报告将其标为 P3-低风险。

**教训**：这是正确的 trade-off 判断——关键是它被明确标注了。

**教训规则**：
- ✅ "知道自己在冒什么险"比"不知道自己缺什么"好 100 倍
- 🟡 MVP/60 分版本也要有明确的质量底线文档：哪些可以不做、哪些绝不能不做
- 🟡 跳过的验证必须在测试报告中标注为"未验证风险"并纳入风险评估
- 🟡 "60 分版本可接受"不等于"永远可接受"——后续迭代要有补齐计划

## 教训 8：内容一致性需要"契约意识"

**踩坑场景**：官网 HTML 将 content-v2.md 定义的三个小分组（组织基建 3 人 + 知识管理 1 人 + 项目管理 1 人）合并为一个"组织基建"大组（5 人）。开发判断合并后视觉更好，但未与文案负责人确认。

**教训规则**：
- 🟡 文案/内容定义文档是否具有"契约"约束力，需要在项目启动时明确
- 🟡 任何偏离文案定义的展示优化，必须获得确认后再实现
- 🟡 代码审查时应对照内容定义文档做一致性核查——这一步容易被跳过
