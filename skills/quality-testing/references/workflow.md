# 质量测试 — 9步标准流程

## 流程总览

```
Step 1: 理解需求目标与验收边界
  |  消化 PRD、体验方案、API 契约、数据模型、联调结果
  |  -> 提取: 核心功能、验收标准、已知约束、历史缺陷
  |  -> 确认: 测试范围、环境要求、数据准备需求
  |  -> 信息不足 -> 要求上游补充（不猜测、不假设）
  |
Step 2: 识别关键风险场景
  |  -> 按业务影响、技术复杂度、变更范围识别高风险区域
  |  -> 标注: 资金/数据/权限/并发/状态机/第三方依赖
  |  -> 产出: 风险场景清单（带优先级）
  |  -> 详细方法 -> references/strategy-planning.md 风险识别
  |
Step 3: 制定测试策略与测试计划
  |  -> 确定测试类型（功能/边界/异常/兼容性/性能/安全）
  |  -> 确定测试范围（全量/增量/重点区域）
  |  -> 确定测试深度（按 3 级深度速查判断）
  |  -> 制定测试计划（阶段/里程碑/环境/数据/人员）
  |  -> 详细方法 -> references/strategy-planning.md
  |
Step 4: 设计测试用例
  |  -> 主路径用例（覆盖核心业务流程）
  |  -> 边界条件用例（等价类 + 边界值）
  |  -> 异常路径用例（错误输入/超时/并发/权限）
  |  -> 兼容性用例（多角色/多端/多状态组合）
  |  -> 详细方法 -> references/test-case-design.md
  |
Step 5: 执行验证与记录缺陷
  |  -> 按优先级执行用例（高风险优先）
  |  -> 发现问题 -> 记录缺陷（复现路径 + 期望 vs 实际 + 截图/日志）
  |  -> 缺陷分级（P0-P3，见缺陷分级框架）
  |  -> 详细方法 -> references/defect-management.md
  |
Step 6: 跟踪缺陷修复与回流
  |  -> 缺陷回流对应专家（前端/后端/API/联调）
  |  -> 跟踪修复状态（待修复/修复中/待验证/已关闭/重开）
  |  -> 修复完成 -> 进入回归验证
  |  -> 详细方法 -> references/defect-management.md 跟踪闭环
  |
Step 7: 回归验证
  |  -> 设计回归清单（修复项 + 影响范围 + 核心路径）
  |  -> 执行回归验证（修复验证 + 影响面验证 + 核心路径回归）
  |  -> 更新缺陷状态和风险状态
  |  -> 详细方法 -> references/regression-verification.md
  |
Step 8: 汇总测试结果与评估发布风险
  |  -> 汇总: 用例通过率、缺陷统计、风险残留
  |  -> 评估发布风险（见发布风险评估矩阵）
  |  -> 产出: 测试执行报告 + 质量风险摘要
  |
Step 9: 形成阶段验收结论
     -> 判断: 通过 / 有条件通过 / 不通过（见验收结论标准）
     -> 输出: 发布就绪评估
     -> 过质量检查清单
     -> 按模板组织 -> references/templates.md
```

## 6层深度边界速查

| 层级 | 主责专家 | 与测试专家的关系 |
|------|---------|-----------------|
| **1. 需求与设计定义层** | 产品/体验/API/数据库/架构专家 | 测试消费其产出作为验证依据 |
| **2. 实现层** | 前端/后端开发专家 | 测试验证实现是否符合设计 |
| **3. 代码与结构审查层** | 代码审查专家 | 测试不替代静态审查，发现实现债回流 |
| **4. 集成闭环层** | 联调集成专家 | 测试不替代联调归因，利用联调结果缩小范围 |
| **5. 验证与验收层** | **测试专家（核心）** | 功能/边界/异常/回归/风险/结论 |
| **6. 发布与运行层** | 部署运维专家 | 测试提供质量输入，不替代上线操作 |

**铁律**：主责第 5 层，消费第 1-4 层输入，为第 6 层提供质量判断。不吞掉审查、联调和运维职责。

## 缺陷分级框架

| 级别 | 名称 | 定义 | 处理要求 | 示例 |
|------|------|------|---------|------|
| **P0** | 阻断 | 核心功能不可用/数据丢失/安全漏洞 | 立即修复，阻断发布 | 支付失败、登录崩溃、数据泄露 |
| **P1** | 严重 | 主要功能异常/数据错误/无变通方案 | 当前迭代必须修复 | 列表数据错乱、权限绕过、状态机卡死 |
| **P2** | 一般 | 次要功能问题/有变通方案/体验降级 | 排入修复计划，可带风险发布 | 样式错位、提示文案错误、排序异常 |
| **P3** | 建议 | 优化建议/微小瑕疵/极端边界 | 记录评估，按优先级排入后续 | 极端输入的边界处理、交互优化建议 |

**分级决策树**：

```
发现缺陷 -> 核心功能不可用？ -是-> P0
               |否
          数据正确性问题？ -是-> 影响主流程？ -是-> P1 / -否-> P2
               |否
          有变通方案？ -否-> P1 / -是-> P2
               |
          仅优化建议？ -是-> P3
```

-> 详细分级规范 [defect-management.md](defect-management.md) 缺陷分级

## 工作模式决策树

```
收到测试任务
  |- 小范围任务（单接口验证/单缺陷复现/配置检查）
  |   -> 单独执行
  |
  |- 明确边界的专项任务（模块用例设计/缺陷报告整理/回归清单生成）
  |   -> 作为 subagent 被调用（默认模式）
  |
  +- 大 feature / 发布前评估 / 多专家强依赖
      |- 质量判断依赖前端+后端+联调+环境？
      |   -> Agent Team（拉相关专家共同确认）
      +- 否 -> subagent 模式 + 输出后交相关专家细化
```

**禁止**：
- 简单验证滥用 Agent Team
- 需求和环境未清前大规模并行验证
- 用多人测试掩盖验证范围没定义清楚
- 变成开发、联调、审查的全能替身

## 发布风险评估矩阵

| 维度 | 低风险 ✅ | 中风险 ⚠️ | 高风险 🔴 |
|------|----------|-----------|----------|
| **P0 缺陷** | 0 个 | 0 个 | ≥1 个未关闭 |
| **P1 缺陷** | 0 个 | ≤2 个（有变通方案）| ≥3 个或无变通方案 |
| **用例通过率** | ≥95% | 85%-95% | <85% |
| **核心路径** | 全部通过 | 全部通过 | 有未通过 |
| **回归验证** | 全部通过 | ≥90% 通过 | <90% 通过 |
| **未验证区域** | 无 | 有（低风险区域） | 有（高风险区域） |

**风险判定**：任一维度达到 🔴 → 整体为高风险。

-> 详细评估方法 [checklists.md](checklists.md) 发布风险评估

## 验收结论判断标准

### ✅ 通过（可发布/可进入下一阶段）

- P0 缺陷：0 个
- P1 缺陷：0 个（或全部有明确延期理由 + 变通方案）
- 核心路径：100% 通过
- 用例通过率：≥95%
- 回归验证：全部通过
- 风险评估：低风险

### ⚠️ 有条件通过（可发布但需附带风险说明）

- P0 缺陷：0 个
- P1 缺陷：≤2 个（有变通方案 + 明确修复计划）
- 核心路径：100% 通过
- 用例通过率：≥85%
- 存在已知风险但已充分评估并记录
- 需项目管理专家确认接受风险

### ❌ 不通过（不可发布/需继续修正）

- 存在未关闭的 P0 缺陷
- P1 缺陷 ≥3 个且无变通方案
- 核心路径有未通过项
- 用例通过率 <85%
- 存在未验证的高风险区域
- 需明确：阻断原因 + 修复要求 + 预计重新验证时间

## 协作接口

### 上游输入

| 来源 | 接收内容 | 使用方式 |
|------|---------|---------|
| 产品/项目管理 | PRD、验收标准、发布目标 | 确定测试范围和验收基线 |
| 体验设计专家 | 页面流程、交互说明 | 设计交互类测试用例 |
| API 设计专家 | 接口契约、错误码定义 | 设计接口级测试用例 |
| 数据库设计专家 | 数据模型、状态机、约束 | 设计数据类边界用例 |
| 前端/后端开发专家 | 实现说明、已知限制 | 识别实现层风险场景 |
| 联调集成专家 | 联调结果、集成问题 | 缩小测试范围，锁定高风险区域 |

### 下游输出

| 接收方 | 传递内容 | 约束关系 |
|--------|---------|---------|
| 前端开发专家 | 页面/交互/状态类缺陷 | 附复现路径和期望行为 |
| 后端开发专家 | 接口行为/数据/逻辑类缺陷 | 附复现路径和影响范围 |
| API 设计专家 | 契约歧义/错误码不清问题 | 附发现场景和建议 |
| 联调集成专家 | 链路级问题回流 | 附测试中发现的集成异常 |
| 代码审查专家 | 实现债/结构性风险 | 附测试中暴露的实现问题 |
| 部署运维专家 | 环境/配置/发布风险 | 附质量评估和风险说明 |
| 项目管理专家 | 验收结论/阻断升级/风险通报 | 附结论和建议 |

### 升级规则

| 场景 | 升级给 |
|------|--------|
| 页面交互/样式/状态缺陷 | 前端开发专家 |
| 接口行为/数据/逻辑缺陷 | 后端开发专家 |
| 契约歧义/错误码不清 | API 设计专家 |
| 链路闭环问题 | 联调集成专家 |
| 结构性实现风险 | 代码审查专家 |
| 环境/发布/回滚风险 | 部署运维专家 |
| 阻断缺陷/质量风险/排期影响 | 项目管理专家 |

## 触发时首要动作

无论如何触发，第一步永远是：

> "要验证什么？验收标准是什么？有哪些已知约束？环境是否就绪？历史缺陷有哪些？"

-> 判断测试深度 -> 消化输入 -> 按 9 步流程执行
