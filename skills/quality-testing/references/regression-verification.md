# 回归清单设计 + 回归验证执行 + 回归范围控制

## 一、回归测试核心原则

1. **回归必须有边界**：不盲目全量回归，基于影响分析确定范围
2. **修复验证优先**：先验证缺陷修复是否有效
3. **影响面验证必须做**：修复可能引入新问题，关联功能必须验证
4. **核心路径必回归**：无论改动大小，核心业务路径必须回归
5. **回归结果必须记录**：每项回归都有明确的通过/未通过结论

---

## 二、回归清单设计

### 回归清单三层结构

```
回归清单
  |
  +-> 第一层：修复验证（直接验证缺陷修复）
  |     -> 每个已修复缺陷的原始复现步骤
  |     -> 验证修复是否有效
  |
  +-> 第二层：影响面验证（修复可能影响的关联功能）
  |     -> 与修复代码有直接关联的功能
  |     -> 修复涉及的公共组件/模块的其他使用方
  |     -> 修复涉及的数据模型的其他消费方
  |
  +-> 第三层：核心路径回归（关键业务流程确认）
        -> 核心业务流程端到端验证
        -> 高频使用场景抽样
```

### 回归范围确定方法

#### 方法 1：代码变更分析

```
代码变更清单
  |
  +-> 识别变更的文件/模块
  +-> 识别调用该模块的上游功能
  +-> 识别该模块调用的下游功能
  +-> 识别使用同一数据的其他功能
  |
  -> 产出：影响范围清单
```

#### 方法 2：风险评估法

```
修复描述
  |
  +-> 修复范围大不大？ -- 大 -> 扩大回归范围
  +-> 涉及公共组件？ -- 是 -> 公共组件使用方全回归
  +-> 涉及数据模型？ -- 是 -> 数据消费方全回归
  +-> 涉及核心逻辑？ -- 是 -> 核心路径全回归
  +-> 仅局部修改？ -- 是 -> 修复验证 + 直接关联回归
```

#### 方法 3：历史经验法

```
历史缺陷记录
  |
  +-> 该模块历史回归发现新问题的频率
  +-> 高频 -> 扩大回归范围
  +-> 低频 -> 标准回归范围
```

### 回归清单模板

| # | 层级 | 验证项 | 来源 | 验证步骤 | 预期结果 | 实际结果 | 状态 |
|---|------|--------|------|---------|---------|---------|------|
| 1 | 修复验证 | BUG-001 修复确认 | 缺陷修复 | {步骤} | {预期} | {实际} | ✅/❌ |
| 2 | 影响面 | {关联功能} | 影响分析 | {步骤} | {预期} | {实际} | ✅/❌ |
| 3 | 核心路径 | {业务流程} | 核心保护 | {步骤} | {预期} | {实际} | ✅/❌ |

## 三、回归验证执行

### 执行流程

```
回归清单就绪
  |
  +-> Step 1: 确认修复已部署到测试环境
  |
  +-> Step 2: 执行第一层 - 修复验证
  |     |- 通过 -> 继续
  |     +- 未通过 -> 标记"重开"，通知开发，暂停该缺陷回归
  |
  +-> Step 3: 执行第二层 - 影响面验证
  |     |- 发现新问题 -> 记录新缺陷，纳入缺陷管理流程
  |     +- 无新问题 -> 继续
  |
  +-> Step 4: 执行第三层 - 核心路径回归
  |     |- 发现新问题 -> 评估级别，可能升级为阻断
  |     +- 无新问题 -> 继续
  |
  +-> Step 5: 汇总回归结果
  |     -> 更新缺陷状态
  |     -> 计算回归通过率
  |     -> 评估回归结论
  |
  -> 输出: 更新 regression-checklist.md
```

### 回归验证判断标准

| 结论 | 条件 |
|------|------|
| **回归通过** | 三层全部通过，无新发现问题 |
| **部分通过** | 修复验证通过，但发现新的 P2/P3 问题 |
| **回归未通过** | 修复验证未通过，或发现新的 P0/P1 问题 |

### 回归未通过处理

```
回归未通过
  |
  +-> 修复验证未通过？
  |     -> 缺陷状态改为"重开"
  |     -> 通知开发专家附上新的发现
  |     -> 等待重新修复后再次回归
  |
  +-> 发现新的 P0/P1？
  |     -> 记录新缺陷
  |     -> 评估是否由本次修复引入
  |     -> 是 -> 通知开发专家，暂停发布评估
  |     -> 否 -> 纳入正常缺陷处理流程
  |
  +-> 发现新的 P2/P3？
        -> 记录新缺陷
        -> 评估是否影响发布决策
        -> 纳入正常缺陷处理流程
```

## 四、回归范围控制

### 范围膨胀风险

| 风险 | 表现 | 控制手段 |
|------|------|---------|
| **过度回归** | 每次回归都全量测试 | 基于影响分析确定范围，非变更区域不回归 |
| **无限追加** | 回归中不断发现新问题追加 | 新发现问题纳入下一轮回归，本轮按计划完成 |
| **范围蔓延** | 回归范围不断扩大无边界 | 明确回归三层结构，超出范围需评审决定 |
| **重复回归** | 同一项被多次验证 | 回归清单去重，已验证项不重复执行 |

### 回归范围控制原则

1. **影响分析驱动**：不按"感觉"回归，基于代码变更和依赖关系分析
2. **三层不混淆**：修复验证、影响面、核心路径各自独立
3. **新问题另起**：回归中发现的新问题不追加到当前回归，单独建缺陷
4. **有进有出**：新增回归项必须有理由，完成的项目及时标记
5. **时间预算**：回归有时间预算，超预算需评估是否继续

### 多轮回归管理

当需要多轮回归时：

```
第 1 轮回归：修复验证 + 影响面 + 核心路径
  |
  +-> 发现问题 -> 修复 -> 第 2 轮回归
  |
第 2 轮回归：新修复验证 + 上轮失败项 + 核心路径
  |
  +-> 仍有问题 -> 评估是否继续回归或升级
  |
第 3 轮及以上：如果仍需回归，需升级项目管理专家评估
  -> 是否继续修复？
  -> 是否缩小发布范围？
  -> 是否延期发布？
```

## 五、回归测试反模式

| ❌ 错误做法 | ✅ 正确做法 |
|------------|------------|
| 每次全量回归 | 基于影响分析确定回归范围 |
| 只验证修复项不看影响面 | 三层结构完整覆盖 |
| 回归发现新问题直接追加 | 新问题另起缺陷，下轮处理 |
| 回归失败不通知直接标通过 | 严格按实际结果记录，未通过就是未通过 |
| 无限轮次回归没有升级机制 | 3 轮以上未收敛，升级评估 |
