# 测试用例设计 + 边界与异常验证

## 一、测试用例设计方法

### 用例设计流程

```
需求/设计文档
  |
  +-> 提取功能点和业务规则
  |
  +-> 识别输入域、状态域、输出域
  |
  +-> 按四类覆盖设计用例
  |     |- 主路径用例（正常流程）
  |     |- 边界条件用例（极值/临界）
  |     |- 异常路径用例（错误/故障）
  |     +- 兼容性用例（组合/多端）
  |
  +-> 标注优先级（P0-P3）
  |
  +-> 关联需求/风险场景
  |
  -> 产出：test-cases.md
```

### 用例设计原则

1. **需求驱动**：每条验收标准至少有一个对应用例
2. **风险驱动**：高风险场景优先设计用例
3. **可执行**：步骤具体、结果明确、可复现
4. **独立性**：用例之间无顺序依赖（除明确标注的流程链）
5. **可追溯**：每个用例关联到需求或风险场景

### 用例质量检查

| 要素 | 要求 |
|------|------|
| 前置条件 | 明确环境/数据/状态要求 |
| 测试步骤 | 具体到每个操作，不用"按要求操作"含糊 |
| 期望结果 | 可验证的具体行为，不用"正常"含糊 |
| 优先级 | P0-P3 明确标注 |
| 关联 | 关联到需求或风险场景 |

## 二、主路径用例设计

### 核心业务流程覆盖

对每个核心业务流程，设计完整的端到端用例：

```
场景: {业务场景名称}
  Given: {前置状态}
  When:  {用户操作序列}
  Then:  {期望结果和状态变化}
```

### 主路径识别方法

1. **从 PRD 中提取**：每个功能描述的主要使用场景
2. **从流程图中提取**：主干路径（非分支路径）
3. **从验收标准中提取**：每条验收标准对应的操作路径
4. **从 API 契约中提取**：每个接口的正常调用场景

### 主路径用例覆盖检查

- [ ] 每个核心功能至少 1 个主路径用例
- [ ] 完整 CRUD 覆盖（创建/读取/更新/删除）
- [ ] 关键状态流转的每个合法路径
- [ ] 核心业务流程的端到端链路

## 三、边界条件测试

### 等价类划分法

将输入域划分为有效等价类和无效等价类，每类至少取一个代表值。

```
输入域: 年龄（1-150 的整数）

有效等价类:
  E1: 1-150 之间的整数 -> 代表值: 25

无效等价类:
  E2: < 1 -> 代表值: 0, -1
  E3: > 150 -> 代表值: 151, 999
  E4: 非整数 -> 代表值: 3.14, "abc"
  E5: 空值 -> 代表值: null, ""
```

### 边界值分析法

对每个边界，取边界上、边界内侧、边界外侧三个值。

```
范围: [1, 150]

下边界:
  0  -> 边界外（无效）
  1  -> 边界上（有效，最小值）
  2  -> 边界内（有效）

上边界:
  149 -> 边界内（有效）
  150 -> 边界上（有效，最大值）
  151 -> 边界外（无效）
```

### 常见边界类型

| 边界类型 | 测试要点 |
|---------|---------|
| **数值范围** | 最小值、最大值、最小值-1、最大值+1 |
| **字符串长度** | 空串、1 字符、最大长度、最大长度+1 |
| **集合大小** | 空集合、1 元素、最大元素数、超出最大数 |
| **日期时间** | 起始时间、结束时间、跨日/跨月/跨年 |
| **分页** | 第 1 页、最后一页、超出总页数、每页 0/1/max |
| **状态转换** | 初始状态、终态、非法转换、并发转换 |
| **权限** | 最低权限、最高权限、无权限、跨角色 |
| **金额** | 0 元、0.01 元、最大金额、负数 |
| **文件** | 空文件、最大文件、非法格式、超大文件 |

## 四、异常路径测试

### 异常场景分类

| 分类 | 场景 | 测试要点 |
|------|------|---------|
| **输入异常** | 非法输入/格式错误/类型错误 | 校验提示是否清晰、是否阻断非法操作 |
| **权限异常** | 未登录/无权限/过期 token | 是否正确拒绝、提示是否准确 |
| **状态异常** | 非法状态转换/重复操作/并发冲突 | 状态机是否防护、幂等是否正确 |
| **网络异常** | 超时/断连/重试/慢网络 | 超时处理、重试策略、用户提示 |
| **数据异常** | 空数据/脏数据/数据不一致 | 空态展示、数据校验、容错处理 |
| **并发异常** | 多用户同时操作/快速重复提交 | 并发控制、幂等处理、乐观锁 |
| **依赖异常** | 第三方服务不可用/返回异常 | 降级策略、错误处理、用户提示 |

### 异常用例设计模板

```
场景: {异常场景名称}
  Given: {前置正常状态}
  When:  {触发异常的操作}
  Then:  {期望的异常处理行为}
    - 错误提示: {具体提示内容}
    - 数据状态: {数据是否被影响}
    - 恢复路径: {用户如何回到正常状态}
```

### 异常测试覆盖检查

- [ ] 每个输入字段的非法值处理
- [ ] 关键操作的权限验证
- [ ] 状态机的非法转换防护
- [ ] 核心接口的超时和错误处理
- [ ] 空数据/空列表的展示处理
- [ ] 并发操作的冲突处理
- [ ] 外部依赖的故障降级

## 五、兼容性测试

### 兼容性维度

| 维度 | 适用场景 | 测试方法 |
|------|---------|---------|
| **多角色** | 系统有多种用户角色 | 每种角色执行核心流程 |
| **多终端** | 支持 PC/移动端/平板 | 各终端核心功能验证 |
| **多浏览器** | Web 应用 | Chrome + 主流浏览器核心验证 |
| **多状态组合** | 功能涉及多个状态 | 关键状态组合的交叉验证 |
| **多数据量** | 列表/搜索/导出 | 空数据/少量/大量/极大量 |

### 组合测试策略

当组合数量过多时，使用正交法或 Pairwise 方法减少用例数：

1. 识别所有变量和取值
2. 确定关键变量对（最可能交互的变量）
3. 保证每对变量的所有取值组合至少覆盖一次
4. 高风险组合全量覆盖，低风险组合 Pairwise 覆盖

## 六、用例设计反模式

| ❌ 错误做法 | ✅ 正确做法 |
|------------|------------|
| 只写主路径，不写边界和异常 | 四类用例全覆盖 |
| 期望结果写"正常" | 写具体可验证的行为 |
| 步骤写"按要求操作" | 写每个具体操作步骤 |
| 用例之间强耦合 | 每个用例独立可执行 |
| 不标优先级 | P0-P3 明确标注 |
| 不关联需求 | 每个用例追溯到需求/风险 |
| 用例数量堆砌不覆盖风险 | 先覆盖高风险，再补充常规 |
