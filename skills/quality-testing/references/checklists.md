## 二、反模式详细防范

### 反模式 1：只测 Happy Path

**表现**：只验证正常流程，不测边界和异常。

**危害**：上线后边界条件和异常场景频繁出错。

**防范**：
- 用例设计时强制四类覆盖：主路径 + 边界 + 异常 + 兼容性
- 检查清单 B3、B4 必须通过
- 风险场景必须有专项用例

### 反模式 2：缺陷描述模糊

**表现**："这里有问题"、"不太对"、"感觉不正常"。

**危害**：开发无法复现、修复方向不清、来回沟通耗时。

**防范**：
- 缺陷必填字段强制检查（见 defect-management.md）
- 无复现步骤的缺陷不允许提交
- 无证据的缺陷不允许提交

### 反模式 3：缺陷不分级

**表现**：所有缺陷平铺列出，不区分严重程度。

**危害**：无法判断哪些必须修、哪些可以缓，影响决策效率。

**防范**：
- 每个缺陷必须标注 P0-P3
- P0/P1 标红，明确处理要求
- 缺陷报告按级别分组展示

### 反模式 4：回归范围失控

**表现**：每次回归全量测试，或回归范围不断蔓延。

**危害**：回归效率低，时间成本高，但未必提高质量。

**防范**：
- 基于影响分析确定回归范围
- 回归清单三层结构明确边界
- 新发现问题另起缺陷，不追加到当前回归
- 详见 regression-verification.md

### 反模式 5：联调通过 = 测试通过

**表现**：联调跑通后直接认为测试通过，不做独立验证。

**危害**：联调验证的是集成链路，不覆盖功能边界和异常。

**防范**：
- 明确区分联调和测试的职责边界
- 联调结果作为测试输入，不替代测试
- 边界/异常/兼容性只能通过测试覆盖

### 反模式 6：代码 Review 通过 = 测试通过

**表现**：代码审查通过后认为不需要测试。

**危害**：代码审查验证结构质量，不验证运行时行为。

**防范**：
- 明确区分代码审查和测试的职责边界
- 代码审查发现结构问题，测试发现行为问题
- 两者互补，不能替代

### 反模式 7：提了就算，不跟踪闭环

**表现**：提了缺陷但不跟踪修复结果，不做回归验证。

**危害**：缺陷"消失"但未真正修复，上线后复现。

**防范**：
- 缺陷必须跟踪到关闭（修复+回归验证）
- P0/P1 缺陷状态每日检查
- 未闭环缺陷在测试报告中明确标注

### 反模式 8：赶进度跳过验证不标风险

**表现**：因为时间紧跳过部分验证，但不在报告中说明。

**危害**：未验证的风险被隐藏，发布后才暴露。

**防范**：
- 跳过任何验证必须在报告中标注"未验证风险"
- 未验证区域纳入风险评估
- 跳过 P0/P1 区域的验证需升级项目管理确认

### 反模式 9：用例数量堆砌不覆盖风险

**表现**：设计了大量简单用例，但高风险场景未覆盖。

**危害**：测试通过率高但实际风险未被发现。

**防范**：
- 先覆盖高风险场景，再补充常规场景
- 风险场景清单的每一项必须有对应用例
- 用例数量不是质量指标，风险覆盖率才是

## 三、发布风险评估详细方法

### 评估维度与评级标准

| 维度 | 评估方法 | 低风险 ✅ | 中风险 ⚠️ | 高风险 🔴 |
|------|---------|----------|-----------|----------|
| **P0 缺陷** | 统计未关闭数 | 0 个 | 0 个 | ≥1 个未关闭 |
| **P1 缺陷** | 统计未关闭数+变通方案 | 0 个 | ≤2 个有变通方案 | ≥3 个或无变通 |
| **用例通过率** | 通过数/总数 | ≥95% | 85%-95% | <85% |
| **核心路径** | 核心流程验证结果 | 全部通过 | 全部通过 | 有未通过 |
| **回归验证** | 回归通过率 | 全部通过 | ≥90% 通过 | <90% 通过 |
| **未验证区域** | 未覆盖的测试范围 | 无 | 有（低风险区域） | 有（高风险区域） |

### 整体风险判定规则

```
检查所有 6 个维度
  |
  +-> 任一维度为 🔴 -> 整体高风险
  +-> 无 🔴 且有 ⚠️ -> 整体中风险
  +-> 全部 ✅ -> 整体低风险
```

### 风险评估输出

评估结论必须包含：
1. **各维度评级表格**
2. **整体风险等级**
3. **主要风险点**：具体列出影响发布的风险
4. **缓解措施**：每个风险的应对方案
5. **建议**：发布/有条件发布/不发布

## 四、验收结论判断详细标准

### 结论类型

| 结论 | 含义 | 适用条件 |
|------|------|---------|
| **✅ 通过** | 可发布/可进入下一阶段 | 质量达标，风险可控 |
| **⚠️ 有条件通过** | 可发布但需附带条件 | 存在已知风险但已评估 |
| **❌ 不通过** | 不可发布/需继续修正 | 存在阻断问题或重大风险 |

### 通过条件详细判断

#### ✅ 通过（全部满足）

- [ ] P0 缺陷：0 个未关闭
- [ ] P1 缺陷：0 个未关闭
- [ ] 核心路径：100% 通过
- [ ] 用例通过率：≥95%
- [ ] 回归验证：全部通过
- [ ] 风险评估：整体低风险
- [ ] 检查清单：D1-D5 全部通过

#### ⚠️ 有条件通过（全部满足）

- [ ] P0 缺陷：0 个未关闭
- [ ] P1 缺陷：≤2 个（每个都有变通方案 + 明确修复计划）
- [ ] 核心路径：100% 通过
- [ ] 用例通过率：≥85%
- [ ] 已知风险已充分评估并记录
- [ ] 项目管理专家确认接受风险
- 附带条件：
  - 遗留缺陷的修复时间承诺
  - 发布后需要监控的指标
  - 回滚方案确认

#### ❌ 不通过（任一满足）

- 存在未关闭的 P0 缺陷
- P1 缺陷 ≥3 个且无变通方案
- 核心路径有未通过项
- 用例通过率 <85%
- 存在未验证的高风险区域
- 不通过报告必须包含：
  - 阻断原因清单
  - 修复要求和预计工作量
  - 预计可重新验证的时间

### 结论输出格式

```
## 验收结论

### 结论：{✅ 通过 / ⚠️ 有条件通过 / ❌ 不通过}

### 判断依据
| 维度 | 实际值 | 要求 | 结果 |
|------|--------|------|------|
| P0 缺陷 | {N} 个 | 0 个 | ✅/❌ |
| P1 缺陷 | {N} 个 | 0 个 | ✅/⚠️/❌ |
| 核心路径 | {N}/{N} 通过 | 100% | ✅/❌ |
| 用例通过率 | {N}% | ≥95% | ✅/⚠️/❌ |
| 回归验证 | {N}% | 100% | ✅/⚠️/❌ |
| 风险等级 | {等级} | 低/中 | ✅/⚠️/❌ |

### 遗留问题
{列出所有遗留问题和处理方案}

### 建议
{下一步行动建议}
```

## 五、测试结果汇总方法

### 汇总步骤

1. **统计用例执行结果**：总数/通过/失败/阻塞/跳过
2. **统计缺陷分布**：按级别、按模块、按状态
3. **计算关键指标**：通过率、缺陷密度、修复率
4. **标注关键发现**：测试过程中的重要发现
5. **评估模块质量**：每个模块的质量状态
6. **整理遗留问题**：未闭环的问题和处理方案

### 关键指标计算

| 指标 | 计算方式 | 说明 |
|------|---------|------|
| 用例通过率 | 通过数 / (总数 - 跳过数) | 排除合理跳过 |
| 缺陷密度 | 缺陷总数 / 功能点数 | 衡量质量水平 |
| 缺陷修复率 | 已修复数 / 缺陷总数 | 衡量修复进度 |
| 回归通过率 | 回归通过数 / 回归总数 | 衡量回归质量 |
| P0 关闭率 | P0 已关闭 / P0 总数 | 必须 100% |
