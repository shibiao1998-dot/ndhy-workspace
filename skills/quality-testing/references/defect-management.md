# 缺陷记录 + 复现路径 + 缺陷分级(P0-P3) + 跟踪闭环

## 一、缺陷记录规范

### 缺陷记录必填字段

每个缺陷必须包含以下信息，缺一不可：

| 字段 | 要求 | 反例 |
|------|------|------|
| **缺陷标题** | 一句话说清问题（模块+现象） | ❌ "有个 bug" |
| **级别** | P0/P1/P2/P3 | ❌ 不分级 |
| **复现步骤** | 编号步骤，具体到每个操作 | ❌ "操作一下就能看到" |
| **期望结果** | 应该发生什么 | ❌ "应该正常" |
| **实际结果** | 实际发生了什么 | ❌ "不正常" |
| **环境** | 浏览器/OS/设备/环境地址 | ❌ 不标环境 |
| **证据** | 截图/日志/网络请求 | ❌ 没有证据 |
| **影响范围** | 影响哪些功能/用户 | ❌ 不说影响 |
| **发现用例** | 关联的测试用例编号 | ❌ 不关联 |

### 缺陷标题写法

```
格式: [模块] 具体现象描述

好的标题:
  [用户管理] 编辑用户信息后保存，手机号字段被清空
  [订单列表] 按创建时间倒序排列时，跨月数据排序错乱
  [登录] 连续输错密码 5 次后，正确密码仍可登录

坏的标题:
  有个 bug
  保存有问题
  页面显示异常
```

### 复现路骤写法

```
好的复现步骤:
  1. 登录系统（账号：test01，角色：管理员）
  2. 进入 [用户管理] -> [用户列表]
  3. 点击用户 "张三" 的 [编辑] 按钮
  4. 修改 [姓名] 字段为 "李四"
  5. 点击 [保存] 按钮

坏的复现步骤:
  1. 编辑用户
  2. 保存
  （缺少登录、定位、具体操作）
```

## 二、缺陷分级详细标准

### P0 - 阻断（Critical/Blocker）

**定义**：核心功能完全不可用、数据丢失或损坏、安全漏洞。

**判断条件**（任一满足即为 P0）：
- 核心业务流程完全无法走通
- 数据丢失、数据被错误覆盖、数据泄露
- 安全漏洞（未授权访问、SQL注入、XSS 等）
- 系统崩溃/无响应/无法启动
- 资金类操作错误（支付、退款、结算）

**处理要求**：
- 立即通知相关开发专家
- 阻断发布，不允许带风险上线
- 升级项目管理专家知晓
- 修复后优先回归验证

**示例**：
- 登录页面 500 错误，所有用户无法登录
- 提交订单后金额计算错误
- 普通用户可访问管理员接口
- 批量删除操作将非选中数据一并删除

### P1 - 严重（Major）

**定义**：主要功能异常、数据错误、无变通方案可绕过。

**判断条件**（任一满足即为 P1）：
- 主要功能（非核心）无法正常使用
- 数据展示错误（影响业务判断）
- 权限控制漏洞（低风险场景）
- 状态机流转异常（非资金相关）
- 无法通过其他方式完成同一目标

**处理要求**：
- 当前迭代必须修复
- 不建议带此级别缺陷发布
- 修复后需回归验证

**示例**：
- 列表筛选结果错误，过滤条件未生效
- 用户角色切换后权限未更新
- 导出 Excel 数据与页面不一致
- 分页跳转到第 2 页显示第 1 页数据

### P2 - 一般（Minor）

**定义**：次要功能问题、有变通方案、体验降级但不影响业务。

**判断条件**：
- 次要功能异常但有变通方案
- UI/样式问题影响体验但不影响功能
- 提示文案错误或不清晰
- 性能轻微下降但可接受
- 排序/格式/显示问题

**处理要求**：
- 排入修复计划
- 可带风险发布（需在风险摘要中记录）
- 按优先级排入后续迭代

**示例**：
- 长文本未做截断，表格列宽错乱
- 成功提示文案有错别字
- 手机号格式校验提示不友好
- 列表默认排序不符合用户习惯

### P3 - 建议（Suggestion）

**定义**：优化建议、微小瑕疵、极端边界情况。

**判断条件**：
- 极端边界下的轻微问题
- 交互体验优化建议
- 不影响功能的视觉微调
- 用户几乎不会触发的场景

**处理要求**：
- 记录评估
- 按优先级排入后续迭代或产品 backlog
- 不影响发布决策

**示例**：
- 输入 10000 个字符时页面轻微卡顿
- 建议增加批量操作功能
- 建议调整按钮间距
- 极端网络下加载动画闪烁

### 分级决策流程图

```
发现问题
  |
  +-> 核心功能不可用？ --是--> P0
  |       |否
  +-> 数据丢失/错误/安全漏洞？ --是--> P0
  |       |否
  +-> 主要功能异常？ --是--> 有变通方案？ --否--> P1
  |       |否                    |是
  |       |                      +-> 影响业务判断？ --是--> P1 / --否--> P2
  +-> 功能可用但体验降级？ --是--> P2
  |       |否
  +-> 仅为优化建议/极端边界？ --是--> P3
```

## 三、缺陷状态流转

### 状态定义

| 状态 | 含义 | 操作者 |
|------|------|--------|
| **新建** | 缺陷刚被记录 | 测试专家 |
| **已确认** | 开发确认是有效缺陷 | 开发专家 |
| **修复中** | 正在修复 | 开发专家 |
| **待验证** | 修复完成，等待回归验证 | 开发专家 → 测试专家 |
| **已关闭** | 回归验证通过 | 测试专家 |
| **重开** | 回归验证未通过或问题复现 | 测试专家 |
| **延期** | 确认延期到后续迭代 | 项目管理专家 |
| **无效** | 非缺陷（设计如此/无法复现/重复） | 测试+开发确认 |

### 状态流转规则

```
新建 -> 已确认 -> 修复中 -> 待验证 -> 已关闭
  |         |                    |
  |         +-> 延期             +-> 重开 -> 修复中
  |         +-> 无效
  +-> 无效
```

### 跟踪闭环原则

1. **缺陷必须有归属**：每个缺陷指定回流对象（前端/后端/API/联调）
2. **修复必须有说明**：修复后需注明修复方案和影响范围
3. **验证必须有结果**：回归验证后更新状态（已关闭/重开）
4. **延期必须有理由**：延期需项目管理专家确认，记录理由和计划时间
5. **无效必须有依据**：标记无效需双方确认，记录判断依据

## 四、缺陷回流规则

### 回流对象判断

| 缺陷类型 | 回流对象 | 示例 |
|---------|---------|------|
| 页面渲染/交互/样式 | 前端开发专家 | 按钮不可点击、样式错乱 |
| 接口行为/数据处理/逻辑 | 后端开发专家 | 返回数据错误、状态流转异常 |
| 契约歧义/错误码不清 | API 设计专家 | 错误码含义不明、字段语义冲突 |
| 前后端不一致/链路问题 | 联调集成专家 | 前后端字段不对应、集成异常 |
| 代码结构/实现债 | 代码审查专家 | 发现明显的实现模式问题 |
| 环境/配置/部署 | 部署运维专家 | 环境配置错误、资源不足 |

### 回流信息要求

回流缺陷时，必须提供：

1. **缺陷编号和标题**
2. **完整复现路径**
3. **期望 vs 实际结果**
4. **证据（截图/日志）**
5. **级别和影响范围**
6. **建议的修复方向**（如果有）
