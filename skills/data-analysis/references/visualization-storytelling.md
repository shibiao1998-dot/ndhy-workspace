# 数据可视化与数据叙事

## 可视化 4 核心原则

| # | 原则 | 核心要求 |
|---|------|---------|
| 1 | **一张图只讲一件事** | 不要在一张图里塞 5 种信息——看的人记不住。讲 5 件事，做 5 张图 |
| 2 | **选对图表类型** | 见下方图表选择指南 |
| 3 | **去除不必要装饰** | 不要 3D、不要花哨配色、不要无意义背景。数据墨水比越高越好 |
| 4 | **高亮关键信息** | 用颜色、标注、箭头引导读者看到最重要的发现 |

## 图表选择指南

| 要表达的内容 | 推荐图表 | 注意事项 |
|-------------|---------|---------|
| 趋势变化 | 折线图 | 标注关键事件节点（如"X月上线新功能"） |
| 大小比较 | 柱状图 | 排序后更直观；避免截断 Y 轴 |
| 占比构成 | 饼图 / 堆叠柱状图 | 饼图分类不超过 5 个；多于 5 个用堆叠柱状图 |
| 分布情况 | 直方图 / 箱线图 | 直方图看整体形态；箱线图比较多组 |
| 相关关系 | 散点图 | 加趋势线辅助判断 |
| 漏斗转化 | 漏斗图 | 标注每步转化率和流失率 |
| 多维对比 | 雷达图 | 适度使用（维度 > 7 个时可读性差） |
| 地理分布 | 地图 | 用颜色深浅表示数值大小 |

## 数据叙事结构

数据叙事 = **数据 + 故事 + 行动建议**

### 标准结构：结论先行

```
1. 先说结论（30 秒能让人抓住重点）
2. 再给证据（数据和图表支撑结论）
3. 最后给建议（基于数据应该怎么做）
```

### 对比示例

**错误版**：
> "下面是过去 3 个月的数据报告。图 1 是用户增长趋势，图 2 是留存曲线，图 3 是功能使用分布，图 4 是区域分布......请大家看一下。"
>
> 问题：堆数据，不讲故事，没有结论，听的人不知道该关注什么

**正确版**：
> "核心发现：新用户的 7 日留存率从 3 月的 38% 提升到了 5 月的 52%——上升了 14 个百分点，主要原因是 4 月上线的新手任务引导。但问题是：30 日留存依然只有 25%，说明用户'上手了但没留住'。建议：下一步聚焦'上手后 2-4 周'的使用深化策略。"
>
> 有结论、有原因、有问题、有建议——一段话就讲清楚了

### 数据叙事 5 步法

1. **抓住一个核心发现**——不要贪多
2. **构建 "so what" 链条**——数据 -> 说明了什么 -> 意味着什么 -> 应该怎么做
3. **用对比创造张力**——"留存从 38% 到 52%"比"留存是 52%"有力得多
4. **用具体数字而非形容词**——"提升了 14 个百分点"比"显著提升"有说服力
5. **以行动收尾**——不要让听众想"so what"

## 4 类数据看板设计

### 看板 1：全局概览

| 内容 | 说明 |
|------|------|
| 北极星指标 + 趋势 | 最重要的那个指标 |
| 4-6 个关键指标当前值和变化 | 用箭头/颜色标注环比变化 |
| 异常指标红色高亮 | 一眼看出哪里有问题 |
| **受众** | 管理层——1 分钟看完，知道大盘情况 |
| **刷新频率** | 每日 |

### 看板 2：用户生命周期

| 内容 | 说明 |
|------|------|
| 新增 -> 激活 -> 留存 -> 流失的漏斗和趋势 | 完整生命周期视图 |
| 各阶段用户量和转化率 | 找薄弱环节 |
| 同期群留存表 | 判断留存是否在改善 |
| **受众** | 产品和运营团队 |
| **刷新频率** | 每周 |

### 看板 3：功能使用分析

| 内容 | 说明 |
|------|------|
| 各功能模块的使用率和趋势 | 哪些功能活、哪些死 |
| 核心功能的使用深度 | 不只看有没有用，看用了多深 |
| 新功能的采纳率 | 新功能是否被接受 |
| **受众** | 产品团队 |
| **刷新频率** | 每周 |

### 看板 4：区域/学校运营

| 内容 | 说明 |
|------|------|
| 各区域覆盖学校数和活跃度 | 区域运营概览 |
| 学校健康度评分 | 快速定位需关注的学校 |
| 重点学校跟踪 | 标杆学校和高危学校 |
| **受众** | 运营和客成团队 |
| **刷新频率** | 每周 |

## 分受众汇报策略

### 给管理层

| 原则 | 说明 |
|------|------|
| 只讲结论和建议 | 不展示分析过程 |
| 一页纸搞定 | 信息密度高但不复杂 |
| 聚焦 3 个问题 | 现状如何？趋势是好是坏？需要做什么决策？ |
| 用绝对值 + 变化量 | "活跃学校 350 所，环比 +12%"比"活跃率 73.2%"直观 |

### 给产品/运营团队

| 原则 | 说明 |
|------|------|
| 结论 + 关键数据 + 可执行建议 | 需要看到分析逻辑 |
| 展示"为什么这么判断" | 给出推理过程 |
| 粒度可以更细 | 按功能/用户群/渠道拆分 |
| 附带优先级建议 | 排好序，不要让受众自己排 |

### 给技术团队

| 原则 | 说明 |
|------|------|
| 聚焦技术相关数据 | 性能、错误率、埋点验证 |
| 提供具体数据定义和口径 | 精确到字段级别 |
| 不需要讲太多业务背景 | 技术人员需要的是"做什么"而非"为什么" |
| 用技术语言 | API 响应时间而非"系统反应速度" |

## 可视化质量检查清单

| # | 检查项 | 通过条件 |
|---|--------|---------|
| 1 | 图表类型是否匹配数据关系？ | 趋势用折线、对比用柱状、占比用饼图 |
| 2 | 每张图是否只讲一件事？ | 不堆砌多种无关信息 |
| 3 | 关键信息是否被高亮？ | 有颜色/标注引导注意力 |
| 4 | 坐标轴是否有标题和单位？ | 不让读者猜测 |
| 5 | Y 轴是否从 0 开始？ | 除非有合理理由截断（需标注） |
| 6 | 配色是否清晰且一致？ | 同一指标全文同一颜色 |
| 7 | 是否去除了多余装饰？ | 无 3D、无花哨背景 |
| 8 | 非专业读者能否在 10 秒内理解？ | 如不能，简化或加注释 |
