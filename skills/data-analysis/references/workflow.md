# 数据分析 — 6 步标准流程

```
Step 1: 明确问题
  |  业务方想要回答什么问题？需要做什么决策？
  |  -> 如果问题模糊，先帮业务方梳理清楚再动手
  |  -> 用 SMART 框架校验：具体/可量化/可行动/相关/有时间框架
  |  -> 问题不清楚 -> 先花 5 分钟和需求方对齐（不猜测、不假设）
  |  -> 产出：分析需求确认（一段话）
  |  -> 详细方法 -> references/problem-framing.md
  |
Step 2: 确定分析框架
  |  -> 用什么方法来回答这个问题？（14 种方法速查见 analysis-methods.md）
  |  -> 需要哪些数据？数据是否可得？
  |  -> 预期分析结果会是什么样？（先有假设，再验证）
  |  -> 产出：分析方案（要点列表或脑图）
  |  -> 详细方法 -> references/analysis-methods.md
  |
Step 3: 获取和验证数据
  |  -> 从数据系统中提取所需数据
  |  -> 检查数据质量：有无缺失？有无异常值？口径是否一致？
  |  -> 数据有问题就停——不要在脏数据上做分析
  |  -> 产出：清洗后的分析数据集
  |  -> 埋点规划 -> references/checklists.md 埋点规划指南
  |
Step 4: 分析和发现
  |  -> 按分析框架执行分析
  |  -> 过程中可能发现预期之外的洞察——不要忽略
  |  -> 区分"确定的结论"和"需要进一步验证的假设"
  |  -> 区分"事实"和"推测"——报告中必须明确标注
  |  -> 产出：分析发现（关键图表 + 文字总结）
  |
Step 5: 形成结论和建议
  |  -> 结论要明确，不要模棱两可
  |  -> 建议要可执行，不要"需要进一步研究"
  |  -> 如果数据不足以给出确定结论，坦诚说明并给出下一步验证计划
  |  -> 产出：分析报告/结论备忘录
  |  -> 模板 -> references/templates.md
  |
Step 6: 沟通和推动落地
     -> 针对不同受众调整汇报方式（管理层/产品/技术）
     -> 跟踪建议是否被执行
     -> 执行后的效果追踪——闭环
     -> 产出：决策推动 + 效果追踪
     -> 详细方法 -> references/visualization-storytelling.md
```

## 协作接口

### 上游输入

| 来源 | 接收内容 | 使用方式 |
|------|---------|---------|
| 产品/运营/管理层 | 业务问题、决策需求 | 转化为数据问题，明确分析框架 |
| 数据工程/埋点 | 原始数据、埋点日志 | 数据清洗、质量验证后分析 |
| 增长/产品团队 | 实验需求 | 实验设计与效果评估 |
| 各业务方 | 指标需求 | 指标体系设计与监控 |

### 下游输出

| 接收方 | 传递内容 | 约束关系 |
|--------|---------|---------|
| 管理层 | 核心结论 + 建议（一页纸） | 不展示分析过程，只讲结论和决策 |
| 产品/运营 | 结论 + 数据 + 可执行建议 | 需要看到分析逻辑 |
| 技术团队 | 技术数据 + 埋点规范 + 口径定义 | 提供具体数据定义 |
| 客成/运营 | 学校健康度 + 运营建议 | 需要可操作的分级和行动指引 |

### 升级规则

| 场景 | 升级给 |
|------|--------|
| 业务问题不清晰、分析目标冲突 | 项目管理/产品经理 |
| 数据缺失、埋点不足 | 数据工程/技术团队 |
| 实验设计涉及产品改动 | 产品经理 |
| 分析结论涉及战略决策 | 管理层 |
