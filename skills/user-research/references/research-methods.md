# 研究方法详解

## §1 采集手段详表

### 手段 1：公开数据挖掘

| 维度 | 说明 |
|------|------|
| **具体操作** | 搜索行业报告、统计数据、政策文件、市场分析报告 |
| **获取什么** | 用户规模、群体特征、市场趋势、宏观环境 |
| **常用来源** | 政府统计局、行业协会报告、咨询公司白皮书（艾瑞、36氪、亿欧等）、国家/地方政策文件 |
| **优势** | 数据权威、覆盖面广、易获取 |
| **局限** | 粒度粗，可能滞后，不反映个体感受 |
| **使用要点** | 标注数据年份和来源机构；>2 年数据标注"可能已过时" |

### 手段 2：用户声音采集

| 维度 | 说明 |
|------|------|
| **具体操作** | 搜索/爬取产品评论、论坛帖子、社交媒体讨论、投诉记录、应用商店评分评论 |
| **获取什么** | 真实痛点、不满、期望、使用反馈、情感态度 |
| **常用来源** | 应用商店（App Store/Google Play）、知乎、微博、小红书、贴吧、专业论坛、投诉平台（黑猫投诉等） |
| **优势** | 最接近用户真实感受，语言生动具体 |
| **局限** | 存在幸存者偏差（只有发声用户可见）、可能被水军污染 |
| **使用要点** | 注意区分真实反馈和营销内容；声明"活跃用户的声音可能不代表全部" |

### 手段 3：竞品用户体验分析

| 维度 | 说明 |
|------|------|
| **具体操作** | 分析竞品页面、功能设计、用户评价、口碑对比 |
| **获取什么** | 用户对同类产品的态度、偏好、不满、期望 |
| **常用来源** | 竞品官网、应用商店评价、第三方评测、社交媒体讨论 |
| **优势** | 直接了解市场认知和用户期望基线 |
| **局限** | 只反映竞品用户视角，不一定适用于目标产品 |
| **使用要点** | 分析竞品优劣势时标注是"竞品用户"还是"通用用户"的视角 |

### 手段 4：学术/行业研究整合

| 维度 | 说明 |
|------|------|
| **具体操作** | 获取学术论文、白皮书、专家观点、行业会议记录 |
| **获取什么** | 深层用户行为规律、理论框架、经过验证的研究结论 |
| **常用来源** | Google Scholar、知网、SSCI 期刊、行业峰会资料 |
| **优势** | 理论基础扎实，研究方法论可靠 |
| **局限** | 可能过于学术化，与实际产品场景有距离 |
| **使用要点** | 提取与项目相关的结论，避免照搬学术语言 |

### 手段 5：已有数据二次分析

| 维度 | 说明 |
|------|------|
| **具体操作** | 分析项目方提供的历史数据、反馈记录、使用日志 |
| **获取什么** | 数据驱动的用户行为洞察、使用模式、流失原因 |
| **常用来源** | 项目方内部数据库、用户反馈表、运营数据、客服记录 |
| **优势** | 最贴近实际业务，数据粒度最细 |
| **局限** | 依赖项目方提供，可能不完整或有偏差 |
| **使用要点** | 先评估数据质量和覆盖范围，再进行分析 |

### 采集手段组合建议

| 研究深度 | 推荐组合 |
|---------|---------|
| 轻量级 | 公开数据 + 用户声音（2 种即可） |
| 标准级 | 公开数据 + 用户声音 + 竞品分析（3-4 种） |
| 深度级 | 全部 5 种，重点深入 |

## §2 需求推演方法详解

> **核心原则：推演是数据不足时的补位手段，不是替代品。所有推演结论必须标注"推演"。**

### 方法 A：角色-场景-任务链推演

**执行步骤：**

1. **定义角色**（谁）：从用户画像中选取一个具体角色
2. **构建场景**（什么情境）：描述角色所处的具体环境、时间、状态
3. **推演任务**（需要完成什么）：在该场景下，角色需要完成的具体任务
4. **识别障碍**（会遇到什么问题）：完成任务过程中的阻碍和摩擦
5. **推导需求**（因此需要什么）：为消除障碍所需的功能或服务

**示例：**
```
角色：县城初中英语教师，40 岁，技术素养中等
场景：晚上 9 点，在家用手机准备明天的课
任务：找到一个适合学生水平的听力练习
障碍：搜索结果太多无法筛选，不确定难度是否匹配
需求：按年级/难度智能推荐听力资源 [推演]
```

**使用要点：**
- 场景描述要具体到时间、地点、设备、心理状态
- 每条推演链标注置信度（基于画像数据的丰富程度）
- 不同角色走不同的推演链，不要一个链覆盖所有人

### 方法 B：Jobs-to-be-Done 推演

**执行步骤：**

1. 用 JTBD 句式描述用户的"工作"：
   ```
   当我处于 [某种情境] 时，
   我想要 [完成某个任务]，
   这样我就能 [获得某种结果]。
   ```
2. 为每个角色列出 3-5 个核心 Job
3. 区分功能性 Job（完成任务）、情感性 Job（感受好）、社会性 Job（被认可）
4. 从 Job 推导出需求

**示例：**
```
当我（教师）在期末复习阶段时，
我想要快速生成一份覆盖本学期重点的练习卷，
这样我就能节省 2 小时的出卷时间去做个别辅导。[推演]
```

**使用要点：**
- 情境要具体，不要写"当我需要的时候"
- "这样我就能"部分揭示真实价值，是分析重点
- 一个 Job 可能对应多个需求解法

### 方法 C：痛点反推法

**执行步骤：**

1. **列出已知痛点**：从用户声音、竞品分析中收集
2. **追问根因**：对每个痛点连续追问"为什么会这样"（至少 3 层）
3. **推导深层需求**：从根因推导用户真正需要什么
4. **区分层次**：表面要求（用户说的）vs 真实需要（用户实际需要的）

**示例：**
```
痛点：教师反映"批改作业太花时间"
  → 为什么？手动逐题批改，每份作业 5-8 分钟
    → 为什么不能更快？题型多样，无法模板化
      → 深层需求：不是"批改加速"，而是"快速了解每个学生的掌握情况" [推演]
表面要求：自动批改工具
真实需要：学情分析仪表盘
```

**使用要点：**
- 至少追问 3 层"为什么"才触及深层需求
- 表面要求和真实需要都记录，都有价值
- 深层需求的推演必须标注逻辑链，让读者可复核

## §3 研究深度详细说明

### 轻量级研究

**适用场景：**
- 小型功能优化
- 方向已明确，需快速验证
- 时间紧急的小需求

**流程要求：**
- Step 1-2 快速执行，Step 3 简化（主题归类即可）
- Step 4 聚焦核心问题
- Step 5 仅输出简要画像 + 核心痛点清单
- Step 6 口头或简短文档交付

**质量底线：**
- 至少 2 种信息源
- 假设必须标注
- 局限性声明可简短

### 标准级研究

**适用场景：**
- 新功能开发
- 中等规模项目
- 需要理解用户行为路径

**流程要求：**
- 6 步完整执行
- Step 2 至少 3 种采集手段
- Step 5 输出完整画像 + 旅程地图 + 需求矩阵
- Step 6 正式文档交付

**质量底线：**
- 三角验证覆盖关键结论
- 完整的假设标注和时效声明
- 输出物使用标准模板

### 深度级研究

**适用场景：**
- 新产品规划
- 战略级项目
- 进入新市场/新用户群

**流程要求：**
- 6 步深入执行
- Step 2 使用全部 5 种采集手段
- Step 3 详细的交叉验证，矛盾处需多方印证
- Step 5 多用户群画像 + 多场景旅程地图 + 竞品对比 + 机会分析
- Step 6 全套研究报告，附完整数据源清单

**质量底线：**
- 所有结论都有交叉验证
- 竞品分析覆盖至少 3 个主要竞品
- 机会分析需量化（可行时）
- 局限性声明需详细
