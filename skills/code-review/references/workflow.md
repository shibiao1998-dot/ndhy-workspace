# 代码审查 — 8步标准流程

## 与 code-quality-gate 的关系

| Skill | 职责 | 使用者 |
|-------|------|--------|
| **code-quality-gate** | 审查**流程编排**：何时审查、谁来审查、双层审查机制、任务分流 | Leader（调度视角） |
| **code-review**（本技能） | 审查**执行方法**：怎么审、审什么、怎么分级、怎么判定 | 代码审查专家（执行视角） |

两者互补：`code-quality-gate` 决定"是否需要审查、派谁审查"；`code-review` 决定"怎么审查、结论是什么"。

## 流程总览

```
Step 1: 理解变更上下文
  |  消化变更目标、代码 diff、上游约束（API 契约/数据模型/架构约束）
  |  -> 确认: 变更范围、影响模块、验收标准、历史上下文
  |  -> 信息不足 -> 要求补充（不猜测、不假设、不硬审）
  |
Step 2: 审查契约一致性
  |  -> 实现是否符合 API 契约（路径/方法/参数/响应/错误码）
  |  -> 实现是否符合数据模型（字段语义/约束/关系）
  |  -> 实现是否符合架构约束（分层/调用方向/依赖规则）
  |  -> 详细方法 -> references/correctness-review.md §契约一致性
  |
Step 3: 审查逻辑正确性与边界条件
  |  -> 核心业务逻辑是否正确（状态转换/计算/条件分支）
  |  -> 边界条件是否覆盖（空值/零值/极值/溢出）
  |  -> 异常路径是否完整（错误处理/回滚/降级/超时）
  |  -> 并发场景是否安全（竞态/死锁/重入）
  |  -> 详细方法 -> references/correctness-review.md §逻辑与边界
  |
Step 4: 审查结构与可维护性
  |  -> 分层是否清晰（handler/service/repository 职责不混）
  |  -> 职责边界是否合理（单一职责/模块边界）
  |  -> 是否存在重复逻辑（copy-paste/散落的相似实现）
  |  -> 命名是否自解释、可读性是否达标
  |  -> 复杂度是否可控（嵌套深度/方法长度/参数数量）
  |  -> 详细方法 -> references/structure-maintainability.md
  |
Step 5: 审查安全与性能风险
  |  -> 输入校验是否完整（注入/XSS/路径遍历）
  |  -> 权限控制是否到位（认证/授权/越权访问）
  |  -> 敏感数据处理是否合规（日志脱敏/传输加密/存储加密）
  |  -> 是否存在性能风险（N+1 查询/无分页/大对象/内存泄漏）
  |  -> 资源使用是否安全（连接泄漏/文件句柄/线程池）
  |  -> 详细方法 -> references/security-performance.md
  |
Step 6: 审查测试充分性
  |  -> 核心逻辑是否有单元测试覆盖
  |  -> 关键路径是否有集成测试
  |  -> 边界条件和异常路径是否有测试
  |  -> 测试质量是否达标（不只断言"不报错"）
  |  -> 详细方法 -> references/test-adequacy.md
  |
Step 7: 分级问题 + 形成放行决策
  |  -> 按 4 级分级所有发现（阻断/需修改/建议/备注）
  |  -> 识别结构性风险，判断是否需要升级
  |  -> 按放行决策树判定：通过 / 有条件通过 / 修改后再审 / 拒绝
  |  -> 生成复审清单（如非直接通过）
  |  -> 问题分级 -> 见下方 §问题分级框架
  |  -> 放行决策 -> 见下方 §放行决策树
  |
Step 8: 输出审查交付包
     -> 6 个标准文档（见下方）
     -> 过审查检查清单（见下方）
     -> 按模板组织 -> references/templates.md
```

## 5层深度边界速查

| 层级 | 主责角色 | 审查专家的关系 |
|------|---------|---------------|
| **需求与设计定义层** | 产品/API/数据库/架构专家 | 消费其输出作为审查基准，不替代其决策 |
| **实现层** | 后端/前端开发专家 | 审查其产出，不替代其实现 |
| **审查与质量闸门层** | **代码审查专家（核心）** | 主责：正确性、契约、边界、可维护、安全、性能、测试 |
| **完整验证与验收层** | 测试专家 | 提出补测建议，不替代验收 |
| **上线与运行层** | 运维专家 | 提出上线风险提示，不替代部署 |

**铁律**：审查专家主责第 3 层。消费第 1-2 层输入，对第 4-5 层提出风险提示，但不替代其职责。

## 问题分级框架

| 级别 | 标识 | 含义 | 处理要求 | 示例 |
|------|------|------|---------|------|
| **P0 阻断** | 🔴 | 必须修复才能放行 | 修复后再审 | 逻辑错误、数据丢失风险、安全漏洞、契约违反 |
| **P1 需修改** | 🟠 | 应当修复，不修复需说明理由 | 本轮修复或附理由延后 | 缺少错误处理、边界未覆盖、测试缺失 |
| **P2 建议** | 🟡 | 改了更好，不改不阻断 | 开发者自行决定 | 命名优化、代码简化、注释补充 |
| **P3 备注** | 🔵 | 记录观察，无需行动 | 仅记录 | 技术债标记、后续优化方向、设计讨论点 |

**分级铁律**：
- 每个问题必须有且仅有一个级别
- 不确定时宁可高估一级
- P0/P1 必须附具体修复建议
- P2/P3 不得阻断放行

> 详细分级标准和边界案例 -> [checklists.md](checklists.md) §问题分级详细标准

## 放行决策树

```
统计审查发现
  |
  |- 存在 P0 阻断？
  |   -> 是 -> ❌ 拒绝放行（修复所有 P0 后再审）
  |
  |- 存在 P1 需修改？
  |   |- 数量 ≤ 3 且无关联风险？
  |   |   -> ✅ 有条件通过（附修复清单，修复后可不再审）
  |   |- 数量 > 3 或存在关联风险？
  |       -> ⚠️ 修改后再审（生成复审清单）
  |
  |- 仅 P2/P3？
  |   -> ✅ 通过（附建议清单）
  |
  +- 零发现？
      -> ✅ 通过
```

### 特殊决策

| 场景 | 决策 |
|------|------|
| 发现结构性风险（跨模块/架构级） | 暂停审查 → 升级技术架构专家 |
| 契约本身存在歧义 | 暂停审查 → 回流 API/数据库专家澄清 |
| 测试严重不足且变更影响核心路径 | 有条件通过 → 同步测试专家补测 |
| diff 过大（>500 行核心变更） | 建议拆分 → 升级项目管理专家协调 |
| 上下文严重不足 | 拒绝审查 → 要求补充变更说明和上游约束 |

## 工作模式决策树

```
收到代码审查任务
  |- 小范围变更（配置/文案/单文件修改 < 50 行）
  |   -> 快速审查，单独执行
  |
  |- 明确边界的功能变更（单模块/单接口/单组件）
  |   -> 标准审查，作为 subagent 被调用（默认模式）
  |
  +- 大范围变更 / 跨模块 / 核心系统
      |- 变更横跨前后端 + 契约 + 数据模型？
      |   -> 深度审查 + 建议拉 Agent Team
      |- 发现结构性问题需多角色判断？
      |   -> 暂停 → 升级架构专家 / 拉 Agent Team
      +- 否 -> 深度审查，subagent 模式
```

## 协作接口

### 上游输入

| 来源 | 接收内容 | 使用方式 |
|------|---------|---------|
| Leader / 项目管理 | 审查任务、验收标准、优先级 | 确定审查范围和深度 |
| 后端/前端开发专家 | 代码 diff、变更说明、自测结果 | 审查对象 |
| API 设计专家 | 接口契约 | 契约一致性审查基准 |
| 数据库设计专家 | 数据模型、字段字典 | 模型一致性审查基准 |
| 技术架构专家 | 架构约束、分层规则 | 结构审查基准 |

### 下游输出

| 接收方 | 传递内容 | 场景 |
|--------|---------|------|
| 开发专家 | 审查报告 + 修改建议 | P0/P1 问题回流修复 |
| 测试专家 | 补测建议 | 测试充分性不足 |
| 架构专家 | 结构性风险报告 | 识别到架构级问题 |
| API/数据库专家 | 契约歧义反馈 | 契约本身需要澄清 |
| Leader | 放行决策 + 风险摘要 | 审查结论交付 |

### 升级规则

| 场景 | 升级给 |
|------|--------|
| 实现逻辑问题需修复 | 后端/前端开发专家 |
| 契约定义不清或冲突 | API 设计专家 |
| 数据模型被破坏或语义偏移 | 数据库设计专家 |
| 结构性复杂度/架构风险 | 技术架构专家 |
| 测试严重不足 | 测试专家 |
| 范围过大/反复返工/排期影响 | 项目管理专家 |

## 触发时首要动作

无论如何触发，第一步永远是：

> "这次变更要做什么？影响哪些模块？上游约束（契约/模型/架构）是什么？验收标准是什么？"

-> 判断审查深度 -> 消化输入 -> 按 8 步流程执行

**教育行业项目**：额外读取 [references/education-review.md](references/education-review.md) 了解行业特殊审查关注点。
