# 测试充分性审查

本文件覆盖测试充分性审查方向：覆盖评估、测试质量判断、补测建议。

---

## 一、测试覆盖评估

### 审查目标

评估当前变更的测试覆盖是否足够支撑安全交付，识别关键的测试缺口。

### 覆盖维度

#### 1. 单元测试覆盖

| 检查项 | 具体审查内容 |
|--------|-------------|
| 核心业务逻辑 | Service 层关键方法是否有单元测试 |
| 计算逻辑 | 涉及数值计算的方法是否有充分的输入输出验证 |
| 条件分支 | if/else/switch 的各分支是否都有测试覆盖 |
| 数据转换 | DTO ↔ Entity ↔ VO 转换逻辑是否有测试 |
| 校验逻辑 | 输入校验规则是否有正向和反向测试 |
| 工具方法 | 公共工具类/方法是否有测试 |

#### 2. 集成测试覆盖

| 检查项 | 具体审查内容 |
|--------|-------------|
| 关键接口 | 核心 API 接口是否有端到端的集成测试 |
| 数据库操作 | Repository 层的 CRUD 操作是否有集成测试 |
| 事务场景 | 涉及事务的操作是否测试了提交和回滚 |
| 外部依赖 | 外部服务调用是否有 mock 测试或集成测试 |
| 权限控制 | 认证授权链路是否有集成测试 |

#### 3. 边界与异常覆盖

| 检查项 | 具体审查内容 |
|--------|-------------|
| 空值/零值 | null、空字符串、0 值输入是否有测试 |
| 极值 | 边界值（最大、最小、临界）是否有测试 |
| 异常路径 | 数据库异常、网络超时、数据不存在等是否有测试 |
| 错误响应 | 各种错误码和错误消息是否有验证 |
| 并发场景 | 关键的并发场景是否有并发测试 |

### 覆盖等级评估

| 等级 | 定义 | 对放行的影响 |
|------|------|-------------|
| **充分** | 核心逻辑+关键边界+异常路径均有测试覆盖 | 正常放行 |
| **基本** | 核心逻辑有测试，但边界和异常覆盖不足 | 有条件通过，标记补测 |
| **不足** | 核心逻辑测试缺失或严重不足 | 建议补测后再审 |
| **缺失** | 无任何测试 | 核心模块变更：阻断；非核心：标记 P1 |

## 二、测试质量判断

### 审查目标

评估已有测试的质量是否达标，确保测试能真正发现问题，而非"为了覆盖率而写"。

### 质量维度

#### 1. 断言质量

| 检查项 | 好的实践 | 坏的实践 |
|--------|---------|---------|
| 断言具体性 | `expect(result.name).toBe("张三")` | `expect(result).toBeTruthy()` |
| 断言完整性 | 验证返回值的关键字段 | 只验证"不报错" |
| 负向断言 | `expect(() => fn()).toThrow(SpecificError)` | 只测正向，不测异常 |
| 状态断言 | 验证操作后的状态变化 | 只验证方法被调用 |

#### 2. 测试独立性

| 检查项 | 具体审查内容 |
|--------|-------------|
| 相互独立 | 每个测试用例是否可独立运行，不依赖其他测试的执行顺序 |
| 数据隔离 | 测试数据是否独立，不共享可变状态 |
| 环境清理 | setUp/tearDown 是否正确清理测试环境 |
| 外部隔离 | 外部依赖是否被正确 mock/stub |

#### 3. 测试可维护性

| 检查项 | 具体审查内容 |
|--------|-------------|
| 测试命名 | 测试名称是否清晰表达"在什么条件下，做什么操作，期望什么结果" |
| 测试结构 | 是否遵循 Arrange-Act-Assert 或 Given-When-Then 模式 |
| 测试数据 | 测试数据是否有工厂方法或 builder，而非硬编码 |
| 重复代码 | 测试间是否有合理的共用辅助方法 |

#### 4. 常见测试质量问题

| 问题 | 示例 | 默认分级 |
|------|------|---------|
| 空断言 | 测试方法中没有 assert/expect | 🟠 P1 |
| 只断言不报错 | `expect(fn()).not.toThrow()` 无其他断言 | 🟠 P1 |
| 测试间耦合 | 测试 B 依赖测试 A 创建的数据 | 🟡 P2 |
| 测试名无意义 | `test("test1")`, `it("should work")` | 🟡 P2 |
| mock 过度 | mock 了被测方法的核心逻辑，测试形同虚设 | 🟠 P1 |
| 测试数据硬编码 | 每个测试中重复构造相同的复杂对象 | 🔵 P3 |

## 三、补测建议生成

### 生成策略

当识别到测试不足时，应输出具体的补测建议，格式如下：

#### 补测建议模板

```markdown
### 补测建议

| # | 被测目标 | 测试类型 | 场景描述 | 优先级 | 理由 |
|---|---------|---------|---------|--------|------|
| 1 | {方法/接口} | 单元/集成 | {具体场景} | 高/中/低 | {为什么需要} |
```

### 补测优先级判断

| 优先级 | 适用场景 | 说明 |
|--------|---------|------|
| **高** | 核心业务逻辑无测试、支付/权限相关无测试 | 必须在本次交付前补充 |
| **中** | 边界条件和异常路径无测试 | 建议本迭代内补充 |
| **低** | 非核心模块测试不足、测试质量可提升 | 可在后续迭代中补充 |

---

## 审查策略

### 评估顺序

1. **先看有没有测试** → 覆盖等级评估
2. **再看测试质量** → 断言是否有效、测试是否独立
3. **最后给补测建议** → 按优先级排序

### 边界判断

- 核心模块完全无测试 → **P1 需修改，建议补测后再审**
- 有测试但质量不足（空断言/mock 过度） → **P1 需修改**
- 边界和异常覆盖不足 → **P2 建议，标记补测方向**
- 测试命名和结构问题 → **P3 备注**
- 需要完整测试计划和验证 → **同步测试专家，不替代其职责**

### 不属于代码审查的范畴

| 不做 | 说明 |
|------|------|
| 编写完整测试用例 | 审查者指出缺口和方向，不替开发者写测试 |
| 执行测试并验证结果 | 执行和验证是开发者/测试专家的职责 |
| 设计完整测试策略 | 整体测试策略由测试专家主导 |
| 评估测试覆盖率数值 | 覆盖率工具和CI是基础设施，不是审查者手动计算 |
