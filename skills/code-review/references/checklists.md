# 完整检查清单与分级详解

本文件包含：完整审查检查清单、问题分级详细标准、反模式详解、放行判断指南。

---

## 一、完整审查检查清单

### A. 审查前准备

| # | 检查项 | 通过条件 |
|---|--------|---------|
| A1 | 变更目标已明确 | 知道这次变更要做什么、为什么做 |
| A2 | 变更范围已确认 | 知道影响哪些文件、模块、接口 |
| A3 | 上游约束已获取 | API 契约/数据模型/架构约束可查阅 |
| A4 | 验收标准已明确 | 知道"做到什么程度算合格" |
| A5 | 审查深度已决定 | 根据变更范围和重要性确定快速/标准/深度 |

### B. 契约一致性（Step 2）

| # | 检查项 | 通过条件 |
|---|--------|---------|
| B1 | API 路径和方法一致 | 实现的路由与契约定义完全一致 |
| B2 | 请求参数/体一致 | 字段名、类型、必填规则与契约一致 |
| B3 | 响应结构一致 | 返回字段、类型、嵌套与契约一致 |
| B4 | 错误码/状态码一致 | 错误场景的响应与契约定义一致 |
| B5 | 数据模型一致 | 字段定义、类型映射、约束与模型一致 |
| B6 | 架构约束一致 | 分层、依赖方向、中间件使用合规 |

### C. 逻辑正确性（Step 3）

| # | 检查项 | 通过条件 |
|---|--------|---------|
| C1 | 核心业务逻辑正确 | 条件判断、计算、状态转换逻辑正确 |
| C2 | 控制流正确 | 循环终止、异步顺序、事务边界正确 |
| C3 | 数据流正确 | 输入→处理→输出完整，无数据丢失 |
| C4 | 类型安全 | 无类型转换丢失精度或语义 |

### D. 边界与异常（Step 3）

| # | 检查项 | 通过条件 |
|---|--------|---------|
| D1 | 空值处理 | null/undefined/空字符串/空集合已处理 |
| D2 | 极值处理 | 超大/超小/零值等边界已处理 |
| D3 | 异常路径完整 | 数据库异常/网络超时/数据不存在等已处理 |
| D4 | 错误处理质量 | 不吞没异常、不泄露内部信息、资源正确释放 |
| D5 | 幂等性 | 写操作（尤其支付类）有幂等保护 |

### E. 结构与可维护性（Step 4）

| # | 检查项 | 通过条件 |
|---|--------|---------|
| E1 | 分层合规 | handler/service/repository 职责不越界 |
| E2 | 职责单一 | 每个类/方法职责可一句话说清 |
| E3 | 无重复逻辑 | 相同业务规则不在 3 处以上独立实现 |
| E4 | 复杂度可控 | 无超长方法(>100行)、超深嵌套(>4层) |
| E5 | 命名自解释 | 变量/方法/类名可读、无歧义 |
| E6 | 依赖关系健康 | 无循环依赖，依赖方向正确 |

### F. 安全与性能（Step 5）

| # | 检查项 | 通过条件 |
|---|--------|---------|
| F1 | 无注入风险 | 使用参数化查询，无字符串拼接 |
| F2 | 授权到位 | 操作前校验权限，无越权风险 |
| F3 | 敏感数据安全 | 日志脱敏、密码加密、密钥不硬编码 |
| F4 | 无 N+1 查询 | 循环中无逐条数据库查询 |
| F5 | 有分页保护 | 列表查询有分页限制 |
| F6 | 资源安全 | 连接/文件/线程在异常路径正确释放 |
| F7 | 外部调用有超时 | HTTP/RPC 调用设置了超时 |
| F8 | 并发安全 | 共享状态有同步保护，关键操作有幂等/锁 |

### G. 测试充分性（Step 6）

| # | 检查项 | 通过条件 |
|---|--------|---------|
| G1 | 核心逻辑有测试 | Service 层关键方法有单元测试 |
| G2 | 关键接口有测试 | 核心 API 有集成测试 |
| G3 | 边界有测试 | 空值/极值/异常输入有测试覆盖 |
| G4 | 测试断言有效 | 非空断言，验证关键字段和状态 |

### H. 分级与决策（Step 7-8）

| # | 检查项 | 通过条件 |
|---|--------|---------|
| H1 | 问题已全部分级 | 每个发现有且仅有 P0-P3 一个级别 |
| H2 | P0/P1 有修复建议 | 阻断和需修改的问题附具体修复方向 |
| H3 | 放行决策已明确 | 通过/有条件通过/修改后再审/拒绝 |
| H4 | 复审清单已生成 | 非直接通过的情况附复审清单 |
| H5 | 结构性风险已识别 | 如有架构级风险，已标记需升级 |

## 二、问题分级详细标准

### P0 阻断 🔴

**定义**：如果不修复就交付，将导致系统不可用、数据丢失、安全事故或严重的功能错误。

**判定标准**（满足任一即为 P0）：
- 核心业务逻辑错误，影响主要功能
- 数据丢失或数据损坏风险
- 安全漏洞（SQL 注入、越权访问、敏感数据泄露）
- 违反 API 契约的关键约定（响应结构、状态码）
- 资源泄漏会导致系统崩溃
- 并发操作可能导致数据不一致（涉及资金、库存等）

**处理**：必须修复，修复后需要复审。

### P1 需修改 🟠

**定义**：不修复不会导致立即崩溃，但会增加风险、降低质量或导致后续返工。

**判定标准**（满足任一即为 P1）：
- 边界条件未处理（空值可能导致异常但不是核心路径）
- 错误处理不完整（异常被吞没或信息不足）
- 测试严重不足（核心逻辑无测试覆盖）
- 明显的重复逻辑（3 处以上相同实现）
- 分层违规（但不影响当前功能正确性）
- N+1 查询或性能隐患（当前可能不显现但将随数据增长暴露）
- 日志中记录了敏感信息

**处理**：应在本次交付中修复。如不修复，需开发者说明理由，审查者确认可接受。

### P2 建议 🟡

**定义**：改了更好，不改不影响功能和安全，但影响代码质量和可维护性。

**判定标准**：
- 命名不够清晰（但不导致误解）
- 代码结构可优化（方法过长但逻辑正确）
- 存在更优的实现方式
- 注释缺失（非关键逻辑）
- 魔数未抽取为常量
- 测试命名或结构可改善

**处理**：开发者自行决定是否修改，不阻断放行。

### P3 备注 🔵

**定义**：记录观察、技术债、后续可关注的方向。无需当前行动。

**判定标准**：
- 已知的技术债但不在本次变更范围
- 可能需要在未来迭代中重构的区域
- 设计讨论点（有多种合理方案）
- 代码风格偏好（不影响可读性）

**处理**：仅记录，无需行动。

### 分级边界案例

| 场景 | 分级 | 理由 |
|------|------|------|
| 支付接口缺少幂等保护 | 🔴 P0 | 可能导致重复扣款 |
| 非核心列表接口缺少分页 | 🟠 P1 | 数据增长后会出问题 |
| 方法命名为 `doStuff()` | 🟡 P2 | 不影响功能但影响可维护性 |
| 有一处 `// TODO: 优化` 注释 | 🔵 P3 | 标记后续关注即可 |
| 错误码不符合契约 | 🔴 P0 | 违反契约 |
| 日志中打印了用户邮箱 | 🟠 P1 | 有隐私风险但非立即崩溃 |
| Service 中有一行直接的 SQL | 🟠 P1 | 分层违规 |
| 变量名 `data` 可以更具体 | 🟡 P2 | 不影响功能 |

## 三、反模式详解

### 反模式 1: 只看命名和格式，不看真实风险

**表现**：审查报告中 80% 是命名建议、格式调整，而忽略了隐藏的逻辑错误。

**危害**：
- 真正的风险被埋没在大量低价值意见中
- 开发者疲于应对格式修改，忽略重要问题
- 给人"审查很仔细"的假象

**正确做法**：
- 先完成 Step 2-6 的实质审查，再看格式
- 格式问题统一放 P2/P3，不混入正确性审查
- 审查报告的前几条必须是最高优先级的问题

### 反模式 2: 所有意见一视同仁不分级

**表现**：审查意见列表中没有优先级标记，阻断问题和格式建议混在一起。

**危害**：
- 开发者不知道要改什么
- 紧急的安全问题可能被忽略
- 审查变成"找茬清单"而非风险管控

**正确做法**：
- 严格使用 P0-P3 四级分类
- P0/P1 放在报告最前面
- 阻断性结论要醒目标注

### 反模式 3: 发现问题直接替作者改掉

**表现**：审查者直接修改了代码，而不是指出问题并给建议。

**危害**：
- 审查变成了开发，角色混乱
- 作者失去了理解和修复问题的机会
- 修改缺乏追踪和可解释性

**正确做法**：
- 指出问题 + 说明原因 + 给修复建议
- 简单到一行的修复可以直接建议代码
- 复杂修复只给方向和要点

### 反模式 4: 把架构缺陷当单点代码修补

**表现**：发现了系统性的架构问题，但只在当前代码中打补丁。

**危害**：
- 根本问题未解决，补丁越来越多
- 系统复杂度持续上升
- 后续维护成本指数增长

**正确做法**：
- 识别问题是局部还是系统性
- 系统性问题 → 升级架构专家
- 当前变更中做最小安全处理，不做大范围重构

### 反模式 5: 忽略边界条件和失败路径

**表现**：只看正常流程（happy path），不审查异常处理。

**危害**：
- 线上 90% 的 bug 来自边界和异常场景
- 系统在非正常条件下崩溃

**正确做法**：
- 每个方法/接口都问"如果输入为空怎么办？如果操作失败怎么办？"
- 异常路径和正常路径同等重要

### 反模式 6: "线上也这么写"当理由放行

**表现**：发现问题后，因为"存量代码也是这样"而默认合理。

**危害**：
- 技术债持续累积
- 新代码继承旧代码的所有问题
- 质量标准持续下降

**正确做法**：
- 每次审查独立判断，不以存量为标准
- 如果确实是遗留问题，标记为 P3 技术债
- 新代码不应重复旧代码的错误

### 反模式 7: 审查变成大规模重写

**表现**：审查意见要求大范围重构，超出了本次变更的范围。

**危害**：
- 审查周期无限延长
- 开发者困惑于"审查还是重新开发"
- 范围蔓延导致项目延期

**正确做法**：
- 审查范围 = 本次变更范围
- 超出范围的改进 → 标记 P3 + 建议单独任务
- 如果重构确实必要 → 升级项目管理专家

### 反模式 8: 上下文不足时硬审硬判

**表现**：缺少 API 契约、数据模型等基准信息，但仍然给出审查意见。

**危害**：
- 审查意见可能基于错误假设
- 浪费双方时间
- 降低审查的可信度

**正确做法**：
- 明确列出缺少什么信息
- 暂停审查，要求补充
- 信息不足时不给硬性结论

## 四、放行判断指南

### 放行决策矩阵

| 条件 | 决策 | 后续动作 |
|------|------|---------|
| 零发现 | ✅ 通过 | 无 |
| 仅 P2/P3 | ✅ 通过 | 附建议清单 |
| P1 ≤ 3 个且无关联 | ⚠️ 有条件通过 | 附修复清单，修复后可不再审 |
| P1 > 3 个或有关联 | 🔄 修改后再审 | 生成复审清单 |
| 存在任何 P0 | ❌ 拒绝放行 | 修复所有 P0 后再审 |
| 上下文严重不足 | ⛔ 拒绝审查 | 要求补充信息 |

### 升级判断

| 场景 | 动作 |
|------|------|
| 发现跨模块的结构性问题 | 暂停审查 → 升级技术架构专家 |
| 契约定义本身有歧义 | 暂停审查 → 回流 API/数据库专家 |
| 核心路径测试完全缺失 | 有条件通过 → 同步测试专家 |
| diff 超过 500 行核心变更 | 建议拆分 → 升级项目管理专家 |
| 审查发现反复出现同类问题 | 标记 → 建议 pm-lead 排查根因 |
| 修复建议可能显著影响排期 | 通知 → 同步项目管理专家 |
