# 正确性审查

本文件覆盖 3 个审查方向：契约一致性审查、逻辑正确性审查、边界条件与异常路径审查。

---

## 一、契约一致性审查

### 审查目标

验证实现代码是否忠实于上游定义的契约（API 契约、数据模型、架构约束），确保"说到做到"。

### 审查要点

#### 1. API 契约一致性

| 检查项 | 具体审查内容 |
|--------|-------------|
| 路径与方法 | 路由路径是否与契约定义一致，HTTP 方法是否正确 |
| 请求参数 | 参数名、类型、必填/可选是否与契约一致 |
| 请求体 | 字段名、类型、嵌套结构、校验规则是否一致 |
| 响应结构 | 响应字段名、类型、嵌套结构是否一致 |
| 状态码 | 成功/失败/异常场景的 HTTP 状态码是否与契约一致 |
| 错误码 | 业务错误码和错误消息是否按契约定义返回 |
| 分页 | 分页参数（page/size/total）和响应格式是否一致 |
| 排序/过滤 | 排序字段、方向、过滤条件是否与契约一致 |

#### 2. 数据模型一致性

| 检查项 | 具体审查内容 |
|--------|-------------|
| 字段语义 | 代码中字段使用是否与字段字典定义的语义一致 |
| 字段类型 | ORM/DAO 中字段类型映射是否与 DDL 一致 |
| 约束传递 | NOT NULL/UNIQUE/FK 等约束是否在代码层体现 |
| 状态机 | 状态转换是否遵循数据模型中定义的合法转换矩阵 |
| 关系映射 | 1:1/1:N/M:N 关系的代码实现是否与模型一致 |
| 默认值 | 代码中的默认值是否与模型定义一致 |

#### 3. 架构约束一致性

| 检查项 | 具体审查内容 |
|--------|-------------|
| 分层调用方向 | 是否遵循 handler → service → repository 单向调用 |
| 依赖规则 | 是否引入了禁止的跨模块直接依赖 |
| 中间件/拦截器 | 是否使用了约定的认证/授权/日志中间件 |
| 配置管理 | 配置项是否按架构规范获取（不硬编码） |

### 常见问题模式

| 问题 | 示例 | 默认分级 |
|------|------|---------|
| 响应字段名与契约不一致 | 契约定义 `user_name`，实现返回 `userName` | 🔴 P0 |
| 错误码未按契约返回 | 契约定义 404 返回 `USER_NOT_FOUND`，实现返回通用 500 | 🟠 P1 |
| 漏实现契约中的可选参数 | 契约定义支持 `sort_by`，实现未处理 | 🟠 P1 |
| 分页格式不一致 | 契约定义 `{items, total, page}`，实现返回 `{data, count}` | 🔴 P0 |

## 二、逻辑正确性审查

### 审查目标

验证核心业务逻辑是否正确实现，确保"做对了"。

### 审查要点

#### 1. 业务逻辑正确性

| 检查项 | 具体审查内容 |
|--------|-------------|
| 条件判断 | if/switch 的条件是否覆盖所有合法分支，是否有遗漏或错误 |
| 计算逻辑 | 数学运算、聚合计算、百分比计算是否正确 |
| 状态转换 | 状态机转换是否覆盖所有合法路径，是否拒绝非法转换 |
| 排序/过滤 | 排序规则和过滤条件是否正确实现 |
| 数据转换 | DTO ↔ Entity ↔ VO 之间的转换是否无遗漏、无错位 |
| 时间处理 | 时区、日期格式、时间比较是否正确 |
| 权限判断 | 权限校验逻辑是否正确，是否可被绕过 |

#### 2. 控制流正确性

| 检查项 | 具体审查内容 |
|--------|-------------|
| 循环终止 | 循环是否一定能终止，有无无限循环风险 |
| 提前返回 | return/break/continue 是否在正确位置 |
| 异步顺序 | async/await 顺序是否正确，有无竞态条件 |
| 事务边界 | 事务开始/提交/回滚时机是否正确 |
| 回调/闭包 | 回调参数是否正确传递，闭包是否捕获了正确的变量 |

#### 3. 数据流正确性

| 检查项 | 具体审查内容 |
|--------|-------------|
| 输入→处理→输出 | 数据是否完整流经所有必要处理步骤 |
| 中间结果 | 中间变量是否被正确赋值和使用 |
| 副作用 | 是否有非预期的数据修改（如修改了传入参数） |
| 数据一致性 | 多表/多步操作的数据一致性是否保证 |

### 常见问题模式

| 问题 | 示例 | 默认分级 |
|------|------|---------|
| 条件反转 | `if (user.isAdmin)` 应为 `if (!user.isAdmin)` | 🔴 P0 |
| 类型转换丢失精度 | 金额从 float 转 int 丢失小数 | 🔴 P0 |
| 事务未回滚 | catch 中未调用 rollback | 🔴 P0 |
| 异步未等待 | 忘记 await 导致后续逻辑用到未完成的结果 | 🟠 P1 |

## 三、边界条件与异常路径审查

### 审查目标

验证代码在非正常输入、极端情况和异常场景下是否安全，确保"出错了也不崩"。

### 审查要点

#### 1. 输入边界

| 检查项 | 具体审查内容 |
|--------|-------------|
| 空值 | null/undefined/空字符串/空数组/空对象是否处理 |
| 零值 | 数值为 0 时逻辑是否正确（0 不等于 null） |
| 极值 | 超大数值、超长字符串、超大集合是否处理 |
| 特殊字符 | SQL 注入字符、HTML 标签、Unicode 特殊字符 |
| 类型边界 | 整数溢出、浮点精度、日期范围 |
| 集合边界 | 空集合、单元素集合、超大集合 |

#### 2. 业务边界

| 检查项 | 具体审查内容 |
|--------|-------------|
| 首次/末次 | 第一次操作和最后一次操作的特殊处理 |
| 并发操作 | 同一资源被多个请求同时操作 |
| 重复提交 | 同一操作被重复执行（幂等性） |
| 权限边界 | 刚好有权限 / 刚好无权限的边界 |
| 时间边界 | 跨天/跨月/跨年、时区切换 |
| 状态边界 | 状态机中非法状态转换的处理 |

#### 3. 异常路径

| 检查项 | 具体审查内容 |
|--------|-------------|
| 数据库异常 | 连接超时、查询失败、唯一约束冲突、死锁 |
| 外部服务异常 | 第三方 API 超时、返回异常、服务不可用 |
| 网络异常 | 请求中断、连接重置、DNS 解析失败 |
| 资源异常 | 磁盘满、内存不足、文件不存在 |
| 业务异常 | 数据不存在、状态不允许、权限不足 |
| 级联失败 | 一个操作失败后，后续依赖操作的处理 |

#### 4. 错误处理质量

| 检查项 | 具体审查内容 |
|--------|-------------|
| catch 范围 | 是否 catch 了过宽的异常类型（如 catch all） |
| 错误信息 | 错误消息是否有诊断价值，是否泄露内部实现 |
| 错误传播 | 异常是否在正确层级处理，不吞没不泄露 |
| 重试策略 | 可重试操作是否有重试机制，是否有退避策略 |
| 降级策略 | 非关键依赖失败时是否有降级方案 |
| 资源清理 | 异常路径中资源（连接/文件/锁）是否被正确释放 |

### 常见问题模式

| 问题 | 示例 | 默认分级 |
|------|------|---------|
| 空值未处理导致 NPE | `user.getName()` 未检查 user 是否为 null | 🔴 P0 |
| catch 吞没异常 | `catch (e) {}` 空 catch 块 | 🟠 P1 |
| 缺少幂等保护 | 支付接口未做重复提交防护 | 🔴 P0 |
| 外部调用无超时 | HTTP 请求未设置超时时间 | 🟠 P1 |
| 资源未关闭 | 数据库连接在异常路径未释放 | 🔴 P0 |
| 错误信息泄露 | 返回包含 SQL 语句的错误消息 | 🟠 P1 |
