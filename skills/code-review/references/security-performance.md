# 安全与性能审查

本文件覆盖 3 个审查方向：基础安全风险审查、性能与资源审查、并发风险审查。

---

## 一、基础安全风险审查

### 审查目标

识别实现中的基础安全风险，包括注入攻击、越权访问、敏感数据泄露等常见安全问题。不替代专业安全审计，但确保基础安全底线。

### 审查要点

#### 1. 输入校验与注入防护

| 检查项 | 具体审查内容 |
|--------|-------------|
| SQL 注入 | 是否使用参数化查询/预编译语句，无字符串拼接 SQL |
| XSS | 用户输入是否在输出前正确转义，前端是否使用安全的渲染方式 |
| 命令注入 | 是否存在拼接用户输入执行系统命令的代码 |
| 路径遍历 | 文件路径是否经过规范化，防止 `../` 穿越 |
| 正则 DoS | 正则表达式是否存在灾难性回溯风险 |
| 输入长度 | 是否对输入长度设置上限，防止缓冲区溢出 |
| 类型校验 | 是否校验输入类型，防止类型混淆攻击 |

#### 2. 认证与授权

| 检查项 | 具体审查内容 |
|--------|-------------|
| 认证检查 | 需要认证的接口是否都经过认证中间件 |
| 授权检查 | 操作前是否校验当前用户的权限 |
| 水平越权 | 是否校验用户只能访问自己的资源（不能通过改 ID 访问他人数据） |
| 垂直越权 | 普通用户是否能调用管理员接口 |
| Token 安全 | Token 是否有过期时间，是否可被重放 |
| 会话管理 | 会话是否安全创建/销毁，无固定会话 ID |

#### 3. 敏感数据处理

| 检查项 | 具体审查内容 |
|--------|-------------|
| 日志脱敏 | 日志中是否输出了密码、Token、身份证号等敏感信息 |
| 响应过滤 | API 响应中是否返回了不该暴露的字段（如密码哈希） |
| 传输加密 | 敏感数据传输是否使用 HTTPS |
| 存储加密 | 密码是否使用不可逆哈希（bcrypt/argon2），不明文存储 |
| 密钥管理 | 密钥/凭证是否硬编码在代码中（应使用环境变量/密钥管理服务） |
| 数据分类 | 不同敏感级别的数据是否有差异化的处理策略 |

### 常见安全问题模式

| 问题 | 示例 | 默认分级 |
|------|------|---------|
| SQL 拼接 | `db.query("SELECT * FROM users WHERE id=" + userId)` | 🔴 P0 |
| 缺少授权检查 | 删除接口未校验是否是资源所有者 | 🔴 P0 |
| 密码明文存储 | `user.password = plaintext` | 🔴 P0 |
| 日志记录密码 | `log.info("登录: user={}, pwd={}", user, pwd)` | 🟠 P1 |
| 密钥硬编码 | `const API_KEY = "sk-xxx"` | 🟠 P1 |
| 缺少输入长度限制 | 文本字段无最大长度校验 | 🟡 P2 |

## 二、性能与资源审查

### 审查目标

识别可能导致性能问题、资源耗尽或系统不稳定的实现模式。

### 审查要点

#### 1. 数据库性能

| 检查项 | 具体审查内容 |
|--------|-------------|
| N+1 查询 | 循环中是否有逐条查询数据库的操作 |
| 全表扫描 | 查询条件是否能命中索引，是否存在无索引的大表查询 |
| 无分页查询 | 列表查询是否有分页限制，防止返回超大结果集 |
| 不必要的字段 | SELECT * 是否可以优化为只查需要的字段 |
| 批量操作 | 大量写入是否使用批量操作而非逐条插入 |
| 事务范围 | 事务是否尽可能短，不在事务中做耗时操作 |

#### 2. 内存与对象

| 检查项 | 具体审查内容 |
|--------|-------------|
| 大对象加载 | 是否一次性加载大量数据到内存（如全量查询后在内存过滤） |
| 对象生命周期 | 大对象是否及时释放，是否存在内存泄漏 |
| 集合膨胀 | Map/List 是否无限增长（如缓存无淘汰策略） |
| 字符串拼接 | 循环中是否使用低效的字符串拼接（应用 StringBuilder/join） |
| 深拷贝 | 是否有不必要的深拷贝大对象操作 |

#### 3. I/O 与网络

| 检查项 | 具体审查内容 |
|--------|-------------|
| 外部调用超时 | HTTP/RPC 调用是否设置超时时间 |
| 串行调用 | 多个独立的外部调用是否可以并行化 |
| 文件 I/O | 大文件是否使用流式处理而非全量加载 |
| 连接复用 | 是否复用 HTTP/数据库连接，而非每次新建 |

#### 4. 资源安全

| 检查项 | 具体审查内容 |
|--------|-------------|
| 连接泄漏 | 数据库/HTTP 连接是否在异常路径正确关闭 |
| 文件句柄 | 打开的文件是否在 finally/defer/using 中关闭 |
| 线程/协程 | 是否存在线程泄漏或协程无限创建 |
| 连接池 | 连接池大小是否合理，是否有耗尽保护 |

### 常见性能问题模式

| 问题 | 示例 | 默认分级 |
|------|------|---------|
| N+1 查询 | 循环中为每条记录单独查关联数据 | 🟠 P1 |
| 无分页查询 | `SELECT * FROM orders` 无 LIMIT | 🟠 P1 |
| 连接泄漏 | try 中获取连接，catch 中未释放 | 🔴 P0 |
| 外部调用无超时 | `httpClient.get(url)` 无超时配置 | 🟠 P1 |
| 内存中全量排序 | 从数据库查出全部数据后在内存排序 | 🟡 P2 |

## 三、并发风险审查

### 审查目标

识别多线程/多请求场景下的并发安全风险。

### 审查要点

| 检查项 | 具体审查内容 |
|--------|-------------|
| 竞态条件 | 检查-然后-执行（check-then-act）是否为原子操作 |
| 共享可变状态 | 是否存在多线程共享的可变变量且无同步保护 |
| 死锁风险 | 多把锁的获取顺序是否一致 |
| 重入问题 | 同一操作被并发触发时是否安全（如双击提交） |
| 乐观/悲观锁 | 并发更新是否使用了版本号或锁机制 |
| 缓存一致性 | 缓存与数据库的一致性策略是否明确 |
| 计数器/余额 | 涉及数量增减的操作是否为原子操作 |

### 常见并发问题模式

| 问题 | 示例 | 默认分级 |
|------|------|---------|
| 先查后写无锁 | 查库存→判断够→扣减，无并发保护 | 🔴 P0 |
| 共享可变 Map 无同步 | 多线程读写同一个 HashMap | 🔴 P0 |
| 双击提交无幂等 | 支付接口无重复提交防护 | 🔴 P0 |
| 缓存不一致 | 更新数据库后未更新/失效缓存 | 🟠 P1 |

## 审查策略

### 安全审查优先级

1. **先审注入和越权**：这是最高风险
2. **再审敏感数据**：泄露影响大，修复成本低
3. **最后审其他**：Token、会话、加密等

### 性能审查优先级

1. **先审数据库性能**：N+1 和无分页是最常见的性能杀手
2. **再审资源安全**：连接泄漏会导致系统崩溃
3. **最后审内存和 I/O**：通常在压测阶段才显现

### 边界判断

- 安全漏洞（注入/越权/数据泄露） → **P0 阻断，必须修复**
- 性能隐患但当前数据量小 → **P1 需修改 + 标记后续关注**
- 并发风险在当前无并发场景 → **P2 建议 + 标记当数据量增长时需处理**
- 需要专业安全审计的深度问题 → **标记 + 升级安全专家**
