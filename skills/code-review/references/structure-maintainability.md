# 结构与可维护性审查

本文件覆盖 2 个审查方向：分层与职责边界审查、可维护性与重复逻辑审查。

---

## 一、分层与职责边界审查

### 审查目标

验证代码结构是否遵循架构分层规范，每层/每模块职责是否清晰单一，确保"各司其职、不越界"。

### 审查要点

#### 1. 分层合规性

| 检查��� | 具体审查内容 |
|--------|-------------|
| Handler/Controller 层 | 只做参数解析、校验、调用 Service、组装响应。不含业务逻辑。 |
| Service 层 | 承载业务逻辑、编排多个 Repository 调用、管理事务。不含数据访问细节。 |
| Repository/DAO 层 | 只做数据访问。不含业务判断。 |
| DTO/VO/Entity | 数据传输对象和实体是否清晰分离，不混用。 |
| 中间件层 | 认证/授权/日志/限流等横切关注点是否在中间件处理，不散落各层。 |

#### 2. 职责边界

| 检查项 | 具体审查内容 |
|--------|-------------|
| 单一职责 | 每个类/函数是否只做一件事，能否用一句话描述其职责 |
| 方法职责 | 方法内是否混入了不属于其抽象层次的逻辑 |
| 模块边界 | 模块间是否通过明确接口通信，无内部实现的直接引用 |
| 跨层调用 | 是否存在 Handler 直接调用 Repository 等跨层情况 |
| 上帝类/上帝方法 | 是否存在承担过多职责的类或方法 |

#### 3. 依赖关系

| 检查项 | 具体审查内容 |
|--------|-------------|
| 依赖方向 | 依赖是否单向流动（上层依赖下层，不反向） |
| 循环依赖 | 是否存在模块/包/类之间的循环依赖 |
| 隐式依赖 | 是否存在通过全局变量、单例等引入的隐式依赖 |
| 接口隔离 | 调用方是否只依赖它需要的接口，不依赖整个实现 |

### 分层违规的常见表现

| 表现 | 具体场景 | 默认分级 |
|------|---------|---------|
| Service 逻辑下沉到 Repository | Repository 方法中包含复杂 if/else 业务判断 | 🟠 P1 |
| Handler 中写业务逻辑 | Controller 方法中直接做数据计算和状态判断 | 🟠 P1 |
| Entity 承担业务逻辑 | Entity 类中有复杂的业务方法（超出简单的值计算） | 🟡 P2 |
| 跨模块直接引用内部类 | Module A 直接 import Module B 的内部实现类 | 🟠 P1 |
| 循环依赖 | Service A 依赖 Service B，Service B 又依赖 Service A | 🟠 P1 |

## 二、可维护性与重复逻辑审查

### 审查目标

评估代码的长期维护成本，识别重复逻辑、过度复杂度和可读性问题，确保"下一个人接得住"。

### 审查要点

#### 1. 重复逻辑

| 检查项 | 具体审查内容 |
|--------|-------------|
| 复制粘贴 | 是否存在大段相似代码（>10 行高度相似） |
| 散落实现 | 相同业务规则是否在多处独立实现 |
| 魔数/硬编码 | 相同的数值/字符串是否在多处出现而未抽取为常量 |
| 重复校验 | 相同的校验逻辑是否在多处实现（应抽取为共用校验器） |
| 重复转换 | 相同的数据转换逻辑是否在多处实现 |

**重复的标���**：
- 两处相同 → 可接受但需标注
- 三处及以上相同 → 必须提取共用逻辑
- 业务规则重复 → 无论几处都应统一来源

#### 2. 复杂度控制

| 检查项 | 具体审查内容 | 阈值建议 |
|--------|-------------|---------|
| 方法长度 | 单个方法行数 | > 50 行关注，> 100 行建议拆分 |
| 嵌套深度 | if/for/try 嵌套层级 | > 3 层关注，> 4 层建议重构 |
| 参数数量 | 方法参数个数 | > 4 个关注，> 6 个建议用对象封装 |
| 分支数量 | 单方法中 if/else/switch 分支 | > 5 个关注，建议策略模式或映射表 |
| 圈复杂度 | 方法中独立路径数 | > 10 关注，> 15 建议拆分 |
| 类体积 | 单文件行数 | > 300 行关注，> 500 行建议拆分 |

#### 3. 可读性

| 检查项 | 具体审查内容 |
|--------|-------------|
| 命名 | 变量/函数/类名是否自解释，无歧义 |
| 注释 | 关键业务逻辑和非显而易见的实现是否有注释 |
| ��码结构 | 是否按逻辑分组，有清晰的段落感 |
| 一致性 | 同一项目中命名风格、编码风格是否一致 |
| 死代码 | 是否存在不可达的代码或注释掉的旧代码 |

#### 4. 可扩展性信号

| 检查项 | 具体审查内容 |
|--------|-------------|
| 硬编码配置 | 配置项是否可外部化（不硬编码在逻辑中） |
| 紧耦合 | 组件间是否松耦合（替换一个实现是否影响其他） |
| 开闭原则 | 新增功能是否能扩展而非修改已有代码 |
| 预留接口 | 已知的变化方向是否有合理的扩展点 |

### 常见问题模式

| 问题 | 示例 | 默认分级 |
|------|------|---------|
| 大段复制粘贴 | 3 处 >20 行的高度相似代码 | 🟠 P1 |
| 上帝方法 | 单方法 200+ 行，嵌套 5 层 | 🟠 P1 |
| 魔数散落 | `if (status === 3)` 出现 5 处 | 🟡 P2 |
| 命名不自解释 | `let a = getData(); let b = process(a);` | 🟡 P2 |
| 死代码 | 大段注释掉的旧实现 | 🟡 P2 |
| 不一致的风格 | 同文件中混用 camelCase 和 snake_case | 🔵 P3 |

## 审查策略

### 分层审查的优先级

1. **先审架构合规**：分层是否正确、依赖方向是否合法
2. **再审职责边界**：每层/每模块是否只做自己的事
3. **最后审内部质量**：可读性、复杂度、重复度

### 可维护性审查的优先级

1. **先审重复逻辑**：相同业务规则是否统一来源
2. **再审复杂度**：是否有过度复杂的方法/类需要拆分
3. **最后审可读性**：命名、注释、代码结构

### 边界判断

- 结构问题涉及架构层面（如分层方式需要调整） → **升级架构专家**
- 重复逻辑源于缺少共用组件/工具 → **建议提取 + 标记 P1**
- 复杂度过高但逻辑正确 → **P2 建议，不阻断当前交付**
- 可读性问题不影响正确性 → **P2/P3，不阻断**
