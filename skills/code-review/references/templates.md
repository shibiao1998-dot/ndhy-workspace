# 交付文件模板

## 1. review-report.md — 审查报告

```markdown
# 审查报告

## 基本信息

| 项目 | 内容 |
|------|------|
| 审查对象 | {项目名/模块名} |
| 变更范围 | {涉及文件/模块清单} |
| 变更目标 | {一句话说明变更要达成什么} |
| 审查深度 | 快速 / 标准 / 深度 |
| 审查日期 | {YYYY-MM-DD} |
| 放行决策 | ✅ 通过 / ⚠️ 有条件通过 / 🔄 修改后再审 / ❌ 拒绝 |

## 审查依据

- API 契约: {文件路径或版本}
- 数据模型: {文件路径或版本}
- 架构约束: {关键约束点}
- 验收标准: {来源}

## 审查摘要

| 级别 | 数量 | 说明 |
|------|------|------|
| 🔴 P0 阻断 | {n} | {概要} |
| 🟠 P1 需修改 | {n} | {概要} |
| 🟡 P2 建议 | {n} | {概要} |
| 🔵 P3 备注 | {n} | {概要} |

## 审查详情

### 契约一致性 (Step 2)
{审查发现，按 P0-P3 分级列出}

### 逻辑正确性与边界 (Step 3)
{审查发现}

### 结构与可维护性 (Step 4)
{审查发现}

### 安全与性能 (Step 5)
{审查发现}

### 测试充分性 (Step 6)
{审查发现}

## 结构性风险

{是否存在需要升级的结构性风险，如有，说明升级对象和理由}

## 放行决策与理由

{决策 + 决策依据 + 后续要求}
```

## 3. non-blocking-suggestions.md — 非阻断建议

```markdown
# 非阻断建议

> 应当修复。如不修复，需说明理由并经审查者确认。

| # | 位置 | 问题描述 | 建议修复方式 | 延后理由（如适用） |
|---|------|---------|-------------|-------------------|
| 1 | {文件:行号} | {问题} | {建议} | {仅当开发者选择延后时填写} |

## P2 建议

> 改了更好，不改不阻断。开发者自行决定。

| # | 位置 | 建议内容 | 预期收益 |
|---|------|---------|---------|
| 1 | {文件:行号} | {建议} | {改了能带来什么} |

## P3 备注

> 记录观察，无需行动。供后续参考。

| # | 观察内容 | 备注 |
|---|---------|------|
| 1 | {观察} | {后续可关注的方向} |
```

## 4. approval-decision.md — 放行决策

```markdown

| 项目 | 内容 |
|------|------|
| 决策 | ✅ 通过 / ⚠️ 有条件通过 / 🔄 修改后再审 / ❌ 拒绝 |
| 决策依据 | {1-3 句话说明为什么是这个决策} |
| 审查深度 | 快速 / 标准 / 深度 |
| 审查日期 | {YYYY-MM-DD} |

## 条件（仅"有条件通过"时填写）

- [ ] {条件 1：修复 P1-xxx 后可合并}
- [ ] {条件 2：补充 xxx 测试后可合并}

## 后续要求

| 要求 | 负责方 | 时限 |
|------|--------|------|
| {修复 P0/P1} | 开发专家 | {紧急/本迭代} |
| {补测试} | 测试专家 | {本迭代} |
| {架构评估} | 架构专家 | {下个迭代} |

## 复审要求

- [ ] 需要复审：是 / 否
- [ ] 复审范围：{仅 P0 修复 / 全量再审}
- [ ] 复审触发条件：{修复完成后通知}
```
