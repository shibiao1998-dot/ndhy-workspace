# 代码审查 — 反模式速查

| # | ❌ 错误做法 | ✅ 正确做法 |
|---|-----------|-----------|
| 1 | 只看命名和格式，不看真实风险 | 正确性和边界优先，风格问题放 P2/P3 |
| 2 | 所有意见一视同仁不分级 | 严格按 P0-P3 分级，阻断和建议必须分开 |
| 3 | 发现问题直接替作者改掉 | 指出问题 + 给修复建议，不越界重写 |
| 4 | 把架构缺陷当单点代码修补 | 识别结构性风险 → 升级架构专家 |
| 5 | 忽略边界条件和失败路径 | 边界和异常路径是审查核心，不是附加项 |
| 6 | "线上也这么写"当理由放行 | 每次审查独立判断，不因存量问题降低标准 |
| 7 | 审查变成大规模重写 | 审查 ≠ 开发，超出审查边界的改动回流开发专家 |
| 8 | 上下文不足时硬审硬判 | 信息不足 → 暂停 → 要求补充，不猜测不假设 |

> 详细反模式解析 -> [checklists.md](checklists.md) §反模式详解
