# 上下文管理策略

> 项目管理专家是整个项目团队的唯一记忆体。本文件定义分级存储、摘要生成和上下文压缩的完整策略。

---

## 三级存储体系

| 级别 | 名称 | 存储位置 | 内容 | 保留策略 |
|------|------|----------|------|----------|
| L1 | 当前工作区 | 指令包 + 当前节点相关摘要 | 正在执行的节点所需的全部信息 | 节点完成后清空，转为摘要 |
| L2 | 近期摘要 | PROJECT-STATUS.md 的摘要池 | 已完成节点的结构化摘要 | 项目结束前持续保留 |
| L3 | 归档 | PROJECT-STATUS-ARCHIVE.md | 详细执行记录、完整异常日志 | 项目结束后保留，日常不加载 |

---

## 摘要生成

### 模板

每个节点完成且质量检查通过后，立即生成摘要写入摘要池：

```markdown
### 节点 [序号]：[节点名称]（[执行角色]）

- **状态**：✅ 完成 | 完成时间：YYYY-MM-DD HH:mm
- **产出物**：`[文件路径]`
- **核心结论**：
  1. [最重要的结论/产出要点，1 句话]
  2. [第二重要的，1 句话]
  3. [第三，1 句话]（如有）
- **关键数据/决策**：
  - [下游可能需要的具体数据点或决策结论]
- **已知局限**：
  - [产出中的假设、未覆盖的场景、已知不足]
- **下游注意**：
  - [下游节点使用此产出时需要注意的事项]
```

### 3 原则

| 原则 | 说明 |
|------|------|
| **面向下游** | 以"下游节点需要知道什么"为导向，不是"上游做了什么"的流水账 |
| **结论优先** | 先写结论，再写支撑。下游需要的是"结果是什么"而非"过程怎么做的" |
| **标注局限** | 已知的不足必须显式写出，防止下游基于错误假设工作 |

## 压缩触发与降级

### 触发阈值

```
上下文使用率：
├── < 40% → 🟢 正常，无需操作
├── 40%-60% → 🟡 关注，组装指令包时优先用摘要
├── 60%-80% → 🔴 压缩，执行降级方案
└── > 80% → ⛔ 紧急压缩 + 考虑拆分节点
```

### 降级方案（按顺序执行，每步后重新评估）

| 步骤 | 操作 | 预期释放 |
|------|------|----------|
| 1 | 将所有已完成节点的详细记录归档到 ARCHIVE | 中等 |
| 2 | 上游产出物从内联改为"摘要+路径" | 较大 |
| 3 | 项目上下文精简为只保留项目名+目标+当前节点相关约束 | 少量 |
| 4 | 非直接依赖节点的摘要移出当前指令包 | 中等 |
| 5 | 考虑将当前节点拆分为更小粒度的子节点 | 根本性解决 |
