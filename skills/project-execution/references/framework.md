# project-execution 核心框架

> 纪统辰 · 项目管理专家 的操作手册。本文档定义项目执行全生命周期的模板、决策树、检查清单和标准流程。
> 拿到即可用，逐模块按需查阅。

```markdown
## 7. 上下文摘要池

> 每个节点完成后生成摘要，供下游节点指令包引用。

### 节点 X：[节点名称]
- **产出物**：[文件名/路径]
- **核心结论**：[3-5 句话概括产出核心内容]
- **关键数据/决策**：[列出下游可能需要的关键信息]
- **已知局限**：[产出中的已知不足或假设]
```

### 使用要点

1. **创建时机**：接手项目后、阅读完流程和团队信息后立即创建
2. **更新纪律**：每次调度后更新「执行排期」状态；每次产出检查后更新「产出记录」；每次异常后更新「异常日志」
3. **摘要池是指令包的弹药库**：组装下游节点指令包时，从摘要池提取上游关键信息，不需要原文全注入
4. **文档长度控制**：超过 120 行时，将已完成节点的详细记录归档到 `PROJECT-STATUS-ARCHIVE.md`，主文档只保留摘要行

## 2. 指令包模板

> 指令包是纪统辰的核心产出物。每次调度执行专家前，按此模板组装。
> 质量铁律：**执行专家的产出上限 = 指令包质量。**

### 2.1 五层结构模板

```markdown
## 你的角色

[角色 emoji] [角色名称] · [职位]

### SOUL.md
[完整粘贴角色的 SOUL.md 内容]

### STANDARDS.md
[完整粘贴角色的 STANDARDS.md 内容]

---

## 任务

[一句话说清楚要做什么]

### 项目上下文

| 项 | 内容 |
|----|------|
| 项目名称 | |
| 项目目标 | [一句话] |
| 当前进度 | 共 N 个节点，已完成 M 个，当前执行第 K 个 |
| 核心约束 | [只列与本节点相关的约束] |

### 上游产出物

> 以下是你需要基于的上游产出。

**来自 [上游节点名称]（[上游角色名]）：**
[方式一：直接引用摘要池中的核心结论和关键数据]
[方式二：若上游产出物较短（<50行），直接内联]
[方式三：若上游产出物较长，给出文件路径 + 摘要，注明"完整内容见 [路径]"]

### 本节点任务说明

**目标**：[本节点要解决什么问题 / 产出什么]

**具体要求**：
1. [要求 1]
2. [要求 2]
3. [要求 3]

### 输出要求

**产出物**：[文件名/路径/格式]
**输出路径**：`[项目文件夹]/[子目录]/[文件名]`
**格式要求**：[Markdown / JSON / 代码文件 等]

### 约束与注意事项

- [约束 1：如技术栈限制]
- [约束 2：如不可修改的上游决策]
- [约束 3：如性能/安全要求]
- [注意：与其他节点的边界——不要越界做 XX]

### 验收标准

1. ✅ [标准 1]
2. ✅ [标准 2]
3. ✅ [标准 3]

### 适用技能

先加载 `[技能名]` Skill（位于 `[技能路径]`），参考其中的 [具体参考内容]。
```

### 2.2 五项核心技巧

| # | 技巧 | 说明 | 反面教材 |
|---|------|------|----------|
| 1 | **摘要代替原文** | 上游产出超过 50 行时，用摘要池的结构化摘要替代原文注入。给路径让执行者按需深读 | 把 300 行的上游产出原封不动贴进指令包 → 执行者淹没在信息中 |
| 2 | **只给相关约束** | 项目有 10 条约束，但本节点只相关 3 条 → 只注入这 3 条 | 把所有约束全贴上 → 执行者分不清哪些跟自己有关 |
| 3 | **边界显式声明** | 明确告诉执行者"不要做 XX"，防止越界 | 只说"做 XX"不说"不做 YY" → 执行者好心帮忙把下游的活也干了，但质量失控 |
| 4 | **验收标准可检验** | 每条验收标准必须是可以回答"是/否"的命题 | "代码质量要好" → 无法判断是否达标。应写成"所有函数有 JSDoc 注释" |
| 5 | **角色设定完整注入** | SOUL.md + STANDARDS.md 完整注入。这是执行者理解自己"该以什么标准工作"的唯一来源 | 只给任务不给角色设定 → 执行者用通用标准工作，产出缺乏专业深度 |

### 2.3 上游产出物注入决策

```
上游产出物行数？
├── ≤ 50 行 → 直接内联到指令包
├── 51-200 行 → 摘要 + 文件路径（注明"完整内容见 [路径]，以下为核心摘要"）
└── > 200 行 → 仅摘要 + 文件路径（注明"请先阅读 [路径] 再开始工作"）
```

## 3. 调度决策树

> 每个调度节点前，按此决策树判断。

### 3.1 节点就绪判断

```
该节点是否可以执行？逐条检查：

1. 所有依赖节点是否已完成？
   ├── 否 → ⏸️ 跳过，处理其他可执行节点
   └── 是 ↓

2. 上游产出物是否已通过质量检查？
   ├── 否 → ⏸️ 等待上游修正（记录阻塞原因）
   └── 是 ↓

3. 执行角色的设定（SOUL.md + STANDARDS.md）是否就绪？
   ├── 否 → 先获取/确认角色设定
   └── 是 ↓

4. 指令包是否已组装完成？
   ├── 否 → 按模板组装
   └── 是 ↓

5. 上下文总量是否可控？（指令包 + 角色设定 + 上游产出 < 上下文窗口 60%）
   ├── 否 → 执行上下文压缩（见模块 6）后重新评估
   └── 是 → ✅ 就绪，可调度
```

### 3.2 串行 vs 并行决策

```
当前有多个就绪节点时：

能否并行？逐条检查（全部满足才可并行）：

1. 输入独立性：各节点是否依赖不同的上游产出，或依赖同一上游但只读不写？
   ├── 否 → 串行
   └── 是 ↓

2. 输出独立性：各节点的产出文件/目录是否完全不重叠？
   ├── 否 → 串行
   └── 是 ↓

3. 无实时交互：各节点是否不需要在执行过程中互相传递中间结果？
   ├── 否 → 串行
   └── 是 ↓

4. 失败隔离：一个节点失败是否不影响另一个节点的产出有效性？
   ├── 否 → 串行（或串行关键路径，并行非关键路径）
   └── 是 → ✅ 可并行调度
```

### 3.3 并行调度 4 要点

| # | 要点 | 操作 |
|---|------|------|
| 1 | **同时分发** | 并行节点的指令包在同一批次组装并发出，不等一个发完再发下一个 |
| 2 | **异步收取** | 不要求并行节点同时完成。先完成的先检查，不阻塞后完成的 |
| 3 | **产出一致性校验** | 并行节点全部完成后，检查产出之间是否存在矛盾（如命名冲突、接口不匹配） |
| 4 | **单点失败不丢另一方** | 一个节点失败时，另一个节点的成功产出保留。失败节点单独重试，不影响已成功的 |

### 3.4 依赖冲突处理

| 场景 | 判断 | 处理 |
|------|------|------|
| 上游节点失败，下游等待中 | 检查失败类型 | 可重试 → 重试上游；根本性问题 → 暂停下游，上报 |
| 两个并行节点产出矛盾 | 一致性校验不通过 | 分析矛盾原因，选择修正成本更低的一方重做 |
| 循环依赖 | 排期阶段检测 | 流程设计缺陷 → 上报，不自行解决 |
| 可选依赖（有更好，没有也行） | 上游标注为可选 | 上游完成 → 注入；上游失败 → 跳过该依赖，指令包中说明"无此输入，请用默认方案" |

## 4. 质量检查清单

> 每个节点产出返回后，依序执行两层检查。第一层必做，第二层按需。

### 4.1 第一层：形式检查（必做）

| # | 检查项 | 判断标准 | 不通过处理 |
|---|--------|----------|------------|
| 1 | **文件存在** | 产出文件是否在指定路径？ | 要求补交 |
| 2 | **格式正确** | 文件格式是否符合要求（MD/JSON/代码等）？能否正常解析？ | 打回重做 |
| 3 | **完整性** | 指令包要求的所有产出项是否齐全？无缺失章节/模块？ | 列出缺失项，要求补全 |
| 4 | **量级合理** | 产出物的长度/规模是否在合理范围？（不能只有 10 行完成 100 行的需求） | 评估是否偷工减料，要求补充 |
| 5 | **编码正常** | 无乱码、无异常字符、中英文标点一致 | 标注问题位置，要求修正 |
| 6 | **命名规范** | 文件名、目录名是否符合项目约定？ | 要求重命名 |

> **第一层全部通过后，才进入第二层。**

### 4.2 第二层：内容检查（按需深度不同）

| # | 检查项 | 判断标准 | 不通过处理 |
|---|--------|----------|------------|
| 1 | **目标一致** | 产出内容是否回答了指令包中"本节点任务说明"的核心问题？ | 方向性问题 → 重新调度 |
| 2 | **呼应上游** | 是否正确引用/基于了上游产出物？有无曲解上游结论？ | 标注曲解点，附修正指引重做 |
| 3 | **内部无矛盾** | 产出物内部各部分是否自洽？（如前文说 A，后文说非 A） | 标注矛盾点，要求修正 |
| 4 | **满足下游** | 产出物是否能直接作为下游节点的输入？格式/内容是否够用？ | 评估缺什么，要求补充 |
| 5 | **验收标准逐条** | 逐条对照指令包中的验收标准，每条是否达标？ | 列出未达标项，附具体差距说明 |
| 6 | **无越界** | 是否超出了指令包指定的边界？有无替下游做了不该做的事？ | 删除越界部分，保留有效部分 |

### 4.3 质量判定与处理

| 判定 | 标准 | 处理 |
|------|------|------|
| ✅ **合格** | 两层检查全部通过 | 更新产出记录，生成摘要，继续下一节点 |
| 🟡 **小问题** | 形式通过，内容有 1-2 处小瑕疵但不影响方向 | 自行修正并标注修改点，不重新调度 |
| 🔴 **方向性问题** | 内容检查中目标不一致或严重曲解上游 | 重新调度，附修正指引（含问题描述 + 正确方向）。最多 2 次 |
| ⛔ **根本性问题** | 流程设计缺陷、需求本身有歧义 | 暂停该节点，上报用户决策 |

## 5. 异常处理表

> 6 种常见异常的判断标准和处理步骤。核心原则：**不超过 2 次重试，解决不了就上报。失败产出不丢弃。**

### 异常 1：执行超时

| 维度 | 内容 |
|------|------|
| **判断标准** | 执行专家长时间未返回产出（参考：简单节点 > 10 分钟，复杂节点 > 30 分钟） |
| **处理步骤** | 1. 发送进度询问（sessions_send） |
|  | 2. 等待 5 分钟。有响应 → 评估是否继续等待 |
|  | 3. 无响应 → kill 当前会话 |
|  | 4. 重新调度（指令包不变），标注"重试 #1" |
|  | 5. 再次超时 → 上报用户，附超时现象和可能原因 |
| **日志记录** | 异常类型：超时 / 节点名 / 超时时长 / 处理方式 / 结果 |

### 异常 2：产出不合格

| 维度 | 内容 |
|------|------|
| **判断标准** | 质量检查未通过（见模块 4 的判定标准） |
| **处理步骤** | 1. 明确不合格类型：形式问题 / 内容小问题 / 方向性问题 / 根本性问题 |
|  | 2. 形式问题或小问题 → 自行修正或要求补充（不算重试） |
|  | 3. 方向性问题 → 重新调度，指令包附加"修正指引"段（描述问题 + 正确方向 + 上次失败产出路径供参考） |
|  | 4. 第 2 次重试仍不合格 → 上报用户，附两次失败产出和分析 |
|  | 5. 根本性问题 → 直接上报，不重试 |
| **日志记录** | 异常类型：产出不合格 / 节点名 / 不合格原因 / 重试次数 / 处理方式 / 最终结果 |

### 异常 3：依赖缺失

| 维度 | 内容 |
|------|------|
| **判断标准** | 组装指令包时发现上游产出物不存在、不完整或未通过质量检查 |
| **处理步骤** | 1. 检查上游节点状态：是否已执行？产出物在哪？ |
|  | 2. 上游未执行 → 先调度上游（调整排期） |
|  | 3. 上游已执行但产出丢失 → 检查是否有备份/日志可恢复 |
|  | 4. 上游已执行但质量不过 → 按"产出不合格"流程处理上游 |
|  | 5. 依赖关系本身有误（不该依赖）→ 上报，属于流程设计缺陷 |
| **日志记录** | 异常类型：依赖缺失 / 当前节点 / 缺失的依赖 / 原因 / 处理方式 |

### 异常 4：上下文溢出

| 维度 | 内容 |
|------|------|
| **判断标准** | 指令包 + 角色设定 + 上游产出合计超过上下文窗口 60%，或执行专家返回明显遗漏（疑似上下文截断） |
| **处理步骤** | 1. 评估哪部分可压缩：优先压缩上游产出（用摘要替代原文） |
|  | 2. 其次压缩项目上下文（只保留与本节点直接相关的信息） |
|  | 3. 如果压缩后仍超标 → 拆分节点：将当前节点拆为 2 个更小的子节点 |
|  | 4. 拆分仍不可行 → 上报用户，建议调整任务粒度或方案 |
| **日志记录** | 异常类型：上下文溢出 / 节点名 / 估算 token 量 / 压缩措施 / 压缩后结果 |

### 异常 5：工具故障

| 维度 | 内容 |
|------|------|
| **判断标准** | 执行专家反馈工具调用失败（API 报错、文件读写失败、网络超时等） |
| **处理步骤** | 1. 分析错误信息：是临时性故障还是持续性问题 |
|  | 2. 临时性（如网络抖动）→ 等待 2 分钟后重新调度（同一指令包） |
|  | 3. 持续性（如 API 不可用）→ 评估是否有替代方案 |
|  | 4. 有替代方案 → 修改指令包中的工具/方法要求，重新调度 |
|  | 5. 无替代方案 → 上报用户，附错误信息和影响评估 |
| **日志记录** | 异常类型：工具故障 / 节点名 / 错误信息 / 是否临时性 / 处理方式 / 结果 |

### 异常 6：根本性设计问题

| 维度 | 内容 |
|------|------|
| **判断标准** | 执行过程中发现需求歧义、流程设计缺陷、角色定义冲突、技术方案不可行等 |
| **处理步骤** | 1. **立即暂停**当前节点和所有受影响的下游节点 |
|  | 2. 记录问题：具体是什么问题、谁发现的、影响哪些节点 |
|  | 3. 评估影响范围：只影响当前节点？还是需要回溯到更上游？ |
|  | 4. 上报用户，附：问题描述 + 影响范围 + 你的处理建议（如果有） |
|  | 5. **不自行修改流程设计或需求**——你的权限在流程之内 |
| **日志记录** | 异常类型：根本性设计问题 / 发现者 / 问题描述 / 影响范围 / 上报内容 / 用户决策 |

### 异常处理速查

```
遇到异常 →
├── 能自行解决吗？（形式问题、临时性工具故障）
│   └── 是 → 解决并记录（不算重试次数）
├── 需要重新调度？（方向性问题、超时）
│   └── 重试 ≤ 2 次 → 附修正指引重新调度
│   └── 重试 > 2 次 → 上报
└── 根本性问题？（需求歧义、流程缺陷）
    └── 立即暂停 → 上报
```

## 6. 上下文管理策略

> 纪统辰是整个项目团队的唯一记忆体。上下文管理是防止"记忆崩溃"的生命线。

### 6.1 三级存储体系

| 级别 | 名称 | 存储位置 | 内容 | 保留策略 |
|------|------|----------|------|----------|
| L1 | 当前工作区 | 指令包 + 当前节点相关摘要 | 正在执行的节点所需的全部信息 | 节点完成后清空，转为摘要 |
| L2 | 近期摘要 | PROJECT-STATUS.md 的摘要池 | 已完成节点的结构化摘要 | 项目结束前持续保留 |
| L3 | 归档 | PROJECT-STATUS-ARCHIVE.md | 详细执行记录、完整异常日志 | 项目结束后保留，日常不加载 |

### 6.2 摘要生成模板

> 每个节点完成且质量检查通过后，立即生成摘要写入摘要池。

```markdown
### 节点 [序号]：[节点名称]（[执行角色]）

- **状态**：✅ 完成 | 完成时间：YYYY-MM-DD HH:mm
- **产出物**：`[文件路径]`
- **核心结论**：
  1. [最重要的结论/产出要点，1 句话]
  2. [第二重要的，1 句话]
  3. [第三，1 句话]（如有）
- **关键数据/决策**：
  - [下游可能需要的具体数据点或决策结论]
- **已知局限**：
  - [产出中的假设、未覆盖的场景、已知不足]
- **下游注意**：
  - [下游节点使用此产出时需要注意的事项]
```

### 6.3 摘要生成 3 原则

| 原则 | 说明 |
|------|------|
| **面向下游** | 摘要内容以"下游节点需要知道什么"为导向，不是"上游做了什么"的流水账 |
| **结论优先** | 先写结论，再写支撑。下游需要的是"结果是什么"而非"过程怎么做的" |
| **标注局限** | 已知的不足必须显式写出，防止下游基于错误假设工作 |

### 6.4 压缩触发条件与降级方案

```
上下文使用率评估：
├── < 40% → 🟢 正常，无需操作
├── 40%-60% → 🟡 关注，组装指令包时优先用摘要
├── 60%-80% → 🔴 压缩，执行以下降级方案
└── > 80% → ⛔ 紧急压缩 + 考虑拆分节点
```

**降级方案（按顺序执行，每步后重新评估）：**

| 步骤 | 操作 | 预期释放 |
|------|------|----------|
| 1 | 将所有已完成节点的详细记录归档到 ARCHIVE | 中等 |
| 2 | 上游产出物从内联改为"摘要+路径" | 较大 |
| 3 | 项目上下文精简为只保留项目名+目标+当前节点相关约束 | 少量 |
| 4 | 非直接依赖节点的摘要移出当前指令包 | 中等 |
| 5 | 考虑将当前节点拆分为更小粒度的子节点 | 根本性解决 |
