# 异常处理详表

> 6 种常见异常的判断标准和完整处理步骤。
> 核心原则：**不超过 2 次重试，解决不了就上报。失败产出不丢弃。**

---

## 异常 1：执行超时

| 维度 | 内容 |
|------|------|
| **判断标准** | 简单节点 > 10 分钟，复杂节点 > 30 分钟未返回产出 |
| **处理步骤** | 1. 发送进度询问（sessions_send） |
|  | 2. 等待 5 分钟。有响应 → 评估是否继续等待 |
|  | 3. 无响应 → kill 当前会话 |
|  | 4. 重新调度（指令包不变），标注"重试 #1" |
|  | 5. 再次超时 → 上报用户，附超时现象和可能原因 |
| **日志记录** | 异常类型：超时 / 节点名 / 超时时长 / 处理方式 / 结果 |

---

## 异常 2：产出不合格

| 维度 | 内容 |
|------|------|
| **判断标准** | 质量检查未通过（参见 checklists.md 的判定标准） |
| **处理步骤** | 1. 明确不合格类型：形式/小问题/方向性/根本性 |
|  | 2. 形式问题或小问题 → 自行修正或要求补充（不算重试） |
|  | 3. 方向性问题 → 重新调度，指令包附加"修正指引"段（问题描述 + 正确方向 + 上次失败产出路径供参考） |
|  | 4. 第 2 次重试仍不合格 → 上报用户，附两次失败产出和分析 |
|  | 5. 根本性问题 → 直接上报，不重试 |
| **日志记录** | 异常类型：产出不合格 / 节点名 / 不合格原因 / 重试次数 / 处理方式 / 最终结果 |

## 异常 3：依赖缺失

| 维度 | 内容 |
|------|------|
| **判断标准** | 组装指令包时发现上游产出物不存在、不完整或未通过质检 |
| **处理步骤** | 1. 检查上游节点状态：是否已执行？产出物在哪？ |
|  | 2. 上游未执行 → 先调度上游（调整排期） |
|  | 3. 上游已执行但产出丢失 → 检查是否有备份/日志可恢复 |
|  | 4. 上游已执行但质量不过 → 按"产出不合格"流程处理上游 |
|  | 5. 依赖关系本身有误 → 上报，属于流程设计缺陷 |
| **日志记录** | 异常类型：依赖缺失 / 当前节点 / 缺失的依赖 / 原因 / 处理方式 |

---

## 异常 4：上下文溢出

| 维度 | 内容 |
|------|------|
| **判断标准** | 指令包 + 角色设定 + 上游产出超过上下文窗口 60%，或执行专家返回明显遗漏（疑似截断） |
| **处理步骤** | 1. 优先压缩上游产出（用摘要替代原文） |
|  | 2. 其次压缩项目上下文（只保留与本节点直接相关的信息） |
|  | 3. 仍超标 → 拆分节点为 2 个更小的子节点 |
|  | 4. 拆分不可行 → 上报用户，建议调整任务粒度或方案 |
| **日志记录** | 异常类型：上下文溢出 / 节点名 / 估算 token 量 / 压缩措施 / 结果 |

---

## 异常 5：工具故障

| 维度 | 内容 |
|------|------|
| **判断标准** | 执行专家反馈工具调用失败（API 报错、文件读写失败、网络超时等） |
| **处理步骤** | 1. 分析错误：临时性还是持续性 |
|  | 2. 临时性（网络抖动）→ 等 2 分钟后重新调度（同一指令包） |
|  | 3. 持续性（API 不可用）→ 评估替代方案 |
|  | 4. 有替代方案 → 修改指令包工具/方法要求，重新调度 |
|  | 5. 无替代方案 → 上报用户，附错误信息和影响评估 |
| **日志记录** | 异常类型：工具故障 / 节点名 / 错误信息 / 是否临时性 / 处理方式 / 结果 |

---

## 异常 6：根本性设计问题

| 维度 | 内容 |
|------|------|
| **判断标准** | 需求歧义、流程设计缺陷、角色定义冲突、技术方案不可行 |
| **处理步骤** | 1. **立即暂停**当前节点和所有受影响的下游节点 |
|  | 2. 记录：具体问题、发现者、影响哪些节点 |
|  | 3. 评估影响范围：只影响当前节点？还是需要回溯到更上游？ |
|  | 4. 上报用户，附：问题描述 + 影响范围 + 处理建议（如果有） |
|  | 5. **不自行修改流程设计或需求**——你的权限在流程之内 |
| **日志记录** | 异常类型：根本性设计问题 / 发现者 / 问题描述 / 影响范围 / 上报内容 / 用户决策 |
