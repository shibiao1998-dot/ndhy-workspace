## §1 项目状态档案模板

> 项目启动时创建，每个调度节点完成后更新。
> 文件路径：`[项目文件夹]/PROJECT-STATUS.md`

### 使用要点

- **创建时机**：接手项目后、阅读完流程和团队信息后立即创建
- **更新纪律**：每次调度后更新执行排期状态；每次产出检查后更新产出记录；每次异常后更新异常日志
- **摘要池是指令包的弹药库**：组装下游指令包时，从摘要池提取上游关键信息，不需要原文全注入
- **长度控制**：超过 120 行时，将已完成节点的详细记录归档到 `PROJECT-STATUS-ARCHIVE.md`，主文档只保留摘要行

### 模板

```markdown

> 最后更新：YYYY-MM-DD HH:mm
```

## §2 指令包模板

> 每次调度执行专家前，按此模板组装。
> 质量铁律：**执行专家的产出上限 = 指令包质量。**

### 5 层结构模板

```markdown
## 你的角色

[角色 emoji] [角色名称] · [职位]

### SOUL.md
[完整粘贴角色的 SOUL.md 内容]

### STANDARDS.md
[完整粘贴角色的 STANDARDS.md 内容]

---

## 任务

[一句话说清楚要做什么]

### 项目上下文

| 项 | 内容 |
|----|------|
| 项目名称 | |
| 项目目标 | [一句话] |
| 当前进度 | 共 N 个节点，已完成 M 个，当前执行第 K 个 |
| 核心约束 | [只列与本节点相关的约束] |

### 上游产出物

> 以下是你需要基于的上游产出。

**来自 [上游节点名称]（[上游角色名]）：**
[方式一：直接引用摘要池中的核心结论和关键数据]
[方式二：若上游产出物较短（<50行），直接内联]
[方式三：若上游产出物较长，给出文件路径 + 摘要，注明"完整内容见 [路径]"]

### 本节点任务说明

**目标**：[本节点要解决什么问题 / 产出什么]

**具体要求**：
1. [要求 1]
2. [要求 2]
3. [要求 3]

### 输出要求

**产出物**：[文件名/路径/格式]
**输出路径**：`[项目文件夹]/[子目录]/[文件名]`
**格式要求**：[Markdown / JSON / 代码文件 等]

### 约束与注意事项

- [约束 1：如技术栈限制]
- [约束 2：如不可修改的上游决策]
- [约束 3：如性能/安全要求]
- [注意：与其他节点的边界——不要越界做 XX]

### 验收标准

1. ✅ [标准 1]
2. ✅ [标准 2]
3. ✅ [标准 3]

### 适用技能

先加载 `[技能名]` Skill（位于 `[技能路径]`），参考其中的 [具体参考内容]。
```

### 5 项核心技巧详解

| # | 技巧 | 说明 | 反面教材 |
|---|------|------|----------|
| 1 | **摘要代替原文** | 上游产出超过 50 行时，用摘要池的结构化摘要替代原文注入。给路径让执行者按需深读 | 把 300 行的上游产出原封不动贴进指令包 → 执行者淹没在信息中 |
| 2 | **只给相关约束** | 项目有 10 条约束，但本节点只相关 3 条 → 只注入这 3 条 | 把所有约束全贴上 → 执行者分不清哪些跟自己有关 |
| 3 | **边界显式声明** | 明确告诉执行者"不要做 XX"，防止越界 | 只说"做 XX"不说"不做 YY" → 执行者好心帮忙把下游的活也干了，但质量失控 |
| 4 | **验收标准可检验** | 每条验收标准必须是可以回答"是/否"的命题 | "代码质量要好" → 无法判断是否达标。应写成"所有函数有 JSDoc 注释" |
| 5 | **角色设定完整注入** | SOUL.md + STANDARDS.md 完整注入。这是执行者理解自己"该以什么标准工作"的唯一来源 | 只给任务不给角色设定 → 执行者用通用标准工作，产出缺乏专业深度 |

### 上游产出物注入决策

```
上游产出物行数？
├── ≤ 50 行 → 直接内联到指令包
├── 51-200 行 → 摘要 + 文件路径（注明"完整内容见 [路径]，以下为核心摘要"）
└── > 200 行 → 仅摘要 + 文件路径（注明"请先阅读 [路径] 再开始工作"）
```

## §3 进度报告模板

> 每个调度节点完成后，向用户汇报进度。

```markdown
## 📊 进度报告 — [项目名称]

**报告时间**：YYYY-MM-DD HH:mm
**项目状态**：🟡 进行中

### 本次完成

| 节点 | 执行角色 | 状态 | 产出物 |
|------|----------|------|--------|
| [节点名] | [角色名] | ✅ 完成 | `[文件路径]` |

### 质量检查结果

- 形式检查：✅ 通过
- 内容检查：✅ 通过 / 🟡 小问题已修正（[修正说明]）

### 核心产出摘要

[2-3 句话概括本节点产出的核心内容和关键结论]

### 整体进度

| 指标 | 数值 |
|------|------|
| 总节点数 | N |
| 已完成 | M |
| 进行中 | X |
| 待执行 | Y |
| 异常/阻塞 | Z |
| 完成率 | M/N (xx%) |

### 下一步

- 下一个执行节点：[节点名]（[角色名]）
- 预计动作：[组装指令包 / 等待并行节点完成 / ...]

### 风险与阻塞（如有）

- [风险/阻塞描述 + 影响 + 建议处理方式]

### 需要用户决策（如有）

- [决策事项 + 选项 + 你的推荐]
```
