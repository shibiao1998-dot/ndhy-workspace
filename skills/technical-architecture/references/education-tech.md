# 教育行业技术架构特别指南

## 教育行业技术特征

教育系统与一般互联网产品的核心差异：

| 维度 | 一般互联网 | 教育行业 |
|------|----------|---------|
| 流量模式 | 相对平稳，增长渐进 | **强周期性**：开学/考试/假期差异巨大 |
| 用户设备 | 主流设备为主 | **低端设备占比高**：县镇学校老旧电脑/平板 |
| 网络环境 | 城市宽带/4G+ | **低带宽**：部分学校 10Mbps 共享 |
| 数据敏感度 | 一般个人信息 | **未成年人数据**：法律保护等级最高 |
| 系统对接 | 标准 API | **教育系统对接**：老旧接口、非标准协议 |
| AI 使用 | 通用内容生成 | **教育适用性审查**：内容必须年龄适当、价值正确 |

## 周期性负载设计

### 流量特征

```
典型教育系统年度流量曲线：

     ┌──┐              ┌──┐
     │  │   ┌──┐       │  │
     │  │   │  │       │  │
─────┘  └───┘  └───────┘  └───── 
 2月    4月   6-8月   9月   12月
开学    期中   暑假   开学   期末
高峰    考试   低谷   高峰   考试
```

### 关键时刻

| 时刻 | 流量特征 | 持续时间 | 设计要求 |
|------|---------|---------|---------|
| **开学季** | 峰值流量，集中注册/选课 | 1-2 周 | 并发注册、批量导入 |
| **考试周** | 高并发，集中提交 | 3-5 天 | 提交不丢、结果准确 |
| **成绩发布** | 瞬时高峰 | 数小时 | 只读扩展、缓存 |
| **假期** | 极低流量 | 1-2 月 | 缩容省成本 |
| **开学前** | 批量数据迁移/导入 | 数天 | 后台任务、不影响前台 |

### 架构应对策略

| 策略 | 说明 |
|------|------|
| **弹性伸缩** | 基于流量自动扩缩容（K8s HPA / 云弹性组） |
| **预热机制** | 开学前 1 周预热缓存和预扩容 |
| **排队机制** | 高峰期入口排队，防止系统过载 |
| **只读模式** | 成绩查询等场景，走只读副本 + CDN |
| **异步提交** | 考试提交等关键操作，先收后处理，确保不丢 |
| **定时缩容** | 假期自动缩容，节省 60%+ 成本 |

## 低带宽与低端设备适配

### 设计原则

| # | 原则 | 具体要求 |
|---|------|---------|
| 1 | **轻量优先** | 首屏资源 < 500KB，首次加载 < 3s（3G 网络） |
| 2 | **渐进增强** | 核心功能在低端设备/低带宽下可用，高级功能渐进加载 |
| 3 | **离线可用** | 关键内容支持离线缓存（Service Worker / 本地存储） |
| 4 | **断点续传** | 大文件上传/下载支持断点续传 |
| 5 | **图片优化** | 响应式图片（WebP + 多尺寸） + 懒加载 |

### 具体措施

| 场景 | 措施 |
|------|------|
| 教学视频 | 多码率转码（360p/720p/1080p）+ 自适应码率（HLS/DASH） |
| 课件下载 | 预下载 + 离线可用 + 断点续传 |
| 实时互动（课堂） | 低带宽降级为文字模式；WebSocket 压缩 |
| 图片展示 | WebP 格式 + 缩略图 + 懒加载 + CDN |
| 批量操作 | 后台任务 + 进度反馈，不阻塞 UI |

### 设备兼容性

| 设备类型 | 最低标准建议 |
|----------|-------------|
| PC 浏览器 | Chrome 80+ / Edge 80+ / Firefox 78+ |
| 移动端 | Android 8.0+ / iOS 13+ |
| 平板 | iPad（2018+）/ Android 平板（2019+） |
| 特殊设备 | 电子白板（WebView 兼容）、教室一体机 |

## 学生数据安全与隐私合规

### 法律要求

| 法规 | 核心要求 | 架构影响 |
|------|---------|---------|
| **《个人信息保护法》** | 知情同意、最小必要、目的限制 | 明确数据收集范围、用户同意机制 |
| **《儿童个人信息网络保护规定》** | 14 岁以下需监护人同意 | 年龄验证 + 监护人授权流程 |
| **《数据安全法》** | 数据分类分级、安全评估 | 数据分级存储 + 访问控制 |
| **教育部相关规范** | 教育数据不出境、脱敏后使用 | 国内部署 + 脱敏方案 |

### 架构要求

| 要求 | 实现方式 |
|------|---------|
| **最小数据收集** | 只收集业务必需的数据，定期审计数据项 |
| **数据分级存储** | 学生身份信息（高敏感）→ 加密存储 + 独立权限 |
| **访问控制** | 教师只能访问本班学生数据；管理员按角色分权 |
| **审计日志** | 所有敏感数据访问有日志，保留 ≥ 6 个月 |
| **数据脱敏** | 测试/分析环境使用脱敏数据 |
| **数据删除** | 支持学生数据删除请求（毕业/转学/退出） |
| **监护人可见** | 家长可查看孩子的数据使用情况 |
| **数据不出境** | 使用国内云服务，AI 模型调用走国内节点 |

### 安全分级

| 数据级别 | 示例 | 存储要求 | 访问控制 |
|----------|------|---------|---------|
| **极高** | 身份证号、生物信息 | 加密存储 + 单独库 | 仅特定管理员 |
| **高** | 姓名、学号、成绩 | 加密存储 | 教师（本班） + 管理员 |
| **中** | 学习行为数据、作业 | 标准存储 | 教师 + 学生本人 |
| **低** | 脱敏统计数据 | 标准存储 | 相关人员 |

## 教育系统对接

### 常见对接场景

| 对接系统 | 典型接口 | 注意事项 |
|----------|---------|---------|
| **教务系统** | 学生/教师/班级数据同步 | 通常是老旧系统，接口非标准 |
| **统一身份认证** | CAS/OAuth/LDAP | 优先用已有认证系统 |
| **成绩系统** | 成绩导入/导出 | 格式不标准，需适配层 |
| **教育云平台** | 资源同步、数据上报 | 各省/市平台不同 |
| **家校通** | 通知推送、作业反馈 | 微信/钉钉/自建 |

### 对接架构设计

**核心原则**：用适配器模式隔离外部系统差异。

```
业务层 ──→ 统一接口（Adapter Interface）
                    │
                    ├── 教务系统适配器
                    ├── 认证系统适配器
                    └── 教育云适配器
```

| 设计要点 | 说明 |
|----------|------|
| **适配器模式** | 每个外部系统一个适配器，屏蔽差异 |
| **数据映射** | 定义标准数据格式，适配器做转换 |
| **重试机制** | 外部系统不稳定，需有重试 + 幂等 |
| **缓存** | 外部数据本地缓存，减少依赖 |
| **监控** | 监控对接接口的可用性和延迟 |

## AI 在教育场景的特殊约束

| 约束 | 要求 | 架构影响 |
|------|------|---------|
| **内容审查** | AI 生成内容必须经教师审核后才展示给学生 | 审核队列 + 状态管理 |
| **不替代判断** | AI 不做评价性判断（成绩评定、品德评价） | 功能边界在 AI 中间层限制 |
| **可追溯** | 学生与 AI 的对话需记录 | 日志 + 管理后台查看 |
| **年龄适当** | 内容必须适合学生年龄段 | 输出过滤 + 年龄分级 |
| **成本可控** | 学校预算有限 | 严格成本控制 + 缓存优化 |
| **公平性** | 不因 AI 能力差异造成教育不公平 | 降级方案保证基础功能可用 |
