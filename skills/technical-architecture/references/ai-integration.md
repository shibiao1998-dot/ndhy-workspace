# AI 能力集成架构

## AI 角色判断

首先判断 AI 在系统中的角色：

| 角色 | 特征 | 架构影响 |
|------|------|---------|
| **核心** | AI 是产品核心价值（AI 教学、AI 批改） | 必须高可用、有降级、有成本控制 |
| **辅助** | AI 增强体验（智能推荐、辅助填写） | 可降级为非 AI 方案，不影响核心流程 |
| **无** | 系统不涉及 AI | 跳过此步骤 |

## AI 中间层架构

### 为什么需要中间层

❌ 直接调用模型 API：
```
业务代码 ──→ OpenAI API
```
问题：成本不可控、无降级、无缓存、Prompt 散落在代码中、换模型要改业务代码。

✅ 通过中间层调用：
```
业务代码 ──→ AI 中间层 ──→ 模型 API
                │
                ├── 路由（选模型）
                ├── 缓存（省钱）
                ├── 限流（控成本）
                ├── 降级（保可用）
                ├── 监控（看数据）
                └── Prompt 管理（版本控制）
```

### 中间层组件设计

#### 1. 路由层
**职责**：根据请求类型、复杂度、成本预算选择模型。

路由策略：
| 策略 | 说明 | 示例 |
|------|------|------|
| **按复杂度** | 简单任务用小模型，复杂任务用大模型 | 分类→GPT-4o-mini，创作→Claude |
| **按成本** | 成本预算内选最优模型 | 月度预算耗尽→切小模型 |
| **按可用性** | 主模型不可用→切备用模型 | OpenAI 宕机→切 Claude |
| **按地域** | 合规要求→选对应区域模型 | 国内数据→用国产模型 |

#### 2. 缓存层
**职责**：相似请求复用结果，减少 API 调用。

缓存策略：
| 类型 | 说明 | TTL 建议 |
|------|------|---------|
| **精确缓存** | 完全相同的 Prompt → 直接返回 | 24h-7d |
| **语义缓存** | 语义相似的请求 → 返回相似结果 | 1h-24h |
| **模板缓存** | 只有参数不同的模板化请求 | 按业务定义 |

预期效果：缓存命中率 30-60% → 成本降低 30-60%。

#### 3. 降级层
**职责**：模型不可用或超成本时的兜底方案。

降级策略梯度：
```
Level 0: 正常 → 调用主模型
Level 1: 主模型慢 → 切备用模型
Level 2: 所有模型慢 → 返回缓存结果（标记非实时）
Level 3: 所有模型不可用 → 降级为规则/模板方案
Level 4: 完全降级 → 提示用户稍后再试
```

各场景降级方案：
| 场景 | 降级策略 | 用户感知 |
|------|---------|---------|
| 模型服务不可用 | 切备用模型 → 返回缓存 → 提示稍后 | 无感知 → 标记"非实时" → 提示 |
| 响应超时（>10s） | 返回部分结果 + "正在生成完整内容" | 渐进式展示 |
| 成本超限 | 切小模型 → 限制请求频率 → 暂停 AI 功能 | 质量略降 → 排队 → 提示 |
| 内容安全问题 | 过滤 → 重新生成 → 返回安全默认内容 | 无感知 → 提示 |
| 结果质量差 | 重试（换 Prompt/换模型） → 人工兜底 | 延迟略增 |

#### 4. 监控层
**职责**：追踪成本、延迟、质量、使用量。

必监控指标：
| 指标 | 说明 | 告警阈值建议 |
|------|------|-------------|
| 日成本 | 每日 API 调用总成本 | > 日预算的 80% |
| 请求延迟 P95 | 95% 请求的延迟 | > 5s |
| 错误率 | 请求失败比例 | > 5% |
| 缓存命中率 | 缓存命中 / 总请求 | < 20%（说明缓存策略需优化） |
| Token 使用量 | 输入/输出 Token 数 | 异常飙升告警 |

#### 5. Prompt 管理
**职责**：Prompt 模板化、版本控制、A/B 测试。

管理规范：
| 规范 | 说明 |
|------|------|
| **代码外管理** | Prompt 不硬编码在业务代码中，存储在配置/数据库 |
| **版本控制** | 每次修改记录版本号，支持回滚 |
| **变量模板化** | `{user_name}` 等变量标准化，不拼字符串 |
| **A/B 测试** | 新 Prompt 灰度发布，对比效果后全量 |
| **质量评估** | 定义评估标准���准确性/相关性/安全性），定期评估 |

Prompt 版本记录格式：
```
prompt-id: course-summary-v3
version: 3.2
created: 2024-03-15
author: architect
change: 增加输出格式约束，减少幻觉
a/b-status: 灰度 30%
metrics:
  accuracy: 87% (v3.1: 82%)
  avg-tokens: 450 (v3.1: 520)
```

## 成本控制策略

### 成本估算方法

```
月成本 = 日均请求数 × 平均 Token 数 × Token 单价 × 30 × (1 - 缓存命中率)
```

### 控制策略清单

| 策略 | 预期节省 | 实施复杂度 |
|------|---------|-----------|
| **请求缓存** | 30-60% | 低 |
| **分级模型路由** | 20-40% | 中 |
| **Prompt 压缩** | 10-30% | 低 |
| **批量处理** | 15-25%（API 折扣） | 中 |
| **限流** | 控上限 | 低 |
| **预算告警 + 自动降级** | 控上限 | 中 |

### 成本预算设计

| 层级 | 说明 |
|------|------|
| **系统级** | 月度总预算上限 |
| **功能级** | 每个 AI 功能的预算占比 |
| **用户级** | 每个用户的日/月调用上限 |
| **请求级** | 单次请求的 Token 上限 |

## AI 内容安全

### 安全防线

| 层级 | 措施 | 说明 |
|------|------|------|
| **输入过滤** | 过滤恶意/敏感输入 | 防 Prompt 注入、过滤违规内容 |
| **输出审查** | 检查生成内容 | 敏感词过滤、事实性检查（可行时） |
| **人工审核** | 高风险场景人工介入 | 面向学生的内容、正式发布��容 |
| **审计日志** | 记录所有 AI 交互 | 可追溯、可审计 |

### 教育场景特殊要求

- AI 生成的教学内容必须经过教师确认后才展示给学生
- AI 不替代教师做评价性判断（成绩评定、品德评语）
- 学生与 AI 的对话需有日志和家长/教师可查看的机制
- 内容需符合教育适用性标准（年龄适当、价值观正确）
