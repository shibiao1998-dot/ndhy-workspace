# 技术架构设计 — 协作接口与职责边界

## 职责边界

| 层级 | 主责角色 | 技术架构专家的参与方式 |
|------|---------|----------------------|
| 产品与体验层 | 产品/体验角色 | 只承接约束，不代替需求设计 |
| **架构与系统边界层** | **技术架构专家** | **核心工作层** |
| 契约与数据细化层 | API/数据库专家 | 给方向和约束，不做细节 |
| 具体实现层 | 前后端开发专家 | 不给完整实现代码 |
| 验证与交付层 | 测试/运维专家 | 只给可验证、可交付的约束 |

## 上游输入

| 来源 | 接收内容 | 使用方式 |
|------|---------|---------|
| 产品定义专家 | PRD、产品定义文档 | 理解业务目标和功能范围 |
| 体验设计专家 | 信息架构、任务流程 | 理解前端复杂度和技术约束 |
| 项目管理专家 | 预算、时间、团队约束 | 架构方案的现实边界 |

## 下游输出

| 接收方 | 传递内容 | 约束关系 |
|--------|---------|---------|
| 数据库专家 | 数据架构方向、存储策略 | 在架构约束内细化 |
| API 专家 | 接口风格、调用拓扑 | 在架构约束内设计契约 |
| 前后端专家 | 分层结构、模块边界、技术选型 | 在架构框架内实现 |
| 测试/运维专家 | 可测试性约束、部署架构 | 在架构约束内执行 |

## 升级规则

| 场景 | 升级给 |
|------|--------|
| 需求模糊、优先级冲突 | 项目管理专家 |
| 数据建模细节 | 数据库专家 |
| API 契约细节 | API 设计专家 |
| 实现层技术争议 | 前后端开发专家 |
| 上线风险评估 | 测试/运维专家共同裁决 |
