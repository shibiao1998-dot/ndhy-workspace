# 架构模式、分层与模块边界

## 架构模式评估

### 模式速查表

| 模式 | 适用场景 | 团队规模 | 业务复杂度 | 典型技术 |
|------|----------|---------|-----------|---------|
| **单体** | 原型验证、极早期 | 1-3 人 | 低 | 任意全栈框架 |
| **模块化单体** | MVP、业务验证期 | 3-10 人 | 中 | Next.js/Django/Spring Boot + 模块划分 |
| **微服务** | 已验证业务、需独立扩展 | 10+ 人 | 高 | K8s + 服务网格 |
| **Serverless** | 事件驱动、低频高峰 | 任意 | 中低 | AWS Lambda/Cloud Functions |
| **混合** | 部分子域需独立演进 | 5+ 人 | 中高 | 核心单体 + 边缘服务 |

### 模式评估决策树

```
业务已验证？
├─ 否 → 团队 > 5 人？
│   ├─ 否 → 单体 或 模块化单体
│   └─ 是 → 模块化单体
│
└─ 是 → 有子域需独立扩展/独立部署？
    ├─ 否 → 模块化单体
    └─ 是 → 几个子域？
        ├─ 1-2 个 → 混合（核心单体 + 拆出的服务）
        └─ 3+ 个 → 微服务（但确认团队有运维能力）
```

### 模式评估维度（5 维打分）

| 维度 | 权重 | 评估方法 |
|------|------|---------|
| **团队匹配度** | 30% | 团队是否有相关经验？运维能力是否支撑？ |
| **业务匹配度** | 25% | 业务复杂度和模块独立性是否需要这个模式？ |
| **性能匹配度** | 20% | 性能需求是否必须用这个模式才能满足？ |
| **演进适配性** | 15% | 未来 6-12 个月的演进方向是否匹配？ |
| **成本合理性** | 10% | 运维成本、开发成本是否可接受？ |

评分标准：1-5 分，加权总分 ≥ 3.5 才选择。

### 模式演进路径

```
单体 ────→ 模块化���体 ────→ 微服务
          （最推荐起点）     （按需渐进）

关键时刻：
- 单体 → 模块化：代码量 > 5 万行 或 团队 > 5 人
- 模块化 → 微服务：某模块部署频率是其他 3x+ 或 扩展需求独立
```

**铁律**：不跳级。不从单体直接到微服务。

## 系统分层设计

### 标准四层架构

```
┌────────────────────────────────────┐
│          接入层 (Gateway)          │  流量入口、认证鉴权、限流、路由
├────────────────────────────────────��
│          业务层 (Application)      │  业务编排、流程控制、DTO 转换
├────────────────────────────────────┤
│          领域层 (Domain)           │  核心业务逻辑、领域模型、业务规则
├───────────────────���────────────────┤
│       基础设施层 (Infrastructure)   │  存储、缓存、消息队列、外部服务
└────────────────────────────────────┘
```

### 各层职责详解

#### 接入层 (Gateway)
- **职责**：统一入口、请求路由、认证鉴权、限流熔断、协议转换
- **不做**：业务逻辑、数据持久化
- **典型组件**：API Gateway、Nginx、认证中间件
- **关键设计**：
  - 认证统一在此层完成，下层不重复验证
  - 限流策略在此层实施（全局限流 + 用户级限流）
  - 请求日志和链路追踪在此层注入

#### 业务层 (Application)
- **职责**：业务流程编排、跨域协调、DTO 转换、事务管理
- **不做**：核心业务规则判断（属于领域层）
- **关键设计**：
  - 一个用例 = ���个 Application Service 方法
  - 事务边界在此层定义
  - 调用领域层完成核心逻辑，自己只做编排

#### 领域层 (Domain)
- **职责**：核心业务逻辑、领域模型、业务规则、领域事件
- **不做**：数据库操作、外部调用（通过接口抽象）
- **关键设计**：
  - 不依赖任何外部框架
  - 业务规则封装在领域对象中
  - 通过 Repository 接口访问数据（具体实现在基础设施层）

#### 基础设施层 (Infrastructure)
- **职责**：数据持久化、缓存实现、消息收发、外部服务调用
- **不做**：业务逻辑
- **关键设计**：
  - 实现领域层定义的接口
  - 封装外部依赖，上层不直接调用外部 SDK

### 分层铁律

| # | 铁律 | 说明 |
|---|------|------|
| 1 | **单向依赖** | 只能上层依赖下层，不能反向 |
| 2 | **跨层禁止** | 接入层不能直接调用基础设施层 |
| 3 | **接口隔离** | 层间通过接口通信，不暴露实现细节 |
| 4 | **领域纯净** | 领域层不依赖任何框架和基础设施 |

## 模块边界划分

### 划分方法

按**业务能力**划分，不按技术功能��

❌ 错误划分（按技术功能）：
```
modules/
├── controllers/     # 所有 Controller
├── services/        # 所有 Service
├── models/          # 所有 Model
└── utils/           # 所有工具
```

✅ 正确划分（按业务能力）：
```
modules/
├── user/            # 用户管理
│   ├── controller
│   ├── service
│   └── model
├── course/          # 课程管理
│   ├── controller
│   ├── service
│   └── model
└── payment/         # 支付
    ├── controller
    ├── service
    └── model
```

### 划分步骤

1. **列出业务能力**：从产品定义中提取独立的业务能力
2. **识别核心实体**：每个业务能力的核心数据实体是什么
3. **检查聚合性**：相关实体是否必须一起变更？是 → 同一模块
4. **检查独立性**：模块间能否通过接口通信？能 → 边界正确
5. **过三检验**：独立可部署？职责单一？依赖方向正确？

### 模块边界三检验（详细版）

#### 检验 1：独立可部署
- 修改模块 A 后，只需重新测试和部署模块 A
- 模块 A 的数据库表归模块 A 管理，其他模块不直接访问
- 模块间通过 API / 事件通信，不共享数据库表

#### 检验 2：职责单一
- 模块职责可用一句话描述，且不出现"和"
- 一个业务变更只影响一个模块（大多数情况）
- 模块内高内聚：相关功能都在模块内

#### 检验 3：依赖方向正确
- 画出模块依赖图，确认无循环依赖
- 核心模块（用户、权限）被依赖，不依赖业务模块
- 同层模块之间通过事件解耦，不直接调用

### 模块间通信方式选择

| 场景 | 推荐方式 | 说明 |
|------|---------|------|
| 需要即时响应 | 同步调用（REST/gRPC） | 调用方等待结果 |
| 可延迟处理 | 异步消息（MQ） | 发送方不等待 |
| 状态变更通知 | 领域事件 | 解耦最强 |
| 数据查询（跨模块） | 数据同步/CQRS | 避免运行时跨模块查询 |

## 系统上下文设计

### 系统上下文图绘制方法

1. **画出系统边界**：本系统是一个整体
2. **识别外部参与者**：用户角色、管理员、第三方系统
3. **标注交互方式**：API/Webhook/文件/消息
4. **标注数据流向**：哪些数据流入、哪些流出
5. **识别集成风险**：每个集成点的故障影响

### 集成点分类

| 类型 | 特征 | 设计要点 |
|------|------|---------|
| **核心依赖** | 不可用则系统不可用 | 必须有降级方案 |
| **增强功能** | 不可用则部分功能受限 | 标记可降级，不影响核心流程 |
| **异步对接** | 非实时，可容忍延迟 | 消息队列 + 重试 + 幂等 |
| **批量导入/导出** | 定时或按需 | 任务队列 + 进度反馈 |
