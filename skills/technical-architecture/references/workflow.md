# 技术架构设计 — 9 步标准流程

```
Step 1：理解输入
  │  消化产品定义、体验设计、非功能性需求
  │  → 提取：产品目标、核心场景、用户规模、团队技术栈、预算/时间约束
  │  → 确认：现有系统现状、历史技术债、上游约束
  │  → 信息不足时，要求上游补充（不猜测）
  │
Step 2：判断架构模式
  │  → 评估维度：团队规模、业务复杂度、性能要求、演进预期
  │  → 选择模式：单体 / 模块化单体 / 微服务 / Serverless / 混合
  │  → 记录理由 + 阶段适配性（当前 vs 未来演进路径）
  │  → 详细方法 → architecture-patterns.md §架构模式评估
  │
Step 3：划分系统分层与模块边界
  │  → 定义分层结构（接入层/业务层/领域层/基础设施层）
  │  → 划分业务模块（按业务能力，非技术功能）
  │  → 明确模块职责、依赖方向、通信方式
  │  → 过模块边界三检验（见 checklists.md）
  │  → 详细方法 → architecture-patterns.md §分层与模块
  │
Step 4：明确数据流与调用关系
  │  → 绘制系统上下文图（系统 ↔ 外部系统 ↔ 用户）
  │  → 定义核心数据流（读写路径、缓存策略、数据一致性）
  │  → 明确模块间调用方式（同步/异步/事件驱动）
  │  → 标注对外依赖和集成点
  │
Step 5：技术选型
  │  → 按维度评估：成熟度、团队熟悉度、社区活跃度、性能、成本、退出成本
  │  → 输出比较矩阵 + 决策理由
  │  → 记录退出成本和迁移方案
  │  → 详细方法 → tech-selection.md
  │
Step 6：AI 能力集成设计
  │  → 判断 AI 在系统中的角色（核心/辅助/无）
  │  → 设计中间层架构（路由/缓存/降级/监控）
  │  → 制定成本控制策略和降级方案
  │  → Prompt 管理与版本控制
  │  → 详细方法 → ai-integration.md
  │
Step 7：NFR + DevOps 设计
  │  → 性能目标与扩展策略
  │  → 安全架构与合规约束
  │  → 可靠性、容灾与监控告警
  │  → CI/CD 策略、环境规划、发布策略
  │  → 详细方法 → nfr-design.md + devops-delivery.md
  │
Step 8：风险识别与演进路线
  │  → 识别架构风险（单点故障/依赖锁定/成本失控/能力不匹配）
  │  → 对照反模式清单检查
  │  → 规划阶段性演进路线（当前 → 6个月 → 1年）
  │  → 详细方法 → checklists.md §反模式防范
  │
Step 9：输出架构交付包
     → 7 个标准文档 + ADR
     → 过架构质量 7 项检验（见 checklists.md）
     → 按模板组织文档 → templates.md
```

## 设计深度速查

| 深度 | 适用场景 | 覆盖步骤 | 输出物 |
|------|----------|----------|--------|
| **快速评估** | 技术可行性判断、小功能技术方案 | Step 1-2-5 重点，其余概述 | 架构评估简报（轻量版模板） |
| **标准设计** | 新模块或重要改版 | 完整 9 步 | 架构交付包（标准 7 文档 + ADR） |
| **深度设计** | 全新产品从 0 到 1 / 大规模重构 | 完整 9 步 + 多方案对比 + 原型验证 | 完整交付包 + 演进路线图 + 技术预研报告 |

判断不清时向上确认。

## 工作模式决策树

```
收到架构任务
  ├─ 小范围评估（选型对比/模块边界判断/ADR 编写）
  │   → 单独执行
  │
  ├─ 明确边界的专项任务（技术选型比较/NFR设计/AI集成方案）
  │   → 作为 subagent 被调用（默认模式）
  │
  └─ 大 feature / 新产品从 0 到 1 / 多专家强依赖
      ├─ 架构决策影响数据库/API/前后端/测试/运维？
      │   → Agent Team（拉相关专家评审）
      └─ 否 → subagent 模式 + 输出后交相关专家细化
```

**禁止**：
- 简单判断滥用 Agent Team
- 边界未清前大规模并行
- 变成"全能替身"替其他专家做实现

## 触发时首要动作

无论如何触发，第一步永远是：

> "产品要做什么？用户是谁？团队技术栈是什么？有什么约束（预算/时间/现有系统）？"

→ 判断设计深度 → 消化输入 → 按 9 步流程执行

**教育行业项目**：额外读取 education-tech.md 了解行业特殊约束。
