# 性能、安全、可靠性设计（NFR）

## 性能与扩展性设计

### 性能目标设定

按场景定义量化指标：

| 指标类型 | 典型指标 | 目标建议 |
|----------|---------|---------|
| **响应时间** | API P50/P95/P99 | P95 < 200ms（读）/ P95 < 500ms（写） |
| **页面加载** | LCP / FCP / TTI | LCP < 2.5s，FCP < 1.5s |
| **吞吐量** | QPS / TPS | 根据预估并发计算 |
| **并发** | 同时在线用户数 | 预估峰值 × 1.5 安全系数 |
| **错误率** | 5xx 比例 | < 0.1% |

**铁律**：不定义具体数字的性能目标 = 没有性能目标。"要快"不是指标。

### 性能瓶颈识别

常见瓶颈排查路径：
```
慢 → 哪里慢？
├── 网络？→ CDN / 压缩 / 减少请求
├── 计算？→ 算法优化 / 缓存 / 异步
├── 数据库？→ 索引 / 读写分离 / 缓存层
├── 外部依赖？→ 超时设置 / 降级 / 异步化
└── 并发？→ 水平扩展 / 限流 / 排队
```

### 扩展策略

| 策略 | 适用场景 | 实施方式 |
|------|---------|---------|
| **垂直扩展** | 早期、成本敏感 | 增加 CPU/内存 |
| **水平扩展** | 应用层无状态 | 负载均衡 + 多实例 |
| **缓存** | 读多写少 | Redis 缓存热数据 |
| **CDN** | 静态资源、全球用户 | ���缘节点加速 |
| **异步化** | 耗时操作 | 消息队列 + 后台处理 |
| **读写分离** | 数据库瓶颈 | 主写从读 |
| **分库分表** | 单库容量上限 | 按业务/时间分片 |

扩展路径规划模板：
```markdown
当前（0-10K 用户）→ 单实例 + Redis 缓存
中期（10K-100K）→ 水平扩展 + 读写分离 + CDN
远期（100K+）→ 按模块拆服务 + 分库分表
```

## 安全架构设计

### 安全设计分层

| 层级 | 关注点 | 典型措施 |
|------|--------|---------|
| **网络层** | 边界防护 | 防火��、WAF、DDoS 防护、HTTPS |
| **应用层** | 业务安全 | 认证鉴权、输入验证、CSRF/XSS 防护、限流 |
| **数据层** | 数据保护 | 加密存储、脱敏显示、访问控制、备份 |
| **运维层** | 操作安全 | 审计日志、密钥管理、最小权限、漏洞扫描 |

### 认证与授权方案

| 方案 | 适用场景 | 说明 |
|------|---------|------|
| **JWT + Refresh Token** | 单体/微服务 API | 无状态，适合前后端分离 |
| **Session** | 传统 Web 应用 | 简单，但需管理 Session 存储 |
| **OAuth 2.0** | 第三方登录/开放平台 | 标准协议，生态完善 |
| **RBAC** | 角色权限管理 | 角色 → 权限映射，最常用 |
| **ABAC** | 细粒度权限控制 | 基于属性（部门/级别/资源），灵活但复杂 |

### 数据安全要求

| 场景 | 要求 | 实现方式 |
|------|------|---------|
| 传输中数据 | 加密 | TLS 1.2+，HTTPS 强制 |
| 存储中数据 | 敏感字段加密 | AES-256 字段级加密 |
| 密码存储 | 单向哈希 | bcrypt/argon2，salt 唯一 |
| 敏感数据显示 | 脱敏 | 手机号/身份证中间位 * 号 |
| 日志 | 脱敏 | 不记录密码、Token、敏感参数 |

### 威胁模型（STRIDE）

| 威胁类型 | 说明 | 典型防护 |
|----------|------|---------|
| **S** Spoofing（伪装） | 冒充他人身份 | 强认证、MFA |
| **T** Tampering（篡改） | 修改数据 | 签名验证、完整性检查 |
| **R** Repudiation（否认） | 否认操作 | 审计日志、操作记录 |
| **I** Information Disclosure（泄露） | 数据泄露 | 加密、访问控制 |
| **D** Denial of Service（拒绝服务） | 服务不可用 | 限流、CDN、DDoS 防护 |
| **E** Elevation of Privilege（提权） | 越权操作 | 最小权限、权限验证 |

## 可靠性与容灾设计

### SLA 定义

| 级别 | 可用性 | 年度停机 | 适用场景 |
|------|--------|---------|---------|
| 基础 | 99% | ~87h | 内部工具、非关键系统 |
| 标准 | 99.9% | ~8.7h | 一般业务系统 |
| 高可用 | 99.95% | ~4.3h | 重要业务系统 |
| 极高 | 99.99% | ~52min | 核心交易系统 |

**铁律**：SLA 目标必须匹配业务价值。不是所有系统都需要 99.99%。

### 故障域划分

按范围递进：
```
代码 Bug → 单实例故障 → 可用区故障 → 区域故障 → 云厂商故障
```

每一层的应对：
| 故障域 | 检测方式 | 恢复方式 | 恢复时间目标 |
|--------|---------|---------|-------------|
| 代码 Bug | 错误监控、告警 | 回滚部署 | < 15 min |
| 单实例 | 健康检查 | 自动重启/替换 | < 1 min |
| 可用区 | 多 AZ 心跳 | 流量切换 | < 5 min |
| 区域 | 跨区域监控 | DNS 切换 | < 30 min |

### 监控告警体系

四层监控：

| 层 | 监控什么 | 典型工具 |
|----|---------|---------|
| **基础设施** | CPU、内存、磁盘、网络 | Prometheus + Grafana |
| **应用** | 错误率、延迟、QPS、活跃线程 | APM（Datadog/SkyWalking） |
| **业务** | 注册转化率、订单成功率、AI 调用成功率 | 自建指标 + 看板 |
| **用户体验** | Core Web Vitals、页面错误 | RUM（Sentry/LogRocket） |

告警��则设计：
| 级别 | 条件 | 响应 |
|------|------|------|
| **P0 Critical** | 服务不可用 / 数据丢失风险 | 立即响应，< 5 min |
| **P1 High** | 核心功能异常 / 性能严重下降 | 30 min 内响应 |
| **P2 Medium** | 非核心功能异常 / 性能轻微下降 | 4h 内响应 |
| **P3 Low** | 告警预警 / 趋势异常 | 下个工作日处理 |

### 回滚策略

| 类型 | 方式 | 回滚时间 |
|------|------|---------|
| **代码回滚** | 部署上一版本（CI/CD 一键） | < 5 min |
| **数据回滚** | 从备份恢复 / 执行反向迁移脚本 | 视数据量 |
| **配置回滚** | 配置中心回退版本 | < 1 min |
| **全量回滚** | 蓝绿部署切换 / 金丝雀收回 | < 5 min |

**铁律**：每次上线前必须确认回滚方案可行，不可行的上线不批准。
