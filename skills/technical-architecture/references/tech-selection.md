# 技术选型评估框架

## 选型原则（铁律）

| # | 原则 | 说明 |
|---|------|------|
| 1 | **团队会用的 > 理论最优的** | 技术再好，团队不会用 = 失败 |
| 2 | **有退出方案的 > 锁定最深的** | 永远考虑：如果要换，代价多大？ |
| 3 | **社区活跃的 > 功能最全的** | 遇到问题能查到答案比功能全更重要 |
| 4 | **够用就好 > 预留过多** | 不为可能永远不来的需求选型 |
| 5 | **已验证的 > 刚发布的** | 生产环境不做先驱者 |

## 选型评估矩阵

### 6 维度评估

| 维度 | 权重 | 评分标准 |
|------|------|---------|
| **成熟度** | 20% | 1=刚发布 2=早期采用 3=主流使用 4=成熟稳定 5=行业标准 |

| **性能** | 15% | 1=不满足 2=勉强 3=满足 4=余量充足 5=远超需求 |

## 退出成��评估

### 退出成本分级

| 级别 | 特征 | 示例 |
|------|------|------|
| **低** | 标准协议/格式，迁移只需改配置 | PostgreSQL → MySQL（标准 SQL）、S3 兼容存储 |
| **中** | 需要改代码，但改动范围可控 | Express → Fastify（API 框架切换） |
| **高** | 深度耦合，迁移需重写大量代码 | DynamoDB 特有 API、Firebase 全家桶 |
| **极高** | 整个架构依赖该技术，基本等于重写 | 用了云厂商独有的编排服务、特定 PaaS |

### 降低退出成本的策略

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **接口抽象** | 用 Repository/Adapter 模式封装外部依赖 | 数据库、存储、缓存 |
| **标准协议** | 优先选择有标准协议的技术（SQL、HTTP、S3 API） | 所有领域 |
| **配置外置** | 连接信息、端点地址配置化 | 所有外部服务 |
| **数据可导出** | 确保数据可以标准格式导出 | 数据库、SaaS 服务 |
| **合同审查** | 检查供应商锁定条款 | 商业软件/云服务 |

## 常见领域选型参考

### 前端框架

| ��景 | 推荐 | 理由 |
|------|------|------|
| 内容型站点 + SEO | Next.js (App Router) | SSR/SSG 成熟，React 生态 |
| 管理后台 | Next.js / Vite + React | 不需 SSR，Vite 开发体验好 |
| 移动端优先 | React Native / Flutter | 跨平台效率 |
| 嵌入式小组件 | Vanilla JS / Preact | 体积小、无框架依赖 |

### 后端框架

| 场景 | 推荐 | 理由 |
|------|------|------|
| 全栈 TypeScript | Next.js API Routes / tRPC | 前后端类型共享 |
| 高性能 API | Go (Gin/Echo) / Rust (Axum) | 低延迟、高并发 |
| 快速开发 | Python (FastAPI) / Node (NestJS) | 开发效率高 |
| 企业级 | Java (Spring Boot) / .NET | 生态成熟、人才充足 |

### 数据库

| 场景 | 推荐 | 理由 |
|------|------|------|
| 通用关系型 | PostgreSQL | 功能全面、免费、JSON 支持好 |
| 高读写吞吐 | MySQL（调优后）/ TiDB | 成熟、人才多 |
| 文档型 | MongoDB | 灵活 schema，快速迭代 |
| 时序数据 | TimescaleDB / InfluxDB | 专门优化 |
| 缓存 | Redis | 事实标准 |

### AI/LLM 服务

| 场景 | 推荐 | 理由 |
|------|------|------|
| 通用对话/生成 | OpenAI / Claude API | 能力最强，API 最成熟 |
| 成本敏感 | 开源模型（Qwen/DeepSeek）+ 自部署 | 长期成本可控 |
| 中国市场合规 | 国内模型（文心/通义/DeepSeek） | 数据不出境 |
| 嵌入/向量检索 | OpenAI Embeddings + pgvector / Milvus | 性价比高 |

> 以上仅为参考方向，实际选型必须结合团队能力和项目约束。

## 选型决策记录格式

每个关键选型决策，输出一条 ADR（架构决策记录）。格式参见 [templates.md](templates.md) §ADR 模板。

选型 ADR 必须包含：
1. 候选方案列表（≥ 2 个）
2. 评估矩阵打分
3. 选择理由（一句话核心论据）
4. 退出方案
5. 已知风险和缓解措施
