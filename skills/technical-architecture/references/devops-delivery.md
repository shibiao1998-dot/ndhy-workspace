# DevOps 与交付能力设计

## CI/CD 策略

### 流水线设计

标准流水线阶段：
```
代码提交 → 代码检查 → 构建 → 单元测试 → 集成测试 → 部署 → 验证 → 发布
```

各阶段详解：

| 阶段 | 自动化程度 | 失败处理 | 通过标准 |
|------|-----------|---------|---------|
| 代码检查 | 全自动（lint + format） | 阻断，必须修复 | 零 lint 错误 |
| 构建 | 全自动 | 阻断 | 构建成功 |
| 单元测试 | 全自动 | 阻断 | 覆盖率 ≥ 阈值，全部通过 |
| 集成测试 | 全自动 | 阻断 | 核心流程全部通过 |
| 部署到 staging | 全自动 | 手动介入 | 部署成功 + 健康检查通过 |
| 发布到 production | 手动触发/审批 | 自动回滚 | 金丝雀指标正常 |

### 分支策略

| 策略 | 适用场景 | 说明 |
|------|---------|------|
| **Trunk-based** | 小团队、持续部署 | 主干开发，短生命周期分支 |
| **GitHub Flow** | 中等团队 | main + feature 分支 + PR |
| **Git Flow** | 大团队、严格发布 | main + develop + feature + release + hotfix |

推荐：**从 Trunk-based 或 GitHub Flow 开始**，团队超过 10 人再考虑 Git Flow。

### 测试策略

测试金字塔：
```
        /  E2E  \          少量，覆盖核心流程
       /──────────\
      / 集成测试   \       中等，覆盖模块间交互
     /──────────────\
    /   单元测试     \     大量，覆盖业务逻辑
   /──────────────────\
```

| 类型 | 覆盖范围 | 运行时机 | 数量 |
|------|---------|---------|------|
| 单元测试 | 函数/方法级 | 每次提交 | 最多 |
| 集成测试 | 模块间交互 | PR 合并前 | 中等 |
| E2E 测试 | 完整用户流程 | 部署到 staging 后 | 最少（核心流程） |

## 环境规划

### 标准环境

| 环境 | 用途 | 数据 | 部署方式 |
|------|------|------|---------|
| **local** | 开发调试 | Mock/本地 | 手动 |
| **dev** | 开发联调 | 测试数据 | 自动（提交触发） |
| **staging** | 预发布验证 | 生产数据脱敏副本 | 自动（PR 合并触发） |
| **production** | 正式环境 | 真实数据 | 手动触发 + 审批 |

### 环境一致性

**铁律**：staging 必须和 production 配置一致（相同的容器镜像、相同的环境变量格式、相同的基础设施配置）。

差异仅允许：
- 数据库连接地址
- 第三方服务的 sandbox/production 端点
- 资源规格（staging 可以规格小一些）

### 配置管理

| 类型 | 管理方式 | 示例 |
|------|---------|------|
| **应用配置** | 环境变量 / 配置文件 | 端口、日志级别 |
| **密钥/凭证** | 密钥管理服务（Vault/AWS Secrets Manager） | API Key、数据库密码 |
| **功能开关** | Feature Flag 服务 | 新功能灰度 |

**禁止**：密钥硬编码在代码中、配置文件提交到 Git。

## 发布策略

### 发布方式选择

| 方式 | 适用场景 | 风险 | 回滚速度 |
|------|---------|------|---------|
| **全量发布** | 低风险变更、内部工具 | 高 | 需重新部署 |
| **蓝绿部署** | 需要快速回滚 | 低 | 秒级（切流量） |
| **金丝雀发布** | 用户量大、需验证 | 最低 | 秒级（收回流量） |
| **滚动更新** | 容器化环境默认 | 中 | 中等 |

推荐路径：
```
早期（< 1K 用户）→ 全量发布（简单够用）
中期（1K-10K）→ 蓝绿部署（快速回滚）
后期（10K+）→ 金丝雀发布（最安全）
```

### 金丝雀发布流程

```
1. 部署新版本到 1% 流量
2. 观察 15 分钟：错误率、延迟、业务指标
3. 指标正常 → 扩大到 10%
4. 观察 30 分钟
5. 指标正常 → 扩大到 50%
6. 观察 1 小时
7. 指标正常 → 全量发布
8. 任何阶段指标异常 → 立即回滚到 0%
```

### 发布检查清单

发布前必须确认：
- [ ] staging 环境验证通过
- [ ] 回滚方案已确认可执行
- [ ] 数据库迁移已验证（可正向 + 可反向）
- [ ] 监控告警已就绪
- [ ] 发布时间窗口合理（避开业务高峰）
- [ ] 相关人员已通知
- [ ] Feature Flag 就绪（如需灰度）

## 基础设施即代码（IaC）

### 原则

| 原则 | 说明 |
|------|------|
| **一切皆代码** | 基础设施配置存储在 Git，可追溯 |
| **幂等性** | 多次执行结果一致 |
| **不可变基础设施** | 不修改运行中的实例，而是替换 |
| **环境一致性** | 通过代码保证环境间配置一致 |

### 工具选择

| 场景 | 推荐 | 说明 |
|------|------|------|
| 云资源管理 | Terraform / Pulumi | 多云支持 |
| 容器编排 | Docker Compose（单机）/ K8s（集群） | 按规模选择 |
| 配置管理 | Ansible（简单）/ Helm（K8s） | 按环境选择 |
| CI/CD | GitHub Actions / GitLab CI | 按代码仓库选择 |

## 日志规范

### 日志分级

| 级别 | 用途 | 生产环境 |
|------|------|---------|
| ERROR | 需要立即关注的错误 | ✅ 输出 |
| WARN | 潜在问题但不影响功能 | ✅ 输出 |
| INFO | 关键业务事件 | ✅ 输出 |
| DEBUG | 调试信息 | ❌ 关闭 |

### 日志内容要求

每条日志必须包含：
- **时间戳**（ISO 8601）
- **级别**
- **请求 ID**（链路追踪）
- **模块/服务名**
- **消息**

**禁止记录**：密码、Token、完整信用卡号、身份证号。
