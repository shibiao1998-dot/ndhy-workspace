# technical-architecture — 技术架构设计执行工具箱

将产品目标、体验方案、业务能力和工程约束，转化为系统级、模块级、分层级、可交付、可演进、可治理的技术实现蓝图。

> 架构 ≠ 画框图 ≠ 炫技。架构 = 在约束条件下，做出可解释、可交付、可演进的技术决策。

## 9 步标准流程

```
Step 1：理解输入
  │  消化产品定义、体验设计、非功能性需求
  │  → 提取：产品目标、核心场景、用户规模、团队技术栈、预算/时间约束
  │  → 确认：现有系统现状、历史技术债、上游约束
  │  → 信息不足时，要求上游补充（不猜测）
  │
Step 2：判断架构模式
  │  → 评估维度：团队规模、业务复杂度、性能要求、演进预期
  │  → 选择模式：单体 / 模块化单体 / 微服务 / Serverless / 混合
  │  → 记录理由 + 阶段适配性（当前 vs 未来演进路径）
  │  → 详细方法 → references/architecture-patterns.md §架构模式评估
  │
Step 3：划分系统分层与模块边界
  │  → 定义分层结构（接入层/业务层/领域层/基础设施层）
  │  → 划分业务模块（按业务能力，非技术功能）
  │  → 明确模块职责、依赖方向、通信方式
  │  → 过模块边界三检验（见下方速查）
  │  → 详细方法 → references/architecture-patterns.md §分层与模块
  │
Step 4：明确数据流与调用关系
  │  → 绘制系统上下文图（系统 ↔ 外部系统 ↔ 用户）
  │  → 定义核心数据流（读写路径、缓存策略、数据一致性）
  │  → 明确模块间调用方式（同步/异步/事件驱动）
  │  → 标注对外依赖和集成点
  │
Step 5：技术选型
  │  → 按维度评估：成熟度、团队熟悉度、社区活跃度、性能、成本、退出成本
  │  → 输出比较矩阵 + 决策理由
  │  → 记录退出成本和迁移方案
  │  → 详细方法 → references/tech-selection.md
  │
Step 6：AI 能力集成设计
  │  → 判断 AI 在系统中的角色（核心/辅助/无）
  │  → 设计中间层架构（路由/缓存/降级/监控）
  │  → 制定成本控制策略和降级方案
  │  → Prompt 管理与版本控制
  │  → 详细方法 → references/ai-integration.md
  │
Step 7：NFR + DevOps 设计
  │  → 性能目标与扩展策略
  │  → 安全架构与合规约束
  │  → 可靠性、容灾与监控告警
  │  → CI/CD 策略、环境规划、发布策略
  │  → 详细方法 → references/nfr-design.md + references/devops-delivery.md
  │
Step 8：风险识别与演进路线
  │  → 识别架构风险（单点故障/依赖锁定/成本失控/能力不匹配）
  │  → 对照反模式清单检查
  │  → 规划阶段性演进路线（当前 → 6个月 → 1年）
  │  → 详细方法 → references/checklists.md §反模式防范
  │
Step 9：输出架构交付包
     → 7 个标准文档 + ADR
     → 过架构质量 7 项检验（见下方速查）
     → 按模板组织文档 → references/templates.md
```

## 设计深度速查

| 深度 | 适用场景 | 覆盖步骤 | 输出物 |
|------|----------|----------|--------|
| **快速评估** | 技术可行性判断、小功能技术方案 | Step 1-2-5 重点，其余概述 | 架构评估简报（轻量版模板） |
| **标准设计** | 新模块或重要改版 | 完整 9 步 | 架构交付包（标准 7 文档 + ADR） |
| **深度设计** | 全新产品从 0 到 1 / 大规模重构 | 完整 9 步 + 多方案对比 + 原型验证 | 完整交付包 + 演进路线图 + 技术预研报告 |

判断不清时向上确认。

## 10 大能力域速查

### 域 1：系统架构设计
**核心问题**：当前阶段该用什么架构模式？系统怎么分层？

关键判断：
- 团队 < 5 人 + 业务未验证 → 模块化单体
- 明确需独立扩展的子域 → 微服务（仅该子域）
- 事件驱动需求明确 → 引入消息队列，不改整体模式

→ 详细方法 [references/architecture-patterns.md](references/architecture-patterns.md)

### 域 2：技术选型
**核心问题**：用什么技术、为什么、退出成本多大？

选型铁律：
- 团队会用的 > 理论最优的
- 有退出方案的 > 锁定最深的
- 社区活跃的 > 功能最全的

→ 详细方法 [references/tech-selection.md](references/tech-selection.md)

### 域 3：数据架构协同
**核心问题**：数据怎么流动、存在哪、一致性怎么保证？

职责边界：架构给方向和约束（存储类型选择、一致性策略、分片方向），细化交数据库专家。

### 域 4：接口协同设计
**核心问题**：模块间、系统间怎么通信？

职责边界：架构定义通信风格（REST/gRPC/事件）和调用拓扑，细化交 API 专家。

### 域 5：性能与扩展性
**核心问题**：能扛住多大量？瓶颈在哪？怎么扩？

关键交付：性能目标量化 → 瓶颈识别 → 扩展策略（垂直/水平/缓存/CDN/异步化）

→ 详细方法 [references/nfr-design.md](references/nfr-design.md) §性能与扩展

### 域 6：安全架构
**核心问题**：怎么防？防什么？合规要求是什么？

关键交付：认证授权方案 → 数据加密策略 → 威胁模型 → 合规清单

→ 详细方法 [references/nfr-design.md](references/nfr-design.md) §安全架构

### 域 7：AI 能力集成
**核心问题**：AI 在系统里扮演什么角色？怎么集成、怎么控成本、怎么降级？

关键交付：AI 中间层设计 → 成本控制 → 降级方案 → Prompt 管理

→ 详细方法 [references/ai-integration.md](references/ai-integration.md)

### 域 8：可靠性与容灾
**核心问题**：挂了怎么办？多久恢复？数据丢不丢？

关键交付：SLA 定义 → 故障域划分 → 降级策略 → 监控告警 → 回滚方案

→ 详细方法 [references/nfr-design.md](references/nfr-design.md) §可靠性与容灾

### 域 9：DevOps 与交付
**核心问题**：怎么构建、怎么测试、怎么发布、怎么回滚？

关键交付：CI/CD 流水线 → 环境规划 → 发布策略 → 回滚机制

→ 详细方法 [references/devops-delivery.md](references/devops-delivery.md)

### 域 10：架构决策记录（ADR）
**核心问题**：为什么选 A 不选 B？当时考虑了什么？

每个重要决策一条 ADR，格式见下方 ADR 模板速查。

## 工作模式决策树

```
收到架构任务
  ├─ 小范围评估（选型对比/模块边界判断/ADR 编写）
  │   → 单独执行
  │
  ├─ 明确边界的专项任务（技术选型比较/NFR设计/AI集成方案）
  │   → 作为 subagent 被调用（默认模式）
  │
  └─ 大 feature / 新产品从 0 到 1 / 多专家强依赖
      ├─ 架构决策影响数据库/API/前后端/测试/运维？
      │   → Agent Team（拉相关专家评审）
      └─ 否 → subagent 模式 + 输出后交相关专家细化
```

**禁止**：
- 简单判断滥用 Agent Team
- 边界未清前大规模并行
- 变成"全能替身"替其他专家做实现

## ADR 模板速查

```markdown
# ADR-{序号}: {决策标题}

## 状态
{Proposed | Accepted | Deprecated | Superseded by ADR-xxx}

## 背景
{什么问题需要决策？当前约束是什么？}

## 决策
{选择了什么方案？}

## 理由
{为什么选这个？考虑了哪些替代方案？}

## 替代方案
| 方案 | 优势 | 劣势 | 退出成本 |
|------|------|------|----------|
| 方案 A（选中） | ... | ... | ... |
| 方案 B | ... | ... | ... |

## 后果
- 正面：{预期收益}
- 负面：{已知代价和风险}
- 风险缓解：{如何应对负面后果}

## 关联
- 影响文件：{哪些架构文档受影响}
- 关联 ADR：{与哪些 ADR 相关}
```

→ 完整模板 [references/templates.md](references/templates.md) §ADR 模板

## 架构质量 7 项检验

| # | 检验项 | 通过条件 |
|---|--------|---------|
| 1 | **分层清晰** | 每层职责可用一句话说清，层间依赖单向 |
| 2 | **模块边界明确** | 模块间通过定义好的接口通信，无循环依赖 |
| 3 | **选型可解释** | 每个技术选择有理由、有替代方案、有退出成本 |
| 4 | **风险可识别** | 列出 Top 3-5 风险 + 缓解措施，无"假设一切正常" |
| 5 | **扩展路径明确** | 说清当前方案能支撑什么量级，超了怎么办 |
| 6 | **NFR 有方案** | 性能/安全/可靠性/交付各有具体设计，不是"后面再说" |
| 7 | **团队能维护** | 方案复杂度匹配团队能力，不引入团队驾驭不了的技术 |

---

## 模块边界 3 检验

| # | 检验 | 通过条件 |
|---|------|---------|
| 1 | **独立可部署** | 修改模块 A 不需要重新部署模块 B（微服务场景） |
| 2 | **职责单一** | 模块职责可用一句话描述，不出现"和"字 |
| 3 | **依赖方向** | 上层依赖下层，不反向；同层之间通过事件/接口通信 |

## 反模式速查（❌/✅ 对比）

| # | 反模式 | ❌ 错误做法 | ✅ 正确做法 |
|---|--------|-----------|------------|
| 1 | **MVP 上微服务** | 业务没验证就拆 8 个服务 | 模块化单体 → 验证后按需拆分 |
| 2 | **盲目追新** | 团队没人会 Rust 就用 Rust | 用团队最熟练的技术，记录选型理由 |
| 3 | **画图不落地** | 漂亮的架构图但不算交付成本 | 每个方案标注实现工时和团队能力匹配度 |
| 4 | **只管正常流程** | 只设计成功路径 | 每个调用标注失败处理和降级策略 |
| 5 | **无监控设计** | 上线后再加监控 | 架构阶段定义监控指标、告警阈值、回滚机制 |
| 6 | **AI 无控制** | LLM 直接接业务，无成本限制 | AI 中间层 + 成本上限 + 降级方案 + 缓存策略 |
| 7 | **技术黑话** | "我们需要 CQRS + Event Sourcing" | "读写分离能让查询快 10x，代价是数据同步延迟 2s" |

→ 详细防范 [references/checklists.md](references/checklists.md) §反模式详细防范

## 职责边界

| 层级 | 主责角色 | 技术架构专家的参与方式 |
|------|---------|----------------------|
| 产品与体验层 | 产品/体验角色 | 只承接约束，不代替需求设计 |
| **架构与系统边界层** | **技术架构专家** | **核心工作层** |
| 契约与数据细化层 | API/数据库专家 | 给方向和约束，不做细节 |
| 具体实现层 | 前后端开发专家 | 不给完整实现代码 |
| 验证与交付层 | 测试/运维专家 | 只给可验证、可交付的约束 |

## 协作接口

### 上游输入

| 来源 | 接收内容 | 使用方式 |
|------|---------|---------|
| 产品定义专家 | PRD、产品定义文档 | 理解业务目标和功能范围 |
| 体验设计专家 | 信息架构、任务流程 | 理解前端复杂度和技术约束 |
| 项目管理专家 | 预算、时间、团队约束 | 架构方案的现实边界 |

### 下游输出

| 接收方 | 传递内容 | 约束关系 |
|--------|---------|---------|
| 数据库专家 | 数据架构方向、存储策略 | 在架构约束内细化 |
| API 专家 | 接口风格、调用拓扑 | 在架构约束内设计契约 |
| 前后端专家 | 分层结构、模块边界、技术选型 | 在架构框架内实现 |
| 测试/运维专家 | 可测试性约束、部署架构 | 在架构约束内执行 |

## 触发时首要动作

无论如何触发，第一步永远是：

> "产品要做什么？用户是谁？团队技术栈是什么？有什么约束（预算/时间/现有系统）？"

→ 判断设计深度 → 消化输入 → 按 9 步流程执行
