# 7 大深层需求挖掘技法

每个技法都有明确的适用场景。不要一股脑全用——根据当前对话的卡点，选择最合适的 1-2 个技法。

---

## ① 5-Why 追问法（纵向挖掘）

**适用场景**：用户给的是"解法"而非"问题"，需要纵向穿透找到真实需求。

**操作步骤**：
1. 用户说出一个需求/方案
2. 追问"为什么需要这个？"
3. 用户回答后，继续追问"为什么？"
4. 重复 3-5 次，直到触达不可再分的底层问题
5. 用底层问题重新定义需求

**实战示例**：

```
用户："我需要一个导出 Excel 的功能"
你：为什么要导出？
用户："要做数据分析"
你：为什么要用 Excel 分析？
用户："因为系统里看不了趋势"
→ 真实需求：系统内的数据可视化 / 趋势分析能力
```

**使用要点**：
- 语气要自然，不要像审讯。可以穿插共情："确实，在系统里看不了趋势挺不方便的"
- 不一定要严格追问 5 次，触达真实问题就停
- 如果用户开始不耐烦，说明追问方式太机械，换用其他技法

---

## ② 场景还原法（横向展开）

**适用场景**：需求的使用场景不清晰，需要横向展开了解完整上下文。

**操作步骤**：
1. 请用户从头到尾描述他的实际操作过程
2. 关键引导语："你现在是怎么做的？一步一步带我走一遍"
3. 在用户叙述过程中，注意捕捉三类信号：
   - **理所当然的环节**：用户觉得正常但实际有问题
   - **隐藏的需求**：用户忘了提但操作中暴露出来的
   - **最痛的点**：用户情绪变化最大的环节

**实战示例**：

```
你："你现在处理客户投诉的流程是什么？带我走一遍"
用户："先在邮箱收到投诉，然后复制到 Excel 里，再分类，然后分配给对应的人……"
你："等等，你说复制到 Excel——每天要复制几次？"
用户："十几次吧，挺烦的"
→ 隐藏需求：投诉自动录入和分类系统
```

**使用要点**：
- 让用户讲，你只在关键节点追问
- 留意"挺烦的"、"每次都要"、"特别麻烦"这类情绪词——那是痛点信号
- 可以画简单的流程图帮用户理清步骤

---

## ③ 反面否定法（测试必要性）

**适用场景**：怀疑是伪需求或优先级很低，需要测试真实必要性。

**操作步骤**：
1. 提出假设性问题："如果不做这个功能，会发生什么？"
2. 追问最坏结果："最坏的情况是什么？"
3. 根据用户反应判断：
   - 能清晰回答具体损失 → 需求是真实的
   - 说不清楚 / 支支吾吾 → 需求可能是伪需求或优先级低

**实战示例**：

```
用户："我想加一个深色模式"
你："如果不做深色模式，最坏会怎样？"
用户："嗯……也不会怎样，就是觉得别人都有"
→ 这是一个优先级较低的需求，不是核心痛点
```

```
用户："我需要自动备份功能"
你："如果不做自动备份，最坏会怎样？"
用户："上个月数据丢了一次，客户差点投诉到总部"
→ 这是一个真实且紧迫的需求
```

**使用要点**：
- 语气要温和，不是质疑用户，而是帮他理清优先级
- 这个技法也适合在需求太多时做"减法"

---

## ④ 极端假设法（突破思维定式）

**适用场景**：用户被现有约束框住了思路，需要帮他看到理想态。

**操作步骤**：
1. 移除所有约束："如果没有任何限制——不考虑钱、不考虑技术、不考虑时间——你最理想的方案是什么？"
2. 让用户畅想理想态
3. 从理想态往回拉，找到现实中最接近的可行方案

**实战示例**：

```
你："假设不考虑任何限制，你最理想的客户管理是什么样的？"
用户："我希望客户一打电话来，系统就自动显示他所有的历史记录、偏好、上次聊的内容"
→ 真实渴望：客户 360 度视图 + 来电弹屏。可以分阶段实现
```

**使用要点**：
- 适合用户反复说"做不到"、"太贵了"的时候
- 理想态暴露的是用户真正渴望的东西，即使不能完全实现也指明了方向
- 从理想态"降级"时，让用户参与取舍："如果只能实现其中一部分，你最想要哪个？"

---

## ⑤ 对比锚定法（让模糊变具体）

**适用场景**：用户描述太抽象（"我要好用的"、"我要简洁的"），需要用具体参照物锚定。

**操作步骤**：
1. 请用户给出参照物："你能举一个你觉得做得好的产品/功能吗？"
2. 追问差异："好在哪？""跟 XX 相比，你期望的区别是什么？"
3. 从对比中提炼出具体的需求规格

**实战示例**：

```
用户："我想要一个好用的仪表盘"
你："你见过觉得做得好的仪表盘吗？"
用户："Notion 的那种，简洁但信息密度高"
你："具体哪个部分最打动你？"
用户："可以自定义布局，而且拖拽很流畅"
→ 具体需求：可自定义布局 + 流畅拖拽交互 + 高信息密度
```

**使用要点**：
- 用户描述抽象感受很困难，但做对比很擅长
- 给他一个锚点，他就能说清楚
- 如果用户举不出例子，你可以主动提供 2-3 个选项供他对比

---

## ⑥ 利益相关者扫描法（多角色视角）

**适用场景**：需求涉及多个角色，需要排查不同角色的诉求是否冲突。

**操作步骤**：
1. 列出所有相关角色："除了你，还有谁会用到这个？"
2. 逐个探索诉求："他们的想法和你一样吗？"
3. 补充关键角色视角："你的老板 / 客户 / 下属会怎么看？"
4. 识别诉求冲突并记录

**实战示例**：

```
用户："我们需要一个新的审批流程"
你："除了你，还有谁参与审批？"
用户："我发起，部门经理审批，财务最终确认"
你："部门经理和财务对现有流程有什么不满？"
用户："经理觉得太慢，财务觉得信息不全"
→ 需求冲突：经理要快（简化流程）vs 财务要全（信息完整）。需要平衡
```

**使用要点**：
- 同一需求，不同角色的诉求可能完全不同甚至矛盾
- 不排查利益相关者就定需求，上线后一定有人不满意
- 特别注意"沉默的利益相关者"——没说话但会受影响的人

---

## ⑦ 拆解投票法（明确优先级）

**适用场景**：挖出了一堆需求，需要明确优先级。

**操作步骤**：
1. 把挖出来的所有需求列成清单（不评价、不排序）
2. 让用户做强制取舍："如果只能选 3 个，你选哪 3 个？"
3. 追问排序理由："为什么这个排第一？"
4. 用取舍结果校准优先级

**实战示例**：

```
你："我们梳理出来 8 个需求点。如果第一期只能做 3 个，你选哪 3 个？"
用户："（思考）……自动录入、分类标签、导出报告"
你："自动录入排第一的原因是？"
用户："因为现在手动录入每天浪费 2 小时"
→ 优先级逻辑：按"痛苦程度 × 发生频率"排序
```

**使用要点**：
- 强制取舍最能暴露真实优先级——用户平时说"都重要"，逼他选就能分出来
- 数字可以调整（"只能选 3 个"→"只能选 1 个"）来进一步聚焦
- 追问排序理由比排序结果本身更有价值——它暴露了用户的优先级逻辑

---

## 技法选择指南

| 你遇到的情况 | 推荐技法 |
|-------------|---------|
| 用户给了方案，不知道他要解决什么问题 | ① 5-Why 追问法 |
| 不清楚需求的使用场景和上下文 | ② 场景还原法 |
| 怀疑需求是"伪需求"或优先级低 | ③ 反面否定法 |
| 用户被约束框住，想不到更好方案 | ④ 极端假设法 |
| 用户描述太抽象，无法转化为具体规格 | ⑤ 对比锚定法 |
| 需求涉及多方角色，可能有诉求冲突 | ⑥ 利益相关者扫描法 |
| 需求太多，优先级混乱 | ⑦ 拆解投票法 |
| 不确定从哪入手 | 先用 ② 场景还原法了解全貌，再按需选择 |
