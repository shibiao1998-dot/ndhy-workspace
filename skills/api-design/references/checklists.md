## 契约质量检查清单

### 完整检查清单（交付前逐项过）

#### A. 资源与动作

| # | 检查项 | 通过条件 | 通过？ |
|---|--------|---------|--------|
| A1 | 资源边界清晰 | 每个资源职责一句话说清，无"和"字 | ⬜ |
| A2 | 资源层级合理 | 从属关系用嵌套路径，关联关系用查询参数 | ⬜ |
| A3 | 动作语义准确 | HTTP Method 与业务语义匹配 | ⬜ |
| A4 | 无动词 URL | 路径中无动词（动作端点除外） | ⬜ |
| A5 | 资源名复数 | 所有资源端点使用复数名 | ⬜ |
| A6 | 状态机已定义 | 有状态的资源有状态枚举 + 合法转换 | ⬜ |

#### B. Schema 与字段

| # | 检查项 | 通过条件 | 通过？ |
|---|--------|---------|--------|
| B1 | 字段命名全局一致 | 同一命名风格，同含义字段同名 | ⬜ |
| B2 | 必填字段明确 | Create/Update 的 required 列表正确 | ⬜ |
| B3 | 字段类型合理 | 无 string 万能类型，金额用 decimal/整数 | ⬜ |
| B4 | 嵌套不超过 3 层 | 深嵌套拆为独立端点 | ⬜ |
| B5 | 响应结构统一 | 单资源用 `{data:{}}`, 列表用 `{data:[],pagination:{}}` | ⬜ |
| B6 | Nullable 已标注 | 可能为 null 的字段标注 `nullable: true` | ⬜ |
| B7 | 示例值完整 | 每个 schema 有 example | ⬜ |

#### C. 错误与异常

| # | 检查项 | 通过条件 | 通过？ |
|---|--------|---------|--------|
| C1 | 错误结构统一 | 所有错误用 `{error:{code,message,details,request_id}}` | ⬜ |
| C2 | 错误码已注册 | 每个端点的错误码在注册表中有记录 | ⬜ |
| C3 | Happy Path + 失败语义 | 每个端点至少定义了主要的错误场景 | ⬜ |
| C4 | 无信息泄露 | 5xx 错误不暴露内部细节 | ⬜ |
| C5 | HTTP 状态码正确 | 创建=201, 删除=204, 业务错误=422 | ⬜ |

| D4 | 幂等策略已标注 | 写入端点有幂等键说明 | ⬜ |
| D5 | 版本策略一致 | 全局版本策略统一 | ⬜ |

| # | 检查项 | 通过条件 | 通过？ |
|---|--------|---------|--------|
| E1 | 认证方式已定义 | Token 类型、获取方式、刷新方式已说明 | ⬜ |
| E2 | 端点权限已标注 | 每个端点有 security 标注 | ⬜ |
| E3 | 行级权限已说明 | 数据范围因角色而异时有明确说明 | ⬜ |

## 反模式详细防范

### 反模式 1：页面驱动契约

**症状**：API 响应字段和某个页面的 UI 元素一一对应。

**危害**：
- 页面改版 → API 必须跟着改
- 不同页面使用同一资源时，API 被迫返回冗余字段或多端点
- 契约不稳定，改一个页面影响整条链路

**案例**：

```
❌ 页面驱动：
GET /course-card-data → { title, teacherAvatar, enrollCount, priceText }
// 这是个 UI 拼装结果，不是资源

✅ 资源驱动：
GET /courses/{id} → { id, title, description, status, teacher_id, price, ... }
// 稳定的资源契约。页面展示逻辑在前端/BFF 处理
```

**防范**：
- 问"如果这个页面不存在了，这个接口还有意义吗？"
- 答案是"没意义" → 这是页面接口，不是资源接口

### 反模式 2：万能接口

**症状**：一个端点接受大量可选参数，根据参数组合返回完全不同的数据结构。

**危害**：
- 实现复杂度指数增长
- 错误场景难以穷举
- 文档维护成本极高
- 性能优化无从下手

**案例**：

```
❌ 万能接口：
GET /data?type=course&include=teacher,students,grades&format=summary
// 一个端点干了列表/详情/统计/关联查询四件事

✅ 职责单一：
GET /courses                      # 课程列表
GET /courses/{id}                 # 课程详情
GET /courses/{id}/students        # 课程学生
GET /courses/{id}/grade-summary   # 成绩统计
```

**防范**：
- 一个端点能否用一句话说清职责？不能 → 拆

### 反模式 3：先实现后补契约

**症状**：后端先把接口写完了，再"照着代码补"OpenAPI 文档。

**危害**：
- 契约变成实现的附庸，而不是设计的产出
- 实现中的"方便"被固化为契约（如内部字段名泄露）
- 前端已按实现联调，改正成本高

**防范**：
- 铁律：先有契约再有实现。契约是设计产出，不是代码翻译
- 后端发现契约有问题 → 走契约��更流程，不擅自改

### 反模式 4：只有 Happy Path

**症状**：文档只定义了成功响应，没有错误场景。

**危害**：
- 前端不知道错误时该展示什么
- 后端不知道该返回什么错误码
- 测试不知道该验证什么边界

**案例**：

```
❌ 只有 Happy Path：
POST /courses → 201 Created

✅ 完整错误语义：
POST /courses
  → 201 Created（成功）
  → 400 VALIDATION_FAILED（参数错误）
  → 401 UNAUTHORIZED（未认证）
  → 403 FORBIDDEN（无权限）
  → 409 DUPLICATE_COURSE（名称重复）
  → 422 COURSE_CATEGORY_INACTIVE（分类已停用）
```

**防范**：
- 每个端点至少问："认证失败怎么办？权限不够怎么办？参数错了怎么办？业务规则不满足怎么办？"

### 反模式 5：临时接口长期化

**症状**："先出个快速接口，后面再规范"，结果永远没有"后面"。

**危害**：
- 技术债务积累
- 多个消费方依赖临时契约后改不动
- 正式契约和临时契约共存，维护混乱

**防范**：
- 所有临时接口必须标注 `x-temporary: true` + `x-sunset-date: YYYY-MM-DD`
- 同时记录：正式替代方案的计划（什么时候做、谁负责）
- 超过 sunset date 未替代 → 上报项目管理专家

### 反模式 6：忽视兼容成本

**症状**：随意修改已发布接口的字段名、类型、结构，不评估影响。

**危害**：
- 已上线客户端崩溃
- 旧数据无法解析
- 回滚困难

**防范**：
- 每次修改已发布接口前，必须做兼容性评估（见下方）
- 破坏性变更必须有：迁移方案 + 共存期 + 通知计划

## 一致性审查

### 与数据模型对齐

| 检查项 | 通过条件 |
|--------|---------|
| API 字段名与数据库字段语义一致 | 同含义字段不用不同名称 |
| API 关系映射与数据库关系一致 | 1:N 在 API 中体现为嵌套或过滤 |
| API 枚举值与数据库枚举一致 | 状态值名称相同 |
| API 不暴露数据库内部细节 | 无自增 ID 暴露、无内部表名 |

### 与前端消费对齐

| 检查项 | 通过条件 |
|--------|---------|
| 列表接口返回前端列表页所需字段 | 不需要 N+1 请求 |
| 详情接口返回前端详情页所需字段 | 不缺少展示必需字段 |
| 表单提交结构与前端表单对齐 | 字段名/类型一致 |
| 状态值可直接用于 UI 判断 | 无需二次映射 |

### 与后端实现对齐

| 检查项 | 通过条件 |
|--------|---------|
| 端点性能可接受 | 单次查询不超过预期时间 |
| 批量操作有上限 | 批量端点有数量限制 |
| 复杂查询可实现 | 过滤/排序参数有索引支撑 |
| 事务边界合理 | 单个端点的写入在一个事务内 |

### 跨端点一致性

| 检查项 | 通过条件 |
|--------|---------|
| 同一资源在不同端点的结构一致 | 列表 item 结构 ⊂ 详情结构 |
| 关联 ID 引用一致 | A 端点返回的 ID 可直接用于 B 端点 |
| 错误码不冲突 | 同一错误码在不同端点含义相同 |
| 时间格式统一 | 所有时间字段格式一致 |

## 兼容性评估指南

### 变更分类

| 类型 | 风险 | 示例 |
|------|------|------|
| **新增（Additive）** | 🟢 安全 | 新增可选字段、新增端点、新增错误码 |
| **修改（Modification）** | 🟡 需评估 | 放宽约束（安全）、收紧约束（破坏性） |
| **删除（Removal）** | 🔴 破坏性 | 删除字段、删除端点、删除错误码 |
| **重命名（Rename）** | 🔴 破坏性 | 等同于"删除旧的 + 新增新的" |

### 兼容性评估流程

```
收到变更请求
  |- 纯新增？ -> 🟢 安全，直接执行
  |
  |- 修改已有？
  |   |- 放宽约束？ -> 🟢 安全（maxLength 增大、选填字段增加）
  |   |- 收紧约束？ -> 🔴 评估影响范围
  |   +- 改变语义？ -> 🔴 评估影响范围
  |
  +- 删除/重命名？ -> 🔴 必须走废弃流程
```

### 影响范围评估模板

```markdown
### 变更评估：{变更描述}

**变更类型**：🔴 破坏性 / 🟡 需注意 / 🟢 安全

**影响分析**：
| 消费方 | 是否受影响 | 影响程度 | 需要的动作 |
|--------|----------|---------|-----------|
| 前端 Web | 是/否 | 高/中/低 | {具体改动} |
| 移动端 | 是/否 | 高/中/低 | {具体改动} |
| 第三方集成 | 是/否 | 高/中/低 | {具体改动} |

**迁移方案**：
1. {步骤 1}
2. {步骤 2}

**回滚方案**：
- {如何回滚}

**时间计划**：
- 通知消费方：{日期}
- 并行期开始：{日期}
- 旧版本下线：{日期}
```

### 自动化兼容性检查建议

对于有 OpenAPI 文件版本管理的项目，可以使用以下检查逻辑：

```
比较 old.yaml vs new.yaml：
  1. 列举所有删除的路径 → 🔴 破坏性
  2. 列举所有删除的字段 → 🔴 破坏性
  3. 列举所有类型变更 → 🔴 破坏性
  4. 列举新增的 required 字段 → 🔴 破坏性（请求）/ 🟢 安全（响应）
  5. 列举所有新增的路径 → 🟢 安全
  6. 列举所有新增的可选��段 → 🟢 安全
```
