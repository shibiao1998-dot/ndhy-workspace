# 检查清单详版

## 合格前端实现检查清单（详版）

### 一、结构与组件

| # | 检查项 | 通过条件 | 严重性 |
|---|--------|---------|--------|
| 1.1 | 页面层次清晰 | Layout → Page → Section → Component 层次分明 | 🔴 必须 |
| 1.2 | 组件职责单一 | 每个组件的职责可以一句话说清 | 🔴 必须 |
| 1.3 | 组件大小合理 | 单个组件文件不超过 200 行（建议 150 行内） | 🟡 建议 |
| 1.4 | Props 接口明确 | 每个组件的 Props 有 TypeScript interface 定义 | 🔴 必须 |
| 1.5 | 命名自解释 | 组件名、文件名、变量名不需要注释就能理�� | 🔴 必须 |
| 1.6 | 无重复逻辑 | 相同逻辑不超过 2 处（超过需抽取 Hook/组件） | 🟡 建议 |
| 1.7 | 路由组织规范 | 路由层级与页面组织一致，无平铺路由 | 🟡 建议 |

### 二、状态管理

| # | 检查项 | 通过条件 | 严重性 |
|---|--------|---------|--------|
| 2.1 | 状态分层正确 | 服务端/客户端/UI/URL 状态使用对应方案 | 🔴 必须 |
| 2.2 | 无状态污染 | UI 状态不混入业务状态 store | 🔴 必须 |
| 2.3 | 服务端状态用专用库 | 使用 React Query/SWR，不手动 useState+useEffect | 🟡 建议 |
| 2.4 | URL 状态可分享 | 筛选/分页条件反映在 URL 中 | 🟡 建议 |
| 2.5 | 无全局状态滥用 | 只有真正需要跨组件共享的才放全局 | 🔴 必须 |
| 2.6 | 表单状态隔离 | 表单状态在表单组件/Hook 内，不在全局 store | 🟡 建议 |

### 三、API 集成

| # | 检查项 | 通过条件 | 严重性 |
|---|--------|---------|--------|
| 3.1 | 契约一致 | endpoint/字段/类型与 API 契约文档完全匹配 | 🔴 必须 |
| 3.2 | 类型定义完整 | 每个 API 的 Request/Response 有 TypeScript 类型 | 🔴 必须 |
| 3.3 | 错误统一处理 | 有统一的错误拦截和转换层 | 🔴 ���须 |
| 3.4 | 无硬编码 URL | API 地址通过环境变量配置 | 🔴 必须 |
| 3.5 | 认证正确 | Token/Cookie 注入方式正确 | 🔴 必须 |
| 3.6 | 有适配层 | 接口返回结构与前端模型不一致时有 adapter | 🟡 建议 |

### 四、全状态覆盖

| # | 检查项 | 通过条件 | 严重性 |
|---|--------|---------|--------|
| 4.1 | Loading 状态 | 每个数据加载点都有 loading 态（骨架屏/spinner） | 🔴 必须 |
| 4.2 | Empty 状态 | 空数据有友好提示和引导操作 | 🔴 必须 |
| 4.3 | Error 状态 | 错误有用户可理解的提示和可操作的恢复方式 | 🔴 必须 |
| 4.4 | Success 反馈 | 操作成功有 toast/redirect/刷新反馈 | 🔴 必须 |
| 4.5 | Disabled 状态 | 无权限/条件不满足时正确禁用 | 🟡 建议 |
| 4.6 | 空态区分 | "无数据" vs "筛选无结果" vs "权限不足"有不同展示 | 🟡 建议 |

### 五、表单与交互

| # | 检查项 | 通过条件 | 严重性 |
|---|--------|---------|--------|
| 5.1 | 校验覆盖完整 | 必填/格式/范围/关联/服务端校验都有 | 🔴 必须 |
| 5.2 | 错误回显友好 | 字段级错误提示在字段旁，不只是顶部一个错误 | 🔴 必须 |
| 5.3 | 提交不清空 | 提交失败不清空用户已填的内容 | 🔴 必须 |
| 5.4 | 防重复提交 | 提交按钮在请求中 loading + disabled | 🔴 必须 |
| 5.5 | 服务端错误映射 | 服务端校验错误能映射到对应表单字段 | 🟡 建议 |
| 5.6 | 弹窗层级合理 | 弹窗嵌套不超过 2 层 | 🟡 建议 |

## 使用方式

### 快速自检（< 5 分钟）

只检查所有 🔴 必须项（约 25 项）。适用于小改动和 Bug 修复。

### 完整检查（15-30 分钟）

检查所有项。适用于新功能和重要改版。

### 审查模式

交给代码审查专家时，按此清单逐项对照。
