# 标准开发流

## 1. 适用场景

- 新产品从 0 到 1 开发
- 现有产品的重大功能模块新增
- 重要改版（涉及架构或交互大幅调整）

**不适用**：线上 bug 修复（→ `quick-fix.md`）、纯调研（→ `research-analysis.md`）

## 2. 流程图

```
Phase 1 — 需求定义
  需求分析 → 战略定位 → ⬡M1

Phase 2 — 方案设计
  产品定义 → 体验设计 → 技术架构 → ⬡M2

Phase 3 — 详细设计（并行）
  [数据库设计 ∥ API设计] → ⬡M3

Phase 4 — 开发实现（并行）
  [后端开发 ∥ 前端开发] → ⬡M4

Phase 5 — 质量保障
  联调集成 → 代码审查 → 测试验证 → ⬡M5

Phase 6 — 发布上线
  [部署运维 ∥ 发布策略] → ⬡M6
```

## 3. 节点详情表

| 节点 | 负责专家 | 输入 | 输出 | 前置依赖 |
|------|----------|------|------|----------|
| 需求分析 | 🎯 需求分析专家 | 人类原始需求/想法 | 结构化需求文档 | 无 |
| 战略定位 | 🧭 战略定位专家 | 需求文档 + 调研数据（若有） | 战略定位方案（方向/定位/边界） | 需求分析 |
| 产品定义 | 📐 产品定义专家 | 战略定位方案 | 产品定义文档（概念/场景/功能/规则/MVP） | 战略定位 |
| 体验设计 | 🎨 体验设计专家 | 产品定义文档 | 体验设计方案（架构/流程/布局/规范） | 产品定义 |
| 技术架构 | 🏛️ 技术架构专家 | 产品定义 + 体验设计方案 | 架构方案（分层/选型/约束/ADR） | 体验设计 |
| 数据库设计 | 🗄️ 数据库设计专家 | 架构方案 + 产品定义 | 数据模型（ER/DDL/字段字典/迁移策略） | 技术架构 |
| API 设计 | 📜 API设计专家 | 架构方案 + 产品定义 | 接口契约（OpenAPI/错误模型/Mock） | 技术架构 |
| 后端开发 | ⚙️ 后端开发专家 | 数据模型 + 接口契约 + 架构约束 | 后端服务代码 + 单元测试 | 数据库设计 + API设计 |
| 前端开发 | 🖥️ 前端开发专家 | 体验方案 + 接口契约 | 前端代码 + 组件测试 | 体验设计 + API设计 |
| 联调集成 | 🔗 联调集成专家 | 前后端代码 + 接口契约 | 联调报告（一致性/问题归因/回归清单） | 后端开发 + 前端开发 |
| 代码审查 | 🔍 代码审查专家 | 前后端代码 + 联调报告 | 审查报告（分级意见/放行决策） | 联调集成 |
| 测试验证 | 🧪 测试专家 | 可运行系统 + 需求文档 | 测试报告（用例/缺陷/回归/发布就绪） | 代码审查 |
| 部署运维 | 🚀 部署运维专家 | 构建产物 + 环境配置 | 发布就绪确认（环境/回滚/监控） | 测试验证 |
| 发布策略 | 📦 发布策略专家 | 产品定义 + 测试结论 | 发布方案（灰度/培训/沟通/评估） | 测试验证 |

## 4. 并行策略

| 并行组 | 节点 | 协调要点 |
|--------|------|----------|
| Phase 3 | 数据库设计 ∥ API设计 | 共享技术架构约束；字段语义需对齐（接口字段 ≠ DB字段，但语义一致） |
| Phase 4 | 后端开发 ∥ 前端开发 | 基于同一份 API 契约；前端可先用 Mock 数据开发；约定联调时间点 |
| Phase 6 | 部署运维 ∥ 发布策略 | 部署运维管技术侧（环境/回滚/监控）；发布策略管产品侧（灰度/培训/沟通） |

**提前并行机会**：
- 体验设计与技术架构可部分并行（体验设计不依赖技术细节时）
- 前端开发可在 API 设计完成后立即启动（不必等数据库设计完成）

## 5. 裁剪指南

| 场景 | 裁剪方式 |
|------|----------|
| 已有 PRD/需求文档 | 跳过需求分析，从战略定位或产品定义开始 |
| 战略方向已定 | 跳过战略定位，从产品定义开始 |
| 小功能迭代 | 合并 Phase 1+2 为简化版；跳过战略定位 |
| 纯后端服务（无界面） | 跳过体验设计和前端开发 |
| 技术方案已定 | 跳过技术架构，直接进入详细设计 |
| 紧急需求 | 代码审查与测试合并执行；发布策略精简 |

## 6. 里程碑建议

| 里程碑 | 节点 | 检查内容 |
|--------|------|----------|
| ⬡ M1 需求锁定 | 战略定位完成后 | 方向清晰、边界明确、验收标准定义 |
| ⬡ M2 方案定稿 | 技术架构完成后 | 产品+体验+架构三方对齐、无冲突 |
| ⬡ M3 契约冻结 | 详细设计完成后 | DB模型与API契约语义对齐、可进入开发 |
| ⬡ M4 开发完成 | 前后端开发完成后 | 代码可编译运行、单元测试通过 |
| ⬡ M5 质量达标 | 测试验证完成后 | 无 P0/P1 缺陷、回归通过、发布就绪 |
| ⬡ M6 发布完成 | 上线并观察后 | 系统稳定、用户可用、监控正常 |
