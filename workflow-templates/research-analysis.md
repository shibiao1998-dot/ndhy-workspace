# 调研分析流

## 1. 适用场景

- 新项目启动前的市场可行性评估
- 需要了解竞争格局、用户需求、市场趋势
- 为战略决策提供数据支撑
- 现有产品面临方向性选择，需要调研辅助决策

**不适用**：方向已定直接开发（→ `standard-dev.md`）、单纯技术调研（直接派 🔬 技术调研专家）

## 2. 流程图

```
Phase 0 — 调研并行（三路并行）
  [用户研究 ∥ 市场评估 ∥ 竞品调研] → ⬡M0

Phase 1 — 需求整合
  需求分析（整合三路调研） → ⬡M1

Phase 2 — 战略收敛
  战略定位 → 产品定义 → ⬡M2
```

## 3. 节点详情表

| 节点 | 负责专家 | 输入 | 输出 | 前置依赖 |
|------|----------|------|------|----------|
| 用户研究 | 🔍 用户研究专家 | 目标用户群描述、已有用户数据 | 用户画像、用户旅程地图、需求优先级 | 无 |
| 市场评估 | 📊 市场评估专家 | 目标市场描述、行业关键词 | 市场规模(TAM/SAM/SOM)、竞争格局、趋势判断 | 无 |
| 竞品调研 | 🔭 竞品调研专家 | 竞品清单（或由市场评估识别） | 竞品拆解报告（功能/策略/定价/差异化） | 无（或市场评估提供竞品清单） |
| 需求分析 | 🎯 需求分析专家 | 三路调研产出 + 人类原始需求 | 结构化需求文档（整合用户/市场/竞品视角） | 用户研究 + 市场评估 + 竞品调研 |
| 战略定位 | 🧭 战略定位专家 | 需求文档 + 三路调研数据 | 战略定位方案（方向/定位/价值主张/边界） | 需求分析 |
| 产品定义 | 📐 产品定义专家 | 战略定位方案 | 产品定义文档（概念/场景/功能/MVP） | 战略定位 |

## 4. 并行策略

| 并行组 | 节点 | 协调要点 |
|--------|------|----------|
| Phase 0 | 用户研究 ∥ 市场评估 ∥ 竞品调研 | 三路独立启动，无直接依赖；项目管理专家统一下发调研范围和关键问题 |

**协调细节**：
- **启动对齐**：三路调研前由项目管理专家统一明确——调研的核心问题是什么、目标市场/用户群边界
- **中间同步**：市场评估可在过程中输出竞品清单给竞品调研专家，形成「先广后深」
- **汇聚规则**：三路全部完成后才进入 Phase 1；若某路延迟，其他路先交付、延迟路追加

## 5. 裁剪指南

| 场景 | 裁剪方式 |
|------|----------|
| 只需竞品分析 | Phase 0 只保留竞品调研；跳过 Phase 1-2 |
| 只需市场判断 | Phase 0 只保留市场评估；结论直接交 Leader |
| 用户群已明确 | 跳过用户研究；Phase 0 保留市场评估 + 竞品调研 |
| 调研后不做产品 | 到 M0/M1 即止，输出调研报告交 Leader 决策 |
| 快速验证方向 | 三路调研精简为快速扫描（1-2 天），战略定位做轻量版 |
| 已有部分调研数据 | 对应调研节点改为"补充调研"，缩短周期 |
| 市场评估后再深挖竞品 | Phase 0 改为串行：市场评估 → 竞品调研（识别 Top 竞品后再深拆） |

## 6. 里程碑建议

| 里程碑 | 节点 | 检查内容 |
|--------|------|----------|
| ⬡ M0 调研完成 | Phase 0 三路汇聚后 | 三路调研产出齐备、核心问题已回答、数据可交叉验证 |
| ⬡ M1 需求锁定 | 需求分析完成后 | 需求已整合三方视角、优先级清晰、无自相矛盾 |
| ⬡ M2 方向确定 | 产品定义完成后 | 战略方向+产品形态已定、可进入设计开发阶段 |

**M0 检查重点**：
- 用户研究、市场评估、竞品调研的结论是否互相印证？
- 有无矛盾信号需要进一步验证？
- 数据质量是否足以支撑战略决策？
