# 快速修复流

## 1. 适用场景

- 线上 bug 修复（影响用户正常使用）
- 小需求/微调（改文案、调样式、加字段等）
- 热修复（Hotfix）紧急上线
- 已有代码库的小范围改动

**不适用**：涉及架构调整（→ `standard-dev.md`）、需要调研的新方向（→ `research-analysis.md`）

## 2. 流程图

```
Phase 1 — 定位
  问题定位 → ⬡M1

Phase 2 — 修复
  修复开发 → ⬡M2

Phase 3 — 验证
  代码审查 → 测试验证 → ⬡M3

Phase 4 — 上线
  部署 → ⬡M4
```

## 3. 节点详情表

| 节点 | 负责专家 | 输入 | 输出 | 前置依赖 |
|------|----------|------|------|----------|
| 问题定位 | ⚙️ 后端开发专家 / 🖥️ 前端开发专家 | Bug 报告 / 需求描述 | 问题原因分析 + 修复方案 | 无 |
| 修复开发 | ⚙️ 后端开发专家 / 🖥️ 前端开发专家 | 修复方案 + 代码库 | 修复代码 + 单元测试 | 问题定位 |
| 代码审查 | 🔍 代码审查专家 | 修复代码 diff | 审查意见（通过/打回） | 修复开发 |
| 测试验证 | 🧪 测试专家 | 修复后系统 + 复现步骤 | 验证结果（修复确认 + 回归通过） | 代码审查 |
| 部署 | 🚀 部署运维专家 | 验证通过的构建产物 | 上线确认 + 回滚方案 | 测试验证 |

**专家选择说明**：
- 问题定位和修复开发：根据问题类型选择后端开发专家或前端开发专家，同一人贯穿
- 简单修复：定位和修复可合并为一个节点

## 4. 并行策略

快速修复流以**串行**为主，原因：
- 修复通常涉及单一模块，并行收益低
- 串行保证每一步质量确认后再进入下一步

**例外并行场景**：

| 场景 | 并行方式 |
|------|----------|
| 前后端同时有 bug | 前端修复 ∥ 后端修复 → 联调确认 → 代码审查 |
| 多个独立 bug 批量修复 | 各 bug 独立走修复流 → 统一代码审查 → 统一测试 → 统一部署 |

## 5. 裁剪指南

| 场景 | 裁剪方式 |
|------|----------|
| 极简修复（改配置/文案） | 跳过代码审查，修复 → 测试 → 部署 |
| 紧急 P0（线上崩溃） | 修复 → 快速验证 → 紧急部署 → 事后补审查 |
| 小需求（非 bug） | 问题定位改为"需求确认"，其余不变 |
| 自信的修复（单行改动） | 代码审查 + 测试合并执行 |
| 前后端联动修复 | 在修复开发后增加联调集成节点（🔗 联调集成专家） |

**P0 紧急通道**：
```
紧急修复 → 最小验证 → 紧急部署 → 事后审查 + 完整回归
```
仅限影响面大、无法等待的场景。事后必须补全审查和回归测试。

## 6. 里程碑建议

| 里程碑 | 节点 | 检查内容 |
|--------|------|----------|
| ⬡ M1 原因明确 | 问题定位完成 | 根因已确认、修复方案可行、影响范围已评估 |
| ⬡ M2 修复完成 | 修复开发完成 | 代码改动最小化、单元测试通过、无引入新问题 |
| ⬡ M3 验证通过 | 测试验证完成 | Bug 已修复、回归无新问题、可安全上线 |
| ⬡ M4 上线确认 | 部署完成 | 线上验证修复有效、监控无异常、回滚方案就绪 |

**快速修复的铁律**：
- 修复范围最小化——只改必须改的，不顺手重构
- 每次修复必须有对应测试用例（防止同类问题再次出现）
- P0 紧急通道用完必须补全质量流程
