# 组织变更流

## 1. 适用场景

- 创建新的 AI 专家角色（SOUL.md + STANDARDS.md）
- 升级/裁撤现有角色
- 开发新的 Skill 或改进现有 Skill
- 流程模板更新、工作手册迭代
- 组织级知识治理（设定 vs 技能的归属调整）

**不适用**：产品开发（→ `standard-dev.md`）、技术调研（直接派 🔬 技术调研专家）

## 2. 流程图

```
Phase 1 — 需求评估
  变更评估 → ⬡M1

Phase 2 — 执行（按类型选择，可并行）
  [角色创造 ∥ Skill开发] → ⬡M2

Phase 3 — 验收
  审查验收 → ⬡M3

Phase 4 — 生效
  全局更新 → ⬡M4
```

## 3. 节点详情表

| 节点 | 负责专家 | 输入 | 输出 | 前置依赖 |
|------|----------|------|------|----------|
| 变更评估 | Leader / 🎯 需求分析专家 | 变更需求描述 | 变更评估结论（做/不做、范围、影响面） | 无 |
| 角色创造 | 🧬 角色创造专家 | 评估结论 + 角色需求描述 | SOUL.md + STANDARDS.md + README 条目 | 变更评估 |
| Skill 开发 | 🔧 Skill开发专家 | 评估结论 + Skill 需求描述 | SKILL.md + 辅助脚本/references | 变更评估 |
| 审查验收 | 🔍 代码审查专家 / Leader | 角色设定 / Skill 文件 | 审查意见（通过/修改/打回） | 角色创造 / Skill开发 |
| 全局更新 | 🏗️ 项目管理专家 / Leader | 审查通过的产出物 | 索引更新、关联文件同步、团队通知 | 审查验收 |

**专家选择说明**：
- Phase 2 根据变更类型选择：角色变更 → 角色创造专家；Skill 变更 → Skill开发专家；两者兼有 → 并行
- 审查验收：Skill 类产出由代码审查专家审查；角色设定类由 Leader 审查
- 全局更新：包括更新 `roles/README.md`、相关 AGENTS.md 引用等

## 4. 并行策略

| 并行组 | 节点 | 协调要点 |
|--------|------|----------|
| Phase 2 | 角色创造 ∥ Skill开发 | 新角色需要新 Skill 时，先确认角色定义再开发 Skill；角色和 Skill 无关联时可完全并行 |

**常见并行场景**：

| 场景 | 并行方式 |
|------|----------|
| 新角色 + 配套 Skill | 角色创造先行（定义专业标准）→ Skill开发跟进（基于标准写操作手册） |
| 多个独立 Skill 开发 | 各 Skill 完全并行 → 统一审查 |
| 角色升级 + Skill 改进 | 先明确角色设定变更 → 再同步调整 Skill |
| 纯 Skill 开发（无角色变更） | Phase 2 只有 Skill开发，跳过角色创造 |

## 5. 裁剪指南

| 场景 | 裁剪方式 |
|------|----------|
| 只创建角色（不需要新 Skill） | Phase 2 只保留角色创造 |
| 只开发/改进 Skill | Phase 2 只保留 Skill 开发 |
| 小改动（修错字/调措辞） | 跳过变更评估，直接修改 → 审查 → 更新 |
| 流程模板更新 | 变更评估 → Leader 直接修改 → 自查 → 更新 |
| 批量角色创建 | 变更评估（统一规划）→ 角色创造（按角色并行）→ 批量审查 |
| 角色裁撤 | 变更评估（影响分析）→ 全局更新（移除索引+归档文件） |

## 6. 里程碑建议

| 里程碑 | 节点 | 检查内容 |
|--------|------|----------|
| ⬡ M1 变更确认 | 变更评估完成 | 变更必要性确认、范围明确、影响面可控 |
| ⬡ M2 产出完成 | 角色/Skill 开发完成 | 产出物符合规范、自检通过 |
| ⬡ M3 审查通过 | 审查验收完成 | 质量达标、无遗漏、与现有体系无冲突 |
| ⬡ M4 全局生效 | 全局更新完成 | 索引已更新、关联引用已同步、团队已知悉 |

**全局更新检查清单**：
- `roles/README.md` 角色索引是否同步？
- 相关 Skill 的 `适用技能` 表是否更新？
- `AGENTS.md` 中是否有需要同步的引用？
- 裁撤的角色/Skill 是否已归档而非删除？
