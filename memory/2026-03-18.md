### ACP 实时监控机制（验证通过，宪法级）
- **问题**：ACP 是黑盒——spawn 后不知道进度、不知道完成、不知道卡死
- **解法三件套**：
  1. `streamTo: "parent"` — spawn 时加，返回 JSONL 日志可实时 tail
  2. `sessions_history` — 可读 ACP 会话完整历史
  3. `sessions_send` — 可向 ACP 追问/纠偏
- **配置变更**：
  - `tools.sessions.visibility`: "tree" → "all"
  - `tools.agentToAgent.enabled`: false → true
  - 两项均为热加载配置，不需要重启 Gateway
- **验证测试**：spawn ACP 任务 + stream 日志读取 + lifecycle 事件追踪，全部通过

## 重要教训（V6 第二轮修复失败）

### ACP 连续修改必须有中间 commit
- **问题**：第一轮 ACP 完成 S3-S7 后直接启动第二轮修复，没有 commit
- **后果**：第二轮导致页面质量下降，回滚时无法只回滚第二轮——git stash 把两轮改动一起回滚
- **规则（宪法级）**：每轮 ACP 修改完成后必须先 git commit 再做下一轮。没有 commit = 没有存档点 = 无法安全回滚
- **当前状态**：两轮修改在 git stash（backup-round2-before-revert）中，页面回到 3/17 EOD 备份

### ACP 浏览器验证循环陷阱
- ACP 在 GSAP pin zone 中反复尝试滚动导致无限循环
- 教训：ACP 浏览器自动化不适合测试 GSAP pin/scrub 页面，改用 DOM 验证 + build 检查即可
