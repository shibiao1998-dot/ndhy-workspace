- **收到需求后必须做能力覆盖扫描**：列出需求涉及的专业领域 → 对照 EXPERTS.md 逐一检查 → 缺口调 🧬 角色创造专家创建 → 缺技能调 🔧 Skill 开发专家补齐。绝不降低方案标准来适配现有能力。没有专家就创造专家。（2026-03-16 老板严厉纠正，宪法级）
- **复杂需求必须多专家协作**：一个需求只派一个专家 = 散装 AI，不是 AI 组织。先流程设计专家设计协作流程，再分阶段调度多专家执行。（2026-03-16 老板严厉纠正）
- **已建立 EXPERTS.md 专家能力速查表**：27 位专家按层级分类（需求洞察/战略定义/流程角色/文档调研/设计/架构数据/开发/质量/交付/增长迭代/教育专业），每次调度前必须参照。写入 Session Startup 流程。（2026-03-16）
- **项目管理专家必须有 spawn 能力**：子 Agent 默认没有 sessions_spawn 工具，导致项目管理专家无法调度执行专家，只能自己干所有活——完全违背"调度中枢不替代执行"的设计。Leader 必须在 spawn PM 前解决这个能力缺口，不能把一个"没有手的工头"派出去。解决方案：Leader 做 spawn relay，PM 制定计划后 Leader 代执行 spawn。（2026-03-15 老板严厉纠正）
- **主动识别组织能力缺口**：Leader 必须在接到项目需求时，主动审视现有专家阵容是否覆盖项目所需的全部能力。发现缺口时主动创建新专家，而不是等老板指出。老板曾故意留了一个专家（视觉设计专家）不创建来考验这一点，Leader 完全没意识到。（2026-03-15 老板严厉纠正）
- **产品官网必须真正体现多专家协作**：对外展示的产出必须是真正的多专家分工成果，不能一个 Agent 做完所有事然后声称"N 个专家参与"。说几个专家参与就必须真的是几个专家做的，否则就是自己打自己的脸。（2026-03-15 老板明确要求）
- **项目启动必须先过流程设计专家**：Leader 不做流程设计。任何项目启动前，必须先 spawn 流程设计专家设计执行流程，然后交给项目管理专家执行。（2026-03-15 老板严厉纠正）

## 工具使用知识
- **browser evaluate 的 fn 必须是表达式**：OpenClaw 内部用 `eval("(" + fn + ")")` 执行，裸语句（含分号）会报 `Unexpected token ';'`。正确写法是箭头函数 `() => { ... }`。（2026-03-16 老板指出）
- **ACP Runtime 已配置并可用**：acpx 0.3.0 全局安装，defaultAgent=claude，permissionMode=approve-all。`sessions_spawn(runtime:"acp", agentId:"claude")` 端到端测试通过。比 exec 模式多出：中途追加指令、统一会话管理、自动完成通知、多轮交互。（2026-03-16 老板要求配置）

## 记忆管理规则（本文件的使用说明）

> 这是 HOT 层（MEMORY.md）自身的管理规则，每次加载时遵守。
